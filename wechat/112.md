上一篇我们把 OpenClaw 拆了个底朝天——架构、故事、安全、中国生态、微信困境，全聊了一遍。

这篇是实战篇。手把手把 OpenClaw 装起来，跑起来，聊起来。

环境：Windows 11 + WSL2 Ubuntu + Docker + Gemini（免费额度）。整个过程不花一分钱（Telegram 注册除外，那个要 10 块钱接码费）。

先说结论：如果你跟着最优路径走，大概 30 分钟能装完。但如果你像我一样每个坑都踩一遍，大概需要一整个下午。

所以这篇文章分两部分：第一部分是最优路径，跟着走就行；第二部分是踩坑记录，解释为什么要这么做。赶时间的看第一部分就够了。

◆ 最优路径：30 分钟装完

━━━━━━━━━━━━━━━━━━━━

────────────────────

**第零步：前置环境**

你需要：
- WSL2 Ubuntu（Windows 搜索栏搜 "Ubuntu"，没装过的去 Microsoft Store 装一个）
- 代理能翻墙（后面要访问 GitHub、Google OAuth、Telegram API）
- 一个 Google 账号（用 Gemini 的免费额度）

代理这事不展开说了，你懂的。假设你的代理跑在 Windows 上，端口 7897，WSL 里通过 `http://127.0.0.1:7897` 访问。

在 WSL 里先把代理环境变量设好：

```bash
export http_proxy=http://127.0.0.1:7897
export https_proxy=http://127.0.0.1:7897
export all_proxy=socks5://127.0.0.1:7897
export no_proxy=localhost,127.0.0.1
```

建议写进 `~/.bashrc`，省得每次开终端都敲。

────────────────────

**第一步：装 Docker**

WSL2 里用 snap 装 Docker，一行命令：

```bash
sudo snap install docker
```

装完之后，你会发现 `docker ps` 报权限错误。snap 版 Docker 默认没有 docker 组，需要手动创建：

```bash
sudo addgroup --system docker
sudo adduser $USER docker
newgrp docker
sudo snap restart docker
sudo chown root:docker /var/run/docker.sock
```

验证一下：

```bash
docker ps
```

不报错就行。

────────────────────

**第二步：装 Node.js 和 Gemini CLI**

OpenClaw 需要 Node.js 22 以上：

```bash
# 如果还没装 Node.js，用 nvm 装
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.3/install.sh | bash
source ~/.bashrc
nvm install 22
```

装 Gemini CLI：

```bash
sudo npm install -g @google/gemini-cli
```

装完跑一次 `gemini`，完成 Google OAuth 登录。这一步很重要——后面要从 Gemini CLI 的本地文件里提取 OAuth 凭据给 OpenClaw 用。

```bash
gemini
```

浏览器会弹出 Google 登录页面，授权完成后终端会显示成功。随便问它一个问题验证能用，然后 Ctrl+C 退出。

────────────────────

**第三步：克隆 OpenClaw 并构建 Docker 镜像**

```bash
mkdir -p ~/work && cd ~/work
git clone https://github.com/openclaw/openclaw.git
cd openclaw
bash docker-setup.sh
```

`docker-setup.sh` 会做这些事：
1. 构建 Docker 镜像 `openclaw:local`——首次构建比较慢，26 万行 TypeScript 项目，依赖多，还要装 Playwright/Chromium。喝杯咖啡的时间
2. 运行 onboarding 引导（交互式配置）
3. 生成 gateway token
4. docker-compose 启动两个服务

────────────────────

**第四步：Onboarding 引导——按这个选**

构建完成后会进入交互式配置。按下面这么选：

**模式**：选 QuickStart

**Model/Auth Provider**：选 Google（Gemini）。会问你 OAuth 还是 API key——这里先选 Skip。原因后面踩坑记录里说，简单讲就是：容器里的 OAuth 流程在 WSL2 + 代理环境下跑不通。

**模型**：Filter 选 google-gemini-cli，模型选 gemini-3-flash-preview（或 gemini-2.5-flash 求稳）

**聊天渠道**：推荐 Telegram (Bot API)。配置方法：
1. 打开 Telegram，搜索 @BotFather
2. 发送 `/newbot`，按提示给 bot 起名
3. 拿到 token（格式：`123456789:ABCdefGHI-jklMNOpqrsTUVwxyz`）
4. 粘贴到终端

不想配 Telegram 的选 Skip for now，Web UI 也能用。

**其余**：Skills、API keys、Hooks 全部 Skip for now。

Onboarding 结束后，会生成一个 gateway token，记下来。

────────────────────

**第五步：改 docker-compose.yml（关键）**

这一步是整个部署最关键的改动。官方的 docker-compose.yml 在中国的 WSL2 + 代理环境下跑不通，需要改三个地方。

打开 `docker-compose.yml`，对两个 service（openclaw-gateway 和 openclaw-cli）都做以下修改：

**1. 网络模式改成 host**

删掉 `ports` 配置（如果有的话），加上：

```yaml
network_mode: host
```

原因：容器默认网络隔离，访问不到 WSL 的代理。host 模式让容器直接用 WSL 的网络栈。

**2. 加代理环境变量**

在 `environment` 段加：

```yaml
http_proxy: ${http_proxy:-}
https_proxy: ${https_proxy:-}
all_proxy: ${all_proxy:-}
no_proxy: ${no_proxy:-}
```

**3. gateway 的 `--bind` 改成 loopback**

host 网络模式下不能用 `--bind lan`，改成：

```yaml
command:
  [
    "node", "dist/index.js", "gateway",
    "--bind", "loopback",
    "--port", "18789",
  ]
```

改完之后，还需要在 `~/.openclaw/openclaw.json` 的 `gateway` 段加一行，解决 allowedOrigins 报错：

```json
"controlUi": {
  "dangerouslyAllowHostHeaderOriginFallback": true
}
```

────────────────────

**第六步：手动配置 Gemini OAuth（最核心的一步）**

这是整个部署最核心的坑，也是我花时间最多的地方。OpenClaw 容器里没有 Gemini CLI，它的交互式 OAuth 流程在 Docker + WSL2 + 代理环境下基本跑不通。**正解是手动构造 auth-profiles.json。**

分六个小步骤：

**6.1 从宿主机的 Gemini CLI 提取 OAuth client credentials：**

```bash
# 提取 Client ID
grep -oP '\d+-[a-z0-9]+\.apps\.googleusercontent\.com' \
  /usr/lib/node_modules/@google/gemini-cli/node_modules/@google/gemini-cli-core/dist/src/code_assist/oauth2.js

# 提取 Client Secret
grep -oP 'GOCSPX-[A-Za-z0-9_-]+' \
  /usr/lib/node_modules/@google/gemini-cli/node_modules/@google/gemini-cli-core/dist/src/code_assist/oauth2.js
```

记下这两个值，后面要用。

**6.2 从宿主机的 Gemini CLI 获取 refresh_token：**

```bash
cat ~/.gemini/oauth_creds.json | python3 -c "import sys,json; print(json.load(sys.stdin)['refresh_token'])"
```

**6.3 用 refresh_token 换取 access_token：**

```bash
curl -s -X POST https://oauth2.googleapis.com/token \
  -d "client_id=<你的CLIENT_ID>" \
  -d "client_secret=<你的CLIENT_SECRET>" \
  -d "refresh_token=<你的REFRESH_TOKEN>" \
  -d "grant_type=refresh_token"
```

返回的 JSON 里有 `access_token` 和 `expires_in`。

**6.4 用 access_token 获取 projectId：**

```bash
curl -s -X POST "https://cloudcode-pa.googleapis.com/v1internal:loadCodeAssist" \
  -H "Authorization: Bearer <ACCESS_TOKEN>" \
  -H "Content-Type: application/json" \
  -H 'Client-Metadata: {"ideType":"ANTIGRAVITY","platform":"PLATFORM_UNSPECIFIED","pluginType":"GEMINI"}' \
  -d '{"metadata":{"ideType":"ANTIGRAVITY","platform":"PLATFORM_UNSPECIFIED","pluginType":"GEMINI"}}'
```

返回的 `cloudaicompanionProject` 就是 projectId。

**6.5 构造 auth-profiles.json：**

创建 `~/.openclaw/auth-profiles.json`：

```json
{
  "version": 1,
  "profiles": {
    "google-gemini-cli": {
      "type": "oauth",
      "provider": "google-gemini-cli",
      "access": "<ACCESS_TOKEN>",
      "refresh": "<REFRESH_TOKEN>",
      "expires": <当前时间戳毫秒 + expires_in * 1000>,
      "projectId": "<PROJECT_ID>",
      "email": "<你的GMAIL>",
      "clientId": "<CLIENT_ID>"
    }
  }
}
```

**注意**：这个文件需要写到两个位置：
- `~/.openclaw/auth-profiles.json`
- `~/.openclaw/agents/main/agent/auth-profiles.json`

两个内容一样，直接复制过去。

**6.6 在 docker-compose.yml 中注入 Gemini OAuth 凭据：**

在两个 service 的 `environment` 段加：

```yaml
GEMINI_CLI_OAUTH_CLIENT_ID: "<你的CLIENT_ID>"
GEMINI_CLI_OAUTH_CLIENT_SECRET: "<你的CLIENT_SECRET>"
```

这样容器里的 OpenClaw 就能自动刷新 access_token 了。

同时把宿主机的 `~/.gemini` 目录挂载到容器里。在 gateway 服务的 `volumes` 段加：

```yaml
- ${HOME}/.gemini:/home/node/.gemini
```

────────────────────

**第七步：启动**

```bash
docker compose down && docker compose up -d
```

启动后访问 Web UI：http://localhost:18789

会要求输入 gateway token（就是 onboarding 生成的那个）。输入后就能看到管理界面了。

如果配了 Telegram，给你的 bot 发一条消息，bot 会回复一个 pairing code。执行：

```bash
docker exec -it openclaw-openclaw-gateway-1 node dist/index.js pairing approve telegram <PAIRING_CODE>
```

配对完成后，就可以在 Telegram 里和你的 AI agent 聊天了。

Web UI 里也有聊天界面，直接对话也行。

完事。

◆ 踩坑记录：为什么要这么做

━━━━━━━━━━━━━━━━━━━━

下面按坑的严重程度排列。如果你跟着最优路径走没出问题，这部分可以当消遣看。

────────────────────

**坑一：Gemini OAuth——整个部署最大的坑**

严重程度：致命。不解决这个，agent 完全没法用。

OpenClaw 的 `openclaw configure` 有一套交互式 OAuth 流程，理论上应该能一键搞定 Gemini 认证。但在 Docker + WSL2 + 代理的环境下，这套流程的每一步都有问题：

1. 容器里没有宿主机的 gemini-cli，OAuth 流程直接跑不起来
2. 在容器里装 gemini-cli？npm 权限问题（non-root 用户），就算装到 `~/local` 目录，容器重建后就丢了
3. 把宿主机的 gemini-cli 挂载进去？snap 版是只读文件系统，复制出来也各种路径问题
4. 用 `openclaw configure` 的交互式 OAuth？WSL2 被检测为 remote 环境，手动贴 URL，终端还会截断 URL，粘贴不完整

试了一圈，全部失败。

最终解法是读 OpenClaw 的源码，发现它不读 `~/.gemini/oauth_creds.json`（那是 gemini-cli 自己的），而是用自己的 `auth-profiles.json` 格式。于是手动调 Google API，一步步拿到所有需要的凭据，直接构造 JSON 文件。

这个过程说起来轻描淡写，实际上包含了：读 gemini-cli 的 oauth2.js 源码 grep 出 client_id 和 client_secret、调 Google 的 token 端点换 access_token、调一个半公开的 `loadCodeAssist` 端点拿 projectId。每一步都是靠读代码和试 API 摸出来的。

所以最优路径里第六步写得那么详细，就是因为这条路是血泪换来的。

────────────────────

**坑二：容器网络隔离——容器上不了网**

严重程度：高。不解决的话容器访问不了任何外部 API。

Docker 默认的 bridge 网络模式下，容器有自己的网络栈。你的代理跑在 Windows 上（通过 WSL 的 127.0.0.1:7897 转发），容器根本访问不到这个地址。

一开始的想法是在 docker-compose 里配代理环境变量指向宿主机 IP。但 WSL2 的网络拓扑是这样的：Windows 有自己的 IP，WSL 有自己的 IP，Docker 容器又有自己的 IP。三层网络，地址转来转去，极其折腾。

最简单的解法：`network_mode: host`。容器直接用 WSL 的网络栈，代理环境变量传进去就能用。一行配置解决三天的网络调试。

副作用是容器没有端口隔离了，但反正是本地开发环境，无所谓。

────────────────────

**坑三：snap 版 Docker 的权限问题**

严重程度：中。初次遇到会懵，但修完就好了。

为什么用 snap 版？因为 `sudo snap install docker` 一行命令，静默安装，不像 Docker Desktop 那样折腾。Docker Desktop 本质上也是在 WSL2 里跑 Linux Docker（它会偷偷创建一个自己的 WSL2 虚拟机），但装起来又重又慢。

snap 版的问题是：没有自动创建 docker 组。你 `docker ps` 会报 permission denied，因为 `/var/run/docker.sock` 的权限是 root:root。

修法就是最优路径第一步写的那些命令。有一个坑中坑：`newgrp docker` 只在当前 shell session 生效。你新开一个终端窗口，又会报权限错误。要么每次 `newgrp docker`，要么直接 `sudo chown root:docker /var/run/docker.sock`。

────────────────────

**坑四：Gateway 启动失败——allowedOrigins**

严重程度：中。报错信息比较清楚，知道了就好改。

docker-compose 默认用 `--bind lan`（绑定所有网络接口），但非 loopback 绑定需要配置 `controlUi.allowedOrigins`，否则 gateway 拒绝启动。

改法两步：`--bind` 改 loopback，然后在配置文件里加 `dangerouslyAllowHostHeaderOriginFallback: true`。名字里有 "dangerously" 看着吓人，但本地环境用完全没问题。

────────────────────

**坑五：Web UI 被锁——无限限流循环**

严重程度：中低。不影响 Telegram 使用，只影响 Web UI。

Web UI 有认证限流机制：10 次认证失败后锁定 5 分钟。听起来很合理。

但实际情况是：Windows 浏览器通过 portproxy 访问 WSL 里的 OpenClaw，IP 不是 loopback（127.0.0.1 本来是免限流的，但 portproxy 转发后 IP 变了），所以会被限流。

更要命的是，浏览器的 WebSocket 会自动重连。你输错一次 token，浏览器在后台疯狂重连，每次重连都用错误的 token，几秒钟就打满 10 次限流。然后等 5 分钟解锁，浏览器又立刻重连打满——永远解不了锁。

你换 token、重启容器都没用，因为浏览器标签页还开着，一重启就自动重连。

解法：
1. 关掉所有浏览器标签页（这一步最关键）
2. 临时改 `~/.openclaw/openclaw.json` 中 `gateway.auth.mode` 为 `"none"`
3. `docker compose down && docker compose up -d`
4. 用无痕模式打开 Web UI，进去后再改回 `"token"` 模式

这个坑不踩不知道，踩了之后你会对"浏览器自动重连"这个平时觉得挺好的功能产生深深的怨念。

────────────────────

**坑六：Telegram 注册——中国手机号收不到验证码**

严重程度：低。花 10 块钱就能解决。

中国手机号注册 Telegram 收不到短信验证码，这是老问题了。解法是用第三方接码服务，找一个海外虚拟号码接收验证码。大概 43 台币，折合人民币 10 块左右。

如果你已经有 Telegram 账号，跳过这个坑。如果不想用 Telegram，Web UI 也能聊天，只是没有"AI 住在聊天工具里"那种感觉。

**关于 Windows 和 Docker 的关系**

很多人以为 Windows 上跑 Docker 需要装 Docker Desktop。其实 Docker Desktop 本质上就是偷偷建一个 WSL2 虚拟机，在里面跑 Linux Docker。你直接在 WSL2 Ubuntu 里 `snap install docker`，效果一样，还少了一层套娃。

套娃层数：Windows -> WSL2 虚拟机 -> Docker 容器 -> OpenClaw -> 代理翻墙 -> Telegram API -> Google OAuth -> Gemini API。

每一层都可能出问题。在中国装 OpenClaw，属于地狱难度。

────────────────────

**关于 Gemini 免费额度**

选 Gemini 而不是 OpenAI 或 Claude 的原因很简单：有免费额度。Gemini CLI 通过 OAuth 登录走 Google 订阅额度，日常使用基本够了。如果用 API key 则按 token 计费。

一个 AI Agent 平台，底层 LLM 不花钱，这个性价比还是可以的。

────────────────────

**关于 `openclaw configure` 的交互式流程**

OpenClaw 设计了一套很完善的交互式配置流程（onboarding + configure），在正常环境下应该是挺顺滑的。但"正常环境"指的是 macOS 或者 Linux 桌面环境，直接装在宿主机上。

换到 Docker + WSL2 + 代理这种套娃环境，这套交互式流程就完全水土不服了。终端截断 URL、OAuth 回调 localhost 不通、容器里没有浏览器……每一步都是坑。

教训是：**自动化流程跑不通的时候，手写 JSON 反而最快。**
