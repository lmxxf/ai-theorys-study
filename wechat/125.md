◆ 第二段：ArkTS ↔ Python 双向管道通信

━━━━━━━━━━━━━━━━━━━━

这是本篇的核心。

────────────────────

【问题：Python 只能被调用，不能调用】

之前的模式：

```
ArkTS → NAPI fork+execve → Python 执行脚本 → 输出结果 → 进程退出
```

**Python 是一个被动的、一次性的工具。** ArkTS 能调 Python，但 Python 不能反过来调 ArkTS。

这意味着：如果你希望 Python 的 AI Agent 能控制鸿蒙设备——比如"帮我拍个照"、"打开蓝牙"、"读一下温度传感器"——**做不到。** Python 跑完就退了，没机会和设备交互。

我们需要的是：

```
ArkTS ←→ Python（常驻进程）
  ArkTS 发消息给 Python（"用户说了什么"）
  Python 发消息给 ArkTS（"我需要拍个照"）
  ArkTS 执行完拍照，把结果发回 Python
  Python 拿到结果，继续处理
```

**双向通信。Python 不退出，作为常驻子进程（daemon）活着。**

────────────────────

💡 **daemon**（守护进程）：在后台持续运行的进程，不会自己退出，等着接活。类比：你叫了个外卖小哥，他不是送完一单就下线，而是一直在线等新订单。

────────────────────

【方案对比】

让 ArkTS 和 Python 在同一台设备上双向通信，有三种主流做法：

| 方案 | 原理 | 优点 | 缺点 |
|:---|:---|:---|:---|
| 文件轮询 | 双方读写共享文件 | 最简单 | 延迟高、要加锁、浪费磁盘 IO |
| stdin/stdout 管道 | 父进程持有子进程的标准输入输出 | 零延迟、系统原生支持、不需要额外端口 | 子进程必须由父进程 fork |
| Unix Socket | 本地套接字通信 | 灵活、支持多客户端 | 需要绑定路径、处理连接生命周期 |

**选了 stdin/stdout 管道。** 理由很简单：

1. Python 本来就是 NAPI fork 出来的子进程——父进程天然持有 stdin/stdout 的管道两端
2. 不需要文件路径，不需要端口号，不需要额外的权限
3. 通信延迟约等于零——就是内核的 pipe buffer，几微秒级别

────────────────────

💡 **stdin/stdout 管道**：Unix 系统的进程间通信基础设施。父进程 fork 出子进程时，可以用 `pipe()` 创建管道——父进程写一头，子进程从另一头读（这是 stdin）；反过来，子进程写一头，父进程从另一头读（这是 stdout）。就像两根单向的水管，一根往里灌水，一根往外流水。

────────────────────

【C++ NAPI 层：新增 5 个函数】

之前的 `exec_napi.cpp` 只有一个函数 `runCommand(cmd)`——跑完就退。现在加了 5 个新函数，构成完整的 daemon 控制 API：

| 函数 | 作用 |
|:---|:---|
| `startDaemon(pythonBase, scriptPath)` | fork Python 子进程，建立 stdin/stdout 双管道 |
| `sendMessage(json)` | 往 Python 的 stdin 写一行 JSON |
| `readMessage()` | 从 Python 的 stdout 读一行 JSON（阻塞等待） |
| `stopDaemon()` | 先 SIGTERM 优雅退出，500ms 后 SIGKILL 强杀 |
| `isDaemonRunning()` | 检查子进程是否还活着（`waitpid` + `WNOHANG`） |

核心是 `startDaemon`。看看它做了什么：

```cpp
// 创建两根管道
int pipe_in[2];   // ArkTS → Python（Python 的 stdin）
int pipe_out[2];  // Python → ArkTS（Python 的 stdout）
pipe(pipe_in);
pipe(pipe_out);

pid_t pid = fork();

if (pid == 0) {
    // 子进程（Python）
    dup2(pipe_in[0], STDIN_FILENO);    // 管道读端 → stdin
    dup2(pipe_out[1], STDOUT_FILENO);  // 管道写端 → stdout
    dup2(pipe_out[1], STDERR_FILENO);  // stderr 也导到管道

    setenv("PYTHONUNBUFFERED", "1", 1);  // 关键：关闭输出缓冲

    execve(pythonBin, args, environ);
}

// 父进程（NAPI / ArkTS）
g_stdin_fd = pipe_in[1];   // 持有写端 → 能往 Python stdin 写
g_stdout_fd = pipe_out[0]; // 持有读端 → 能从 Python stdout 读
```

**两根管道，四个端口。** 父进程关掉不用的那两个，子进程关掉不用的那两个。剩下的就是一条干净的双向通道。

────────────────────

有一个细节容易踩坑：**`PYTHONUNBUFFERED=1`**。

Python 默认会缓冲 stdout 输出——攒够一定量才真正写出去。如果不关缓冲，NAPI 那边 `readMessage()` 会一直阻塞，因为 Python 的输出还留在缓冲区里，没真正写进管道。

**设 `PYTHONUNBUFFERED=1`，或者用 `python -u`，让每次 `print()` 立刻写出去。**

────────────────────

【Python 端：agent_daemon.py】

Python 这边很简单——一个消息循环：

```python
import sys
import json

def send(obj):
    """发 JSON 到 ArkTS（写 stdout）"""
    sys.stdout.write(json.dumps(obj, ensure_ascii=False) + '\n')
    sys.stdout.flush()

def recv():
    """从 ArkTS 收 JSON（读 stdin）"""
    line = sys.stdin.readline()
    if not line:
        return None  # 管道关了
    return json.loads(line.strip())

def main():
    send({"type": "ready", "python_version": sys.version.split()[0]})

    while True:
        msg = recv()
        if msg is None:
            break

        if msg['type'] == 'chat':
            handle_chat_with_tools(msg['message'])
        elif msg['type'] == 'ping':
            send({"type": "pong"})
        elif msg['type'] == 'shutdown':
            send({"type": "shutdown_ack"})
            break
```

**启动 → 发 ready → 进入循环 → 等消息 → 处理 → 回消息 → 继续等。** 不退出。

────────────────────

【通信协议：JSON Lines】

💡 **JSON Lines** 是一种简单的文本协议——每行一个完整的 JSON 对象，用换行符 `\n` 分隔。不需要包头、不需要长度字段、不需要二进制解析。

```
{"type": "chat", "message": "你好"}       ← ArkTS 发给 Python
{"type": "response", "content": "你好！"}  ← Python 回给 ArkTS
```

为什么选 JSON Lines 不选 protobuf 或 MessagePack？

**因为够用。** 管道通信是同一台设备上两个进程之间的事，带宽不是瓶颈。JSON 可读可调试，出了问题 `LOGI` 一打就能看到完整的消息内容。复杂协议在这个阶段是过度设计。

────────────────────

【消息类型一览】

**ArkTS → Python：**

| type | 说明 |
|:---|:---|
| `chat` | 用户输入。附带 `message` 字段 |
| `tool_result` | ArkTS 执行完工具后返回的结果。附带 `tool` 和 `result` |
| `ping` | 心跳检测 |
| `shutdown` | 要求 Python 退出 |

**Python → ArkTS：**

| type | 说明 |
|:---|:---|
| `ready` | 启动成功，附带 Python 版本 |
| `response` | 普通回复。附带 `content` |
| `tool_call` | 请求 ArkTS 执行设备操作。附带 `tool` 和 `params` |
| `pong` | 心跳回应 |
| `error` | 出错了。附带 `message` |

────────────────────

【验证：Tool Calling 拍照流程】

这是最关键的验证——**Python 能不能反过来"调用"ArkTS 的能力？**

Python 端的 `handle_chat_with_tools` 函数做了个简单的关键词匹配：如果用户说了"拍照"、"camera"、"摄像头"之类的词，就发一个 `tool_call` 请求给 ArkTS。

完整的消息流：

```
用户输入："帮我拍个照"

[1] ArkTS → Python:
    {"type": "chat", "message": "帮我拍个照"}

[2] Python 识别到关键词"拍个照"
    Python → ArkTS:
    {"type": "tool_call", "tool": "camera", "params": {"action": "capture"}}

[3] ArkTS 收到 tool_call，执行拍照（目前是模拟）
    ArkTS → Python:
    {"type": "tool_result", "tool": "camera",
     "result": {"path": "/data/.../fake_photo.jpg", "status": "simulated"}}

[4] Python 收到 tool_result，拼最终回复
    Python → ArkTS:
    {"type": "response",
     "content": "[Python Agent] Photo captured! Path: .../fake_photo.jpg"}

[5] ArkTS UI 显示：
    [You] 帮我拍个照
    [Tool Call] camera
    [Agent] Photo captured! Path: .../fake_photo.jpg
```

**P7885 板子上验证通过。** 整个流程——发消息、Python 识别意图、发 tool_call、ArkTS 执行、返回结果、Python 拼回复——全部在管道里跑完，延迟几乎感受不到。

────────────────────

【ArkTS 端怎么写的】

pythonRunner.ets 里的 `sendChat()` 方法处理了完整的 tool_call 流程：

```typescript
// 发消息给 Python
execNapi.sendMessage(JSON.stringify({type: 'chat', message: userMsg}));

// 读 Python 的回复
let reply = execNapi.readMessage();
let parsed = JSON.parse(reply);

if (parsed.type === 'tool_call') {
    // Python 要调设备能力
    let toolName = parsed.tool;  // 比如 "camera"

    // 执行工具（目前模拟，以后接真 API）
    let result = {path: filesDir + '/fake_photo.jpg', status: 'simulated'};

    // 把结果发回 Python
    execNapi.sendMessage(JSON.stringify({
        type: 'tool_result', tool: toolName, result: result
    }));

    // 读 Python 的最终回复
    let finalReply = execNapi.readMessage();
    // 显示到 UI
}
```

**ArkTS 端只需要知道两件事：怎么发 JSON、怎么读 JSON。** 不需要了解 Python 内部的逻辑。管道把两边完全解耦了。

◆ 第三段：这意味着什么

━━━━━━━━━━━━━━━━━━━━

先退一步看整个架构：

```
用户
  ↓ 打字
ArkTS UI (pythonRunner.ets)
  ↓ sendMessage
C++ NAPI (exec_napi.cpp) ← 持有 stdin/stdout 管道
  ↓ write(stdin_fd)
Python daemon (agent_daemon.py) ← 常驻进程
  ↓ 需要设备能力时
  ↓ stdout 写 tool_call
C++ NAPI 读到 tool_call
  ↓ 回传给 ArkTS
ArkTS 执行设备操作（拍照 / 蓝牙 / 传感器 / GPS / ...）
  ↓ sendMessage(tool_result)
Python 收到结果
  ↓ 继续处理
  ↓ stdout 写最终 response
ArkTS UI 显示结果
```

**这条管道意味着：Python Agent 可以调用鸿蒙的任何系统能力。**

目前 `tool_call: camera` 是模拟的——ArkTS 端返回一个假路径。但把那段模拟代码换成真的 `@ohos.multimedia.camera` API 调用，**就是完整的"AI Agent 控制摄像头"**。

同理：

- `tool_call: bluetooth` → `@ohos.bluetooth` → 扫描 / 配对 / 连接
- `tool_call: sensor` → `@ohos.sensor` → 读加速度计 / 陀螺仪 / 温度
- `tool_call: brightness` → `@ohos.display` → 调屏幕亮度
- `tool_call: location` → `@ohos.geoLocationManager` → 获取 GPS 定位

**管道的两端是通用的。** Python 端只管发 `{"type": "tool_call", "tool": "xxx"}`，ArkTS 端只管收到后调对应的鸿蒙 API。加一个新 tool 就是在 ArkTS 端加一个 `if (toolName === 'xxx')` 分支。

────────────────────

【和 LangChain 的天然契合】

LangChain 框架有一套成熟的 **Tool Calling** 机制——你定义一组 tool（函数描述），大模型根据用户意图决定调哪个 tool，框架执行 tool 拿到结果，再让大模型根据结果生成回复。

现在的 `agent_daemon.py` 用关键词匹配模拟了这个过程。**但把关键词匹配换成 LangChain Agent + DeepSeek 的 Tool Calling，流程完全一样：**

```
用户："帮我拍个照"
  → DeepSeek 分析意图 → 返回 tool_call: camera
  → Python 发 tool_call 给 ArkTS
  → ArkTS 执行拍照 → 返回 tool_result
  → Python 把 tool_result 交给 DeepSeek
  → DeepSeek 生成最终回复："已帮您拍好照片，保存在 /xxx/photo.jpg"
```

**LangChain 的 Agent 机制天然适配这种"Python 做大脑、设备做四肢"的架构。**

────────────────────

【这是一个基础设施】

不要把它看成"一个能模拟拍照的 demo"。

**这是在鸿蒙上实现"AI Agent 控制物理世界"的基础管道。**

管道两端是解耦的——Python 端可以跑 LangChain Agent、可以跑 AutoGPT、可以跑任何 Python AI 框架。ArkTS 端可以接鸿蒙的任何系统 API——摄像头、蓝牙、传感器、文件系统、网络、通知、日程。

**Python 生态的 AI 能力 + 鸿蒙的设备控制能力，通过 stdin/stdout 管道连起来了。**

你在手机上对着一个 AI 说"帮我查一下今天的步数，然后根据运动量推荐晚餐"——AI 调鸿蒙的健康传感器 API 拿步数，调大模型分析运动量，调美食 API 推荐晚餐。**这不再是概念，而是一条已经能跑通的技术栈。**

几个值得记住的设计决策：

**stdin/stdout 管道而不是 Unix Socket。** 因为 Python 本来就是 fork 出来的子进程，管道是最自然、最轻量的选择。不需要多余的配置。

**JSON Lines 而不是二进制协议。** 因为可读可调试，出了问题直接打日志看。在设备端 AI Agent 这个阶段，调试效率比传输效率重要得多。

**PYTHONUNBUFFERED=1。** 不加这个，管道通信会莫名其妙卡住。Python 的 stdout 缓冲是个隐蔽的坑——在终端里跑没问题（终端默认行缓冲），但管道里默认全缓冲。

**Tool 的执行放在 ArkTS 端，不放在 Python 端。** 因为鸿蒙的系统 API 都是 ArkTS 写的。Python 只负责"决策"（调不调 tool、调哪个），ArkTS 负责"执行"（调具体的 @ohos.xxx API）。**大脑和四肢分离。**

◆ 术语速查

━━━━━━━━━━━━━━━━━━━━

| 术语 | 解释 |
|:---|:---|
| daemon | 守护进程。在后台持续运行、不主动退出的进程。等着接收消息并处理 |
| stdin/stdout 管道 | 进程间通信的基础设施。fork 时用 pipe() 创建，父子进程各持一端，实现双向通信 |
| JSON Lines | 文本协议。每行一个 JSON 对象，用换行符分隔。简单可读，适合管道通信 |
| tool_call | Agent 请求执行外部工具的消息。包含工具名和参数，由宿主环境执行后返回 tool_result |
| Tool Calling | LangChain / 大模型的能力——模型根据用户意图决定调用哪个工具函数 |
| certifi | Python 的 SSL 证书包。内置 Mozilla 的 CA 根证书列表，HTTPS 开箱即用 |
| PYTHONUNBUFFERED | Python 环境变量。设为 1 则关闭 stdout 缓冲，每次 print 立刻写出 |
| NAPI | Native API。鸿蒙的 C/C++ 原生模块机制，让 ArkTS 调 C++ 函数 |
| fork + execve | Unix 最底层的进程创建方式。fork 复制进程，execve 替换为新程序 |
| pipe() | Unix 系统调用。创建一对文件描述符，一端写一端读，用于进程间通信 |
| dup2 | Unix 系统调用。把一个文件描述符复制到另一个编号上。用来把管道"接"到 stdin/stdout |
| SIGTERM / SIGKILL | Unix 信号。SIGTERM 礼貌请求退出，SIGKILL 强制杀死进程 |
| waitpid + WNOHANG | 检查子进程状态。WNOHANG 表示不阻塞——看一眼就走，进程还活着就返回 0 |
| HAP | Harmony Ability Package。鸿蒙 App 安装包格式，类似 Android APK |
| rawfile | HAP 里的资源目录。打包时放进去，运行时 App 可以读取 |
| ArkTS | 鸿蒙开发语言，基于 TypeScript，加了声明式 UI |
| site-packages | Python 的第三方包安装目录。pip install 的包都放这里 |
| pydantic_core | pydantic v2 核心引擎，Rust 编写的 .so。LangChain 的硬依赖 |
