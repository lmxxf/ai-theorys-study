【统一内存跑AI】RTX Spark笔记本要来了？先把这个坑了解一下

6 月 1 日，Computex 2026，老黄掏出了 **RTX Spark**——NVIDIA 头一回把 Grace Blackwell 超级芯片塞进消费级笔记本和迷你台式机。20 个 Arm CPU 核 + Blackwell GPU + **最高 128GB 统一内存** + 1 petaflop AI 算力，号称本地能跑 AI agent、能创作、能打游戏。戴尔、惠普、联想、微软、华硕、微星，30 多款笔记本，今年秋天就上。

听着挺香。不过这类统一内存的设备有个不太起眼的坑，今天想先跟你聊聊——以后真要拿它本地跑大模型，知道这个能少走点弯路。

其实这套芯片不是凭空冒出来的。去年（2025）NVIDIA 就发过一台给开发者用的"小金盒子"，叫 **DGX Spark**，同样是 GB10 / Grace Blackwell 思路、同样最高 128GB 统一内存，首发 3999 美元，今年 2 月因为内存涨价涨到了 4699。RTX Spark 可以看成这条路线的 PC / 消费平台化版本——把同源架构塞进笔记本和小台式机，卖给创作者、开发者和游戏玩家。

注意，这两代共享的核心 DNA，就是那 **128GB 统一内存**。而这个统一内存，恰恰是今天要讲的坑的源头。

──────

假设秋天到了，你买了一台 RTX Spark，统一内存 128GB。你没有用 NVIDIA 或厂商给的专用推理框架，而是自己写了个小程序，直接调 transformers，兴冲冲地要跑一个 68GB 的模型。

128 > 68，内存够啊。

结果 `from_pretrained`（加载模型的标准函数）跑到一半，整台机器卡死，或者直接 OOM（Out Of Memory，内存不足）被系统杀掉。

你换了 transformers 4.57、5.0，甚至 nightly 版（加载模型用的库，换了好几个版本），全挂。

这通常不是模型本身放不下，而是**现有加载路径 + 统一内存 + 模型接近内存上限**共同造成的峰值问题。

今天把这件事从底层讲清楚：为什么"内存够"不等于"放得下"，以及怎么绕过去。

◆ 先搞清楚：统一内存和独显到底差在哪

━━━━━━━━━━━━━━━━━━━━

要理解这个坑，必须先理解两种硬件架构的根本区别。

【独显（NVIDIA RTX / A100 / H100）：两个池子】

你的电脑里有两块独立的物理内存：

- **CPU 内存（RAM）**：比如 64GB DDR5，插在主板上
- **GPU 显存（VRAM）**：比如 24GB GDDR6X，焊在显卡上

这两块内存是物理隔离的，各管各的。加载模型的流程是：

```python
# 第一步：mmap 到 CPU 内存
model = AutoModelForCausalLM.from_pretrained("some-model")
# 此时权重在 CPU RAM 里

# 第二步：拷贝到 GPU 显存
model = model.to("cuda")
# 权重从 CPU RAM 复制到 GPU VRAM
```

💡 **mmap（内存映射）**：操作系统的一个机制——不是把整个文件读进内存，而是"假装"这个文件已经在内存里了。你真正去读某一页的时候，操作系统才把那一页从硬盘搬进物理内存。这个"搬进来的页"叫**页缓存（page cache）**。

关键点：CPU RAM 和 GPU VRAM 是两个独立的池子。mmap 页缓存占的是 CPU RAM，GPU tensor 占的是 GPU VRAM。**两份各在各的池子里，互不干扰。**

64GB RAM 装得下 mmap 页缓存，24GB VRAM 装得下模型权重，完美。

这里藏着一个你过去从来没注意过、却一直在保护你的事实：**独显时代，你的内存（RAM）几乎总是比显存（VRAM）大。**

想想看，常见配置大概是：16GB 内存配 8GB 显卡、32GB 内存配 24GB 显卡、64GB 内存配 32GB 的 5090……CPU 内存和显存大小接近，或者内存略大一点，但关键是**它们是两块独立池子**。模型加载时先在内存里铺开那份 mmap 副本，内存这边先兜住；真正吃紧的显存那边，只放最终权重。**内存就像一个单独的缓冲区，默默接住了 mmap 那一份，你根本感觉不到它的存在。**

所以这个坑过去不存在——不是因为大家都意识到了这里有问题，而是因为硬件结构天然就帮你挡住了，你压根没机会撞上。

可一旦换成统一内存，这个"内存比显存大"的安全垫，没了。

【统一内存（Mac M 系列 / DGX Spark / AMD AI Max）：一个池子】

这些设备的 CPU 和 GPU **共享同一块物理内存**。

Mac Studio M4 Ultra 的 128GB，不是"CPU 64GB + GPU 64GB"——是 CPU 和 GPU 共用同一个 128GB 的物理内存池。DGX Spark（Grace Blackwell，128GB）和 AMD AI Max 395+（128GB）同理。

💡 **统一内存（Unified Memory）**：CPU 和 GPU 不再各有各的内存，而是共享同一块物理内存。好处是不用在 CPU 和 GPU 之间来回拷贝数据（理论上 GPU 可以直接读 CPU 的数据）。坏处嘛——就是今天要讲的这个坑。

问题来了。同样的加载流程：

```python
model = AutoModelForCausalLM.from_pretrained("some-model")
model = model.to("cuda")  # 或 .to("mps") 对 Mac
```

mmap 页缓存和 GPU tensor **都住在同一个 128GB 池子里**。

一个 68GB 的模型：mmap 页缓存 68GB + GPU tensor 68GB = **136GB**。

128GB 的池子装不下 136GB。

**爆了。**

──────

打个比方就是：

加载模型，本质上是"复印"。原件（硬盘上的文件）先摊到内存里，再复印一份给 GPU 用。关键是——**原件不会消失，复印件又多出来一份，两份同时存在。**

独显就像你家有两张桌子：原件摊在书桌上（CPU RAM），复印件放到工作台上（GPU VRAM）。一张桌子放一份，互不挤占。

统一内存就像你家只有一张桌子：原件和复印件都得摊在这一张桌子上。你以为"复印件只占半张桌子，桌子够大"——但你忘了原件还在桌上摊着没收走，两份加起来，桌子就摊不下了。

━━━━━━━━━━━━━━━━━━━━

◆ OOM 的三个根因

━━━━━━━━━━━━━━━━━━━━

不只是"两份"这么简单。实际上有三个东西在同时吃你的内存：

【根因一：safetensors 的 mmap 页缓存】

现在主流的模型权重格式叫 **safetensors**。它的加载方式就是 mmap——打开文件后，操作系统把访问过的页缓存进物理内存。

💡 **safetensors**：Hugging Face 设计的模型权重存储格式，比老的 pickle 格式更安全（不能执行任意代码）、更快（支持 mmap 直接映射）。现在几乎所有开源模型都用这个格式。

关键问题：这些页缓存**不会在 Python 对象释放时立刻消失**。它们是操作系统管理的可回收缓存，内存紧张时理论上可以被回收；但什么时候回收、能不能及时回收，不由你的加载代码精确控制。在独显上这通常无所谓（反正占的是 CPU RAM，GPU VRAM 不受影响）；在统一内存上，这些页缓存和 GPU tensor 抢的是同一个池子。

【根因二：compressed-tensors 解压的中间态】

很多量化模型（INT8、INT4）用 **compressed-tensors** 格式存储。加载时需要解压：从压缩格式还原成 GPU 能用的 tensor。

这个解压过程会产生中间态——一份压缩数据 + 一份解压后的数据，短暂地同时存在于内存中。在独显上，中间态在 CPU RAM 里，解压完拷到 GPU VRAM，CPU 那份释放掉，没事。在统一内存上，中间态和最终 tensor 都在同一个池子里，峰值内存会短暂飙高。

【根因三：from_pretrained 的内部流程】

很多常见的 `from_pretrained` 加载路径，都会经历类似这样的中转过程：

```
磁盘上的 safetensors 文件
    → mmap 映射到虚拟内存
    → 访问时 fault 进物理内存（页缓存）
    → 创建 CPU tensor（读取页缓存中的数据）
    → .to("cuda") 创建 GPU tensor
```

在统一内存上，"页缓存"和"GPU tensor"这两步产生的数据，都落在同一个物理内存池里。**就算 transformers / accelerate 内部做了优化（比如 meta init、逐层加载、分片加载、用完释放），页缓存的回收时机仍然由操作系统决定，不是 Python 能精确控制的。**

──────

说人话就是，三个吃内存的家伙同时挤在一个池子里：

1. **页缓存**：操作系统读文件时留下的"缓存副本"，不一定立刻走
2. **解压中间态**：量化模型解压时短暂出现的"半成品"
3. **GPU tensor**：模型真正要用的权重

在独显上，1 和 2 在 CPU 那张桌子上，3 在 GPU 那张桌子上，各不相干。在统一内存上，1、2、3 全摊在同一张桌子上——**峰值接近模型体积的两倍**。

◆ 多机的意外好处，不只是"内存变多"

━━━━━━━━━━━━━━━━━━━━

先说清楚一件容易被忽略的事：用两台 128GB 的机器，硬件总量是 256GB，装一个 136GB 的模型，本来就该装得下——这没什么神奇的。

真正值得说的是另一点。

一个 68B 的 BF16 模型（体积约 136GB，68×2），用两台机器做 **TP=2（张量并行）**，每台承担一半，理论上每台 68GB。

但你应该记得前面的结论：68GB 的模型，在单台 128GB 统一内存上，也会因为"页缓存 + GPU tensor 双份 = 136GB"而 OOM。

那问题来了：每台明明也是放 68GB，凭什么双机就不 OOM，单机就爆？**这说明双机省下的不只是"总量"，它的加载方式本身就和单机不一样。**

💡 **TP（Tensor Parallelism，张量并行）**：把模型的每一层切成 N 份，分散到 N 台机器（或 N 块 GPU）上。每台只算自己那一份，算完通信汇总。不是"前半层在 A、后半层在 B"，而是"每一层都切两半，A 算左半、B 算右半"。

答案藏在新版 transformers 的 TP 加载路径里。

TP 的加载路径和单机 `from_pretrained` **完全不同**。它用的是 safetensors 库的 `get_slice()` 方法：

```python
# TP 加载路径（简化）
with safe_open("model.safetensors", framework="pt") as f:
    # 不是把整个 tensor 读进来，而是拿一个 lazy 引用
    slice_obj = f.get_slice("model.layers.0.self_attn.q_proj.weight")
    
    # 每个 rank 只读自己那一半的字节
    if rank == 0:
        my_chunk = slice_obj[0:half, :]   # 前半行
    else:
        my_chunk = slice_obj[half:, :]    # 后半行
```

💡 **get_slice()**：safetensors 提供的懒加载接口。它不读数据，只返回一个"引用"——告诉你"这个 tensor 在文件的第几字节到第几字节"。你用切片语法（`[start:end]`）去取的时候，才真正把那一段字节从磁盘 mmap 进物理内存。

关键在这里：**每个 rank 用 `slice_obj[start:end]` 只读自己那一半的字节。**

mmap 是按页触发的——你访问哪一页，操作系统才把那一页从磁盘搬进物理内存（这个动作叫 **page fault**）。你没访问的页，根本不会被 fault 进物理内存。

所以在 TP=2 的情况下，从整个模型来看：

- rank 0 只访问前半个 tensor 的页 → 总访问量约 68GB
- rank 1 只访问后半个 tensor 的页 → 总访问量约 68GB
- 每台的 GPU tensor 也只有约 68GB

你可能会问：每台机器的峰值不还是 **约 68GB 页缓存 + 68GB GPU tensor = 136GB** 吗？不对，关键不在"总共访问过多少页"，而在"同一时刻压着多少页"。

不。这里有个细节：TP 的加载是**逐层进行**的。加载完一层，那一层的页缓存就不再被引用了，操作系统可以在内存压力下回收它。而且 GPU tensor 是直接从 mmap 切片创建的，中间不需要一个完整的 CPU tensor 副本。

我自己实际跑下来：这样一个 136GB 的模型，双机 TP=2，每台峰值只有 70GB 出头，128GB 绰绰有余。

──────

说穿了，道理特别朴素。

单机加载，历来是"无脑"的——`from_pretrained` 一把梭，框架替你把整个模型哗啦倒进内存，你从来不用操心加载的过程。这个习惯在独显时代没事，到了统一内存就翻车。

而双机部署，**本来就麻烦**。你不得不安排"哪台机器加载哪一部分"，被逼着一片一片地处理权重。结果呢？这个"被迫精细化"的过程，顺手就把 mmap 翻倍的坑给绕过去了。

所以不是双机有什么魔法，是它麻烦得不得不"一点点加载"，而单机用户图省事，习惯了"一口气加载"——**坑就藏在这个习惯里。**

──────

再换个画面感的说法。

单机加载就像搬家：一堆箱子全堆进客厅（页缓存），然后把每个箱子里的东西取出来、摆进卧室（GPU tensor）。注意——**箱子不会扔掉，它还堆在客厅里，东西又在卧室摆了一份。** 客厅和卧室是同一个房子（统一内存），箱子和摆出来的东西同时占着地方，房子塞不下。

TP 双机加载就像两家人各搬各的：每家只分到一半箱子，而且是**取一个、空一个**——从客厅拿一个箱子，把东西摆进卧室，随手就把空箱子拆了扔掉。任何时候客厅里只堆着一两个箱子，永远不会堆满。

◆ 解决方案：模型多大，就用什么招

━━━━━━━━━━━━━━━━━━━━

以 128GB 统一内存设备为基准，按模型权重文件大小分档：

【≤60GB：直接加载】

按通用默认路径估算，峰值约为模型大小的两倍（页缓存 + GPU tensor）。60×2 = 120GB，给系统和运行时留一点余量，128GB 设备大概率还能扛住。

```python
from transformers import AutoModelForCausalLM

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-3.1-8B",
    torch_dtype="auto",
    device_map="auto"
)
```

这个区间基本不用动脑子，直接上。

【60-128GB：流式加载，逐 shard / 逐 tensor 控制峰值】

模型一旦超过总内存的一半（这里是 60GB 左右），翻倍峰值就接近甚至超过 128GB，直接 `from_pretrained` 就开始有翻车风险了。这一整个区间最稳的方向是：**流式 / 分片 / 低峰值加载**。

核心思路：**不要让"整份 CPU 权重 + 整份 GPU 权重 + 一堆页缓存"同时存在。** 具体做法是先用 meta init 建一个空模型骨架，再用 safetensors / accelerate / 专用 loader 逐 shard、逐 tensor 把权重放到目标设备上，尽量减少 CPU 侧中间副本，并把峰值控制在逐 tensor / 逐 shard 粒度。

```python
# 伪代码：重点是加载策略，大概就这么个意思

# 1. 先在 meta device 上建空模型，不分配真实权重内存
with init_empty_weights():
    model = AutoModelForCausalLM.from_config(config)

# 2. 逐 shard / 逐 tensor 读取 safetensors
for shard in shard_files:
    with safe_open(shard, framework="pt", device=target_device) as f:
        for name in f.keys():
            tensor = f.get_tensor(name)   # 只拿当前 tensor
            assign_into_model(model, name, tensor)  # 直接赋到目标参数
            del tensor

    gc.collect()
```

💡 **shard（分片）**：大模型的权重文件通常被拆成多个文件（比如 `model-00001-of-00004.safetensors`），每个叫一个 shard。这样下载和加载都可以分批进行。

这里 `safe_open(..., device="cuda")` 会让 safetensors 尽量直接把 tensor 创建在 GPU 上，减少 CPU 侧的中间副本。注意，文件页缓存仍然可能存在；它不是被"消灭"了，而是峰值被压到了当前 tensor / 当前 shard 的粒度。每处理完一个 shard 就触发垃圾回收，只是给 Python 和操作系统一个回收机会，不是强制命令。

真正写工程代码时还有一个坑：不要先把所有 tensor 攒成一个 `state_dict`，再 `load_state_dict`。那样很可能又制造一份完整中间副本。要么用 `assign=True` / meta init 这类直接赋值路径，要么用 accelerate、vLLM、SGLang、llama.cpp、MLX 之类框架自己的低峰值 loader。

说句实话——即使用 shard 级加载，一个 shard 往往也有三五个 GB，一次性读进来，峰值还是会多吃这么一坨。**更合理的做法是切到 slice（单个 tensor、甚至 tensor 的一部分）粒度**，读一片、扔一片，峰值能压得更低——这其实就是前面讲双机 TP 时用的那套 `get_slice()` 思路。对单机来说 shard 级一般也够用了（多吃几个 GB 在 128GB 池子里无所谓），但你要追求极致，slice 级才是正解。

【超过 128GB：得多机联手了】

模型比单机内存还大，一台放不下，就得用两台甚至更多机器做张量并行（TP），把模型切开分摊。前面那节讲的 `get_slice()` 懒加载，正是多机 TP 不 OOM 的关键。

不过这一档基本和大多数消费者无关——多机 TP 需要机器之间有高速互联（DGX Spark 用的是 200Gbps 的 ConnectX-7 网卡直连）。DGX Spark 这种个人 AI 小盒子为了开发者场景，可能还会保留类似能力；但 RTX Spark 笔记本要做轻薄、成本和续航，这种接口大概率会被阉割掉。所以这块就不展开了，真要折腾多机的人，自然会去啃专门的文档。

◆ 总结：一张表记住

━━━━━━━━━━━━━━━━━━━━

如果你是自己写小程序、走通用默认加载路径，可以先记住这个保守公式：

```
峰值内存 ≈ 模型文件大小 × 2
（页缓存一份 + GPU tensor 一份，共享同一个物理池）
```

所以 128GB 统一内存，按这个保守估算，安全线不是 128GB，而是它的一半 **约 60GB**。超过这条线，就老老实实换加载方式，或者直接用厂商/框架已经适配好的专用 loader。

以 128GB 设备为例，按模型大小这么选，基本不会错：

| 模型大小 | 怎么做 | 一句话 |
|---------|-------|--------|
| ≤ 一半（约 60GB 以内） | 直接 `from_pretrained` | 不用想，上 |
| 一半到全部（60-128GB） | 低峰值加载（meta init / streaming / 专用 loader） | 别一口气倒进去 |
| 超过全部（>128GB） | 多机 TP（`get_slice()` 懒加载天然不翻倍） | 大多数消费者基本用不到 |

统一内存是好东西——不用买昂贵的 GPU 显存，CPU 和 GPU 共享一个大池子，带宽高、延迟低。但它有个隐含的代价：**所有的中间态都摊在同一张桌子上，没有第二张桌子给你腾挪。**

──────

最后说个判断。

DGX Spark 这类机器（未来的 RTX Spark 大概率也一样）适合干什么？**适合做本地实验。** 比如跑一些激活值 dump、小规模 agent、模型行为观察、工具链验证——这些场景看重的是"我桌上就有一台能摸的机器"，不是极限吞吐。

但你要把它当 **AI 推理机**，那基本是个伪需求。统一内存再大，GPU 算力和内存带宽也不是数据中心卡；真跑起 30B 稠密模型推理，都能卡得你怀疑人生。真要认真做推理吞吐，别幻想小盒子和笔记本，老老实实上独显工作站卡，RTX PRO 6000 这种级别起步，价格基本就是 10 万往上。

当然，这不等于 RTX Spark 没用。拿来打游戏、做日常创作、本地跑轻量 AI 工具，应该还是可以的。它的游戏性能大概率比 RTX 5070 独显慢一些，但基本够用。只是别被"128GB 统一内存"这几个字骗了，以为它突然就能变成便宜的数据中心推理卡。

短期内，统一内存大概率不会成为加载模型的最高优先级路径。原因很简单：**真正训练和部署大模型的数据中心，用的是 H100、B200 这种独显——HBM 显存和系统内存严格分开，根本不是统一内存架构。** 各种主流框架（transformers、vLLM、SGLang）的默认加载逻辑，优先服务的是数据中心那套独显模型，统一内存适配会有，但不会排在最前面。

这意味着什么？**如果你用的是通用框架的默认路径**——比如直接拿 transformers 一把梭 `from_pretrained`——未来相当长一段时间里，可能都得自己针对统一内存做适配：流式加载、手动管理峰值、少用一把梭式加载。

但如果你用的是 **RTX Spark 专用推理框架**、NVIDIA 自己的 demo，或者厂商为这类机器定制过的 loader，那就另说了。它们当然应该知道 GB10 / 统一内存的坑，加载路径大概率会专门处理峰值问题。本文讲的是：**不能把数据中心独显时代的通用默认加载习惯，原封不动搬到消费级统一内存机器上。**

换句话说，今天讲的这个坑，不是一次性的 bug，而是消费级统一内存设备一个长期存在的"水土不服"。早点知道，早点习惯绕着走。

━━━━━━━━━━━━━━━━━━━━

"128GB 内存，68GB 模型，一把梭加载也可能放不下。"

"不是内存不够，是你的两份数据在同一个房间里打架。"
