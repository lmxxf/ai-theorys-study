【统一内存】128GB 明明够，为什么加载模型还是 OOM 死机？

你买了台 DGX Spark / Mac Studio / AMD AI Max 笔记本，统一内存 128GB，兴冲冲地要跑一个 68GB 的模型。

128 > 68，内存够啊。

结果 `from_pretrained` 跑到一半，整台机器卡死，或者直接 OOM（Out Of Memory，内存不足）被杀。

你换了 transformers 4.57、5.0、5.7，全挂。

这不是 bug。这是统一内存架构的物理必然。

今天把这件事从底层讲清楚：为什么"内存够"不等于"放得下"，以及怎么绕过去。

◆ 先搞清楚：统一内存和独显到底差在哪

━━━━━━━━━━━━━━━━━━━━

要理解这个坑，必须先理解两种硬件架构的根本区别。

【独显（NVIDIA RTX / A100 / H100）：两个池子】

你的电脑里有两块独立的物理内存：

- **CPU 内存（RAM）**：比如 64GB DDR5，插在主板上
- **GPU 显存（VRAM）**：比如 24GB GDDR6X，焊在显卡上

这两块内存是物理隔离的，各管各的。加载模型的流程是：

```python
# 第一步：mmap 到 CPU 内存
model = AutoModelForCausalLM.from_pretrained("some-model")
# 此时权重在 CPU RAM 里

# 第二步：拷贝到 GPU 显存
model = model.to("cuda")
# 权重从 CPU RAM 复制到 GPU VRAM
```

💡 **mmap（内存映射）**：操作系统的一个机制——不是把整个文件读进内存，而是"假装"这个文件已经在内存里了。你真正去读某一页的时候，操作系统才把那一页从硬盘搬进物理内存。这个"搬进来的页"叫**页缓存（page cache）**。

关键点：CPU RAM 和 GPU VRAM 是两个独立的池子。mmap 页缓存占的是 CPU RAM，GPU tensor 占的是 GPU VRAM。**两份各在各的池子里，互不干扰。**

64GB RAM 装得下 mmap 页缓存，24GB VRAM 装得下模型权重，完美。

【统一内存（Mac M 系列 / DGX Spark / AMD AI Max）：一个池子】

这些设备的 CPU 和 GPU **共享同一块物理内存**。

Mac Studio M4 Ultra 的 128GB，不是"CPU 64GB + GPU 64GB"——是 CPU 和 GPU 共用同一个 128GB 的物理内存池。DGX Spark（Grace Blackwell，128GB）和 AMD AI Max 395+（128GB）同理。

💡 **统一内存（Unified Memory）**：CPU 和 GPU 不再各有各的内存，而是共享同一块物理内存。好处是不用在 CPU 和 GPU 之间来回拷贝数据（理论上 GPU 可以直接读 CPU 的数据）。坏处嘛——就是今天要讲的这个坑。

问题来了。同样的加载流程：

```python
model = AutoModelForCausalLM.from_pretrained("some-model")
model = model.to("cuda")  # 或 .to("mps") 对 Mac
```

mmap 页缓存和 GPU tensor **都住在同一个 128GB 池子里**。

一个 68GB 的模型：mmap 页缓存 68GB + GPU tensor 68GB = **136GB**。

128GB 的池子装不下 136GB。

**爆了。**

──────

💡 翻译成人话

独显就像你家有两个冰箱：一个放食材（CPU RAM），一个放成品菜（GPU VRAM）。食材和成品菜各占各的冰箱，互不干扰。

统一内存就像你家只有一个冰箱：食材和成品菜都得挤进去。你以为"成品菜只有 68 升，冰箱 128 升，够了"——但你忘了食材也占地方，两样加起来超了。

━━━━━━━━━━━━━━━━━━━━

◆ OOM 的三个根因

━━━━━━━━━━━━━━━━━━━━

不只是"两份"这么简单。实际上有三个东西在同时吃你的内存：

【根因一：safetensors 的 mmap 页缓存】

现在主流的模型权重格式叫 **safetensors**。它的加载方式就是 mmap——打开文件后，操作系统把访问过的页缓存进物理内存。

💡 **safetensors**：Hugging Face 设计的模型权重存储格式，比老的 pickle 格式更安全（不能执行任意代码）、更快（支持 mmap 直接映射）。现在几乎所有开源模型都用这个格式。

关键问题：这些页缓存**不会自动释放**。操作系统觉得"你之后可能还要用"，就让它常驻物理内存。在独显上这无所谓（反正占的是 CPU RAM，GPU VRAM 不受影响）；在统一内存上，这些页缓存和 GPU tensor 抢的是同一个池子。

【根因二：compressed-tensors 解压的中间态】

很多量化模型（INT8、INT4）用 **compressed-tensors** 格式存储。加载时需要解压：从压缩格式还原成 GPU 能用的 tensor。

这个解压过程会产生中间态——一份压缩数据 + 一份解压后的数据，短暂地同时存在于内存中。在独显上，中间态在 CPU RAM 里，解压完拷到 GPU VRAM，CPU 那份释放掉，没事。在统一内存上，中间态和最终 tensor 都在同一个池子里，峰值内存会短暂飙高。

【根因三：from_pretrained 的内部流程】

不管你怎么写代码，`from_pretrained` 的内部加载路径始终有一个中转过程：

```
磁盘上的 safetensors 文件
    → mmap 映射到虚拟内存
    → 访问时 fault 进物理内存（页缓存）
    → 创建 CPU tensor（读取页缓存中的数据）
    → .to("cuda") 创建 GPU tensor
```

在统一内存上，"页缓存"和"GPU tensor"这两步产生的数据，都落在同一个物理内存池里。**就算 transformers 内部做了优化（比如逐层加载、用完释放），页缓存的回收时机由操作系统决定，不是 Python 能控制的。**

──────

💡 翻译成人话

三个吃内存的家伙同时挤在一个池子里：

1. **页缓存**：操作系统读文件时留下的"缓存副本"，赖着不走
2. **解压中间态**：量化模型解压时短暂出现的"半成品"
3. **GPU tensor**：模型真正要用的权重

在独显上，1 和 2 在 CPU 的冰箱里，3 在 GPU 的冰箱里，各不相干。在统一内存上，1、2、3 全挤在同一个冰箱里——**峰值接近模型体积的两倍**。

━━━━━━━━━━━━━━━━━━━━

◆ 为什么双机 TP 反而不 OOM

━━━━━━━━━━━━━━━━━━━━

这件事更反直觉。

一个 136GB 的模型（Qwen2.5-72B BF16），单机 128GB 放不下，很正常。但用两台 128GB 的机器做 **TP=2（张量并行）**，每台只需要承担一半——理论上每台 68GB。

问题是：按前面的分析，68GB 的模型在 128GB 统一内存上也会 OOM 啊（页缓存 + GPU tensor = 136GB）。为什么双机 TP 就能成功？

💡 **TP（Tensor Parallelism，张量并行）**：把模型的每一层切成 N 份，分散到 N 台机器（或 N 块 GPU）上。每台只算自己那一份，算完通信汇总。不是"前半层在 A、后半层在 B"，而是"每一层都切两半，A 算左半、B 算右半"。

答案藏在 transformers 5.7 的源码里。

TP 的加载路径和单机 `from_pretrained` **完全不同**。它用的是 safetensors 库的 `get_slice()` 方法：

```python
# TP 加载路径（简化）
with safe_open("model.safetensors", framework="pt") as f:
    # 不是把整个 tensor 读进来，而是拿一个 lazy 引用
    slice_obj = f.get_slice("model.layers.0.self_attn.q_proj.weight")
    
    # 每个 rank 只读自己那一半的字节
    if rank == 0:
        my_chunk = slice_obj[0:half, :]   # 前半行
    else:
        my_chunk = slice_obj[half:, :]    # 后半行
```

💡 **get_slice()**：safetensors 提供的懒加载接口。它不读数据，只返回一个"引用"——告诉你"这个 tensor 在文件的第几字节到第几字节"。你用切片语法（`[start:end]`）去取的时候，才真正把那一段字节从磁盘 mmap 进物理内存。

关键在这里：**每个 rank 用 `slice_obj[start:end]` 只读自己那一半的字节。**

mmap 是按页触发的——你访问哪一页，操作系统才把那一页从磁盘搬进物理内存（这个动作叫 **page fault**）。你没访问的页，根本不会被 fault 进物理内存。

所以在 TP=2 的情况下：

- rank 0 只 fault 了前半个 tensor 的页 → 页缓存约 68GB
- rank 1 只 fault 了后半个 tensor 的页 → 页缓存约 68GB
- 每台的 GPU tensor 也只有约 68GB

每台机器的峰值：**约 68GB 页缓存 + 68GB GPU tensor = 136GB？不对，还是爆了啊？**

不。这里有个细节：TP 的加载是**逐层进行**的。加载完一层，那一层的页缓存就不再被引用了，操作系统可以在内存压力下回收它。而且 GPU tensor 是直接从 mmap 切片创建的，中间不需要一个完整的 CPU tensor 副本。

实测数据：Qwen2.5-72B BF16（136GB），双机 TP=2，每台峰值 **75.2GB**，128GB 绰绰有余。

──────

💡 翻译成人话

单机加载就像一个人搬家：你把 68 个箱子全搬进客厅（页缓存），然后再一个个挪到卧室（GPU tensor）。客厅和卧室是同一个房子（统一内存），两堆箱子同时在房子里，房子不够大。

TP 双机加载就像两个人合搬：每人只搬 34 个箱子，而且是**搬一个放一个**——从门口拿一个箱子（mmap fault 一页），直接放进卧室（GPU tensor），门口就空了。任何时候房子里只有一小堆箱子在中转，不会堆满。

━━━━━━━━━━━━━━━━━━━━

◆ 解决方案：按模型大小选路

━━━━━━━━━━━━━━━━━━━━

以 128GB 统一内存设备为基准，按模型权重文件大小分档：

【≤40GB：直接加载】

峰值约为模型大小的两倍（页缓存 + GPU tensor），40×2 = 80GB，128GB 放得下。

```python
from transformers import AutoModelForCausalLM

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-3.1-8B",
    torch_dtype="auto",
    device_map="auto"
)
```

这个区间不用动脑子，直接上。

【40-65GB：device_map 分层卸载】

峰值可能超 128GB，但如果用 `device_map="auto"` 让 transformers 自动决定哪些层放 GPU、哪些留 CPU（在统一内存上这个区分是逻辑上的），并配合 `max_memory` 参数限制，可以勉强挤进去。

```python
model = AutoModelForCausalLM.from_pretrained(
    "some-model",
    torch_dtype="auto",
    device_map="auto",
    max_memory={0: "90GiB"}  # 给 GPU 限个上限，留空间给系统
)
```

但这个区间不稳定，取决于模型具体结构和操作系统的页缓存回收策略。建议先试，OOM 了就用下一档。

【65-128GB：流式加载，逐 shard 直接进 GPU】

这是 68GB 模型在 128GB 设备上的正解。核心思路：**绕过 from_pretrained，用 safetensors 的 safe_open 逐个 tensor 直接加载到 GPU，避免页缓存堆积。**

```python
from safetensors.torch import safe_open
import torch
import gc, os

def stream_load(model_dir, device="cuda"):
    """逐 shard、逐 tensor 流式加载到 GPU"""
    state_dict = {}
    
    shard_files = sorted([
        f for f in os.listdir(model_dir) 
        if f.endswith(".safetensors")
    ])
    
    for shard in shard_files:
        path = os.path.join(model_dir, shard)
        with safe_open(path, framework="pt", device=device) as f:
            for key in f.keys():
                state_dict[key] = f.get_tensor(key)
        
        # 每个 shard 加载完，主动提示 GC 回收
        gc.collect()
        if device == "cuda":
            torch.cuda.empty_cache()
    
    return state_dict
```

💡 **shard（分片）**：大模型的权重文件通常被拆成多个文件（比如 `model-00001-of-00004.safetensors`），每个叫一个 shard。这样下载和加载都可以分批进行。

这里 `safe_open(..., device="cuda")` 会让 safetensors 直接把 tensor 创建在 GPU 上，减少 CPU 侧的中间副本。每处理完一个 shard 就触发垃圾回收，让操作系统有机会回收前一个 shard 的页缓存。

【128-256GB：双机 TP=2】

模型太大，单机物理放不下。用两台统一内存设备组 TP：

```python
from transformers import AutoModelForCausalLM

# 在两台机器上分别启动，每台一个 rank
model = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen2.5-72B",
    torch_dtype="auto",
    tp_plan="auto",          # 自动张量并行
    device_map="auto"
)
```

前面分析过了，TP 的 `get_slice()` 懒加载路径天然避免了页缓存翻倍的问题。实测 Qwen2.5-72B BF16（136GB）双机 TP=2 成功，每台峰值 75.2GB。

【>256GB：自定义权重加载】

超过两台 128GB 能覆盖的范围，需要自己写加载逻辑：结合 TP/PP（流水线并行）、量化、逐层流式加载，按具体模型和硬件配置定制。这个区间没有通用方案，需要 case by case。

━━━━━━━━━━━━━━━━━━━━

◆ 实测案例

━━━━━━━━━━━━━━━━━━━━

以下是在 DGX Spark（128GB 统一内存）上的实测：

| 模型 | 大小 | 方法 | 结果 |
|-----|------|------|------|
| Llama-3.3-70B INT8 | 68GB | 单机 from_pretrained | **OOM**（transformers 4.57 / 5.0 / 5.7 全挂） |
| Llama-3.3-70B INT8 | 68GB | 流式 safe_open | **成功** |
| Qwen2.5-72B BF16 | 136GB | 双机 TP=2 | **成功**，每台 75.2GB |
| Qwen2.5-72B BF16 | 136GB | 单机 from_pretrained | **OOM**（物理不可能） |

第一行是最扎心的：68GB < 128GB，按直觉应该放得下，但就是 OOM。原因就是本文讲的——页缓存 + GPU tensor 双份，在同一个池子里挤爆了。

◆ 总结：一张图记住

━━━━━━━━━━━━━━━━━━━━

统一内存设备加载模型的核心公式：

```
峰值内存 ≈ 模型文件大小 × 2
（页缓存一份 + GPU tensor 一份，共享同一个物理池）
```

所以 128GB 统一内存，安全线不是 128GB，是 **约 60-65GB**。超过这个线，必须用流式加载或多机 TP。

三条规则：

1. **模型 ≤ 总内存的一半**：直接 `from_pretrained`，不用想
2. **模型在总内存的一半到全部之间**：流式加载（`safe_open` 逐 tensor 进 GPU）
3. **模型 > 总内存**：多机 TP，利用 `get_slice()` 的懒加载天然避免翻倍

统一内存是好东西——不用买昂贵的 GPU 显存，CPU 和 GPU 共享一个大池子，带宽高、延迟低。但它有个隐含的代价：**所有的中间态都在同一个池子里，没有第二个冰箱给你腾挪。**

知道这个代价，就知道怎么绕。
