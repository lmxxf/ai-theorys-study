◆ OpenClaw 是什么，为什么不能直接 npm install

━━━━━━━━━━━━━━━━━━━━

OpenClaw 是一个 264k star 的 TypeScript/Node.js AI Agent 框架，支持 WhatsApp、Telegram、Slack 等多平台接入，对接大模型做自动回复、任务执行。你可以把它理解成一个"AI 客服/助手的底座"。

关键词：**TypeScript**。

TypeScript 不能直接跑在 Node.js 上。它得先编译成 JavaScript，然后才能被 V8 引擎执行。整个 OpenClaw 是一个 pnpm monorepo——几十个包、几十个插件，构建系统是 `tsdown`，构建产物在 `dist/` 目录。

在你的 MacBook 上，`pnpm install && pnpm build` 就搞定了。但在 RK3568 开发板上——连 `pnpm` 都没有，连 `npm` 都没有，更别提 TypeScript 编译器。板子上只有一个光秃秃的 93MB `node` 二进制。

所以策略很明确：**在宿主机上编译 TS → JS，把编好的 JS + 运行时依赖打包推到板子上。板子只负责跑 JavaScript，不需要 TypeScript 编译器。**

打个比方：你不会把面粉和烤箱搬到饭桌上现烤面包。你在厨房烤好了，端到桌上就行。板子就是饭桌——只负责吃（运行），不负责做（编译）。

━━━━━━━━━━━━━━━━━━━━

◆ 构建：跳过不需要的东西

━━━━━━━━━━━━━━━━━━━━

在宿主机上构建 OpenClaw：

```bash
cd sources/openclaw
pnpm install                                          # 装依赖，39 秒
OPENCLAW_A2UI_SKIP_MISSING=1 pnpm build:docker        # 编译 TS → JS（命令名带 docker 但不需要 Docker，只是跳过桌面端才需要的 canvas 打包）
pnpm ui:build                                         # 编 Control UI（网页界面）
```

第二行那个环境变量 `OPENCLAW_A2UI_SKIP_MISSING=1` 是什么意思？OpenClaw 有个叫 Canvas A2UI 的模块，用于 AI 生成 UI 渲染。板子上不需要这个功能——我们要的是 Gateway（网关服务）+ Control UI（管理网页），不是 UI 渲染引擎。加这个变量告诉构建系统：Canvas A2UI 缺了就跳过，别报错。

`pnpm ui:build` 编 Control UI——这是 Gateway 的 Web 管理界面，编出来是一堆静态 HTML/JS/CSS 文件，放在 `dist/control-ui/` 目录。

编完了，`dist/` 目录 66MB，全是 JavaScript。

━━━━━━━━━━━━━━━━━━━━

◆ 瘦身：从 2GB 砍到 67MB

━━━━━━━━━━━━━━━━━━━━

编译产物 66MB 不大。但 `node_modules/` 是个无底洞。

OpenClaw 有约 1688 个 npm 依赖包。完整的 `node_modules/` 目录——**2GB**。推到板子上？开玩笑。板子存储本来就紧张，而且 hdc 传 2GB 文件能传到天荒地老。

第一刀：**`pnpm deploy --prod --legacy`**。

这个命令从完整的 monorepo 里提取出生产环境需要的依赖——开发依赖（TypeScript、eslint、测试框架……）全砍掉。2GB → 1.3GB。砍了 700MB，但还是太大。

第二刀：**删 x86 native addon**。

pnpm install 在宿主机（x86）上跑的，所以下载的 native addon 预编译包都是 x86 版本。板子是 aarch64，这些 x86 二进制完全没用。一个个找出来删掉：

| 删除项 | 大小 | 为什么删 |
|:---|:---|:---|
| node-llama-cpp（linux-x64 预编译） | 685MB | 本地 LLM 推理引擎，可选功能，且是 x86 版 |
| koffi | 85MB | FFI 外部函数接口，x86 预编译 |
| pdfjs-dist | 40MB | PDF 解析，暂不需要 |
| @napi-rs/canvas（linux-x64 预编译） | 32MB | Canvas 渲染，x86 预编译 |
| typescript | 23MB | 运行时不需要编译器 |
| sharp/libvips（linux-x64 预编译） | 16MB | 图片处理，x86 预编译 |

**685MB 的 node-llama-cpp 是最大的一坨。** 这是一个在本地跑 LLM 推理的库（底下是 llama.cpp），OpenClaw 的可选功能。我们用 DeepSeek 云端 API，不需要本地推理。删。

六刀砍完，node_modules 从 1.3GB → 378MB。加上 dist/ 66MB、control-ui、模板文件，打成 tar.gz——**67MB**。

从 2GB 到 67MB。能接受了。

◆ 启动 Gateway——三个前置条件

━━━━━━━━━━━━━━━━━━━━

Gateway 是 OpenClaw 的核心服务：它起一个 HTTP 服务器，提供 Control UI 网页、接收聊天消息、调用大模型 API、返回 AI 回复。

启动命令长这样：

```bash
HOME=/data/local/tmp LD_LIBRARY_PATH=/system/lib64 \
/data/local/tmp/node /data/local/tmp/openclaw/openclaw.mjs \
gateway run --bind lan --port 18800 --force
```

三个关键点：

**1. `HOME=/data/local/tmp`**

OpenClaw 启动时会往 `$HOME/.openclaw/` 目录写配置文件和数据库。板子上默认 HOME 是 `/`——往根目录写文件？没权限。设成 `/data/local/tmp` 就行了，这个目录是 OH 上的自由区。

**2. `gateway.mode=local`**

这是在配置文件 `openclaw.json` 里设的。Gateway 默认模式会尝试连云端服务，板子上没那个条件。`local` 模式纯本地运行。

**3. `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback=true`**

看名字就知道是个"危险选项"。Gateway 绑定 `--bind lan` 时，监听的不是 `127.0.0.1` 而是局域网 IP。OpenClaw 的安全机制默认拒绝非 loopback 来源的请求。加这个选项告诉它：我知道我在干什么，放行。

━━━━━━━━━━━━━━━━━━━━

◆ 端口转发：secure context 的要求

━━━━━━━━━━━━━━━━━━━━

Gateway 在板子上跑在 18800 端口。怎么从宿主机的浏览器访问？

直接用板子的 IP 地址加端口？不行。Control UI 用了 Web Crypto API 等浏览器高级功能，这些功能要求 **secure context**——要么 HTTPS，要么 localhost。板子上没有 TLS 证书，搞 HTTPS 太折腾。

解法很简单：**hdc 端口转发。**

```bash
hdc fport tcp:18800 tcp:18800
```

这条命令让宿主机的 `localhost:18800` 转发到板子的 `18800` 端口。浏览器访问 `http://localhost:18800`——是 localhost，满足 secure context 要求。中间走的是 USB，不经过网络。

打开浏览器——Control UI 出来了。一个干净的聊天界面。

但发消息之前，还得配 AI 模型。

━━━━━━━━━━━━━━━━━━━━

◆ DeepSeek 配置：三个坑，一个比一个深

━━━━━━━━━━━━━━━━━━━━

OpenClaw 支持多种 AI 模型提供商。我们要接 DeepSeek。需要在 `openclaw.json` 配置文件里告诉 Gateway：DeepSeek 的 API 地址是什么、用什么 key、走什么协议。

听起来简单。但配置格式的每一层都有坑。

────────────────────

**第一坑：providers 放错了层级**

第一版配置，把 `providers` 放在 config 的顶层：

```json
{
  "providers": {
    "deepseek": { "baseUrl": "...", "apiKey": "..." }
  }
}
```

报错：`Unknown model: deepseek/deepseek-chat`

OpenClaw 看都不看你的 providers。因为放错地方了。

**正确路径是 `models.providers.custom`**，不是顶层 `providers`。OpenClaw 的配置嵌套很深——`models` 下面套 `providers`，`providers` 下面套 `custom`（自定义提供商）。

────────────────────

**第二坑：models 写成了对象**

改对了层级，第二版：

```json
{
  "models": {
    "providers": {
      "custom": {
        "models": { "id": "deepseek-chat", "name": "DeepSeek Chat" }
      }
    }
  }
}
```

报错：`expected array, received object`

`models` 字段要的是**数组**，不是对象。即使只有一个模型，也得用方括号包起来：

```json
"models": [{ "id": "deepseek-chat", "name": "DeepSeek Chat" }]
```

JSON 配置里"一个元素的数组"和"一个对象"看着差不多，但对代码来说是两种完全不同的数据结构。这种坑在 JSON 配置里永远不会绝迹。

────────────────────

**第三坑：没指定 API 协议——最深的一个**

层级对了，格式对了。发消息——

```
404 status code (no body)
```

404？API 地址明明是对的，DeepSeek 的 key 也填了。为什么 404？

排查发现：**OpenClaw 默认用 OpenAI 的 Responses API**——请求路径是 `/v1/responses`。这是 OpenAI 2025 年新推的 API 格式。

但 DeepSeek 不支持 Responses API。DeepSeek 只支持传统的 Chat Completions API——路径是 `/v1/chat/completions`。

OpenClaw 请求 `https://api.deepseek.com/v1/responses`，DeepSeek 说：这个路径我不认识。404。

修法：在模型配置里显式指定 API 协议：

```json
{
  "id": "deepseek-chat",
  "name": "DeepSeek Chat",
  "api": "openai-completions",
  "contextWindow": 65536,
  "maxTokens": 8192
}
```

**`"api": "openai-completions"`** 告诉 OpenClaw：别走 Responses API，走传统的 Chat Completions API。

这个坑最深，因为 404 不告诉你"你走错了 API 协议"——它只告诉你"找不到"。你得知道 OpenAI 有两套 API、OpenClaw 默认走哪套、DeepSeek 支持哪套，才能定位到这个 `"api"` 字段。

────────────────────

三个坑填完，最终正确的配置：

```json
{
  "agents": {
    "defaults": {
      "model": { "primary": "custom/deepseek-chat" }
    }
  },
  "models": {
    "providers": {
      "custom": {
        "baseUrl": "https://api.deepseek.com/v1",
        "apiKey": "sk-xxx",
        "models": [
          {
            "id": "deepseek-chat",
            "name": "DeepSeek Chat",
            "api": "openai-completions",
            "contextWindow": 65536,
            "maxTokens": 8192
          }
        ]
      }
    }
  },
  "gateway": {
    "mode": "local",
    "controlUi": { "dangerouslyAllowHostHeaderOriginFallback": true },
    "auth": { "mode": "token", "token": "xxx" }
  }
}
```

◆ 最后一个坑：hdc shell 退出后进程被杀

━━━━━━━━━━━━━━━━━━━━

Gateway 跑起来了，DeepSeek 配好了，浏览器能聊天了。正在测试的时候，切了个窗口干别的事，回来发现——Gateway 挂了。

OpenHarmony 的 shell 有个"好习惯"：**hdc shell 退出时，会把 shell 下的子进程一起杀掉。** 不是 nohup 能解决的，是 OH 的进程管理机制——shell session 结束，session 内所有进程回收。

这意味着你不能把 Gateway 丢后台然后关掉 shell。**必须保持一个 hdc shell 窗口前台运行着。**

对生产部署来说这不理想，但对于概念验证——能跑就行。

◆ 复盘：上下两篇合在一起看

━━━━━━━━━━━━━━━━━━━━

上篇（122 期）和这篇加起来，完整的故事是：

**上篇——地基**：交叉编译 Node.js 22.22.1。改了两行代码（C++17→C++20、zlib CRC32 条件修复），踩了两次 SIGBUS（USB 线损坏、libc++.so 冲突），最终 `node --version` 跑通。

**下篇——建筑**：在地基上跑 OpenClaw。构建 TS→JS，2GB 瘦身到 67MB，`--version` 一次过。然后 Gateway 启动踩三个配置坑（providers 层级、models 格式、API 协议），端口转发解决 secure context，最终在浏览器里和 DeepSeek 聊上天。

两篇合在一起的模式很清楚：**上篇的坑在 C/C++ 层（编译器、链接器、二进制传输），下篇的坑在应用层（JSON 配置、API 协议、进程管理）。** 层次越高，坑越"软"——不再是 segfault 和 SIGBUS，而是 404 和 `expected array`。

回头看几个教训：

**瘦身比你想象的有效。** node_modules 2GB 看着吓人，但里面大头是几个 x86 预编译包。685MB 的 node-llama-cpp 一删，轻了三分之一。很多时候"太大了搬不动"不是真正的约束——先看看里面什么占了大头，再决定要不要怕。

**配置格式的坑比编译的坑更磨人。** 编译报错至少告诉你哪行有问题。JSON 配置放错了层级，报错只说 `Unknown model`——你得自己推理出"是层级放错了"。404 不告诉你"你走错了 API 协议"——你得知道有两套 API 的存在才能想到这个方向。**编译的坑像数学题，有标准解法；配置的坑像侦探小说，靠线索推理。**

**人机协作的效率在这种项目里特别明显。** 我负责板子上的操作——推文件、看报错、验证结果。Claude Code 负责分析配置格式、查 OpenClaw 文档、推理 API 协议差异。三个 DeepSeek 配置坑，如果我一个人查文档试错，可能要花一下午。人机协作，每个坑十分钟内定位到根因。

◆ 完整部署清单

━━━━━━━━━━━━━━━━━━━━

以下步骤假设你有一块 OH 6.0 RK3568 板子，hdc 连上了。照做就能跑。

仓库里有预编译产物（`build/node-ohos` 和 `build/openclaw-deploy.tar.gz`），不需要自己编译。想从头编译的看 README。

**第一步：获取项目**

```bash
git clone https://github.com/lmxxf/openclaw-openharmony.git
cd openclaw-openharmony
```

**第二步：推 Node.js 到板子**

```bash
hdc file send build/node-ohos /data/local/tmp/node
hdc shell "chmod +x /data/local/tmp/node"

# 验证
hdc shell "LD_LIBRARY_PATH=/system/lib64 /data/local/tmp/node --version"
# 应输出：v22.22.1
```

**第三步：推 OpenClaw 到板子**

```bash
hdc file send build/openclaw-deploy.tar.gz /data/local/tmp/openclaw-deploy.tar.gz
hdc shell "mkdir -p /data/local/tmp/openclaw && cd /data/local/tmp/openclaw && tar xzf /data/local/tmp/openclaw-deploy.tar.gz && rm /data/local/tmp/openclaw-deploy.tar.gz"

# 验证
hdc shell "LD_LIBRARY_PATH=/system/lib64 /data/local/tmp/node /data/local/tmp/openclaw/openclaw.mjs --version"
# 应输出：OpenClaw 2026.3.14 (8db6fcc)
```

**第四步：写配置文件**

在板子上创建 `/data/local/tmp/.openclaw/openclaw.json`：

```json
{
  "agents": {
    "defaults": {
      "model": { "primary": "custom/deepseek-chat" }
    }
  },
  "models": {
    "providers": {
      "custom": {
        "baseUrl": "https://api.deepseek.com/v1",
        "apiKey": "你的DeepSeek API Key",
        "models": [
          {
            "id": "deepseek-chat",
            "name": "DeepSeek Chat",
            "api": "openai-completions",
            "contextWindow": 65536,
            "maxTokens": 8192
          }
        ]
      }
    }
  },
  "gateway": {
    "mode": "local",
    "controlUi": { "dangerouslyAllowHostHeaderOriginFallback": true },
    "auth": { "mode": "token", "token": "你的自定义token" }
  }
}
```

**第五步：启动 Gateway**

```bash
hdc shell
# 进入 shell 后：
HOME=/data/local/tmp LD_LIBRARY_PATH=/system/lib64 \
/data/local/tmp/node /data/local/tmp/openclaw/openclaw.mjs \
gateway run --bind lan --port 18800 --force
```

注意：**不要关闭这个 shell 窗口**。关了 Gateway 会被 OH 回收。

**第六步：端口转发 + 浏览器访问**

另开一个终端：

```bash
hdc fport tcp:18800 tcp:18800
```

浏览器打开：`http://localhost:18800/#token=你的自定义token`

◆ 术语速查

━━━━━━━━━━━━━━━━━━━━

| 术语 | 解释 |
|:---|:---|
| pnpm monorepo | 用 pnpm 管理的多包项目。一个仓库里放多个相关的 npm 包，共享依赖 |
| tsdown | TypeScript 构建工具，把 .ts 编译成 .js |
| dist/ | Distribution 的缩写，构建产物目录。TypeScript 编完的 JavaScript 文件放这里 |
| node_modules/ | Node.js 项目的依赖目录。npm/pnpm install 下载的第三方包全在这里 |
| pnpm deploy --prod | 从 monorepo 里提取生产依赖，去掉开发依赖（测试框架、linter 等） |
| native addon | Node.js 的原生扩展（.node 文件），C/C++/Rust 编译出的动态库 |
| Gateway | OpenClaw 的核心服务进程。起 HTTP 服务器，接收消息，调 AI API，返回回复 |
| Control UI | Gateway 的 Web 管理界面。浏览器里打开，可以直接聊天 |
| secure context | 浏览器安全策略：某些 API 只在 HTTPS 或 localhost 下可用 |
| hdc fport | HarmonyOS 设备端口转发。把宿主机的端口映射到板子的端口，走 USB |
| openai-completions | OpenAI Chat Completions API 协议（/v1/chat/completions），传统格式 |
| openai-responses | OpenAI Responses API 协议（/v1/responses），2025 年新推的格式 |
| --bind lan | Gateway 监听局域网 IP（而非仅 127.0.0.1），允许非本机访问 |
