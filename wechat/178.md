◆ 前情提要

━━━━━━━━━━━━━━━━━━━━

176 期（ https://mp.weixin.qq.com/s/e1GKi53IociZnfHX1Ie9_w ）讲了 GPU 编程的五层塔——从纯 PyTorch 到原生 CUDA，我们在 DGX Spark（sm_121）上被迫一层层往下爬。那一期的核心结论有两个：

第一，DeepSeek V4 Flash 280B 在 vLLM 里跑推理时，FP4 MoE 路径走的 Marlin kernel 在 sm_120+（DGX Spark、RTX 5090、RTX 6000 Pro 等消费级 Blackwell 系列）上会**静默算错**——不报错，但输出是垃圾。

第二，我们用 CUTLASS 的 sm_120+ 模板写了一个验证实验（Consumer-DeepGEMM），走完全不同的计算路径替换 Marlin——输出终于正确了。这证明了问题确实在 Marlin 的 FP4 kernel，而不是模型权重、tokenizer 或其他环节。但验证归验证，速度只有 0.8 tok/s（每个 token 要 launch 120 次 CUTLASS kernel），不能当日常工具用。

这一期讲后续：社区怎么解决这个问题的，我们怎么跟进的，中间走过哪些弯路，以及目前 DGX Spark 上跑 DeepSeek V4 Flash 280B 的真实可用状态。

如果你没看 176 期也没关系——这一期是独立可读的。需要的背景知识我会重新交代。

◆ 问题回顾：Marlin 在 sm_120+ 上静默算错

━━━━━━━━━━━━━━━━━━━━

简单回顾一下根因。不看 176 期也能看懂这段。

先解释一下 Marlin。176 期讲了 GPU 编程的五层塔——从纯 PyTorch 到原生 CUDA。Marlin 在五层塔里属于**第 1 层（原生 CUDA）**，和 DeepGEMM 同一层级。它是 IST-DASLab（奥地利格拉茨工业大学的研究组）写的一组手写 CUDA kernel，专门为量化模型推理加速——支持 INT4、FP8、FP4 等多种量化格式，每种格式有对应的 kernel 实现。vLLM 把 Marlin 集成在框架里，作为量化推理的默认加速方案。和 CUTLASS（第 2 层，C++ 模板库，"声明算什么让编译器决定怎么算"）不同，Marlin 是手写的 kernel 成品——直接操控线程、共享内存、寄存器，性能极致但移植性差。

Marlin 的 FP4 kernel 用了数据中心 GPU（sm_90/sm_100）专属的 tensor core 指令来做矩阵乘法。这些指令在 sm_120+ 上不是"不存在"——而是"存在但行为不一致"。具体来说，某些 FP4 MMA（Matrix Multiply-Accumulate）指令在 sm_120+ 上的数据布局解释和数据中心卡不同，算出来的数是错的，但 CUDA runtime 不会报任何错误。

表现很有迷惑性：

- 中文短回复（"2+2 等于几"→"4"）正常，回答正确
- 中文中等长度回复（"介绍万里长城"）正常
- 英文回复、代码生成、长输出 → 从第一个 token 开始跑飞，全是乱码或重复字符

如果你只测中文短问答，会以为一切正常——直到用户问了一个英文问题。这也是为什么很多人在 Reddit 和 GitHub issue 里报告"DeepSeek V4 在我的 RTX 5090/6000 Pro 上跑得好好的啊"——他们只测了中文或短回复。**静默错误是最危险的**——它不像 crash 一样让你立刻知道出了问题，而是让你在错误的结果上继续工作。

为什么中文对、英文错？不是因为模型"中文能力强"——是因为 MoE（Mixture of Experts）架构的路由机制。

简单解释 MoE：传统 transformer 每一层有一个大 FFN（前馈网络），所有 token 都过同一个网络。MoE 把这个大 FFN 拆成 256 个小 FFN（"专家"），每个 token 只过其中 6 个（DeepSeek V4 Flash 的 top-6 策略）。哪 6 个由一个"路由网络"决定——路由网络看 token 的语义嵌入，算一个 softmax，选得分最高的 6 个专家。

不同语言、不同语义的输入经过路由网络后，会激活不同的专家组合。中文 prompt 激活的那组专家恰好避开了 Marlin 算错的那几个；英文 prompt 激活了另一组专家，恰好命中了算错的那些。256 个专家里可能只有十几个的 FP4 MMA 会算错——但只要命中一个，那一层的输出就歪了，歪了的输出传到下一层，继续歪，最终的 logits 已经面目全非。

定位到 Marlin 后，我们需要**验证**——真的是 Marlin 算错，而不是别的环节？最直接的验证方法：换一条完全不同的计算路径，看输出是否正确。

176 期讲过，我们用 CUTLASS 的 sm_120 block-scaled MX 模板写了一个验证实验（Consumer-DeepGEMM）。CUTLASS 走的是 `mma.sync` + block-scaled MX 这条指令路径，不依赖 Marlin 用的那些有问题的指令。结果：**输出正确了**——"万里长城"测试完美通过，finish_reason=stop。

这就是铁证：**换掉 Marlin 的计算路径，同样的模型权重、同样的 tokenizer、同样的 prompt，输出从垃圾变成正确。问题确实在 Marlin 的 FP4 kernel，不在别处。**

但 Consumer-DeepGEMM 不能当日常工具用——0.8 tok/s，每生成一个 token 要 launch 120 次 CUTLASS grouped GEMM，每次都有 CPU↔GPU 同步等待。它的价值是验证根因，不是替代方案。

◆ 我们是如何定位到 Marlin 的

━━━━━━━━━━━━━━━━━━━━

静默错误的定位比 crash 难得多——程序不报错，不崩溃，就是输出不对。你得自己证明"哪一层出了问题"。

**第一轮：排除"不是 kernel 的问题"**

拿到垃圾输出后，首先怀疑的不是 GPU kernel——而是更上层的东西：

1. **chat template**——DeepSeek V4 的 tokenizer_config.json 没有 chat_template 字段（V4 设计如此，用 `--tokenizer-mode deepseek_v4` 代替），怀疑模板格式拼错了。手动拼正确格式的 prompt 走 `/v1/completions` 端点（绕过 chat template），**还是出垃圾**——排除
2. **tokenizer**——怀疑特殊 token 编码有误。`PreTrainedTokenizerFast` 直接编码验证，`<｜User｜>` → 128803 单 token，完全正确——排除
3. **`</think>` bug**——发现 `deepseek_v4_encoding.py` 第 403 行在 chat 模式下错误插入了 `</think>`。修了，prompt_tokens 从 14 降到 13——修复生效，**但垃圾依旧**。这是一个真实的 bug，但不是根因
4. **权重文件**——46 shard、148GB、config 正常——排除
5. **镜像差异**——换回 Docker Hub 原版镜像，**同样出垃圾**——bug 一直存在，之前短对话没暴露

到这一步，上层原因全部排除。问题在 vLLM 的推理计算层。

**第二轮：日志指向 Marlin**

vLLM 启动日志里有一行字：

```
Using 'MARLIN' Mxfp4 MoE backend
```

**这行日志直接告诉你——FP4 MoE 计算走的是 Marlin backend。**拿着 "Marlin SM120" 搜 GitHub，搜到两个已知 issue：vllm#40928（DeepSeek V4 Flash SM120 Triton fallback 请求）和 cutlass#3096（CUTLASS NVFP4 在 SM120 上初始化失败）。社区已经知道 Marlin 在 SM120 上有问题，但没人修。

**第三轮：穷举验证"没有替代路径"**

确认 Marlin 有问题后，第一反应是换一个 MoE backend——vLLM 支持好几种。全部试了一遍：

| 尝试 | 结果 |
|------|------|
| `--moe-backend triton` | Triton MoE 不支持 SM120 |
| `--moe-backend emulation` | 不支持 MXFP4 权重格式转换 |
| `--moe-backend cutlass` | 不在 MXFP4 可选后端列表 |
| `--moe-backend flashinfer_cutlass` | 不支持 SM120 量化格式 |

**SM120 上只有 Marlin 能跑 FP4 MoE，但 Marlin 在 SM120 上算错。其他所有 backend 要么不支持 SM120，要么不支持 MXFP4。死胡同。**

所有现成的路都走不通。这就是为什么我们从 CUTLASS 模板层开始写 Consumer-DeepGEMM——不是为了做一个替代库，是为了验证"换一条计算路径是否能解决问题"。答案是能。也是为什么 jasl 后来选择用 Triton 重写——思路一样，绕过 Marlin，但选了一个更适合日常使用的层次。

定位过程的关键教训：**先看日志，再搜 issue，最后穷举验证**。启动日志里那一行 `Using 'MARLIN' Mxfp4 MoE backend` 价值千金——没有它，我们可能还在怀疑 tokenizer 或者权重文件。vLLM 的日志系统在这方面做得不错，每个 kernel 路径的选择都会打印出来。

◆ 错误长什么样，我们怎么验证修复的

━━━━━━━━━━━━━━━━━━━━

定位到 Marlin、穷举验证没有替代路径之后，下一步是理解错误的本质——然后自己修。

先看 vLLM 里 DeepSeek V4 的 MXFP4 MoE 走的到底是哪段代码。这个 kernel 在 vLLM 源码的 `csrc/libtorch_stable/quantization/fp4/mxfp4_blockwise_moe_kernel.cu`，468 行 CUTLASS C++ 模板代码。注意——这不是 Marlin 本体（Marlin 在 `csrc/quantization/marlin/` 下面），而是 vLLM 为 MXFP4 格式单独写的 CUTLASS grouped GEMM。两者的关系是 fallback：vLLM 先尝试这个 CUTLASS kernel，被拒绝后退到 Marlin。

核心部分长这样（从 468 行里截取关键片段）：

```cpp
// 第一部分：声明计算类型——和 176 期 CUTLASS 那段几乎一样
using ElementA = cutlass::mx_float4_t<cutlass::float_e2m1_t>;   // 激活：MXFP4
using ElementB = cutlass::mx_float4_t<cutlass::float_e2m1_t>;   // 权重：MXFP4
using ElementC = OutType;   // 输出：BF16 或 FP16
using ArchTag = cutlass::arch::Sm100;   // ← 硬编码 sm_100

// 第二部分：kernel 调度策略——这里是 sm_100 专属
using KernelSchedule =
    cutlass::gemm::KernelPtrArrayTmaWarpSpecialized1SmMxf4Sm100;
//                                                       ^^^^^
//   注意名字里的 "TmaWarpSpecialized" 和 "Sm100"
//   TMA = 176 期讲过的 Tensor Memory Accelerator（硬件自动搬运张量数据）
//   WarpSpecialized = Warp 特化调度（不同 warp 分工——有的搬数据，有的算）
//   Sm100 = 只给数据中心 Blackwell 生成代码
```

176 期讲五层塔第 2 层（CUTLASS）时说过——CUTLASS 的本质是"你声明算什么，编译器决定怎么算"。这里声明的就是"MXFP4 × MXFP4 → BF16，在 sm_100 上用 TMA + Warp Specialization 策略"。CUTLASS 根据这些类型参数，在编译期自动展开出使用 TMA 硬件搬运和 `tcgen05` tensor core 指令的 kernel 代码。

然后是架构检查——kernel 入口处的门卫：

```cpp
// 运行时检查：只放行 sm_100~sm_119
int32_t version_num = get_sm_version_num();
if (version_num >= 100 && version_num < 120) {
    run_mxfp4_blockwise_scaled_group_mm_sm100<OutType>(...);  // 放行
    return;
}
// sm_120+ 到这里直接报错退出
STD_TORCH_CHECK_NOT_IMPLEMENTED(false,
    "No compiled cutlass_mxfp4_group_mm kernel for CUDA device capability: ",
    version_num, ". Required capability: 100");
```

`version_num >= 100 && version_num < 120`——sm_100 到 sm_119 放行，sm_120+ **直接被挡**。DGX Spark 的 sm_121？不好意思，`Required capability: 100`。

这不是 bug——是设计。写这个 kernel 的人知道 sm_120+ 没有 TMA 和 `tcgen05`，跑了也是算错，不如直接拒绝。问题是**没有提供 sm_120+ 的替代路径**——sm_120+ 用户到了这里就死路一条。

vLLM 对 FP4 MoE 有多条 fallback 路径。CUTLASS 这条明确拒绝 sm_120+，vLLM 就退到下一条——Marlin 的 FP4 量化通路。前面说过 Marlin 是一组手写 CUDA kernel 的集合，它里面有处理 FP4 的 kernel，但**没有做 sm_120+ 的架构检查**——不拒绝，直接跑，然后静默算错。所以实际情况是：
- CUTLASS MXFP4 kernel 对 sm_120+ **拒绝执行**（正确行为——不能算就别算）
- Marlin 的 FP4 kernel 对 sm_120+ **允许执行但结果错误**（这才是祸根）

vLLM 的 fallback 逻辑是：**CUTLASS 拒绝 → 退到 Marlin → Marlin 不拒绝 → 算错 → 输出垃圾**。如果 Marlin 也像 CUTLASS 一样直接拒绝，用户至少会看到报错而不是静默乱码。

这段 kernel 代码里还有一个细节值得注意——偏移量计算 kernel `__mxfp4_get_group_gemm_starts`：

```cpp
// MoE 的 256 个专家，每个专家的权重矩阵在显存里连续存放
// 这个 kernel 计算每个专家的矩阵起始地址（pointer array）
__global__ void __mxfp4_get_group_gemm_starts(
    ElementAB** a_offsets, ElementAB** b_offsets, ...
    const int32_t* expert_offsets, ...) {
  int64_t expert_id = threadIdx.x;
  // 每个专家的权重偏移量
  b_offsets[expert_id] = b_base + expert_id * n * half_k;
  // 每个专家的 scale factor 偏移量
  b_scales_offsets[expert_id] = b_scales_base + expert_id * n * group_k;

  // TMA 要求 128 字节对齐——又是 TMA
  assert((reinterpret_cast<uintptr_t>(a_scales_offsets[expert_id]) % 128) == 0
         && "TMA requires 128-byte alignment");
}
```

这是 grouped GEMM 的 pointer array 设置——MoE 的 256 个专家矩阵在显存里连续存放，kernel 要算出每个专家的起始地址。注意 `TMA requires 128-byte alignment` 这行 assert——连偏移量计算都依赖 TMA 的对齐要求。**整个 kernel 从头到尾都假设 TMA 存在**——不是"加个 if 判断就能绕过"的程度，是基础架构完全绑定。

这就是为什么不能简单地"给这个 kernel 加个 sm_120 分支"——**需要重写整个计算策略**，换一套不依赖 TMA 的数据搬运方案和不依赖 `tcgen05` 的矩阵乘法指令。

**所以我们重写了。** Consumer-DeepGEMM（核心 CUDA 文件：https://github.com/lmxxf/Consumer-DeepGEMM/blob/main/csrc/cutlass_mxfp8_mxfp4_probe.cu ）做了四件事：

1. **换计算指令**：把 `tcgen05`（sm_100 专属）换成 `mma.sync` + block-scaled MX（sm_120 支持）
2. **换数据搬运**：把 TMA 硬件自动搬运换成手动构造 pointer array——每个专家的权重起始地址、scale factor 偏移量，自己算、自己对齐、自己搬
3. **换 scale factor 布局**：CUTLASS sm_120 的 block-scaled MX kernel 对 scale factor 有特殊的 SfAtom tile-interleaved 内存布局要求（176 期详细讲过），和 vLLM 传来的 row-major `[M, K/128]` 不同，要做 `repeat_interleave(4)` 扩展 + 按 tile 结构重排
4. **接入 vLLM**：写 Python shim 包装层，让 vLLM 的 MoE dispatch 逻辑能调用我们的 CUTLASS kernel，替换掉 Marlin

结果：**输出正确了**。"万里长城"从垃圾变成完美中文，英文 quicksort 也正确——因为我们替换的是最底层的 GEMM 计算，不管哪个专家被激活都走 CUTLASS 路径，所有 256 个专家的计算结果都是对的。

**这就是铁证：换掉计算路径，同样的权重、同样的 tokenizer、同样的 prompt，输出从垃圾变正确。问题在 Marlin 的 FP4 kernel，不在别处。**

但速度只有 0.8 tok/s——每生成一个 token 要 launch 120 次 CUTLASS grouped GEMM（60 层 × 每层 2 个 FC），每次都有 CPU↔GPU 同步等待。Consumer-DeepGEMM 的价值是**定位根因 + 验证修复方向**，不是日常可用的替代方案。后面 jasl 用 Triton 走同样的"绕过 Marlin"思路，把这个验证结论变成了 14 tok/s 的日常可用方案。

◆ vLLM 官方做了什么、没做什么

━━━━━━━━━━━━━━━━━━━━

在我们折腾 Consumer-DeepGEMM 的同时，vLLM 官方也没闲着——但方向不在 sm_120+。

**vLLM 0.20.0**（2026-04-27 发布）：
- 合入了 DeepSeek V4 基础模型支持（PR #40860）——模型定义、tokenizer、MLA 注意力、Hyper-Connection 等基础设施到位
- 加了 SM120 CUTLASS blockwise FP8 GEMM 优化——普通 FP8 linear 路径在 sm_120+ 上跑通了
- 加了 MXFP4 W4A4 CUTLASS MoE——但只针对 **sm_100**（数据中心 Blackwell），sm_120+ 没份

**vLLM 0.20.1**（2026-05-04 发布）：
- DeepSeek V4 多流 pre-attention GEMM 优化
- PTX 加速 FP32→FP4 转换
- BF16/MXFP8 all-to-all FlashInfer 支持
- **仍然没有 sm_120+ 的 MXFP4 MoE 修复**

换句话说：vLLM 0.20.x 让 DeepSeek V4 在数据中心 GPU（H100/B200/B300）上跑得更好了，但在 sm_120+（DGX Spark、RTX 5090、RTX 6000 Pro）上的核心问题——**Marlin FP4 MoE 静默算错**——原封不动。

vLLM 的 issue 挂了两个，都 Open 着，没人 assign：
- **#40928**：要求给 DeepSeek V4 Flash 在 sm_120 上加 Triton fallback
- **#41063**：列出了 DeepGEMM 在 sm_12.x 上的全部 kernel 覆盖缺口

社区给 vLLM 提了 PR，也没合：
- **PR #40991**（jasl）：用 Triton 重写 DeepSeek V4 的关键 kernel，sm_120+ 上验证通过——draft 状态，还在 review
- **PR #40082**（meena-at-work）：FlashInfer CuTeDSL b12x backend——但只给 NVFP4 模型做的，对 DeepSeek V4 的 MXFP4 路径无效（后面详讲）

另一边，DeepSeek 自家的 DeepGEMM 库更直接——issue #236 里明确表示**没有计划支持 sm_120+**。合理——DeepGEMM 是给 DeepSeek 自家数据中心写的，消费级卡不在他们的业务范围内。

官方的态度可以理解——sm_120+ 用户是少数，数据中心客户才是 vLLM 的核心用户群。但对我们这些花了几万块买 DGX Spark 的人来说，"等官方排期"不是选项。

所以社区自己动手了。

GitHub 用户 jasl 专门给 sm_120 系列做了 vLLM 适配，维护在 `github.com/jasl/vllm` 的 `ds4-sm120` 分支上。他的方法不是给 Marlin 打补丁，也不是用 CUTLASS 替换 Marlin——而是**用 Triton 重写了 DeepSeek V4 用到的关键 kernel**。

但我们 5-02 晚第一次用 jasl 的代码时**失败了**——那时候 jasl 的分支基于更早版本的 vLLM，和我们用的 eugr 镜像内部 API 不兼容。尝试用 Docker volume 挂载替换个别 .py 文件，结果 `ImportError: cannot import name 'dequantize_combined_sparse_mla_decode_kv'`——两个版本的 vLLM 内部 API 对不上，部分替换行不通。整个 fork 从源码编译又要一小时，Docker 里还访问不了 GitHub。

走不通，才退回去自己做 Consumer-DeepGEMM 验证根因。

然后 5-04，jasl 大幅更新了代码（+31000 行，454 文件变更），rebase 到了 vLLM 0.20.x 基础上。eugr/spark-vllm-docker 也更新了构建脚本。**两边版本终于对齐了**——两天前对不齐，现在对齐了，时间窗口的问题。

这次 rebase 不只是对齐 API——jasl 的 Triton 重写覆盖范围也更完整了。4-29 我们用的早期版本里，很多计算路径还是 fallback 到 Marlin（所以中文长回复也出垃圾）；rebase 到 0.20.x 后，更多路径被 Triton 接管，中文 prompt 激活的专家组合全部走 Triton 路径，不再碰 Marlin——中文长回复变正确了。英文还有问题说明仍有少量残余路径 fallback 到 Marlin（后面详讲）。

jasl 的 fork 里用 Triton 重写了 DeepSeek V4 在 vLLM 中用到的这些关键 kernel：
- **sparse MLA**（Multi-head Latent Attention，DeepSeek V4 特有的注意力机制——把 KV cache 压缩到低秩空间，省显存）
- **FP8 einsum**（激活值的矩阵运算——MoE 层里每个专家的前馈网络核心计算）
- **paged MQA**（分页多查询注意力，vLLM 的 KV cache 管理核心——让多个请求共享 GPU 显存）
- **MoE routing + dispatch**（路由网络决定哪些 token 发给哪些专家，以及实际的分发和收集）

用 Triton 重写而不是给 Marlin 打补丁，和我们 Consumer-DeepGEMM 的思路本质相同——**绕过 Marlin，走一条 sm_120+ 上正确的计算路径**。区别在于层次选择：我们用 CUTLASS（五层塔第 2 层）验证了"换路径就能修"，jasl 用 Triton（第 4 层）把这个结论做成了日常可用的方案。176 期讲过，Triton 是五层塔里唯一天然跨硬件的——编译器根据目标 GPU 自动生成对应的机器码，不存在 Marlin 那种"数据中心指令拿到消费级卡上算错"的问题。

jasl 把这个工作提交给了 vLLM 官方——PR #40991，目前是 draft 状态，还没合进主线。我们的实测结果见下一节。

◆ 我们的实测

━━━━━━━━━━━━━━━━━━━━

我们用 eugr/spark-vllm-docker 的构建工具，配合 jasl 源码编译。

eugr/spark-vllm-docker 是另一位社区开发者做的 Docker 构建框架——把 vLLM 在 DGX Spark 上编译所需的所有依赖（CUDA toolkit 13.0、PyTorch nightly、flash-attention、Triton 等）打包成 Dockerfile，一键构建。DGX Spark 是 ARM64 架构（Grace CPU），很多 Python 包没有预编译的 ARM wheel，必须从源码编译——如果不用 Docker 统一管理依赖链，你会在各种 `gcc` 版本不匹配和 `cmake` 找不到库的报错里浪费一整天。

具体操作就是改 Dockerfile 里一行 `git clone` 的 URL，从 vLLM 官方仓库换成 jasl 的 fork，分支切到 `ds4-sm120`。然后 `docker build`。ARM64 上编译约 30 分钟（主要时间花在 flash-attention 和 vLLM 的 C++ 扩展编译上）。相比之下，我们之前编译 Consumer-DeepGEMM 的 CUTLASS 模板每次改参数要 40 分钟——Triton 的 JIT 编译模式在开发迭代上也有优势。

实测结果：

```text
测试 1："2+2等于几"
→ 正确回答 "4"
→ 10 tokens, 16.8s（首次请求含模型 warmup + JIT 编译）
→ 后续请求相同问题：<1s

测试 2："请用500字介绍万里长城"
→ 完美输出，流畅中文，内容准确
→ 414 tokens, 29.3s（~14.1 tok/s）
→ finish_reason=stop（模型自然结束，不是被截断）

测试 3："写一个Python快速排序函数"
→ 完美输出，代码正确可运行
→ 500 tokens, 36.5s（~13.7 tok/s）
→ finish_reason=stop

测试 4："What is quicksort? Explain with code."
→ 开头出现垃圾 token（乱码字符和重复片段）
→ 后续英文全程有问题，不会自己修正回来
→ 代码部分混入不合法的 Python 语法
```

以上是 5-04 的测试结果。5-05 jasl 又推了几个 commit（MLA prefix cache 修复、prefill workspace 预留、MoE clamp limit 等），我们拉了最新代码重新编译测试：

```text
5-05 最新版测试：

测试 1："请用500字介绍万里长城"
→ 完美输出，336 tokens，finish_reason=stop ✅

测试 2："What is quicksort? Explain with code."
→ 开头仍然是垃圾（法语 plutôt、中文"而非""司空"、乱标点混在一起）
→ 但比 5-04 好一点：模型自己意识到了乱码，输出了 "Ah, I see that my previous
  attempt ended up generating a lot of garbled text"，然后自己修正回来给出了
  正确的 quicksort 解释和代码
→ 开头垃圾仍在，Marlin 残留路径没修干净 ❌
```

结论不变：**中文完全可用，英文仍有问题。** 5-05 的更新让模型的自我修正能力好了一点（能意识到自己输出了垃圾并尝试纠正），但根因——Marlin FP4 MoE 在 sm_120+ 上的残余 fallback 路径——还在。

中文场景完全可用——无论是问答、写作还是代码生成（只要用中文 prompt）。英文场景还有问题——这是社区已知的 "occasional spurious tokens" 现象。

对比一下速度：

- **vLLM 官方 main**：输出是垃圾，速度无意义
- **Consumer-DeepGEMM**（验证用，CUTLASS 路径）：正确，**0.8 tok/s**
- **jasl/vllm fork**（Triton 重写）：中文正确，**14 tok/s**

从 0.8 到 14，**快了约 18 倍**。从"每个字等 1.2 秒"到"每秒 14 个 token"——前者是折磨，后者是正常的聊天体验。

为什么 Triton 重写比 CUTLASS 替换快这么多？不是因为 Triton kernel 本身比 CUTLASS kernel 快（实际上单次 kernel 执行，CUTLASS 可能更快），而是因为：

1. **减少了 kernel launch 次数**。jasl 的 Triton kernel 把多个操作融合在一起，一个 token 的生成不需要 120 次独立 launch
2. **消除了 CPU-GPU 同步**。我们 Consumer-DeepGEMM 的瓶颈不在 GPU 计算，而在每次 launch 之间的 CPU 等待。0.8 tok/s 里，GPU 实际计算时间可能只占 10%，剩下 90% 是 CPU 在等 GPU 返回结果
3. **更好的流水线重叠**。Triton 编译器自动做了 kernel 之间的流水线调度，数据搬运和计算可以重叠执行

176 期最后说"瓶颈是 m_indices 的 CPU 同步不是 reorder 本身"——jasl 的方案从根本上绕过了这个问题，因为他重写的 Triton kernel 里没有这种 CPU 同步点。

打个比方：Consumer-DeepGEMM 是请了 120 个翻译，每翻译完一段都要回报一次进度（CPU 同步），虽然每个翻译都很准确（CUTLASS kernel 正确），但回报进度的时间比翻译本身还长。jasl 的方案是请了一个能同时处理多段的翻译（融合 Triton kernel），翻完整篇再汇报一次——总工作量差不多，但等待时间大幅减少。

◆ 为什么英文还有问题

━━━━━━━━━━━━━━━━━━━━

这不是 jasl 的 bug，是底层同一个根因的残留。

前面说过：Marlin 的 FP4 MMA 指令在 sm_120+ 上不是"全错"——是**特定的专家组合算错**。280B MoE 模型有 256 个专家，每次推理只激活其中 6 个（DeepSeek V4 Flash 的 top-6 路由策略）。哪 6 个被激活取决于输入 token 的语义嵌入向量经过路由网络后的 softmax 排名。

关键在于：jasl 用 Triton 重写了**大部分**计算路径，但不一定覆盖了 vLLM 里 DeepSeek V4 推理链路上的**每一个**操作。vLLM 的推理管线很长——tokenizer → embedding → 61 层 transformer（每层包含 attention + MoE FFN）→ output head。jasl 重写的是最核心的几个 kernel，但可能有少量边缘路径（比如某些特殊的 epilogue 操作、scale 计算的 corner case）仍然 fallback 到 vLLM 内置的实现——其中某些可能间接调用了有问题的 Marlin 代码路径。

这些残余路径在中文 prompt 下不被触发（因为激活的专家组合避开了出错的那些），在英文 prompt 下被触发时，垃圾 token 就出来了。

这也解释了为什么 Consumer-DeepGEMM 验证时中英文输出都正确——我们替换的是底层 GEMM 计算本身，不管哪个专家被激活都走 CUTLASS 路径，所以所有专家的计算结果都是正确的。jasl 替换的是更上层的 kernel 调度，大部分路径走了 Triton，但可能有少量没覆盖到。

根本解决办法有两条：
1. 把所有残余的 Marlin 路径都替换掉（jasl 还在做）
2. 等 vLLM 官方修复 Marlin 对 sm_120+ 的兼容性（issue #40928）

◆ 我们还试了什么（走不通的路）

━━━━━━━━━━━━━━━━━━━━

在找到 jasl fork 之前，我们还试了另一个方案：vLLM PR #40082（分支 `meena-at-work/vllm` 的 `integrate-flashinfer-b12x-moe`），基于 FlashInfer 的 CuTeDSL backend。eugr/spark-vllm-docker 也为这个 PR 提供了运行时补丁（`mods/exp-b12x`）。

这个方案来自 vLLM 的 PR #40082。PR 标题看着很对路——"Add FlashInfer B12x backend for sm_120+ support"。但实测：对 DeepSeek V4 没用。

原因在 vLLM 的 backend 选择逻辑里。vLLM 根据模型的量化格式选择不同的计算后端：

```text
模型量化格式                        →  vLLM oracle    →  选择的 backend
────────────────────────────────────────────────────────────────────
NVFP4（如 nvidia/DeepSeek-R1-NVFP4）→  nvfp4_oracle  →  flashinfer_b12x (PR #40082) [正常]
MXFP4（如 DeepSeek V4 Flash 原版）   →  mxfp4_oracle  →  marlin [算错]（sm_120+）
```

PR #40082 引入的 `flashinfer_b12x` backend 只注册在 **NVFP4** oracle 中。而 DeepSeek V4 Flash 用的量化格式是 **MXFP4**（MX-compliant FP4，微软/ARM/高通/NVIDIA 联合定义的开放标准），不是 NVFP4（NVIDIA 专有的 FP4 格式）。

这里要区分"原生格式"和"后量化格式"。DeepSeek V4 Flash 发布时权重就是 MXFP4——这是 DeepSeek 训练时选择的原生格式。而 NVFP4 是 NVIDIA 拿别人的模型用 ModelOpt 工具做的后量化产物，发布在 HuggingFace 的 `nvidia/` 命名空间下（比如 `nvidia/DeepSeek-R1-NVFP4`、`nvidia/Qwen3.5-397B-A17B-NVFP4`）。两者的区别：

- **NVFP4**：NVIDIA 专有格式，每 16 个元素共享一个 E4M3 scale factor + 全局 FP32 scale（两级缩放），精度更高但只有 NVIDIA 后量化工具能生成
- **MXFP4**：OCP 开放标准格式，每 32 个元素共享一个 E8M0 scale factor（单级缩放，power-of-two），DeepSeek V4、GPT-OSS 等模型原生使用

同样是 4-bit 浮点数，但 scale factor 的位宽、存储方式、和 GPU 硬件指令的对接方式完全不同。在 vLLM 代码里，这两种格式走的是完全不同的管线——从模型加载时的 weight loader、到运行时的 oracle 选择、到最终的 kernel dispatch，代码路径完全分开，互不影响。你给 NVFP4 管线加了 sm_120 支持，MXFP4 管线不会得到任何好处。

所以 PR #40082 的 `flashinfer_b12x` backend 是给 NVFP4 模型做的 sm_120 适配，对 DeepSeek V4 的 MXFP4 路径没有任何影响。我们花了半天编译构建，启动后发现根本没走到那个 backend——白忙一场。

这个坑的教训：**不能只看 PR 标题里的 "sm_120 support"，要看它支持的是哪种量化格式**。vLLM 里 FP4 有两条完全独立的管线（NVFP4 和 MXFP4），给一条管线加 sm_120 支持，另一条纹丝不动。DeepSeek V4 偏偏用了 MXFP4——目前社区对这条管线的 sm_120 适配，只有 jasl 一个人在做。

```text
## What is Quicksort?

Quicksort is a **divide-and-conquer** sorting algorithm that works by:
1. Selecting a "pivot" element from the array
2. Partitioning the array so elements smaller than pivot go left, larger go right
```

说实话，需要英文的话还不如直接用一个正常的小模型。比如最近发布的 Qwen 3.6-27B——27B 稠密模型在 SWE-bench Verified 上打了 77.2%，超过了 15 倍参数的 Qwen 3.5-397B MoE，被社区称为"消费级 GPU 上能跑的最强模型"。单台 DGX Spark 128GB 跑 BF16 全精度只要 ~56GB，剩 72GB 给 KV cache，零 kernel 兼容性问题，vLLM 直接支持。费这么大劲跑一个英文半残的 280B，不如老老实实用一个全功能的小模型。

◆ 怎么复现

━━━━━━━━━━━━━━━━━━━━

硬件要求：两台 DGX Spark（Grace Blackwell，ARM64 CPU + 128GB GPU 显存，每台），CX7 200Gbps NVLink 直连。

为什么需要两台？DeepSeek V4 Flash 280B 的 FP4 权重文件约 148GB。单台 Spark 128GB 显存放不下——还没算 KV cache 和运行时 buffer 的开销。双机 tensor parallel 把模型参数对半切，每台 GPU 加载约 74GB 权重，剩下 54GB 给 KV cache 和临时计算空间。

构建步骤（详见各 GitHub 仓库的 README）：

```bash
# 1. 获取构建工具
git clone https://github.com/eugr/spark-vllm-docker.git
cd spark-vllm-docker

# 2. 获取 jasl 的 vLLM sm_120+ 适配源码
git clone -b ds4-sm120 https://github.com/jasl/vllm.git vllm-sm120

# 3. 修改 Dockerfile 中的 vLLM 源码地址（一行改动）
# 将第 190 行: git clone --recursive https://github.com/vllm-project/vllm.git
# 改为:        git clone --recursive https://github.com/jasl/vllm.git

# 4. 构建镜像并同步到 slave（约 30 分钟编译 + 1 分钟同步）
./build-and-copy.sh -t vllm-node-jasl -c <slave_ip> --vllm-ref ds4-sm120

# 5. 启动双机推理服务
HF_HOME=/path/to/your/model/parent \
./launch-cluster.sh -n <host_ip>,<slave_ip> -t vllm-node-jasl exec \
  vllm serve /root/.cache/huggingface/deepseek-v4-flash \
  --tensor-parallel-size 2 \
  --distributed-executor-backend ray \
  --gpu-memory-utilization 0.85 \
  --kv-cache-dtype fp8 \
  --max-model-len 131072 \
  --enforce-eager
```

推荐使用 jasl/vllm 的 `ds4-sm120` 分支——这是目前社区验证过的、在 DGX Spark 上跑 DeepSeek V4 Flash 的最佳方案。eugr/spark-vllm-docker 负责处理 ARM64 编译链路和双机 Ray 集群管理，jasl 的分支负责 sm_120+ 的 kernel 适配。两个项目配合使用。

关键参数解释：

- `--enforce-eager`：禁用 CUDA graph capture。sm_120+ 上 graph capture 有兼容性问题（某些 Triton kernel 不支持被 graph 捕获），用 eager mode 逐个 dispatch kernel
- `--kv-cache-dtype fp8`：KV cache 用 FP8 存储而不是默认的 FP16。128K context 的 KV cache 很大，FP8 省一半显存
- `--gpu-memory-utilization 0.85`：预留 15% 显存给临时 buffer 和碎片。两台 128GB = 256GB 总显存，模型参数占 148GB，KV cache + workspace 用剩下的。如果 OOM 降到 0.80
- `--max-model-len 131072`：DeepSeek V4 Flash 支持的最大上下文长度 128K

几个实际使用中的注意事项：

- 首次请求会很慢（我们测到 16.8s 才出 10 个 token）——这是 Triton kernel 的 JIT 编译时间。后续请求不受影响，因为编译结果会被缓存
- 如果遇到 OOM（Out of Memory），先降 `--gpu-memory-utilization` 到 0.80，或者降 `--max-model-len` 到 65536（64K context 对大多数场景够用）
- 双机 tensor parallel 需要两台 Spark 的 NVLink 直连正常工作。用 `nvidia-smi nvlink --status` 检查链路状态
- 日志里看到 `Using Triton kernel for ...` 的字样说明走对了路径；如果看到 `Using Marlin kernel` 则说明某些操作还在走有问题的路径
- 验证方法：先用中文问一个需要长回复的问题（如"详细介绍量子力学的基本原理"），确认输出流畅且 `finish_reason=stop`；再用英文问同样的问题，观察是否有垃圾 token。中文正确 + 英文有问题 = 和我们的结果一致，部署成功
- 14 tok/s 是生成阶段（decode）的速度。首 token 延迟（TTFT, Time To First Token）在长 prompt 下会更高——128K context 的 prefill 可能需要十几秒。短 prompt（几十个 token）的 TTFT 通常在 1-2 秒

━━━━━━━━━━━━━━━━━━━━

◆ 下一步

━━━━━━━━━━━━━━━━━━━━

几件值得关注的事：

1. **jasl PR #40991 合入 vLLM 主线**——目前 draft 状态。合入后所有 DGX Spark 用户直接用官方 vLLM 就能跑 DeepSeek V4，不需要自己编译 fork
2. **vLLM 官方修复 Marlin sm_120+ 静默错误**——issue #40928 挂着，排期不明。这是根本解决方案，但需要 NVIDIA 或 vLLM 核心开发者介入，理解 sm_120+ tensor core 指令的具体行为差异
3. **DeepGEMM 支持 sm_120+**——DeepSeek 在 issue #236 中明确表示没有计划。这是合理的——DeepGEMM 是给自家数据中心写的，消费级卡不在他们的业务范围内
4. **英文垃圾 token 问题修复**——需要社区继续排查 jasl fork 中残余的 Marlin fallback 路径，逐一替换成 Triton 实现

现阶段结论：**中文日常使用已经完全够了，英文需要等社区继续修复**。14 tok/s 的速度对问答、写作辅助、代码生成（中文 prompt）来说足够流畅——和 ChatGPT 网页版的体感差不多。等英文问题修好，这两台 DGX Spark 就是一台完整的 280B 大模型本地推理服务器，不依赖任何云服务。

从 171 期第一次在 DGX Spark 上跑通 DeepSeek V4 Flash（短回复正常，没发现问题），到发现长输出垃圾、定位 Marlin 根因，到 176 期用 CUTLASS 修正输出（正确但 0.8 tok/s），到这一期用 jasl fork 达到 14 tok/s——整个过程大约三周。这三周里社区的进展速度远超 vLLM 官方。开源的力量就在这里：官方排期遥遥无期的时候，总有人自己动手。

回看这段经历，有几个值得记住的点：

第一，**静默错误比 crash 可怕得多**。如果 Marlin 在 sm_120+ 上直接报错，我们第一天就知道该换方案。恰恰因为它"看起来正常"（中文短回复没问题），我们花了好几天才定位根因。测试大模型推理服务时，一定要用多种语言、多种长度的 prompt 做全面验证——别只问"1+1 等于几"就觉得部署成功了。

第二，**选对抽象层比写极致代码更重要**。我们选了 CUTLASS（第 2 层），写出了正确的结果，但被 launch 开销卡住了。jasl 选了 Triton（第 4 层），抽象层更高，单次 kernel 可能没有 CUTLASS 快，但编译器帮他做了流水线优化和 launch 融合——最终端到端速度快 18 倍。不是"越底层越好"——是"选对层次最好"。

第三，**176 期的五层塔不是白讲的**。理解了五层塔，才能理解为什么 jasl 选 Triton、为什么 exp-b12x 对 DeepSeek V4 无效、为什么 Consumer-DeepGEMM 正确但慢。没有这个框架，你面对的就是一堆互相矛盾的社区 issue 和 PR，不知道该试哪个。
