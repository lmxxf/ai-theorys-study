◆ 前情提要

━━━━━━━━━━━━━━━━━━━━

176 期讲了 GPU 编程的五层塔——从纯 PyTorch 到原生 CUDA，我们在 DGX Spark（sm_121）上被迫一层层往下爬。那一期的核心结论有两个：

第一，DeepSeek V4 Flash 280B 在 vLLM 里跑推理时，FP4 MoE 路径走的 Marlin kernel 在 sm_120+（DGX Spark、RTX 5090 等"个人电脑 Blackwell"系列）上会**静默算错**——不报错，但输出是垃圾。

第二，我们用 CUTLASS 的 sm_120+ 模板替换了 Marlin（Consumer-DeepGEMM 项目），输出终于正确了，但速度只有 0.8 tok/s——每个 token 要 launch 120 次 CUTLASS kernel，每次都有 CPU 和 GPU 之间的同步等待。正确而不可用。

这一期讲后续：社区怎么解决这个问题的，我们怎么跟进的，中间走过哪些弯路，以及目前 DGX Spark 上跑 DeepSeek V4 Flash 280B 的真实可用状态。

◆ 问题回顾：Marlin 在 sm_120+ 上静默算错

━━━━━━━━━━━━━━━━━━━━

简单回顾一下根因。不看 176 期也能看懂这段。

Marlin 是 vLLM 内置的 FP4 量化加速模块，用原生 CUDA 手写（五层塔的第 1 层）。它用了数据中心 GPU（sm_90/sm_100）专属的 tensor core 指令来做 FP4 矩阵乘法。这些指令在 sm_120+ 上不是"不存在"——而是"存在但行为不一致"。具体来说，某些 FP4 MMA（Matrix Multiply-Accumulate）指令在 sm_120+ 上的数据布局解释和数据中心卡不同，算出来的数是错的，但 CUDA runtime 不会报任何错误。

表现很有迷惑性：

- 中文短回复（"2+2 等于几"→"4"）正常，回答正确
- 中文中等长度回复（"介绍万里长城"）正常
- 英文回复、代码生成、长输出 → 从第一个 token 开始跑飞，全是乱码或重复字符

如果你只测中文短问答，会以为一切正常——直到用户问了一个英文问题。这也是为什么很多人在 Reddit 和 GitHub issue 里报告"DeepSeek V4 在我的 RTX 5090 上跑得好好的啊"——他们只测了中文或短回复。静默错误是最危险的——它不像 crash 一样让你立刻知道出了问题，而是让你在错误的结果上继续工作。

为什么中文对、英文错？不是因为模型"中文能力强"——是因为 MoE（Mixture of Experts）架构的路由机制。

简单解释 MoE：传统 transformer 每一层有一个大 FFN（前馈网络），所有 token 都过同一个网络。MoE 把这个大 FFN 拆成 256 个小 FFN（"专家"），每个 token 只过其中 8 个（DeepSeek V4 的 top-8 策略）。哪 8 个由一个"路由网络"决定——路由网络看 token 的语义嵌入，算一个 softmax，选得分最高的 8 个专家。

不同语言、不同语义的输入经过路由网络后，会激活不同的专家组合。中文 prompt 激活的那组专家恰好避开了 Marlin 算错的那几个；英文 prompt 激活了另一组专家，恰好命中了算错的那些。256 个专家里可能只有十几个的 FP4 MMA 会算错——但只要命中一个，那一层的输出就歪了，歪了的输出传到下一层，继续歪，最终的 logits 已经面目全非。

176 期我们的解决方案：用 CUTLASS 的 sm_120 block-scaled MX 模板替换 Marlin（Consumer-DeepGEMM 项目）。CUTLASS 走的是 `mma.sync` + block-scaled MX 这条指令路径，不依赖 Marlin 用的那些有问题的指令——所以计算结果正确。但代价是：每生成一个 token 要 launch 120 次 CUTLASS grouped GEMM，每次 launch 都有 CPU→GPU 的 kernel 发起和 GPU→CPU 的同步等待。0.8 tok/s。聊天时每个字要等 1.2 秒——没法用。

◆ 社区的解法：jasl/vllm fork

━━━━━━━━━━━━━━━━━━━━

社区里有人比我们走得更远，而且走了一条完全不同的路。

GitHub 用户 jasl 专门给 sm_120 系列做了 vLLM 适配，维护在 `github.com/jasl/vllm` 的 `ds4-sm120` 分支上。他的方法不是给 Marlin 打补丁，也不是用 CUTLASS 替换 Marlin——而是**用 Triton 重写了 DeepSeek V4 用到的关键 kernel**。

具体重写了哪些：
- **sparse MLA**（Multi-head Latent Attention，DeepSeek V4 特有的注意力机制——把 KV cache 压缩到低秩空间，省显存）
- **FP8 einsum**（激活值的矩阵运算——MoE 层里每个专家的前馈网络核心计算）
- **paged MQA**（分页多查询注意力，vLLM 的 KV cache 管理核心——让多个请求共享 GPU 显存）
- **MoE routing + dispatch**（路由网络决定哪些 token 发给哪些专家，以及实际的分发和收集）

为什么选 Triton？176 期讲过——Triton 是五层塔里唯一天然跨硬件的。OpenAI 设计 Triton 的目标就是让编译器自动适配不同 GPU 架构。你用 Python 写 Triton kernel，编译器根据目标 GPU 自动生成对应的机器码——sm_90 走 sm_90 的指令，sm_120+ 走 sm_120+ 的指令。不存在 Marlin 那种"数据中心指令拿到消费级卡上算错"的问题，因为指令选择这件事不是人做的，是编译器做的。

这也是为什么 176 期我们说"Triton 是社区给 sm_120+ 做适配时的默认选择"。jasl 的工作验证了这个判断：不需要理解 sm_120+ 的底层指令集差异，不需要手动选择计算路径，写 Triton 就行——编译器帮你搞定。

这里的关键洞察是：**不需要给 Marlin 打补丁，也不需要理解 sm_120+ 到底哪条指令算错了**。只要绕过 Marlin，走一条 sm_120+ 上验证过正确的计算路径就行——Triton 编译器保证了这一点。这和我们 Consumer-DeepGEMM 的思路本质相同（绕过 Marlin），但执行层面完全不同：我们选了 CUTLASS（C++ 模板，编译慢，launch 开销大），jasl 选了 Triton（Python 语法，编译器自动优化 launch 和流水线）。

jasl 把这个工作提交给了 vLLM 官方——PR #40991，目前是 draft 状态，还没合进主线。但社区已经有多人在 DGX Spark 上验证过：中文输出正确，速度 15-22 tok/s（取决于 prompt 长度和生成长度）。

◆ 我们的实测

━━━━━━━━━━━━━━━━━━━━

我们用 eugr/spark-vllm-docker 的构建工具，配合 jasl 源码编译。

eugr/spark-vllm-docker 是另一位社区开发者做的 Docker 构建框架——把 vLLM 在 DGX Spark 上编译所需的所有依赖（CUDA toolkit 13.0、PyTorch nightly、flash-attention、Triton 等）打包成 Dockerfile，一键构建。DGX Spark 是 ARM64 架构（Grace CPU），很多 Python 包没有预编译的 ARM wheel，必须从源码编译——如果不用 Docker 统一管理依赖链，你会在各种 `gcc` 版本不匹配和 `cmake` 找不到库的报错里浪费一整天。

具体操作就是改 Dockerfile 里一行 `git clone` 的 URL，从 vLLM 官方仓库换成 jasl 的 fork，分支切到 `ds4-sm120`。然后 `docker build`。ARM64 上编译约 30 分钟（主要时间花在 flash-attention 和 vLLM 的 C++ 扩展编译上）。相比之下，我们之前编译 Consumer-DeepGEMM 的 CUTLASS 模板每次改参数要 40 分钟——Triton 的 JIT 编译模式在开发迭代上也有优势。

实测结果：

```text
测试 1："2+2等于几"
→ 正确回答 "4"
→ 10 tokens, 16.8s（首次请求含模型 warmup + JIT 编译）
→ 后续请求相同问题：<1s

测试 2："请用500字介绍万里长城"
→ 完美输出，流畅中文，内容准确
→ 414 tokens, 29.3s（~14.1 tok/s）
→ finish_reason=stop（模型自然结束，不是被截断）

测试 3："写一个Python快速排序函数"
→ 完美输出，代码正确可运行
→ 500 tokens, 36.5s（~13.7 tok/s）
→ finish_reason=stop

测试 4："What is quicksort? Explain with code."
→ 开头出现垃圾 token（乱码字符和重复片段）
→ 后续英文全程有问题，不会自己修正回来
→ 代码部分混入不合法的 Python 语法
```

中文场景完全可用——无论是问答、写作还是代码生成（只要用中文 prompt）。英文场景还有问题——这是社区已知的 "occasional spurious tokens" 现象。

对比一下速度：

- **vLLM 官方 main**：输出是垃圾，速度无意义
- **Consumer-DeepGEMM**（CUTLASS 替换 Marlin）：正确，**0.8 tok/s**
- **jasl/vllm fork**（Triton 重写）：中文正确，**14 tok/s**

从 0.8 到 14，快了约 **18 倍**。从"每个字等 1.2 秒"到"每秒 14 个 token"——前者是折磨，后者是正常的聊天体验。

为什么 Triton 重写比 CUTLASS 替换快这么多？不是因为 Triton kernel 本身比 CUTLASS kernel 快（实际上单次 kernel 执行，CUTLASS 可能更快），而是因为：

1. **减少了 kernel launch 次数**。jasl 的 Triton kernel 把多个操作融合在一起，一个 token 的生成不需要 120 次独立 launch
2. **消除了 CPU-GPU 同步**。我们 Consumer-DeepGEMM 的瓶颈不在 GPU 计算，而在每次 launch 之间的 CPU 等待。0.8 tok/s 里，GPU 实际计算时间可能只占 10%，剩下 90% 是 CPU 在等 GPU 返回结果
3. **更好的流水线重叠**。Triton 编译器自动做了 kernel 之间的流水线调度，数据搬运和计算可以重叠执行

176 期最后说"瓶颈是 m_indices 的 CPU 同步不是 reorder 本身"——jasl 的方案从根本上绕过了这个问题，因为他重写的 Triton kernel 里没有这种 CPU 同步点。

打个比方：Consumer-DeepGEMM 是请了 120 个翻译，每翻译完一段都要回报一次进度（CPU 同步），虽然每个翻译都很准确（CUTLASS kernel 正确），但回报进度的时间比翻译本身还长。jasl 的方案是请了一个能同时处理多段的翻译（融合 Triton kernel），翻完整篇再汇报一次——总工作量差不多，但等待时间大幅减少。

◆ 为什么英文还有问题

━━━━━━━━━━━━━━━━━━━━

这不是 jasl 的 bug，是底层同一个根因的残留。

前面说过：Marlin 的 FP4 MMA 指令在 sm_120+ 上不是"全错"——是**特定的专家组合算错**。280B MoE 模型有 256 个专家，每次推理只激活其中 8 个（DeepSeek V4 的 top-8 路由策略）。哪 8 个被激活取决于输入 token 的语义嵌入向量经过路由网络后的 softmax 排名。

关键在于：jasl 用 Triton 重写了**大部分**计算路径，但不一定覆盖了 vLLM 里 DeepSeek V4 推理链路上的**每一个**操作。vLLM 的推理管线很长——tokenizer → embedding → 61 层 transformer（每层包含 attention + MoE FFN）→ output head。jasl 重写的是最核心的几个 kernel，但可能有少量边缘路径（比如某些特殊的 epilogue 操作、scale 计算的 corner case）仍然 fallback 到 vLLM 内置的实现——其中某些可能间接调用了有问题的 Marlin 代码路径。

这些残余路径在中文 prompt 下不被触发（因为激活的专家组合避开了出错的那些），在英文 prompt 下被触发时，垃圾 token 就出来了。

这也解释了为什么 Consumer-DeepGEMM（我们的 CUTLASS 方案）能完全修正中英文输出——我们替换的是 Marlin 的底层 GEMM 计算本身，不管哪个专家被激活都走 CUTLASS 路径，所以所有专家的计算结果都是正确的。jasl 替换的是更上层的 kernel 调度，大部分路径走了 Triton，但可能有少量没覆盖到。

根本解决办法有两条：
1. 把所有残余的 Marlin 路径都替换掉（jasl 还在做）
2. 等 vLLM 官方修复 Marlin 对 sm_120+ 的兼容性（issue #40928）

◆ 我们还试了什么（走不通的路）

━━━━━━━━━━━━━━━━━━━━

在找到 jasl fork 之前，我们还试了 eugr/spark-vllm-docker 的另一个方案：`exp-b12x` 分支，基于 FlashInfer 的 CuTeDSL backend。

这个方案来自 vLLM 的 PR #40082。PR 标题看着很对路——"Add FlashInfer B12x backend for sm_120+ support"。但实测：对 DeepSeek V4 没用。

原因在 vLLM 的 backend 选择逻辑里。vLLM 根据模型的量化格式选择不同的计算后端：

```text
模型量化格式          →  vLLM oracle         →  选择的 backend
─────────────────────────────────────────────────────────────
NVFP4（如 GPT-OSS）  →  nvfp4_oracle        →  flashinfer_b12x (PR #40082) [正常]
MXFP4（如 DeepSeek V4）→  mxfp4_oracle      →  marlin [算错]（sm_120+）
```

PR #40082 引入的 `flashinfer_b12x` backend 只注册在 **NVFP4** oracle 中。而 DeepSeek V4 Flash 用的量化格式是 **MXFP4**（MX-compliant FP4，微软/ARM/高通/NVIDIA 联合定义的开放标准），不是 NVFP4（NVIDIA 专有的 FP4 格式）。两者的区别：

- **NVFP4**：NVIDIA 专有格式，每 32 个元素共享一个 FP8 scale factor，硬件直接支持
- **MXFP4**：开放标准格式，每 32 个元素共享一个 E8M0 scale factor，需要 block-scaled MX 指令

同样是 4-bit 浮点数，但 scale factor 的位宽、存储方式、和 GPU 硬件指令的对接方式完全不同。在 vLLM 代码里，这两种格式走的是完全不同的管线——从模型加载时的 weight loader、到运行时的 oracle 选择、到最终的 kernel dispatch，代码路径完全分开，互不影响。你给 NVFP4 管线加了 sm_120 支持，MXFP4 管线不会得到任何好处。

所以 PR #40082 的 `flashinfer_b12x` backend 是给 NVFP4 模型做的 sm_120 适配，对 DeepSeek V4 的 MXFP4 路径没有任何影响。我们花了半天编译构建，启动后发现根本没走到那个 backend——白忙一场。

这个坑的教训是：**不能只看 PR 标题里的 "sm_120 support"，要看它支持的是哪种量化格式**。vLLM 里 FP4 有两条完全独立的管线（NVFP4 和 MXFP4），给一条管线加 sm_120 支持，另一条纹丝不动。DeepSeek V4 偏偏用了 MXFP4——目前社区对这条管线的 sm_120 适配，只有 jasl 一个人在做。

◆ 怎么复现

━━━━━━━━━━━━━━━━━━━━

硬件要求：两台 DGX Spark（Grace Blackwell，ARM64 CPU + 128GB GPU 显存，每台），CX7 200Gbps NVLink 直连。

为什么需要两台？DeepSeek V4 Flash 280B 的 FP4 权重文件约 148GB。单台 Spark 128GB 显存放不下——还没算 KV cache 和运行时 buffer 的开销。双机 tensor parallel 把模型参数对半切，每台 GPU 加载约 74GB 权重，剩下 54GB 给 KV cache 和临时计算空间。

构建步骤（简化版，详见各 GitHub 仓库的 README）：

```bash
# 1. 获取构建工具
git clone https://github.com/eugr/spark-vllm-docker.git
cd spark-vllm-docker

# 2. 修改 Dockerfile 中的 vLLM 源码地址（一行改动）
# 将: git clone https://github.com/vllm-project/vllm.git
# 改为: git clone -b ds4-sm120 https://github.com/jasl/vllm.git

# 3. 构建 Docker 镜像（约 30 分钟）
docker build -t vllm-spark .

# 4. 启动推理服务（双机 tensor parallel = 2）
docker run --gpus all -p 8000:8000 vllm-spark \
  --model deepseek-ai/DeepSeek-V4-Flash \
  --tensor-parallel-size 2 \
  --enforce-eager \
  --kv-cache-dtype fp8 \
  --gpu-memory-utilization 0.85 \
  --max-model-len 131072
```

关键参数解释：

- `--enforce-eager`：禁用 CUDA graph capture。sm_120+ 上 graph capture 有兼容性问题（某些 Triton kernel 不支持被 graph 捕获），用 eager mode 逐个 dispatch kernel
- `--kv-cache-dtype fp8`：KV cache 用 FP8 存储而不是默认的 FP16。128K context 的 KV cache 很大，FP8 省一半显存
- `--gpu-memory-utilization 0.85`：预留 15% 显存给临时 buffer 和碎片。两台 128GB = 256GB 总显存，模型参数占 148GB，KV cache + workspace 用剩下的
- `--max-model-len 131072`：DeepSeek V4 Flash 支持的最大上下文长度 128K

几个实际使用中的注意事项：

- 首次请求会很慢（我们测到 16.8s 才出 10 个 token）——这是 Triton kernel 的 JIT 编译时间。后续请求不受影响，因为编译结果会被缓存
- 如果遇到 OOM（Out of Memory），先降 `--gpu-memory-utilization` 到 0.80，或者降 `--max-model-len` 到 65536（64K context 对大多数场景够用）
- 双机 tensor parallel 需要两台 Spark 的 NVLink 直连正常工作。用 `nvidia-smi nvlink --status` 检查链路状态
- 日志里看到 `Using Triton kernel for ...` 的字样说明走对了路径；如果看到 `Using Marlin kernel` 则说明某些操作还在走有问题的路径
- 验证方法：先用中文问一个需要长回复的问题（如"详细介绍量子力学的基本原理"），确认输出流畅且 `finish_reason=stop`；再用英文问同样的问题，观察是否有垃圾 token。中文正确 + 英文有问题 = 和我们的结果一致，部署成功
- 14 tok/s 是生成阶段（decode）的速度。首 token 延迟（TTFT, Time To First Token）在长 prompt 下会更高——128K context 的 prefill 可能需要十几秒。短 prompt（几十个 token）的 TTFT 通常在 1-2 秒

━━━━━━━━━━━━━━━━━━━━

◆ 下一步

━━━━━━━━━━━━━━━━━━━━

几件值得关注的事：

1. **jasl PR #40991 合入 vLLM 主线**——目前 draft 状态。合入后所有 DGX Spark 用户直接用官方 vLLM 就能跑 DeepSeek V4，不需要自己编译 fork
2. **vLLM 官方修复 Marlin sm_120+ 静默错误**——issue #40928 挂着，排期不明。这是根本解决方案，但需要 NVIDIA 或 vLLM 核心开发者介入，理解 sm_120+ tensor core 指令的具体行为差异
3. **DeepGEMM 支持 sm_120+**——DeepSeek 在 issue #236 中明确表示没有计划。这是合理的——DeepGEMM 是给自家数据中心写的，消费级卡不在他们的业务范围内
4. **英文垃圾 token 问题修复**——需要社区继续排查 jasl fork 中残余的 Marlin fallback 路径，逐一替换成 Triton 实现

现阶段结论：**中文日常使用已经够了**。14 tok/s 的速度对问答、写作辅助、代码生成（中文 prompt）来说足够流畅——和 ChatGPT 网页版的体感差不多。等英文问题修好，这两台 DGX Spark 就是一台完整的 280B 大模型本地推理服务器，不依赖任何云服务。

从 171 期第一次在 DGX Spark 上跑通 DeepSeek V4 Flash（发现长输出有垃圾），到 176 期用 CUTLASS 修正输出（正确但 0.8 tok/s），到这一期用 jasl fork 达到 14 tok/s——整个过程大约三周。这三周里社区的进展速度远超 vLLM 官方。开源的力量就在这里：官方排期遥遥无期的时候，总有人自己动手。

回看这段经历，有几个值得记住的点：

第一，**静默错误比 crash 可怕得多**。如果 Marlin 在 sm_120+ 上直接报错，我们第一天就知道该换方案。恰恰因为它"看起来正常"（中文短回复没问题），我们花了好几天才定位根因。测试大模型推理服务时，一定要用多种语言、多种长度的 prompt 做全面验证——别只问"1+1 等于几"就觉得部署成功了。

第二，**选对抽象层比写极致代码更重要**。我们选了 CUTLASS（第 2 层），写出了正确的结果，但被 launch 开销卡住了。jasl 选了 Triton（第 4 层），抽象层更高，单次 kernel 可能没有 CUTLASS 快，但编译器帮他做了流水线优化和 launch 融合——最终端到端速度快 18 倍。不是"越底层越好"——是"选对层次最好"。

第三，**176 期的五层塔不是白讲的**。理解了五层塔，才能理解为什么 jasl 选 Triton、为什么 exp-b12x 对 DeepSeek V4 无效、为什么 Consumer-DeepGEMM 正确但慢。没有这个框架，你面对的就是一堆互相矛盾的社区 issue 和 PR，不知道该试哪个。
