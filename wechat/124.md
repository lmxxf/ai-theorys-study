◆ 第一段：P7885 新设备，老问题换了个马甲

━━━━━━━━━━━━━━━━━━━━

换了一台 P7885 设备。硬件参数：

| 项目 | RK3568（旧） | P7885（新） |
|:---|:---|:---|
| 芯片 | Cortex-A55 | Cortex-A55 |
| 内核 | Linux 6.6.101 | Linux 5.10.184 |
| musl 版本 | 未知 | 1.2.5 |
| 存储 | 14G 可用 | 228G 可用 |

同一套 ARM 指令集，同一个 OpenHarmony 6.0，理论上部署包直接搬过来就能跑。

把上次在 RK3568 上验证通过的部署包推上去，跑验证脚本——

```
import ssl  →  Signal 11 (SIGSEGV)
import hashlib  →  Signal 11 (SIGSEGV)
```

**又 segfault 了。**

────────────────────

💡 Signal 11（SIGSEGV）就是段错误——程序访问了不该访问的内存地址，操作系统直接把它杀掉。没有错误信息，没有 traceback，只有"死亡"。

────────────────────

【老朋友回来了】

上一篇花了大半天才找到的根因：**OHOS 的 musl dlopen 不继承调用者进程的符号表。** 每个 `.so` 动态库必须自己声明"我需要 libpython"。

当时的修法是在 CPython Makefile 的 `BLDSHARED` 变量里加 `-lpython3.11`。这个变量控制的是通过 Makefile 编译 `.so` 扩展模块时的链接命令。改完之后，RK3568 上全部通过。

**但有 6 个模块漏了。**

用 `readelf`（一个查看二进制文件信息的工具）检查：

```bash
readelf -d _sha256.cpython-311-aarch64-linux-ohos.so | grep NEEDED
# 结果：没有 libpython3.11.so
```

这 6 个模块是：`_sha1`、`_sha256`、`_sha3`、`_sha512`、`_hashlib`、`_ssl`。

它们通过 CPython 的 `Setup.stdlib` 配置文件编译——走的是一条和 `setup.py` 不同的编译路径。上次修 `BLDSHARED` 的时候，只覆盖了 `setup.py` 那条路；**`Setup.stdlib` 那条路完全没碰到。**

────────────────────

💡 **Setup.stdlib** 是 CPython 3.11 新增的机制——一些核心模块（hash、ssl 等）从 `setup.py` 的动态检测模式改成了 `Setup.stdlib` 的静态声明模式。编译时走 Makefile 的不同规则。这意味着你改了 `setup.py` 那边的链接参数，这边不会自动跟。

────────────────────

**那为什么在 RK3568 上没崩？**

RK3568 内核版本 6.6，P7885 是 5.10。两台设备的 musl 版本不同。**RK3568 上的 musl 在 dlopen 时对符号继承更宽松**——虽然 `.so` 没显式声明依赖 libpython，但进程里已经加载了 libpython，旧版 musl 碰巧能找到符号。P7885 的 musl 1.2.5 更严格，不继承就是不继承。

**同一个根因，不同的 musl 版本表现不同。** 这是跨平台移植最阴损的坑——你在设备 A 上"验证通过"了，搬到设备 B 就炸。因为"通过"不代表"正确"，有时候只是"碰巧没出事"。

────────────────────

【patchelf 修复——不用重新编译】

上次碰到 dlopen 符号问题，修法是改 Makefile 重新编译。这次有个更省事的工具——**patchelf**。

💡 **patchelf** 是一个修改 ELF 二进制文件（Linux 上的可执行文件和 `.so` 格式）的工具。它能直接修改已编译好的 `.so`，往里面补一条"我需要 libpython3.11.so"的声明。不用重新编译。

```bash
for f in _hashlib _sha1 _sha256 _sha3 _sha512 _ssl; do
    patchelf --add-needed libpython3.11.so \
        "lib-dynload/${f}.cpython-311-aarch64-linux-ohos.so"
done
```

6 个模块，一个 for 循环。

验证一下：

```bash
readelf -d _sha256.cpython-311-aarch64-linux-ohos.so | grep libpython
# → Shared library: [libpython3.11.so]  ✅
```

推上 P7885，跑验证——

```
Python: 3.11.11
ssl: OpenSSL 3.3.2 3 Sep 2024
hashlib: OK
pydantic_core: 2.41.5
langchain_core: OK

=== All imports OK ===
```

**全通了。**

────────────────────

【一键部署脚本】

每次手动 patchelf + 打包 + hdc 推送 + 解压 + 校时 + 验证，步骤太多容易漏。做了一个一键部署脚本 `deploy-langchain.sh`：

```bash
./deploy-langchain.sh
```

脚本自动完成：
1. 检查 hdc 连接，确认设备是 aarch64
2. 检查 patchelf，自动给 6 个 `.so` 补依赖（幂等——已 patch 的跳过）
3. 从编译产物目录打包（排除 test、`__pycache__` 等垃圾）
4. hdc 推到板子、解压
5. 校准板子时间（刷机后时间重置到 2008 年，SSL 证书会验证失败）
6. 跑验证脚本，确认全部 import OK

**RK3568 和 P7885 通用。**

────────────────────

【三次碰同一个根因】

到这里，**OHOS musl dlopen 不继承符号** 这个问题已经出现了三次：

| 次数 | 场景 | 修法 |
|:---|:---|:---|
| 第 1 次 | CPython 自己的 .so 模块（`_sha256` 等） | Makefile `BLDSHARED` 加 `-lpython3.11` |
| 第 2 次 | 第三方 Rust 扩展（pydantic-core） | `RUSTFLAGS="-l python3.11"` |
| 第 3 次 | 换设备后 Setup.stdlib 编译的模块漏了 | `patchelf --add-needed` |

**同一个根因，三种不同的表现形式，三种不同的修法。** 但本质都是一件事：在 OHOS 上，每个 `.so` 必须自己声明所有依赖，不能靠"进程里已经有了"来碰运气。

以后往 OHOS 移植任何 Python 扩展，**第一件事就是 `readelf -d xxx.so | grep libpython`**，没有就补。

◆ 第二段：在 App 里跑 Python

━━━━━━━━━━━━━━━━━━━━

到目前为止，Python 都是通过 hdc shell 在板子上手动跑的。这对开发者调试没问题，但对用户来说不行——**谁会用命令行？得有 App、有界面、点个按钮就能跑。**

目标很明确：在 OpenHarmony 的 App 里，点一个按钮，后台启动 Python 子进程，拿到输出，显示到界面上。

听起来不难。事实上，从第一行代码到最终跑通，踩了六个坑。

────────────────────

【第一个想法：process.runCmd()】

ArkTS（鸿蒙的开发语言）有个 `process.runCmd()` API，可以在 App 里执行系统命令，类似 Android 的 `Runtime.exec()`。

代码写好，编译——

```
error: process.runCmd is not a function
```

**SDK 20 删了这个 API。**

💡 OpenHarmony SDK 每个版本都在收紧安全策略。`process.runCmd()` 在 SDK 19 还能用，SDK 20 直接删了。官方的态度是：App 不应该直接执行系统命令。

这条路堵死了。ArkTS 层面没有任何执行外部命令的 API。

────────────────────

【第二个想法：NAPI C++ 层 popen】

ArkTS 不让执行命令，那就绕到 C++ 层。鸿蒙的 **NAPI**（Native API）机制允许 App 加载 C/C++ 编写的 `.so` 动态库，从 ArkTS 调 C++ 函数。

💡 **NAPI** 的意思是 Node-API——鸿蒙借鉴了 Node.js 的原生模块机制。你用 C++ 写一个函数，编译成 `.so`，注册到 NAPI 框架，ArkTS 就能调。

写了个 NAPI 模块，用 C 标准库的 `popen()` 执行命令：

```cpp
FILE* fp = popen("echo hello", "r");
// 读输出...
pclose(fp);
```

编译，推上板子，一跑——

```
exit code: 127
```

127 是"命令找不到"。echo 都找不到？

查了一下，**App 进程的 PATH 环境变量是空的。** 鸿蒙 App 进程启动时不继承系统的 PATH。popen 内部调 `/bin/sh -c "echo hello"`，sh 找不到 echo（因为 PATH 为空）。

加了 `setenv("PATH", "/bin:/system/bin", 1)` 之后——还是 127。

popen 在鸿蒙 App 进程里**行为异常**。不只是 PATH 问题，popen 调出来的 sh 本身就跑不起来。

────────────────────

【第三个想法：fork + execve】

popen 不行，那就自己 fork。`fork()` 创建子进程，`execve()` 执行命令，通过 pipe 拿输出。这是 Unix 进程创建的最底层 API，绕过了 popen 的封装。

```cpp
pid_t pid = fork();
if (pid == 0) {
    // 子进程
    setenv("PATH", "/bin:/system/bin:/vendor/bin", 1);
    const char* argv[] = {"/bin/sh", "-c", cmd, nullptr};
    execve("/bin/sh", (char* const*)argv, environ);
    _exit(127);
}
// 父进程通过 pipe 读输出
```

推上去，执行 `echo hello`——

```
hello
```

**能跑了。** fork + execve 在鸿蒙 App 进程里是可用的。

那就执行 Python 试试——Python 部署在 `/data/local/tmp/python-home/`，路径是确定的：

```
ls /data/local/tmp/  →  No such file or directory
```

────────────────────

**等一下。`/data/local/tmp/` 不存在？**

在 hdc shell 里看得到，在 App 里看不到。

────────────────────

【核心发现：App 沙箱的 mount namespace】

这是本篇最重要的技术发现。

💡 **mount namespace** 是 Linux 内核的隔离机制——每个进程可以有自己的"文件系统视图"。同一个物理路径，不同进程可能看到不同内容，或者根本看不到。

**鸿蒙的每个 App 都运行在独立的 mount namespace 里。** 普通 App 看到的文件系统是被裁剪过的：

- `/data/local/tmp/` → **不存在**（App 看不到）
- `/system/` → 只读，能看到
- `/data/storage/el2/base/...` → App 自己的沙箱目录

这不是权限问题（不是"没权限访问"），而是**根本不存在**——App 的 mount namespace 里压根没挂载 `/data/local/tmp/` 这个路径。

**Python 部署在 `/data/local/tmp/`，App 看不到这个目录，自然也跑不了。**

────────────────────

【转向系统 App】

普通 App 的沙箱隔离太严格，那**系统 App**呢？

💡 **系统 App** 是预装在鸿蒙系统里的 App（比如"设置"），拥有更高的权限和更宽松的沙箱。它们签名用的是系统签名，不是普通开发者签名。

我们选了 `com.ohos.settings`（设置 App）作为宿主。在设置 App 的代码里加一个 "Python Runner" 页面。

系统 App 的沙箱比普通 App 宽松，**但 `/data/local/tmp/` 还是看不到**。不过系统 App 有自己的沙箱目录，而且这个目录的物理路径可以通过 hdc 访问：

| 视角 | 路径 |
|:---|:---|
| ArkTS 代码（App 内部） | `/data/storage/el2/base/haps/phone/files` |
| hdc shell（物理路径） | `/data/app/el2/100/base/com.ohos.settings/haps/phone/files` |

两者是同一个目录——通过 mount namespace 映射。

**把 Python 推到物理路径，App 通过沙箱路径访问。**

```bash
# 从 WSL 通过 hdc 推到物理路径
hdc file send python-runtime.tar.gz \
    /data/app/el2/100/base/com.ohos.settings/haps/phone/files/

# App 里通过沙箱路径访问
# filesDir = /data/storage/el2/base/haps/phone/files
```

────────────────────

【fork+execve + 沙箱路径 → 能跑！】

把 Python 推到 App 沙箱的物理路径，然后在 NAPI 的 fork+execve 里用沙箱路径执行：

```cpp
// 子进程里
setenv("LD_LIBRARY_PATH", ".../files/python/lib", 1);
setenv("PYTHONHOME", ".../files/python", 1);
const char* cmd = ".../files/python/bin/python3.11 -c 'print(\"hello\")'";
execve("/bin/sh", {"-c", cmd}, environ);
```

P7885 板子上，设置 App 里点 "Run Python"——

```
Status: Python OK
Output: Python 3.11.11 (main, Mar 11 2026, 15:46:41) [Clang 15.0.4]
        Hello from OpenHarmony System App!
```

**在鸿蒙系统 App 里，Python 跑起来了。**

────────────────────

【技术路线总结】

```
ArkTS UI (pythonRunner.ets)
    │ import 'libexec_napi.so'
    ↓
C++ NAPI (exec_napi.cpp)
    │ fork() → 子进程: setenv + execve("/bin/sh", "-c", cmd)
    │ pipe 读 stdout + stderr
    ↓
LD_LIBRARY_PATH=.../python/lib
PYTHONHOME=.../python
python3.11 -c "print(...)"
    ↓
UI 显示结果
```

**关键决策一览：**

| 尝试 | 结果 | 原因 |
|:---|:---|:---|
| ArkTS `process.runCmd()` | 编译失败 | SDK 20 删除了该 API |
| NAPI C++ `popen()` | 所有命令 exit 127 | App 进程 PATH 为空 + popen 行为异常 |
| NAPI C++ `fork+execve` | echo 成功 | 最底层 Unix API，绕过封装 |
| Python 放 `/data/local/tmp/` | 文件不存在 | App mount namespace 隔离 |
| 普通 App 沙箱 | 权限不足 | 完全隔离，连 ls 都看不到外部目录 |
| **系统 App + 沙箱 filesDir** | **成功** | 系统 App 权限更高 + Python 部署到沙箱内 |

◆ 第三段：自包含部署——装个 App 就有 Python

━━━━━━━━━━━━━━━━━━━━

上面的方案有一步需要手动操作：`hdc file send python-runtime.tar.gz ...`。用户的板子上不一定连着 hdc。**能不能装个 App 就自带 Python，不需要手动推文件？**

能。

────────────────────

【rawfile 机制】

💡 鸿蒙 App 的 HAP（Harmony Ability Package，类似 Android 的 APK）里有个 `rawfile` 目录。放进去的文件会随 App 安装包一起打进去。App 运行时可以读取这些文件。

把 Python 运行时打成一个 `python-runtime.tar.gz`（26MB），放到项目的 `resources/rawfile/` 目录下。编译 HAP 时自动打进去。

App 里加一个 **Install Python** 按钮：

1. 读 rawfile 里的 `python-runtime.tar.gz`（24MB 压缩包）
2. 写到 App 沙箱的 `filesDir`
3. 执行 `tar xzf` 解压
4. `chmod 755` 加执行权限

用户的操作流程变成了：

```
安装 HAP → 打开设置 → Python Runner → 点 "Install Python"
→ 自动解压（几秒钟）→ 点 "Run Python" → 看到输出
```

**从"hdc + 命令行 + 手动推文件"简化到"装个 App + 点两下按钮"。**

────────────────────

【打包细节】

Python 运行时包含三部分：

| 内容 | 大小 | 说明 |
|:---|:---|:---|
| CPython 二进制 + 标准库 | ~15MB（压缩后） | `python3.11` + `lib/python3.11/` |
| libpython3.11.so | 4.8MB | Python 核心动态库 |
| libssl.so.3 + libcrypto.so.3 | 5.1MB | OpenSSL，HTTPS 必须 |

压缩后 26MB。HAP 总大小 33MB（含 Python 运行时 + App 本体 + 其他资源）。

────────────────────

【踩的坑】

| 坑 | 根因 | 解法 |
|:---|:---|:---|
| `aboutToAppear` 里 async 解压不生效 | ArkTS 的 `aboutToAppear` 生命周期不等 async Promise | 改成手动点按钮触发解压 |
| `fs.accessSync` 检测文件不准 | 文件不存在时没抛异常（OH 的 fs 行为和 Node.js 不完全一致） | 改用 `runCmd('ls ...')` 检测 |
| OpenSSL 没打进 tar | 打包时 `lib/` 路径不在目标目录下 | 把 libssl/libcrypto 复制到 `python-home/lib/` 再打包 |

────────────────────

【最终效果】

P7885 板子，安装 HAP，打开设置 App：

```
设置 → Python Runner → Install Python (23MB)
  rawfile read: 24006554 bytes → written to disk → tar: OK
  → Ready (installed!)

设置 → Python Runner → Run Python
  Status: Python OK
  Output: Python 3.11.11 (main, Mar 11 2026, 15:46:41) [Clang 15.0.4]
          Hello from OpenHarmony System App!
```

**一个 33MB 的 HAP，自带 Python 运行时。** 装上就能在鸿蒙设备上跑 Python 代码。

◆ 回头看这一路

━━━━━━━━━━━━━━━━━━━━

从上一篇的"hdc shell 手动跑 Python"到这一篇的"装个 App 就有 Python"，中间经历了：

**P7885 部署**——同一个根因第三次出现，但这次 10 分钟解决。patchelf 比重新编译优雅得多。

**App 沙箱探索**——6 次尝试，6 个坑。最值得记住的教训是 **mount namespace 隔离**：你在 hdc shell 里看到的文件系统和 App 里看到的**不是同一个**。普通 App 甚至看不到 `/data/local/tmp/`。这不是权限问题，是"不存在"。

**NAPI fork+execve**——ArkTS 层面没有执行外部命令的 API（SDK 20 删了 `process.runCmd()`），只能往下走到 C++ 层。popen 在 App 进程里不可用，fork+execve 可以。这是最底层的 Unix 进程创建方式，绕过了所有封装。

**系统 App 是必须的**——普通 App 的沙箱太严格。系统 App 有更宽松的权限和更大的文件系统视野。代价是需要系统签名。

**rawfile 自包含**——26MB 的 Python 运行时打进 HAP 的 rawfile，App 内一键解压到沙箱。用户体验从"连 hdc + 推文件 + 敲命令"变成"装 App + 点按钮"。

────────────────────

**这个项目从开始到现在，每一步都在和"理所当然"作斗争。**

glibc 上理所当然能跑的 dlopen，在 OHOS musl 上不行。

理所当然能用的 `process.runCmd()`，被 SDK 20 删了。

理所当然能看到的 `/data/local/tmp/`，在 App 的 mount namespace 里不存在。

**每一个"理所当然"背后，都是一个具体的系统设计决策——有人决定了 musl 的 dlopen 不继承符号，有人决定了 SDK 20 删除命令执行 API，有人决定了 App 的 mount namespace 隔离策略。**

搞明白这些决策的逻辑，比记住具体的修法更重要。因为下一个坑一定不是同一个坑——但做决策的逻辑是通用的。

◆ 术语速查

━━━━━━━━━━━━━━━━━━━━

| 术语 | 解释 |
|:---|:---|
| patchelf | 修改已编译的 ELF 二进制文件的工具。可以给 .so 补加依赖声明，不用重新编译 |
| readelf | 查看 ELF 文件信息的工具。`readelf -d xxx.so` 能看它依赖了哪些库 |
| mount namespace | Linux 内核的隔离机制。每个进程可以有自己的文件系统视图，看不到其他进程挂载的目录 |
| NAPI | Native API，鸿蒙的 C/C++ 原生模块机制。让 ArkTS 调 C++ 函数 |
| fork + execve | Unix 进程创建的最底层 API。fork 复制进程，execve 替换为新程序 |
| popen | C 标准库函数，内部调 fork+sh 来执行命令。是 fork+execve 的高层封装 |
| HAP | Harmony Ability Package，鸿蒙 App 的安装包格式，类似 Android APK |
| rawfile | HAP 里的资源目录。放进去的文件随 App 安装，运行时可读取 |
| filesDir | App 沙箱的私有文件目录。App 有读写权限，其他 App 不能访问 |
| 系统 App | 预装在系统里的 App（如"设置"），使用系统签名，权限高于普通 App |
| ArkTS | 鸿蒙的开发语言，基于 TypeScript，加了声明式 UI 语法 |
| Setup.stdlib | CPython 3.11 新增的模块配置文件，核心模块通过它声明编译方式 |
| BLDSHARED | CPython Makefile 里的变量，定义编 .so 时的链接命令 |
| dlopen | C 语言的动态库加载函数。Python 的 `import _ssl` 底层就是调它 |
| musl | 轻量级 C 标准库。Alpine Linux 和 OpenHarmony 都用它（但 OH 有魔改） |
| Signal 11 / SIGSEGV | 段错误信号。程序访问非法内存，操作系统直接杀掉进程 |
