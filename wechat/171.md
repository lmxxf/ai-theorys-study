先说结论：**两台 DGX Spark，48 小时，18 个坑，最终跑通了 DeepSeek V4 Flash 280B。**

如果你在社交媒体上看到"DGX Spark 一键跑大模型"的宣传，请做好心理准备——那说的是跑已有生态支持的模型。DeepSeek V4 是 2026 年 4 月 24 日发布的，发布五天后我开始折腾，整个开源生态还没跟上。

这篇文章不是教程（教程在 GitHub readme 里），而是一份踩坑记录。重点讲三件事：DGX Spark 双机的真实架构是什么样的、为什么 V4 在 Spark 上特别难跑、以及最终方案是怎么拼出来的。

◆ 先搞清楚硬件：256GB 不是你想的那个 256GB

━━━━━━━━━━━━━━━━━━━━

NVIDIA 宣传两台 DGX Spark 组成"256GB 共享内存"。第一次看到这句话，我以为是像 Mac 的统一内存一样——应用程序看到一块 256GB 的大内存，系统自动管理。

**不是的。**

实际架构是这样的：

- 两台独立的机器，每台 128GB 统一内存（CPU 和 GPU 共用）
- 两台之间通过一根 QSFP56 DAC 线缆直连，走 ConnectX-7 网卡，带宽 200Gbps
- GPU 之间的通信走 NCCL 协议，通过这根线缆做 AllReduce
- **模型权重两台各存一份**——不是"一份权重分布在两台上"

💡 翻译成人话：这不是一台大电脑，而是一个两节点的分布式集群。你踩的每一个分布式系统的坑，这里一个不少。

────────────────────

【Tensor Parallel 怎么工作】

DeepSeek V4 Flash 有 280B 参数，原生 FP4 精度发布，模型文件约 158GB。两台 Spark 用 Tensor Parallel（张量并行）来分工：

- 每一层的矩阵乘法被水平切成两半
- host 算前半部分，slave 算后半部分
- 算完后通过 NCCL AllReduce 把结果同步
- **每一层都要同步一次**——61 层就是 61 次跨机通信

200Gbps 听起来很快，但同机 NVLink 是 900Gbps，差了 4.5 倍。好在 V4 是 MoE 架构，每个 token 只激活 280B 中的约 13B 参数，跨机通信量相对可控。

────────────────────

【三个角色分工】

| 组件 | 干什么 | 类比 |
|------|--------|------|
| **Ray** | 调度员：管进程启停、资源分配 | 工地项目经理 |
| **NCCL** | 对讲机：GPU 之间传张量 | 工人之间喊话 |
| **vLLM** | 干活的：实际跑推理 | 搬砖的工人 |

◆ 为什么 DeepSeek V4 在 Spark 上这么难跑

━━━━━━━━━━━━━━━━━━━━

两个原因叠加，导致这个组合特别棘手。

────────────────────

【原因一：V4 太新，生态没跟上】

DeepSeek V4 是 2026-04-24 发布的。NVIDIA 官方 NGC 容器 `nvcr.io/nvidia/vllm:26.03-py3` 是 3 月份的，里面的 transformers 4.57.5 压根不认识 `deepseek_v4` 这个模型架构。

你可能会想：pip 升级一下不就行了？

**踩了才知道不行。** NGC 容器里的 PyTorch 是 NVIDIA 针对 ARM64 + Blackwell 架构深度定制的版本。你一 `pip install --upgrade vllm`，它连带把 torch 换成了社区版，然后 torch inductor 编译器直接崩了。inductor 是 PyTorch 2.0 引入的即时编译器（JIT），把 Python 写的模型自动编译成高效的 GPU 机器码——类似 C++ 编译器，但编译的是神经网络计算图。定制版和社区版的 inductor 在底层实现上不兼容，报错 `AssertionError: auto_functionalized was not removed`。

💡 翻译成人话：NGC 容器是一个精密的积木塔，你不能只换中间一块——拆一块就全塌了。

────────────────────

【原因二：sm_121 孤儿架构】

这是更深层的问题。DGX Spark 用的 GPU 是 NVIDIA GB10，CUDA 架构代号 **sm_121**。

在 NVIDIA 的产品线里，Blackwell 架构分两条线：

- **sm_100**：数据中心级 Blackwell（B100/B200/GB200），所有主流框架都支持
- **sm_120**：桌面级 Blackwell（RTX 5090 等），PyTorch 社区版支持到这里
- **sm_121**：DGX Spark 专属——**既不是数据中心级，也不完全是桌面级**

DeepSeek V4 依赖一个叫 DeepGEMM 的高性能矩阵乘法库。这个库的 hyperconnection kernel 硬编码了架构检查：只支持 sm_90（Hopper）和 sm_100（数据中心 Blackwell），**sm_121 不在列表里**。

你设环境变量 `VLLM_USE_DEEP_GEMM=0` 想绕过？没用。hyperconnection 是独立的代码路径，环境变量管不到。

💡 翻译成人话：sm_121 是个孤儿。大厂的 kernel 不管它，社区的 kernel 也没覆盖到它。你拿着一颗"看起来像 Blackwell"的 GPU，但所有为 Blackwell 写的代码都不认它。

━━━━━━━━━━━━━━━━━━━━

◆ 最终方案：三个项目拼出来的

━━━━━━━━━━━━━━━━━━━━

单靠任何一个开源项目都跑不通。最终方案是三个项目的组合：

1. **eugr/spark-vllm-docker**：提供 Docker 构建基础设施——Dockerfile、编排脚本、双机启动流程。这个项目专门为 DGX Spark 做的，构建逻辑很成熟。

2. **jasl/vllm 的 ds4-sm120 分支**：有人专门 fork 了 vLLM，给 sm_120 系列做了支持。核心改动是用 Triton kernel 替代了 DeepGEMM 的 hyperconnection。Triton 是 OpenAI 开源的 GPU 编程语言，你用 Python 写 kernel，它自动编译成目标 GPU 的机器码——不挑 CUDA 架构，换 GPU 不用重写。

3. **sm_120 前向兼容**：PyTorch 社区版只编译到 sm_120 的 CUDA kernel，但 sm_121 是 sm_120 的超集——**sm_120 的 kernel 可以直接在 sm_121 上跑**。编译时指定 `TORCH_CUDA_ARCH_LIST="12.0"` 加 `CMAKE_CUDA_ARCHITECTURES="120-real"` 就行。

整个过程：把 eugr 的 Dockerfile 里预编译 whl 替换成 jasl 的源码编译，加上 sm_120 架构参数，构建出一个新镜像 `vllm-node-sm120`。

◆ 最有价值的 6 个坑

━━━━━━━━━━━━━━━━━━━━

48 小时踩了 18 个坑，挑 6 个最有普遍价值的。

────────────────────

【坑 1：NGC 容器里不能 pip 升级 torch】

前面讲过了。NGC 的 torch 是定制版，pip 升级会换成社区版，inductor 编译器直接崩。

**教训**：碰到 NGC 容器，要么用它原装的版本，要么完全不用它——别试图"部分升级"。

────────────────────

【坑 2：DeepGEMM 不支持 sm_121】

`RuntimeError: Assertion error (csrc/apis/hyperconnection.hpp:56): Unsupported architecture`

这个报错出现在你以为"终于快成了"的时候，特别打击人。DeepGEMM 的 C++ 代码里写死了架构白名单，sm_121 不在里面。设环境变量 `VLLM_USE_DEEP_GEMM=0` 也没用——hyperconnection 是独立的代码路径，环境变量管不到。

**解法**：用 jasl/vllm 的 ds4-sm120 分支，它用 Triton + TileLang 重写了 hyperconnection kernel，绕过了 DeepGEMM 的架构检查。这也是我们最终方案的核心。

**教训**：对于新硬件，先检查 CUDA kernel 的架构兼容性，再花时间编译。

────────────────────

【坑 3：Docker 里编译 vLLM 需要 1 小时】

vLLM 不只是 Python 包——它有大量 C++/CUDA 扩展代码（注意力 kernel、量化 kernel、自定义算子等），`pip install .` 的时候要用 nvcc 把这些 CUDA 代码编译成 GPU 可执行的机器码。这一步在 ARM64 Grace CPU 上特别慢，MAX_JOBS=16 并行编译也要约 1 小时。我总共编译了 4 次——每次改一行 Dockerfile 就得等一小时。

期间遇到的依赖问题清单：
- 缺 `setuptools_scm`、`pybind11`、`cmake`——逐个加到 Dockerfile
- `setuptools` 版本冲突——torch 要 `<82`，vllm 要 `<81`，新 pyproject.toml 要 `>=75`
- requirements.txt 里的 hash 锁定——用 `sed` 删掉所有 `--hash` 行

**教训**：编译慢的项目，每次试错成本极高。改 Dockerfile 之前先把所有依赖理清楚，争取一次改对。

────────────────────

【坑 4：Docker 里访问不了 GitHub】

我的科学上网方式是 ShellCrash 的 Redir 模式（iptables 透明代理）。宿主机不需要设 proxy 环境变量就能访问外网，但 Docker build 的网络是隔离的，不走宿主机的 iptables。

配 Docker 代理又是另一个噩梦：`127.0.0.1` 在容器内不可达，必须用宿主机 IP；`noProxy` 要排除 `*.ubuntu.com`、`*.nvidia.com`、`*.pypi.org` 等不需要科学上网的域名……

**最终方案**：放弃在 Docker 里配代理。GitHub 上需要的东西（NCCL、vLLM、DeepGEMM）全部在宿主机预先 clone 好，COPY 进容器。简单粗暴，但管用。

────────────────────

【坑 5：jasl fork 不能部分替换文件】

第一反应是用 Docker `-v` 挂载把 jasl fork 的几个 .py 文件替换到 eugr 容器里——只换改过的文件，不用重新编译。结果报 `ImportError: cannot import name 'dequantize_combined_sparse_mla_decode_kv'`——两个版本的 vLLM 内部 API 完全不兼容，部分替换行不通。

**解法**：老老实实把 jasl 的整个 vLLM 源码 COPY 进 eugr 的 Dockerfile 里，从头编译。多花一小时，但保证所有内部 API 一致。

**教训**：大项目不同 fork 之间的内部 API 差异很大，"只替换几个文件"是幻想，只能整体替换源码重新编译。

────────────────────

【坑 6：sm_120 前向兼容——最后一个也是最关键的坑】

所有代码问题解决后，运行时报 `CUDA error: no kernel image is available for execution on the device`。

根因：PyTorch 社区版编译时生成的 CUDA kernel 只包含到 sm_120，而 DGX Spark 的 GPU 实际报告自己是 sm_121（Capability 12.1）。torch 找不到 sm_121 的 kernel，就崩了。

解法出奇简单：编译时指定 `TORCH_CUDA_ARCH_LIST="12.0"`。sm_121 是 sm_120 的超集，能跑 sm_120 的所有 kernel。**前向兼容**——高版本芯片能跑低版本的代码。

这是最后一个坑。改完重新编译 43 分钟，跑通了。

━━━━━━━━━━━━━━━━━━━━

◆ 最终效果

━━━━━━━━━━━━━━━━━━━━

首次请求返回：

> "你好，我是DeepSeek，一个由深度求索公司打造的AI助手，乐于为你解答问题、提供信息与创意灵感。"

看到这行字的时候，已经是 4 月 29 日下午。从 28 号晚上开始折腾，整整 48 小时。

速度嘛——说实话，慢。体感大概 20 tok/s 左右，打字一个一个蹦出来。原因有两个：一是 `--enforce-eager` 模式关掉了 CUDA graph 优化（torch.compile 和 sm_121 还不兼容），相当于没开涡轮增压；二是双机 Tensor Parallel 每一层都要跨 CX7 做一次 AllReduce，200Gbps 虽然不慢但跟同机 NVLink 900Gbps 比还是差了 4 倍多。等 vLLM 官方把 sm_121 的 torch.compile 支持合进去，速度会有明显提升。

不过 280B 模型能在两台桌面小主机上跑起来本身就是件不错的事——毕竟这东西设计上是给 H100/B200 集群跑的。

启动命令：

```bash
HF_HOME=/home/lmxxf/work/deepseek-v4-flash-deployment \
./launch-cluster.sh -t vllm-node-sm120 exec \
  vllm serve /root/.cache/huggingface/deepseek-v4-flash \
  --tensor-parallel-size 2 \
  --distributed-executor-backend ray \
  --gpu-memory-utilization 0.85 \
  --kv-cache-dtype fp8 \
  --max-model-len 1000000 \
  --enforce-eager
```

两台 Spark 各分到 73.85 GiB 显存，模型权重每台 73.85 GiB，剩余约 58GB 给 KV cache。DeepSeek V4 的 KV cache 极小（仅为传统模型的 2%），1M 上下文 fp8 KV cache 只需约 5GB，所以 `--max-model-len` 直接拉满 1M 没问题。

◆ 附：DGX Spark 双机部署 DeepSeek V4 Flash 完整步骤

━━━━━━━━━━━━━━━━━━━━

国内几乎没有 DGX Spark 双机部署的中文教程，这里完整记录一遍。已经做好了预构建镜像，照着做不需要自己编译。

────────────────────

【第一步：双机组网】

两台 Spark 用 QSFP56 线缆连接 ConnectX-7 网卡后，两台都执行：

```bash
# 配置 CX7 网络
sudo wget -O /etc/netplan/40-cx7.yaml \
  https://github.com/NVIDIA/dgx-spark-playbooks/raw/main/nvidia/connect-two-sparks/assets/cx7-netplan.yaml
sudo chmod 600 /etc/netplan/40-cx7.yaml
sudo netplan apply

# 自动发现对方 + 配置 SSH 免密
wget https://github.com/NVIDIA/dgx-spark-playbooks/raw/refs/heads/main/nvidia/connect-two-sparks/assets/discover-sparks
chmod +x discover-sparks
./discover-sparks
```

完成后两台机器会分到 `169.254.x.x` 的 CX7 直连 IP，互相 ping 验证。

────────────────────

【第二步：下载模型权重】

158GB 的权重，**两台都需要有一份**（不是共享的，前面讲过了）。

在 host 上下载（国内用 hf-mirror 镜像）：

```bash
export HF_ENDPOINT=https://hf-mirror.com
huggingface-cli download deepseek-ai/DeepSeek-V4-Flash \
  --local-dir ./deepseek-v4-flash
```

然后 rsync 到 slave（走 CX7 200Gbps，581MB/s，五分钟搞定）：

```bash
rsync -avP ./deepseek-v4-flash/ \
  user@<slave的CX7_IP>:./deepseek-v4-flash/
```

────────────────────

【第三步：拉取 Docker 镜像】

我已经构建好了预编译镜像，直接拉取，不需要自己编译一小时：

```bash
docker pull lmxxf/vllm-deepseek-v4-dgx-spark:latest
```

两台都要拉。或者 host 拉完后用 `docker save | ssh ... docker load` 走 CX7 传给 slave。

还需要 eugr 的启动脚本：

```bash
git clone https://github.com/eugr/spark-vllm-docker.git
cd spark-vllm-docker
```

────────────────────

【第四步：启动推理服务】

```bash
HF_HOME=/path/to/weights/parent \
./launch-cluster.sh \
  -t lmxxf/vllm-deepseek-v4-dgx-spark exec \
  vllm serve /root/.cache/huggingface/deepseek-v4-flash \
  --tensor-parallel-size 2 \
  --distributed-executor-backend ray \
  --gpu-memory-utilization 0.85 \
  --kv-cache-dtype fp8 \
  --max-model-len 8192 \
  --enforce-eager
```

`HF_HOME` 指向模型权重所在目录的**上一级**。脚本会自动在两台机器上启动 Docker 容器、组建 Ray 集群、加载模型。首次启动加载权重约 2 分钟。

────────────────────

【第五步：测试】

另开终端：

```bash
curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "/root/.cache/huggingface/deepseek-v4-flash",
    "messages": [{"role":"user","content":"你好"}],
    "max_tokens": 100
  }'
```

看到 JSON 返回就成功了。vLLM 暴露的是 OpenAI 兼容 API，可以接任何支持自定义 API 地址的聊天前端（Open WebUI、ChatBox、LobeChat 等）。

────────────────────

💡 翻译成人话：总共五步——接线配网、下权重、拉镜像、启动、测试。如果用预构建镜像，不需要编译，整个过程最慢的是下载 158GB 的权重。

◆ 写在最后

━━━━━━━━━━━━━━━━━━━━

整个过程里，AI 帮了大忙——报错信息贴给 Claude 分析、让它帮忙定位 Dockerfile 里的依赖冲突、让它解释 sm_121 和 sm_120 的关系。但每一步操作还是人来执行的。AI 分析报错，我来拍板改哪里、编译、测试。

几个值得记住的结论：

**DGX Spark 双机不是一台大电脑**——是分布式集群，模型权重两边各存一份，所有分布式的坑一个不少。

**不要在 NGC 容器里 pip 升级 torch**——定制版和社区版不兼容，拆一块就全塌。

**sm_121 是孤儿架构**——数据中心 Blackwell 的 kernel 不管它，桌面 Blackwell 的 kernel 差一点也不管它。需要 Triton fallback 或者 sm_120 前向兼容来救。

**新模型 + 新硬件 = 地狱级配置**——V4 发布五天后生态还没跟上，sm_121 生态本身就薄弱，两者叠加就是 48 小时。等生态成熟了，可能就是一条命令的事。
