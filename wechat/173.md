【张量并行】169 期的前向传播，在两台机器上到底怎么切

◆ 从"水平切成两半"说起

━━━━━━━━━━━━━━━━━━━━

169 期（ https://mp.weixin.qq.com/s/mnbaXhQmQjDAyGD1dDdq5Q ）走完了 DeepSeek V4-Pro 的整条前向传播路径——从 token ID 到 embedding，到 61 层 Transformer，到 softmax 采样出"北京"。每一步标了维度变化，一个矩阵乘法都不漏。

171 期（ https://mp.weixin.qq.com/s/JPxR8rHSb1Pko3vq2ukY3A ）在两台 DGX Spark 上跑通了 V4 Flash 280B。但关于张量并行只说了一句话：**"每一层的矩阵乘法被水平切成两半，算完后通过 NCCL AllReduce 把结果同步。"**

具体怎么切？哪些矩阵切了、哪些没切？AllReduce 在哪发生、传多少数据？这篇展开讲。

169 期用的是 V4 Pro 的参数，171 期跑的是 V4 Flash。Pro 和 Flash 的 TP 切法原理完全一样，区别只是维度大小不同。**本篇统一用 V4 Flash 的参数**（毕竟 171 期实际跑的就是 Flash），和 Pro 的关键差异列在下面：

| 参数 | V4 Pro | V4 Flash（本文） |
|------|--------|-----------------|
| hidden_size | 7168 | **4096** |
| 注意力头数 | 128 | **64** |
| 层数 | 61 | **43** |
| q_lora_rank | 1536 | **1024** |
| 路由专家数 | 384 | **256** |
| 专家中间维度 | 3072 | **2048** |
| O 投影分组 | 16 组 | **8 组** |
| head_dim | 512 | 512（相同） |

**前置知识**：你需要熟悉 169 期的前向传播流程（每一步的维度变化）和 171 期的硬件架构（两台 DGX Spark 通过 CX7 200Gbps 直连）。169 期用的是 Pro 的维度，本篇改成了 Flash 的维度，但每一步的结构和逻辑完全一致——只是数字不同。

◆ 张量并行的基本原理：一个矩阵乘法怎么分到两台机器上

━━━━━━━━━━━━━━━━━━━━

先从最简单的场景开始：一个矩阵乘法 `Y = X × W`，怎么分到两台机器上算？

两种切法。

────────────────────

【Column Parallel：按列切 W，结果拼接】

把权重矩阵 W 按列切成两半，每台各存一半：

```text
完整: X [1, 5, 4096] × W [4096, 4096] → Y [1, 5, 4096]

TP=2 切法:
  GPU 0: X [1, 5, 4096] × W₀ [4096, 2048] → Y₀ [1, 5, 2048]  ← 前 2048 列
  GPU 1: X [1, 5, 4096] × W₁ [4096, 2048] → Y₁ [1, 5, 2048]  ← 后 2048 列
```

两台各算出结果的一半。要得到完整的 Y，把两半**拼接**（concatenate）起来：`Y = [Y₀ | Y₁]`。

注意：两台的输入 X 必须是完整的、一模一样的。

────────────────────

【Row Parallel：按行切 W，结果相加】

把权重矩阵 W 按行切成两半，对应地，输入也要切成两半：

```text
完整: X [1, 5, 4096] × W [4096, 4096] → Y [1, 5, 4096]

TP=2 切法:
  GPU 0: X₀ [1, 5, 2048] × W₀ [2048, 4096] → Y₀ [1, 5, 4096]  ← 前 2048 行
  GPU 1: X₁ [1, 5, 2048] × W₁ [2048, 4096] → Y₁ [1, 5, 4096]  ← 后 2048 行
```

两台各算出一个**部分和**，大小和完整的 Y 一样。要得到完整的 Y，把两个部分和**相加**：`Y = Y₀ + Y₁`。

这个"相加"不是本地操作——两台机器各自只有自己的部分和，必须通过网络把两个部分和合并。**这就是 AllReduce。**

────────────────────

【为什么要配对使用】

实际使用中，Column Parallel 和 Row Parallel 总是成对出现。原因很简单：

- Column Parallel 的输出是"切好的"——GPU 0 有前半，GPU 1 有后半
- Row Parallel 的输入恰好需要"切好的"——GPU 0 用前半行，GPU 1 用后半行

**前一层 Column Parallel 的输出，正好是后一层 Row Parallel 的输入。** 中间不需要任何通信。

**AllReduce 只在 Row Parallel 的输出端出现**——因为 Row Parallel 的两个部分和必须加起来才能得到正确结果。Column Parallel 的输出不需要通信，各自保留各自的半边就行。

一对 Column + Row 的完整流程：

```text
X [1,5,4096]（两台相同）
  ↓ Column Parallel（按列切 W，无通信）
GPU 0: [1,5,2048]          GPU 1: [1,5,2048]
  ↓ Row Parallel（按行切 W，无通信）
GPU 0: [1,5,4096] 部分和    GPU 1: [1,5,4096] 部分和
  ↓ AllReduce（跨机通信，两个部分和相加）
GPU 0: [1,5,4096] 完整      GPU 1: [1,5,4096] 完整
```

💡 翻译成人话：Column Parallel 是"你算左半边我算右半边，各自留着"。Row Parallel 是"你算上半部分我算下半部分，然后加起来"。加起来这一步需要通信——这就是 AllReduce。整个张量并行里，通信只发生在"加起来"这一步。

◆ 工业机上怎么切——我们看不见，但逻辑是一样的

━━━━━━━━━━━━━━━━━━━━

DeepSeek 训练 V4 用的是自己的 GPU 集群——至少几千张卡，TP 切法写在内部训练框架里。这部分代码没有开源，我们看不见。

但 DeepSeek 开源的**推理代码** `inference/model.py` 里，TP 切分逻辑是完整的。打开文件就能看到 `ColumnParallelLinear`、`RowParallelLinear`、MoE 专家按 `world_size` 均分、`dist.all_reduce` 同步——和上一节讲的 Column + Row 配对原理一模一样。启动方式是 `torchrun --nproc-per-node ${MP}`，标准的 PyTorch 分布式多进程。这不是单卡代码——V4 280B 的权重连单张 H100 80GB 都塞不下，官方推理代码天然就是多卡设计的。

171 期我们用的 vLLM 也有自己的一套 TP 实现（`vllm/model_executor/models/deepseek_v4.py`），和 DeepSeek 官方推理代码是独立实现的。但打开两边的代码对比一下，**切法完全一致**——哪些矩阵用 Column Parallel、哪些用 Row Parallel、哪些不切完整复制、MoE 按专家编号分——一模一样。

这不是巧合，是因为对给定的模型架构来说，合理的 TP 切法基本只有一种。哪些维度可以按头切、哪些是瓶颈维度不值得切、AllReduce 放在哪——这些决策被模型架构本身约束住了。你换成 SGLang、TensorRT-LLM 或者任何一个推理框架，切法也是一样的。

那 vLLM 的价值在哪？不在 TP 切法本身，而在**推理优化**——continuous batching（多用户请求动态拼批）、PagedAttention（KV cache 按页管理，避免显存碎片）、CUDA graph（编译计算图减少调度开销）、FP8 KV cache 量化。DeepSeek 的 `inference/model.py` 是"能跑"，vLLM 是"跑得快、能上生产"。171 期我们用 vLLM 而不是 DeepSeek 官方代码，原因就在这。

━━━━━━━━━━━━━━━━━━━━

◆ 代码里怎么表示"从哪个方向切"和"不切"

━━━━━━━━━━━━━━━━━━━━

模型代码里不会手动写 `W[:, :32768]` 来切矩阵。TP 的切分是**声明式**的——模型定义时用不同的 API 声明每个矩阵的切法，运行时根据 `world_size`（几台机器）自动适配。

三种 API，对应三种切法：

────────────────────

【ColumnParallelLinear——按列切】

```python
# 模型代码里这样写
self.W_q_up = ColumnParallelLinear(1024, 32768)
```

内部只做一件事——把输出维度除以机器数：

```python
# ColumnParallelLinear 的 __init__
self.output_size_per_partition = output_size // world_size
# world_size=2 → 每台的权重变成 [1024, 16384]
# world_size=4 → 每台的权重变成 [1024, 8192]
# world_size=8 → 每台的权重变成 [1024, 4096]
```

加载权重时，每个 rank 根据自己的编号取对应的列片段——rank 0 取前 16384 列，rank 1 取后 16384 列。`forward()` 只用自己那一片权重做矩阵乘法，输出自然是完整结果的 1/world_size。

**TP=2 还是 TP=8，模型代码不用改一个字。** 切几份完全由启动时的 `world_size` 决定。

────────────────────

【RowParallelLinear——按行切】

```python
self.wo_b = RowParallelLinear(1024, 4096)
```

和 Column 反过来——按输入维度切：

```python
# RowParallelLinear 的 __init__
self.input_size_per_partition = input_size // world_size
# world_size=2 → 每台的权重变成 [512, 4096]
```

`forward()` 最后调一次 `dist.all_reduce()` 把部分和加起来。**AllReduce 写在 RowParallelLinear 的 forward 里**——模型代码不用自己写通信逻辑，用了 Row Parallel 就自动带通信。

────────────────────

【ReplicatedLinear——不切】

```python
# 方式一：直接用 ReplicatedLinear
self.W_kv = ReplicatedLinear(4096, 512)

# 方式二：ColumnParallelLinear 加 disable_tp=True（vLLM 的写法）
self.W_kv = ColumnParallelLinear(4096, 512, disable_tp=True)

# 方式三：DeepSeek 官方代码里就用普通 nn.Linear
self.W_kv = nn.Linear(4096, 512)
```

不管哪种写法，效果都一样：**每台存完整的权重，算完整的结果，不通信。** `world_size` 变了也不影响——不切就是不切。

────────────────────

【MoE 专家的分配】

MoE 不是切矩阵，是分专家。逻辑同样自动：

```python
# DeepSeek 官方 inference/model.py
self.n_local_experts = n_routed_experts // world_size
self.expert_start = rank * self.n_local_experts
# world_size=2 → 每台 128 个专家
# world_size=4 → 每台 64 个专家
# world_size=8 → 每台 32 个专家
```

加载权重时每个 rank 只加载自己负责的专家。forward 完了 `dist.all_reduce()` 合并。

────────────────────

所以整个 TP 切分的代码骨架长这样：

```python
class TransformerLayer:
    def __init__(self):
        # 不切——瓶颈维度太小 / MLA 只有 1 份 KV / Router 必须一致
        self.W_q_down = ColumnParallelLinear(4096, 1024, disable_tp=True)
        self.W_kv     = ColumnParallelLinear(4096, 512,  disable_tp=True)
        self.W_gate   = ReplicatedLinear(4096, 256)

        # 按列切——输出维度大，可以按头/按组分
        self.W_q_up   = ColumnParallelLinear(1024, 32768)
        self.wo_a     = ColumnParallelLinear(4096, 1024)

        # 按行切——和上面的 Column 配对，带 AllReduce
        self.wo_b     = RowParallelLinear(1024, 4096)
```

**每个矩阵的切法在模型定义时就确定了，运行时自动适配并行度。** 模型代码告诉框架"这个矩阵按列切、那个按行切、这个不切"，框架根据 `world_size` 自动算切片大小、自动加载对应的权重片段、自动在该通信的地方插入 AllReduce。

下面沿着 169 期的前向传播路径，逐步标注每个矩阵的切法和通信点。

━━━━━━━━━━━━━━━━━━━━

◆ Attention 层怎么切

━━━━━━━━━━━━━━━━━━━━

────────────────────

【Q 低秩投影】

169 期里的 Q 投影分四步：`h → W_q_down → RMSNorm → W_q_up → 逐头RMSNorm`。

**W_q_down [4096, 1024]：不切，两台各存一份完整的。**

```text
GPU 0: h [1,5,4096] × W_q_down [4096,1024] → q_latent [1,5,1024]  完整
GPU 1: h [1,5,4096] × W_q_down [4096,1024] → q_latent [1,5,1024]  完整
```

为什么不切？1024 维已经是压缩过的瓶颈维度了。如果按列切成 512+512，后面 RMSNorm 要在完整的 1024 维上算均方根——切了就没法独立做 RMSNorm，还得通信一次把两半拼起来。切 1024 维省不了多少计算，反而多一次通信。不划算。

**W_q_up [1024, 32768]：按列切（Column Parallel）。**

32768 = 64 头 × 512 维/头。TP=2 按头切——GPU 0 算头 0~31，GPU 1 算头 32~63。

```text
GPU 0: q_latent [1,5,1024] × W_q_up₀ [1024,16384] → q [1,5,32,512]  ← 头 0~31
GPU 1: q_latent [1,5,1024] × W_q_up₁ [1024,16384] → q [1,5,32,512]  ← 头 32~63
```

每台只存半张 W_q_up，只算 32 个头的 Q。这是标准的 Column Parallel——按头切输出维度。

逐头 RMSNorm 在各自的 32 头上独立算，不需要通信。

────────────────────

【KV 投影】

**W_kv [4096, 512]：不切，两台各存一份完整的。**

```text
GPU 0: h [1,5,4096] × W_kv [4096,512] → kv [1,5,1,512]  完整
GPU 1: h [1,5,4096] × W_kv [4096,512] → kv [1,5,1,512]  完整
```

这是 MLA 在张量并行下的核心约束。169 期讲过，V4 的 KV 只有 **1 份**（`num_key_value_heads=1`），不像 MHA 有 128 份（Pro）或 64 份（Flash）可以按头切。1 份 KV 没法切——切了就不完整了。

这带来一个代价：**KV cache 在 TP 下翻倍。** 两台各存一份完整的 KV latent，总量是单机的两倍。

对传统 MHA 来说，TP 是 KV cache 的救星——64 份 KV 按头切分到各台，每台只存 32 份，KV cache 显存减半。长上下文场景下 KV cache 动辄几十 GB，TP=8 能把每台的 KV cache 压到 1/8，这是大规模部署的关键。

**MLA 放弃了这个红利。** 但 MLA 把每个 token 的 KV 从 16384 维（如果 Flash 用传统 MHA：64 头 × 128 维/头 × 2 份 K+V）压到了 512 维，再加上 CSA/HCA 的压缩（大部分 token 被 4:1 或 128:1 压缩，只有滑动窗口内最近 128 个 token 存原始 KV），1M 上下文 fp8 KV cache 每台只需约 3GB。底子已经小到翻倍也不疼。

────────────────────

【注意力计算】

每台用自己的 32 个 Q 头 × 完整的 1 份 KV → 各自得到 32 头的注意力输出。

```text
GPU 0: q[1,5,32,512] × kv[1,5,1,512] → attn_out [1,5,32,512]  ← 头 0~31
GPU 1: q[1,5,32,512] × kv[1,5,1,512] → attn_out [1,5,32,512]  ← 头 32~63
```

两台完全独立计算，不需要通信。每台算的是不同的头——头 0~31 和头 32~63 之间本来就没有交互。

────────────────────

【O 分组低秩投影——第一次 AllReduce】

Flash 的 64 头分 8 组，每组 8 头。TP=2 时，每台 4 组（32 头 / 8 头每组 = 4 组）。

```text
GPU 0: 4 组 × wo_a [4096, 1024] → sum → [1,5,1024]
GPU 1: 4 组 × wo_a [4096, 1024] → sum → [1,5,1024]
```

wo_a 是 Column Parallel——每台持有自己负责的 4 组的 wo_a 权重。

接下来的 wo_b 是 Row Parallel。完整的操作是"8 组各自通过 wo_a 压到 1024 维，8 组加起来得到一个 1024 维向量，再通过共享的 wo_b [1024, 4096] 展开回 4096 维"。

在 TP=2 下，8 组的 sum 被拆成两步：GPU 0 把自己的 4 组加起来得到一个部分和 [1,5,1024]，GPU 1 也是。**两台各自持有完整的 wo_b [1024, 4096]**，各自把自己的部分和 [1,5,1024] 过 wo_b：

```text
GPU 0: [1,5,1024](4组sum) × wo_b [1024, 4096] → [1,5,4096] 部分和
GPU 1: [1,5,1024](4组sum) × wo_b [1024, 4096] → [1,5,4096] 部分和
```

两台的 [1,5,4096] 各自只包含 4 组的贡献，加起来才是完整的 8 组结果。然后——

**AllReduce #1：两台把 [1,5,4096] 的部分和相加，得到完整的注意力输出。**

```text
AllReduce 传输量: 1 × 5 × 4096 × 2 bytes(BF16) = 40,960 bytes ≈ 40 KB
```

这是整层 Transformer 的**第一次跨机通信**。

━━━━━━━━━━━━━━━━━━━━

◆ MoE 层怎么切

━━━━━━━━━━━━━━━━━━━━

────────────────────

【Router：不切】

**W_gate [4096, 256]：不切，两台各存一份完整的。**

```text
GPU 0: h [1,5,4096] × W_gate [4096,256] → scores [1,5,256]
GPU 1: h [1,5,4096] × W_gate [4096,256] → scores [1,5,256]
```

两台必须算出完全相同的 256 个分数，才能选出相同的 top-6 专家。如果两台的分数不一样——哪怕差一个 bit——选出的专家就可能不同，后面的结果就对不上了。Router 的权重必须完整且一致。

────────────────────

【专家分配：按专家编号分，不是切专家内部】

TP=2 时，256 个专家分成两组：

- GPU 0 持有专家 0~127
- GPU 1 持有专家 128~255

每个专家内部的 SwiGLU FFN（W1 [4096,2048]、W2 [2048,4096]、W3 [4096,2048]）完整地存在一台机器上。**不是把一个专家的矩阵切成两半分到两台**，而是整个专家放在一台上。

每个 token 的 top-6 专家可能分布在两台上。比如某个 token 的 top-6 选中了专家 [23, 105, 150, 200, 230, 250]：

```text
GPU 0 算专家 23 和 105    （2 个在 0~127 范围内）
GPU 1 算专家 150、200、230、250  （4 个在 128~255 范围内）
```

两台各自算完自己负责的专家，结果乘以路由权重加起来——但这只是各自的部分和。GPU 0 的结果缺了专家 150/200/230/250 的贡献，GPU 1 的结果缺了专家 23/105 的贡献。

**AllReduce #2：两台把 MoE 的部分和 [1,5,4096] 相加，得到完整的 MoE 输出。**

```text
AllReduce 传输量: 1 × 5 × 4096 × 2 bytes(BF16) = 40,960 bytes ≈ 40 KB
```

这是整层 Transformer 的**第二次跨机通信**。

────────────────────

【共享专家：和普通 SwiGLU FFN 一样按 TP 切】

共享专家是一个标准的 SwiGLU FFN（4096→2048→4096），按 Column + Row Parallel 切：

- W1（gate）和 W3（up）：按列切（Column Parallel），每台算 1024 维中间结果
- W2（down）：按行切（Row Parallel），输出做 AllReduce

共享专家的 AllReduce 和路由专家的 AllReduce 可以合并——两者的输出都是 [1,5,4096] 的部分和，先本地相加再做一次 AllReduce 就行。所以 MoE 层总共只需要**一次 AllReduce**，不是两次。

💡 翻译成人话：路由专家的切法像分工——你负责编号 0~127 的专家，我负责 128~255 的。而不是两个人合作切一个专家。谁手下有被选中的专家，谁就算，算完结果加起来。

━━━━━━━━━━━━━━━━━━━━

◆ mHC（流形超连接）在 TP 下怎么处理

━━━━━━━━━━━━━━━━━━━━

169 期讲过，mHC 维护 4 份 hidden state 副本，维度是 `[1, 5, 4, 4096]`。

TP 不切 4 这个维度。4 是副本数（hc_mult），不是一个可以按头或按专家切分的并行维度——4 份副本之间有动态混合（Sinkhorn 矩阵），切了就没法独立算混合权重。

TP 切的是 4096（hidden_size）这个维度吗？也不是。hc_pre 和 hc_post 的 Sinkhorn 混合矩阵操作的是 `[4]` 和 `[4,4]` 维度的标量权重，和 hidden_size 的大小无关。**每台机器在 AllReduce 之后拿到完整的 `h [1,5,4096]`，然后独立计算 Sinkhorn 混合权重，独立执行 hc_pre/hc_post 的合并和展开。**

mHC 不引入额外通信。

━━━━━━━━━━━━━━━━━━━━

◆ 一层 Transformer 的完整 TP 流程

━━━━━━━━━━━━━━━━━━━━

沿用 169 期的维度变化风格，把 TP=2 的完整流程标出来：

```text
GPU 0:                                    GPU 1:
h [1,5,4096] 完整                          h [1,5,4096] 完整（开始时两台一样）

━━ hc_pre + RMSNorm ━━
h [1,5,4096] 完整                          h [1,5,4096] 完整

━━ Q 低秩投影 ━━
W_q_down：完整 [4096,1024]                  W_q_down：完整 [4096,1024]
q_latent [1,5,1024] 完整                   q_latent [1,5,1024] 完整
W_q_up：前半 [1024,16384]                   W_q_up：后半 [1024,16384]
q [1,5,32,512] ← 头 0~31                   q [1,5,32,512] ← 头 32~63

━━ KV 投影 ━━
W_kv：完整 [4096,512]                       W_kv：完整 [4096,512]
kv [1,5,1,512] 完整                        kv [1,5,1,512] 完整

━━ Attention ━━
32 头 × 1 份 KV → [1,5,32,512]            32 头 × 1 份 KV → [1,5,32,512]
（独立计算，不通信）                          （独立计算，不通信）

━━ O 分组低秩 ━━
4 组 → wo_a → [1,5,1024]                  4 组 → wo_a → [1,5,1024]
wo_b → [1,5,4096] (部分和)                 wo_b → [1,5,4096] (部分和)
          ←──── AllReduce #1 ────→

━━ hc_post + hc_pre + RMSNorm ━━
h [1,5,4096] 完整                          h [1,5,4096] 完整

━━ MoE ━━
Router：完整，选出相同的 top-6              Router：完整，选出相同的 top-6
专家 0~127 在此                            专家 128~255 在此
各算各的专家 → 部分和                       各算各的专家 → 部分和
共享专家按 TP 切 → 部分和                   共享专家按 TP 切 → 部分和
          ←──── AllReduce #2 ────→
moe_out [1,5,4096] 完整                    moe_out [1,5,4096] 完整
```

**每层 2 次 AllReduce，每次约 40 KB（BF16）。43 层共 86 次 AllReduce。**

◆ 性能分析——200Gbps 够不够

━━━━━━━━━━━━━━━━━━━━

先算总通信量：

```text
43 层 × 2 次/层 × 40 KB/次 = 3,440 KB ≈ 3.4 MB
```

200Gbps = 25 GB/s 的线速带宽。3.4 MB 的数据，纯带宽时间：

```text
3.4 MB / 25 GB/s = 0.14 ms
```

如果瓶颈是带宽，那 0.14 毫秒就能传完所有通信数据。**但实际瓶颈不是带宽，是延迟。**

每次 AllReduce 有一个固定的启动开销（startup cost）：建立连接、同步信号、DMA 传输初始化。不管传 40 KB 还是 40 bytes，这个开销都要付。对于 CX7 走 RDMA over Ethernet（RoCE），单次 AllReduce 延迟大约 10~30 微秒。

```text
86 次 × 20 μs（取中间值）= 1,720 μs ≈ 1.72 ms
```

对比同机 NVLink 的延迟（约 1~2 微秒）：

```text
NVLink: 86 次 × 1.5 μs ≈ 0.13 ms
CX7:    86 次 × 20  μs ≈ 1.72 ms
```

**CX7 的延迟惩罚约 13 倍。** 注意不是带宽差——200Gbps vs NVLink 900Gbps 只差 4.5 倍。但 40 KB 的数据量太小了，根本吃不满带宽，瓶颈完全在延迟上。

这就是 171 期说的"NVLink 900Gbps vs CX7 200Gbps"的**真正影响**：对于 V4 Flash 这种每次只传 40 KB 的模型，带宽差距无所谓，**每次 AllReduce 的启动开销才是真凶**。如果一次传 70 MB 而不是 40 KB，带宽才会成为瓶颈——但 Flash 的 hidden_size 只有 4096，一次 AllReduce 的数据量就是这么小。

1.72 ms 多不多？要看推理总时间。V4 Flash 在 DGX Spark 双机上跑大约 20 tok/s，即每个 token 50 ms。通信延迟占 50 ms 的约 3.4%——不算大头，但也不可忽略。真正的时间大头是计算本身（MoE 专家的矩阵乘法）和显存带宽（加载专家权重）。

━━━━━━━━━━━━━━━━━━━━

◆ 为什么不用更简单的流水线并行

━━━━━━━━━━━━━━━━━━━━

张量并行（TP）是层内并行——同一层的矩阵乘法切成两半，两台同时算。还有一种更简单的并行方式：**流水线并行（Pipeline Parallel，PP）**——按层切，前 21 层放 GPU 0，后 22 层放 GPU 1（以 Flash 43 层为例）。

PP 听起来简单得多：不需要 Column/Row Parallel 原语，不需要每层 AllReduce，只需要一层算完把 hidden state 传给下一台。但工业界的推理框架（vLLM、SGLang、TensorRT-LLM）对 PP 的支持要么没有要么半成品，DeepSeek 官方推理代码也只实现了 TP。

原因很直接：**工业机不需要 PP。** 数据中心里 GPU 之间有 NVLink（900Gbps，延迟 1~2 μs），TP 的每层 2 次 AllReduce 开销极小。而 PP 有一个致命问题——**气泡**：GPU 0 算完一层才能传给 GPU 1，GPU 1 等着的时间就是浪费。TP 两台同时算，PP 一台算另一台等。对于有 NVLink 的机器来说，TP 全面碾压 PP。

但反过来想：**如果机器之间的连接很慢呢？**

TP 每层 2 次 AllReduce，43 层就是 86 次通信。每次通信都有启动延迟——在 CX7（200Gbps RDMA）上是 10~30 μs，如果换成普通千兆网卡就是几百微秒，USB 更别提了。86 次乘上去，延迟直接爆炸。

PP 就不一样了。43 层只需要在层 21 和层 22 之间传一次 hidden state（`[1, seq_len, 4096]`），而且是一大块数据一次性传——能吃满带宽，启动延迟只付一次。**通信次数少、每次传输量大**，这恰好是慢速网络的最优模式。

讽刺的是：**PP 最适合的场景恰恰是普通人家里的硬件——几台破电脑用网线或 USB 连起来。** 但工业界不需要 PP，所以没人把推理框架的 PP 做好。家用玩家最需要的功能，工业界没动力做。

DGX Spark 双机的 CX7 200Gbps 刚好卡在中间：比 NVLink 慢 4.5 倍，但比家用网络快两个数量级。**快到 TP 还能用（1.72 ms 延迟可以接受），慢到能感觉到延迟惩罚（比 NVLink 慢 13 倍）。** 如果连接再慢一个数量级——比如两台机器之间只有千兆网卡——TP 就不可用了，PP 才是正确选择。但到那时候，你会发现没有推理框架能帮你。

━━━━━━━━━━━━━━━━━━━━

◆ 为什么 MoE 救了 DGX Spark

━━━━━━━━━━━━━━━━━━━━

换个角度想：如果 V4 是 280B 参数的 dense 模型（不是 MoE），会怎样？

**第一，单机放不下。** 280B 参数用 FP4 存约 140GB。单机 128GB 放不下一半的权重——连 TP=2 都跑不起来（每台需要至少 70GB 权重 + KV cache + 激活值 + 系统开销）。实际上 V4 Flash 280B FP4 的权重约 158GB，TP=2 时每台 79GB，加上 KV cache 和运行时开销刚好塞进 128GB——已经很紧了。

**第二，计算量爆炸。** Dense 280B 意味着每个 token 要走完所有 280B 参数。MoE 280B 只激活约 13B。计算量差 20 倍。

**第三，通信量其实差不多。** AllReduce 传的是 hidden_size 维度的结果向量，和模型总参数量无关。Dense 模型也是传 [1,5,4096]，和 MoE 一样。但 dense 模型的计算时间远长于 MoE，通信占比更低——所以通信不是 dense 模型的主要问题，**存储才是**。

MoE 的真正优势在于：**用大量参数换知识密度，但推理时只激活一小部分。** 256 个专家各自 3 个矩阵（W1、W2、W3），参数量巨大，但每个 token 只过 6+1=7 个专家。这种"参数大但计算小"的特点天生适合存储够但算力有限的部署场景——比如两台 DGX Spark。

如果 V4 是 dense 的，280B 参数全激活，每个 token 的推理时间大约是 MoE 版的 20 倍——从 50 ms/token 变成 1 秒/token，只有 1 tok/s。那就真没法用了。

💡 翻译成人话：MoE 就像一个有 256 个专科医生的大医院。每个病人只需要看 6 个科室，不用 256 个科室全跑一遍。医院虽然大（280B 参数），但每个病人的看诊时间很短。两台 Spark 刚好能装下整个医院的编制（权重），而每个病人的看诊速度还可以接受。

━━━━━━━━━━━━━━━━━━━━

◆ 总结

━━━━━━━━━━━━━━━━━━━━

把这篇的要点收拢。

**张量并行的核心模式**：Column Parallel（按输出维度切 W，结果拼接）和 Row Parallel（按输入维度切 W，结果相加）配对使用。AllReduce 只在 Row Parallel 的输出端发生。一层 Transformer 两次 AllReduce——注意力输出一次，MoE 输出一次。

**哪些矩阵不切**：
- W_q_down [4096, 1024]——瓶颈维度太小，切了反而多通信
- W_kv [4096, 512]——MLA 只有 1 份 KV，没法切
- W_gate [4096, 256]——Router 两台必须选出相同的 top-6

**MLA 的 TP 代价**：KV cache 无法按 TP 切分。MHA 的 64 份 KV 在 TP=2 时每台存 32 份，MLA 的 1 份 KV 在 TP=2 时每台还是存完整的 1 份。KV cache 绝对大小已经很小（512 维），但不享受 TP 的存储红利。

**MoE 的 TP**：按专家编号分（GPU 0 持有 0~127，GPU 1 持有 128~255），不是切专家内部。每个 token 的 top-6 专家可能分布在两台上，各自算完后 AllReduce 合并。

**真正的瓶颈不是带宽而是延迟**：每次 AllReduce 只传 40 KB，200Gbps 的带宽远远吃不满。但 86 次 AllReduce（43 层）的启动开销累计 1.72 ms，是同机 NVLink 的约 13 倍。

169 期最后讲了一个观察：好的架构设计是"需要时才介入"——CSA/HCA 在短输入时静静退化，不加额外开销。张量并行的 AllReduce 也是同样的设计哲学：**只在 Row Parallel 输出、必须合并结果的地方才通信。** 能本地算的绝不跨机，能延迟通信的绝不提前通信。Column Parallel 的输出自然地切好了、Row Parallel 的输入自然地对上了，中间零通信。通信点压到最少——每层只有 2 次。这不是偶然的巧合，是 Megatron-LM 在 2019 年就定下的设计范式：**让计算图的天然结构来决定通信边界，而不是在任意位置插入同步点。**
