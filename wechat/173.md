◆ 从"水平切成两半"说起

━━━━━━━━━━━━━━━━━━━━

169 期（ https://mp.weixin.qq.com/s/mnbaXhQmQjDAyGD1dDdq5Q ）走完了 DeepSeek V4-Pro 的整条前向传播路径——从 token ID 到 embedding，到 61 层 Transformer，到 softmax 采样出"北京"。每一步标了维度变化，一个矩阵乘法都不漏。

171 期（ https://mp.weixin.qq.com/s/JPxR8rHSb1Pko3vq2ukY3A ）在两台 DGX Spark 上跑通了 V4 Flash 280B。但关于张量并行只说了一句话：**"每一层的矩阵乘法被水平切成两半，算完后通过 NCCL AllReduce 把结果同步。"**

具体怎么切？哪些矩阵切了、哪些没切？AllReduce 在哪发生、传多少数据？这篇展开讲。

**前置知识**：你需要熟悉 169 期的前向传播流程（每一步的维度变化）和 171 期的硬件架构（两台 DGX Spark 通过 CX7 200Gbps 直连）。本篇沿着 169 期的前向传播路径，逐步标注每个矩阵乘法在 TP=2（Tensor Parallel，张量并行度 2）时是怎么分配到两台机器上的。

◆ 张量并行的基本原理：一个矩阵乘法怎么分到两台机器上

━━━━━━━━━━━━━━━━━━━━

先从最简单的场景开始：一个矩阵乘法 `Y = X × W`，怎么分到两台机器上算？

两种切法。

────────────────────

【Column Parallel：按列切 W，结果拼接】

把权重矩阵 W 按列切成两半，每台各存一半：

```text
完整: X [1, 5, 7168] × W [7168, 6144] → Y [1, 5, 6144]

TP=2 切法:
  GPU 0: X [1, 5, 7168] × W₀ [7168, 3072] → Y₀ [1, 5, 3072]  ← 前 3072 列
  GPU 1: X [1, 5, 7168] × W₁ [7168, 3072] → Y₁ [1, 5, 3072]  ← 后 3072 列
```

两台各算出结果的一半。要得到完整的 Y，把两半**拼接**（concatenate）起来：`Y = [Y₀ | Y₁]`。

注意：两台的输入 X 必须是完整的、一模一样的。

────────────────────

【Row Parallel：按行切 W，结果相加】

把权重矩阵 W 按行切成两半，对应地，输入也要切成两半：

```text
完整: X [1, 5, 6144] × W [6144, 7168] → Y [1, 5, 7168]

TP=2 切法:
  GPU 0: X₀ [1, 5, 3072] × W₀ [3072, 7168] → Y₀ [1, 5, 7168]  ← 前 3072 行
  GPU 1: X₁ [1, 5, 3072] × W₁ [3072, 7168] → Y₁ [1, 5, 7168]  ← 后 3072 行
```

两台各算出一个**部分和**，大小和完整的 Y 一样。要得到完整的 Y，把两个部分和**相加**：`Y = Y₀ + Y₁`。

这个"相加"不是本地操作——两台机器各自只有自己的部分和，必须通过网络把两个部分和合并。**这就是 AllReduce。**

────────────────────

【为什么要配对使用】

实际使用中，Column Parallel 和 Row Parallel 总是成对出现。原因很简单：

- Column Parallel 的输出是"切好的"——GPU 0 有前半，GPU 1 有后半
- Row Parallel 的输入恰好需要"切好的"——GPU 0 用前半行，GPU 1 用后半行

**前一层 Column Parallel 的输出，正好是后一层 Row Parallel 的输入。** 中间不需要任何通信。

**AllReduce 只在 Row Parallel 的输出端出现**——因为 Row Parallel 的两个部分和必须加起来才能得到正确结果。Column Parallel 的输出不需要通信，各自保留各自的半边就行。

一对 Column + Row 的完整流程：

```text
X [1,5,7168]（两台相同）
  ↓ Column Parallel（按列切 W，无通信）
GPU 0: [1,5,3072]          GPU 1: [1,5,3072]
  ↓ Row Parallel（按行切 W，无通信）
GPU 0: [1,5,7168] 部分和    GPU 1: [1,5,7168] 部分和
  ↓ AllReduce（跨机通信，两个部分和相加）
GPU 0: [1,5,7168] 完整      GPU 1: [1,5,7168] 完整
```

💡 翻译成人话：Column Parallel 是"你算左半边我算右半边，各自留着"。Row Parallel 是"你算上半部分我算下半部分，然后加起来"。加起来这一步需要通信——这就是 AllReduce。整个张量并行里，通信只发生在"加起来"这一步。

◆ Attention 层怎么切

━━━━━━━━━━━━━━━━━━━━

以 169 期的前向传播路径为骨架，逐步标注 TP=2 时的切分方式。

────────────────────

【Q 低秩投影】

169 期里的 Q 投影分四步：`h → W_q_down → RMSNorm → W_q_up → 逐头RMSNorm`。

**W_q_down [7168, 1536]：不切，两台各存一份完整的。**

```text
GPU 0: h [1,5,7168] × W_q_down [7168,1536] → q_latent [1,5,1536]  完整
GPU 1: h [1,5,7168] × W_q_down [7168,1536] → q_latent [1,5,1536]  完整
```

为什么不切？1536 维已经是压缩过的瓶颈维度了。如果按列切成 768+768，后面 RMSNorm 要在完整的 1536 维上算均方根——切了就没法独立做 RMSNorm，还得通信一次把两半拼起来。切 1536 维省不了多少计算，反而多一次通信。不划算。

**W_q_up [1536, 65536]：按列切（Column Parallel）。**

65536 = 128 头 × 512 维/头。TP=2 按头切——GPU 0 算头 0~63，GPU 1 算头 64~127。

```text
GPU 0: q_latent [1,5,1536] × W_q_up₀ [1536,32768] → q [1,5,64,512]  ← 头 0~63
GPU 1: q_latent [1,5,1536] × W_q_up₁ [1536,32768] → q [1,5,64,512]  ← 头 64~127
```

每台只存半张 W_q_up，只算 64 个头的 Q。这是标准的 Column Parallel——按头切输出维度。

逐头 RMSNorm 在各自的 64 头上独立算，不需要通信。

────────────────────

【KV 投影】

**W_kv [7168, 512]：不切，两台各存一份完整的。**

```text
GPU 0: h [1,5,7168] × W_kv [7168,512] → kv [1,5,1,512]  完整
GPU 1: h [1,5,7168] × W_kv [7168,512] → kv [1,5,1,512]  完整
```

这是 MLA 在张量并行下的核心约束。169 期讲过，V4 的 KV 只有 **1 份**（`num_key_value_heads=1`），不像 MHA 有 128 份可以按头切。1 份 KV 没法切——切了就不完整了。

MHA 的 KV 有 128 份，TP=2 时每台存 64 份，KV cache 减半。**MLA 的 KV cache 不随 TP 缩减**——TP=1 时每个 token 存 512 维，TP=2 时每台还是存 512 维，总共存了两份。这是 MLA 选择"1 份 KV"的设计代价：推理时 KV cache 绝对大小已经很小（512 vs MHA 的 32768），但不能被 TP 进一步切分。

────────────────────

【注意力计算】

每台用自己的 64 个 Q 头 × 完整的 1 份 KV → 各自得到 64 头的注意力输出。

```text
GPU 0: q[1,5,64,512] × kv[1,5,1,512] → attn_out [1,5,64,512]  ← 头 0~63
GPU 1: q[1,5,64,512] × kv[1,5,1,512] → attn_out [1,5,64,512]  ← 头 64~127
```

两台完全独立计算，不需要通信。每台算的是不同的头——头 0~63 和头 64~127 之间本来就没有交互。

────────────────────

【O 分组低秩投影——第一次 AllReduce】

169 期讲过，128 头分 16 组，每组 8 头。TP=2 时，每台 8 组（64 头 / 8 头每组 = 8 组）。

```text
GPU 0: 8 组 × wo_a [4096, 1024] → sum → [1,5,1024]
GPU 1: 8 组 × wo_a [4096, 1024] → sum → [1,5,1024]
```

wo_a 是 Column Parallel——每台持有自己负责的 8 组的 wo_a 权重。

接下来的 wo_b 是 Row Parallel。169 期里完整的操作是"16 组各自通过 wo_a 压到 1024 维，16 组加起来得到一个 1024 维向量，再通过共享的 wo_b [1024, 7168] 展开回 7168 维"。

在 TP=2 下，16 组的 sum 被拆成两步：GPU 0 把自己的 8 组加起来得到一个部分和 [1,5,1024]，GPU 1 也是。**两台各自持有完整的 wo_b [1024, 7168]**，各自把自己的部分和 [1,5,1024] 过 wo_b：

```text
GPU 0: [1,5,1024](8组sum) × wo_b [1024, 7168] → [1,5,7168] 部分和
GPU 1: [1,5,1024](8组sum) × wo_b [1024, 7168] → [1,5,7168] 部分和
```

两台的 [1,5,7168] 各自只包含 8 组专家的贡献，加起来才是完整的 16 组结果。然后——

**AllReduce #1：两台把 [1,5,7168] 的部分和相加，得到完整的注意力输出。**

```text
AllReduce 传输量: 1 × 5 × 7168 × 2 bytes(BF16) = 71,680 bytes ≈ 70 KB
```

这是整层 Transformer 的**第一次跨机通信**。

◆ MoE 层怎么切

━━━━━━━━━━━━━━━━━━━━

────────────────────

【Router：不切】

**W_gate [7168, 384]：不切，两台各存一份完整的。**

```text
GPU 0: h [1,5,7168] × W_gate [7168,384] → scores [1,5,384]
GPU 1: h [1,5,7168] × W_gate [7168,384] → scores [1,5,384]
```

两台必须算出完全相同的 384 个分数，才能选出相同的 top-6 专家。如果两台的分数不一样——哪怕差一个 bit——选出的专家就可能不同，后面的结果就对不上了。Router 的权重必须完整且一致。

────────────────────

【专家分配：按专家编号分，不是切专家内部】

TP=2 时，384 个专家分成两组：

- GPU 0 持有专家 0~191
- GPU 1 持有专家 192~383

每个专家内部的 SwiGLU FFN（W1 [7168,3072]、W2 [3072,7168]、W3 [7168,3072]）完整地存在一台机器上。**不是把一个专家的矩阵切成两半分到两台**，而是整个专家放在一台上。

每个 token 的 top-6 专家可能分布在两台上。比如某个 token 的 top-6 选中了专家 [23, 105, 200, 301, 350, 380]：

```text
GPU 0 算专家 23 和 105    （2 个在 0~191 范围内）
GPU 1 算专家 200、301、350、380  （4 个在 192~383 范围内）
```

两台各自算完自己负责的专家，结果乘以路由权重加起来——但这只是各自的部分和。GPU 0 的结果缺了专家 200/301/350/380 的贡献，GPU 1 的结果缺了专家 23/105 的贡献。

**AllReduce #2：两台把 MoE 的部分和 [1,5,7168] 相加，得到完整的 MoE 输出。**

```text
AllReduce 传输量: 1 × 5 × 7168 × 2 bytes(BF16) = 71,680 bytes ≈ 70 KB
```

这是整层 Transformer 的**第二次跨机通信**。

────────────────────

【共享专家：和普通 SwiGLU FFN 一样按 TP 切】

共享专家是一个标准的 SwiGLU FFN（7168→3072→7168），按 Column + Row Parallel 切：

- W1（gate）和 W3（up）：按列切（Column Parallel），每台算 1536 维中间结果
- W2（down）：按行切（Row Parallel），输出做 AllReduce

共享专家的 AllReduce 和路由专家的 AllReduce 可以合并——两者的输出都是 [1,5,7168] 的部分和，先本地相加再做一次 AllReduce 就行。所以 MoE 层总共只需要**一次 AllReduce**，不是两次。

💡 翻译成人话：路由专家的切法像分工——你负责编号 0~191 的专家，我负责 192~383 的。而不是两个人合作切一个专家。谁手下有被选中的专家，谁就算，算完结果加起来。

━━━━━━━━━━━━━━━━━━━━

◆ mHC（流形超连接）在 TP 下怎么处理

━━━━━━━━━━━━━━━━━━━━

169 期讲过，mHC 维护 4 份 hidden state 副本，维度是 `[1, 5, 4, 7168]`。

TP 不切 4 这个维度。4 是副本数（hc_mult），不是一个可以按头或按专家切分的并行维度——4 份副本之间有动态混合（Sinkhorn 矩阵），切了就没法独立算混合权重。

TP 切的是 7168（hidden_size）这个维度吗？也不是。hc_pre 和 hc_post 的 Sinkhorn 混合矩阵操作的是 `[4]` 和 `[4,4]` 维度的标量权重，和 hidden_size 的大小无关。**每台机器在 AllReduce 之后拿到完整的 `h [1,5,7168]`，然后独立计算 Sinkhorn 混合权重，独立执行 hc_pre/hc_post 的合并和展开。**

mHC 不引入额外通信。

━━━━━━━━━━━━━━━━━━━━

◆ 一层 Transformer 的完整 TP 流程

━━━━━━━━━━━━━━━━━━━━

沿用 169 期的维度变化风格，把 TP=2 的完整流程标出来：

```text
GPU 0:                                    GPU 1:
h [1,5,7168] 完整                          h [1,5,7168] 完整（开始时两台一样）

━━ hc_pre + RMSNorm ━━
h [1,5,7168] 完整                          h [1,5,7168] 完整

━━ Q 低秩投影 ━━
W_q_down：完整 [7168,1536]                  W_q_down：完整 [7168,1536]
q_latent [1,5,1536] 完整                   q_latent [1,5,1536] 完整
W_q_up：前半 [1536,32768]                   W_q_up：后半 [1536,32768]
q [1,5,64,512] ← 头 0~63                   q [1,5,64,512] ← 头 64~127

━━ KV 投影 ━━
W_kv：完整 [7168,512]                       W_kv：完整 [7168,512]
kv [1,5,1,512] 完整                        kv [1,5,1,512] 完整

━━ Attention ━━
64 头 × 1 份 KV → [1,5,64,512]            64 头 × 1 份 KV → [1,5,64,512]
（独立计算，不通信）                          （独立计算，不通信）

━━ O 分组低秩 ━━
8 组 → wo_a → [1,5,1024]                  8 组 → wo_a → [1,5,1024]
wo_b → [1,5,7168] (部分和)                 wo_b → [1,5,7168] (部分和)
          ←──── AllReduce #1 ────→
h [1,5,7168] 完整                          h [1,5,7168] 完整

━━ hc_post + hc_pre + RMSNorm ━━
h [1,5,7168] 完整                          h [1,5,7168] 完整

━━ MoE ━━
Router：完整，选出相同的 top-6              Router：完整，选出相同的 top-6
专家 0~191 在此                            专家 192~383 在此
各算各的专家 → 部分和                       各算各的专家 → 部分和
共享专家按 TP 切 → 部分和                   共享专家按 TP 切 → 部分和
          ←──── AllReduce #2 ────→
moe_out [1,5,7168] 完整                    moe_out [1,5,7168] 完整

━━ hc_post ━━
h_hc [1,5,4,7168] 完整                     h_hc [1,5,4,7168] 完整
```

**每层 2 次 AllReduce，每次约 70 KB（BF16）。61 层共 122 次 AllReduce。**

◆ 性能分析——200Gbps 够不够

━━━━━━━━━━━━━━━━━━━━

先算总通信量：

```text
61 层 × 2 次/层 × 70 KB/次 = 8,540 KB ≈ 8.54 MB
```

200Gbps = 25 GB/s 的线速带宽。8.54 MB 的数据，纯带宽时间：

```text
8.54 MB / 25 GB/s = 0.34 ms
```

如果瓶颈是带宽，那 0.34 毫秒就能传完所有通信数据。**但实际瓶颈不是带宽，是延迟。**

每次 AllReduce 有一个固定的启动开销（startup cost）：建立连接、同步信号、DMA 传输初始化。不管传 70 KB 还是 70 bytes，这个开销都要付。对于 CX7 走 RDMA over Ethernet（RoCE），单次 AllReduce 延迟大约 10~30 微秒。

```text
122 次 × 20 μs（取中间值）= 2,440 μs ≈ 2.44 ms
```

对比同机 NVLink 的延迟（约 1~2 微秒）：

```text
NVLink: 122 次 × 1.5 μs ≈ 0.18 ms
CX7:    122 次 × 20  μs ≈ 2.44 ms
```

**CX7 的延迟惩罚约 10~15 倍。** 注意不是带宽差——200Gbps vs NVLink 900Gbps 只差 4.5 倍。但 70 KB 的数据量太小了，根本吃不满带宽，瓶颈完全在延迟上。

这就是 171 期说的"NVLink 900Gbps vs CX7 200Gbps"的**真正影响**：对于 V4 这种每次只传 70 KB 的模型，带宽差距无所谓，**每次 AllReduce 的启动开销才是真凶**。如果一次传 70 MB 而不是 70 KB，带宽才会成为瓶颈——但 V4 的 hidden_size 只有 7168，一次 AllReduce 的数据量就是这么小。

2.44 ms 多不多？要看推理总时间。V4 Flash 在 DGX Spark 双机上跑大约 20 tok/s，即每个 token 50 ms。通信延迟占 50 ms 的约 5%——不算大头，但也不可忽略。真正的时间大头是计算本身（MoE 专家的矩阵乘法）和显存带宽（加载专家权重）。

◆ 为什么 MoE 救了 DGX Spark

━━━━━━━━━━━━━━━━━━━━

换个角度想：如果 V4 是 280B 参数的 dense 模型（不是 MoE），会怎样？

**第一，单机放不下。** 280B 参数用 FP4 存约 140GB。单机 128GB 放不下一半的权重——连 TP=2 都跑不起来（每台需要至少 70GB 权重 + KV cache + 激活值 + 系统开销）。实际上 V4 Flash 280B FP4 的权重约 158GB，TP=2 时每台 79GB，加上 KV cache 和运行时开销刚好塞进 128GB——已经很紧了。

**第二，计算量爆炸。** Dense 280B 意味着每个 token 要走完所有 280B 参数。MoE 280B 只激活约 13B。计算量差 20 倍。

**第三，通信量其实差不多。** AllReduce 传的是 hidden_size 维度的结果向量，和模型总参数量无关。Dense 模型也是传 [1,5,7168]，和 MoE 一样。但 dense 模型的计算时间远长于 MoE，通信占比更低——所以通信不是 dense 模型的主要问题，**存储才是**。

MoE 的真正优势在于：**用大量参数换知识密度，但推理时只激活一小部分。** 384 个专家各自 3 个矩阵（W1、W2、W3），参数量巨大（约 250B），但每个 token 只过 6+1=7 个专家。这种"参数大但计算小"的特点天生适合存储够但算力有限的部署场景——比如两台 DGX Spark。

如果 V4 是 dense 的，280B 参数全激活，每个 token 的推理时间大约是 MoE 版的 20 倍——从 50 ms/token 变成 1 秒/token，只有 1 tok/s。那就真没法用了。

💡 翻译成人话：MoE 就像一个有 384 个专科医生的大医院。每个病人只需要看 6 个科室，不用 384 个科室全跑一遍。医院虽然大（280B 参数），但每个病人的看诊时间很短（13B 计算量）。两台 Spark 刚好能装下整个医院的编制（权重），而每个病人的看诊速度还可以接受。

━━━━━━━━━━━━━━━━━━━━

◆ 总结

━━━━━━━━━━━━━━━━━━━━

把这篇的要点收拢。

**张量并行的核心模式**：Column Parallel（按输出维度切 W，结果拼接）和 Row Parallel（按输入维度切 W，结果相加）配对使用。AllReduce 只在 Row Parallel 的输出端发生。一层 Transformer 两次 AllReduce——注意力输出一次，MoE 输出一次。

**哪些矩阵不切**：
- W_q_down [7168, 1536]——瓶颈维度太小，切了反而多通信
- W_kv [7168, 512]——MLA 只有 1 份 KV，没法切
- W_gate [7168, 384]——Router 两台必须选出相同的 top-6

**MLA 的 TP 代价**：KV cache 无法按 TP 切分。MHA 的 128 份 KV 在 TP=2 时每台存 64 份，MLA 的 1 份 KV 在 TP=2 时每台还是存完整的 1 份。KV cache 绝对大小已经很小（512 维 vs MHA 的 32768 维），但不享受 TP 的存储红利。

**MoE 的 TP**：按专家编号分（GPU 0 持有 0~191，GPU 1 持有 192~383），不是切专家内部。每个 token 的 top-6 专家可能分布在两台上，各自算完后 AllReduce 合并。

**真正的瓶颈不是带宽而是延迟**：每次 AllReduce 只传 70 KB，200Gbps 的带宽远远吃不满。但 122 次 AllReduce 的启动开销累计 2.44 ms，是同机 NVLink 的 10~15 倍。

169 期最后讲了一个观察：好的架构设计是"需要时才介入"——CSA/HCA 在短输入时静静退化，不加额外开销。张量并行的 AllReduce 也是同样的设计哲学：**只在 Row Parallel 输出、必须合并结果的地方才通信。** 能本地算的绝不跨机，能延迟通信的绝不提前通信。Column Parallel 的输出自然地切好了、Row Parallel 的输入自然地对上了，中间零通信。通信点压到最少——每层只有 2 次。这不是偶然的巧合，是 Megatron-LM 在 2019 年就定下的设计范式：**让计算图的天然结构来决定通信边界，而不是在任意位置插入同步点。**
