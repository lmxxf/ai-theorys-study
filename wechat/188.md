【Qwen3.6】27B 打赢 397B——阿里这一代到底改了什么

━━━━━━━━━━━━━━━━━━━━

◆ 先把事情说清楚

━━━━━━━━━━━━━━━━━━━━

2026 年 4 月，阿里发了 Qwen 3.6。

不是一个模型，是一个家族：

| 型号 | 参数量 | 类型 | 开源 | 备注 |
|---|---|---|---|---|
| Qwen3.6-27B | 27B | Dense | 是 | 质量担当，24GB VRAM 能跑 |
| Qwen3.6-35B-A3B | 35B 总 / 3B 激活 | MoE | 是 | 速度担当，16GB VRAM 能跑 |
| Qwen3.6-Plus | 未公开 | API only | 否 | 云服务中端 |
| Qwen3.6-Max-Preview | ~1T（推测） | MoE，API only | 否 | 旗舰，对标前沿 |

开源只放了两个小的。大的全是 API——你可以用，但看不到权重。这个策略后面第七节会讲。

核心卖点：**Qwen3.6-27B 在多个编程和推理 benchmark 上打赢了上一代旗舰 Qwen 3.5-397B。** 一个 27B 的 dense 模型，跑在单张 24GB 显卡上，干掉了一个 400B 级别的 MoE。参数量差 15 倍。

这事到底是怎么发生的？

━━━━━━━━━━━━━━━━━━━━

◆ 架构：真正的大跳是 3 → 3.5，3.6 是训练迭代

━━━━━━━━━━━━━━━━━━━━

拿到 Qwen 3.6 的第一件事是看 config.json。打开一看——`model_type: "qwen3_5_moe"`。

等等，3.6 的模型类型是 3.5？

是的。Qwen 3.5 和 3.6 **共享完全一样的架构定义**，代码层面没有任何区别。3.6 不是架构迭代，是训练迭代——更好的数据、更长的训练、更强的 RL。

真正的架构大跳发生在 Qwen 3 到 3.5 之间。列个表看得清楚：

| 规格 | Qwen 3 (30B-A3B) | Qwen 3.5/3.6 (35B-A3B) |
|---|---|---|
| 注意力 | 标准 GQA（全部层） | 混合：3:1 Gated DeltaNet + Full Attention |
| 层数 | 48 | 40 |
| hidden dim | 2048 | 2048 |
| Q/KV 头 | 32/4 | 16/2 |
| head dim | 128 | 256 |
| 专家数 | 128 总/8 路由 | 256 总/8 路由 + 1 共享 |
| 词表 | 151,936 | 248,320 |
| 最大上下文 | 40,960 | 262,144（YaRN 扩展到 ~1M） |
| 视觉 | 无 | 有（ViT） |
| MTP | 无 | 有 |

变化很猛。注意力机制从纯标准 GQA 换成了混合线性注意力；层数反而少了 8 层；head dim 翻倍；专家数翻倍还加了共享专家；词表扩了 60%；上下文从 4 万直接拉到 26 万原生、YaRN 外推到百万级。

**Qwen 3.6 比 3.5 强，不是因为换了架构，是因为训练数据和 RL 更好。** 架构这张牌在 3.5 就打完了，3.6 打的是训练牌。

下面重点讲架构里最关键的那个改动——混合注意力。

127 期（127 期《线性注意力》）已经介绍过 Mamba-3 和线性注意力家族，Gated DeltaNet 也提过。这一期不重复讲 DeltaNet 的数学推导（下一期专门讲），只讲"为什么用它、怎么用它"。

━━━━━━━━━━━━━━━━━━━━

◆ 75% 线性注意力 + 25% 标准注意力：为什么是 3:1

━━━━━━━━━━━━━━━━━━━━

打开 Qwen 3.5/3.6 的 config.json，有一行：

```json
"full_attention_interval": 4
```

意思是：**每 4 层里，3 层用 Gated DeltaNet（线性注意力），1 层用标准 GQA（二次注意力）。**

为什么要混？

Gated DeltaNet 是线性复杂度的——处理 n 个 token 的计算量和 n 成正比，不是 n²。这意味着上下文从 4K 拉到 256K，计算量只涨 64 倍（线性），而不是 4096 倍（二次方）。省算力，省显存，省钱。127 期讲过这个家族的基本原理：用一个固定大小的状态矩阵（类似"压缩记忆"）代替存储所有历史 token 的 KV cache。新 token 进来时更新这个状态，不需要回头看所有历史。

但 127 期也讲过 SSM/线性注意力家族的致命弱点：**精确检索和状态追踪能力差。**

什么叫精确检索？"第 17 段第 3 句话说了什么"——你需要从压缩记忆里精确提取某个位置的信息。线性注意力的压缩记忆是有损的，越早的信息越容易被后来的信息覆盖。

什么叫状态追踪？"这段代码里变量 x 被赋值了几次，最后的值是多少"——你需要跟踪一个实体在上下文中的状态变化轨迹。这要求模型精确地知道"第 5 次出现的 x"和"第 3 次出现的 x"是同一个变量但值不同。

标准 Transformer 的 O(n²) 注意力做这两件事很自然——每个 token 都能直接看到所有历史 token，想查谁查谁。代价是 n² 的计算量。

Qwen 3.5/3.6 的混合策略：**75% 的层用线性注意力省钱，25% 的层用标准注意力补短板。** 3 层 DeltaNet 处理大部分"理解语义、建立关联"的工作，1 层标准 GQA 负责"精确检索"和"状态追踪"。

代价是：不再是纯线性了。25% 的层仍然是 O(n²) 的，整体复杂度比纯 Transformer 低很多，但不是严格线性。

换来的收益：**KV cache 只需要给 25% 的标准注意力层存。** 跑 1M 上下文时，KV cache 大约 25GB——纯 Transformer 要 4 倍。这也是为什么一个 27B 模型能原生支持 262K 上下文、外推到 1M 的关键：纯 Transformer 的 27B 做不到这个上下文长度，KV cache 就把显存撑爆了。

这个 3:1 的比例和 Qwen 3.5 完全一样，3.6 没有调。

◆ 27B Dense：为什么这个型号火了

━━━━━━━━━━━━━━━━━━━━

先看数字。

| Benchmark | Qwen3.6-27B | Qwen3.5-397B | Opus 4.6 | 备注 |
|---|---|---|---|---|
| SWE-bench Verified | 77.2 | — | 80.8 | 编程 |
| SWE-bench Pro | 53.5 | 50.9 | — | 编程进阶 |
| SkillsBench | 48.2 | 30.0 | — | Agent 能力 |
| AIME 2026 | 94.1 | — | — | 数学竞赛 |
| MMLU-Pro | 86.2 | — | — | 综合知识 |
| GPQA Diamond | 87.8 | — | — | 研究生级科学 |
| Terminal-Bench 2.0 | 59.3 | — | — | 和 Opus 4.5 打平 |

SWE-bench Pro 上 27B 直接赢了自家 397B。SkillsBench（Agent 任务）48.2 vs 30.0，更是碾压。这在 LocalLLaMA 社区炸开了锅——本地一张 24GB 的 4090/5090 跑个 27B，编程水平接近 Opus 4.6（SWE-bench Verified 差 3.6 个点），还要什么 API？

但要给这些数字打个折。

Qwen 官方的 benchmark 用的是**自己的 agent scaffold**（即模型外面套的那层"工具调用 + 循环重试 + 错误修复"的框架）。scaffold 本身也贡献分数——同一个模型换不同的 scaffold，SWE-bench 分数能差 10 个点以上。第三方用标准化 scaffold 做的独立验证还比较少，目前社区反馈的实际体感和官方数字之间有一定 gap。

所以合理的判断是：Qwen3.6-27B 确实很强，编程和推理能力在 27B 这个量级里断档领先。但"27B 达到 Opus 4.6 水平"需要加定语——**在特定 benchmark 上、用 Qwen 自己的 scaffold**。

### 27B 和 35B-A3B 的定位差异

这两个开源模型不是大小号的关系，是两种不同的取舍。

**27B Dense：质量担当。** 所有 27B 参数每个 token 都参与计算，推理质量高，但算力需求也高。需要 24GB VRAM（4-bit 量化后刚好塞进一张 4090/5090）。适合对质量要求高、batch size 小、不介意速度的场景——比如本地编程助手。

**35B-A3B MoE：速度担当。** 总参数 35B 但每个 token 只激活 3B，计算量只有 27B 的约 1/9。16GB VRAM 就能跑，推理速度比 27B 快 3-5 倍。适合需要高吞吐、低延迟的场景——比如 Agent 调用（一个任务要调模型几十次）。代价是同等上下文下推理质量不如 27B。

◆ MTP：训练时多猜几个 token

━━━━━━━━━━━━━━━━━━━━

Qwen 3.5/3.6 的架构表里有一项"MTP"——Multi-Token Prediction，多 token 预测。

标准 Transformer 训练时，每一步只预测下一个 token。看了"今天天气"，预测"真"。看了"今天天气真"，预测"好"。一次一个。

MTP 让模型每一步同时预测后面好几个 token。看了"今天天气"，同时预测"真"、"好"、"啊"。

**这是训练技巧，不是推理技巧。** 训练时多预测几个 token 的目的是给模型更强的梯度信号——不只是"下一个词对不对"，而是"接下来一串词的趋势对不对"。这让模型学到更好的内部表征（internal representation）。推理时模型仍然可以一个一个 token 生成，但因为训练阶段学到了更强的表征，每一步的预测质量更高。

不过 MTP 在推理时也有妙用：配合 **speculative decoding**（投机解码）加速。

投机解码的原理：用 MTP 的预测头一次性"猜"好几个 token，然后用主模型验证这几个猜测。猜对了就全接受（一步走好几个 token），猜错了就回退到猜错的位置重新生成。Qwen 3.6 官方数据：全局接受率 72.53%——10 次猜测里大约 7 次猜对，吞吐量提升 +24.2%。

35B-A3B 配合 MTP 在 12GB VRAM 上能跑到 **80 tok/s**。对比一下：你在 ChatGPT 网页版打字的阅读速度大概是 5-10 tok/s。80 tok/s 已经是"模型在等你看完"的速度了。

💡 翻译成人话：训练时多做几道题（MTP），学到的解题思路更好。考试时先快速把答案蒙上去（speculative decoding），蒙对了省时间，蒙错了再仔细做。蒙对率 73%。

◆ Max-Preview：阿里的旗舰有多强

━━━━━━━━━━━━━━━━━━━━

Qwen3.6-Max-Preview 是这一代的天花板。推测参数量在 1T 级别的 MoE，API only，看不到架构细节。

阿里号称它拿了六项 SOTA（State of the Art，各 benchmark 的最高分），集中在编程和 Agent 场景：SWE-Bench Pro、Terminal-Bench 2.0、SkillsBench、TAU-bench、Agentic Coding 等。

但也要看看它输在哪儿：

- **SWE-bench Verified**——Opus 4.7 领先（87.6 vs Max-Preview 的约 82-85）。这是编程 benchmark 里公信力最高的一个，第三方验证最多。
- **GPQA Diamond**——Gemini 3.1 Pro 按着打（94.3 vs Max-Preview 不到 90）。这个 benchmark 测的是研究生级别的物理/化学/生物知识，考的是知识密度和精确推理。

画面比较清楚：**Max-Preview 在编程和 Agent 场景（需要工具调用、多步推理、错误修复）很强，纯知识密度和精确推理还差前沿一档。** 编程/Agent 能力更依赖训练策略（RL、scaffold 协同优化），知识密度更依赖模型规模和预训练数据量——阿里在前者跟得很紧，后者还有差距。

◆ 这一代的战略意图

━━━━━━━━━━━━━━━━━━━━

把 Qwen 3.6 的产品线摊开，策略很清晰：**开源小模型 + API 大模型的双轨打法。**

小模型（27B Dense、35B-A3B MoE）全部开源，抢两个市场：
- **开源社区**——LocalLLaMA 的人看到"27B 接近 Opus 4.6 编程水平"就坐不住了，马上开始跑量化版、做微调、接各种框架。社区活跃度直接转化为生态粘性。
- **本地部署**——企业不想把代码发到云上，24GB 显存跑个 27B 做内部编程助手，数据不出门。这个需求在中国市场特别大。

大模型（Plus、Max-Preview）全部 API only，抢云服务市场：
- 想用旗舰？来阿里云调 API。
- 定价比 Opus/Gemini 低一截，用量大的客户算账算得过来。

混合注意力架构在这个策略里扮演关键角色：**它让小模型能在 262K 上下文下跑得起来。** 纯 Transformer 的 27B 做不到这个上下文长度——KV cache 太大，24GB 显存装不下。75% 的线性注意力层不需要 KV cache，等于把"长上下文"这个原本只有大模型才能玩的特性下放到了 27B 这个量级。一个能在单卡上跑、同时支持 262K 上下文的 27B 编程模型——对本地部署场景来说，这就是杀手级组合。

阿里在架构创新上跟得紧。从 Qwen 3 的纯 Transformer 到 3.5/3.6 的混合线性注意力，跟上了 Mamba-3/DeltaNet 这一波混合架构的浪潮，而且是第一批把混合注意力做进大规模商用模型并开源的厂商之一。训练策略上也不含糊——同一个架构，3.6 靠训练数据和 RL 就把 benchmark 拉高了一截，说明他们的数据飞轮和 RL pipeline 在快速迭代。

下一期预告：**Gated DeltaNet 的数学原理。** 127 期（127 期《线性注意力》）讲了 Mamba-3 用复数旋转编码时序，DeltaNet 用"记忆覆写"更新状态矩阵——它们到底有什么本质区别？为什么 Qwen 选了 DeltaNet 而不是 Mamba-3？下一期把公式拆开。
