【后 RAG 时代的 Agent 记忆】压缩、索引、遗忘——2026 年的新选择

◆ 记忆问题的本质不是"装不下"

━━━━━━━━━━━━━━━━━━━━

第 156 期我们测了 MiroFish 多智能体推演项目，后台被追问最多的就是那个 ZEP_API_KEY——"这是啥？能换本地的吗？"

这个问题背后藏着一个更深的问题：**Agent 的记忆到底在解决什么问题？**

很多人的第一反应是：上下文窗口不够长。Gemini 都 200 万 token 了，把历史全塞进去不就完了？

不行。问题的本质不是"装不下"，而是"找不到"。

Transformer 的注意力机制是 O(n^2) 的——上下文越长，每个 token 分到的注意力越稀薄。你把 10 万条对话历史全塞进 prompt，大模型不会变成博闻强识的学者，它会变成一个在图书馆里迷路的人，手里拿着一支越来越暗的手电筒。

这就是"注意力稀释"（attention dilution）。窗口再大，有效信息的密度在下降。

所以，所有记忆方案本质上都在做三件事：

- **压缩**：把 10 万 token 的历史变成 1000 token 的摘要
- **索引**：知道去哪里找到需要的那条记忆
- **遗忘**：知道什么该扔掉，不让噪声淹没信号

这三个操作的不同取舍，决定了每家方案的性格。

在看新方案之前，先回到原点——大多数人接触 Agent 记忆的起点是 RAG。

◆ 基线：RAG 做对了什么，又缺了什么

━━━━━━━━━━━━━━━━━━━━

RAG（Retrieval-Augmented Generation）是最经典的外挂记忆方案，流程很直觉：

1. 把文档切成几百 token 的小块（chunking）
2. 每个块用 embedding 模型算出一个向量，存进向量数据库（Chroma、Pinecone、Milvus……）
3. 用户提问时，把问题也算成向量，用余弦相似度找到最相关的几个块
4. 把这几个块塞进 prompt，让大模型基于检索结果回答

这套流程解决了"索引"问题——至少不用把所有东西都塞进上下文了，只捞最相关的。

但 RAG 在另外两个维度上几乎是零分：

**没有压缩。** 检索回来的是原文块，该多长还多长。10 个相关块可能就占掉 5000 token，而里面真正有用的可能就两句话。

**没有遗忘。** 向量库是只增不删的。你三年前说"我喜欢苹果"，今天说"我对苹果过敏"，两条都躺在库里，检索时可能都被捞出来。RAG 不知道哪条过期了——它没有时间概念，也没有冲突检测。

**索引只有一个维度。** 余弦相似度只能回答"什么东西长得像"，不能回答"什么东西跟什么东西有关系"。你问"张三的合作伙伴是谁"，如果库里没有一个块同时提到张三和他的合作伙伴，纯向量检索就抓瞎了。

RAG 是 2023 年的正确答案。但到了 2026 年，Agent 需要的记忆能力远超"检索相关文档"——它需要能处理矛盾、能追踪时间线、能沿着关系链推理。

下面的每个方案，都是在 RAG 的某个短板上做升级。

但在看具体方案之前，先补两个前置知识。后面几个方案都重度依赖知识图谱，不讲清楚来龙去脉，后面全是空中楼阁。

━━━━━━━━━━━━━━━━━━━━

◆ 前置知识一：图数据库——比 RAG 更古老的技术

━━━━━━━━━━━━━━━━━━━━

其实"用图结构存关系"这件事，比 RAG 还早得多。

RAG 用的向量数据库存的是**点**——每条记忆是高维空间里的一个坐标，检索靠的是"谁离我最近"。

图数据库存的是**点和线**——点是实体（人、公司、事件），线是关系（投资、竞争、隶属）。数据结构是三元组：（张三，投资了，公司A）、（公司A，竞争对手，公司B）。

本质区别：

- **向量检索回答"什么东西长得像"**——语义相似度，适合模糊匹配
- **图谱检索回答"什么东西跟什么东西有关系"**——沿着边遍历，适合结构化推理

向量检索只能做单跳——问题进去，最相似的结果出来。图谱检索能做多跳——从 A 跳到 B，再跳到 C，把整条链路返回。打个比方：向量库是百度搜索（输入关键词，返回最相关的网页），图数据库是百度百科右侧的关系卡片（点进"张三"，能看到他的公司、合作伙伴、竞争对手，再点进去又是一张新的关系网）。

这个领域最老牌的选手是 **Neo4j**，2007 年发布，查询用一种叫 Cypher 的语言：

MATCH (a:Person)-[:INVESTED]->(b:Company) WHERE a.name = '张三' RETURN b

意思是：找到张三投资的所有公司。要多跳——"张三投资的公司的竞争对手"——再接一层 MATCH 就行。Neo4j 在社交网络、推荐系统、风控领域用了快二十年，技术很成熟。

但它有一个致命门槛：**图谱要人工建。** 实体怎么定义、关系怎么分类、三元组怎么抽取——全靠开发者自己写规则、写正则、写 ETL 管道。数据量一大，维护成本爆炸。所以 Neo4j 一直是大厂专属玩具，普通开发者碰都不想碰。

◆ 前置知识二：LLM 让知识图谱平民化了

━━━━━━━━━━━━━━━━━━━━

大模型改变了这个局面。

以前三元组要人工抽取，现在不用了。2026 年的标准做法是：**让 LLM 自己读文档，输出三元组。** 你喂进去一段话"张三在 2024 年投资了公司 A"，LLM 输出（张三，投资，公司A，时间=2024）。本质上就是把非结构化文本翻译成结构化图谱——翻译员是大模型本身。

这意味着以前只有大厂才玩得起的知识图谱，现在普通开发者也能用了。后面的 Zep、Cognee 都是基于这个能力构建的。

换句话说：**RAG 是一个无脑系统——切块、embedding、检索，全程不需要理解内容。知识图谱是一个依赖 LLM 理解力的系统——建图靠理解，检索靠推理。** 能力上限更高，但每一步都多了一个"LLM 可能犯错"的风险敞口。2026 年的共识是两者混合用：向量做语义粗筛（广度），图谱做关系精排（深度）。

搞清楚这个区别，下面的方案就好理解了。

━━━━━━━━━━━━━━━━━━━━

◆ Mem0：用会幻觉的裁判管记忆的生死

━━━━━━━━━━━━━━━━━━━━

GitHub 48K stars，YC 孵化，2400 万美元 A 轮。目前生态最好的记忆层，没有之一。

────────────────────

Mem0 的核心不是存，是**冲突解决**。

你对 Agent 说"我喜欢苹果"，一个月后说"我对苹果过敏了"。传统 RAG 会把两条都存进向量库，下次检索时两条都捞出来，大模型当场精分。

Mem0 的做法是：每当新记忆进来，先用向量检索找到最相似的历史记忆，然后通过 tool call 让 LLM 做裁判——判定这条新记忆应该 ADD（新增）、UPDATE（覆盖旧的）、DELETE（删掉旧的）、还是 NOOP（什么都不做）。

这个设计解决了记忆一致性问题，但要点出一个事实：**裁判本身是个会幻觉的大模型。** 它可能把"我不再吃苹果了"误判为 NOOP，也可能把"我今天吃了苹果派"误判为 UPDATE。没有确定性保障——记忆的正确性取决于 LLM 那一瞬间的判断力。

2026 年 4 月，Mem0 发了 token-efficient memory 算法，单次检索从 25000+ token 降到 7000 以下。这是在"压缩"维度上的一次大幅进步。

────────────────────

Mem0 是开源 + 商业双轨。开源版可以自己部署，支持基本的向量记忆和冲突解决——上面说的 ADD/UPDATE/DELETE/NOOP 机制，开源版就有。但图记忆（Graph Memory，实体关系、多跳查询）锁在商业版每月 249 美元的 Pro 层——从 19 美元的 Basic 到 249 美元的 Pro，中间没有过渡。社区最大的吐槽就是这个跳跃：我需要图谱能力，但我买不起。

**适合：** 快速验证、轻量 Agent、个人助理。和 LangChain/CrewAI 绑得很深，接入成本最低。

━━━━━━━━━━━━━━━━━━━━

◆ Zep / Graphiti：给记忆加上时间轴

━━━━━━━━━━━━━━━━━━━━

先说清楚关系：**Zep 是商业平台，Graphiti 是它开源出来的核心引擎。** 类似 Elasticsearch 和 Lucene 的关系——Graphiti 是开源的时序知识图谱引擎（GitHub 20K stars），Zep 是基于它的云服务，2026 年转型叫"Context Engineering Platform"。Zep 的社区版已经停止维护，团队全力押注 Graphiti 开源。所以如果你要用，直接看 Graphiti。

────────────────────

Graphiti 的核心差异化是**时序知识图谱**。

普通知识图谱存的是"A 认识 B"这种静态关系。Graphiti 存的是"A 在 2024 年 3 月投资了 B，2025 年 8 月撤资了，2026 年 1 月起诉了 B"。每条边都带时间戳和有效期。

这意味着你可以问 Agent："A 和 B 现在是什么关系？"——它不会把三年前的投资关系当成现状。时间维度上的"遗忘"是自动的：过期的边权重自然衰减。

在 LongMemEval 基准测试上，Zep 平台拿到 63.8%，Mem0 是 49.0%。差距主要来自需要时间线推理的题目。

────────────────────

**适合：** 需要时间线推理的场景——金融分析、多方博弈推演、合同生命周期管理。

━━━━━━━━━━━━━━━━━━━━

◆ Cognee：不只是检索，是推理

━━━━━━━━━━━━━━━━━━━━

GitHub 12K stars，750 万美元种子轮。

────────────────────

Cognee 的管道叫 ECL——Extract（抽取实体和关系）、Cognify（构建认知图谱）、Load（加载到图数据库）。

它和其他家最大的区别在检索端：**多跳图谱检索**。

Mem0 和普通 RAG 做的是单跳——你问一个问题，它找最相关的记忆，返回。Cognee 做的是：找到 A 节点后，沿着图的边跳到 B，再跳到 C，把整条推理链路返回。

举个例子："张三的投资人的竞争对手是谁？"——这需要跳两次：张三 -> 投资人 -> 竞争对手。普通向量检索做不了这件事，必须靠图结构。

默认图数据库用的是嵌入式的 Kuzu（不用单独部署），数据量大了可以换 Neo4j。

────────────────────

**适合：** 需要复杂关系推理的场景——人物关系网分析、供应链溯源、多方利益博弈。

━━━━━━━━━━━━━━━━━━━━

◆ Evermind / EverOS：跑分最猛的本地派

━━━━━━━━━━━━━━━━━━━━

完全本地部署，数据不出内网。

────────────────────

Evermind 设计了三阶段记忆生命周期，对标人类认知心理学：

- **Episodic Trace**（情景痕迹）：原始对话记录，相当于你的短期记忆
- **Semantic Consolidation**（语义巩固）：提取核心语义，类似睡眠时的记忆整理
- **Reconstructive Recollection**（重构性回忆）：检索时不是原样返回，而是根据当前语境重新组织

跑分上，LoCoMo 基准 93%、LongMemEval 83%——在本地部署方案里遥遥领先。

────────────────────

**适合：** 数据不能出内网的场景——政企、金融机构内部系统。

━━━━━━━━━━━━━━━━━━━━

◆ 泼一盆冷水：知识图谱的软肋

━━━━━━━━━━━━━━━━━━━━

上面这些方案看着都很美，但它们共享一个根本问题。

前面说过，2026 年的知识图谱是 LLM 自动建的。这意味着图谱的质量天花板就是 LLM 的理解力——抽错了实体、漏了关系、搞混了时间线，后面的检索和推理全跟着错。

而且资料量一大，还有一个更隐蔽的问题：**跨文档冲突。** A 文档说"苹果发布了新产品"，B 文档说"苹果富含维生素C"——同一个词，一个是公司一个是水果，LLM 抽出来可能挂到同一个节点上。再比如 A 文档说"张三是 X 公司 CEO"，B 文档说"张三是 Y 公司 CEO"——可能是同名不同人，也可能同一个人换了工作。这就是经典的**实体消歧**问题。

RAG 反而没有这个烦恼——它不理解内容，切块就存，检索时把原文甩给大模型，让大模型自己去判断"这个苹果是哪个苹果"。无脑有无脑的好处。

知识图谱越大，这类冲突越多，目前没有完美解法。建图靠 LLM，冲突解决靠 LLM，记忆筛选还是靠 LLM——**每一个关键判断都外包给了同一个会幻觉的大脑。**

━━━━━━━━━━━━━━━━━━━━

◆ 回到三个操作

━━━━━━━━━━━━━━━━━━━━

拉回到开头的框架：

**压缩**——Mem0 的 token-efficient memory、Evermind 的 Semantic Consolidation，都是在把冗长的历史压成精华。

**索引**——向量数据库擅长语义相似性检索（"找最像的"），图数据库擅长关系跳转（"找关联的"）。2026 年的共识是混合架构：向量做粗筛，图谱做精排。

**遗忘**——Zep 用时间衰减自动淘汰过期关系，Mem0 用 LLM 判断 UPDATE/DELETE，Evermind 用语义巩固过滤噪声。

这些工具都在解决"Agent 怎么记住过去"的问题。但真正难的不是存储，是**判断**——什么值得记，什么该忘，什么时候该改写旧记忆。

这个判断力，目前全部外包给了 LLM 自己。

用一个会幻觉的大脑来管理自己的记忆——这本身就是 2026 年最大的未解问题。
