◆ 哪里不对：INT8 测试的意外发现

━━━━━━━━━━━━━━━━━━━━

事情的转折点出现在 NLA（Natural Language Autoencoder，自然语言自编码器）实验中。

181 期我们在 DGX Spark 上做 NLA 70B AV 模型的量化部署。原模型是 BF16 的 Llama-3.3-70B，我们先做了 FP8 量化，后来又做了 INT8 量化。INT8 量化的动机很简单：compressed-tensors 格式（NeuralMagic 的量化标准，vLLM 原生支持）在加载 INT8 权重时，走的是一条"INT8 反量化到 BF16 + BF16 matmul"的路径。也就是说——INT8 权重存着省空间，但实际计算时先转成 BF16 再做矩阵乘法。

等等。如果 INT8 也要反量化到 BF16 再算，那跟 FP4 的处境不是一样吗？都是"硬件不支持低精度计算，只能软件反量化到 BF16"？

但 INT8 矩阵乘法在 GPU 上可是有悠久历史的东西。`mma.sync.aligned.m16n8k32.s32.s8.s8.s32` 这条 INT8 tensor core 指令，从 sm_80（Ampere，A100 那一代）就有了。sm_121 没道理不支持。

于是我们写了一个最小验证：

```python
@triton.jit
def int8_matmul_kernel(a_ptr, b_ptr, c_ptr, M, N, K, ...):
    acc = tl.zeros((BLOCK_M, BLOCK_N), dtype=tl.int32)
    for k in range(0, K, BLOCK_K):
        a = tl.load(a_ptr + ...)
        b = tl.load(b_ptr + ...)
        acc += tl.dot(a, b)  # int8 x int8 -> int32
    tl.store(c_ptr + ..., acc)
```

`tl.dot(int8, int8)` —— 让 Triton 编译器自己决定用什么指令。如果 sm_121 上 INT8 tensor core 不存在或者行为不对，结果应该会出错。

结果：

```
Max diff: 0.0
Triton INT8 tensor core matmul on sm_121: PERFECT
```

**max_diff = 0.0。和 PyTorch 的 float 参考结果逐元素完全一致。**

INT8 tensor core matmul 在 sm_121 上完美运行。不是"差不多对"，是零误差。

这就引出了一个问题：**如果 INT8 tensor core 完整支持，那之前"FP8 残废"的结论还对吗？** INT8 比 FP8 更老的指令都能用，FP8 反而不能用？这不合理。

于是我们去查了文档。不是博客文章，不是 Reddit 帖子，不是 GitHub issue 里的猜测——去查了 NVIDIA 的 PTX ISA 9.2 官方文档。

━━━━━━━━━━━━━━━━━━━━

◆ PTX ISA 9.2：748 页文档里的真相

━━━━━━━━━━━━━━━━━━━━

PTX（Parallel Thread Execution）是 NVIDIA GPU 的中间指令语言——介于 CUDA C++ 和 GPU 硬件机器码之间。PTX ISA（Instruction Set Architecture）文档定义了每条 PTX 指令的语法、语义、以及**在哪些 SM 架构上可用**。

PTX ISA 9.2 版本于 2026 年 4 月发布，与 CUDA Toolkit 13.2 配套，748 页。我们重点看了 `mma`（Matrix Multiply-Accumulate）指令族的架构支持表。

────────────────────

【INT8 mma】

指令格式：`mma.sync.aligned.m16n8k32.s32.s8.s8.s32`

含义：16x8x32 的矩阵块，输入 int8（s8），输出 int32（s32），同步执行。

架构要求：**sm_80+**（Ampere 及以后所有架构）。

sm_121 是 sm_80 之后的架构。INT8 mma 支持，毫无悬念。这和我们的 Triton 测试结果吻合。

────────────────────

【FP8 mma】

指令格式：`mma.sync.aligned.m16n8k32.f32.e4m3.e4m3.f32`

含义：输入 FP8 e4m3（4 位指数 3 位尾数），输出 FP32。

架构要求：**sm_89+**（Ada Lovelace 及以后）。

sm_121 > sm_89。**FP8 mma 完整支持。**

等一下。sm_89 是 RTX 4090 那一代。RTX 4090 都能跑的 FP8 tensor core 指令，DGX Spark 没道理不能跑。

我们之前说"FP8 也残的"，到底是哪来的结论？回头翻 179 期原文：

> "SM121 上的 FP8 矩阵乘法指令同样是残缺的——还是老黄的刀法，消费级卡的 FP8 tensor core 支持不完整。"

这句话没有引用任何文档或测试数据。是我们基于"FP4 不行所以 FP8 大概也不行"的推测。推测错了。

────────────────────

【FP4 mma】

这里更关键。FP4 在 PTX ISA 里对应的数据类型是 `.e2m1`（2 位指数 1 位尾数），还有相关的 FP6 类型 `.e3m2`（3 位指数 2 位尾数）和 `.e2m3`（2 位指数 3 位尾数）。

指令格式：`mma.sync.aligned.m64n32k64.f32.e2m1.e2m1.f32`（以及 block_scale 变体）

架构要求：文档里列了两种 SM 子类型。

先解释 sm_120a 和 sm_120f 的区别。NVIDIA 从 Blackwell 开始引入了 SM 子类型标记：
- **sm_120a**（accelerated）：支持该架构的全部加速指令
- **sm_120f**（full）：支持该架构的全部基础指令，部分加速指令从特定 PTX 版本起支持

PTX ISA 9.2 文档明确写道：

- `.e2m1` / `.e3m2` / `.e2m3` 类型的 mma 指令：sm_120a 支持；sm_120f 从 PTX 8.8 起支持
- `block_scale` 修饰符（每块独立 scale factor，MXFP4 需要的特性）：sm_120a 支持；sm_120f 也支持

DGX Spark 的 sm_121 属于 sm_120 家族。Triton 编译器针对 sm_121 生成代码时，会走 sm_120f 的代码路径。PTX 8.8 对应 CUDA 12.8——我们用的 CUDA Toolkit 是 13.0+，远超这个版本要求。

**结论：sm_121 的 FP4 mma 硬件指令存在。**

不是"不存在"，不是"残废"，是**完整存在**。INT8、FP8、FP4 的 tensor core mma 指令在 sm_121 上全部可用。

────────────────────

把三条指令的架构支持放在一起看：

| 指令类型 | PTX 指令 | 最低架构要求 | sm_121 支持? |
|---------|---------|------------|-------------|
| INT8 mma | `mma.sync...s8.s8` | sm_80+ | 支持 |
| FP8 mma | `mma.sync...e4m3.e4m3` | sm_89+ | 支持 |
| FP4 mma | `mma.sync...e2m1.e2m1` | sm_120f (PTX 8.8+) | 支持 |
| FP4 block_scale | `mma.sync...block_scale` | sm_120a / sm_120f | 支持 |

全部支持。三期文章里的"FP4 硬件指令不存在""FP8 也残的""只有 BF16 完整可用"——每一条都是错的。

◆ 那之前为什么不行？

━━━━━━━━━━━━━━━━━━━━

硬件指令都在，那 176/178/179 三期里我们遇到的所有问题——FP4 反量化到 BF16、FP8 走 BF16 matmul、Marlin 静默算错——到底是怎么回事？

**全是软件栈的问题。** 逐个拆解：

────────────────────

【问题 1：torch._scaled_mm 不支持 INT8】

PyTorch 提供了一个 `torch._scaled_mm` API，专门做带 scale factor 的低精度矩阵乘法。但这个 API 的参数签名只接受 `float8_e4m3fn` 和 `float8_e5m2` 两种输入类型——**它完全没有封装 INT8 mma 指令**。

这不是 INT8 tensor core 不存在，是 PyTorch 没有暴露这条代码路径。硬件从 sm_80 就支持 INT8 mma，PyTorch 选择只封装 float8 路径。

结果就是：如果你通过 PyTorch 高层 API 做 INT8 矩阵乘法，它不会走 INT8 tensor core——要么走 cuBLAS 的通用路径，要么让上层框架自己做反量化。compressed-tensors 选择了后者：INT8 反量化到 BF16，再用 BF16 matmul。

────────────────────

【问题 2：compressed-tensors 选择了反量化路径】

NeuralMagic 的 compressed-tensors 库是 vLLM 处理量化模型的标准格式层。它在加载 INT8 权重时，走的是"INT8 反量化 -> BF16 -> BF16 matmul"的路径。这个设计可能是为了跨架构兼容——BF16 matmul 在所有现代 GPU 上都能用，不需要检测目标 GPU 的 INT8 tensor core 支持情况。

代价是白白浪费了 INT8 tensor core 的算力。INT8 tensor core 做矩阵乘法比 BF16 更快（INT8 精度更低，硬件每周期能处理更多元素），但 compressed-tensors 把 INT8 数据膨胀成 BF16 再算——等于花了 INT8 的存储空间，拿了 BF16 的计算速度，两头没占到便宜。

我们的 Triton INT8 matmul 绕过了 compressed-tensors 的反量化路径——直接 `tl.dot(int8, int8) -> int32`，然后在 kernel 内部乘 scale。结果完美。

────────────────────

【问题 3：Marlin FP4 kernel 的数据布局错误】

178 期讲过，Marlin 的 FP4 MoE kernel 在 sm_120+ 上静默算错——不报错，但输出是垃圾。我们当时把这归因于"sm_120+ 没有 FP4 硬件指令"。

现在知道了：sm_121 上 FP4 mma 指令完整存在。Marlin 算错不是因为指令不存在，是因为**Marlin 的 FP4 数据布局解释在 sm_120+ 上和数据中心卡不一致**。

具体来说：Marlin 用了 sm_90/sm_100 上特定的 FP4 MMA 变体（涉及 `tcgen05` 调度），这些变体在 sm_120 上不存在。sm_120 有自己的 FP4 mma 路径（`mma.sync` + block_scale），但数据在寄存器里的布局方式不同。Marlin 按照 sm_100 的布局去解释 sm_120 上的寄存器数据——同一个 byte 里的两个 FP4 值被分配到了错误的矩阵位置——所以结果是垃圾。

这是软件 bug（对目标架构的数据布局理解错误），不是硬件缺失。

────────────────────

【问题 4：我们自己的 FP8 量化路径不匹配】

179 期做 FP4 优化时，我们提到"SM121 上的 FP8 矩阵乘法指令同样是残缺的"。这个判断的来源是什么？是我们用 `torch._scaled_mm` 做 FP8 matmul 时遇到了问题。

但问题不是 FP8 tensor core 不存在——是我们的 FP8 权重格式（per-block scale）和 `torch._scaled_mm` 要求的格式（per-tensor 或 per-row scale）不兼容。`torch._scaled_mm` 期望输入是 `(float8_tensor, float8_tensor, scale_a, scale_b)` 的形式，scale 是标量或一维向量。我们的 FP8 block quantization 用的是二维 scale 矩阵（每个 128x128 块一个 scale），和 API 不匹配，调用直接报错或出错误结果。

我们把这解读成了"FP8 tensor core 残废"。实际上是 API 格式不匹配。

────────────────────

总结一下：

| 我们之前的结论 | 真实原因 |
|-------------|---------|
| "FP4 硬件指令不存在" | Marlin 的数据布局 bug + 标准 Triton 库用了 sm_100 专属 PTX |
| "FP8 也残的" | torch._scaled_mm 的 scale 格式不兼容我们的量化方案 |
| "只有 BF16 完整可用" | BF16 确实可用，但 INT8/FP8/FP4 也可用 |
| "消费级卡被物理阉割" | tcgen05/TMA 确实被砍了，但 FP4/FP8 mma 指令没砍 |

◆ sm_100 vs sm_120 的真正差异（更正版）

━━━━━━━━━━━━━━━━━━━━

之前三期文章里，我们把 sm_100 和 sm_120 的差异描述为"砍了哪些指令"——FP4 砍了、FP8 砍了半刀。这个描述不准确。

更准确的说法是：**sm_100 和 sm_120 用了不同的指令族来实现同样的功能。**

────────────────────

【sm_100（数据中心）：tcgen05 指令族】

`tcgen05` 是第 5 代 tensor core 指令生成器（Tensor Core GENeration 05）。它不是单条指令，而是一套硬件调度机制：

- 使用 **TMEM**（Tensor Memory，256KB 专用张量内存）作为操作数来源
- 通过 **TMA**（Tensor Memory Accelerator）硬件单元自动搬运数据到 TMEM
- 支持 TMA multicast（一次广播数据到多个 SM）

`tcgen05` 的设计目标是最大化训练和大规模推理的吞吐——TMEM + TMA 让数据搬运完全由硬件自动化，SM 只管算。

────────────────────

【sm_120（消费级）：mma.sync.aligned 指令族】

sm_120 用的是 `mma.sync.aligned`——和 Ampere（sm_80）、Hopper（sm_90）同族的指令。这套指令的特点：

- 使用**寄存器**作为操作数来源（不需要 TMEM）
- 数据搬运走传统的 load/store 或 LDGSTS（不需要 TMA）
- 不支持 TMA multicast

关键的一点：**sm_120 的 mma.sync 指令族覆盖了 INT8、FP8、FP4 所有精度的矩阵乘法。** 它不是 sm_80/sm_89 那些旧版 mma.sync 的简单延续——sm_120 新增了 `.e2m1`（FP4）、`.e4m3`（FP8）、`block_scale` 等新类型和修饰符。

────────────────────

【差异在哪？】

| 特性 | sm_100（数据中心） | sm_120（消费级） |
|-----|-------------------|-----------------|
| 指令族 | tcgen05 | mma.sync.aligned |
| 操作数来源 | TMEM（256KB 专用） | 寄存器 |
| 数据搬运 | TMA 硬件自动 | 手动 load/store |
| TMA multicast | 支持 | 不支持 |
| INT8 mma | 支持 | 支持 |
| FP8 mma | 支持 | 支持 |
| FP4 mma + block_scale | 支持 | 支持 |

**FP4/FP8/INT8 的 tensor core 矩阵乘法，两边都支持。** 差异在于数据搬运和调度的效率——TMEM + TMA 是专用硬件做搬运，寄存器 + 手动 load 是通用路径做搬运。前者峰值吞吐更高，后者灵活但效率低一些。

这些差异影响的是**峰值性能**，不是**功能有无**。

打个比方：sm_100 是有高速传送带的自动化工厂（TMEM + TMA 自动送料），sm_120 是靠工人手动送料的工厂（寄存器 + load/store）。两个工厂的机器都能加工同样的零件（FP4/FP8/INT8 矩阵乘法），只是送料效率不同。我们之前说的"消费级卡被阉割了 FP4 计算"，相当于说"手动送料的工厂不能加工零件"——机器明明在那，只是你没往里面塞料。

◆ 教训

━━━━━━━━━━━━━━━━━━━━

**1. "torch API 不支持" 不等于 "硬件不支持"**

PyTorch 是高层封装，不代表硬件能力的全部。`torch._scaled_mm` 没有 INT8 路径，不等于 sm_121 没有 INT8 tensor core。`torch._scaled_mm` 的 FP8 路径和我们的 scale 格式不兼容，不等于 sm_121 的 FP8 tensor core "残废"。

这个错误的推理链条是：
1. 用 PyTorch API 测试 -> 不行
2. 结论：硬件不支持

正确的做法应该是在步骤 1 和 2 之间加一步：查 PTX ISA 文档确认硬件是否真的不支持。

**2. 查 PTX ISA 文档应该是第一步**

PTX ISA 文档是 NVIDIA GPU 指令集的权威参考。每条指令支持哪些架构、什么 PTX 版本起可用，文档里写得清清楚楚。

我们折腾了三个月——176 期到 182 期，从 CUTLASS 到 Triton 到 Fused Kernel——一直基于"FP4/FP8 硬件不支持"的假设在做软件模拟。如果一开始就查 PTX ISA 9.2，翻到 mma 指令的架构支持表，十分钟就能确认 sm_121 支持 FP4/FP8/INT8 的全部 mma 指令。然后我们的优化方向就完全不同了——不是"怎么在没有硬件指令的情况下用软件模拟"，而是"怎么让 Triton 正确生成这些硬件指令"。

三个月的弯路，本可以省掉。

**3. sm_120 系列是新硬件，软件还在追赶**

DGX Spark 2026 年初上市。PyTorch、Triton、vLLM 对 sm_120 的适配还在进行中。

这意味着"框架不支持"是一个时间窗口问题，不是永久性限制。sm_120 的 INT8/FP8/FP4 硬件指令都在，只是高层软件还没有适配好——`torch._scaled_mm` 没有封装 INT8 路径、compressed-tensors 选择了反量化路径、Marlin 的数据布局没有适配 sm_120 的 mma.sync 变体。

**"软件没跟上"不等于"硬件砍了"。** 这是我们犯的最核心的归因错误。

━━━━━━━━━━━━━━━━━━━━

◆ 接下来可以做什么

━━━━━━━━━━━━━━━━━━━━

既然 sm_121 的 INT8/FP8/FP4 tensor core 全部可用，之前被我们判死刑的一些优化方向现在可以重新考虑了。

**1. Triton INT8 matmul 推理引擎**

我们已经验证了 `tl.dot(int8, int8)` 在 sm_121 上完美运行。NLA AV 模型的推理已经用上了这条路径——INT8 权重直接走 tensor core，不需要反量化到 BF16。

下一步可以把这个方案通用化：写一个不依赖 compressed-tensors 反量化路径的 INT8 推理引擎。对所有 INT8 量化模型都适用，不只是 NLA。

**2. Triton FP8 matmul 替换 PyTorch dequant**

181 期做的 FP8 量化，加载到 vLLM 时走的也是反量化路径。现在知道 FP8 tensor core 可用，可以用 Triton 写一个直接走 FP8 mma 的 matmul kernel——跳过反量化，省掉那 56MB 中间 tensor 的显存读写。

关键是要处理好 per-block scale 的格式——`torch._scaled_mm` 不支持二维 scale，但 Triton kernel 里可以自己做。

**3. Triton FP4 matmul 验证 Marlin 的 bug**

最有意思的方向。Marlin 在 sm_120+ 上算错，之前归因于"硬件不支持"。现在知道 FP4 mma 指令存在，可以用 Triton 写一个最小的 FP4 matmul kernel——`tl.dot` 配合 block_scale——验证 Marlin 的问题是不是真的只是数据布局的软件 bug。

如果 Triton FP4 matmul 在 sm_121 上跑出正确结果，就能彻底证实：Marlin 的 FP4 错误是纯软件问题，不是硬件限制。这对整个 sm_120+ 社区都有价值——目前 vLLM issue #40928 里大家都还在按"硬件不支持"的方向讨论。

━━━━━━━━━━━━━━━━━━━━

◆ 一句话总结

━━━━━━━━━━━━━━━━━━━━

**sm_121 的 INT8/FP8/FP4 tensor core 全部完整存在。之前三期文章说的"FP4 硬件不存在""FP8 残废""只有 BF16 能用"全错了——错的不是硬件，是软件栈没适配好，而我们把软件的问题归咎于硬件。**

查文档应该是第一步，不是折腾三个月后才想到。
