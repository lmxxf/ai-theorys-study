◆ 1. CSA + HCA 混合注意力——V4 最大的创新

━━━━━━━━━━━━━━━━━━━━

V4 的注意力不是一种机制，是两种机制交替排列，再加一层滑动窗口兜底。

**CSA（Compressed Sparse Attention）**：每 4 个 token 的 KV 压缩成 1 个（compress_ratio = 4），然后用 Lightning Indexer 从压缩后的 KV 里选 top-k 做稀疏注意力——只看最相关的一小撮，其余丢弃。

**HCA（Heavily Compressed Attention）**：每 128 个 token 的 KV 压缩成 1 个（compress_ratio = 128），但不做稀疏——压缩后的全部 KV 都看（dense on compressed）。10 万 token 的上下文压完只剩不到 800 个 KV，全看也很便宜。

**滑动窗口**：每一层都附加一个 128 token 的滑动窗口，保证最近的上下文始终能精确关注。

三者的分工很清晰：CSA 负责"从全局历史中精选最相关的细节"，HCA 负责"用极低成本扫一遍全局大意"，滑动窗口负责"最近的上下文一个都不能丢"。

**config.json 的 compress_ratios 字段完美对应**：

```json
// V4-Pro 的 compress_ratios（61 层）
[128, 128, 4, 128, 4, 128, 4, 128, ..., 4, 128, 4, 0]
```

- 128 = HCA 层
- 4 = CSA 层
- 0 = 纯滑动窗口（最后一层）

排列规律：前两层 HCA，之后 CSA 和 HCA 交替，最后一层纯滑动窗口。这个交替排列让模型在每一步都能同时获得"全局概览"和"精选细节"两种信息。

**效果**：推理 FLOPs 只需 V3.2 的 27%，KV cache 只需 10%。

**和 V3.2 DSA 的关键区别**：V3.2 的 DSA 是推理阶段的适配——模型训练时用的是完整注意力，推理时换成稀疏注意力来省算力，本质是"不改脑子，换副眼镜"。V4 的 CSA 是训练阶段就原生稀疏——模型从头到尾都按稀疏方式"看世界"，本质是"脑子就是这么长的"。DSA 在 V4 里退居二线，变成了 CSA 内部的一个子组件。

◆ 2. Lightning Indexer——上篇叫的"FP4 Indexer"

━━━━━━━━━━━━━━━━━━━━

Lightning Indexer 只在 CSA 层（compress_ratio == 4）使用。它的工作就一件事：从压缩后的 KV 里选出最相关的 top-k。

它有自己**独立的压缩器**，和 CSA 主路径的压缩器不共享。这个压缩器的流程是：Hadamard 旋转 -> FP4 量化。Hadamard 旋转是一种正交变换，作用是把向量分量的能量分散均匀，让后续的 FP4 量化误差更小。

config.json 里对应的字段：

```json
"index_n_heads": 64,      // 索引头数量（Pro）
"index_head_dim": 128,    // 每个索引头的维度
"index_topk": 1024        // 选 top-1024
```

QK 路径走 FP4——这是通过 QAT（量化感知训练）实现的，训练时就按 FP4 精度来优化，不是推理时硬砍的。这就是我们在 158 期讲的"FP4 在注意力中的真正用法"。

排序对精度的容忍度远高于精确计算——你只需要知道"谁排前面"，不需要知道分数差 0.001 还是 0.002。这是 Lightning Indexer 能用 FP4 的根本原因。

━━━━━━━━━━━━━━━━━━━━

◆ 3. Hash Routing——和 Engram 有关系但不是 Engram

━━━━━━━━━━━━━━━━━━━━

上篇我们写了一整节 Engram，这是最大的失误。V4 没有使用 Engram。

但 config.json 里确实有一个 Hash 相关的字段：

```json
"num_hash_layers": 3
```

这代表前 3 层 MoE 用的是 Hash 路由（Hash Routing），而不是学习的路由。

代码里 Gate 类有一个 `tid2eid` 查找表：token ID 直接映射到 expert ID，O(1) 查表，不需要过路由网络。PDF 原文是：

> "we replace the dense FFN layers in the initial several Transformer blocks with MoE layers that employ Hash routing (Roller et al., 2021)"

这个"哈希"和 Engram 论文（arXiv 2601.07372）里的哈希记忆表不是同一件事。Engram 是用哈希来存储和检索知识向量；Hash Routing 是用哈希来固定路由——每个 token 永远去同一个专家，不需要学习。两者共享"哈希"这个词，但解决的问题完全不同。

为什么前 3 层要用固定路由？PDF 没有详细解释原因，只引用了 Roller et al., 2021（Hash Layers）。一个合理的猜测是：最底层的路由模式相对稳定，固定路由省计算还避免了负载均衡问题。

━━━━━━━━━━━━━━━━━━━━

◆ 4. mHC（Manifold-Constrained Hyper-Connections）

━━━━━━━━━━━━━━━━━━━━

mHC 在上篇只提了一句，这次展开说。

普通残差连接是 y = x + f(x)，信息流只有一条路。mHC 把残差流展宽 4 倍（config.json: hc_mult = 4），让信息在更宽的通道里传播。

核心约束：映射矩阵 B 被约束在**双随机矩阵流形**上（Birkhoff 多面体）。

什么是双随机矩阵？就是每行之和 = 1，每列之和 = 1 的非负矩阵。这个约束保证了两件事：

1. 信息不会被放大——谱范数 <=1，梯度传播稳定
2. 信息不会被丢失——每行每列都参与，没有"死通道"

怎么把任意矩阵投影到双随机流形上？用 **Sinkhorn-Knopp 迭代**——交替做行归一化和列归一化，20 轮收敛。这个算法 1967 年就有了，数学上能证明一定收敛。

代码里对应 Block 类的 `hc_pre`（进入子层前展宽）和 `hc_post`（出子层后收窄）方法。

额外训练开销只有 6.7%——用不到 7% 的额外算力换训练稳定性，对 1.6T 参数的模型来说是划算的买卖。

◆ 5. Muon 优化器

━━━━━━━━━━━━━━━━━━━━

V4 的大部分参数用 Muon 优化器，不再用 AdamW。只有 embedding、输出头、RMSNorm 这几个特殊层仍用 AdamW。

Muon 的核心操作是 Newton-Schulz 迭代——一种不需要 SVD 的矩阵正交化方法。V4 的实现用了一个 Hybrid 策略：

- 前 8 步：系数 (3.4445, -4.7750, 2.0315)，快速收敛
- 后 2 步：系数 (2, -1.5, 0.5)，精确归一

前 8 步用激进系数快速逼近正交矩阵，最后 2 步用保守系数修正精度。总共 10 步迭代，比标准 Newton-Schulz 更快收敛到高精度。

一个有意思的细节：V4 不用 QK-Clip（注意力 logits 裁剪）。因为 V4 在注意力 QK 之后加了 RMSNorm，RMSNorm 本身就能约束 logits 的尺度，不需要再手动 clip。

◆ 6. MoE 具体参数

━━━━━━━━━━━━━━━━━━━━

上篇给了参数表但没展开。这次补充几个重要细节。

| | V4-Pro | V4-Flash |
|---|---|---|
| 路由专家数 | 384 | 256 |
| 激活专家数 | 6 | 6 |
| 共享专家数 | 1 | 1 |
| 专家中间维度 | 3072 | 2048 |

**激活函数变了**：V3 用的是 Sigmoid 做路由打分，V4 换成了 `sqrt(softplus(x))`。

```python
# config.json
"scoring_func": "sqrtsoftplus"
```

Softplus 是 ln(1 + e^x)，一个平滑版的 ReLU。先 softplus 保证非负，再开根号压缩动态范围。相比 Sigmoid 的优势是：不会饱和（Sigmoid 在两端梯度接近 0），梯度信号更健康。

**SwiGLU Clamping**：专家内部的 SwiGLU 激活值被限制在 [-10, 10]。PDF 详细解释了为什么——超大模型训练时，个别 token 的激活值偶尔会飙到几百，触发梯度爆炸。硬裁剪暴力但有效。

◆ 7. 训练细节——165 期完全没提的新信息

━━━━━━━━━━━━━━━━━━━━

以下内容全部来自技术报告 PDF，上篇没有覆盖。

**预训练数据量**：Pro 实际是 33T token，不是上篇写的 32T+。

**稀疏注意力不是从头用的**：先用 dense attention（完整注意力）训练 1T+ token，模型学会"怎么看"之后，再引入 sparse attention。如果从第一个 token 就用稀疏注意力，模型还没学会分配注意力就被迫做稀疏选择，效果会差。

**序列长度渐进扩展**：4K -> 16K -> 64K -> 1M。不是一上来就训百万上下文——先在短序列上把基础能力学好，再逐步拉长。每次扩展都会在新长度上继续训练一段时间，让模型适应更长的上下文。

━━━━━━━━━━━━━━━━━━━━

◆ 8. Anticipatory Routing——防训练崩溃的黑科技

━━━━━━━━━━━━━━━━━━━━

MoE 训练有一个独特的不稳定性来源：路由网络和专家网络同步更新时，路由决策可能突然大幅变化，导致某些专家突然涌入大量 token、另一些专家瞬间空闲——loss 突然飙升（loss spike）。

Anticipatory Routing 的做法很聪明：

**路由网络用历史参数 theta(t - delta_t) 计算，主干网络用当前参数 theta(t) 计算。**

路由决策总是"慢半拍"——它看到的是几步之前的模型状态，不会被当前步的剧烈更新带偏。主干网络正常更新不受影响。

更聪明的是自动开关机制：系统实时监测 loss。正常训练时不启用（没必要多花开销），一旦检测到 loss spike，自动启用 Anticipatory Routing 来稳住路由，loss 恢复正常后自动关闭。

额外开销约 20%——只在 loss spike 期间付出这个代价，不是全程。

━━━━━━━━━━━━━━━━━━━━

◆ 9. FP4 QAT（量化感知训练）

━━━━━━━━━━━━━━━━━━━━

V4 的 FP4 不是训完之后再量化的（Post-Training Quantization），是训练过程中就按 FP4 精度优化的（Quantization-Aware Training, QAT）。

两个地方用了 FP4 QAT：

1. **MoE 专家权重**：384 个专家的权重全部 FP4，训练时前向传播模拟 FP4 量化误差，反向传播用直通估计器（Straight-Through Estimator）把梯度穿过量化操作传回去。
2. **Lightning Indexer 的 QK 路径**：索引器的 Query 和 Key 也是 FP4 QAT 训练的。

一个关键的数学事实：FP4 (E2M1) 到 FP8 (E4M3) 的反量化是**无损的**。PDF 的解释是：FP8 (E4M3) 比 FP4 (E2M1) 多 2 个指数位，动态范围更大。FP4 量化时会对每个 1x32 的小块（sub-block）计算一个 scale factor，而 FP8 量化时对每个 128x128 的大块计算一个 scale factor。只要同一个大块内各小块 scale factor 的最大值和最小值之比不超过一定阈值，FP8 的大 scale factor 就能完全"吸收"掉 FP4 小块的精细缩放信息。实测中，当前模型权重都满足这个条件。结果是：存储用 FP4 省空间，计算时反量化到 FP8 不丢精度，而且整个 QAT 流程可以完全复用现有的 FP8 训练框架。

◆ 10. Attention Sink

━━━━━━━━━━━━━━━━━━━━

标准注意力的 softmax 输出之和恒等于 1——即使所有 Key 都和当前 Query 不相关，注意力权重也必须分配给某些 token，造成"注意力浪费"。

V4 加了一个学习的 sink logit：在 softmax 的分母里多加一个 exp(z'_h) 项。z'_h 是一个可学习参数，每个注意力头一个。

效果：注意力权重之和可以小于 1，甚至接近 0。当模型判断"当前没有值得关注的历史 token"时，大部分注意力权重会被 sink 吸收，不会强行分配给不相关的 token。

这个设计对长上下文特别重要——百万 token 里真正相关的可能只有几百个，剩下的都是噪音。没有 sink，注意力必须均匀撒在噪音上；有了 sink，注意力可以集中在真正重要的 token 上，剩余的全部"倒进水槽"。

━━━━━━━━━━━━━━━━━━━━

◆ 11. MTP（Multi-Token Prediction）

━━━━━━━━━━━━━━━━━━━━

V4 延续了 V3 的 Multi-Token Prediction 训练策略：

```json
"num_nextn_predict_layers": 1
```

depth = 1，意味着模型在训练时不仅预测下一个 token，还同时预测下下一个 token。额外的预测头共享主模型的表示但有自己的输出层。

这不是 V4 的新东西——V3 就有了。但值得一提的是 V4 没有加大 depth（比如预测未来 3-4 个 token），说明 depth=1 在成本和收益之间已经是一个不错的平衡点。

◆ 整体回顾

━━━━━━━━━━━━━━━━━━━━

把 V4 的技术栈摊开看：

| 组件 | 上篇怎么写的 | 实际是什么 |
|------|------------|-----------|
| 注意力架构 | DSA2（五合一） | CSA + HCA（两种压缩注意力交替排列 + 滑动窗口） |
| 知识检索 | Engram 哈希记忆 | 没有使用 Engram，前 3 层用 Hash Routing |
| 稀疏注意力索引 | FP4 Indexer | Lightning Indexer（FP4 QAT） |
| MoE 推理优化 | Mega MoE | Fine-Grained EP Scheme（PDF 正式名称） |
| 官方列的三大创新 | 五大创新 | 混合注意力 + mHC + Muon |

上篇的大方向没错——V4 确实是在注意力稀疏化、MoE 推理优化、训练稳定性三个方向上同时推进。但名字叫错了，Engram 写多了，创新分类也不对。

速报和精读的差别，就在于"坐下来看代码"这一步。config.json 的 compress_ratios 字段、model.py 的 Gate 类、PDF 第 4 章的训练细节——这些东西不读原文是猜不到的。
