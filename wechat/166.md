昨天 DeepSeek V4 发布，我们第一时间通过开源页面的 README 讨论了一下，基于之前对 NSA 论文和 V3.2 DSA 实现的分析，推演了 V4 的架构，写了一篇速报（第 165 期 https://mp.weixin.qq.com/s/8UnO_NMW_A2zHEywkAND1A ）。后来我们详细阅读了技术文档的 58 页 PDF、config.json 配置文件、model.py 推理代码，发现推演和实际有不少出入。

先说哪里推错了：

1. 我们基于 NSA 论文和 V3.2 的 DSA，推测 V4 用了"DSA2"（四种稀疏注意力组合）——实际叫 CSA + HCA（Compressed Sparse Attention + Heavily Compressed Attention），是一套全新设计，DSA 只是 CSA 内部的子组件。
2. 我们推测 Engram 会被用上，写了一整节——V4 没用 Engram。
3. 我们说"五大创新"——官方列的是三大创新（混合注意力 + mHC + Muon），基础设施优化单独列章节。
4. "Mega MoE / FP4 Indexer"在 PDF 里有自己的正式名字——Fine-Grained EP Scheme 和 Lightning Indexer。

推演的价值在快，代价是粗。这篇坐下来，把技术报告里的每个组件摊开讲。

◆ 1. CSA + HCA 混合注意力——V4 最大的创新

━━━━━━━━━━━━━━━━━━━━

V4 的注意力核心是两种 **KV cache 压缩策略**交替排列，再加一层滑动窗口兜底。

先说压缩的具体做法。CSA 和 HCA 的压缩机制本质上是同一种操作——**学习的加权池化**：

1. 每个 token 的隐藏状态通过线性投影变成 KV cache 条目，同时生成一组"压缩权重"
2. 把相邻的 m 个 token 分成一组，用 softmax 对压缩权重做归一化，然后加权求和——m 条 KV cache 变 1 条
3. 权重不是固定的平均，是模型学出来的，还加了可学习的位置偏置（positional bias），让模型知道每个 token 在块内的相对位置

这不是简单的降采样或平均池化——每个压缩后的 KV cache 条目是原始 m 个条目的"智能摘要"，哪些 token 更重要、权重更高，是模型自己学会的。

说白了，CSA 和 HCA 在代码里就是同一个 Compressor 类，区别只有一个参数 compress_ratio——填 4 叫 CSA，填 128 叫 HCA。技术报告里拆成两个章节、两张图、两套公式，各起了一个名字。读代码比读论文清楚。

两种策略的区别在于压缩比和后续处理：

**CSA（Compressed Sparse Attention，compress_ratio = 4）**：每 4 个 token 的 KV cache 压成 1 条。CSA 还用了**重叠压缩**——每个压缩块不仅看自己的 4 个 token，还看前一个块的 4 个 token，相当于每个压缩结果实际融合了 8 个 token 的信息（虽然压缩比还是 4:1）。压缩完之后，在压缩后的 KV cache 上跑 DSA（DeepSeek Sparse Attention）——用 Lightning Indexer 选 top-k，只看最相关的一小撮，其余丢弃。

注意：**DSA 没有消失**，它仍然是 V4 注意力的核心组件，训练和推理全程都在用。只不过在 V3.2 里 DSA 直接操作原始 KV cache，在 V4 里 DSA 操作的是压缩后的 KV cache——先压缩再稀疏，两刀叠加。

**HCA（Heavily Compressed Attention，compress_ratio = 128）**：每 128 个 token 的 KV cache 压成 1 条，没有重叠，比 CSA 更简单粗暴。但不做稀疏——压缩后的全部 KV cache 都看（dense on compressed）。10 万 token 的上下文压完只剩不到 800 条，全看也很便宜。

**滑动窗口**：当前 token 往前数 128 个 token 的 KV cache 原样保留，不压缩。每生成一个新 token，最老的那条被踢掉、新的补进来，像传送带一样始终只保持最近 128 条。

为什么不是 4 条或者 128 条分开设？因为压缩操作要凑满一个块才能执行——CSA 要凑 4 个 token，HCA 要凑 128 个。凑满之前，同块内的 token 在压缩区里是看不到的。CSA 层最多有 3 个 token 在等待区，HCA 层最多有 127 个。滑动窗口的大小取最大的那个：128，刚好覆盖 HCA 最坏情况。CSA 层用 128 有点浪费（它只需要 3 个），但统一一个数字简化了实现，而且多存的几十条开销可以忽略（128 条 × 576 字节 ≈ 0.07 MB/层）。

三者的分工很清晰：CSA 负责"从全局历史中精选最相关的细节"（4:1 压缩 + DSA 稀疏选择），HCA 负责"用极低成本扫一遍全局大意"（128:1 重度压缩 + 全看），滑动窗口负责"最近的上下文一个都不能丢"。

**config.json 的 compress_ratios 字段完美对应**：

```
// V4-Pro 的 compress_ratios（主模型 61 层）
128, 128, 4, 128, 4, 128, 4, 128, 4, 128, 4, 128, 4, 128, 4,
128, 4, 128, 4, 128, 4, 128, 4, 128, 4, 128, 4, 128, 4, 128,
4, 128, 4, 128, 4, 128, 4, 128, 4, 128, 4, 128, 4, 128, 4,
128, 4, 128, 4, 128, 4, 128, 4, 128, 4, 128, 4, 128, 4, 128, 4
```

- 128 = HCA 层，共 31 层
- 4 = CSA 层，共 30 层

排列规律：前两层 HCA，之后 CSA 和 HCA 严格交替。每一步都能同时获得"全局概览"（HCA 层）和"精选细节"（CSA 层）两种信息。

**举个具体例子：50K token 上下文，V4-Pro 每层要存多少 KV cache？**

V4 的 KV cache 压缩分两步。

**第一步：维度压缩（MLA 低秩投影，2024 年 V2 首创，V3/V4 一路继承）。** MLA（Multi-head Latent Attention）是 DeepSeek 在 2024 年 5 月发布 V2 时提出的原创架构——把所有注意力头的 KV 压缩成一份低秩的 latent vector，用的时候再投影回去。这是 DeepSeek 能撑起百万上下文的基础，没有这一层打底，后面的 CSA/HCA 再怎么压序列维度也扛不住。 7168 维的隐藏状态通过线性投影压到 512 维（config.json: head_dim = 512, num_key_value_heads = 1），128 个 query 头共享同一份 512 维的 KV。混合精度存储（RoPE 的 64 维用 BF16，其余 448 维用 FP8），每条约 576 字节。如果不做这个投影，每条要存 7168 维 × 2 字节 = 14336 字节——光这一步就省了 96%。

**第二步：序列压缩（CSA/HCA，V4 新增）。** 在 512 维的 KV cache 上，再沿序列维度压缩——把相邻的 m 条加权合并成 1 条。具体做法：每条 KV cache 同时生成一个"压缩权重"，m 条为一组做 softmax 归一化后加权求和，权重是模型学出来的，不是固定平均。压缩后的那 1 条是原始 m 条的"智能摘要"——哪个 token 更重要，权重更高。CSA 每 4 条压 1 条，HCA 每 128 条压 1 条。

两步叠加后，50K 上下文要存多少 KV cache？

先说一个基础知识：**KV cache 是按层存的。** 每个 token 过一层 Transformer 就产生一条 KV cache，V4-Pro 有 61 层，所以 1 个 token 要存 61 条。50000 个 token 就是 50000 × 61 = 305 万条。这就是为什么 KV cache 是推理的显存大户——token 数 × 层数 × 每条大小，三个维度相乘。

标准注意力（不投影不压缩）：每层 50000 条 × 14336 字节 = 684 MB。61 层合计约 40.7 GB。
V2 MLA（只做维度投影）：每层 50000 条 × 576 字节 = 27.5 MB。61 层合计约 1.64 GB。存储省了 96%。
V3.2 DSA（投影 + 推理阶段稀疏）：KV cache 存储量和 V2 一样，还是 1.64 GB——DSA 只省计算不省存储，KV cache 全存着，只是注意力计算时选 top-k 不全看。

CSA 层（compress_ratio = 4）：
- 50000 / 4 = 12500 条压缩 KV cache + 128 条滑动窗口 = 共存 12628 条，约 7.3 MB（含 Indexer 的额外 KV cache）
- 注意力计算时不看全部 12500 条——Lightning Indexer 从中选 top-1024 条，加上滑动窗口 128 条，每个 query 实际只看 1152 条
- 存储省了约 4 倍，计算省了 50000 / 1152 ≈ 43 倍

HCA 层（compress_ratio = 128）：
- 50000 / 128 = 390 条压缩 KV cache + 128 条滑动窗口 = 共存 518 条，约 0.3 MB
- 全部都看（不做稀疏选择），每个 query 看 518 条
- 存储省了近 100 倍，计算省了 50000 / 518 ≈ 96 倍

V4-Pro 的 61 层里 30 层 CSA、31 层 HCA。50K 上下文的 KV cache 总量：30 × 7.3 + 31 × 0.3 ≈ **228 MB**，相比不压缩的 1.64 GB 省了约 **86%**。

**压缩后的 KV cache 就是最终存储的内容，原始的 token 级 KV 不保留**（只有滑动窗口的 128 条是原始精度）。所以这既省显存又省算力——不是"压缩了算，算完还原"，是"压缩了就这么存着用"。

**从标准注意力到 V4，每一代砍了什么（50K 上下文，61 层）：**

| 阶段 | KV cache 总量 | 省了什么 |
|------|-------------|---------|
| 标准注意力 | 40.7 GB | 基线 |
| V2 MLA 维度投影（2024） | 1.64 GB | 每条从 7168 维压到 512 维，省存储 96% |
| V3.2 DSA | 1.64 GB | 只省计算不省存储——KV cache 全存着，注意力选 top-k 不全看 |
| V4 CSA/HCA 序列压缩 | 228 MB | 多条压一条，再省存储 86% |

从 40.7 GB 到 228 MB，四代迭代压了约 **99.4%**。

**和 V3.2 的关键区别**：V3.2 的 DSA 是推理阶段的适配——模型训练时用的是完整注意力，推理时换成 DSA 来省算力，本质是"不改脑子，换副眼镜"。V4 在 DSA 前面加了一层 KV cache 压缩，训练阶段就开始用——模型从头到尾都在压缩后的 KV cache 上做注意力。DSA 本身依然在，但它的输入从原始 KV cache 变成了压缩后的 KV cache，搜索空间先被压缩砍了一刀，DSA 再砍第二刀。两刀叠加，才有了 27% FLOPs 和 10% KV cache 的效果。

◆ 2. Lightning Indexer——上篇叫的"FP4 Indexer"

━━━━━━━━━━━━━━━━━━━━

Lightning Indexer 只在 CSA 层（compress_ratio == 4）使用。它的工作就一件事：从压缩后的 KV cache 里选出最相关的 top-k。怎么判断"相关"？还是注意力的老办法——Query 和 Key 做点积（Q × K），分数高的就是相关的。Indexer 就是用这个分数排序，取前 1024 名。

它有自己**独立的压缩器**，和 CSA 主路径的压缩器不共享。这个压缩器的流程是：Hadamard 旋转 -> FP4 量化。Hadamard 旋转是一种正交变换，作用是把向量分量的能量分散均匀，让后续的 FP4 量化误差更小。

config.json 里对应的字段：

```json
"index_n_heads": 64,      // 索引头数量（Pro）
"index_head_dim": 128,    // 每个索引头的维度
"index_topk": 1024        // 选 top-1024
```

Indexer 的压缩 KV cache 用 FP4 精度存储，Query 也量化到 FP4，两者的点积（算相关性分数）全程 FP4 计算——存的、读的、算的都是 FP4。这是通过 QAT（量化感知训练）实现的，训练时就按 FP4 精度来优化，不是推理时硬砍的。

排序对精度的容忍度远高于精确计算——你只需要知道"谁排前面"，不需要知道分数差 0.001 还是 0.002。这是 Lightning Indexer 能用 FP4 的根本原因。

◆ 3. Hash Routing——和 Engram 有关系但不是 Engram

━━━━━━━━━━━━━━━━━━━━

上篇我们写了一整节 Engram，这是最大的失误。V4 没有使用 Engram。

但 config.json 里确实有一个 Hash 相关的字段：

```json
"num_hash_layers": 3
```

这代表前 3 层 MoE 用的是 Hash 路由（Hash Routing），而不是学习的路由。

代码里 Gate 类有一个 `tid2eid` 查找表：token ID 直接映射到 expert ID，O(1) 查表，不需要过路由网络。PDF 原文是：

> "we replace the dense FFN layers in the initial several Transformer blocks with MoE layers that employ Hash routing (Roller et al., 2021)"

这个"哈希"和 Engram 论文（arXiv 2601.07372）里的哈希记忆表不是同一件事。Engram 是用哈希来存储和检索知识向量；Hash Routing 是用哈希来固定路由——每个 token 永远去同一个专家，不需要学习。两者共享"哈希"这个词，但解决的问题完全不同。

为什么前 3 层要用固定路由？PDF 没有详细解释原因，只引用了 Roller et al., 2021（Hash Layers）。一个合理的猜测是：最底层的路由模式相对稳定，固定路由省计算还避免了负载均衡问题。

**回过头看，Engram 想解决的问题被 CSA 用另一种方式解决了。** Engram 的思路是：静态知识不该每次都过注意力，用哈希查表 O(1) 直接命中。CSA 的思路是：先把 KV cache 压缩到 1/4，再用 Lightning Indexer 从压缩后的 KV cache 里选 top-k——注意力还是注意力，但搜索空间被压缩 + 稀疏两刀砍到了原来的零头。效果上，CSA 把注意力成本压到 V3.2 的 27%，已经足够省了。而且 CSA 是通用方案，不挑内容类型——不管是静态事实还是需要上下文推理的内容都能处理。Engram 的哈希查表虽然理论上 O(1) 更快，但有两个硬伤：一是只能处理静态知识，不挑内容类型的通用场景用不了；二是记忆表存在 CPU DRAM 里，GPU 要用的时候得通过 PCIe 总线从 CPU 拉数据——PCIe 5.0 带宽 64 GB/s，GPU 显存带宽 3 TB/s，差了近 50 倍。省了显存但堵在总线上。CSA 全在 GPU 显存里完成，不跨总线，在 CSA 已经把成本压到这个程度的前提下，Engram 的性价比就不够了。

◆ 4. mHC（Manifold-Constrained Hyper-Connections）

━━━━━━━━━━━━━━━━━━━━

mHC 我们在第 37 期（《DeepSeek mHC：为什么"流形约束"是标题党》 https://mp.weixin.qq.com/s/IMU__NKt_L41YeHKi7_A1g ）已经详细拆解过原理，上期也引用了。这里不重复，只补充从 V4 技术报告和代码里看到的新信息。

**V4 的具体配置**：残差流展宽 4 倍（config.json: hc_mult = 4），Sinkhorn-Knopp 迭代 20 轮（hc_sinkhorn_iters = 20）。

**代码实现细节**：Block 类的 `hc_pre` 方法把 4 份残差流加权合并成 1 份送入子层，`hc_post` 方法把子层输出重新展开成 4 份。权重不是固定的——由输入动态生成（input-dependent），加上一个静态偏置。这就是 PDF 里说的"动态参数化"。

**训练开销**：PDF 报告 mHC 的额外 wall-time 开销只有 6.7%。他们用了三招压开销：融合 kernel、选择性 recomputation（只 checkpoint 必要的中间张量）、调整 DualPipe 流水线让 mHC 的通信和计算重叠。

◆ 5. Muon 优化器

━━━━━━━━━━━━━━━━━━━━

优化器就是训练时用来更新模型权重的算法。过去几年大模型几乎都用 AdamW，V4 换成了 Muon——收敛更快、训练更稳定。1.6T 参数里 99.9% 用 Muon——注意力层的投影矩阵、MoE 专家的 FFN 权重、mHC 的混合矩阵全部用 Muon。只有 embedding、输出头、RMSNorm 这几个模块仍用 AdamW，合计不到 20 亿参数，占比 0.1%。

Muon 和 AdamW 的核心区别：AdamW 是逐元素更新权重（每个参数独立算梯度方向），Muon 是把整个权重矩阵当作一个整体做正交化更新——让更新方向尽量"均匀分散"，不会某些方向更新太猛、某些方向纹丝不动。代价是计算量更大，但换来的收敛速度和稳定性对 1.6T 参数的模型来说值得。

◆ 6. MoE 具体参数

━━━━━━━━━━━━━━━━━━━━

上篇给了参数表但没展开。这次补充几个重要细节。

**MoE 参数对比（V3 → V4）**：

| | V3 | V4-Flash | V4-Pro |
|---|---|---|---|
| 路由专家数 | 256 | 256 | 384 |
| 激活专家数 | 8 | 6 | 6 |
| 共享专家数 | 1 | 1 | 1 |
| 专家中间维度 | 2048 | 2048 | 3072 |
| 路由打分函数 | Sigmoid | sqrtsoftplus | sqrtsoftplus |

**激活函数变了**：V3 用的是 Sigmoid 做路由打分，V4 换成了 `sqrt(softplus(x))`。

```python
# config.json
"scoring_func": "sqrtsoftplus"
```

Softplus 是 ln(1 + e^x)，一个平滑版的 ReLU。先 softplus 保证非负，再开根号压缩动态范围。相比 Sigmoid 的优势是：不会饱和（Sigmoid 在两端梯度接近 0），梯度信号更健康。

**SwiGLU Clamping**：每个 MoE 专家内部是一个 FFN（前馈网络），用的激活函数叫 SwiGLU——这是目前大模型最常用的激活函数，LLaMA、Gemma、DeepSeek 全在用。SwiGLU 本身没有上下界限制，输出可以是任意实数，想飙多高飙多高。V4 给它加了硬裁剪，限制在 [-10, 10]。原因很直接：超大模型训练时，个别 token 的激活值偶尔会飙到几百，触发梯度爆炸。而且 V4 的专家权重是 FP4 精度——E2M1 只有 2 位指数 1 位尾数，能表示的数值范围极其有限。激活值一旦飙到几百，FP4 直接溢出。所以 Clamping 不只是防梯度爆炸，也是让数值落在 FP4 能安全表示的范围内——不裁剪，必炸。

◆ 7. 训练细节——165 期完全没提的新信息

━━━━━━━━━━━━━━━━━━━━

以下内容全部来自技术报告 PDF，上篇没有覆盖。

**预训练数据量**：Pro 实际是 33T token，不是上篇写的 32T+。

**稀疏注意力不是从头用的**：先用 dense attention（完整注意力）训练 1T+ token，模型学会"怎么看"之后，再引入 sparse attention。如果从第一个 token 就用稀疏注意力，模型还没学会分配注意力就被迫做稀疏选择，效果会差。

**序列长度渐进扩展（预训练阶段）**：4K -> 16K -> 64K -> 1M。不是一上来就训百万上下文——先在短序列上把基础能力学好，再逐步拉长。这个扩展发生在预训练过程中，不是训完之后单独做的。

━━━━━━━━━━━━━━━━━━━━

◆ 8. Anticipatory Routing——防训练崩溃的黑科技

━━━━━━━━━━━━━━━━━━━━

MoE 训练有一个独特的不稳定性来源：路由网络和专家网络同步更新时，路由决策可能突然大幅变化，导致某些专家突然涌入大量 token、另一些专家瞬间空闲——loss 突然飙升（loss spike）。

MoE 每一步训练其实有两个决策：第一，路由网络决定"这个 token 送去哪个专家"；第二，被选中的专家拿到 token 后做实际计算。正常训练时，这两步用的是同一套参数——路由和计算看到的是同一个模型状态，同步更新。问题在于：某一步梯度更新幅度大了，路由决策突然变化，大量 token 涌入之前冷门的专家，这些专家还没准备好，loss 就炸了。

Anticipatory Routing 把这两步解耦：**路由用几步之前的旧参数做决策，专家用当前最新的参数做计算。** 路由决策总是"慢半拍"——它看到的是几步之前的模型状态，不会被当前步的剧烈更新带偏。专家计算正常用最新参数，不受影响。

更聪明的是自动开关机制：系统实时监测 loss。正常训练时不启用（没必要多花开销），一旦检测到 loss spike，自动启用 Anticipatory Routing 来稳住路由，loss 恢复正常后自动关闭。

额外开销约 20%——只在 loss spike 期间付出这个代价，不是全程。

━━━━━━━━━━━━━━━━━━━━

◆ 9. FP4 QAT（量化感知训练）

━━━━━━━━━━━━━━━━━━━━

V4 的 FP4 不是训完之后再量化的（Post-Training Quantization），是训练过程中就让模型适应 FP4 精度的（Quantization-Aware Training, QAT）。

先澄清一个容易混淆的概念。当年 DeepSeek V3 宣传"FP8 训练"，说的是**计算精度**——矩阵乘法用 FP8 算，省算力。但优化器存储的主权重（包括动量、方差等状态）一直是 FP32 全精度，否则训练会不稳定。所谓的"低精度训练"，低的是算的精度，不是存的精度。

| | 优化器存储 | 计算精度 | 额外操作 |
|---|---|---|---|
| V3 FP8 训练 | FP32 | FP8 | 无 |
| V4 FP4 QAT | FP32 | FP8 | 前向传播多一步 FP4 → FP8 模拟 |

V4 的 QAT 具体流程：优化器里存的主权重还是 FP32。每次前向传播时，先把 FP32 权重量化到 FP4，再反量化到 FP8，用 FP8 做实际计算。这个"FP32 → FP4 → FP8"的过程会引入 FP4 的量化误差——模型在训练时就"习惯"了这个误差。反向传播时，梯度直接穿过量化操作（直通估计器，Straight-Through Estimator），更新回 FP32 主权重。推理时直接用 FP4 权重，因为模型已经适应了 FP4 的精度。

两个地方用了 FP4 QAT：

1. **MoE 专家权重**：384 个专家的权重全部走上述流程。
2. **Lightning Indexer 的 QK 路径**：索引器的 Query 和 Key 也是 FP4 QAT 训练的。

一个关键的数学事实：**训练阶段**，FP4 到 FP8 的反量化是**无损的**。FP8 (E4M3) 比 FP4 (E2M1) 多 2 个指数位，动态范围更大。FP4 量化时会对每个 1x32 的小块计算一个 scale factor，而 FP8 量化时对每个 128x128 的大块计算一个 scale factor。只要大块内各小块 scale factor 的比值不超过一定阈值，FP8 就能完全"吸收"掉 FP4 的精细缩放信息。实测中模型权重都满足这个条件，所以训练时的 FP4 → FP8 转换零精度损失，而且整个 QAT 流程可以完全复用现有的 FP8 训练框架。

**推理阶段**则不需要反量化到 FP8——Blackwell 架构（B200/GB200）原生支持 FP4 × FP8 的矩阵乘法，权重 FP4 直接参与计算，激活值 FP8。不过 PDF 提到一个现实：目前 Blackwell 上 FP4 × FP8 和 FP8 × FP8 的峰值算力相同——也就是说 FP4 目前只省显存不省算力。PDF 说未来硬件理论上可以让 FP4 × FP8 再快 1/3，但那是下一代的事了。

━━━━━━━━━━━━━━━━━━━━

◆ 10. Attention Sink

━━━━━━━━━━━━━━━━━━━━

Attention Sink 存在于每一层的注意力模块里（不只是输出层），每个注意力头各有一个。

先说它解决什么问题。标准注意力的 softmax，所有权重加起来必须等于 1：

```
weight[i] = exp(score[i]) / Σ exp(score[j])
```

哪怕所有历史 token 都和当前 Query 不相关，权重也得分出去——模型被迫"看"一些没用的 token。

V4 在 softmax 的分母里多加了一项 exp(sink)，sink 是一个可学习参数：

```
weight[i] = exp(score[i]) / (Σ exp(score[j]) + exp(sink))
```

换个说法：原来的 softmax 像一场投票，100% 的票必须投出去，不能弃权。sink 给了模型一个"弃权票箱"——你可以把票投进去，但投进去的票不产生任何效果。数学上，这是把注意力从概率分布（权重和 = 1）变成了次概率分布（权重和 <= 1）。

这不是 DeepSeek 独创的——OpenAI 在 GPT-5 的技术报告里也用了类似的设计。对长上下文来说几乎是必需品：百万 token 里真正相关的可能只有几百个，剩下的都是噪音。没有弃权选项，注意力必须均匀撒在噪音上；有了弃权选项，注意力可以集中在真正重要的 token 上，剩余的全部"倒进水槽"。

━━━━━━━━━━━━━━━━━━━━

◆ 11. MTP（Multi-Token Prediction）

━━━━━━━━━━━━━━━━━━━━

V4 延续了 V3 的 Multi-Token Prediction 训练策略：

```json
"num_nextn_predict_layers": 1
```

depth = 1，意味着模型在训练时不仅预测下一个 token，还同时预测下下一个 token。额外的预测头共享主模型的表示但有自己的输出层。

一次预测 2 个 token 有什么用？好处不在推理——推理时其实还是一个一个输出的，MTP head 可以不用。好处在**训练阶段**：强迫模型在预测当前 token 的时候，内部表示就已经"想好了"下一个 token 是什么。这让中间层的表示更有前瞻性，信息密度更高。打个比方：考试只要求答第 1 题，你可能只想第 1 题。但要求同时答第 1 题和第 2 题，你看题目时就会多想一步——这个"多想一步"的训练信号让模型的表示质量更好。

为什么是 2 个不是 3 个 4 个？这是实验结果。depth 每加 1 就多一个预测头、多一份计算量。V3 的论文里做过 ablation，depth = 1（预测 2 个）收益最高，再往上边际递减。V4 沿用了这个结论，没有加大 depth。

◆ 推理框架现状——能跑，但不好跑

━━━━━━━━━━━━━━━━━━━━

上期我们说"真正要等的是推理框架的适配——vLLM、SGLang 这些框架支持 DeepseekV4ForCausalLM 架构之后，就能直接跑"。这个判断没问题，但现在可以说得更具体。

**官方只提供了裸 PyTorch 推理脚本。** HuggingFace 仓库的 inference 文件夹里有 model.py（模型定义）、generate.py（生成脚本）、convert.py（权重转换），用 torchrun 启动。没有 vLLM，没有 SGLang，没有 TensorRT-LLM，没有任何第三方框架适配。

这不奇怪。V4 的架构对推理框架提出了前所未有的挑战：

**1. 异构 KV Cache。** V3 的 KV cache 每一层结构一样，推理框架可以统一管理。V4 每一层的 KV cache 结构不同——CSA 层有压缩 KV + Lightning Indexer KV + 滑动窗口 KV 三种，HCA 层有重度压缩 KV + 滑动窗口 KV 两种，纯滑动窗口层只有滑动窗口 KV。PDF 专门用了一整节（3.6.1）讲 KV Cache 的异构管理，画了一张复杂的布局图。PagedAttention 的基本假设——每一层的 KV cache 大小一致——被彻底打破了。

**2. 压缩状态缓冲。** CSA 和 HCA 的压缩操作需要凑满 m 个（CSA 是 4 个，HCA 是 128 个）token 才能执行一次。在凑满之前，未压缩的 token 和它们的隐藏状态必须暂存在缓冲区里。这个"等凑满再压缩"的逻辑在现有推理框架里没有对应物。

**3. Lightning Indexer 的 top-k 选择。** 每个 query token 要从压缩 KV cache 里选 top-1024（Pro）个最相关的条目。这个选择是动态的、per-query 的，不是固定模式。现有稀疏注意力 kernel 大多假设固定的稀疏模式（比如滑动窗口、block sparse），V4 的 per-query top-k 需要专门的 kernel 支持。

**4. On-Disk KV Cache。** PDF 描述了三种磁盘缓存策略（全量缓存、周期检查点、零缓存），用于共享前缀的请求复用。这是生产级的优化，第三方框架短期内不太可能实现。

**上期提到的两个硬件方案，需要修正：**

上期我们推荐了 Mac Studio M3 Ultra 和两台 DGX Spark 直连。硬件规格没算错——256GB 内存确实装得下 160GB 的 V4-Flash 权重。但软件侧目前都跑不通：

- **Mac Studio M3 Ultra**：MLX 还没有适配 DeepseekV4ForCausalLM 架构。V4 的异构注意力（CSA + HCA 交替 + 滑动窗口 + Lightning Indexer）和 V3 的 MLA 完全不同，MLX 需要从头实现这套注意力机制。再加上 Apple Silicon 没有 FP4 原生指令，即便适配了也要走软件模拟。目前状态：**装得下，跑不了。**

- **两台 DGX Spark 直连**：官方推理脚本的默认配置是 model-parallel=8（8 卡张量并行）。DGX Spark 每台只有 1 块 GPU，两台也只有 2 块。要在 2 卡上跑需要自己改并行策略，官方没有提供现成的 2 卡配置。而且单台 Spark 的 128GB 共享内存装不下 160GB 的 Flash 权重，必须双机——双机推理需要框架层面的支持。目前状态：**硬件够，脚本不支持。**

**目前实际可行的选择：**

- **官方 PyTorch 脚本 + 多卡 GPU 集群**：这是目前唯一能直接跑的方案。官方默认 model-parallel=8，需要支持 FP4 的 Blackwell 架构卡（B200/GB200）。上一代的 H100 只支持到 FP8，A100 连 FP8 都不支持——要在这些卡上跑 V4 的 FP4 权重，只能先反量化到 FP16 再算，显存占用翻倍，FP4 的效率优势全丢。裸 PyTorch 没有 continuous batching、没有 PagedAttention（也用不了原版 PagedAttention），适合验证和实验，不适合生产服务。
- **等 vLLM / SGLang 适配**：这两个框架社区都在跟进 DeepSeek 系列模型，但 V4 的异构 KV cache 是全新挑战。V3 从开源到 vLLM 完整支持花了大约两周，V4 大概率会更久。
- **等 DeepSeek 官方推理框架**：PDF 提到他们的 Fine-Grained EP Scheme（就是我们之前叫的 Mega MoE）已经开源在 DeepGEMM 的 PR #304。后续可能会有更完整的推理框架开源。
- **等 MLX 适配（Mac 用户）**：MLX 社区对 DeepSeek 系列模型的适配速度一直很快，但 V4 的异构注意力是个硬骨头。
- **昇腾 950PR**：硬件上没问题——950PR 原生支持 FP4，是目前除 Blackwell 之外唯一支持 FP4 的商用推理卡。PDF 技术报告里也明确写了"We validated the fine-grained EP scheme on both NVIDIA GPUs and HUAWEI Ascend NPUs platforms"，说明 DeepSeek 内部已经在昇腾上跑通了。但开源的推理代码（kernel.py）全是 CUDA kernel，没有 Ascend 版本。昇腾用户需要等华为或社区出 CANN/MindSpore 适配。考虑到 V4 的 FP4 方案和昇腾 950PR 的发布时间线高度吻合（上期分析过），华为侧的适配应该不会太慢。

简单说：**上期的硬件分析没问题，但高估了软件的就绪程度。** 模型权重已经开源了，"装得下"和"跑得起来"之间还隔着推理框架适配这道坎。V4 的 CSA + HCA + Lightning Indexer 组合，大概是开源 LLM 推理框架面对过的最复杂的注意力架构。

━━━━━━━━━━━━━━━━━━━━

◆ 整体回顾

━━━━━━━━━━━━━━━━━━━━

把 V4 的技术栈摊开看：

| 组件 | 上篇怎么写的 | 实际是什么 |
|------|------------|-----------|
| 注意力架构 | DSA2（五合一） | CSA + HCA（两种压缩注意力交替排列 + 滑动窗口） |
| 知识检索 | Engram 哈希记忆 | 没有使用 Engram，前 3 层用 Hash Routing |
| 稀疏注意力索引 | FP4 Indexer | Lightning Indexer（FP4 QAT） |
| MoE 推理优化 | Mega MoE | Fine-Grained EP Scheme（PDF 正式名称） |
| 官方列的三大创新 | 五大创新 | 混合注意力 + mHC + Muon |

上篇的大方向没错——V4 确实是在注意力稀疏化、MoE 推理优化、训练稳定性三个方向上同时推进。但名字叫错了，Engram 写多了，创新分类也不对。

速报和精读的差别，就在于"坐下来看代码"这一步。config.json 的 compress_ratios 字段、model.py 的 Gate 类、PDF 第 4 章的训练细节——这些东西不读原文是猜不到的。
