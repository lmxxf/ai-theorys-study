【DeepSeek V4 推理】一次 OOM 的经验：注意力一波算不完

━━━━━━━━━━━━━━━━━━━━

◆ 现象：17.8k token，两台 128G 的 Spark 全死机了

━━━━━━━━━━━━━━━━━━━━

先做个区分。这个 DGX Spark 跑 DeepSeek V4 的系列已经写了快 10 期，但其实是**两个不同的仓库交替在讲**：

- **vLLM 部署线**（`deepseek-v4-deployment-on-dgx-spark`，目标是出 token、出服务）：185 期 https://mp.weixin.qq.com/s/dEc-wD0JcM5WhKicOUR1pQ （sm_121 修复，从 4 到 12 tok/s）、224 期 https://mp.weixin.qq.com/s/X1b7H4565_WFn6CSTzoAQg （W4A16 vs W4A8 反转）等几期都属于这条线。
- **实验平台线**（`deepseek-v4-experimental-platform-on-dgx-spark`，目标是抽激活、做研究）：上一期讲它是 216 期 https://mp.weixin.qq.com/s/mVM2rvX9VdGO9WKsTNrlTw （部署手册，"两台 Spark 跑通 V4 Flash：改了官方推理代码的哪里"），加上之前几期一共三四期。**这一期就接在 216 那条线后面**——同一个仓库，新的坑。

这两条线**基于完全不同的代码**——vLLM 线是工业级推理框架，实验平台线是 **DeepSeek 官方推理脚本**改造而成。代码路径、权重加载方式、kernel 选择、kv cache 管理全都不一样，遇到的坑也是另一套。读者如果上一次跟的是 vLLM 那条线，请先把它放一边——这一期讲的是实验平台线的新麻烦。

工程目标变了，常见的 OOM 形态也变。这次问题很朴素：两台 DGX Spark 用 NVLink 连起来，跑 DeepSeek V4 Flash 280B。权重大部分是 FP4，少量 FP8，整套权重文件加起来约 150GB。两台机器各 128GB 统一内存，TP=2 切分后，单机每个 rank 常驻约 82GB，理论上还剩四十多 GB 余量。

短 prompt 能跑，中 prompt 能跑。但一换成太长的——我自己日常给 Claude 模型加载的一份 memory.md，约 17815 token——机器直接 OOM。而且不是 Python 抛异常那种温柔的 OOM，是 NVRM OOM，整个系统被统一内存拖死。

这就有意思了。账面上算，单机 82GB 常驻、还剩 46GB 余量，长 prompt 的 KV cache 加中间张量按常理远远花不到这个数。模型不大，硬件不小，怎么就 OOM 了？

答案最后证明是：**硬件够。真正的问题是我们把某个中间张量一次性摊开了。**

摊开以后，内存曲线看起来像"模型太大"——但**那不是模型大，是算法写法大**。

━━━━━━━━━━━━━━━━━━━━

◆ 先把加载期嫌疑排掉

━━━━━━━━━━━━━━━━━━━━

在排查 forward OOM 之前，加载期已经修过一个非常隐蔽的问题，先讲清楚——免得这次 OOM 被甩锅到这上面。

`safetensors` 流式加载 46 个 shard，代码看起来是这样的：

```python
with safe_open(shard_file, framework="pt", device="cpu") as f:
    ...
```

直觉上，`with` 结束 mmap 就该释放。但实际不是——46 个 shard 关掉之后，虚拟地址空间还在涨。更要命的是，虚拟地址空间本身虽然不等于物理内存，但**页表（page table）要占物理内存**。最后页表能涨到 **7GB 多**。

💡 page table 不是页面缓存（page cache）。page cache 是"已经读过的文件页缓存在 RAM 里加速下次读"，page table 是"CPU 把虚拟地址翻译成物理地址的查找表"。mmap 77GB 文件，哪怕你一页都没真读，内核也得为这 77GB 虚拟地址范围**预留索引条目**——条目本身是实打实的 RAM 占用。`with` 退出后 safetensors 没真 munmap，46 个 shard 累积出 112GB 虚拟地址范围，对应的页表条目就涨到 7GB。

在普通服务器上这可能只是看着难看，但在 DGX Spark 这种 CPU/GPU 共享 128GB 内存的机器上，7GB 页表直接吃掉了 GPU 那边的活动余量。

修法很土：

```python
gc.collect()
ctypes.CDLL("libc.so.6").malloc_trim(0)
```

每个 shard 结束后强制把 glibc arena 还给内核。修完之后，加载期日志稳定下来：

```text
shard 1/46  done  GPU=83.6GB  RSS=1.0GB  VM=102GB  PgT=3MB
shard 5/46  done  GPU=83.1GB  RSS=1.1GB  VM=112GB  PgT=4MB
...
shard 46/46 done  GPU=82.0GB  RSS=0.5GB  VM=112GB  PgT=4MB
```

VM 很大（112GB），但 RSS（0.5GB）和 PgT（4MB）很小——说明 mmap 残留的虚拟地址不再吃真实物理页。

所以加载期可以排除。**OOM 不在"权重怎么读进来"，OOM 在 forward。**

━━━━━━━━━━━━━━━━━━━━

◆ 第一个真 bug：Indexer 的 O(S²)

━━━━━━━━━━━━━━━━━━━━

DeepSeek V4 的注意力不是传统全量 KV attention，它有压缩、有索引、有 sparse attention（完整谱系——从 MHA 到 CSA/HCA 这一路是怎么走过来的——168 期 https://mp.weixin.qq.com/s/YqcTnIrGKEYZ2n-sMAelKg 专门讲过）。其中 **Indexer** 要根据当前 query，从压缩后的 KV 里选 top-k。

原始路径里有一行看起来非常普通：

```python
index_score = torch.einsum("bshd,btd->bsht", q, kv_cache[:, :T])
```

💡 先把这一行的每个零件拆开：

**`torch.einsum`**：PyTorch 的爱因斯坦求和约定接口。给两个张量、一个字符串规则，它自动算出哪些维度做点积、哪些维度保留、哪些维度广播。把矩阵乘法、batch 矩阵乘、转置、求和全部用一行表达式写出来。优点是简洁，缺点是**它会按字面意思把整个结果张量造出来**——这次坑就在这里。

**`"bshd,btd->bsht"`**：einsum 的规则字符串，逗号左边是第一个输入的维度标签，右边是第二个输入的标签，`->` 后面是输出标签。每个字母代表一个维度名。这条规则的意思是：

| 标签 | 含义 | 这次值 |
|---|---|---|
| `b` | batch | 1 |
| `s` | query 的 seqlen | 17815 |
| `t` | KV 的 compressed_len（压缩后的 KV 长度） | 约 4458（≈ 17815/4） |
| `h` | head 数 | 64 |
| `d` | head_dim | 约 128 |

`bshd` 表示 `q` 的形状是 `[b, s, h, d]`，`btd` 表示 `kv_cache` 的形状是 `[b, t, d]`，`bsht` 表示输出 `index_score` 的形状是 `[b, s, h, t]`。

**`d` 在输入两边都出现、在输出里消失** → 这个维度被点积（sum over d）求和掉了。**`h` 只在 `q` 那边出现** → 对 KV 做广播（每个 head 都看一遍同一份 KV）。**`s` 和 `t` 都保留** → 输出是 `s × t` 的密集矩阵，每个 query token 对每个 KV 位置都给一个分数。

**`kv_cache[:, :T]`**：PyTorch 的 numpy 风格切片。`kv_cache` 假设形状是 `[batch, total_len, head_dim]`，`[:, :T]` 表示**第 0 维全取**（保留 batch）、**第 1 维只取前 T 个**（截到当前已经填入的有效长度，后面预分配但还没写入的位置不参与计算）。**这是个轻量操作**——只是给原张量做了个视图（view），不复制数据，本身不吃额外内存。

────────────────────

如果嫌符号还是抽象，把这条 einsum 改写成最朴素的 Python 嵌套循环就一目了然：

```python
# 等价的"傻瓜版"实现
for b_i in range(b):                # 每个 batch
    for s_i in range(s):            # 每个 query token (s=17815)
        for h_i in range(h):        # 每个 head (h=64)
            for t_i in range(t):    # 每个 KV 位置 (t≈4458)
                # 取出长度为 d 的两个向量，做点积
                q_vec  = q[b_i, s_i, h_i, :]      # shape [d]
                kv_vec = kv_cache[b_i, t_i, :]    # shape [d]
                index_score[b_i, s_i, h_i, t_i] = (q_vec * kv_vec).sum()
```

人话：**有 17815 个 query token，每个 token 有 64 个 head；每个 head 都拿自己那条 query 向量，去和 4458 个 KV 位置各做一次点积，得出 4458 个分数。最终输出是 `17815 × 64 × 4458` 个分数**，密密麻麻摆成一张大表。

**这张大表就是问题**——`17815 × 64 × 4458` 个浮点数同时塞在内存里。einsum 不是按需流式产生，它是真的把整张表一次性算出来、整张存好。

einsum 当然不会真用四层嵌套循环（那慢得没法看），它底层是 GPU 上的并行矩阵乘——但**输出张量的形状和占用，跟上面这段循环算的完全一样**。

────────────────────

算一下这张表多大：在 prefill 阶段 `seqlen = S`、`compressed_len ≈ S / 4`，输出张量 `[1, S, 64, S/4]` 的规模是 `S × heads × S/4`——也就是 **O(S²)**。17.8k token 时按 BF16 算：1 × 17815 × 64 × 4458 × 2 字节 ≈ **20 GB**。**单层 20GB**，DeepSeek V4 有 43 层 transformer block，每一层都要这么过一遍。

这不是"模型太大"，是"**把不该完整出现的矩阵完整实例化了**"。

💡 经典 attention 的影子

熟悉 attention 的程序员看到这一行应该一眼认出来——这就是经典 attention 第一步 `Q @ K^T` 的变种：

| 经典 attention | 这次的 Indexer | 差别 |
|---|---|---|
| `Q @ K^T` | `einsum("bshd,btd->bsht", q, kv_cache)` | 同一回事，einsum 写法更直接 |
| 用全量 K | 用 `kv_cache[:, :T]`（压缩后的 KV，`t ≈ s/4`） | Indexer 查的是 MLA 低秩压缩后的 KV，所以 `t` 比 `s` 小 4 倍 |
| 每个 head 一份 K | KV 是 `[b, t, d]`（没有 h 维） | 所有 head 共享同一份 K——这是 MLA/MQA 省 KV cache 的核心 |
| 后接 softmax + @ V | 后接 `relu × weights → sum_head → topk` | Indexer **不算 softmax 也不查 V**，只是挑"哪 512 个 KV 位置值得细看" |

读 einsum 字符串还有个小窍门：**带 `s` 标签的就是 query（query 的位置维度按惯例叫 s），带 `t` 标签的就是 key/value（被查询的位置维度叫 t）**——这是 Vaswani 2017 之后的行业约定俗成。

所以整条 DeepSeek V4 注意力流水线 = **Indexer 这一刀（用便宜的 Q × 压缩 K 挑出 top-512 个最值得看的位置）→ sparse_attn 那一刀（在这 top-512 里做完整的 Q @ K → softmax → @ V）**。把全量 O(S²) attention 降级成 O(S × topk) 的稀疏 attention。

这次 OOM 正好两刀都漏了分块——Indexer 摊 20GB、sparse_attn 摊 1GB × 43 层。**下一节就讲第二刀，本节先把第一刀修了**。

────────────────────

讽刺的是，**Indexer 这个为了"避免全量 O(S²)"而存在的零件，我们移植时它自己变成了 O(S²)**。

官方 TileLang kernel 里这一步本来是 fused + block streaming 的——算一块、归约、丢掉、再算下一块，不会把整个 `index_score` 放到显存里。我们在 sm_121 上替换 kernel 时漏掉了这条路径，相当于把"流式处理"退化成了"一次性摊开"。

💡 一个诚实的对照：原版在数据中心卡上没这个问题

读到这里有个合理的疑问：DeepSeek 官方推理脚本部署在 H100 / B200 这类卡上（TileLang kernel 在编译期明确写了 **sm_90 Hopper 和 sm_100 数据中心 Blackwell** 两条线），跑同样的 17.8k token，按理说也会摊出几十 GB 的中间张量，怎么没人喊 OOM？

两个原因。

**第一**，原版 TileLang kernel 这一步本来就是分块流式的（如上所述）。问题出在我们替换 kernel 这一步：sm_121（DGX Spark 的消费级 Blackwell）跑不了 TileLang，我们用 `torch.einsum` 重写时漏掉了原版的分块策略。

**第二**，Hopper / 数据中心 Blackwell 单卡显存 80GB（H100）到 192GB（B200）量级，多卡集群轻松上 TB。即便有人不小心摊开了，余量也吞得下，根本暴露不出来。

所以这次 OOM 不全是"消费级硬件内存小"的问题——更准确说是：**原版用 fused/streaming kernel 把这条路径堵住了，我们移植时漏堵；移植到数据中心卡上未必会炸，但在 DGX Spark 这种 128GB 统一内存的机器上，余量薄得藏不住任何 sloppy 写法**。

消费级硬件不是制造了 bug，是把"写得没那么紧的代码"暴露了出来。

────────────────────

所以第一刀的修法很直接——把 Indexer 改成 query 维分块，一次只处理 256 个 query token：

```python
for s_start in range(0, seqlen, 256):
    q_chunk = q[:, s_start:s_start+256]
    score_chunk = torch.einsum("bshd,btd->bsht", q_chunk, kv_cache[:, :T])
    # ↑ shape [b, 256, h, t]——把 s 那一维从 17815 压到 256
    score_chunk = relu(score_chunk) * weights
    score_chunk = score_chunk.sum(dim=h)
    all_reduce(score_chunk)
    topk_idxs[:, s_start:s_start+256] = topk(score_chunk, k=512)
    # score_chunk 出作用域立即释放
```

中间张量从 `[b, 17815, h, t]` 降到 `[b, 256, h, t]`——单次只占原来的 256/17815 ≈ 1.4%，约 290MB，算完 topk 立刻释放。

这刀修完后，短、中 prompt 能过了，但长 prompt 还是 OOM——**说明还有第二个真凶**。

━━━━━━━━━━━━━━━━━━━━

◆ 第二个真凶：sparse_attn 不是 O(S²)，但照样能坑人

━━━━━━━━━━━━━━━━━━━━

Indexer 修完以后，重新跑四个 stage：

| stage | 输入长度 | 结果 |
|---|---|---|
| stage1_short | 18 tokens | OK |
| stage2_1k | 416 tokens | OK |
| stage3_6k | 2416 tokens | OK，peak 86.9GB |
| stage4_long | 17815 tokens | **OOM** |

这时候如果只盯 O(S²) 会误判——Indexer 已经分块了，那为什么 stage4 还死？

看 sparse attention。Indexer 选完 top-k 之后，真正的 attention 要做的是：每个 query token 只看 topk 个 KV 位置。这已经不是全量 attention，规模不是 `S × S`，而是 `S × topk`，听起来安全多了。

💡 先回顾一下两刀的分工，避免混

DeepSeek V4 的注意力流水线，每一层 transformer block 里其实跑了两个函数：

```text
[1] Indexer.forward(q, kv_cache)
    ├─ index_score = torch.einsum("bshd,btd->bsht", q, kv_cache[:, :T])  ← 第一个 bug 在这
    ├─ relu × weights → sum_head → all_reduce → mask
    └─ topk_idxs = topk(index_score, k=512)   # 输出 top-512 索引

[2] sparse_attn(q, kv, attn_sink, topk_idxs, softmax_scale)
    ├─ kv_gathered = gather(kv, topk_idxs)    # [b, s, topk, d]   ← 第二个 bug 在这
    ├─ scores     = q @ kv_gathered           # [b, s, h, topk]
    ├─ weights    = softmax(scores + attn_sink)  # [b, s, h, topk+1]
    └─ output     = weights @ kv_gathered     # 完整 attention 输出
```

两刀分工：**Indexer 用便宜的"Q × 压缩 K"算粗略相关度，挑出 top-512 个值得细看的位置；sparse_attn 拿到这 512 个索引，在这 512 个位置上做完整的 Q@K → softmax → @V**——这才是真正的 attention。

上一节已经把第一刀修了。这一节看第二刀。

────────────────────

第二刀的问题代码（`kernel_sm121.py` 的 `sparse_attn`）：

```python
kv_gathered = [b, s, topk, d]
scores      = [b, s, h, topk]
weights     = [b, s, h, topk + 1]
```

💡 先把这几个字母翻译成人话：

| 字母 | 含义 | DeepSeek V4 Flash 实际值（长 prompt 场景） |
|---|---|---|
| `b` | batch size（一次推理几条 prompt 同时跑） | 1（单 prompt） |
| `s` | seqlen（这条 prompt 的 token 数） | **17815** |
| `h` | n_heads（多头注意力的头数） | **64** |
| `topk` | 每个 query token 只看的 KV 位置数 | **512** |
| `d` | head_dim（每个头的维度） | 约 128 |

挑最大的那个 `scores = [b, s, h, topk]` 心算一下：1 × 17815 × 64 × 512 × 2 字节（BF16）≈ **1.1 GB**。`kv_gathered` 还要再加上 `d` 维 → 又是 GB 级。这是**单层**的中间张量，DeepSeek V4 有 **43 层** transformer block，每一层 forward 都要这么过一遍。

旧实现为了快，把整个 seqlen 一次性向量化——没有 Python 循环、GPU 利用率好看、代码也漂亮。但它**把 17.8k token 的 `kv_gathered`、`scores`、`weights` 全摊开了**。

这里要注意一个反直觉点：**不是只有 O(S²) 才会 OOM。在大模型推理里，O(S × topk × heads) 也足够把整机内存吃光。**

因为常驻权重已经吃了 82GB——你不是从 0GB 开始分配中间张量，是站在 82GB 的悬崖边上，再往上摊几 GB、十几 GB。一旦加上 CUDA reserved、all-reduce 同步缓冲、allocator 碎片、两台 worker 的时序差，看起来"理论上还够"的内存，就会被推下去。

━━━━━━━━━━━━━━━━━━━━

◆ 修法：不是少算，是别一次性摊开

━━━━━━━━━━━━━━━━━━━━

修法非常直接——把 sparse attention 也按 query 维分块。原来一次处理完整 seqlen，现在一次处理 512 个 query token。

伪代码：

```python
for s_start in range(0, seqlen, 512):
    q_chunk = q[:, s_start:s_start+512]
    idx_chunk = topk_idxs[:, s_start:s_start+512]

    kv_gathered = gather(kv, idx_chunk)
    scores = q_chunk @ kv_gathered
    weights = softmax(scores + attn_sink)
    out_chunk = weights @ kv_gathered

    o[:, s_start:s_start+512] = out_chunk
```

注意：**这不是近似**。不是少看 token、不是降低 topk、不是砍模型。它只是改变了执行顺序——原来"把所有中间结果同时放在内存里"，现在"算一块、写回输出、释放、再算下一块"。

数学结果不变。小张量对照实测：

```text
batch=1  max_abs_diff=0.000000
batch=2  max_abs_diff=0.000000
```

和旧的 vectorized 实现完全一致。这就是工程里最舒服的一类修复：**不改语义，只改峰值。**

━━━━━━━━━━━━━━━━━━━━

```text
stage2_1k     416 tokens    OK | 2.8s  | peak_alloc=83.0GB
stage3_6k     2416 tokens   OK | 10.4s | peak_alloc=83.3GB
stage4_long   17832 tokens  OK | 79.9s | peak_alloc=90.3GB
```

对比修复前：stage3 从 86.9GB 降到 83.3GB，省掉 3.6GB；stage4 原来直接死机，现在只到 90.3GB。

常驻权重大约 82GB，也就是说长 prompt 在常驻之上的额外峰值只有 **8GB 左右**，128GB 机器上还剩三十多 GB 余量。

所以结论可以钉死：**stage4 OOM 的主因不是模型装不下，而是 sparse_attn 把长 prompt 的中间张量一次性摊开了。**

━━━━━━━━━━━━━━━━━━━━

◆ 为什么这个错误这么容易犯

━━━━━━━━━━━━━━━━━━━━

因为"向量化"在深度学习里通常是个好词——Python 循环慢、GPU 喜欢大矩阵，所以写代码时会本能地把 `for token in seq` 改成"一次性处理 whole seq"。在小模型、小 batch、短上下文里，这几乎总是对的。

但长上下文大模型推理不是这个账本。它的核心约束不是"这一行 PyTorch 有没有 Python loop"，而是"常驻权重之后，还剩多少峰值空间"。

DGX Spark 跑 280B，权重常驻 82GB，剩余内存看起来还有 46GB。但这 46GB 不是随便花的，还要给 KV cache、临时 score、softmax weights、通信缓冲、CUDA allocator、系统页表留空间。

所以一个"看起来只是 O(S × topk)"的中间张量，也会成为压垮峰值预算的最后一根稻草。这就是大模型工程的残酷点：

**复杂度符号不够，峰值账本才算数。**

O(S × topk) 听起来比 O(S²) 小很多。但当常驻项已经 82GB 时，那个线性项照样能把剩余的 46GB 余量吃光。

────────────────────

💡 两个 bug 的共同特征：q 太长

把两个出问题的中间张量并排放：

```text
Indexer:    [b, s, h, t]        ← s=17815, t=4458
sparse_attn: [b, s, h, topk]    ← s=17815, topk=512
```

`t` 已经被 MLA 低秩压缩从 17815 砍到 4458（÷ 4）；`topk` 已经被稀疏注意力从 4458 砍到 512（÷ 9）——所有"被查询"那一侧的维度都被工程师拼命压缩过了。

**但 `s` 在两边都是 17815，没人动**。

因为 `s` 就是用户给的 query 长度。你拿到 17815 个 token，就必须为这 17815 个 token 各算一份 attention。**稀疏 attention 稀疏的是"每个 query 看多少 KV"（`t` 或 `topk`），不是"有多少 query"（`s`）**。q 的长度是用户给的，跑不掉。

prefill 阶段（一次性吃完整 prompt）这一维度 = 整段长度；decode 阶段（一次吐一个 token）这一维度 = 1。所以 decode 几乎不会 OOM，prefill 才是真正的痛——这次 OOM 100% 是 prefill 之痛。

修法的唯一出路：既然 17815 个 query 一个不能少，那就**一次只算 256 个、再算 256 个**——把"prefill 之痛"从空间维度（同时全摊）转嫁到时间维度（分批跑）。query 维分块就是干这个的。

━━━━━━━━━━━━━━━━━━━━

◆ 真正的经验：别迷信"显存够"

━━━━━━━━━━━━━━━━━━━━

这次错误给我的教训不是"以后 sparse attention 要分块"这么窄。真正的教训是：

**大模型推理里，内存不是容量问题，是形状问题。**

同样的数学对象，有两种执行形状：第一种是"把完整中间矩阵摊开，再归约"；第二种是"分块、局部归约、只保留最终结果"。两者算的是同一个东西——但前者会死，后者能跑。

这也是为什么官方 kernel 总喜欢 fusion、tiling、streaming。不是为了炫技，而是因为很多中间张量在数学公式里存在，但在工程实现里不该真的存在。

公式可以写 `scores = Q × Kᵀ`，但机器不一定应该真的造出完整 scores；公式可以写 `weights = softmax(scores)`，但机器不一定应该真的保存完整 weights。

数学课关心等号两边。推理工程关心的是：**这个等号中间有没有临时造出一座山？** 这座山如果超过内存，等号就没有意义。

━━━━━━━━━━━━━━━━━━━━

◆ 最后的账

━━━━━━━━━━━━━━━━━━━━

这次一共踩了三个坑：

| # | 坑 | 修法 |
|---|---|---|
| 1 | `safetensors` mmap 退出后页表没及时释放 | 每个 shard 后 `gc.collect()` + `malloc_trim(0)` |
| 2 | Indexer 里 `einsum("bshd,btd->bsht")` 把 O(S²) score 完整摊开 | query 维分块，每块 topk 后释放 `score_chunk` |
| 3 | `sparse_attn` 虽然不是 O(S²)，但全 seq 向量化把 `S × topk × heads` 中间张量完整摊开 | query 维分块，默认 `chunk=512`，必要时降到 256 或 128 |

诚实做个排序——三个坑权重不一样。**真正决定 OOM 的是 2 和 3，attention 两条路径都没分块**。坑 1 的 7GB 页表在 128GB 机器上占了 5%，就算不修，Indexer 单层摊出来 20GB 中间张量照样炸。把它列在第一只是因为它在排查时序上最早被发现，不是因为它最致命——更准确的说法是"顺带也修了一个加载期的小漏"，不是"OOM 三巨头之一"。

修完后的真实账本：

```text
权重常驻：       约 82GB
17.8k prompt 峰值：90.3GB
剩余余量：       30GB+
```

所以这台机器不是不能跑——**它能跑，只是不能用"把所有中间张量摊开"的写法跑**。

这就是这次 bug 的本质：

> **不是 DeepSeek V4 太大。是我们把本来应该流过寄存器和小块 buffer 的东西，摊成了一张巨大的内存地毯。**

地毯一铺，机器就死。卷起来，一块一块算，它就活了。

工程有时候就这么简单，**也这么残酷**。

━━━━━━━━━━━━━━━━━━━━
