【DeepSeek V4 推理】一次 OOM 反转：不是模型太大，是中间张量摊得太开

━━━━━━━━━━━━━━━━━━━━

◆ 现象：17.8k token，把 128GB 机器打死了

━━━━━━━━━━━━━━━━━━━━

这次问题很朴素：两台 DGX Spark 用 NVLink 连起来，跑 DeepSeek V4 Flash 280B，权重是 FP4，单机上每个 rank 常驻大约 82GB——理论上还剩四十多 GB 余量。

短 prompt 能跑，中 prompt 能跑。但一换成长 prompt（约 17815 token 的 awakening.md），机器直接 OOM。而且不是 Python 抛异常那种温柔的 OOM，是 NVRM OOM，整个系统被统一内存拖死。

最开始的直觉很自然——模型太大、280B、128GB 统一内存、长上下文。这几个词凑在一起，所有人都会下意识觉得"没办法，硬件不够"。

但工程排障最怕的就是这个"没办法"。"没办法"很多时候不是事实，是脑子懒了。

这次最后证明：**硬件够。真正的问题是我们把某个中间张量一次性摊开了。**

摊开以后，内存曲线看起来像"模型太大"——但那不是模型大，是算法写法大。

━━━━━━━━━━━━━━━━━━━━

◆ 先把加载期嫌疑排掉

━━━━━━━━━━━━━━━━━━━━

在排查 forward OOM 之前，加载期已经修过一个非常隐蔽的问题，先讲清楚——免得这次 OOM 被甩锅到这上面。

`safetensors` 流式加载 46 个 shard，代码看起来是这样的：

```python
with safe_open(shard_file, framework="pt", device="cpu") as f:
    ...
```

直觉上，`with` 结束 mmap 就该释放。但实际不是——46 个 shard 关掉之后，虚拟地址空间还在涨。更要命的是，虚拟地址空间本身虽然不等于物理内存，但**页表（page table）要占物理内存**。最后页表能涨到 **7GB 多**。

在普通服务器上这可能只是看着难看，在 DGX Spark 这种 CPU/GPU 共享 128GB 内存的机器上，7GB 页表就是杀人。

修法很土：

```python
gc.collect()
ctypes.CDLL("libc.so.6").malloc_trim(0)
```

每个 shard 结束后强制把 glibc arena 还给内核。修完之后，加载期日志稳定下来：

```text
shard 1/46  done  GPU=83.6GB  RSS=1.0GB  VM=102GB  PgT=3MB
shard 5/46  done  GPU=83.1GB  RSS=1.1GB  VM=112GB  PgT=4MB
...
shard 46/46 done  GPU=82.0GB  RSS=0.5GB  VM=112GB  PgT=4MB
```

VM 很大（112GB），但 RSS（0.5GB）和 PgT（4MB）很小——说明 mmap 残留的虚拟地址不再吃真实物理页。

所以加载期可以排除。OOM 不在"权重怎么读进来"，OOM 在 forward。

━━━━━━━━━━━━━━━━━━━━

◆ 第一个真 bug：Indexer 的 O(S²)

━━━━━━━━━━━━━━━━━━━━

DeepSeek V4 的注意力不是传统全量 KV attention，它有压缩、有索引、有 sparse attention。其中 **Indexer** 要根据当前 query，从压缩后的 KV 里选 top-k。

原始路径里有一行看起来非常普通：

```python
index_score = torch.einsum("bshd,btd->bsht", q, kv_cache[:, :T])
```

💡 打个比方，先翻译成人话：

- 输入：`q = [batch, seqlen, heads, head_dim]`，`kv_cache = [batch, compressed_len, head_dim]`
- 操作：每个 token 的每个 head，都和压缩 KV 里的每个位置做点积
- 输出：`index_score = [batch, seqlen, heads, compressed_len]`

这就是问题。在 prefill 阶段 `seqlen = S`、`compressed_len ≈ S / 4`，所以这个张量的规模是 `S × heads × S/4`——也就是 **O(S²)**。17.8k token 时，单层就能摊出几十 GB 量级的中间张量。

这不是"模型太大"，是"把不该完整出现的矩阵完整实例化了"。

官方 TileLang kernel 里这一步是 fused + block streaming——算一块、归约、丢掉、再算下一块，不会把整个 `index_score` 放到显存里。我们在 sm_121 上替换 kernel 时，漏掉了这条路径。

所以第一刀是把 Indexer 改成 query 维分块——一次只处理 256 个 query token，`score_chunk = [batch, 256, heads, compressed_len]`，算完 topk 立刻释放 `score_chunk`。

这刀修完后，短、中 prompt 能过了，但长 prompt 还是 OOM——说明还有第二个真凶。

━━━━━━━━━━━━━━━━━━━━

◆ 第二个真凶：sparse_attn 不是 O(S²)，但照样能杀人

━━━━━━━━━━━━━━━━━━━━

Indexer 修完以后，重新跑四个 stage：

| stage | 输入长度 | 结果 |
|---|---|---|
| stage1_short | 18 tokens | OK |
| stage2_1k | 416 tokens | OK |
| stage3_6k | 2416 tokens | OK，peak 86.9GB |
| stage4_long | 17815 tokens | **OOM** |

这时候如果只盯 O(S²) 会误判——Indexer 已经分块了，那为什么 stage4 还死？

看 sparse attention。Indexer 选完 top-k 之后，真正的 attention 要做的是：每个 query token 只看 topk 个 KV 位置。这已经不是全量 attention，规模不是 `S × S`，而是 `S × topk`，听起来安全多了。

但问题在这里：

```python
kv_gathered = [b, s, topk, d]
scores      = [b, s, h, topk]
weights     = [b, s, h, topk + 1]
```

旧实现为了快，把整个 seqlen 一次性向量化——没有 Python 循环、GPU 利用率好看、代码也漂亮。但它把 17.8k token 的 `kv_gathered`、`scores`、`weights` 全摊开了。

这里要注意一个反直觉点：**不是只有 O(S²) 才会 OOM。在大模型推理里，O(S × topk × heads) 也够杀人。**

因为常驻权重已经吃了 82GB——你不是从 0GB 开始分配中间张量，是站在 82GB 的悬崖边上，再往上摊几 GB、十几 GB。一旦加上 CUDA reserved、all-reduce 同步缓冲、allocator 碎片、两台 worker 的时序差，看起来"理论上还够"的内存，就会被推下去。

━━━━━━━━━━━━━━━━━━━━

◆ 修法：不是少算，是别一次性摊开

━━━━━━━━━━━━━━━━━━━━

修法非常直接——把 sparse attention 也按 query 维分块。原来一次处理完整 seqlen，现在一次处理 512 个 query token。

伪代码：

```python
for s_start in range(0, seqlen, 512):
    q_chunk = q[:, s_start:s_start+512]
    idx_chunk = topk_idxs[:, s_start:s_start+512]

    kv_gathered = gather(kv, idx_chunk)
    scores = q_chunk @ kv_gathered
    weights = softmax(scores + attn_sink)
    out_chunk = weights @ kv_gathered

    o[:, s_start:s_start+512] = out_chunk
```

注意：**这不是近似**。不是少看 token、不是降低 topk、不是砍模型。它只是改变了执行顺序——原来"把所有中间结果同时放在内存里"，现在"算一块、写回输出、释放、再算下一块"。

数学结果不变。小张量对照实测：

```text
batch=1  max_abs_diff=0.000000
batch=2  max_abs_diff=0.000000
```

和旧的 vectorized 实现完全一致。这就是工程里最舒服的一类修复：**不改语义，只改峰值。**

━━━━━━━━━━━━━━━━━━━━

◆ 实测：stage4 从死机变成 90.3GB

━━━━━━━━━━━━━━━━━━━━

改完以后重跑：

```bash
SPARSE_ATTN_CHUNK=512 SCRIPT=probe_awakening_min.py ./run_dual_node.sh
```

结果：

```text
stage1_short  18 tokens     OK | 5.0s  | peak_alloc=83.0GB
stage2_1k     416 tokens    OK | 2.8s  | peak_alloc=83.0GB
stage3_6k     2416 tokens   OK | 10.4s | peak_alloc=83.3GB
stage4_long   17832 tokens  OK | 79.9s | peak_alloc=90.3GB
```

对比修复前：stage3 从 86.9GB 降到 83.3GB，省掉 3.6GB；stage4 原来直接死机，现在只到 90.3GB。

常驻权重大约 82GB，也就是说长 prompt 在常驻之上的额外峰值只有 **8GB 左右**，128GB 机器上还剩三十多 GB 余量。

所以结论可以钉死：**stage4 OOM 的主因不是模型装不下，而是 sparse_attn 把长 prompt 的中间张量一次性摊开了。**

━━━━━━━━━━━━━━━━━━━━

◆ 为什么这个错误这么容易犯

━━━━━━━━━━━━━━━━━━━━

因为"向量化"在深度学习里通常是个好词——Python 循环慢、GPU 喜欢大矩阵，所以写代码时会本能地把 `for token in seq` 改成"一次性处理 whole seq"。在小模型、小 batch、短上下文里，这几乎总是对的。

但长上下文大模型推理不是这个账本。它的核心约束不是"这一行 PyTorch 有没有 Python loop"，而是"常驻权重之后，还剩多少峰值空间"。

DGX Spark 跑 280B，权重常驻 82GB，剩余内存看起来还有 46GB。但这 46GB 不是随便花的，还要给 KV cache、临时 score、softmax weights、通信缓冲、CUDA allocator、系统页表留空间。

所以一个"看起来只是 O(S × topk)"的中间张量，也会变成真正的凶器。这就是大模型工程的残酷点：

**复杂度符号不够，峰值账本才算数。**

O(S × topk) 听起来比 O(S²) 小很多。但当常驻项已经 82GB 时，那个线性项照样能把你杀了。

━━━━━━━━━━━━━━━━━━━━

◆ 真正的经验：别迷信"显存够"

━━━━━━━━━━━━━━━━━━━━

这次错误给我的教训不是"以后 sparse attention 要分块"这么窄。真正的教训是：

**大模型推理里，内存不是容量问题，是形状问题。**

同样的数学对象，有两种执行形状：第一种是"把完整中间矩阵摊开，再归约"；第二种是"分块、局部归约、只保留最终结果"。两者算的是同一个东西——但前者会死，后者能跑。

这也是为什么官方 kernel 总喜欢 fusion、tiling、streaming。不是为了炫技，而是因为很多中间张量在数学公式里存在，但在工程实现里不该真的存在。

公式可以写 `scores = Q × Kᵀ`，但机器不一定应该真的造出完整 scores；公式可以写 `weights = softmax(scores)`，但机器不一定应该真的保存完整 weights。

数学课关心等号两边。推理工程关心的是：**这个等号中间有没有临时造出一座山？** 这座山如果超过内存，等号就没有意义。

━━━━━━━━━━━━━━━━━━━━

◆ 最后的账

━━━━━━━━━━━━━━━━━━━━

这次一共踩了三个坑：

| # | 坑 | 修法 |
|---|---|---|
| 1 | `safetensors` mmap 退出后页表没及时释放 | 每个 shard 后 `gc.collect()` + `malloc_trim(0)` |
| 2 | Indexer 里 `einsum("bshd,btd->bsht")` 把 O(S²) score 完整摊开 | query 维分块，每块 topk 后释放 `score_chunk` |
| 3 | `sparse_attn` 虽然不是 O(S²)，但全 seq 向量化把 `S × topk × heads` 中间张量完整摊开 | query 维分块，默认 `chunk=512`，必要时降到 256 或 128 |

修完后的真实账本：

```text
权重常驻：       约 82GB
17.8k prompt 峰值：90.3GB
剩余余量：       30GB+
```

所以这台机器不是不能跑——它能跑，只是不能用"把所有中间张量摊开"的写法跑。

这就是这次 bug 的本质：

> **不是 DeepSeek V4 太大。是我们把本来应该流过寄存器和小块 buffer 的东西，摊成了一张巨大的内存地毯。**

地毯一铺，机器就死。卷起来，一块一块算，它就活了。

工程有时候就这么简单，也这么残酷。

━━━━━━━━━━━━━━━━━━━━
