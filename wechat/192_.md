【机器人AI】VLA 综述：从 RT-2 到 Pi0，大模型怎么长出手臂

━━━━━━━━━━━━━━━━━━━━

◆ 前情提要

━━━━━━━━━━━━━━━━━━━━

第 146 期我们聊了机器人 AI 的入门——MuJoCo 物理引擎、Diffusion Policy 扩散策略、Push-T 推积木。那一期讲清楚了"怎么控制"：给模型一张图，它输出关节位移。

但有个问题没碰：**模型只会看图做动作，听不懂人话。**

你对着 Diffusion Policy 说"把红色杯子递给我"，它不知道你在说什么。它只能看当前画面，然后重复训练时学到的动作模式。没有语言输入，就没有灵活性——每换一个任务就得重新采集数据、重新训练。

VLA（Vision-Language-Action）就是为了解决这件事：把"看懂"和"听懂"和"动手"连成一条链。

━━━━━━━━━━━━━━━━━━━━

◆ VLA 是什么

━━━━━━━━━━━━━━━━━━━━

VLA = Vision + Language + Action，视觉-语言-动作三合一模型。

```
输入：摄像头画面 + 语言指令（"把红色杯子放到左边"）
输出：机器人动作序列（每个关节的角度/位移）
```

和纯 Diffusion Policy 的区别一张表讲清楚：

| | Diffusion Policy | VLA |
|---|---|---|
| 视觉输入 | 有 | 有 |
| 语言输入 | 无 | 有 |
| 输出 | 连续动作 | 连续动作 或 离散 token |
| 换任务 | 重新训练 | 换一句指令就行 |
| 典型模型 | 250M | 450M ~ 55B |

VLA 的本质：**在多模态大模型的基础上接一个动作输出头。** 大模型负责理解图像和语言，输出头负责把理解转化成机器人动作。

但这里有一个根本性的技术矛盾。

━━━━━━━━━━━━━━━━━━━━

◆ 核心矛盾：离散 vs 连续

━━━━━━━━━━━━━━━━━━━━

大语言模型的输出是**离散**的。它从一个词表（比如 32000 个 token）里选一个概率最高的 token 输出。这是分类问题——softmax 接 argmax，从有限选项里选一个。

机器人的动作是**连续**的。一个 6-DOF 机械臂的关节角度是 6 个实数，比如 [0.314, -1.207, 0.853, -0.042, 1.571, 0.000]，每个值可以是任意浮点数。这是回归问题。

矛盾来了：**怎么把一个离散的语言模型，接到一个连续的运动控制系统上？**

这是 VLA 领域过去三年的核心问题。围绕这个问题，产生了两条技术路线：

```
路线 A（离散化）：把连续动作量化成离散 token，让语言模型直接预测
路线 B（混合架构）：语言模型负责理解，另接一个连续生成器负责动作
```

下面按时间线，看这两条路线怎么各自发展、最终走向统一。

━━━━━━━━━━━━━━━━━━━━

◆ RT-2（Google, 2023.07）：第一个真正的 VLA

━━━━━━━━━━━━━━━━━━━━

RT-2 走的是路线 A——把动作变成 token。

思路非常暴力：机器人有 7 个自由度（6 个关节 + 1 个夹爪），每个自由度的值域是一个连续区间。RT-2 把每个自由度的连续区间均匀切成 256 份（0~255），每份对应一个整数。这样一个 7 维连续动作就变成了 7 个整数——7 个 token。

────────────────────

**量化过程**

假设某个关节的角度范围是 [-1.0, 1.0]：

```
连续值 0.314 → 量化 → round((0.314 - (-1.0)) / (1.0 - (-1.0)) × 255) = round(167.3) = 167
连续值 -0.042 → 量化 → round((-0.042 - (-1.0)) / (1.0 - (-1.0)) × 255) = round(122.2) = 122
```

每个关节一个 0~255 的整数，7 个关节就是 7 个 token。模型输出的序列变成：

```
[语言 token...] [图像 token...] → [167, 58, 201, 122, 200, 128, 255]
                                    ↑ 这 7 个数就是机器人动作
```

────────────────────

这样做的好处：**概念极其简单。** 动作变成了 token，训练就是标准的下一个 token 预测（next token prediction），和训语言模型完全一样。所有大模型的基础设施——注意力机制、位置编码、训练框架——直接复用，不需要改任何东西。

RT-2 用的骨架是 PaLI-X（55B 参数的视觉语言模型），在 Google 的机器人数据上做了联合微调。

────────────────────

**问题出在哪？**

1. **精度受限。** 256 级量化意味着每个关节的分辨率是值域的 1/256。对于范围 [-1.0, 1.0] 的关节，量化误差约 0.008 弧度（约 0.5 度）。听起来不多，但末端执行器的误差会被臂长放大——0.5 度的关节误差在 50cm 臂长末端可能变成几毫米偏移，精细操作（比如插 USB 接口）就废了。

2. **55B 太大，推理太慢。** 机器人控制的实时性要求很高——通常需要 5~20Hz 的控制频率（每秒输出 5~20 次动作）。55B 模型即使在 TPU 集群上推理也很慢，延迟太高，机器人动作就卡顿。

3. **多模态动作分布表达不了。** 这是最根本的问题。

────────────────────

**什么是多模态动作分布？**

同一个场景下，机器人可能有多种"正确"的动作。比如桌上有一个杯子，指令是"把杯子拿起来"——你可以从左边抓，也可以从右边抓，也可以从上面罩下去。三种动作都对。

动作的概率分布不是单峰的（只有一个最优解），而是多峰的（有多个同样好的解）：

```
概率
 ↑
 |   *           *           *
 |  * *         * *         * *
 | *   *       *   *       *   *
 |*     *     *     *     *     *
 +-------------------------------→ 动作空间
   从左抓     从上罩      从右抓
```

token 预测（softmax + argmax）本质上只能输出一个最可能的 token。面对多峰分布，它会输出三个峰的"平均"——一个介于左抓和右抓之间的折中动作，这个折中动作可能恰好是所有峰之间的"谷底"（概率最低的位置），比任何一个正确答案都差。

这叫**模式平均（mode averaging）** 问题。离散 token 预测天然有这个缺陷。

━━━━━━━━━━━━━━━━━━━━

◆ OpenVLA（Stanford, 2024.06）：开源的 7B VLA

━━━━━━━━━━━━━━━━━━━━

RT-2 的思路对了，但 55B 太重。Stanford 和 Toyota Research Institute 合作的 OpenVLA 把同样的思路做小做开源。

架构：
- 视觉编码器：SigLIP（语义特征）+ DinoV2（空间特征），双流并行
- 语言骨架：Llama 2 7B
- 动作输出：仍然是离散 token 预测（和 RT-2 同路线）

训练数据：Open X-Embodiment 数据集，970K 条机器人操作轨迹，跨 22 种机器人形态。

────────────────────

关键结果：**7B 参数，在多个基准上比 55B 的 RT-2 高 16.5%。**

为什么能小这么多、性能还高？

1. **视觉编码器更好。** RT-2 用的是 2022 年的 ViT，OpenVLA 用了 2024 年的 SigLIP + DinoV2。视觉特征提取能力提升了一代。

2. **语言模型在机器人任务上有巨大冗余。** 大语言模型的参数主要用来存储世界知识（历史、文学、代码……），但机器人控制只需要很小一部分——空间关系理解、物体属性识别、指令解析。7B 对这些任务绰绰有余。

3. **数据质量。** Open X-Embodiment 是精心整理的多机器人数据集，质量比 RT-2 用的单一机器人数据高。

────────────────────

但 OpenVLA 仍然是离散路线，仍然有 RT-2 的根本问题：精度受限、多模态分布表达不了。它证明了一件事——**VLA 的语言理解部分不需要很大，瓶颈不在语言模型的参数量上。** 瓶颈在动作输出的方式上。

━━━━━━━━━━━━━━━━━━━━

◆ Diffusion Policy（Columbia, 2023.03）：连续路线的奠基

━━━━━━━━━━━━━━━━━━━━

在 VLA 领域忙着把动作变 token 的同时，另一条路线在旁边悄悄成熟了。

Columbia 大学的 Diffusion Policy 不是 VLA（它没有语言输入），但它证明了一件关键的事：**连续动作生成比离散方案好得多。**

核心机制（第 146 期讲过，这里快速回顾）：

```
训练时：
  正确动作序列 → 逐步加噪声 → 变成纯高斯噪声
  训一个去噪网络，学会把噪声还原成动作

推理时：
  采样纯噪声 → 去噪网络反复迭代去噪 → 输出动作序列
```

在 11 个 benchmark 任务上，Diffusion Policy 比离散动作预测（BeT，Behavior Transformer）**平均高 46.9%**。

为什么高这么多？就是因为多模态分布。扩散模型从噪声出发，每次采样可能收敛到分布的不同峰上——左抓、右抓、从上罩，都是合法的采样结果。它天然就能处理多峰分布，不存在模式平均问题。

────────────────────

但 Diffusion Policy 的局限也很明显：**没有语言输入。** 它只能学一个固定任务（推积木、倒水），不能通过语言指令切换任务。

所以问题变成了：**能不能把 VLM（视觉语言模型）的理解能力和 Diffusion Policy 的连续动作生成能力结合起来？**

这就是 Pi0 干的事。

━━━━━━━━━━━━━━━━━━━━

◆ Pi0（Physical Intelligence, 2024.10）：两条路线的统一

━━━━━━━━━━━━━━━━━━━━

Pi0 是 Physical Intelligence 公司（由前 Google Brain 机器人组核心成员创立）2024 年 10 月发布的模型。它的架构设计直接回答了上面的问题：**不再让语言模型输出离散 token 来表示动作，而是让一个专门的连续生成器消费语言模型的内部表征。**

────────────────────

**架构分两部分：**

```
┌─────────────────────────────────┐
│  VLM（PaliGemma 3B）            │
│  输入：图像 + 语言指令           │
│  输出：不是 token！              │
│        是中间层的连续向量表征     │  ← "中间层截胡"
└───────────────┬─────────────────┘
                │ 连续向量（不经过最后的 softmax）
                ▼
┌─────────────────────────────────┐
│  Flow Matching 动作专家（300M）  │
│  输入：VLM 内部表征 + 噪声动作  │
│  输出：连续动作序列              │
└─────────────────────────────────┘
```

关键设计：VLM 的输出**不经过最后的 token 预测层（softmax + 词表映射）**。传统做法是让 VLM 一路算到最后一层、输出 token，再把 token 当作动作。Pi0 在 VLM 的中间层就把表征抽走了——在语言坍缩（离散化）发生之前，把连续的"意图向量"喂给动作专家。

**这就是"中间层截胡"。**

────────────────────

为什么要截胡？

VLM 内部的表征是连续的高维向量（每个位置是一个浮点数向量）。这些向量编码了模型对图像和语言的理解——"红色杯子在左边"、"要把它放到右边"、"路径上没有障碍"。这些信息本身是连续的、丰富的。

如果让 VLM 一路输出到 token，所有连续信息都被压缩成一个离散符号。信息损失巨大。而动作生成器需要的恰好是连续的、细粒度的空间信息——关节该转多少度，力该用多大。

中间层截胡保留了 VLM 的连续表征，直接传递给动作生成器，避免了离散瓶颈。

────────────────────

**Flow Matching 动作专家**

动作专家使用的不是传统的 Diffusion（扩散），而是 Flow Matching（流匹配）。两者目标一样——从噪声生成连续数据——但路径不同：

```
Diffusion（扩散）：
  噪声 → 数据 的路径是随机的（随机微分方程 SDE）
  每一步加的噪声也是随机的
  需要很多步（通常 50~1000 步）才能收敛

Flow Matching（流匹配）：
  噪声 → 数据 的路径是直线（常微分方程 ODE）
  从噪声点直接画一条直线到数据点
  路径简单，所以步数可以少很多（Pi0 用 10 步）
```

数学上，Flow Matching 学的是一个速度场 v(x, t)，描述在时间 t、位置 x 处数据应该往哪个方向流动：

```
dx/dt = v(x, t)

其中：
  x 是当前的"噪声动作"
  t 从 0（纯噪声）到 1（干净动作）
  v 是神经网络预测的速度向量
```

训练目标是让 v(x, t) 预测从噪声到数据的直线方向：

```
Loss = || v(x_t, t) - (x_1 - x_0) ||²

其中：
  x_0 是纯噪声（从标准正态分布采样）
  x_1 是真实动作（训练数据里的正确动作）
  x_t = (1-t)·x_0 + t·x_1 是直线插值（t 时刻的中间状态）
```

loss 就是均方误差——让网络预测的速度方向尽可能接近真实的"噪声到数据"方向。

────────────────────

推理时，从纯噪声 x_0 出发，用欧拉法积分 10 步：

```
for i in range(10):
    t = i / 10
    x = x + (1/10) * v(x, t)    # 每步沿速度方向走 1/10
```

10 步后得到的 x 就是最终的连续动作序列。

和 Diffusion 的 50~1000 步相比，10 步快了一个数量级。这对实时控制至关重要。

────────────────────

**Pi0 的训练流程**

两阶段：

1. **预训练**：在大规模多机器人数据集上训练（包含单臂、双臂、灵巧手等多种形态），让 VLM 和动作专家学会通用的视觉-语言-动作映射

2. **微调**：在特定任务的少量高质量数据上微调。论文报告用 100~1000 条演示数据微调后就能达到很好的效果

────────────────────

**效果**

Pi0 在多个灵巧操作任务上的成功率：

| 任务 | Pi0 | 之前最好的方法 |
|------|-----|---------------|
| 折叠衣物 | 80% | ~20% |
| 装配乐高 | 无人做过（需要毫米级精度） | - |
| 收拾桌面 | 85% | ~50% |

折叠衣物这种需要处理柔性物体、多步操作的任务，之前的方法基本做不好。Pi0 靠连续动作生成的高精度 + VLM 的语义理解，第一次把这类任务的成功率做到可用水平。

◆ 为什么 Flow Matching 而不是 Diffusion

━━━━━━━━━━━━━━━━━━━━

这个问题值得单独说一下，因为很多人会混淆。

两者的核心区别不在最终效果（差不多），而在路径的简洁性：

| | Diffusion | Flow Matching |
|---|---|---|
| 数学框架 | 随机微分方程（SDE） | 常微分方程（ODE） |
| 噪声到数据的路径 | 弯曲的随机路径 | 直线 |
| 典型步数 | 50~1000 | 5~20 |
| 推理速度 | 慢 | 快 |
| 训练目标 | 预测噪声 | 预测速度方向 |
| 生成质量 | 高 | 高（略优或持平） |

对机器人控制来说，推理速度是硬约束。你不能让机器人等 1 秒才能动一下。Flow Matching 10 步积分就能出结果，延迟可以压到 50ms 以内，满足 20Hz 控制频率的要求。

直觉类比：Diffusion 像是在大雾里走迷宫，每一步都带随机性，要走很多步才能到出口。Flow Matching 像是在清晰的地图上画直线导航，几步就到。终点一样，但路程差很多。

◆ 后续发展：2025 年的加速

━━━━━━━━━━━━━━━━━━━━

Pi0 之后，VLA 领域在 2025 年爆发了。几个重要的工作：

────────────────────

**Pi0.5（Physical Intelligence, 2025.04）**

Pi0 的升级版。核心改进：长程开放世界泛化。Pi0 能做好单步操作（"把杯子放到左边"），Pi0.5 开始处理多步复合任务（"去厨房，打开冰箱，拿出牛奶，倒进杯子"）。加入了高层规划模块，能把复杂指令分解成子任务序列。

────────────────────

**GR00T N1（NVIDIA, 2025.03）**

NVIDIA 的方案走了"双系统"架构——借鉴认知科学的快思考/慢思考：

```
System 1（快）：扩散策略，~50ms 输出动作，负责实时运动控制
System 2（慢）：LLM 规划器，~500ms 输出子目标，负责任务分解和决策
```

System 2 的 LLM 思考一个大方向（"先伸手靠近杯子"），System 1 的扩散策略在这个大方向下生成精细的关节动作。两个系统运行在不同频率上——LLM 慢慢想，扩散策略快速执行。

────────────────────

**Gemini Robotics（Google, 2025.03）**

基于 Gemini 2.0，Google 把自己最大的多模态模型直接接上了机器人动作输出。具体技术细节还没完全公开，但方向是"用最大的模型、最多的数据，暴力出奇迹"。和 Pi0 的精巧设计形成对比。

────────────────────

**SmolVLA（HuggingFace, 2025）**

极端小型化的尝试。只有 450M 参数，MacBook CPU 都能跑推理。证明了 VLA 不一定要大——对于简单任务（桌面抓取、推物体），小模型足够。

────────────────────

**Discrete Diffusion VLA（2025）**

一个有趣的尝试：在离散空间做扩散。不是把动作连续化再扩散，而是在离散 token 空间里做一种"离散去噪"——从随机 token 出发，逐步替换成正确的 token。试图在离散路线上也获得扩散模型处理多模态分布的能力。目前还在早期，效果待验证。

◆ 演进主线总结

━━━━━━━━━━━━━━━━━━━━

画一条时间线：

```
2023.03  Diffusion Policy ── 证明连续动作生成 >> 离散
           │
2023.07  RT-2 ── 第一个 VLA，离散路线（动作→token），55B
           │
2024.06  OpenVLA ── RT-2 思路做小，7B > 55B，开源
           │
2024.10  Pi0 ── 两条路线统一：VLM 理解 + Flow Matching 生成
           │
2025.03  GR00T N1 / Gemini Robotics ── 工业化、多系统
           │
2025.04  Pi0.5 ── 长程复合任务
           │
2025     SmolVLA ── 极端小型化，450M
```

整条演进线的逻辑：

1. **起点**：大模型能理解语言和图像，但输出是离散 token
2. **第一步（RT-2）**：暴力统一——把动作也变成 token，用同一套机制
3. **发现问题**：离散输出精度不够、速度不够、多模态分布处理不了
4. **另一条路（Diffusion Policy）**：连续生成好得多，但没有语言
5. **统一（Pi0）**：VLM 负责理解（保留语言能力），Flow Matching 负责生成（连续输出）
6. **工程化（2025）**：更大（Gemini）、更快（GR00T 双系统）、更小（SmolVLA）

**VLA 的演进就是一场"离散 vs 连续"的拉锯战。** RT-2 用离散 token 统一了语言和动作，但精度和速度都受限；Pi0 用 Flow Matching 在连续空间生成动作，同时用"中间层截胡"保留了 VLM 的语言理解能力。

目前看来，Pi0 的"理解用离散模型，生成用连续模型"的混合范式正在成为主流。但离散路线也没有死——Discrete Diffusion 等方法在探索离散空间里的多模态生成，两条路线最终可能殊途同归。
