【AI的子空间】200行Python复刻Kimi Agent集群——小模型的提示词陷阱

━━━━━━━━━━━━━━━━━━━━

◆ 起因

━━━━━━━━━━━━━━━━━━━━

在 B 站看到 Kimi K2.6 的 Agent 集群演示：输入"调研 100 个城市的美食"，蹦出一堆角色卡——"成都美食探员""长沙辣味特工"——每个 agent 独立干活，最后汇总成报告。评论区都在喊"太帅了"。

花了 199 块订了 Kimi 的 Allegretto 计划，亲手试了一圈。角色卡确实好看，拆解过程也流畅。但日常干活用的是御三家（Claude Max、GPT Pro、Gemini Ultra），Kimi 的订阅纯粹是为了搞清楚它到底做了什么。

搞清楚之后，下一个问题就是：能不能用开源模型复刻一个？

答案是能。用 Qwen3.6-27B（硅基流动 API，也可以本地 GPU 跑）+ FastAPI + 单文件 HTML，大约 200 行 Python 就能跑。

188 期（ https://mp.weixin.qq.com/s/jcQlCc3XtaCEEnjUaxw93w ）拆过 Qwen3.6-27B 的架构——75% Gated DeltaNet + 25% 标准 GQA 的混合注意力，编程/Agent benchmark 打赢了上一代旗舰 Qwen3.0-397B MoE，参数少了 15 倍。195 期（ https://mp.weixin.qq.com/s/C-vsE_ceIv6FYVzeZWFBYQ ）用 eRank+TwoNN 测了它的权重几何结构。那两期是"它为什么强"，这期是"拿它干活时踩了什么坑"。

三个坑都指向同一件事：**27B 级别的小模型在 multi-agent 调度场景下的行为边界。**

代码开源：`https://github.com/lmxxf/agent-swarm-on-personal-gpu`

━━━━━━━━━━━━━━━━━━━━

◆ Kimi Agent 集群做了什么

━━━━━━━━━━━━━━━━━━━━

K2.6 支持 300 个子 agent 并行、4000 步协作。产品形态是角色卡 + 任务树 + 实时进度条。技术上的核心是三件事：

1. **调度 LLM 把大任务拆成子任务**（fork）
2. **多个 LLM 调用同时跑**（parallel execution）
3. **调度 LLM 合并子结果**（join）

本质是 fork/join 并行模型的产品化包装。角色卡和工牌动画是锦上添花——截图传播力很强，但不影响底层机制。

从拓扑结构看，Kimi 做的是 DAG（有向无环图）依赖调度：子 agent 之间可以有依赖关系，A 的输出喂给 B。我们做的是更简单的**星型拓扑**：所有子 agent 互不通信，各自独立执行，最后统一汇总。星型拓扑覆盖了大多数"尴尬并行"场景——20 个城市的美食调研，城市之间不需要通信。

━━━━━━━━━━━━━━━━━━━━

◆ 架构：浏览器 -- SSE -- FastAPI -- async -- LLM API

━━━━━━━━━━━━━━━━━━━━

整个系统只有一个 server.py 和一个 index.html。

数据流：

```
浏览器
  |  POST /api/task （提交任务）
  |  GET  /api/task/{id}/stream （SSE 长连接）
  v
FastAPI
  |  Step 1: decompose_task() → 调用 LLM 拆任务
  |  Step 2: asyncio.gather() → 并发调用 N 个 LLM
  |  Step 3: call_llm(summary) → 串行汇总
  v
OpenAI-compatible API（硅基流动 / 本地 vLLM）
```

SSE（Server-Sent Events）是关键选择：浏览器端不需要 WebSocket 的复杂握手，服务端 yield 一个 dict 就推送一条消息。每个 agent 创建、开始运行、完成，都是一个 SSE 事件，前端实时渲染。

━━━━━━━━━━━━━━━━━━━━

◆ 并发控制：Semaphore + asyncio.gather

━━━━━━━━━━━━━━━━━━━━

20 个 agent 同时启动意味着 20 路并发 LLM 调用。云端 API 会被限流，本地 GPU 问题更大——每路请求都要独占一份 KV cache，20 路同时跑的显存根本装不下。

做法就几行：

```python
semaphore = asyncio.Semaphore(8)             # 最多 8 路同时在飞

async def call_llm(messages):
    async with semaphore:                     # 第 9 个请求会在这里等
        return await client.chat.completions.create(model=MODEL, messages=messages)

await asyncio.gather(*(run_agent(sub) for sub in subtasks))  # 全部并发启动
```

Semaphore 卡住的请求会等前面的完成再上。

对云端 API 来说这防限流；对本地 GPU 来说这是生死线——8 路并发的 KV cache 已经是单卡显存的极限，放开到 20 路直接 OOM。即使用 vLLM 的 PagedAttention 做动态显存管理，显存不够时也要把部分请求的 KV cache 换出，下次轮到它时重新 prefill——并发越多，换出越频繁，每路请求的延迟越高。

◆ 27B 模型的提示词适配：三个坑

━━━━━━━━━━━━━━━━━━━━

这是文章最有价值的部分。大模型（70B+）能"听话"的提示词，在 27B 上会出各种意外。

────────────────────

### 坑一：模型偷懒合并子任务

提示词说"调研 20 个城市的美食，每个城市一个 agent"。期望拆出 20 个子任务。

27B 模型反复只拆出 5 个——按地区合并：
- "华北美食研究员"（负责北京、天津、石家庄）
- "华东美食研究员"（负责上海、杭州、南京）
- ...

加了"绝对不要合并""N 个城市就是 N 个子任务"，时灵时不灵。同一个 prompt，curl 命令行测试出 21 个，前端跑出 5 个——因为采样的随机性（temperature > 0 时每次生成不同）。

**根因**：27B 模型在 JSON 输出模式下有"偷懒"倾向。生成 20 个结构化对象比生成 5 个费力得多，模型会倾向于找捷径合并，尤其当指令里同时有"合理拆解"和"每个城市一个"这种看起来矛盾的要求时。

────────────────────

### 坑二：汇总 agent 不能和其他 agent 并行

最初设计是让 LLM 拆任务时自己加一个"汇总编辑"角色，和其他 20 个 agent 一起并行启动。

结果汇总 agent 跑的时候其他人还没出结果，等于空转——它拿不到任何输入，只能编一些不存在的内容。

**解法**：去掉汇总 agent，改为所有 agent 完成后串行调用汇总。在代码里直接过滤掉 LLM 自作主张加的汇总角色：

```python
subtasks = [s for s in subtasks if "汇总" not in s.get("role_name", "")]
```

这是星型拓扑最简单的约束：没有依赖关系的节点并行，有依赖关系的节点串行。如果要做 DAG 依赖（A 完成后 B 才启动），需要拓扑排序 + 事件驱动调度，复杂度上一个台阶。

────────────────────

### 坑三：汇总步骤超时

20 个 agent 的结果拼起来 token 量很大。默认超时 120 秒不够，汇总步骤频繁报 timeout。

**解法**：汇总步骤单独加大超时到 300 秒：

```python
summary = await call_llm([
    {"role": "system", "content": SUMMARY_SYSTEM},
    {"role": "user", "content": f"原始任务：{prompt}\n\n各专家结果：\n\n{all_results_text}"},
], timeout=300)   # 汇总专用超时
```

看起来是工程问题不是 AI 问题，但它暴露了一个设计原则：**multi-agent 系统里不同阶段的资源预算应该独立配置。** 调度阶段需要快响应（2-5 秒），执行阶段需要中等超时（30-60 秒），汇总阶段需要长超时（300 秒）+ 大 context window。

◆ 双策略拆解：最终方案

━━━━━━━━━━━━━━━━━━━━

三个坑里最难的是坑一。最终方案是**双策略拆解**：

```python
expected = extract_expected_count(prompt)           # 正则提取"20个"→ 20
subtasks = llm_plan(prompt)                         # 策略一：让 LLM 直接拆

if expected and len(subtasks) < expected * 0.6:     # 拆少了？走策略二补救
    entities = llm_extract_entities(prompt)          # 只让 LLM 列清单
    subtasks = [make_agent(e) for e in entities]     # 代码机械生成 agent

subtasks = [s for s in subtasks if "汇总" not in s["role_name"]]  # 过滤汇总角色
```

核心思路：

- **策略一**（PLAN_SYSTEM prompt）：让 LLM 自己拆子任务。灵活，什么任务都能处理。但 27B 模型在实体类任务上容易偷懒合并。
- **策略二**（EXTRACT_SYSTEM prompt）：让 LLM 只列实体清单——"北京、上海、广州..."。这是一个更简单的任务，模型输出更稳定。然后代码机械地为每个实体生成 agent。
- **切换条件**：从用户 prompt 里正则提取期望数量（"20 个城市"→ 20），如果策略一拆出的数量不到期望的 60%，自动走策略二。

两个 system prompt 的设计意图完全不同：

PLAN_SYSTEM 要求模型做"任务拆解"——理解任务结构、分配角色、写具体指令。这对 27B 来说是一个复杂的推理任务。

EXTRACT_SYSTEM 只要求模型做"实体列举"——列出北京、上海、广州...。这几乎是纯知识检索，27B 做得很稳。

**降级的本质是把一个复杂任务（结构化拆解）拆成一个简单任务（列清单）+ 一段确定性代码（为每个实体生成 agent）。** 用代码的确定性弥补小模型的不确定性。

◆ 本地 GPU 可行性

━━━━━━━━━━━━━━━━━━━━

我们用的是硅基流动 API（云端），但整个系统兼容任何 OpenAI 兼容 API。改 config.json 的 base_url 就能切到本地。

本地跑 Qwen3.6-27B 的关键瓶颈不是单次推理速度，而是**并发 KV cache 的显存**。8 路并发意味着 8 套独立的 KV cache 同时存在。

不过 Qwen3.6-27B 在这里有个天然优势：188 期讲过，它 75% 的层是 DeltaNet，不产生 KV cache（维护的是固定大小的状态矩阵，不随上下文增长）。只有 25% 的标准 GQA 层需要 KV cache。相比纯 Transformer 模型，KV cache 显存占用直接砍到约 1/4。

即便如此，8 路并发的 KV cache 仍然可观：

- **RTX 5090（32GB）**：FP8 量化下模型约 14GB，剩余约 18GB。虽然 KV cache 只有 1/4 的层在产生，但 8 路并发仍然吃紧，需要降到 3-4 路并发。
- **RTX PRO 6000（96GB）**：FP8 下模型约 14GB，剩余约 80GB 全给 KV cache，8 路并发绰绰有余。这张卡做 agent 集群的本地推理非常合适。

推理服务用 vLLM 或 SGLang，它们自带 PagedAttention 做 KV cache 的动态分配，比手动管理高效得多。

换个角度看：**云端 API 的核心优势不是单卡性能，是弹性并发。** 硅基流动的 Qwen3.6-27B API 背后可能有几十张卡做负载均衡，20 路并发对它来说毫无压力。本地单卡永远追不上这个并发密度。Agent 集群这种"并发密集、单次计算轻量"的场景，天然适合云端。

◆ Kimi 做了什么我们没做的

━━━━━━━━━━━━━━━━━━━━

差距不在代码量，在工程深度：

**可训练的 orchestrator。** Kimi 的调度器本身用 RL（强化学习）训练过，知道怎么拆任务、分配多少资源、什么时候该合并。我们的调度器是一段 system prompt + 一个正则 fallback。

**DAG 依赖调度。** 子 agent 之间可以有依赖关系——"先调研市场规模，再分析竞争格局"。我们只能做互不通信的星型拓扑。

**SFT 过的角色扮演。** K2.6 大概率对模型做过多角色切换的 SFT（监督微调），让模型在不同 system prompt 下稳定地扮演不同角色。27B 通用模型的角色一致性明显弱于此。

**产品级 UI。** Kimi 的角色卡动画、工牌设计打磨得很细。我们的前端是一个单文件 HTML，有暗色主题和角色卡动画，够用但不够精致。

但也不必妄自菲薄。星型拓扑 + 双策略拆解已经覆盖了"N 个独立实体的并行调研"这一类最常见的场景。200 行 Python 能做到这个程度，说明 multi-agent 的**机制门槛不高，工程门槛很高**。

◆ 总结

━━━━━━━━━━━━━━━━━━━━

Multi-agent 集群的核心不是"很多 agent 一起干活"——那只是并发调用 LLM API。核心是三个问题：

1. **怎么拆任务？** 用 LLM 拆，但小模型会偷懒。双策略降级：先试结构化拆解，不行就退化成实体列举 + 代码生成。
2. **怎么控制并发？** Semaphore 限流 + asyncio.gather 并发。不同阶段独立配置超时。
3. **怎么汇总结果？** 串行调用，给足超时，给足 context window。

最大的教训是：**提示词工程在小模型上的失效模式和大模型不同。** 大模型的问题通常是"理解了但做得不够好"，小模型的问题是"理解了但选择偷懒"——它知道你要 20 个，但生成 5 个更省力，采样随机性给了它偷懒的机会。应对策略不是写更长的提示词，而是用代码的确定性包住模型的不确定性。
