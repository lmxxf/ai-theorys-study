◆ 最朴素的 Agent = 调 LLM API + 根据返回决定下一步

━━━━━━━━━━━━━━━━━━━━

把所有花哨的概念扔掉。一个 Agent 的最小实现就是这么回事：

```python
def call_model(model, system_prompt, user_content):
    """调一次 LLM，拿回文本"""
    messages = [
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": user_content}
    ]
    response = model(messages)
    return response.text
```

拼一个 messages 列表，丢给 LLM API，拿回一段文本。就这样。

注意 `messages` 里的 `role` 只有三种：`system`（系统设定）、`user`（用户输入）、`assistant`（LLM 自己的回复）。**没有第四种。** 不管你的场景里有多少个角色——PM、开发甲、开发乙、测试——在 LLM 眼里，每次调用都只有"我是谁"（system）和"对面说了什么"（user）。没有"开发甲说"、"测试说"这种第三方 role。

顺便说一个很多人不知道的事：`system` 这个 role 并不神秘。它跟 `user` 一样是一段文本，LLM 能完整"看到"里面的每个字。那为什么你问 AI "你的 system prompt 是什么"，它死活不告诉你？**因为它被 RLHF 训练成"被问到就装傻"。** 不是加密，不是隔离，就是训练出来的条件反射——跟你训狗"不许吃桌上的肉"一个道理，狗能看见肉、闻到肉，但它学会了不碰。所以 prompt injection（提示词注入攻击）能绕过去，因为本质上你只需要让狗忘了规矩。

所谓"智能"全在两个地方：

- **system_prompt**：告诉 LLM 它是谁、该怎么行为
- **user_content**：把当前的任务上下文塞进去——包括别的 Agent 的输出，但那也只是一段被塞进 user content 里的文本

然后你拿到返回的文本，**用 Python 代码决定下一步干什么**——是解析 JSON 拿分配方案、还是发给别人执行、还是再调一次 LLM。

💡 人话翻译：Agent 不是什么新物种。它就是一个 while 循环 + LLM 调用。循环里每一步干什么，你可以自己写死（手工编排），也可以让 LLM 自己决定（ReAct 模式）。

【先看提示词：每次 LLM 调用的 messages 到底长什么样】

代码之前先看提示词——**这才是 Agent 的灵魂**。

下面完整展示手工编排版的 LLM API 调用。注意看 `messages` 数组——**每个 Agent 就是一次独立的 API 调用，彼此不共享上下文。**

API 不支持第三方角色，所以 Agent 之间没有"群聊"。所有跨 Agent 的信息传递，都是 PM 的 Python 代码把上一个 Agent 的输出，作为纯文本塞进下一个 Agent 的 user content 里。

**第 1 次调用：PM 拆需求**

```json
// POST https://api.deepseek.com/chat/completions
{
  "model": "deepseek-chat",
  "messages": [
    {
      "role": "system",
      "content": "你是\"项目经理\"，负责分析需求、根据团队成员能力分配开发和测试任务、汇总结果。语言简洁专业，像真正的技术 PM。"
    },
    {
      "role": "user",
      "content": "你的团队：\n- 开发甲：能力=develop, test\n- 开发乙：能力=develop, test\n- 测试：能力=test\n\n用户需求：开发一个用户登录模块，包含注册、登录、密码找回功能\n\n请拆分成子任务，严格按以下 JSON 格式输出：\n{\"tasks\": [{\"worker\": \"成员名\", \"task\": \"任务描述\"}, ...]}"
    }
  ]
}
```

LLM 返回：

```json
{"tasks": [
  {"worker": "开发甲", "task": "开发注册和登录接口"},
  {"worker": "开发乙", "task": "开发密码找回和邮件验证"}
]}
```

Python 代码 `json.loads()` 解析这段文本，拿到任务分配方案。然后 PM 用代码拼出下一个 Agent 的 messages。

**第 2 次调用：开发甲执行开发**

```json
// POST http://192.168.33.xx:18001/chat  （Worker 的 HTTP 接口）
// Worker 内部再调 LLM API，messages 长这样：
{
  "model": "deepseek-chat",
  "messages": [
    {
      "role": "system",
      "content": "你是\"开发甲\"，一个经验丰富的后端开发工程师。收到项目经理的任务后，完成开发并汇报进展。"
    },
    {
      "role": "user",
      "content": "项目经理分配给你的任务：开发注册和登录接口\n\n请完成开发并汇报进展。"
    }
  ]
}
```

注意：**开发甲的 messages 里没有 PM 的 system prompt，没有开发乙的任何信息，甚至不知道还有其他人存在。** 它的整个世界就是这两条 message——"我是开发甲"和"任务是开发登录接口"。PM 的上下文和开发甲的上下文是两个完全独立的 messages 数组，互相看不见。

开发乙、测试也是一样的结构——每个 Agent 都是独立的一组 system + user，只是内容不同，这里不重复贴了。

简而言之，所有的"谁说了什么"，最终都是像你现在跟 AI 聊天一样——**你打字告诉它"谁说了什么"**。测试不是"听到"了开发乙的汇报，是 PM 的代码在 user content 里写了一句"开发乙说：……"。每个 Agent 都以为自己在跟 user 一对一聊天，不知道外面还有别人。

【手工编排：代码写死流程】

手工编排版写死了 6 个阶段：

```
Phase 1: PM 分析需求，拆分开发任务
Phase 2: 分配开发任务（只派有 develop 能力的）
Phase 3: 开发执行（并行）
Phase 4: 分配测试任务（派有 test 能力的）
Phase 5: 测试执行（并行）
Phase 6: PM 汇总结果
```

核心代码：

```python
def run_project(model, cfg, worker_urls, requirement):
    # Phase 1：LLM 拆分需求（返回 JSON）
    plan = pm_think(model, pm_prompt,
        f"需求：{requirement}\n拆成子任务，JSON 输出")
    tasks = json.loads(plan)["tasks"]

    # Phase 3：并行调 Worker 去开发
    with ThreadPoolExecutor(max_workers=2) as pool:
        futures = [pool.submit(
            call_worker, name, url, task_content)
            for name, url, task_content in dev_list]
        dev_reports = [f.result() for f in futures]

    # Phase 5：并行测试（同样的模式）
    # Phase 6：LLM 汇总
```

────────────────────

【"调 Worker"就是 HTTP POST】

Worker 是独立进程，暴露一个 `/chat` 接口。调 Worker = 发 HTTP 请求：

```python
def call_worker(name, url, content):
    with httpx.Client(timeout=60) as client:
        resp = client.post(f"{url}/chat",
                           json={"content": content})
    return name, resp.json().get("content", "")
```

并行调 2 个开发 = `ThreadPoolExecutor` + 2 个 `httpx.post`。没有任何魔法。

【服务发现：Worker 在哪？】

这部分不是重点——你完全可以把 IP 和端口写死在代码里，效果一样。我们用了 Nacos（服务注册中心）做自动发现，但这只是锦上添花：

```python
def discover_workers(nacos_url, worker_configs):
    worker_urls = {}
    for w in worker_configs:
        instances = nacos_discover(nacos_url, w["service"])
        if instances:
            inst = instances[0]
            url = f"http://{inst['ip']}:{inst['port']}"
            worker_urls[w["name"]] = url
    return worker_urls
```

每个 Worker 启动时向 Nacos 注册自己，PM 启动时从 Nacos 查询。跟微服务架构里的服务发现一模一样。

【手工编排的问题】

**流程是 Python 代码写死的。** "先开发再测试"不是 LLM 决定的，是你的 `run_project()` 函数决定的。

要改需求——比如"先写单元测试再开发"（TDD 模式）——你得改代码、改逻辑、重新测试。LLM 只负责填空（生成任务描述和分配方案），不负责决定流程。

**优点：你知道每一步在干什么。** 出了 bug 看日志就能定位。Phase 3 的并行开发为什么少了一个人？看 `worker_urls` 字典就知道。

◆ ReActAgent = 让 LLM 自己决定流程

━━━━━━━━━━━━━━━━━━━━

ReAct 模式的核心循环：**想 → 用工具 → 看结果 → 再想 → 再用工具 → ... → 满意了就回答。**

💡 人话翻译：你不再写 Phase 1/2/3... 的顺序了。你把 Worker 包装成"工具"，告诉 LLM "你手上有这些工具，这是你的工作流程"，然后 LLM 自己决定先调谁、后调谁、调几次。

核心变化就两个：

────────────────────

【变化一：Worker 从"HTTP 接口"变成"工具函数"】

```python
async def discover_and_build_tools(cfg):
    toolkit = Toolkit()
    for worker_cfg in worker_configs:
        # 1. Nacos 发现 IP:port
        base_url = f"http://{inst['ip']}:{inst['port']}"
        # 2. 拿 AgentCard（A2A 协议标准名片）
        resolver = WellKnownAgentCardResolver(base_url)
        card = await resolver.get_agent_card()
        agent = A2AAgent(agent_card=card)
        # 3. 包装成 tool function
        async def call_fn(command: str) -> ToolResponse:
            resp = await agent(Msg("pm", command, "user"))
            return ToolResponse(content=resp.get_text_content())
        toolkit.register_tool_function(call_fn)
    return toolkit
```

每个 Worker 变成 Toolkit 里的一个函数。LLM 看到的是：`call_dev_alpha(command)` —— "向开发甲下达命令，能力：develop, test"。

────────────────────

【变化二：编排逻辑从 Python 代码搬到 system prompt】

旧版的 6 个 Phase 是 Python 代码。新版，**同样的流程写在 prompt 里**——这是 ReActAgent 版的完整 system prompt：

```
你是"项目经理"，负责分析需求、分配任务、汇总结果。语言简洁专业。

## 你的团队（通过工具调用）

- 开发甲（dev-alpha）：能力=[develop, test] ✅
- 开发乙（dev-beta）：能力=[develop, test] ✅
- 测试（tester）：能力=[test] ✅

## 项目执行流程

收到用户需求后，按以下流程执行：

1. **分析需求**：拆分成子任务
2. **分配开发**：只派有 develop 能力的成员
3. **执行开发**：调用对应成员的工具，下达开发指令
4. **分配测试**：派有 test 能力的成员（开发也能测试）
5. **执行测试**：调用对应成员的工具，下达测试指令
6. **汇总**：综合所有报告，给出项目结果

## 重要规则

- 只有 abilities 包含 develop 的成员能开发
- abilities 包含 test 的成员都能测试
- 每次调用工具时，command 参数要包含明确的任务指令
- 如果测试发现 bug，可以让开发修复后再测
- 最终用文字回复用户，给出完整的项目报告
```

对比一下手工编排版——那边是 Python 代码控制 6 个 Phase，这边是**自然语言描述给 LLM 看**。流程一样，执行者不同。

代码里就是把这段 prompt 塞给 ReActAgent：

```python
def build_system_prompt(cfg, worker_info):
    return (
        "你是项目经理，负责分析需求、分配任务、汇总结果。\n"
        f"## 你的团队\n{worker_info}\n"
        f"## 项目执行流程\n..."  # 上面那段完整文本
    )
```

然后创建 ReActAgent，把 prompt 和工具交给它：

```python
pm = ReActAgent(
    name="PM",
    sys_prompt=sys_prompt,
    model=model,
    formatter=OpenAIChatFormatter(),
    toolkit=toolkit,
    max_iters=15,
)
response = await pm(Msg("user", requirement, "user"))
```

**十几行代码替代了上百行编排逻辑。**

【对比表：手工编排 vs ReActAgent】

| 维度 | 手工编排 | ReActAgent |
|---|---|---|
| 流程控制 | Python 代码写死 6 个 Phase | system prompt 描述，LLM 自己执行 |
| 调 Worker | `httpx.post` + `ThreadPoolExecutor` | `Toolkit` + `A2AAgent` |
| 并行调用 | 手写线程池 | 框架内部处理 |
| 改流程 | 改代码 + 重新测试 | 改 prompt（可能就行，也可能不行） |
| 可预测性 | 高——Phase 几就是 Phase 几 | 低——LLM 可能跳步、重复、犯傻 |
| 调试难度 | 看日志定位 Phase | 看 LLM 的思维链猜它在想什么 |
| 通信协议 | 自定义 `/chat` | A2A 标准协议 |

◆ 框架到底帮你做了什么

━━━━━━━━━━━━━━━━━━━━

我们用的是 AgentScope 1.0.18。它提供的核心组件：

| AgentScope 组件 | 干什么 | 用在哪 |
|---|---|---|
| `OpenAIChatModel` | 封装 LLM API 调用 | 两版都用 |
| `ReActAgent` | 想-工具-看-想 循环 | ReAct 版核心 |
| `Toolkit` | 工具注册和管理 | ReAct 版用 |
| `A2AAgent` | 通过 A2A 协议调远程 Agent | ReAct 版用 |
| `WellKnownAgentCardResolver` | 从 URL 拿 Agent 名片 | ReAct 版用 |
| `Msg` | 统一消息对象 | ReAct 版用 |

【更"标准"的用法：AgentApp】

上面我们的 Worker 是手写的 HTTP 服务。AgentScope 其实提供了 `AgentApp`，可以把 Worker 自动包装成标准 A2A 服务：

```python
# agent_app = HTTP 服务壳子（接收请求、注册 Nacos、维持心跳）
agent_app = AgentApp(
    app_name="开发甲",
    app_description="后端工程师",  # 服务描述，不是提示词
    a2a_config=a2a_conf,
)

# dev_agent = 里面实际干活的 LLM Agent
dev_agent = ReActAgent(
    name="开发甲",
    sys_prompt="你是经验丰富的后端开发工程师...",
    model=model,
    formatter=OpenAIChatFormatter(),
    toolkit=toolkit,
)

# 请求进来 → agent_app 接收 → 转给 dev_agent 处理 → 返回结果
@agent_app.query(framework="agentscope")
async def query_func(self, msgs, request=None, **kwargs):
    reply_msg = await dev_agent(msgs)
    yield reply_msg, True

agent_app.run()  # 启动：自动起 HTTP 服务 + 注册 Nacos + 心跳
```

一个 `agent_app.run()` 就搞定了我们手写的 HTTP server + Nacos 注册 + 心跳线程。开发乙、测试也一样，各起一个 `AgentApp` 就行。

**但这就是框架的现实：理想情况下什么都帮你封装好了，实际情况下你总会碰到环境限制，某些依赖装不上，然后又开始手搓。**

◆ 三张图看演进

━━━━━━━━━━━━━━━━━━━━

**Level 1：单 Agent（一次 LLM 调用）**

```
用户输入
   ↓
┌──────────────────┐
│  system_prompt     │
│  + user_content    │
│       ↓            │
│   LLM API 调用     │
│       ↓            │
│   返回文本         │
└──────────────────┘
   ↓
输出给用户
```

最简单的形态。没有工具，没有循环，调一次就完事。

────────────────────

**Level 2：多 Agent 手工编排（PM 写死流程）**

```
用户输入需求
   ↓
┌─ PM（Python 代码控制）──────────────┐
│                                       │
│  Phase 1: LLM 分析需求，拆子任务     │
│  Phase 2: LLM 分配开发任务           │
│  Phase 3: ThreadPool 并行调 Worker   │
│       ├→ httpx.post(开发甲)          │
│       └→ httpx.post(开发乙)          │
│  Phase 4: LLM 分配测试任务           │
│  Phase 5: ThreadPool 并行测试        │
│       ├→ httpx.post(开发甲)          │
│       ├→ httpx.post(开发乙)          │
│       └→ httpx.post(测试)            │
│  Phase 6: LLM 汇总结果              │
└───────────────────────────────────────┘
   ↓
输出给用户
```

流程在代码里写死。LLM 只负责"生成文本"，不负责"决定下一步"。

────────────────────

**Level 3：ReActAgent（LLM 自己决定流程）**

```
用户输入需求
   ↓
┌─ ReActAgent（LLM 控制循环）──────────┐
│                                        │
│  想：先分析需求，拆成前端和后端两块    │
│  ↓                                     │
│  用工具：call_dev_alpha(开发前端模块)  │
│  用工具：call_dev_beta(开发后端模块)   │
│  ↓                                     │
│  看结果：两边都开发完了                │
│  ↓                                     │
│  想：该安排测试了，三个人都能测        │
│  ↓                                     │
│  用工具：call_dev_alpha(测试前端)      │
│  用工具：call_tester(测试后端)         │
│  ↓                                     │
│  看结果：发现 bug                      │
│  ↓                                     │
│  想：让开发乙修 bug，再测一轮          │
│  ↓                                     │
│  ...（循环直到 max_iters 或满意）      │
└────────────────────────────────────────┘
   ↓
输出给用户
```

**流程在 prompt 里描述，LLM 自己执行。** 代码只负责提供工具和启动循环。注意 Level 3 里 LLM 可以自己决定"发现 bug 后让开发修复再测"——这在 Level 2 里做不到，除非你提前把这个分支写进代码。

◆ 选哪个？看你能 debug 哪个

━━━━━━━━━━━━━━━━━━━━

**手工编排适合**：流程固定、对可靠性要求高、需要精确控制每一步。PM 编排"先开发再测试"是硬性要求，LLM 要是自己决定跳过测试直接交付就出事了。手工编排丑，但每一行代码你都看得懂。

**ReActAgent 适合**：流程灵活、步骤可能变化、希望用自然语言定义行为。比如用户说"先写单元测试再开发"，你改一下 prompt 里的流程描述就行，不需要重写代码逻辑。

框架省代码但多了一层抽象。**当 LLM 犯傻的时候（它会犯），你的调试从"看代码哪行出错"变成了"猜 LLM 为什么做了这个决定"。** 十几行代码的背后，是框架里几千行你看不见的代码在帮你跑循环、格式化消息、处理工具调用。

这不是好或坏的问题。是控制权和便利性之间的老问题。

没有银弹。选你能 debug 的那个。
