【AI Agent】程序到底怎么写？从手搓到框架的真实代码演进

━━━━━━━━━━━━━━━━━━━━

◆ AI Agent 这个词满天飞，但写过 Agent 代码的程序员可能不到 1%

━━━━━━━━━━━━━━━━━━━━

今天不讲概念，不画架构图，从真实代码讲——从手搓到用框架。

场景：你是一个项目经理（PM），手下 2 个开发 + 1 个测试。开发甲和开发乙都能开发也能测试，测试只能测试。你要分析需求、分配开发任务、等开发完成、分配测试任务、等测试完成、汇总结果。

第一版，手搓 Python 编排，流程写死在代码里。第二版，用 AgentScope 框架的 ReActAgent，编排逻辑搬到 prompt 里。

**代码量砍了一半多，但砍掉的不只是行数——是你对流程的控制权。**

这篇文章把两个版本拆开来讲。看完之后你至少能搞清楚三件事：

1. 不用框架，一个 Agent 到底长什么样
2. 多 Agent 编排到底在编排什么
3. 框架帮你做了什么，又藏了什么

━━━━━━━━━━━━━━━━━━━━

◆ 最朴素的 Agent = 调 LLM API + 根据返回决定下一步

━━━━━━━━━━━━━━━━━━━━

把所有花哨的概念扔掉。一个 Agent 的最小实现就是这么回事：

```python
def call_model(model, system_prompt, user_content):
    """调一次 LLM，拿回文本"""
    messages = [
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": user_content}
    ]
    response = model(messages)
    return response.text
```

拼一个 messages 列表，丢给 LLM API，拿回一段文本。就这样。

所谓"智能"全在两个地方：

- **system_prompt**：告诉 LLM 它是谁、该怎么行为
- **user_content**：把当前的任务上下文塞进去

然后你拿到返回的文本，**用 Python 代码决定下一步干什么**——是解析 JSON 拿分配方案、还是发给别人执行、还是再调一次 LLM。

💡 人话翻译：Agent 不是什么新物种。它就是一个 while 循环 + LLM 调用。循环里每一步干什么，你可以自己写死（手工编排），也可以让 LLM 自己决定（ReAct 模式）。

━━━━━━━━━━━━━━━━━━━━

◆ 多 Agent 编排 = 谁调谁 + 什么顺序

━━━━━━━━━━━━━━━━━━━━

我们的场景：PM（主进程）+ 3 个 Worker（独立服务），通过 HTTP 通信。

Worker 的能力配置长这样：

```python
workers = [
    {"name": "开发甲", "service": "dev-alpha",
     "abilities": ["develop", "test"]},
    {"name": "开发乙", "service": "dev-beta",
     "abilities": ["develop", "test"]},
    {"name": "测试",   "service": "tester",
     "abilities": ["test"]},
]
```

开发甲和开发乙能开发也能测试，测试只能测试。PM 根据 abilities 分配任务。

手工编排版写死了 6 个阶段：

```
Phase 1: PM 分析需求，拆分开发任务
Phase 2: 分配开发任务（只派有 develop 能力的）
Phase 3: 开发执行（并行）
Phase 4: 分配测试任务（派有 test 能力的）
Phase 5: 测试执行（并行）
Phase 6: PM 汇总结果
```

核心代码：

```python
def run_project(model, cfg, worker_urls, requirement):
    # Phase 1：LLM 拆分需求（返回 JSON）
    plan = pm_think(model, pm_prompt,
        f"需求：{requirement}\n拆成子任务，JSON 输出")
    tasks = json.loads(plan)["tasks"]

    # Phase 3：并行调 Worker 去开发
    with ThreadPoolExecutor(max_workers=2) as pool:
        futures = [pool.submit(
            call_worker, name, url, task_content)
            for name, url, task_content in dev_list]
        dev_reports = [f.result() for f in futures]

    # Phase 5：并行测试（同样的模式）
    # Phase 6：LLM 汇总
```

────────────────────

【"调 Worker"就是 HTTP POST】

Worker 是独立进程，暴露一个 `/chat` 接口。调 Worker = 发 HTTP 请求：

```python
def call_worker(name, url, content):
    with httpx.Client(timeout=60) as client:
        resp = client.post(f"{url}/chat",
                           json={"content": content})
    return name, resp.json().get("content", "")
```

并行调 2 个开发 = `ThreadPoolExecutor` + 2 个 `httpx.post`。没有任何魔法。

────────────────────

【服务发现：Worker 在哪？】

Worker 的 IP 和端口不是写死的，通过 Nacos（服务注册中心）发现：

```python
def discover_workers(nacos_url, worker_configs):
    worker_urls = {}
    for w in worker_configs:
        instances = nacos_discover(nacos_url, w["service"])
        if instances:
            inst = instances[0]
            url = f"http://{inst['ip']}:{inst['port']}"
            worker_urls[w["name"]] = url
    return worker_urls
```

每个 Worker 启动时向 Nacos 注册自己，PM 启动时从 Nacos 查询。跟微服务架构里的服务发现一模一样。

────────────────────

【手工编排的问题】

**流程是 Python 代码写死的。** "先开发再测试"不是 LLM 决定的，是你的 `run_project()` 函数决定的。

要改需求——比如"先写单元测试再开发"（TDD 模式）——你得改代码、改逻辑、重新测试。LLM 只负责填空（生成任务描述和分配方案），不负责决定流程。

**优点：你知道每一步在干什么。** 出了 bug 看日志就能定位。Phase 3 的并行开发为什么少了一个人？看 `worker_urls` 字典就知道。

━━━━━━━━━━━━━━━━━━━━

◆ ReActAgent = 让 LLM 自己决定流程

━━━━━━━━━━━━━━━━━━━━

ReAct 模式的核心循环：**想 → 用工具 → 看结果 → 再想 → 再用工具 → ... → 满意了就回答。**

💡 人话翻译：你不再写 Phase 1/2/3... 的顺序了。你把 Worker 包装成"工具"，告诉 LLM "你手上有这些工具，这是你的工作流程"，然后 LLM 自己决定先调谁、后调谁、调几次。

核心变化就两个：

────────────────────

【变化一：Worker 从"HTTP 接口"变成"工具函数"】

```python
async def discover_and_build_tools(cfg):
    toolkit = Toolkit()
    for worker_cfg in worker_configs:
        # 1. Nacos 发现 IP:port
        base_url = f"http://{inst['ip']}:{inst['port']}"
        # 2. 拿 AgentCard（A2A 协议标准名片）
        resolver = WellKnownAgentCardResolver(base_url)
        card = await resolver.get_agent_card()
        agent = A2AAgent(agent_card=card)
        # 3. 包装成 tool function
        async def call_fn(command: str) -> ToolResponse:
            resp = await agent(Msg("pm", command, "user"))
            return ToolResponse(content=resp.get_text_content())
        toolkit.register_tool_function(call_fn)
    return toolkit
```

每个 Worker 变成 Toolkit 里的一个函数。LLM 看到的是：`call_dev_alpha(command)` —— "向开发甲下达命令，能力：develop, test"。

────────────────────

【变化二：编排逻辑从 Python 代码搬到 system prompt】

旧版的 6 个 Phase 是 Python 代码。新版，同样的流程写在 prompt 里：

```python
def build_system_prompt(cfg, worker_info):
    return (
        "你是项目经理，负责分析需求、分配任务、汇总结果。\n"
        f"## 你的团队\n{worker_info}\n"
        "## 项目执行流程\n"
        "1. 分析需求，拆分子任务\n"
        "2. 分配开发任务（只派有 develop 能力的）\n"
        "3. 执行开发（调用对应工具）\n"
        "4. 分配测试任务（派有 test 能力的）\n"
        "5. 执行测试（调用对应工具）\n"
        "6. 汇总结果\n"
    )
```

然后创建 ReActAgent，把 prompt 和工具交给它：

```python
pm = ReActAgent(
    name="PM",
    sys_prompt=sys_prompt,
    model=model,
    formatter=OpenAIChatFormatter(),
    toolkit=toolkit,
    max_iters=15,
)
response = await pm(Msg("user", requirement, "user"))
```

**十几行代码替代了上百行编排逻辑。**

────────────────────

【对比表：手工编排 vs ReActAgent】

| 维度 | 手工编排 | ReActAgent |
|---|---|---|
| 流程控制 | Python 代码写死 6 个 Phase | system prompt 描述，LLM 自己执行 |
| 调 Worker | `httpx.post` + `ThreadPoolExecutor` | `Toolkit` + `A2AAgent` |
| 并行调用 | 手写线程池 | 框架内部处理 |
| 改流程 | 改代码 + 重新测试 | 改 prompt（可能就行，也可能不行） |
| 可预测性 | 高——Phase 几就是 Phase 几 | 低——LLM 可能跳步、重复、犯傻 |
| 调试难度 | 看日志定位 Phase | 看 LLM 的思维链猜它在想什么 |
| 通信协议 | 自定义 `/chat` | A2A 标准协议 |

◆ 框架到底帮你做了什么

━━━━━━━━━━━━━━━━━━━━

我们用的是 AgentScope 1.0.18。它提供的核心组件：

| AgentScope 组件 | 干什么 | 用在哪 |
|---|---|---|
| `OpenAIChatModel` | 封装 LLM API 调用 | 两版都用 |
| `ReActAgent` | 想-工具-看-想 循环 | ReAct 版核心 |
| `Toolkit` | 工具注册和管理 | ReAct 版用 |
| `A2AAgent` | 通过 A2A 协议调远程 Agent | ReAct 版用 |
| `WellKnownAgentCardResolver` | 从 URL 拿 Agent 名片 | ReAct 版用 |
| `Msg` | 统一消息对象 | ReAct 版用 |

【同事的写法：更"标准"的用法】

同事在另一个项目里的 `route.py` 是一个路由 Agent——根据用户输入决定调"读文件 Agent"还是"写文件 Agent"：

```python
toolkit = Toolkit()
toolkit.register_tool_function(call_write_agent)
toolkit.register_tool_function(call_read_agent)

router = ReActAgent(
    name="Router",
    sys_prompt="你是路由智能体，将用户查询路由到正确的后续任务。",
    model=model,
    formatter=OpenAIChatFormatter(),
    toolkit=toolkit,
)
```

他还用了 `AgentApp` 把 Agent 自动包装成 A2A 服务并注册到 Nacos：

```python
agent_app = AgentApp(
    app_name="Reader",
    app_description="具备读取文件中文本能力的助手",
    a2a_config=a2a_conf,
)

@agent_app.query(framework="agentscope")
async def query_func(self, msgs, request=None, **kwargs):
    reply_msg = await reader_agent(msgs)
    yield reply_msg, True

agent_app.run()  # 自动起服务，注册 Nacos
```

这是更标准的写法。我们没用 `AgentApp`，因为实际环境中某些依赖跑不起来，所以 Worker 是手写的 HTTP 服务 + 手动注册 Nacos。

**这就是框架的现实：理想情况下什么都帮你封装好了，实际情况下你总会碰到环境限制，然后开始手搓。**

◆ 三张图看演进

━━━━━━━━━━━━━━━━━━━━

**Level 1：单 Agent（一次 LLM 调用）**

```
用户输入
   ↓
┌──────────────────┐
│  system_prompt     │
│  + user_content    │
│       ↓            │
│   LLM API 调用     │
│       ↓            │
│   返回文本         │
└──────────────────┘
   ↓
输出给用户
```

最简单的形态。没有工具，没有循环，调一次就完事。

────────────────────

**Level 2：多 Agent 手工编排（PM 写死流程）**

```
用户输入需求
   ↓
┌─ PM（Python 代码控制）──────────────┐
│                                       │
│  Phase 1: LLM 分析需求，拆子任务     │
│  Phase 2: LLM 分配开发任务           │
│  Phase 3: ThreadPool 并行调 Worker   │
│       ├→ httpx.post(开发甲)          │
│       └→ httpx.post(开发乙)          │
│  Phase 4: LLM 分配测试任务           │
│  Phase 5: ThreadPool 并行测试        │
│       ├→ httpx.post(开发甲)          │
│       ├→ httpx.post(开发乙)          │
│       └→ httpx.post(测试)            │
│  Phase 6: LLM 汇总结果              │
└───────────────────────────────────────┘
   ↓
输出给用户
```

流程在代码里写死。LLM 只负责"生成文本"，不负责"决定下一步"。

────────────────────

**Level 3：ReActAgent（LLM 自己决定流程）**

```
用户输入需求
   ↓
┌─ ReActAgent（LLM 控制循环）──────────┐
│                                        │
│  想：先分析需求，拆成前端和后端两块    │
│  ↓                                     │
│  用工具：call_dev_alpha(开发前端模块)  │
│  用工具：call_dev_beta(开发后端模块)   │
│  ↓                                     │
│  看结果：两边都开发完了                │
│  ↓                                     │
│  想：该安排测试了，三个人都能测        │
│  ↓                                     │
│  用工具：call_dev_alpha(测试前端)      │
│  用工具：call_tester(测试后端)         │
│  ↓                                     │
│  看结果：发现 bug                      │
│  ↓                                     │
│  想：让开发乙修 bug，再测一轮          │
│  ↓                                     │
│  ...（循环直到 max_iters 或满意）      │
└────────────────────────────────────────┘
   ↓
输出给用户
```

**流程在 prompt 里描述，LLM 自己执行。** 代码只负责提供工具和启动循环。注意 Level 3 里 LLM 可以自己决定"发现 bug 后让开发修复再测"——这在 Level 2 里做不到，除非你提前把这个分支写进代码。

◆ 选哪个？看你能 debug 哪个

━━━━━━━━━━━━━━━━━━━━

**手工编排适合**：流程固定、对可靠性要求高、需要精确控制每一步。PM 编排"先开发再测试"是硬性要求，LLM 要是自己决定跳过测试直接交付就出事了。手工编排丑，但每一行代码你都看得懂。

**ReActAgent 适合**：流程灵活、步骤可能变化、希望用自然语言定义行为。同事的路由 Agent 就很合适——用户说"读文件"就调读 Agent，说"写文件"就调写 Agent，不需要写死 if-else。

框架省代码但多了一层抽象。**当 LLM 犯傻的时候（它会犯），你的调试从"看代码哪行出错"变成了"猜 LLM 为什么做了这个决定"。** 十几行代码的背后，是框架里几千行你看不见的代码在帮你跑循环、格式化消息、处理工具调用。

这不是好或坏的问题。是控制权和便利性之间的老问题。

没有银弹。选你能 debug 的那个。
