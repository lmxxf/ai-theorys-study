◆ 问题一：总损失变负了，是 bug 吗？

━━━━━━━━━━━━━━━━━━━━

先看现象。从 epoch 4 开始，avg_train_loss = -0.016109，avg_val_loss = -0.013113——总损失变负了。但三个子任务的损失全程为正：type_loss 在 0.88~0.98 之间，quantity_loss 在 0.13~0.14 之间，leverage_loss 在 0.11~0.16 之间。

总损失是三个子任务损失的加权组合之和，子任务损失全是正的，总损失怎么能是负的？

回忆一下上一篇的公式。总损失的完整展开：

```
L_total = [exp(-s1) * L_CE + s1]                           # 分类
        + [(1/2) * exp(-s2) * L_MSE_q + (1/2) * s2]        # 回归 quantity
        + [(1/2) * exp(-s3) * L_MSE_l + (1/2) * s3]        # 回归 leverage
```

每个任务的贡献由两部分组成：**加权损失项**（exp(-s)*L，永远为正）和**正则项**（s 或 (1/2)*s）。

关键：**s = log(σ²) 可以是任意实数，包括负数。**

上一篇推过，s 的均衡点是 s = log(L)。当任务损失 L 小于 1 时，log(L) < 0，s 就是负数。

看他的数据：

```
# 均衡点近似（s ≈ log(L)）：
type:     L_CE ≈ 0.88  →  s1 ≈ log(0.88) ≈ -0.128
quantity: L_MSE ≈ 0.135 →  s2 ≈ log(0.135) ≈ -2.00
leverage: L_MSE ≈ 0.158 →  s3 ≈ log(0.158) ≈ -1.845
```

三个 s 全是负数。

正则项的贡献：

```
分类：    s1 ≈ -0.128
回归 q：  (1/2) * s2 ≈ (1/2) * (-2.00) = -1.00
回归 l：  (1/2) * s3 ≈ (1/2) * (-1.845) = -0.923
正则项总和 ≈ -0.128 + (-1.00) + (-0.923) = -2.051
```

加权损失项呢？上一篇推过，均衡点处 exp(-s)*L = 1（分类）或 exp(-s)*L = 1（回归，因为 (1/2) 同时出现在两项中，对均衡点的乘积无影响，均衡点处也是 1）。所以：

```
加权损失项总和 ≈ 1 + (1/2)*1 + (1/2)*1 = 2.0
```

总损失 ≈ 2.0 + (-2.051) = **-0.051**

和日志里 epoch 6 的 avg_val_loss = -0.042875 量级吻合。

💡 **人话翻译：** 当任务损失 L < 1 时，s = log(L) < 0，正则项变成负数。正则项的负值大过了加权损失项的正值，总损失就成了负数。**不是 bug，是数学上的必然。**

────────────────────

**那总损失为负，会不会影响训练？**

不会。梯度下降不关心损失的绝对值是正还是负——它关心的是**梯度方向**。损失从 -0.04 降到 -0.05，梯度的方向和从 1.04 降到 1.03 完全一样。优化器看的是"朝哪个方向走能让损失更小"，不关心损失本身是正是负。

你也可以这样理解：给总损失加一个常数 C（比如 +10），`L_total + C` 和 `L_total` 的梯度完全相同——常数的导数是零。损失函数的"零点"是可以随便选的，正负没有物理意义。

**结论：完全正常。负的总损失只说明正则项 s 的贡献超过了加权后的原始损失。梯度依然正确，训练依然有效。真正有意义的，是各子任务自己的损失值。**

◆ 问题二：早停应该锚定哪个指标？

━━━━━━━━━━━━━━━━━━━━

网友的早停条件是：avg_val_loss（总验证损失）连续 2 轮下降阈值小于 1e-2。最佳检查点在第 6 轮，第 8 轮触发早停。

但问题来了——总损失被正则项 s 搅浑了，**它反映的不再是"模型预测质量"，而是"模型预测质量 + 可学习参数 s 的动态"。**

看日志：

```
avg_val_loss:      epoch 6 = -0.042875 → epoch 7 = -0.046221 → epoch 8 = -0.048622
                   差值：         -0.003346             -0.002401
                   （都在下降，但幅度很小，触发早停）

avg_val_type_loss: epoch 6 = 0.883855  → epoch 7 = 0.882970  → epoch 8 = 0.879770
                   差值：         -0.000885             -0.003200
                   （还在缓步下降，没有止住）
```

总损失说"已经不怎么动了，该停了"。但分类任务（业务最优先）的验证损失还在往下走——0.883855 → 0.882970 → 0.879770，三轮降了 0.004。

**总损失的变化里混杂了 s 的变化。** 当 s 还在调整时，总损失的变化并不纯粹反映模型预测能力的改善。s 的调整和模型权重的调整是两套动力学，但它们被混在同一个数字里了。

────────────────────

**锚点应该是什么？**

既然分类任务有**绝对业务优先级**，早停应该盯的是 **avg_val_type_loss**——分类任务在验证集上的原始交叉熵损失，不含 s，不含正则项。

为什么用"原始损失"而不是"加权后的损失"？因为加权后的损失 exp(-s)*L_CE 在均衡点处恒等于 1（上一篇推过的），它几乎不变——你用它做早停，什么也看不出来。只有**不带 s 的原始损失 L_CE** 才真实反映模型在这个任务上的预测质量。

```
# 正确的早停逻辑（伪代码）：
best_type_loss = float('inf')
patience_counter = 0
patience = 2
threshold = 1e-3  # 分类损失量级较小，阈值也可以调小

for epoch in range(max_epochs):
    train(...)
    val_type_loss = evaluate_type_loss(val_set)  # 只看分类任务的验证损失

    if best_type_loss - val_type_loss > threshold:
        best_type_loss = val_type_loss
        patience_counter = 0
        save_checkpoint()
    else:
        patience_counter += 1
        if patience_counter >= patience:
            break
```

────────────────────

**如果多个任务都重要呢？**

那就用子任务验证损失的加权组合——**不带 s 的原始损失**按业务权重加起来：

```
val_metric = w1 * val_type_loss + w2 * val_quantity_loss + w3 * val_leverage_loss
# w1, w2, w3 是业务权重，反映你对各任务的重视程度
# 注意：这里的权重是人工设定的业务权重，不是训练中 s 控制的那个权重
```

关键区别：训练时用同调不确定性加权（让 s 自动平衡梯度），早停时用人工业务权重（反映你真正在乎的东西）。训练和评估用的指标可以不一样，也应该不一样。

────────────────────

**补充建议：启用策略 3 的 α**

上一篇最后给了处方：如果分类任务有绝对优先级，应该把分类任务的正则项系数 α 调大——换一根更硬的弹簧，让分类任务在总损失中占更大的份额。

网友目前还没启用 α（等于 α=1，默认值）。从日志看，分类的 type_loss 从 0.978 降到 0.880，降了 10%；两个回归的 quantity_loss 从 0.142 降到 0.135，leverage_loss 从 0.107 降到 0.158（甚至略微上升过）。如果分类是绝对优先，可以把分类的 α 调到 2 或 3，让模型把更多"注意力"给分类任务，换取分类精度的进一步提升——代价是两个回归任务可能稍微变差。

这就是上一篇那根"弹簧"的用途：你已经知道了弹簧的原理，现在该调弹簧的刚度了。

◆ 问题三：同调不确定性加权到底怎么协调梯度的？

━━━━━━━━━━━━━━━━━━━━

这是最深的一个问题。前面两个是"现象怎么解释"，这个是"机理是什么"。

先看最朴素的做法——直接把三个损失加起来：

```
L_total = L1 + L2 + L3
```

反向传播时，对共享参数 θ 的梯度：

```
∂L_total/∂θ = ∂L1/∂θ + ∂L2/∂θ + ∂L3/∂θ
```

三个梯度直接叠加。看起来很公平？不公平。

梯度的"大小"（范数）跟损失的量级直接相关。看网友的数据（取 epoch 6 的验证损失）：

```
type_loss     ≈ 0.884    （交叉熵）
quantity_loss ≈ 0.135    （MSE）
leverage_loss ≈ 0.159    （MSE）
```

分类损失是回归损失的 **6~7 倍**。直接相加时，分类任务产生的梯度范数也大致是回归任务的 6~7 倍——共享网络的参数更新被分类任务主导，回归任务的信号被淹没。

（实际上，梯度范数的差异可能比损失值的差异更大。损失值只是梯度链上的一个因子，还要乘以从输出头到共享层的整条链。但损失值的量级差异是最直接的驱动力。）

────────────────────

**同调不确定性加权怎么改变这件事？**

加了 s 之后，总损失变成：

```
L_total = exp(-s1)*L1 + s1 + (1/2)*exp(-s2)*L2 + (1/2)*s2 + (1/2)*exp(-s3)*L3 + (1/2)*s3
```

对共享参数 θ 求梯度（注意 s 不依赖于 θ，它是独立的可学习参数）：

```
∂L_total/∂θ = exp(-s1) * ∂L1/∂θ + (1/2)*exp(-s2) * ∂L2/∂θ + (1/2)*exp(-s3) * ∂L3/∂θ
```

正则项 s1、s2、s3 对 θ 的导数是零（它们不含 θ），所以正则项不参与共享参数的梯度计算。**正则项只影响 s 自身的更新，不影响模型权重的更新。**

关键在 exp(-s_i) 这个系数。它对每个任务的梯度做了缩放：

```
任务 i 对共享参数的梯度贡献 = exp(-s_i) × ∂L_i/∂θ
```

上一篇推过，均衡点处 exp(-s_i) ≈ 1/L_i。代入：

```
任务 i 的梯度贡献 ≈ (1/L_i) × ∂L_i/∂θ
```

**每个任务的梯度被它自己的损失值反除了。** 用网友的数据算：

```
type:     exp(-s1) ≈ 1/0.884 ≈ 1.13   → 梯度被放大 1.13 倍
quantity: exp(-s2) ≈ 1/0.135 ≈ 7.41   → 梯度被放大 7.41 倍
leverage: exp(-s3) ≈ 1/0.159 ≈ 6.29   → 梯度被放大 6.29 倍
```

分类的损失大（0.884），但缩放系数小（1.13）——大损失配小权重。回归的损失小（0.135），但缩放系数大（7.41）——小损失配大权重。

最终效果：

```
# 不加权时，梯度范数比例大致为：
type : quantity : leverage ≈ 0.884 : 0.135 : 0.159 ≈ 6.5 : 1 : 1.2

# 加权后，梯度范数比例大致为：
type : quantity : leverage ≈ (0.884 × 1.13) : (0.135 × 7.41) : (0.159 × 6.29)
                           ≈ 1.0 : 1.0 : 1.0
```

**三个任务的梯度贡献被拉到同一量级。** 这就是"自动梯度归一化"——不是通过显式计算梯度范数（那是 GradNorm 的做法），而是通过 exp(-s) 的自适应缩放间接实现的。

💡 **人话翻译：** 三个人在同一条绳子上拉（共享参数），力气大的（损失大的任务）被戴上阻力套，力气小的（损失小的任务）被装上助力器。最终三个人对绳子的拉力差不多大。

────────────────────

**s 大的任务少说话，s 小的任务多说话**

再换一个角度理解。s 代表"不确定性"——s 越大，模型对这个任务越不确定。

```
s 大 → exp(-s) 小 → 梯度被压小 → 不确定的任务"少说话"
s 小 → exp(-s) 大 → 梯度被放大 → 确定的任务"多说话"
```

这很合理：你不确定的事情，不应该大声嚷嚷着要改方向；你确定的事情，应该坚定地说"往这边走"。

从网友的日志看，随着训练推进，三个任务的损失都在下降——模型对所有任务都越来越确定。s 跟着下降，exp(-s) 升高，梯度信号整体增强。但关键是**相对大小**始终保持平衡：损失下降快的任务，s 也下降快，但它的 exp(-s) 只是从"大"变成"更大"——和其他任务保持同步。

────────────────────

**三个维度的效果**

网友具体问了梯度范数、损失下降、泛化差距这三个方面。逐个看：

**1. 梯度范数：自动归一化**

上面已经分析了。不加权时，type 任务的梯度主导共享网络；加权后，三个任务的梯度贡献被拉平。从日志看，两个回归任务的损失确实在稳步下降（quantity 从 0.142 降到 0.135，leverage 从 0.107 稳定在 0.158 左右），说明它们的梯度信号没有被分类任务淹没。

**2. 损失下降：节奏同步**

看八轮训练的下降趋势：

```
type_loss:     0.978 → 0.880   下降 10.0%
quantity_loss: 0.142 → 0.135   下降 4.9%
leverage_loss: 0.107 → 0.158   先升后稳（可能有特殊原因，比如任务本身较难，或数据分布问题）
```

分类和回归都在下降，没有出现"一个任务暴降、另一个任务纹丝不动"的情况。这就是梯度归一化的效果——每个任务都能分到足够的梯度信号来更新共享参数。

**3. 泛化差距（过拟合程度）**

泛化差距 = 训练损失与验证损失的差。看分类任务：

```
epoch 1: train_type_loss = 0.980, val_type_loss = 0.978  → 差距 0.002
epoch 6: train_type_loss = 0.848, val_type_loss = 0.884  → 差距 0.036
epoch 8: train_type_loss = 0.817, val_type_loss = 0.880  → 差距 0.063
```

差距在扩大——训练损失持续下降，验证损失在 epoch 5 之后下降变缓。这是经典的过拟合信号。分类任务的模型开始"记住"训练数据，而不是学到泛化特征。

两个回归任务的泛化差距呢？

```
quantity:
  epoch 1: train 0.148, val 0.142 → 差距 0.006
  epoch 8: train 0.133, val 0.135 → 差距 0.002（差距反而缩小了）

leverage:
  epoch 1: train 0.173, val 0.107 → 差距 0.066
  epoch 8: train 0.155, val 0.158 → 差距 0.003（差距也缩小了）
```

有趣——回归任务的泛化差距在缩小，而分类任务的泛化差距在扩大。这说明分类任务的容量（模型用来拟合这个任务的参数自由度）可能不够，或者分类数据中有更多噪声。这进一步支持了"应该启用 α 来给分类任务更多权重"的建议。

◆ 总结：弹簧在真实训练中的动态行为

━━━━━━━━━━━━━━━━━━━━

上一篇用弹簧比喻讲了同调不确定性加权的静态原理：弹簧的刚度（正则项系数）决定均衡点，外力（损失权重）会被弹性抵消。

这一篇看到的是**弹簧在真实训练中的动态行为**——

- **弹簧可以压过零点**：s 可以是负数，正则项可以是负数，总损失可以是负数。这不是故障，是弹簧的行程范围比你以为的大。
- **弹簧的振动不适合当指南针**：总损失里混杂了弹簧（s）的动态和模型预测能力的动态，两者纠缠在一起。早停要盯的是模型的预测能力（子任务原始损失），不是弹簧系统的总能量（总损失）。
- **弹簧的真正作用是梯度归一化**：exp(-s) 把三个量级不同的任务梯度拉到同一水平，让每个任务都能在共享参数上"说上话"。不确定的少说，确定的多说。

三个问题，一个比一个深。第一个是现象（负值），第二个是工程决策（早停），第三个是机理（梯度协调）。串起来就是：**同调不确定性加权不是一个简单的"自动权重"工具——它是一个有内部动力学的系统，用起来要理解它的行为，而不只是理解它的公式。**

━━━━━━━━━━━━━━━━━━━━

技术名词速查（本篇新增）：

- **早停（Early Stopping）**：训练过程中监控验证集指标，当指标不再改善时停止训练，防止过拟合。是最简单也最常用的正则化手段
- **梯度范数（Gradient Norm）**：梯度向量的长度（L2 范数），反映参数更新的"力度"。梯度范数大的任务对共享参数的影响大
- **泛化差距（Generalization Gap）**：训练集表现与验证集表现的差异。差距越大，过拟合越严重
- **梯度归一化（Gradient Normalization）**：把不同任务的梯度缩放到相近的量级，防止某个任务的梯度主导共享参数的更新
