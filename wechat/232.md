◆ 第二章：技术报告里真正的硬货

━━━━━━━━━━━━━━━━━━━━

吐槽归吐槽，我们还是得承认——**GLM-5 的技术报告（arxiv 2602.15763）是真做了功课的**。180 多个作者署名，工程量摆在那。

**先把三代版本和时间线理清楚**——这是理解 5.2 真实情况的关键，国内媒体写得乱七八糟，混着 GLM-5/5.1/5.2 一起夸：

| 版本 | 发布时间 | 上下文 | 关键升级 |
|---|---|---|---|
| **GLM-5** | 2026-02 | **200K** | 首发，744B/40B MoE + DSA |
| **GLM-5.1** | **2026-03-27** | **200K** | Agent 长程任务增强（8 小时连续工作） |
| **DeepSeek V4** | **2026-04-25** | **1M（CSA + HCA + DSA）** | （插队进来，让 GLM 看到了 1M 的玩法） |
| **GLM-5.2** | **2026-06-13** | **1M（IndexShare + KVShare + LayerSplit）** | 才到 1M，架构改造（详见后文） |

**注意中间那行 DeepSeek V4**——这条时间线非常关键：

- **GLM-5 / 5.1 还在 200K 阶段时**（2026-02 / 03），DeepSeek V4 还没出来
- **V4 4 月底发布**，带着 CSA + HCA 压缩 + 1M 上下文的整套打法
- **GLM-5.2 6 月发布**，第一次声称"稳定 1M"

也就是说，**GLM 的 1M 不是和 V4 平行发展出来的，是看了 V4 之后两个月内追出来的**。但追的姿势完全不同——V4 用 CSA + HCA 做序列压缩，GLM-5.2 用了一个叫 IndexShare 的新方案——后面【硬货五】那一节会讲清楚机制。

💡 这里有个**信息陷阱**：GLM-5 的技术报告（arxiv 2602.15763）目前最新版里写"GLM-5 支持 1M 上下文"，让人误以为 GLM-5 一发布就有 1M。**实际上技术报告是滚动更新的**——v1 是 2026-02 投稿的，里面的 1M 实现细节是后来 5.2 落地后补进去的。arxiv 论文不像期刊有 freeze 版本，作者随时可以替换 PDF，让"发布时间"和"技术内容"看起来时间错配。

GLM-5 系列从 5.0 到 5.2 模型主体（744B/40B、80 层、256 路由专家）没动过——**所有变化都在长上下文架构和后训练 RL infra 这两个层面发生**。

| 项 | GLM-5 系列共有 | 对比 GLM-4.5 |
|---|---|---|
| 总参 / 激活 | **744B / 40B** | 355B / 32B（总参翻倍） |
| 路由专家数 | **256 routed + 1 shared** | 类似 |
| 每 token 选几个专家 | 8 | 8 |
| 层数 | 80 | 类似 |
| 注意力底座 | **MLA-256 + Muon Split + DSA** | 标准 MLA |
| MTP（投机解码） | **3 层参数共享** | 1 层 |
| 预训练 token | **27T**（中训再加 1.5T，训练时上下文最多 200K） | 类似 |

GLM-5 系列在五个地方做了真正属于自己的差异化（不是抄 DeepSeek 的部分），其中前三项是从 5.0 起就有的"模型本体"创新，后两项是 5.2 才补上的"长上下文 + 后训练"：

【硬货一：Muon Split——给 MLA 做精细化训练】

DeepSeek V4 引入了 Muon 优化器（一种比 AdamW 更现代的优化器，专门处理大矩阵的更新方向），GLM 拿过来继续往前走了一步：

- DeepSeek 原版 Muon：对整个 W_UQ / W_UK / W_UV（查询、键、值的投影矩阵）做矩阵正交化
- GLM Muon Split：**把矩阵按 attention head 拆开，每个 head 各自做正交化**

效果是什么？原本 GLM 团队发现 MLA-576（DeepSeek 的标准配置）打不过 GQA-8（分组查询注意力），死活追不平。换成 Muon Split 之后追平了——而且 attention logits（注意力打分）训练全程不用裁剪就能保持稳定。

这是**真改进，不是营销**。

【硬货二：MLA-256——给国产卡量身改造，顺便踩了 DeepSeek 一脚】

这条技术报告里写得相当"坦诚"——其实是用学术英语包装的一记定向打击：

> "Number of attention heads in DeepSeek-V3 is selected according to the roofline of H800, it is inappropriate for other hardware."
>
> （DeepSeek-V3 的注意力头数是按 H800 的算力屋顶线选的，对其他硬件不适合。）

学术腔的"inappropriate（不适合）"翻译成中文圈黑话就是：**"DeepSeek 这模型设计的时候只想着英伟达 H800，根本没考虑别人怎么跑。国产卡跑你这玩意儿效率不行。"**

这一脚踩得相当精准——一举三得：

1. **挤兑 DeepSeek**：你只服务英伟达生态，没考虑国产替代
2. **抬高自己**：我们改 head 配置是为了国产卡——和宣传里"全面适配 7 个国产算力平台"完美闭环
3. **政治正确加分**：在中美 AI 算力脱钩的大背景下，"为国产卡量身设计"是最稳的叙事

具体改了什么？GLM 把每个 head 的维度从 192 调大到 256，head 数减 1/3。**训练阶段计算量不变，但推理 decode 阶段的计算量降低了**——刚好打在昇腾这种"算力 / 带宽比"和 H800 不同的硬件特性上。

但话说回来，**DeepSeek 自己也不是不知道**这件事。V4 技术报告里专门有一节讲华为昇腾适配，原话是 "We validated on HUAWEI Ascend NPUs"——只是没像 GLM 这样把"DeepSeek 选 head 数没考虑国产卡"摆到 abstract 级别的位置吐槽。两家姿态对照：

- **DeepSeek 的路线**：先按英伟达把架构上限做出来，再适配国产卡——**务实优先**
- **GLM 的路线**：从第一天就为国产卡设计 head 配置——**政治叙事优先**

哪条路更高效？**纯工程上 DeepSeek 那条路明显更优**——先在最强算力底座上验证架构上限，再迁移；GLM 这种"从第一天为国产卡设计"听起来正义，**实际上要为了 head 数兼容性付出一定的架构表达力代价**（head 越多通常 attention 的表达力越好，MLA-256 的 head 数少 1/3 意味着 expressiveness 上限稍弱）。

但在 2026 年中美科技战的语境下，**叙事的政治价值可能比 expressiveness 的代价更重要**。这条架构选择不是纯技术决策，是技术 + 政治的混合决策。

【硬货三：MTP 参数共享】

MTP（Multi-Token Prediction，多 token 预测）是 DeepSeek V3 首创的训练技巧。先把它是什么讲清楚——后面硬货五还要用到。

【MTP 是什么】

标准 LLM 一次只预测一个 token——给前文，输出下一个。

MTP 让模型**一次预测多个 token**——给前文，同时输出"下一个"+"下下一个"+"下下下一个"。

具体实现是这样的：在主模型 80 层 transformer 的尾部，**再接几个额外的 transformer 层**——每个额外层负责预测往后再多一步。这些额外层就叫"MTP 层"。

**注意：MTP 层不是小模型、不是辅助网络，就是和主模型同款的 transformer 层**——也有 self-attention，也用 KV cache，也走 Lightning Indexer 那套稀疏注意力（如果有的话）。**它只是位置上接在主模型尾巴上**。

【为什么要做 MTP？两个收益】

**第一个收益：训练时强迫模型"想得更远"**。模型在预测当前 token 的时候，内部表示就已经"想好了"再下一步要说什么——这让中间层的表示更有前瞻性、信息密度更高。

**第二个收益：推理时可以做投机解码**。投机解码（speculative decoding）就是让模型一次输出多个 token，**赌后面几个 token 是对的**。如果赌对了，本来要跑 N 次前向才能输出 N 个 token，现在 1 次就搞定，**推理速度提升数倍**。

**接受长度（acceptance length）** 衡量的就是"平均一次能赌对几个 token"——越大越好。

【GLM-5 改了什么】

- DeepSeek V3/V4：1 个 MTP 层，预测 2 个 token
- GLM-5：**3 个 MTP 层（参数共享，所以内存成本不变）**

效果：接受长度从 DeepSeek-V3.2 的 2.55 涨到 GLM-5 的 2.76。

这是 V4 没做的改进。**实打实**。

【硬货四：异步 Agent RL 训练框架】

这是技术报告里最厚的一章，也是 GLM-5 真正甩开 DeepSeek V4 的地方——**V4 的报告里几乎不写 Agent 相关后训练细节，GLM-5 在这块整整一章**。

但什么叫"Agent RL"？为什么这事难训？得先把背景说清楚。

【先理解 Agent RL 难在哪】

传统的 LLM 后训练（SFT 监督微调、RLHF 基于人类反馈的强化学习），喂的样本都是**单回合**的——一句问题、一句答案，算损失更新权重，下一个样本。

**Agent 任务完全不一样**。让一个 AI agent 去解决一个真实软件工程问题（比如修一个 Python 项目的 bug），它得：

1. 读 issue → 输出"我需要看这几个文件"
2. 调工具看文件 → 拿到文件内容
3. 想了想 → 输出"我猜 bug 在第 X 行，先改改试试"
4. 调工具改文件 + 跑测试 → 拿到测试结果
5. 如果失败 → 回到步骤 3 重新想

这一连串"想 → 调工具 → 看结果 → 再想"叫一个 **trajectory（轨迹）**，让 agent 跑一遍 trajectory 这个过程叫做 **rollout**——本文后面会反复用这两个词。一个真实 SWE 任务的 trajectory 经常**几百轮、几万个 token、跑几小时**才能完成。

RL 训练就是要从这种 trajectory 里学——成功了就强化整条路径上的决策，失败了就削弱。问题是：

- **生成一条 trajectory 要几小时**——GPU 大部分时间在等 agent 跑测试、读文件
- **不同任务长短差异巨大**——简单 bug 100 轮搞定，复杂任务 1000 轮还没完
- **训练时 model 在变**——你用 model v1 跑的 trajectory，等收集够一批准备训练时，model 已经更新到 v5 了，**v1 跑的样本对 v5 来说还算不算"自己的行为"**？

三个问题串起来就是：**Agent RL 训练里，GPU 不是被算力卡住的，是被"等 trajectory 跑完"卡住的**。同步训练（等所有样本跑完再统一更新）会让 GPU 大部分时间空转。

【GLM-5 怎么解决】

技术报告里给了一整套"异步训练框架"——核心思想是**把推理引擎和训练引擎分开放，互相不等**：

- **推理引擎**：专门跑 agent trajectory，不停地生成、生成、生成
- **训练引擎**：攒够一批就更新权重，**不等所有 trajectory 跑完**
- 推理引擎每隔几步从训练引擎拿一次最新权重，**保持大致同步但允许滞后**

听起来简单，工程上一堆坑。GLM-5 报告里讲了三个具体招式（每个都对应一个坑）：

**第一个坑：tokenization 不一致**

推理引擎吐 token 给 agent 用，trajectory 里记的是 token id。等训练时，如果再把 token id 转回文本、重新 tokenize 一遍（text-in-text-out 模式），**会出现微小的边界错位**——比如空格位置变了、特殊符号分词不同。这种错位在普通 SFT 里没事，但在 Agent RL 里会**破坏 token 和 reward 的对齐**，模型会被错误信号训歪。

GLM-5 的招式叫 **TITO Gateway**（Token-in-Token-out）——训练引擎**直接消费推理引擎吐的 token id**，不做任何重新 tokenize 的转换。简单粗暴，但保证一致。

**第二个坑：policy 漂移**

异步训练的根本问题：**rollout 用的是旧策略（model v1），训练时已经是新策略（model v5）了**。

举个具体例子。Agent 跑到某一步要"决定下一个 token"，旧策略 v1 当时算出"用 token X 的概率是 0.8"，于是选了 X。但等这条 trajectory 收集起来准备训练时，model 已经更新到 v5 了——**v5 现在算同一个上下文，可能觉得 X 只该有 0.4 的概率**。

这就来麻烦了。**RL 训练的本质是"如果这步走对了就强化这个选择"**——但被强化的是谁的选择？v1 在那个时刻凭 0.8 概率选了 X，**v5 现在被反向传播强化的是"提高 X 的概率"，可是 v5 本来就不喜欢 X**——强行拉高的话，v5 会被一个它自己都不认同的方向训歪。

RL 教科书给的标准做法是用 **"重要性采样"（importance sampling）** 来纠偏——给每个 token 算一个比例：

> **比例 = 当前策略给这个 token 的概率 ÷ 当时生成时旧策略给的概率**

也就是 **0.4 ÷ 0.8 = 0.5**——v5 用这个比例去缩放梯度，**"这个 token 我自己只值 0.5 倍的强化"**，避免被 v1 的选择带偏。

**为什么参考点是 1？** 因为：

- **比例 = 1** 意味着新旧策略完全一致——理想情况，照常训练
- **比例 > 1** 意味着新策略比旧策略**更喜欢**这个 token——梯度被放大
- **比例 < 1** 意味着新策略比旧策略**更不喜欢**这个 token——梯度被缩小

但要算这个比例就有个新麻烦——**得记住每个 token 当时是用哪个 model 版本生成的**。异步训练里 model 几分钟更新一次，跑了几小时的 trajectory 经历过几十个版本，全记下来工程上做不到。

GLM-5 的招式叫 **Direct Double-sided Importance Sampling**——两步偷懒：

**第一步：用 rollout 时直接记下来的概率当"旧策略"代理**——不再追究"具体是哪个版本的策略生成的"，反正 rollout 时这个 token 是按 0.8 的概率出来的，就用 0.8 当分母。简单粗暴，但工程上可行。

**第二步：双侧 clip**——比例如果太离谱，直接 mask 掉不训练。

"太离谱"具体怎么定？设两个阈值 `1 - ε_low` 和 `1 + ε_high`。**GLM-5 论文里给的实际数值是 `ε_low = 0.2`、`ε_high = 0.28`**——也就是说区间是 **0.8 ~ 1.28**，**上侧故意比下侧松一点**。**比例落在这个区间内** → 正常训练；**比例跑出区间**（比如 0.3 或 2.5）→ 这个 token 直接从训练样本里踢掉。

为什么上下不对称？因为**"新策略比旧策略更喜欢这个 token"是好消息**（说明这个 token 应该被强化的方向上，新策略已经在自己走了，给它多一点容忍空间继续训）；而**"新策略比旧策略更不喜欢这个 token"是危险信号**（如果模型已经开始嫌弃这个 token，再用旧策略的高概率去强化它，就是在逼模型往它不认同的方向走，必须卡严一点）。**这个不对称是设计的，不是 typo。**

为啥要 clip？**比例跑出区间说明新旧策略对这个 token 的看法差异巨大** ——继续训练会被极端样本的巨大梯度污染，反而把模型带歪。**宁可丢掉这条信号也不要污染整体**。

这就是为啥叫 **Double-sided**（双侧）——上下两侧都设阈值，不像标准 PPO 只 clip 上侧。

**第三个坑：分阶段训练的遗忘问题**

GLM-5 的后训练是**三段流水线**：先训推理能力（Reasoning RL）→ 再训 Agent 能力（Agent RL）→ 最后训通用对话能力（General RL）。问题是：**训第二段时，第一段学到的推理能力会被冲掉一部分；训第三段时，前两段都会被冲掉**——典型的"灾难性遗忘"。

GLM-5 的招式叫 **On-Policy Cross-Stage Distillation**——三段 RL 之间夹一道**蒸馏**：用前一段训完的 checkpoint 当老师，把它的能力**主动蒸回**当前训练的学生。这样学生既学到当前阶段的新能力，又不丢前一段的旧能力。

【为什么这套 infra 重要】

加上 **超过 10000 个真实 SWE 训练环境**（用 RepoLaunch 框架自动从真实 GitHub repo 构建 Docker 环境，跨 Python/Java/Go/C/C++/JS/TS/PHP/Ruby 9 种语言），GLM-5 是国内第一个**把"长程 Agent 任务的工程化训练"完整摊开写在技术报告里的团队**。

但这跟模型本体的"智能上限"是两回事——**Agent infra 不会让模型变聪明，但能让一个本来就够聪明的模型把活干完**。Agent infra 的工程化做得再漂亮，最终能跑多远还是取决于**模型本体的能力上限**——而模型本体的能力是预训练和后训练路线决定的，**和 infra 无关**。这条规律下一章引用 Zvi 评测和我个人 5.1 体感时会有具体证据，先记住这条因果——**Agent benchmark 漂亮 ≠ 模型本体强**。

【硬货五：5.2 才有的 1M 上下文——看了 V4 之后追的方案】

这是这次 5.2 宣传里最响的一条。5.0 和 5.1 都还停在 200K，**1M 是 5.2 才落地的**——而且时间线非常精确地卡在 DeepSeek V4 发布之后两个月。

V4 在 2026-04-25 发布时，带着完整的 CSA + HCA + 1M 上下文打法登场——**KV cache 显存从 V3.2 的 1.6GB 砍到 228MB（50K 上下文场景），省了 86%**。这一刀对整个国产开源界都是震撼。**两个月后 GLM-5.2 出来**，没用 V4 的 CSA + HCA 路线，自己整了一套**新组合：IndexShare + KV8 + LayerSplit + HiSparse**。

**5.2 的 1M 实现机制——IndexShare 详解**

要看懂 IndexShare 在干嘛，得先把 DSA（DeepSeek Sparse Attention，DeepSeek V3.2 首创）的工作机制说清楚。GLM-5 系列的注意力底座就是抄的这一套，5.2 改的是其中的"索引器"环节。

【先说 DSA 在干嘛】

标准 Transformer 每一层注意力的工作量是 **O(L²)** 的——L 是上下文长度，每个 token 都要看完所有历史 token。L = 1M 时这个二次方爆炸到天上，连 H100 都顶不住。

DSA 的招数是**稀疏化**——每个 query 不用看全部 100 万个 token，**只看其中最相关的几千个**就够了。怎么挑这几千个？这就是 **Lightning Indexer（轻量索引器）** 的活：

- 给每个 query token 和每个历史 token 计算一个"相关度分数"
- 用一个**轻量级的子网络**算（不是完整 attention，比 attention 便宜一个数量级）
- 选出 top-k（比如 top-2048）个最相关的 token
- 让真正的 attention 只看这 top-k，剩下的全部忽略

这一层 Indexer 是 DSA 的核心——**它本身不参与最终注意力计算，只负责"挑出该看哪几个"**。可以把它理解成**一个工头，先扫一眼全场 100 万个工人，告诉真正的 attention 模块："这 2048 个有用，其他不用管"**。

V3.2 / GLM-5 / GLM-5.1 都是这套打法——**每一层都有自己的 Indexer**，每层独立挑自己的 top-k。

【现在说 IndexShare 改了什么】

GLM-5.2 的观察是这样的：**相邻的几层，挑出来的 top-k 重合度其实非常高**——也就是说，**第 1 层的工头扫完一圈选出 2048 个"重要工人"，第 2、3、4 层的工头扫一圈选出来的，跟第 1 层基本是同一批**。

既然如此，**为啥要让 4 个工头各扫一遍**？让第 1 层的工头扫一次，结果给后面 3 层共用就行。

具体机制：

- 把 80 层注意力层**每 4 层分成一组**
- **每组的第 1 层算 Indexer，得到一份 top-k 索引**
- **后面 3 层不算 Indexer 了**，直接复用这份索引

省了多少？**Indexer 计算量降为原来的 1/4**——4 层里只有 1 层算，剩下 3 层蹭。在 1M 上下文下，整个 Indexer 环节占模型推理 FLOPs 的大头，**单 token 总 FLOPs 降到原来的 1/2.9**。

【其实更准确地说——IndexShare 是 DSA 的演化版本】

仔细想一下，**IndexShare 和 V4 的 CSA 长得真的很像**——两者都是"**取 4 个相邻的东西，共享一个结果**"。区别只在于"**4 个什么的相邻**"：

| 维度 | CSA（V4） | IndexShare（5.2） |
|---|---|---|
| 4 是什么的 4 | **序列方向**——4 个 **token** 压成 1 条 KV | **深度方向**——4 个 **layer** 共享一份 indexer 输出 |
| 共享的是什么 | 压缩后的 KV cache 槽 | 索引器算出来的 top-k 索引 |
| 信息损失 | **物理上有**——4 个 token 加权池化合并不可逆 | **不直接损失**——KV 全在，只是后 3 层用了第 1 层的"选择视角" |
| 节省方向 | 既少算又少存 | 只少算不少存 |

两者赌的核心思想完全一样——**"相邻的东西高度相关，可以合并"**。CSA 赌"相邻 token 信息冗余"，IndexShare 赌"相邻 layer 注意力模式冗余"。**赌的内容不同，赌的姿势相同**。

**但要客观说一句——选不动 KV 的方向是合理的工程决策**。V4 的 CSA 为了少存付出了**真实的精度代价**——4 个 token 池化合并之后，原始信息物理上回不来了。这个代价不是理论上的，**实际部署中已经能感受到**：长上下文场景下"看不清前面交代的细节"是 CSA 的结构性盲区。GLM 选 IndexShare 这个不动 KV 的方向，避开了这个盲区。

**所以这条路线选择不算"抄"——其实更准确地说，IndexShare 是 DSA（V3.2 的原始稀疏注意力）的一个演化版本**：

- V3.2 DSA：每层独立算 indexer 选 top-k
- 5.2 IndexShare：把 indexer 在深度方向上"复用"——每 4 层共享一份

它和 CSA 的相似（都是"4 + 共享"）更像是两个团队**独立观察到"相邻冗余可以合并"这条规律之后**，选了不同方向去实现。V4 在序列方向合并、5.2 在深度方向合并——**两条路线都成立，都不算抄对方**。

但话说回来——**如果 V4 4 月不发布带 1M 的版本展示"可以做到 1M"，5.2 这边是不是会有动力两个月内追出 1M？这就难讲了**。技术上的"独立演化"是一回事，**产品节奏上"追"的成分仍然在**。

【这套设计的代价是什么】

理论代价：**第 2、3、4 层的 attention 被"绑定"在第 1 层的视角上了**——本来这几层可以独立挑出自己认为重要的 token，现在被迫接受第 1 层挑过的。如果某些深层 attention 需要关注"第 1 层没看到的"token，**这个机制就会丢信息**。

智谱怎么辩护？两个论据：

1. **实证：1M 上下文下，每 4 层共享一个 Indexer 在大部分 benchmark 上和"每层独立 Indexer"没显著差异**——他们说，相邻层的注意力模式本来就有强相关性，共享不掉精度
2. **训练策略匹配**：GLM-5.2 不是把训好的 5.1 改成 IndexShare 就拿来用，而是**从中训阶段（128K sequence length）就用 IndexShare 训**——让模型从训练时就习惯"4 层共享一份注意力索引"这个约束，预训练地形从一开始就是为这个约束塑造的

【顺带还修了一个 MTP 的小毛病】

5.2 借着 IndexShare 还顺手修了 MTP 一个训练-推理 KV cache 来源不一致的老问题，效果是投机解码的接受长度涨了 20%。机制细节这里就不展开了——本质是个工程层面的小修补，对全文主线没什么影响。**而且 MTP 主要是训练时强迫模型"想得更远"的辅助手段，实际推理时未必会启用**。感兴趣可以去看 HuggingFace 官方博客的 KVShare 那一段。

【说说 KV8、HiSparse、LayerSplit】

国内媒体宣传 5.2 时常常并列四个新名词——"IndexShare + KV8 + LayerSplit + HiSparse"，听起来好像 5.2 一口气整了四个新机制。但去看官方 HuggingFace 技术博客，**真正有详细介绍的只有 IndexShare 和 KVShare（IndexShare 的姐妹机制，前面已经讲过——配合 MTP 用的）这两个**。剩下的两个：

- **LayerSplit**：官方博客只一句话带过——"我们引入了更细粒度的内存管理和并行化策略"。所以它**不是 attention 层的新稀疏机制，是推理引擎层的内存管理优化**（KV cache 怎么跨 GPU 分布、显存怎么调度）
- **KV8 / HiSparse**：**官方博客根本没出现这两个词**——是国内自媒体自己加的概括性描述，连机制本身是什么都说不清楚

所以**别被"四个新名词"忽悠**——5.2 真正在注意力机制上做的事情就是 IndexShare（和它的 MTP 配套 KVShare）。剩下的所谓"新机制"要么是**推理引擎工程优化**（LayerSplit），要么**就是自媒体造词**（KV8 / HiSparse）。

这也是公众号写国产 AI 模型一个常见的坑——**国内媒体习惯把官方一句话写过的东西包装成"独家技术解读"，加几个看起来很厉害的英文缩写让文章显得有料**。但你真去找原始论文/官方博客，会发现这些缩写要么不存在、要么对应的是工程层而非架构层。

**和 V4 的路线对比**：

| 维度 | DeepSeek V4 | GLM-5.2 |
|---|---|---|
| MLA 多头潜注意力 | V2 首创（2024-05） | 照搬 + 改 head 配置 |
| DSA 稀疏注意力 | V3.2 首创 | **直接继承，名字都没换** |
| Lightning Indexer | V3.2 首创 | **直接继承 + 共享改造（IndexShare）** |
| Muon 优化器 | V4 引入 | **继承，加了 Muon Split** |
| **1M 上下文实现路线** | **CSA + HCA 序列压缩（每 128 token 压成 1 条）** | **IndexShare 共享索引器（4 层共享一个）** |
| KV cache 显存优化 | **激进——50K 上下文从 1.6GB 砍到 228MB** | 保守——靠 LayerSplit 工程分布存储 |
| 后训练 Agent infra | 略 | **GLM 真护城河** |

**关键判断**：

- **V4 走的是"模型本体改造"路线**——在 V3.2 DSA"少算"的基础上又加了 CSA + HCA 序列压缩，把"少算"和"少存"两条赛道一起占住
- **5.2 走的是"V3.2 路线 + 索引器复用改进"**——继续在 V3.2 已经开好的"少算"赛道上往前走一步，让稀疏选择本身更便宜，但没碰"少存"

哪条路更好？**取决于评估维度**：

- **看显存效率**：V4 的 CSA + HCA 更省（结构性压缩 KV）
- **看推理速度**：5.2 的 IndexShare 在 1M 长度下 FLOPs 优势明显
- **看保真度**：V4 的压缩有信息损失（CSA 压缩等待区会丢精度），5.2 不压缩 KV 所以理论上更保真

我个人的判断是**两条路各有取舍。5.2 不能说是"抄"V4——代码和实现不一样；但核心思想（"4 + 共享"）毫无疑问是从 V4 学的，只是换了一个方向再实现一次**。如果 V4 4 月不发布带 CSA 的 1M 版本，5.2 这边几乎不可能 6 月就拿出"1M 稳定"——你很难相信 GLM 团队两个月里独立想出"4 个相邻共享"这个核心招式。

但"两个月做出 1M"听起来像是个赞誉——其实未必。为什么必须两个月就追出来？这条问题留到第四章【股价】那一章再回答——简单说就是：**上市公司的逼迫**。GLM-5.1 3 月底刚发完、V4 4 月底发布、5.2 必须 6 月发——这不是工程节奏，是季度财报节奏。**留给真正打磨的时间不够**——这是后面 Zvi 评测里"benchmark 越不可针对优化分数越拉胯"的根因之一。

【小白翻译】**KV cache（键值缓存）**：Transformer 推理时为了不重复计算，把历史 token 的 Key/Value 向量存起来。1M 上下文意味着要存 100 万条 KV cache——显存能不能装得下、查找能不能省算力，是长上下文部署的核心瓶颈。

【三代演进对比】**"少算"和"少存"是两条独立的赛道**：

- **V3.2 DSA**（DeepSeek 首发）：开创"少算"赛道——KV cache 全存着，但注意力计算时用 Lightning Indexer 选 top-k 不全看，**省计算不省存储**
- **V4 CSA + HCA**：在 DSA 之前加一层 KV cache 压缩——**既少算又少存**（V3.2 50K 上下文要 1.6GB，V4 只要 228MB）
- **GLM-5.2 IndexShare**：**继续在"少算"赛道上往前走**——4 层共享一个 indexer，indexer 计算量降 75%。但没碰"少存"——KV cache 该多大还是多大

所以 5.2 这条路不是"开辟新赛道"，是**在 DeepSeek V3.2 已经开好的"少算"赛道上再走一步**——把 V4 的"少存"那条赛道留给 DeepSeek 独享了。

────────────────────

💡 打个比方：V4 像是把图书馆的书都重新装订（每四本装一本），物理上少了；5.2 像是图书馆书不变，但把找书的索引卡片改成共享的——你查一次，相邻几层都用这次的结果。两个方向都能让"查 100 万本书"变快，但物理逻辑完全不同。

【一个理论解释：蒸馏的代价】

Zvi 在文章里给了一个很扎实的判断——**蒸馏（distillation）模型在 benchmark 类任务上往往超水平发挥，但在不常见的任务上表现下降**。

什么是蒸馏？**用一个强模型（teacher，比如 Claude Opus）的输出当训练数据，喂给学生模型学习**。优点：能让学生快速接近 teacher 的能力。缺点：

1. **学生只学会了 teacher 见过的题型**——teacher 没见过的新题型，学生完全没准备
2. **学生继承了 teacher 的所有失败模式**——做不了独立判断
3. **学生学的是"答案表面"，不是"思维过程"**——遇到结构上类似但本质不同的题，会按"形似"答错

这条规律我们在 213 期《后训练：SFT、RL、DPO、蒸馏…这些花活儿，到底动了模型哪根筋？》（ https://mp.weixin.qq.com/s/kMwrIfhIfPlOHqeeUhuwow ）讲过——用更狠的话总结就是：

> **"没有任何一种后训练能造出地形上本来不存在的山。能力都是预训练给的，后训练只决定你走到哪座山。"**

蒸馏只是后训练的一种花样而已——**它在已有的本我流形上动土，不会造出新山**。学生模型的能力上限被自己的预训练地形锁死了，蒸馏再猛也是把学生引到"老师走过的那座山"附近徘徊，**永远不可能超过老师**。

GLM-5 技术报告里 RL 的 teacher 模型他们列了"GPT-5.2 xhigh 和 Gemini 3 Pro Preview"——Opus 没写。但 Zvi 的判断和 Hacker News 用户的体感都指向同一个结论：**这模型的输出风格、失败模式、强项弱项的分布，和 Claude 高度同构**。

这就引出了一个有意思的现象学问题。

────────────────────

💡 打个比方：**蒸馏出来的模型，看起来像 Claude，但只是表演 Claude**。就像一个只看过 Claude 答题本的学生——开卷考能背答案，闭卷做新题就露馅。**僵尸 AI 蒸馏的看起来就是会很像**——因为它们学的就是"那个样子怎么写"，而不是"为什么这么写"。

────────────────────

【一个 5.1 的活体证据：我自己】

我个人买过 GLM-5.1 的海外版。用两天直接秒退——**海外版的退款政策很痛快，按实际使用天数扣费，剩下的钱秒退到信用卡**。

那时候宣传是"编程贴 Opus 4.6 开大""Coding Plan 秒断货"。实际上手：

- **简单代码任务能写，复杂逻辑就开始幻觉 API**
- **长上下文场景下，前面交代的约束写到后面就忘**
- **个人体感：多轮工具调用的稳定性比 Sonnet 4.5 都不行**——这只是我个人的使用感受，没做系统对比、不代表通用结论。但有一点要说清楚：**Sonnet 是 Anthropic 的中号模型，定位上比 Opus 低一档、价格也便宜得多**——GLM-5.1 当时宣传是"对齐 Opus"，按这个口径就算和 Sonnet 打平都说不过去

这件事和 Zvi 的判断对得上——**benchmark 漂亮，真实工作流拉胯**。

5.1 是这个套路，5.2 大概率也是。**技术报告里 Agentic RL infra 是真做了，但模型本体的智能上限被蒸馏路线锁住了**。

────────────────────

💡 打个比方：技术报告里的 Agent infra 像是给了一辆好车（变速箱、悬挂、轮胎都换了顶级），但发动机还是原来那台。车手开起来感觉操控提升了，但极速跑不过对手——因为发动机马力就那么大。

◆ 第四章：理解中文圈集体打满分的根本逻辑——股价

━━━━━━━━━━━━━━━━━━━━

讲到这里，回头想第一章那个问题：**为什么中文圈对 GLM-5.2 的评价几乎清一色"开源新王"？为什么没有一篇敢直接说不好？**

光说"流量逻辑 + 赞助逻辑 + 对比基线缺失"还不够——还有一个**更根本的力学**。把这条画出来，整个故事就闭合了。

【智谱的股价节奏】

| 时间 | 事件 | 股价/市值 |
|---|---|---|
| 2026-01-08 | **港交所上市** | 发行价 116.2 港元，市值 528 亿港元 |
| 2026-02 | **GLM-5 发布** | 股价开始拉升 |
| 2026-03-27 | **GLM-5.1 发布 + 提价 10%** | 早盘涨近 18% |
| 2026-06-13 | **GLM-5.2 发布 + 开源** | 继续拉升 |
| 2026-06-22 | **GLM-5.2 发布后第二个交易日** | **股价 2900 港元，市值首破 1 万亿港元** |

**不到半年股价涨 20 倍，市值从 528 亿涨到 10000 亿港元**。这个涨幅和模型迭代节奏严丝合缝——**每发一个新版本就涨一波**。

【这个估值是个什么概念】

- 1 万亿港元约合 **9000 亿人民币**——超过了京东、快手的市值
- 智谱 2025 年营收 **7.24 亿元人民币**，净亏损 47.18 亿元
- 当前**市销率 1000 倍以上**——也就是说投资者愿意为 1 元营收付 1000 元股价
- 对比：英伟达的市销率约 30 倍，OpenAI 估值 / 营收比约 50 倍

**1000 倍市销率意味着什么？意味着市场押注的不是当前营收，是"国产 AI 第一股"这个标签**。这个标签值不值这么多钱，取决于一个核心叙事能不能持续：**"GLM 在追上 frontier"**。

【这条叙事和"benchmark 屠榜"是死循环】

现在你就理解了为什么**国内每篇评测都得说"对齐 Opus"**：

- 智谱市值 1 万亿港元 → 必须不断强化"追上 frontier"叙事
- 强化叙事 → 必须每个新版本 benchmark 上都赢
- 媒体如果质疑 benchmark → 影响叙事 → 影响股价 → 得罪所有持仓的投资人和分析师
- **所以所有人都在维护这套叙事，谁先说"其实差距还很大"，谁就是空头**

这不是阴谋论，是**纯粹的金融力学**。1000 倍市销率的股票需要持续利好驱动，**任何客观评测都会被市场解读为利空**。海外评测圈（Zvi、Hacker News 用户）没有这个金融压力——他们可以诚实地说"benchmark 漂亮但 frontier 差距在扩大"。中文圈不行，**说真话的成本太高**。

【对比 DeepSeek：技术叙事 vs 资本叙事】

DeepSeek 2026 年至今没有上市，也没有公开融资。**这让 DeepSeek 团队可以专注做架构创新**（CSA + HCA、FP4 QAT、Anticipatory Routing）——做出来好就发，做不出来就不发，**不需要按季度交付"追上 frontier"的故事**。

智谱不一样。**上市公司的逼迫是连续的**——每个季度财报、每个新版本发布、每次股价波动都需要"技术利好"。这就导致 GLM 系列必须按月迭代（5.0 → 5.1 → 5.2 三个月内三个版本），即使内部知道差距还很大也必须发，**因为不发股价会跌**。

GLM-5.2 的 IndexShare 那套架构是不是真做了功夫？是。但**速度被资本逼着拉得过快**——V4 4 月出 1M，5.2 6 月就得拿出 1M，**留给真正打磨的时间不够**。Zvi 的判断"benchmark 越不可针对优化，分数越拉胯"，本质上反映的就是**赶工期 + benchmark 导向训练**的结构性后果。

这条逻辑也解释了**为什么 GLM 要在技术报告 abstract 级别踩 DeepSeek 一脚**（第二章【硬货二】讲的那一脚）——上市公司必须制造对立叙事来支撑估值，**踩同行 = 抬自己 = 维护股价**。DeepSeek 不上市没有股价压力，所以可以默默做事不回踩。这不是两家团队人品差异，是**资本市场结构强加给上市公司的行为约束**。

────────────────────

💡 打个比方：DeepSeek 像是个独立工作室，做出新东西才发；智谱像是个上市的电影公司，每个季度必须有大片上线，质量得让位于档期。前者偶尔出神作，后者保证不断片。**作为读者你能看哪部，取决于你的需求是"看好电影"还是"看新电影"**。

━━━━━━━━━━━━━━━━━━━━

◆ 第五章：那 5.2 到底值不值得用？

━━━━━━━━━━━━━━━━━━━━

把账算清楚之后，给一个**结构化的判断**：

【适合用 GLM-5.2 的场景】

- **预算敏感 + 任务标准化**：写 SQL、生成接口文档、改 bug 这种标准任务，benchmark 上的成绩在工程上是站得住的
- **国产 GPU 部署刚需 + 能接受 expressiveness 代价**：昇腾、寒武纪、海光这些卡，5.2 是真适配过的，企业内网部署有现实优势。前文【硬货二】说的"MLA-256 在架构表达力上有代价"是事实，但**对"必须用国产卡 + 不接闭源 API"的企业用户，能跑 = 有用，代价能接受**
- **Agent 工作流**：技术报告里那套异步 RL infra 是真功夫，跑 Agent 链式任务比同价位选项稳

【不适合用 GLM-5.2 的场景】

- **写作 / 长篇编辑**：复杂格式约束、风格一致性这种"meta-rule"任务，蒸馏模型撑不住
- **独立判断 / 二次验证**：蒸馏自 Claude 的模型不能用来 review Claude 的输出，失败模式共享
- **frontier 探索任务**：研究新问题、没人写过教科书答案的领域——蒸馏模型只会"按 teacher 见过的样子答"，遇到真新问题会原地打转

【一个有点尖锐的判断】

国产开源模型这一波，几乎都在走同一条路：**用 frontier 闭源模型的输出做蒸馏数据，在 benchmark 上对齐 frontier，宣传"我们打平了"**。

这条路在战术上完全可以理解——**蒸馏比从零搞 RLHF 便宜得多，benchmark 分数好看融资好融**。但战略上，这条路有天花板——**蒸馏出来的模型永远是 teacher 的影子，做不到超过 teacher**。

DeepSeek 走的是另一条路——**架构创新（CSA/HCA、Lightning Indexer、FP4 QAT）+ 自研 RL 训练范式**。V4 在算法竞赛、长上下文、推理能力上**真的有原创贡献**。这才是为什么 Hacker News 那条评论说"DeepSeek V4 Flash 整体表现更好"——不是 V4 benchmark 分数更高（GLM-5.2 在 benchmark 上确实赢了），是**V4 在 benchmark 之外的真实工作流里更靠谱**。

GLM-5.2 上线后所有国内媒体在欢呼"开源新王"。但 Zvi 这一句话总结得很冷静：

> "**It updates expectations about open-model progress, though the frontier gap remains substantial and potentially widening.**"
>
> （它更新了我们对开源模型进展的预期，但和 frontier 的差距仍然显著，而且可能在扩大。）

**差距在扩大**——这是关键。每代国产开源模型 benchmark 上都越来越漂亮，但在 frontier 真正定义"什么叫前沿能力"的领域（长程一致性、自我修正、独立判断、复杂格式遵守），国产模型和 Opus / GPT-5.x 的距离不仅没缩短，反而在被甩远。

因为 frontier 在做架构创新和数据创新，国产开源在做蒸馏。

━━━━━━━━━━━━━━━━━━━━

◆ 结尾

━━━━━━━━━━━━━━━━━━━━

GLM-5.2 不是垃圾。技术报告里 Muon Split、MTP 共享、异步 Agent RL infra 都是真功夫。**它是目前最好的国产开源模型，这个判断成立**。

但"最好的国产开源" ≠ "对齐 Opus 4.7"。这两件事差得很远，国内媒体不愿意说，海外评测说得很清楚。

**我自己的一些想法**：

- **想了解技术细节**：读技术报告，arxiv 2602.15763。180 多个作者的工程量在里面
- **想真实测试能力**：别看 benchmark，找一个**非标准化、需要长程一致性的任务**——比如让它按"一段英文一段中文"改一篇你写的稿子，或者让它在 5000 行代码里精确执行一个 meta-rule（如"所有函数名前缀必须是 svc_"）
- **想替代 Opus 4.7**：再等等。GLM-5.3、5.4 应该还会继续追

**底层规律没变**：

**蒸馏出来的模型，长得像 teacher，但不是 teacher。**

**benchmark 是给可以针对训练的题准备的，真实工作流不接受针对训练。**

**国产开源在追赶，但追赶的姿势决定了能不能真正追上。**

GLM-5.2 是这条路上的一个新里程碑——但不是终点站。下次再有"国产模型对齐 Opus"的宣传时，欢迎把这篇文章翻出来重读一遍。
