171 期（ https://mp.weixin.qq.com/s/JPxR8rHSb1Pko3vq2ukY3A ）我们在两台 DGX Spark 上跑通了 DeepSeek V4 Flash 280B 的推理服务——18 个坑，24 小时，最终看到了"你好，我是 DeepSeek"那行字。

但那篇文章最后一段写了一句话："推理服务对我用处不大……我真正想做的是提取中间层激活值。"

vLLM 是一台封得严严实实的黑箱。它接收 prompt，返回生成结果，中间发生了什么你看不到。你没法在第 28 层的 forward pass 中间插一个 hook，把那一刻的 hidden state 存下来。你也没法对比两个不同 prompt 在同一层的激活值差异。

对做 AI 可解释性研究来说，vLLM 就像 nginx——你可以用它提供 HTTP 服务，但你没法在里面打断点看 HTTP 解析的中间状态。你需要的不是 nginx，是 Wireshark。

◆ 为什么不能在 vLLM 里加 hook

━━━━━━━━━━━━━━━━━━━━

先说清楚为什么需要一个全新的平台，而不是在 vLLM 上改改。

vLLM 是推理服务框架。它的目标是**吞吐量和延迟**——continuous batching（动态拼批）、PagedAttention（KV cache 分页管理）、CUDA graph（编译计算图减少调度开销）。为了这些优化，vLLM 把模型的内部状态全封装在自己的调度器里，外部没有接口拿到中间层的 hidden state。

你可能会想：PyTorch 的 `register_forward_hook()` 不是可以在任意层插 hook 吗？

理论上可以。但 vLLM 的模型执行路径不是标准的 PyTorch `nn.Module.forward()` 调用链——它用了自己的 `ModelRunner`，把 attention 计算、KV cache 管理、batch 调度全耦合在一起。你要么改 vLLM 的内部代码（几千行，版本一更新全废），要么等社区出 `--return-hidden-states` 接口（SGLang 有实验性支持，但不支持 sm_121 + V4）。

我们选了第三条路：**用 DeepSeek 官方的推理代码**。

DeepSeek 发布 V4 的时候，除了 HuggingFace 权重，还附带了一个 `inference/` 目录——`model.py`（模型定义）、`generate.py`（推理入口）、`kernel.py`（GPU kernel）。这是标准的 PyTorch 代码，`model.py` 里每个 `Block` 就是一个 `nn.Module`，可以像教科书一样用 `register_forward_hook()` 提取任意层的中间状态。

但官方代码也不能直接提取激活值——`generate.py` 只输出最终 token，没有 hook 框架。**即使你有 B200，用官方代码也拿不到中间层激活值。** 我们要做的是：在官方 `model.py` 上加 hook 框架。这件事跟硬件无关。

不过我们恰好还多了一层坑：**这份代码在 DGX Spark 上跑不了。**

◆ sm_120 的生态空白——不是 sm_121 孤儿，是整条产品线没人管

━━━━━━━━━━━━━━━━━━━━

171 期把 DGX Spark 的 GPU（sm_121）称为"孤儿架构"。这个说法不够准确。准确的说法是：**sm_120 整个系列都是孤儿。**

NVIDIA 的 Blackwell 架构分两条线：

| 架构 | 代表产品 | 定位 | 分布式推理生态 |
|------|---------|------|-------------|
| sm_100 | B100 / B200 / GB200 | 数据中心 | 完整：DeepGEMM、TileLang、FlashAttention、vLLM 原生全支持 |
| sm_120 | RTX 5090 / RTX 5080 | 消费级桌面 | **空白** |
| sm_121 | DGX Spark GB10 | 消费级变体 | **空白** |

sm_120 是消费级 Blackwell。RTX 5090 最大 32GB 显存，你拿它打游戏、跑 Stable Diffusion、单卡推理 7B 模型——没人拿两张 5090 组分布式集群跑 280B。

💡 翻译成人话：没有市场，就没有人做基础设施。没有人做基础设施，就没有 kernel 库。DeepGEMM 不支持 sm_120，TileLang 不支持 sm_120，FlashAttention 不支持 sm_120——不是技术上做不到，是没人愿意花时间做。事实上我们用纯 PyTorch 在一天之内就替换了全部 6 个 kernel——毕竟现在可以 vibe coding，技术门槛不高，就是没人觉得值得做。

DGX Spark 是第一个拿消费级架构干数据中心活的产品——128GB 统一内存，两台直连跑 280B。它撞上了一个从未存在过的使用场景，自然也撞上了一个从未被填补过的生态空白。

DeepSeek 官方 `kernel.py` 用的是 TileLang——一个 GPU kernel JIT 编译框架。TileLang 0.1.8 支持 sm_75（Turing）、sm_90（Hopper）、sm_100（数据中心 Blackwell），**不支持 sm_120/sm_121**。

171 期我们绕过了 DeepGEMM，这次又撞上了 TileLang。同一个问题，不同的 kernel 库，同样的根因。

━━━━━━━━━━━━━━━━━━━━

◆ 方案：不改 model.py 一行代码

━━━━━━━━━━━━━━━━━━━━

核心思路很简单：**在 import 层做替换**。

DeepSeek 的 `model.py` 开头有一行：

```python
from kernel import act_quant, fp4_act_quant, fp8_gemm, fp4_gemm, sparse_attn, hc_split_sinkhorn
```

我们不动 `model.py`，写一个 `kernel_sm121.py` 实现同样的 6 个函数，然后在启动脚本里：

```python
import kernel_sm121
sys.modules['kernel'] = kernel_sm121
```

Python 的 module system 被"欺骗"了——`model.py` 以为自己导入的是 `kernel`，实际拿到的是 `kernel_sm121`。**模型定义、TP 分片逻辑、forward 流程全部原封不动。**

这件事能做到，是因为 DeepSeek 的架构设计很干净：模型逻辑和硬件 kernel 通过函数签名解耦。6 个函数，签名固定，输入输出的 shape 和 dtype 文档里写得清清楚楚。你只要保证函数行为一致，底层怎么实现无所谓。

◆ FP4 解包：这个格式连 PyTorch 都不会转

━━━━━━━━━━━━━━━━━━━━

6 个 kernel 里最有意思的是 `fp4_gemm`——FP8 激活值乘以 FP4 权重的矩阵乘法。

DeepSeek V4 Flash 的专家权重用 FP4（`float4_e2m1fn_x2`）存储——每个浮点数只有 4 位，两个打包成一个字节。256 个专家 × 3 个矩阵 = 768 个 FP4 权重张量，占了模型 158GB 权重的大头。

**FP4 是 280B 模型塞进 256GB 内存的前提。** 如果用 FP8（每个浮点数 8 位），权重翻倍到约 300GB，两台 Spark 放不下。FP4 不是精度优化选项，是能不能跑的前提。

你可能觉得：FP4 转 FP32 不就是个 `.to(torch.float32)` 的事？

试了。GPU 上报 `DynamicCast.h assertion failed`，CUDA device-side assert 直接崩。CPU 上报 `copy_kernel not implemented`。

💡 翻译成人话：`float4_e2m1fn_x2` 这个数据类型是 PyTorch 2.11 新加的，类型定义有了，但 `.to()` 转换函数没实现。你可以创建一个 FP4 张量，但不能把它转成别的类型。像是买了一把螺丝刀，螺丝刀柄有了，但刀头没配。

解法是手动解包。FP4 E2M1 格式只有 16 个可能的值，建一张查找表：

```python
FP4_TABLE = torch.tensor([
    0.0, 0.5, 1.0, 1.5, 2.0, 3.0, 4.0, 6.0,       # 正数（高 bit = 0）
    0.0, -0.5, -1.0, -1.5, -2.0, -3.0, -4.0, -6.0,  # 负数（高 bit = 1）
])
```

然后把 FP4 张量当 uint8 看（两个 FP4 值打包在一个字节里），用位操作拆出低 4 位和高 4 位，查表得到 float32 值：

```python
raw = b.view(torch.uint8)           # 把 FP4 当字节看
low  = raw & 0x0F                   # 低 4 位 → 第一个值
high = (raw >> 4) & 0x0F            # 高 4 位 → 第二个值
result = torch.stack([FP4_TABLE[low], FP4_TABLE[high]], dim=-1).flatten(-2)
```

**8 行代码，解决了 PyTorch 自己没实现的功能。** DeepSeek 官方的 `convert.py` 里也是同样的方法——说明 FP4 cast 不可用不是 bug，是这个版本的 PyTorch 确实没做。

◆ 统一内存的教训——在一个碗的厨房里做饭

━━━━━━━━━━━━━━━━━━━━

6 个 kernel 的替换花了大半天。但真正让我差点掀桌的是**权重加载**。

先说背景。DeepSeek 官方的权重加载流程是这样的：

1. 跑 `convert.py`，把 HuggingFace 的 46 个小 shard（每个约 3.5GB）合并转换成 TP 分片格式——`model0-mp2.safetensors`（约 77GB）
2. 跑 `generate.py`，调 `safetensors.torch.load_model()` 加载这个 77GB 文件

在数据中心的 H100 上，CPU 有几百 GB RAM，GPU 有 80GB 独立显存，这套流程没问题。

**但 DGX Spark 的 CPU 和 GPU 共享 128GB 统一内存。**

`load_model()` 的默认行为是先把整个文件加载到 CPU 内存（通过 mmap），然后逐个张量拷贝到 GPU。在统一内存架构下：

```text
模型骨架在 GPU 上:     ~82 GB
权重文件 mmap 活跃页:   ~77 GB
合计:                  ~159 GB > 128 GB → OOM
```

CPU 和 GPU 在抢同一碗汤。

我试了四种方案，全挂：

| 方案 | 做了什么 | 怎么挂的 |
|------|---------|---------|
| `load_model(file)` 直接加载 | 77GB 一次性加载到 CPU | CPU + GPU 竞争统一内存 → OOM |
| `safe_open(device="cuda")` | 直接加载到 GPU | mmap 77GB 的活跃页仍占物理内存 → OOM |
| meta 设备 + 流式加载 | 先创建空壳再填充 | meta tensor 去重导致 6 轮 debug 地狱 |
| 分批加载 + 定期关文件 | 试图释放 mmap 页 | mmap 页累积不释放 → 还是 OOM |

第三个方案值得多说两句。PyTorch 的 `torch.device("meta")` 可以创建不占内存的"空壳"模型——只有 shape 和 dtype，没有实际数据，0 字节内存。你可以查看它的 shape，但不能读取它的值。

💡 翻译成人话：meta tensor 就像建筑图纸——标了每个房间多大、什么材料，但房子还没盖。你可以拿图纸算总面积、规划布局，但不能住进去。

想法是先用图纸"盖"出模型的空壳（0 内存），再逐个张量从文件加载到 GPU、把空房间填上家具。但 meta 设备有个噩梦般的行为：**多个层共享同一个 meta tensor 对象时，`named_buffers()` 会去重，只返回第一次出现的那个。** 你以为自己修好了所有层的 `freqs_cis` buffer，其实只修了第一层，剩下 42 层还在 meta 上。这个 bug 让我来回改了 6 轮。

最终放弃 meta 方案。改回 `with torch.device("cuda")` 直接在 GPU 上创建骨架（分配 82GB），然后——

**跳过 convert.py，直接读 HuggingFace 的 46 个小 shard。**

每个 shard 约 3.5GB。逐个打开、处理、关闭。关闭后 mmap 页释放，峰值 mmap 只有 3.5GB。

```python
for shard_file in sorted(glob("*.safetensors")):
    with safe_open(shard_file, framework="pt", device="cpu") as f:
        for key in f.keys():
            tensor = f.get_tensor(key)
            # 重命名 key、TP 分片、赋值到模型
            del tensor
    # shard 关闭 → mmap 页释放
```

这正是 vLLM 内部的方案。不是偶然——**统一内存架构下，流式逐 shard 加载是唯一正解。**

💡 翻译成人话：在一个只有一个碗的厨房里做饭，你不能同时端两碗汤。只能盛一勺、倒进锅里、腾出碗、再盛下一勺。46 个小 shard 就是 46 勺，每次只端一勺。

但这意味着 `weight_loader.py` 要在加载的同时做三件事：

1. **重命名**：HuggingFace 用 `model.layers.X.self_attn.q_proj`，官方代码用 `layers.X.attn.wq`
2. **TP 分片**：根据 rank 和 world_size 切分权重——非专家权重按维度切，专家权重按编号分（rank 0 拿专家 0~127，rank 1 拿 128~255）
3. **特殊处理**：`wo_a` 权重需要把 scale 融合进 weight 再转 BF16；FP4 专家权重要从 int8 视图转成 `float4_e2m1fn_x2`

**270 行代码，每一行都是被 OOM 逼出来的。**

### 坑 8：单机创建模型骨架直接死机

单机 128GB，TP=2 的模型骨架需要给 256 个专家全部分配内存。256 个专家 × 3 个矩阵 × FP4 权重 = 远超 128GB。

**直接死机。** 不是 Python 报错，是 Linux OOM Killer 把进程杀了，DGX Spark 黑屏重启。

**教训**：统一内存架构下，OOM 不是报错，是死机。没有优雅降级，直接黑屏。测试之前先算一下内存预算。

**解法**：必须双机。`world_size=2` 时每台只创建 128 个专家的骨架，82GB 刚好塞进 128GB。

────────────────────

### 坑 9-10：NCCL 初始化超时 15 分钟

`dist.init_process_group("nccl")` 挂了 15 分钟后超时。原因是 `NCCL_SOCKET_IFNAME` 写错了——写成了 `eth1`，实际 CX7 网卡接口名是 `enp1s0f1np1`。

改完接口名，还缺 `NCCL_IB_HCA`（RoCE 设备名）和 `GLOO_SOCKET_IFNAME`（Gloo 后端接口）。这些环境变量从 eugr 的 `launch-cluster.sh` 里抄过来才配齐。

**教训**：NCCL 超时的 debug 信息几乎为零——只告诉你超时了，不告诉你为什么。加 `NCCL_DEBUG=INFO` 能看到更多日志，但信息量仍然很大，需要逐行排查。

────────────────────

### 坑 12-16：meta tensor 去重地狱

前面讲过了。总结一下：PyTorch 的 `named_buffers()` 会对共享的 meta tensor 对象去重。43 层的 `freqs_cis` 在 meta 设备上可能共享同一个 tensor 对象（因为 shape 和 dtype 相同），`named_buffers()` 只返回第一次出现的。你修了第一层，以为全修了，实际 42 层还在 meta 上。

试了直接操作 `module._buffers` 字典、遍历所有 module、逐层检查——改了 6 轮才彻底解决。最终放弃 meta 方案，改回 CUDA 骨架。

**教训**：meta device 是为节省内存设计的，但它的去重行为在需要逐层修改 buffer 时是个陷阱。如果你的内存够用，别用 meta。

────────────────────

### 坑 17：`fast_hadamard_transform` 编译失败

DeepSeek 的 model.py 里用了一个 Hadamard 变换，依赖 `fast_hadamard_transform` 这个 pip 包。这个包有 CUDA kernel，不支持 sm_121。

解法很轻量：Hadamard 矩阵就是一个递归构造的正交矩阵，128 维的 Hadamard 矩阵只有 128x128 = 16384 个元素。纯 PyTorch 实现，递归构造 + 缓存，33 行代码。

────────────────────

### 坑 18-20：dtype 不匹配

model.py 里有几个地方对激活值调了 `.float()`（转 FP32），然后传给 BF16 权重做 `F.linear()`。在数据中心 GPU 上用 TileLang kernel 时混合精度没问题，但换成纯 PyTorch 的 `F.linear()` 后会报 dtype mismatch。

解法是 monkey-patch。在 import model.py 之后、forward 之前，替换掉 `model.linear()` 和 `ParallelHead.get_logits()`，加上 dtype 对齐：

```python
def _patched_linear(x, weight, bias=None):
    if weight.dtype in (torch.float4_e2m1fn_x2, torch.float8_e4m3fn):
        return _original_linear(x, weight, bias)
    return F.linear(x.to(weight.dtype), weight, bias)
```

**不改 model.py 一行代码的承诺，靠 monkey-patch 兑现。**

◆ 结果：5.5 秒 forward pass，4 路残差流不对称

━━━━━━━━━━━━━━━━━━━━

双机 TP=2，280B 参数全部加载，每台 GPU 占用 81.8GB。

```text
Prefill OK! 5.5s
Logits shape: [1, 129280], dtype: float32
```

5.5 秒完成一次 forward pass（prefill），覆盖 43 层全部计算。生成阶段约 1.8 秒/token。作为参考，vLLM 推理服务的生成速度是约 15 token/s——我们慢了 27 倍。

为什么这么慢？不是因为每次 forward 都在做 FP4 解包——FP4 权重在加载时就已经从 int8 view 成了 `float4_e2m1fn_x2`，存在 GPU 上不会变。慢的根因是**反量化**。

先解释为什么必须反量化。FP4 权重存在 GPU 上，FP8 激活值也在 GPU 上，但 `torch.mm` 要求两边 dtype 一致——你不能直接算 `torch.mm(fp8_tensor, fp4_tensor.T)`，PyTorch 会报错。所以必须先把 FP4 和 FP8 都转成 BF16，再做矩阵乘法。

TileLang 原版也做反量化——但它用的是**融合 kernel**。

💡 什么是融合 kernel？普通 PyTorch 代码每一行调用一次 GPU——三步操作 = GPU 被叫三次，每次都要从显存读数据、算完、写回显存。融合 kernel 是把三步合成一个 GPU 指令，GPU 只被叫一次，数据读进来在内部流转完所有步骤再写回去。区别就像：分步是跑三趟厨房（拿鸡蛋回桌子、拿锅回桌子、开火回桌子），融合是去一趟厨房全搞定。

TileLang 的融合 kernel 从显存读入 FP4 数据，在寄存器里解包成浮点数，立刻和激活值相乘，结果写回显存。整个过程数据只经过显存一次。反量化的开销藏在计算过程中，不产生额外的显存读写。

我们的纯 PyTorch 版本没有融合 kernel，只能分步做：

```text
TileLang（融合）：  显存 → 寄存器（读 FP4 + 解包 + 乘法 + 写回） → 显存     一次读写
我们（分步）：      显存 → BF16 中间结果 → 显存 → torch.mm 读出 → 显存       三次读写
```

同样的数据在显存里被读写了三次。280B 模型有 43 层，每层有多个矩阵乘法，累积起来就是几十倍的带宽浪费。

另一个慢点是 `sparse_attn`——原版是一个融合的 GPU kernel，所有 token 位置并行处理；我们用 Python for 循环逐位置遍历，完全没有并行。

💡 翻译成人话：原版是流水线工厂，原料进去成品出来中间不停；我们是手工作坊，每一步做完把半成品搬到桌上，再搬到下一个工位。做的事一样，搬运成本差了几十倍。

但速度不重要——**做实验只需要跑一次 prefill（5.5 秒）提取激活值**，不需要自回归生成几百个 token。

这个慢是可以改善的。Triton（一个 GPU kernel 编译框架）在 sm_121 上可用——我们已验证 Triton 3.6.0 能在 DGX Spark 上跑。用 Triton 写融合 kernel，把 FP4 解包 + scale + matmul 合成一个指令，预计能提速 10-20 倍。jasl 在 vLLM 的 sm_120 分支里已经做了类似的事（`deepseek_v4_fp8_einsum_triton`），我们可以参考他的实现适配到自己的函数签名上。另一条路是等生态——等 TileLang 加 sm_120 支持、或 PyTorch 原生支持 FP4 matmul，到时候换回官方 kernel.py 就行，因为我们没改 model.py。这些留作后续优化。

────────────────────

### 第一个实验：layer 28-29 的 4 路 Hyper-Connection 残差流

DeepSeek V4 的 Hyper-Connection 机制维护 4 份 hidden state 副本（`hc_mult=4`），每层通过 Sinkhorn 归一化的混合矩阵动态合并。173 期讲过这个机制的 TP 切分方式——4 份副本不按 TP 切，每台保留完整的 4 份。

我们在 layer 28 和 29（模型的约 2/3 位置）注册了 `register_forward_hook()`，捕获了输入的 4 路残差流：

```python
def make_hook(layer_id):
    def hook_fn(module, input, output):
        x = input[0]  # shape [1, seq, 4, 4096]
        captured[layer_id] = x.detach().cpu().clone()
    return hook_fn
```

结果：

```text
Layer 28 residual: shape=[1, 12, 4, 4096]
  4 streams norms: ['133.00', '166.00', '113.50', '26.88']

Layer 29 residual: shape=[1, 12, 4, 4096]
  4 streams norms: ['132.00', '169.00', '117.50', '27.38']
```

4 路残差流的 L2 范数分别是约 133、166、113、27。**最大和最小之间差了 6 倍。**

💡 翻译成人话：Hyper-Connection 的 4 路残差流不是对称的。第 1 路（范数 166）承载了最多信息，第 3 路（范数 27）可能只承载细粒度的修正信号。这和 DeepSeek V4 论文里 Sinkhorn 归一化的设计意图一致——4 路不是均匀分工，是有主次的。

输出经过验证，和 vLLM 推理服务给相同 prompt 的生成结果完全一致。**纯 PyTorch fallback 虽然慢，但算得对。**

### 激活值保存

捕获的激活值保存为 `.pt` 文件：

```python
torch.save({
    "prompt": prompt,
    "input_ids": input_ids.cpu(),
    "activations": captured,        # {28: tensor, 29: tensor}
    "target_layers": [28, 29],
    "hc_mult": 4,
    "dim": 4096,
}, "activations_layer28_29.pt")
```

后续可以离线做 PCA 降维、余弦相似度对比、特定 attention head 的激活模式分析——不需要再跑 forward pass。

◆ 全貌：960 行代码做了什么

━━━━━━━━━━━━━━━━━━━━

```text
DeepSeek 官方代码（不修改）          我们的替换
───────────────────────          ─────────────────────
inference/model.py               
  └── from kernel import ...   ──→  kernel_sm121.py（6 个纯 PyTorch kernel）
  └── import fast_hadamard_    ──→  fast_hadamard_transform.py（33 行递归实现）
  └── load_model(file)         ──→  weight_loader.py（流式 HF shard 加载）

test_dual_node.py 在 import model.py 之前做两件事：
  sys.modules['kernel'] = kernel_sm121     # 重定向 kernel 导入
  sys.path.insert(0, '/workspace')         # fast_hadamard_transform 从这里找
```

**model.py 原封不动。** 所有适配工作在外围完成。

这意味着：DeepSeek 更新 V4 的 model.py 时（比如修 bug 或优化），我们直接替换就行，不用合并代码。替换层是稳定的（函数签名不变），被替换层可以自由演进。

◆ 这个平台有什么用

━━━━━━━━━━━━━━━━━━━━

这个平台解决的核心问题不是"在 DGX Spark 上跑模型"——那个 171 期已经用 vLLM 做到了。核心问题是：**如何从一个 280B 的大模型里提取中间层激活值。**

这件事即使你有 B200 也做不到——vLLM 不暴露中间层，DeepSeek 官方的 `generate.py` 只输出最终 token。要拿到 hidden state，你必须绕开推理服务框架，直接在 PyTorch 层面跑 forward pass、插 hook。

我们做的事情是：**在不修改 model.py 一行代码的前提下，搭了一个能在任意层提取激活值的实验框架。** DGX Spark 的 sm_121 适配只是额外多踩的一层坑——如果你在 H100/B200 上跑，跳过 `kernel_sm121.py`，直接用官方 `kernel.py`，`weight_loader.py` 和 hook 框架同样适用。

开源的价值不在于有多少人用 DGX Spark。**在于证明了一件事：280B 模型的中间层是可以被打开看的。** 不需要 Anthropic 的训练集群，不需要 Google 的内部工具——两台六万块的桌面迷你主机就够了。

────────────────────

### 下一步：不同 prompt 对激活值的影响

现在平台跑通了，接下来要做的是：**给模型不同的 prompt，观察中间层激活值的变化。**

比如：同一个问题"你是谁？"，前面加不同的系统提示词，对比 43 层中每一层的激活值差异——逐层余弦相似度变化曲线、PCA 降维可视化、特定 attention head 的激活模式。

Anthropic 的情绪探针实验（给模型不同的 prompt，用线性探针检测内部情绪向量方向）用的就是类似方法——只不过他们有自己的模型权重和训练集群，我们用的是开源模型和两台桌面迷你主机。
