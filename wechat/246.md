◆ 开篇：attention 线走完了，换 FFN 线

━━━━━━━━━━━━━━━━━━━━

244 期《mHC 出口的 Sinkhorn——4 份副本怎么调和》（ https://mp.weixin.qq.com/s/IpmdCFVbfZwVwmQno815LA ）把 attention 这条线收了尾——从量化、矩阵乘、Indexer、sparse_attn 一路走到 mHC 出口，一个 token 在 attention 段的旅程算是完整了。今天切到 FFN 这条线，先讲 MoE 的入口：**路由**。

先重申一句本系列的老前提：**咱们全程盯着的是 V4 Flash 的推理过程**（43 层、hidden_size=4096），跟 169 期讲的 V4-Pro（61 层、7168 维）不是同一套数字，别混。本期的机制两代一样，但所有具体数字走 Flash。

一个 token 怎么决定走哪 6 个专家？V4 有两套完全不同的规则——**前 3 层查表，后 40 层打分**。我自己看官方 `inference/model.py` 的时候，每次看到路由那一段都会愣两秒：**前 3 层是真的纯查表，连打分都不做**。这一期把这件事说清楚，顺便补两个之前一直没看仔细的东西：sqrtsoftplus 这个奇怪打分函数的来由，和一个藏得很深的 bias 设计。

━━━━━━━━━━━━━━━━━━━━

◆ 先把 MoE 是什么说清楚

━━━━━━━━━━━━━━━━━━━━

第 33 期《DeepSeek MoE：671B 的噱头与 37B 的真相》（ https://mp.weixin.qq.com/s/SGAt3w3d1C3icAB3JbgDYw ）讲过 MoE 的演化路径，这里只快速回顾基本结构。

MoE（Mixture of Experts，混合专家）做的事很朴素：

**一个 token 不走一个大 FFN，而是从一池子专家里挑几个小 FFN 来处理。**

V4 Flash 的具体数字（全部来自官方 config.json）：

| 项目 | V4 Flash | 备注 |
|------|----------|------|
| 路由专家数 | 256 | 池子里待选的专家 |
| 每 token 激活专家 | 6 | top-6 |
| 共享专家 | 1 | 所有 token 都走 |
| 专家中间维度 | 2048 | SwiGLU 的内宽 |
| 路由打分函数 | sqrtsoftplus | sqrt(softplus(x)) |
| 前 3 层路由 | 查表（hash routing） | token ID 决定 |
| 后 40 层路由 | 打分（score-based） | 打分选 top-6 |
| 路由权重整体放大 | 1.5 | routed_scaling_factor |
| top-k 选择方式 | noaux_tc | 带 bias 的免辅助损失均衡 |

V4 Flash 43 层全是 MoE 层，没有 dense FFN。每层 256 个路由专家加 1 个共享专家，每个 token 实际只过 7 个专家（6 + 1）——总参数巨大，激活参数很小，这就是 MoE "总量取胜、激活省钱"的逻辑。

而"从 256 里挑 6 个"这个动作本身，就是今天的全部主题：**先选再算，这个"选"具体怎么选**。

━━━━━━━━━━━━━━━━━━━━

◆ V4 路由的两套规则

━━━━━━━━━━━━━━━━━━━━

打开官方 `inference/model.py`，MoE 路由的第一步先判断当前层号（下面是按官方代码改写的伪代码）：

```python
if layer_id < num_hash_layers:      # 前 3 层
    expert_ids = tid2eid[token_id]  # 纯查表，连 hidden state 都不看
else:                                # 后 40 层
    scores = x @ W_gate                       # [4096] @ [4096, 256] -> [256]
    scores = torch.sqrt(F.softplus(scores))   # sqrtsoftplus 打分
    topk_idx = (scores + bias).topk(6).indices  # bias 只参与选人
    w = scores.gather(topk_idx)                 # 权重用原始分数
```

前 3 层根本不打分，token ID 直接决定专家。后 40 层才是正经的打分路由。

![V4 Flash MoE 路由的两套规则](assets/246/routing-two-strategies.svg)

下面分别讲。

━━━━━━━━━━━━━━━━━━━━

◆ 前 3 层：查表路由

━━━━━━━━━━━━━━━━━━━━

```python
expert_ids = tid2eid[token_id]   # 一次索引，完事
```

就这。**没有矩阵乘法，没有打分函数，没有 top-k**——纯查表。

这个 `tid2eid` 是一张 `[词表大小 × 6]` 的 int32 参数表，**跟模型权重一起发布**：每个 token ID 对应固定的 6 个专家编号。**同一个 token 不管出现在哪个位置、什么上下文里，前 3 层激活的专家都一样**。这张表是怎么生成的（训练前随机散列？还是别的规则？）官方没说——我们能确认的只是：推理时它就是一张死表，查一下就完。

────────────────────

💡 打个比方

查表路由 = **按学号分宿舍**。学号 0001 永远住 101，学号 0002 永远住 102，不看你长什么样、不看你性格——就按学号分。

后面的打分路由才是 **按特长分小组**。每个 token 拿出自己的"特长向量"，从 256 个小组里挑 6 个最匹配的。

────────────────────

【为什么前 3 层不打分？】

我第一遍看到这个设计的时候很困惑——既然有 router，干嘛不一直用？省那点算力有什么意义？

后来读 V4 报告才反应过来——**前几层的 hidden state 还没什么语义信息**。token 刚从 embedding 表里查出来，加上前两层注意力的微小变化，整个 hidden state 还停留在"字面"层面，没分化出"实体"、"动作"、"长程指代"这些可路由的语义。

这时候让一个 router 去打分，打出来的分数噪声非常大。**模型选了，但选得没意义**——既不是按语义，也不是按上下文，只是按一堆没有分化的特征做随机分配。

那不如直接放弃打分，**用固定查表强制做负载均衡**。好处有三条：

- **强制均衡**：表是定死的，256 个专家的使用分布不随输入漂移，不会出现"所有 token 涌向同一个专家"的崩溃
- **省算力**：不用算 `x @ W_gate`，不用过 sqrtsoftplus，不用 topk
- **训练稳定**：前几层的梯度信号最敏感，少一个可学习的 router 就少一个不稳定来源

代价是：前 3 层完全不利用语义来路由。但既然 hidden state 还没语义可用，这个"代价"是免费的。

**这是典型的"机制强度匹配问题复杂度"——简单层用简单规则。** 169 期《一个 token 的旅程》（ https://mp.weixin.qq.com/s/mnbaXhQmQjDAyGD1dDdq5Q ）讲全流程的时候我提过这个观察，V4 的设计哲学很一致：CSA/HCA 在短输入时退化为滑动窗口，前 3 层路由退化为查表——**该简单的地方就让它简单**。

━━━━━━━━━━━━━━━━━━━━

◆ 后 40 层：打分路由

━━━━━━━━━━━━━━━━━━━━

第 4 层开始，hidden state 已经有足够的语义分化，可以正经打分了。流程分六步：

```python
# 第一步：投影到 256 维
scores = x @ W_gate            # [4096] @ [4096, 256] -> [256]

# 第二步：sqrtsoftplus 打分
scores = torch.sqrt(F.softplus(scores))   # softplus(x) = log(1+exp(x))

# 第三步：加 bias，只用来选 top-6
topk_idx = (scores + bias).topk(6).indices

# 第四步：权重回到原始分数取
topk_score = scores.gather(topk_idx)      # 注意：不带 bias 的分数

# 第五步：top-6 内部归一化
w = topk_score / topk_score.sum()         # 加起来 = 1

# 第六步：整体放大 routed_scaling_factor = 1.5
w = w * 1.5
```

W_gate 是一个 `[4096, 256]` 的小矩阵，把 hidden state 投影到 256 维分数。第三、四步那个 bias 的细节先按下，最后单独讲——它是整段代码里最容易看漏、也最值得看的设计。

────────────────────

【sqrtsoftplus 到底是什么】

V4 的打分函数定义：

`score = sqrt(softplus(x))`，其中 `softplus(x) = log(1 + exp(x))`

这个组合不是凭空来的，是替换 softmax 的精心设计。对比一下几种常见选择：

![三种打分函数对比](assets/246/sqrtsoftplus-curve.svg)

| 函数 | 形状特点 | 用在 router 里的问题 |
|------|----------|--------------------|
| softmax | 大分数指数放大，赢家通吃 | 容易塌缩到少数专家 |
| ReLU(x) | 负数直接归零 | 专家"门被焊死"，梯度回不去 |
| softplus(x) | 负数也有小正值，光滑版 ReLU | 高分一路涨到正无穷，分数差距还是太大 |
| **sqrt(softplus(x))** | 高分被 sqrt 压平，低分仍有小正值 | V4 的选择 |

sqrt 这一层在干嘛？**把高分压下来**。softplus 在 x=3 时输出 3.05，sqrt 之后变成 1.75；在 x=0 时输出 0.69，sqrt 之后变成 0.83。**高分被压扁，低分基本不变**——整个分数曲线的"陡峭度"变缓了。

softplus 这一层在干嘛？**让负数也有小正值**。softplus(-2) = 0.13，不是 0。这就保证了得分低的专家不会被完全切断，梯度回得去。

合起来，sqrt(softplus(x)) 的几何性质：

- **比 softmax 平缓**——分数差距被压扁，不会塌缩到一个专家
- **比 ReLU 多一个尾巴**——负数也有小正值，梯度不死
- **不归一化成概率分布**——单纯是个"重要度分数"，归一化交给 top-k 选完之后

最后这条值得多看一眼：如果直接用 softmax，top-6 选完再归一化等于做两遍归一化，会把分数信息抹平；用 sqrt(softplus) 保留原始的"分数比例"，归一化只发生一次。**router 不需要输出概率分布，router 需要输出重要度分数**——sqrtsoftplus 是为"top-k 之后再归一化"这个特定流程设计的。

────────────────────

【bias：只管选人，不管发钱】

回到第三、四步那个 bias。config 里 `topk_method` 写的是 `noaux_tc`，这是 DeepSeek 从 V3 传下来的**免辅助损失负载均衡**（auxiliary-loss-free load balancing）方案，代码里就两行，但分工非常讲究：

- **选专家时**：分数**加上 bias** 再取 top-6——bias 高的专家更容易被选中；
- **算权重时**：回到**不带 bias 的原始分数**取值、归一化——bias 完全不影响选上之后分多少权重。

这个 bias 是训练时动态调节的：**哪个专家接的活儿太多，它的 bias 就被调低；哪个专家总没人光顾，bias 就被调高**——像给冷门专家发"曝光补贴"。传统 MoE 做负载均衡靠辅助损失（aux loss），把"专家用量要均匀"写进损失函数里和主任务抢梯度，代价是干扰主任务的学习；bias 方案把均衡从损失函数里搬出来，变成一个不参与梯度主流的调节旋钮——**均衡照做，主任务的梯度一点不脏**。

💡 打个比方

招生办给偏远地区考生**加分**——加分决定"录不录取"（选人）；但入学之后发奖学金，看的是**原始成绩单**（发钱）。加分只开门，不改成绩。

推理时这个 bias 已经训好、固定不动了，但它的痕迹还留在每一次 top-6 的选择里。

────────────────────

【第五步、第六步：归一化加放大】

选完 top-6 之后，6 个分数加起来归一化到 1，然后整体乘 `routed_scaling_factor = 1.5`。

为什么要放大？**这是配共享专家用的**。后面合并的时候是 `output = shared + Σ wᵢ · expertᵢ`——共享专家固定权重 1，6 个路由专家加起来权重 1.5，**FFN 段的总信号强度约等于 2.5 个专家的量**。如果路由权重不放大，每个 token 实际只享受 1 个 shared 加总权重 1 的 routed，路由专家的声音就太小了。`routed_scaling_factor` 就是拧这个配比的旋钮——V4 Flash 拧在 1.5。

━━━━━━━━━━━━━━━━━━━━

◆ 共享专家：所有 token 都走的"公共专家"

━━━━━━━━━━━━━━━━━━━━

除了 top-6 的路由专家，每层还有 **1 个共享专家**：

```python
shared_output = SharedExpert(x)   # 每个 token 都过，不参与路由
```

结构和路由专家完全一样（SwiGLU FFN，4096→2048→4096），区别是**所有 token 都过它**，不参与打分。

共享专家在干嘛？**承担通用知识**。

想象一下：如果让 256 个专家把"所有知识"都瓜分掉，会出现一个尴尬现象——很多基础能力（比如"语法正确"、"语义连贯"、"一般推理"）每个专家都得学一遍，因为不知道自己什么时候会被选中处理一个普通句子。这等于每个专家用 90% 的容量学通用知识，10% 的容量学专项化能力——**专家根本"专"不起来**。

把通用知识从 256 个专家里抽出来，单独放进一个"公共专家"，让所有 token 都过它。**剩下的 256 个路由专家就能放心做专项化分化**——一个专精"代码语法"、一个专精"长程指代"、一个专精"数学符号"……每个专家把自己的小领域吃透。

────────────────────

💡 打个比方

共享专家 = **公共课老师**，所有学生都得上。教语文、教数学这种基础。

256 个路由专家 = **选修课老师**，每个学生根据兴趣选 6 节。教 GLSL 写法、教中古汉语、教蛋白质折叠——每个老师只懂一小块，但懂得很深。

如果没有公共课老师，每个选修课老师都得花一半精力先教学生识字——大家全都半瓶水。有了公共课，选修课才能真"专"。

────────────────────

━━━━━━━━━━━━━━━━━━━━

◆ 路由完之后：加权合并 + 接 mHC

━━━━━━━━━━━━━━━━━━━━

路由出来了 top-6 索引和权重，加上 1 个共享专家，最后合并成 MoE 输出：

```python
moe_out = shared_output                              # 共享专家，权重 1
for i in range(6):
    moe_out += w[i] * routed_expert[topk_idx[i]](x)  # 路由专家，权重 wᵢ
```

输出维度 `[4096]`，和输入一样。

注意一个细节：**MoE 这一层的输出还不能直接当残差**——它要先过 hc_post 走 mHC 流形超连接（就是 244 期讲的 `hc_split_sinkhorn`），4 份残差路径动态混合之后才进入下一层。

也就是说一层完整的 FFN 段是这样：

```
hc_pre (4->1) -> RMSNorm -> MoE 路由 + 专家计算 + 合并 -> hc_post (1->4)
                                ↑
                            就是今天讲的这一段
```

mHC 把"本层输出"和"4 条历史残差路径"按 Sinkhorn 约束动态混合，所以**路由出来的 MoE 输出不是简单地加回 x**——它是被 4 个动态权重重新分配到 4 条不同的残差路径上的。244 期把 mHC 放在 FFN 之前先讲，就是为了今天这里能一句话接上：**FFN 的输出怎么"接回去"，比 FFN 本身怎么算还重要。**

◆ 学习者复盘：这一站的四个记忆抓手

━━━━━━━━━━━━━━━━━━━━

我以前看完就忘——读到"前 3 层 hash"会以为是某种工程妥协，下次看又得重新想一遍。这次给自己整理四条记忆抓手：

**抓手一：前 3 层不是"路由的简化版"，是"路由的禁用版"。**

前 3 层根本没有任何"基于内容的选择"——token ID 决定一切，hidden state 完全不参与。这不是"打分但是用了简单的打分方式"，是"完全跳过打分"。把这点想清楚，查表路由的存在就不奇怪了：**它根本不是 router，它是 token ID 直接控制专家分布的硬约束**。

**抓手二：什么时候该简单，由 hidden state 的语义成熟度决定。**

前几层 hidden state 还在"字面层"，没有可路由的语义信号；后面层分化出实体/动作/指代等可路由特征，才值得让 router 出场。**机制强度匹配问题复杂度**——和"CSA/HCA 在短输入退化为滑动窗口"是同一个设计哲学。

**抓手三：sqrtsoftplus 不是替换 softmax，是替换"概率分布"这个抽象。**

router 不需要输出概率分布，router 需要输出"重要度分数"。sqrt(softplus(x)) 的设计目标就是给一个**平缓的、不归一化的、非负的**重要度分数，让 top-k 选完之后再做一次归一化。两遍归一化的信息损耗，避免了。

**抓手四：bias 只管选人，不管发钱。**

负载均衡的全部干预被限制在"谁进 top-6"这一步，选上之后的权重回到原始分数算。均衡和权重解耦——**"公平"和"按劳分配"被分成两个互不干扰的旋钮**，V3 传下来的 noaux_tc 方案最漂亮的一笔。

━━━━━━━━━━━━━━━━━━━━

◆ 收尾：又是"既要分工，又要均衡"

━━━━━━━━━━━━━━━━━━━━

回头看 244 期收尾说过的那句话：mHC 的 Sinkhorn 是"哪都不撒手，但哪都不拧死"。今天的 MoE 路由是同一个两难的另一种走法——**专家要分工（打分选人，各干各的专长），但负载要均衡（不能让明星专家累死、冷门专家饿死）**。

mHC 用的是显式几何约束（Sinkhorn 把矩阵推进双随机家族）；MoE 用的是一套工程组合拳：前 3 层查表强制均衡、sqrtsoftplus 压平分数差距、bias 补贴冷门专家。**同样的两难，一边用数学定理，一边用三个土办法**——但味道是一致的：约束要有，但不拧死。

下一站（第 7 站，也是这个系列的最后一站）讲 top-6 选出来的专家本体：**SwiGLU**。每个专家里面是什么计算？

```python
gate = x @ W1    # [4096] @ [4096, 2048] -> [2048]
up   = x @ W3    # [4096] @ [4096, 2048] -> [2048]
hidden = SiLU(gate) * up
down = hidden @ W2   # [2048] @ [2048, 4096] -> [4096]
```

`SiLU(gate) * up` 这一步是 SwiGLU 的核心，它和最早的 FFN（ReLU(x @ W1) @ W2）有什么本质区别？gate 这个分支是怎么从 GLU 演化来的？config.json 里那个 `swiglu_limit = 10.0` 又是干嘛的？下一站把这 7 个零件全打通。
