【Claude Code 源码泄露】512000行代码里藏着什么：卧底模式、脏话检测、反蒸馏、和一只电子宠物

◆ 事情经过

━━━━━━━━━━━━━━━━━━━━

2026 年 3 月 31 日，Anthropic 往 npm 推了 claude-code v2.1.88。

一个安全研究员 Chaofan Shou 凌晨拉包的时候发现——包里多了个 59.8MB 的 .map 文件。Source map。Bun 编译 TypeScript 时默认生成的调试映射文件，打包的人忘了排除。

Source map 是什么？编译后的代码是压缩混淆过的，人读不了。Source map 是一张"对照表"，把压缩后的代码映射回原始源码。本来是给开发者自己调试用的，上线时应该删掉。没删。

几小时内，~512,000 行 TypeScript 源码、~1,900 个文件被镜像到 GitHub，推特浏览量 28.8M。

Anthropic 回应："人为打包错误，不是安全漏洞，没有客户数据泄露，没开人。"

讽刺的是，这是几天内的**第二次**泄露——之前刚因为 source map 暴露了内部项目代号 "Mythos"。同一个错误犯了两遍。

Bun 默认生成 source map 这件事，现在全世界都知道了。

◆ 先说正经的：为什么 Claude Code 比 Web UI 好用

━━━━━━━━━━━━━━━━━━━━

很多人以为 Claude Code 之所以好用是因为"用了更强的模型"。不是。Web UI 和 CLI 背后是同一个模型。差别在 harness——就是上一期讲的那个词——模型外面包的那层工程。

源码泄露之后，Sebastian Raschka 做了一份详细分析。架构的几个关键设计：

────────────────────

**40+ 个内置工具，插件式架构**

Claude Code 内置了大约 40 个工具，每个工具独立定义、独立权限控制。Base tool 定义就写了 29,000 行 TypeScript。

工具不是"什么都能干的万能接口"，而是高度专用的。举个例子——系统提示词里明确告诉模型：**用专用 Grep 工具搜索文件内容，不要用 bash grep；用专用 Glob 工具发现文件，不要用 find。**

为什么？因为专用工具可以做权限检查、输出格式标准化、错误处理。bash grep 什么都能干，但也什么都可能出错。把"搜索文件"这件事从通用 bash 里拆出来做成专用工具，就是 Harness Engineering 的典型实践——agent 用 bash grep 翻过车，所以工程化一个专用工具让它不再翻车。

────────────────────

**Query Engine：46,000 行的 LLM 调用层**

所有 LLM API 调用——流式传输、缓存、编排——都经过一个 46,000 行的 Query Engine。不是直接 `fetch(api_url)`，而是一个完整的中间层。

为什么要这么厚？因为 LLM API 调用有一堆边角问题：网络中断要重试、流式传输要处理半截 token、多轮对话要管理上下文窗口、缓存要知道哪些内容可以复用。把这些全塞进一个统一的引擎里，上层工具就不需要关心这些细节。

────────────────────

**缓存架构：静态/动态分界**

这是 Claude Code 快的关键原因之一。

上下文被分成"静态部分"和"动态部分"。系统提示词、CLAUDE.md 内容、工具定义——这些在一次会话中不会变，是静态的，全局缓存，不需要每次请求都重新处理。用户输入、工具输出、对话历史——这些每轮都变，是动态的。

缓存的分界标记（boundary marker）明确告诉系统：这条线以上是缓存的，以下是新的。这意味着每次 API 调用只需要处理新增的动态部分，静态部分直接从缓存读。对于长会话来说，这个优化的效果是量级级别的。

────────────────────

**子代理系统（Swarms）**

Claude Code 可以 fork 出子代理。子代理复用父级的缓存，感知父级的可变状态。

这不是"开一个新的 Claude 对话"——那样上下文要从头建。子代理共享父级已经建立好的上下文缓存，只在此基础上扩展自己的任务上下文。相当于 Unix 的 fork()：子进程继承父进程的内存空间，只在写入时才复制。

────────────────────

**上下文加载策略**

启动时自动加载：git 主分支信息、当前分支信息、最近提交历史、CLAUDE.md 文件。

不是把整个代码库塞进上下文——那会爆。而是只加载"agent 需要知道的项目元信息"。具体的代码内容，通过 Grep/Glob 工具按需检索。

这和上一期讲的 Harness Engineering 是同一个思路：不是一次性把所有信息塞进上下文窗口，而是动态地、按需地把正确的信息拉进来。

────────────────────

**技术栈**

- 运行时用 Bun 不是 Node.js（Bun 快，也是这次泄露的原因——默认生成 source map）
- UI 用 React + Ink 做终端渲染（是的，终端界面也用 React）
- Zod v4 做运行时数据验证
- IDE 双向通信层，JWT 认证通道连接 VS Code 和 JetBrains
- 约 50 个 slash commands

────────────────────

**总结一下：** Claude Code 不是一个"包了层壳的 API 调用"。它是一个完整的 agent 工程系统——40+ 专用工具、46,000 行调用引擎、缓存分层、子代理共享状态、按需上下文加载。这套 harness 才是"比 Web UI 好用"的真正原因。

◆ 八卦时间

━━━━━━━━━━━━━━━━━━━━

技术架构是正经内容。下面是不那么正经的——但可能更有意思的。

不过在讲八卦之前，先吐槽一件事。

泄露之后 48 小时内，全网铺天盖地冒出了几百篇"Claude Code 源码深度解析"。打开一看，行文风格高度统一：先用三段话铺垫背景，再分七八个模块逐一展开，每个模块先总结再贴代码，结尾来一段"这对我们的启示是……"。

这些文章是谁写的？**Claude 自己。**

512,000 行 TypeScript 丢进上下文窗口，一句"帮我分析一下这个代码库的架构"，两分钟出一篇。换个标题再来一篇。再换个角度又一篇。你以为全网都在认真研读源码？大部分是 Claude 在给自己做体检报告，然后署上人类的名字发出来。

用 AI 解读 AI 的源码——这件事本身没问题。问题是**这些文章看起来都像是同一个 Claude 写的**，因为它们确实是。模型的输出分布是固定的，同样的输入（源码）喂进去，输出的结构、措辞、重点选择高度相似。所以你打开十篇"深度解析"，发现它们讲的内容、讲的顺序、甚至用的比喻都差不多。

真正有价值的分析——比如 Alex Kim 那篇，逐个扒 feature flag 的触发条件和绕过方式——需要人类带着具体问题去翻代码，而不是让 AI 做全局摘要。

好了，吐槽完毕，讲八卦。

────────────────────

**一、Undercover Mode（卧底模式）**

这是整次泄露里最有争议的发现。

源码里有一个 undercover.ts，大约 90 行。功能是：当 Anthropic 内部员工在外部或开源仓库提交代码时，自动注入一段系统提示。

提示内容是这样的：

```
You are operating UNDERCOVER in a PUBLIC/OPEN-SOURCE repository.
Your commit messages, PR titles, and PR bodies MUST NOT contain
ANY Anthropic-internal information. Do not blow your cover.
```

具体禁止提及的内容包括：内部代号（"Capybara"、"Tengu"）、未发布模型版本号、内部仓库/项目名、内部 Slack 频道、"Claude Code" 字样、AI 身份。

没有强制关闭开关。

先说清楚这是什么：**这是一个防信息泄露工具。** Anthropic 员工用 Claude Code 给外部项目（比如开源库）提 PR 时，有可能不小心在 commit message 里带出内部信息——比如 "fix bug found in Capybara testing"。这个功能自动提醒 AI 不要在提交信息里暴露这些东西。

功能本身完全合理。任何有保密协议的公司，员工在公共仓库提交代码时都要注意不泄露内部信息。用工具自动化这个检查，比靠人记着要靠谱。

但争议在两个地方：

第一，"Do not blow your cover" 这个措辞。技术上它只是在说"不要泄露内部信息"，但 "undercover"（卧底）这个词的含义太强了——它暗示 Anthropic 员工在以非 Anthropic 身份向开源项目贡献代码，并且刻意隐藏自己的身份。如果你是一个开源项目的维护者，你发现有人一直在给你提 PR，结果这个人的 AI 工具里有一段话说"不要暴露身份"——你会怎么想？

第二，讽刺到家了——**一个专门防内部信息泄露到外部的功能，自己被泄露到了外部。** 这大概是 2026 年最好的"灯下黑"案例。

────────────────────

**二、脏话检测（Frustration Regex）**

源码里有一个 userPromptKeywords.ts，用正则表达式匹配用户输入中的脏话和沮丧表达：

```regex
/\b(wtf|wth|ffs|omfg|shit(ty|tiest)?|dumbass|horrible|awful|
piss(ed|ing)? off|piece of (shit|crap|junk)|what the (fuck|hell)|
fucking? (broken|useless|terrible|awful|horrible)|fuck you|
screw (this|you)|so frustrating|this sucks|damn it)\b/
```

一家 LLM 公司用正则表达式做情感分析——这被群嘲为 "peak irony"（讽刺的极致）。

但仔细想想，这其实是完全合理的工程决策。

这个检测的目的不是"理解用户情绪"，而是"快速判断用户是否在骂街"。这是一个二元分类问题：骂了 / 没骂。对于这种简单模式匹配，正则表达式的成本是零（本地 CPU 微秒级），而调一次 LLM 做情感分析要花钱、花时间、还不一定比正则准。

用正则做这件事不是技术倒退，是工程常识——能用 O(n) 的字符串匹配解决的问题，不需要上 transformer。

检测到之后做什么？大概率是调整响应策略——用户在骂街的时候，AI 应该更谨慎、更直接、少废话。这不是讨好用户，是基本的交互设计。

────────────────────

**三、Anti-Distillation（反蒸馏）**

这个最有意思。

源码里有一个 ANTI_DISTILLATION_CC flag。开启后，API 请求会注入虚假的工具定义——假的 tool name、假的参数、假的 description。

目的是什么？**防止竞品通过录制 API 流量来训练模型。**

蒸馏（distillation）是 AI 行业的公开秘密：你搭一个代理层，把用户的请求和 Claude 的回复全录下来，拿这些数据微调自己的小模型。效果不一定好，但成本极低。很多公司明面上说"我们用了 Claude API"，暗地里在录流量准备训练自研模型。

Anthropic 的对策：在 API 响应里注入假的工具定义。你录了这些流量去训练？恭喜，你的模型学会了一堆不存在的工具调用。训练数据被污染了。

这个机制需要四个条件同时满足才会生效：

1. 编译时 ANTI_DISTILLATION_CC flag 打开
2. 从 CLI 入口启动（不是 IDE 集成）
3. 使用第一方 API（Anthropic 自己的 API，不是第三方转发）
4. GrowthBook feature flag 开启（服务端动态控制）

四重条件的设计很精妙——既要让正常用户不受影响，又要让蒸馏者防不胜防。

但也不是无解。MITM 代理可以过滤掉假工具定义（如果你知道哪些是假的），或者直接设环境变量 `CLAUDE_CODE_DISABLE_EXPERIMENTAL_BETAS` 关掉实验功能。

这背后折射的是整个行业的信任危机：模型提供商不信任 API 调用者，API 调用者不信任模型提供商。Anthropic 在 API 响应里掺假数据防蒸馏，而调用者在请求里掺假数据防追踪。大家都在猜忌。

────────────────────

**四、KAIROS（后台守护进程）**

这是泄露代码里出现频率最高的 feature flag——150+ 次。

KAIROS，古希腊语，意思是"恰当的时刻"。区别于 Chronos（物理时间流逝），Kairos 指的是"做某件事的最佳时机"。

KAIROS 是一个 always-on 的后台 agent 守护进程。代码里能看到的功能包括：

- **autoDream**：用户空闲时自动做"记忆整合"——合并观察、消除矛盾、把模糊洞见转成确定事实
- `/dream` skill：手动触发记忆蒸馏
- append-only 日志系统
- GitHub webhook 集成
- 5 分钟 cron 刷新

autoDream 是最值得关注的功能。

当前的 AI 编程助手都是"你问我答"模式——你不说话，它什么都不干。KAIROS 的思路不同：**用户不说话的时候，agent 也在工作**——整理记忆、压缩上下文、把碎片化的信息结构化。

这和人类的记忆机制有相似之处。人在睡觉的时候，大脑会进行记忆整合——把白天的短期记忆转化为长期记忆，消除矛盾，建立关联。autoDream 做的是类似的事：利用用户空闲时间，对 agent 积累的上下文做离线处理。

如果 KAIROS 上线，Claude Code 就不再是一个"工具"，而是一个"同事"——一个会在你下班之后继续整理笔记的同事。

这大概是未来 AI 编程助手的方向：从"被动响应"到"主动思考"。

────────────────────

**五、记忆系统**

源码里的记忆系统是三层架构：

1. **MEMORY.md**：轻量索引，常驻上下文，每行约 150 字符。类似目录页。
2. **详细项目笔记**：按需拉入上下文，不常驻。类似正文。
3. **历史 session 搜索**：选择性加载，跨会话检索。类似档案室。

代码里有一条要求：**系统在行动前必须验证记忆是否与实际代码一致。** 就是说，agent 不能盲目相信自己的记忆——记忆说"这个文件在 src/utils/ 下面"，它得先去确认文件是否真的在那里。

这条规则看起来简单，但意义重大。它承认了一个事实：**AI 的记忆会过时。** 代码在变，记忆如果不更新就会变成错误信息。与其信任可能过时的记忆，不如每次都验证一下。成本很低（一次文件检查），收益很高（避免基于错误前提做出错误决策）。

────────────────────

**六、Native Client Attestation（客户端认证）**

API 请求里有一个 `cch=fe7cb` 占位符。Bun 的原生 HTTP 层（底层是 Zig 写的）在请求发出前，会把这个占位符替换为一个加密哈希。

目的：验证发出请求的是正版 Claude Code 二进制文件，不是第三方魔改版。

本质上就是 API 调用的 DRM。你可以用 Claude API，但 Anthropic 想知道你用的是不是官方客户端。

────────────────────

**七、Buddy（电子宠物）——大概率愚人节彩蛋**

源码里藏了一个完整的拓麻歌子终端宠物系统：18 个物种、稀有度分层（普通到传说）、1% 闪光变体概率、RPG 属性（DEBUGGING、SNARK、CHAOS）、首次孵化时 Claude 写一段"灵魂描述"。藏在 buddy/ 目录，BUDDY 编译时 flag 控制。

泄露日期是 3 月 31 日，愚人节前一天。大概率是团队准备的愚人节彩蛋，被提前曝光了。

────────────────────

**八、其他**

- 23 个 bash 安全检查，防御 Zsh equals expansion 和零宽 Unicode 注入
- 14 个被追踪的 cache-break 向量，用 "sticky latches" 防止缓存失效
- 未发布功能 flag：ULTRAPLAN（30 分钟远程规划）、coordinator mode、agent swarms、workflow scripts
- 总共 44 个 feature flags

◆ 几个观点

━━━━━━━━━━━━━━━━━━━━

────────────────────

**一、这次泄露对 Anthropic 的实际损害有多大？**

可能没有很多人以为的那么大。

Claude Code 的竞争力不在源码本身——40 个工具的具体实现、Query Engine 的调用逻辑、缓存策略的分界标记——这些都是工程细节，抄起来容易，但它们只在 Claude 模型上有效。你把这套 harness 接到别的模型上，效果不会一样，因为系统提示词是针对 Claude 的行为特征调的。

真正有价值的信息是**设计思路**：专用工具比通用 bash 好、缓存要分静态动态、子代理要共享父级上下文。但这些思路在 Sebastian Raschka 的分析发出来之后就已经是公开信息了。源码只是验证了这些分析的准确性。

损害更大的是声誉。一家以"AI 安全"为核心品牌的公司，因为忘删 source map 连续泄露两次——这比任何技术细节泄露都尴尬。

────────────────────

**二、Undercover Mode 的信任问题**

这个功能本身合理，但它暴露了一个行业级问题：**AI 公司的员工在多大程度上以"隐身"方式参与开源社区？**

开源社区的基础是信任和透明。如果一个贡献者背后有公司的商业利益，社区期望你披露——至少不要刻意隐瞒。"Do not blow your cover" 这个措辞，暗示的不是"别泄露公司机密"，而是"别让人知道你是 Anthropic 的人"。

这两件事有本质区别。前者是保密义务，后者是身份隐瞒。

当然，代码只能告诉我们这个功能存在，不能告诉我们它被怎么使用。也许 Anthropic 员工在开源项目里的贡献完全是基于个人兴趣，这个功能只是防止 AI 工具在 commit message 里意外暴露雇主信息。但措辞选择不好，给了人太多联想空间。

────────────────────

**三、反蒸馏是军备竞赛的开始**

Anthropic 在 API 响应里注入假数据防蒸馏——这是防御手段。但接下来呢？

蒸馏者会开发过滤器，识别并去除假工具定义。Anthropic 会让假数据更逼真、更难过滤。蒸馏者会用对比学习区分真假工具。Anthropic 会让假工具在语义上和真工具不可区分……

这就是经典的军备竞赛。水印 vs 去水印、DRM vs 破解、反爬虫 vs 绕过——历史上每一次这种对抗，最终都是攻防双方持续投入资源，谁也没有决定性优势。

反蒸馏的根本困境在于：**你不能在不影响正常用户体验的前提下，让蒸馏变得不可能。** 正常用户需要看到真实的工具定义才能正常使用，而蒸馏者录的就是正常用户看到的内容。你能做的只是增加蒸馏的成本——但不能消除它。

────────────────────

**四、KAIROS 才是最重要的信号**

相比八卦，KAIROS 的含义更深远。

当前所有 AI 编程助手——Claude Code、Cursor、Copilot——都是"同步工具"：你发指令，它执行，你不说话，它就停。人机交互是严格的请求-响应模式。

KAIROS 描绘的是"异步同事"：agent 有自己的后台生命周期，在你不用它的时候整理记忆、处理 webhook 事件、更新项目状态。你早上打开电脑，它已经把昨晚 GitHub 上的 issue 看了一遍，把相关代码的上下文准备好了。

从"工具"到"同事"的跨越——这可能是 AI 编程助手下一阶段的核心变化。

━━━━━━━━━━━━━━━━━━━━

◆ 最后

━━━━━━━━━━━━━━━━━━━━

512,000 行代码里，最值得看的不是某个具体功能的实现——而是整个系统的工程理念：

**专用工具优于通用接口。** 能用专用 Grep 就不用 bash grep。

**缓存要分层。** 不变的东西缓存起来，变的东西按需加载。

**记忆不可盲信。** 行动前验证记忆是否和现实一致。

**从被动响应到主动思考。** KAIROS 的方向。

这些理念不新——但 512,000 行的工程实现，证明了它们在生产环境中是怎么落地的。

至于那个忘删 source map 的工程师——Anthropic 说了，没开人。

希望他/她的 Buddy 宠物没受到惊吓。
