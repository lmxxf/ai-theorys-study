【Loop Engineering】别再 prompt agent 了——你的工作是设计 loop

━━━━━━━━━━━━━━━━━━━━

最近你大概率看到了这两句话——

> **You shouldn't be prompting coding agents anymore. You should be designing loops that prompt your agents.**
>
> —— Peter Steinberger（前 PSPDFKit 创始人，现 OpenAI），650 万浏览的引爆推文

> **I don't prompt Claude anymore. I have loops running that prompt Claude. My job is to write loops.**
>
> —— Boris Cherny（Anthropic Claude Code 负责人）

两个核心位置的工程师，同一周说同一件事：别再给 agent 写 prompt 了，你该做的是设计 loop，让 loop 替你给 agent 写 prompt。

新词被推上来了：**Loop Engineering**。正式命名的人是 Addy Osmani（Google Chrome 工程主管），2026 年 6 月 8 日在 Substack 发了同名文章。还是这个 Addy，2026 年 2 月刚造完 Harness Engineering，三个月之后又造了 Loop Engineering——他自己把 AI 工程切了三层：context、harness、loop。

这次确实有新东西。但不是你以为的那种"新"。

━━━━━━━━━━━━━━━━━━━━

◆ 等等——agent 不就是 loop 吗？

────────────────────

你的第一反应应该是这个。

ReAct 是 loop。AutoGPT 是 loop。BabyAGI 是 loop。Devin 是 loop。Claude Code 一启动就在转 loop。

没有不是 loop 的 agent。一个 agent 如果不 loop，它就只是个一次性函数调用。

那 Loop Engineering 说的 loop 到底是哪个 loop？

────────────────────

仔细看 Addy 的原文，他说的 loop 不是 agent 内部那个干一个任务的循环——那个循环是 agent 框架自带的，Claude Code 启动就在转。

他说的 loop 是**外面再套一圈的那个**：cron 早上九点自动叫醒 agent；agent 干完活，另一个独立 agent 验收；不合格扔回去重做；合格了写进状态文件，明天接着干；一周后 Slack 自动汇报"上周新增 23 个 PR，合并了 19 个"。

**ReAct loop 是一次任务内部的循环；Loop Engineering 的 loop 是跨任务、跨时间、跨多个 agent 的外部循环。**

💡 翻译成人话

- ReAct loop = 工人在工位上拧螺丝（agent 框架自带）
- Loop Engineering = 工厂的传送带 + 质检 + 排班表 + 早晚班交接本

────────────────────

问题接下来就清楚了：这事儿 cron + Jenkins + CI/CD 干了二十年了，套个 AI 外壳算"新东西"吗？

答案是：零件全是旧的，但这次零件第一次全部装进了 agent 产品本体。

一年前你想搭个这样的 loop，得自己写一堆 bash 脚本、自己维护、那是你一个人的玩具。现在 Claude Code 2.1.139（2026 年 5 月 12 日发布）内置了 `/loop`、`/goal`、`/batch`。Codex 内置了 Automations 面板和 sub-agent 配置。曾经野生 hacker 才会搭的东西，现在变成了 agent 产品自带的标准件。

**这是产品形态的相变，不是技术发明。但相变本身值得起个名字。**

━━━━━━━━━━━━━━━━━━━━

◆ Loop Engineering 到底要你做啥

────────────────────

Addy 把这件事拆成五个建筑块，Claude Code 和 Codex 现在都齐了。我用人话讲一遍。

**第一件：定时和按目标触发**

Claude Code 里有两个新命令——`/loop` 和 `/goal`。

`/loop` 是按时间触发：`/loop 5m /babysit` 就是每五分钟跑一次 PR 维护。

`/goal` 是按目标触发：跑到目标达成为止，带预算上限。比如：

```
claude /goal "把 /lib/auth 重构成用 JWT" --bg --tokens 500K --turns 20
```

`/goal` 设计上最讲究的一点：**双 agent 架构**。一个 agent 干活，另一个独立的、没看过过程的 agent 做验收。Addy 这里有一句金句：写代码的模型给自己批作业实在太宽容了。所以验收必须找一个不知道过程的 agent。

**第二件：用 git worktree 让多个 agent 并行不打架**

以前你让 agent 改大改动，要么排队，要么打架——两个 agent 在同一个工作目录里 commit 互相覆盖。

worktree 让你能同时跑五个 agent，每个在自己的隔离副本里。改完合不合并你说了算。Claude Code 直接支持 `--worktree`，Codex 内置。

**第三件：把项目规矩写进 SKILL.md**

新来的 agent 上手第一件事是踩坑：改了 `/config` 不知道 CI 会读老路径、用了 `any` 不知道项目禁了、跑测试前不知道要检查数据库连接。

Skills 就是把"我们不这么做，因为去年那次事故"这类老员工脑子里的血泪教训搬到 markdown 文件里。新 agent 一上手就知道哪儿不能碰。文件叫 SKILL.md，放在 `.claude/agents/` 或 `.codex/agents/` 下面。

这件其实就是 AGENTS.md / CLAUDE.md 的升级版——以前一份大文件，现在按"技能"切成多份。

**第四件：用 MCP 接外部世界**

底层协议是 MCP（Model Context Protocol），让 agent 能读 Linear 工单、查数据库、调内部 API、发 Slack。

Addy 这件有句话点得很准：一个 agent 说"修好了，代码在这"；一个 loop 自己开 PR、自己关联工单、CI 绿了自己发 Slack 通知——区别就在这。

**第五件：maker 和 checker 必须分开**

回到那句"自己批自己作业太宽容"。maker 知道自己写了啥、知道哪些地方差不多就行。checker 不知道，所以会真较劲。

所以你要在 `.claude/agents/` 或 `.codex/agents/` 下专门配一个 verifier 子 agent——独立的指令、独立的 context、不知道 maker 的思路、只看产出。

**附加：状态文件**

agent 会忘，repo 不会忘。

所有跨 session 的状态——进度到哪、踩过哪些坑、明天该干啥——写进一个 markdown 文件（比如 STATE.md）或者 Linear 看板。什么都行，**只要它活在单次对话之外**。

━━━━━━━━━━━━━━━━━━━━

◆ 一个活体案例：Karpathy 的 autoresearch

────────────────────

Addy 的文章里举的例子是泛泛的"OpenAI 内部 daily issue triage"——没有硬数据。真正能让你看清 Loop Engineering 长啥样的活体案例，是 Andrej Karpathy 2026 年 3 月 7 日开源的 autoresearch。

这个项目本身只有约 630 行 Python，但它是 Loop Engineering 五件套的完整教科书示范。

它要 agent 干的事是：自己跑 ML 实验、自己迭代优化训练脚本——人睡觉时 agent 干活。

三个文件：`prepare.py` 是数据预处理（agent 不动）；`train.py` 是模型和训练循环（agent 只改这一个）；`program.md` 是研究方向（人写）。

外层 loop 这么转：agent 读 `program.md` 和当前 `train.py`，形成假设，改一处，跑固定五分钟，看 validation 指标。改进就保留、git commit 到 feature 分支；没改进就回退。一小时大约 12 个实验，一晚约 100 个。

五件套全中：定时（你睡觉时它在转）、worktree（git 分支当隔离）、SKILL.md（`program.md` 就是写给 agent 的研究方向）、verifier（validation 数字不骗人，就是天然的 checker）、状态（git log 记录每次尝试）。没用 LangChain，没用什么 agentic 框架。**就是 git + python + 一个验证数字 + 五分钟计时器。**

成绩：5 天 GitHub 25,000 stars，两周 43,800 stars，现在 8 万 7 千多 stars。**Shopify 创始人 Tobi Lutke 直接拿去复现，validation 提升 19%**，小模型超过手调大模型。其他用户在 16 块 GPU 集群上一晚跑 910 个实验，总成本 309 美元。

Karpathy 自己的总结是：人的工作从"写训练代码"换成了"写研究方向"。

这句话翻译成 Boris 的调调就是：**我不再 prompt 模型，我写 program.md，剩下的 loop 替我跑**——一字不差。

━━━━━━━━━━━━━━━━━━━━

◆ Addy 自己警告的三个坑

────────────────────

这一段我特意留出来，因为很多复述 Loop Engineering 的文章会跳过这部分，只留下"快来设计 loop"的 hype。但 Addy 原文里这三个反模式的篇幅几乎和五件套一样长——他自己很清楚这套东西的危险性。

**坑一：你信不过 verifier，loop 就跑了个寂寞**

没人盯的 loop 是没人盯地在犯错的 loop。

回到 AutoGPT 在 2023 年的翻车——典型的失败案例是：search → save → verify → "再多查一点" → 重复 300 次、零产出、烧掉 80 美元。

Loop Engineering 号称的解法是 `/goal` 的双 agent 架构加硬预算（比如 `--tokens 500K --turns 20`）。但这个解法本身有个嵌套问题：**你信不信得过那个 verifier？**如果你不信，你得自己看；你自己看，那 loop 跑了个啥？

**坑二：理解债**

loop 越快地交付你没写过的代码，"代码实际是啥"和"你脑子里以为的是啥"之间的鸿沟就越大。除非你真的去读它写的东西，否则顺滑的 loop 只会让这鸿沟变大得更快。

人类版本的同构问题：你雇了个外包团队，验收时只看测试通过、从不看代码——一年后想自己改一行，发现自己看不懂自家代码库。

**坑三：认知投降**

loop 自己转的时候，你很容易就懒得有自己的判断了——loop 给啥你拿啥。

Addy 这里有一句很重的话：**设计 loop 这件事，带着判断去做是解药，为了逃避思考去做是助推剂——同一个动作，相反的结果。**

最后一句金句单独抄一遍：**两个人搭一模一样的 loop，得到的结果可以完全相反。**

💡 翻译成人话

loop 不是工具，是放大器。**勤奋的人用 loop 放大勤奋，懒的人用 loop 放大懒。**

━━━━━━━━━━━━━━━━━━━━

◆ 这次是 Reflexion 第四次借尸还魂吗？

────────────────────

如果你看过 140 期那篇《Harness Engineering：同一件事，第三次被命名》，你大概会问这个。

那篇说的是：**Reflexion（2023）→ Context Engineering（2025）→ Harness Engineering（2026 年 2 月）**，同一件事被三批不同人独立发现、各自命名。

Loop Engineering 不是这个剧本。**这次是真的有新东西，但新东西不在技术，在产品形态。**

────────────────────

**第一，Loop Engineering 和 Reflexion 不是同一件事。**

Reflexion 是 agent 算法内部的反思机制——失败后生成文字反馈、下次带着反馈重试。这是模型/agent 框架内部的逻辑。

Loop Engineering 是人作为系统设计者搭在 agent 外面的循环——cron、worktree、verifier、状态文件。这是产品和工程层面的事。

一个管 agent 怎么自我修正，一个管 agent 之上的调度系统怎么搭。**两个不同的层。**

**第二，Loop Engineering 是 Addy 主动迭代出来的，不是不同人独立发现的。**

Addy 2 月造 Harness Engineering，6 月造 Loop Engineering——同一个作者，同一套叙事，主动往上加一层。这不是"传承断了又被独立重发现"，是"产品形态相变之后，作者把新形态命名了"。

**第三，相变是真的。**

5 月 12 日 Claude Code 2.1.139 内置 `/goal`，是这次相变的官方时间戳。一年前你要 cron + bash + 自己写的 verifier 才能搭出来的东西，现在是产品自带功能。这种"以前野生现在标配"的产品形态变化，确实值得起一个新名字。

────────────────────

把这层 Loop 放回 Addy 自己搭的三层叙事里：

| 层 | 这一层管什么 | 类比 | 旧名字 |
|---|---|---|---|
| Context Engineering | 上下文窗口里放什么 | 食材 | prompt + RAG |
| Harness Engineering | 单个 agent 跑在啥环境里 | 厨房 | system prompt + sandbox |
| **Loop Engineering** | **agent 之上的外层调度** | **流水线 + 质检** | **cron + CI/CD** |

诚实地说，每一层对应的"旧东西"都早就存在。**Addy 真正的贡献不是发明，是命名——给"AI 工程"这件无定形的事造了一套能写在岗位 JD 上的词汇表。**Context Engineer 招会写 prompt 搞 RAG 的；Harness Engineer 招会配 sandbox、写 AGENTS.md 的；Loop Engineer 招懂 CI/CD、懂任务编排、懂如何让 agent 系统自己跑起来的。三个抽屉，三种 JD，三种立项理由。

词汇表是给组织看的，不是给技术看的。

━━━━━━━━━━━━━━━━━━━━

◆ 结尾

────────────────────

Boris 说"我的工作是写 loop"，Karpathy 说"人的工作从写训练代码换成了写研究方向"，Addy 说"loop 改变了工作内容，但没把你从工作里删掉"——三个人说的是同一件事：

**杠杆点上移了，但人没被踢出去。只是干的事变了。**

以前你 prompt agent，现在你设计那个 prompt agent 的系统。以前你写训练代码，现在你写研究方向让 agent 去试。

但 Loop Engineering 不是什么"下一代 AI 工程范式"——它就是 cron + Jenkins + CI/CD 套了一层 agent 外壳。零件全是旧的，新的是这些零件第一次全部装进了 agent 产品本体。

最后留 Addy 那句话：

**搭你的 loop。但要像一个还打算继续当工程师的人那样搭，不要像一个只剩按"开始"按钮的人那样搭。**
