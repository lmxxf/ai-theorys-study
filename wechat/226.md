◆ 开篇：三个人三种 loop，谁说的是对的？

━━━━━━━━━━━━━━━━━━━━

最近你大概率看到了这三句话——

> **You shouldn't be prompting coding agents anymore. You should be designing loops that prompt your agents.**
>
> —— Peter Steinberger（前 PSPDFKit 创始人，现 OpenAI），650 万浏览的引爆推文

> **I don't prompt Claude anymore. I have loops running that prompt Claude. My job is to write loops.**
>
> —— Boris Cherny（Anthropic Claude Code 负责人）

> **Loop engineering is replacing yourself as the person who prompts the agent.**
>
> —— Addy Osmani（Google Chrome 工程主管），2026 年 6 月 8 日同名文章开篇

Addy 这篇文章的发布日期是 **2026 年 6 月 8 日**，平台是 Substack（美国版"公众号 + 邮件订阅"平台，大 V 用来写长文章）。正是这篇文章，把 Peter 和 Boris 那两条推文里冒出来的 "loop" 正式推上方法论的位置。还是这个 Addy——2026 年 2 月他刚造完 Harness Engineering，三个月后又造了 Loop Engineering，自己把 AI 工程切成三层：context、harness、loop。

之后**一个月之内**：

- O'Reilly Radar 联名发了一篇
- The New Stack 写了"Claude Code 之父抛弃了 prompt"
- TechTalks 警告"loopmaxxing"风险
- 中文圈知乎、腾讯新闻、思否、CSDN、菜鸟教程全部跟进
- GitHub 出现了名为"loop-engineering-orange-book"的中英文小册子，号称"橙皮书"

**新词的传播速度堪比一次 IPO 路演**。

但我把这些文章每篇都读了一遍，发现一件挺尴尬的事——

**没有任何两个人，对"loop"的定义是完全一致的**。

Peter 说的 loop 和 Boris 说的 loop 不是一个东西；Addy 写文章那天又改了一遍定义；O'Reilly 把它包装成"新范式"，TechTalks 警告"循环烧光预算"；中文媒体大部分直接搬运 + 加点工业气息名词。

**三个发明者，四种定义，没人对得上**（第四种来自 Anthropic 官方文档——最朴素的那个，没人提）。

那这个词到底指什么？这套东西到底有没有新东西？这篇就把它拆清楚——**从 Claude Code 的 `/loop` 命令切入，对照原作者的三种定义，再看中英文媒体怎么各自跑偏的**。

◆ 先从 Claude Code 的 `/loop` 切入：Anthropic 自己怎么说

━━━━━━━━━━━━━━━━━━━━

讨论 Loop Engineering 之前，先看一件**最不浪漫**的事实——

Claude Code **2.1.139** 版本（2026-05-12 上线）内置了一个 slash command 叫 `/loop`。这是 Anthropic 官方文档原话：

> **`/loop`**：Run a prompt or slash command on a recurring interval (e.g. `/loop 5m /foo`). **Omit the interval to let the model self-pace.**

两种用法：

- **`/loop 5m /foo`** —— 每 5 分钟跑一次 `/foo`（按时间触发）
- **`/loop`**（不带 interval） —— **让模型自己定节奏**（按目标 / 递归）

注意 Anthropic 官方的措辞——**"let the model self-pace"**。**朴素得不像话**。没有"递归目标"、没有"自主智能体"、没有"代理范式"——就是六个字：让模型自己定节奏。

记住这句官方原话。后面对比的时候你会看到，它是判断 **"包装"含量**的标尺——同一个东西，**包装得越花，离 Anthropic 这句官方原话越远**。

━━━━━━━━━━━━━━━━━━━━

◆ 原作者三家的"loop"，各指不同的东西

━━━━━━━━━━━━━━━━━━━━

把三个发明者**到底在说什么**对照清楚——

**Peter** 那条推文的原话是 "design loops that prompt your agents"——注意这个 "loops that prompt"，是个**抽象的拟人化比喻**：把"自动化系统"叫成"loop"，因为听着比"system"或"script"更工程师范儿。Peter 自己没给 loop 任何精确定义，他只是想表达**抽象层级上移**——从 prompt 升到设计 prompt 的人。**"loop" 在他这里是修辞，不是技术名词**。

**Boris** 描述的真实使用场景具体得多：他有**几百个 Claude 实例**在并行跑，主动读 GitHub issues、扫 Twitter 反馈、翻 Slack 频道，自动浮现"接下来该做什么"。这是真正在干活的 loop——不是 prompt 的递归，是**生产环境的多实例编排**：cron + Claude + GitHub API + Slack webhook + 一堆调度脚本。**Boris 在描述工程实践，Peter 在创造修辞**。

**Addy** 在 6 月 8 日的文章里给了一个**正式定义**："A loop here can be thought of a recursive goal where you define a purpose and the AI iterates until complete."——翻译过来：你定义一个目标，AI 一直迭代直到完成。**这和 Peter 的修辞不重合，也不完全是 Boris 的"几百实例并行"**——Addy 描述的是**单个 agent 内部的目标驱动循环**。

把三家放一起，再加上 Anthropic 官方的朴素版本：

| 人 | 他的 loop 指什么 | 类比对象 |
|---|---|---|
| Peter | 设计 agent 的人（抽象比喻） | 一种"工程师姿态" |
| Boris | 几百实例的并行调度 | cron + 多 Claude 实例 + API |
| Addy | 递归目标直到完成 | ReAct / AutoGPT |
| **Anthropic 官方** | **"让模型自己定节奏"** | **`/loop` 一个命令** |

**四个人四种 loop**。最朴素的反而是 Anthropic 自己——他们一个 slash command 就把 Addy 的"递归目标"和 Boris 的"按时间触发"两种用法都涵盖了。**Addy 的整个增量，就是把 Anthropic 一个命令的两种用法 + 三个相邻功能（worktree、skills、MCP），包装成"一种新工程范式"**。

◆ 中文世界主要媒体怎么解读

━━━━━━━━━━━━━━━━━━━━

我把中文圈主要的解读分成三派——

**【A 派：忠实搬运派】（菜鸟教程、CSDN、思否、橙皮书）**

直接把英文原文翻译过来，加点"循环工程"的中文术语，**对原概念零质疑**。橙皮书甚至把这件事定调为：

> "**一周起源故事**——多位顶级技术人员**独立发现**了同一个模式（2026 年 6 月），这是趋同发现，不是重新包装。"

注意这个叙事——**"独立发现"四个字**。三个住在 X 上、互相 follow、互相点赞、互相转推的硅谷工程师在**同一周**说同一件事，被这套叙事描述成"独立发现"。

**【B 派：技术分析派】（鹤啸九天个人技术博客）**

这一派比 A 派强，至少能看出技术脉络：

> "Loop = **cron + 决策器**。"
>
> 追溯到 ReAct (2022) → AutoGPT (2023) → Ralph (2025) → 现代实现。

承认它比 prompt engineering 难十倍、token 消耗大、必须外部验证。**但仍然把它当成"真正的系统设计"**——没追问"既然就是 cron + 决策器，为什么不叫 cron engineering？"。

**【C 派：腾讯新闻派】**

腾讯新闻那篇标题《Loop Engineering：循环工程有新东西吗？》一看以为是质疑稿，**点进去发现还是吹**——只是吹得有点犹豫：

> "本质是工程意义上的进化…**它是 cost model first, productivity model second**。"
>
> 承认存在 bug 烧 token、自我验证偏差、可观测性差的问题，最后归结到"需要精心设计的系统"。

**没有任何一篇中文报道，把"`/loop` 就是一个 30 字符 slash command、剩下四件全是外围"这层戳穿**。

────────────────────

中文圈整体的盲区是：**默认西方造词背后有真东西**。

中文 AI 媒体生态有几个客观约束：

1. **流量逻辑**：写"美国新方法论"是流量密码，写"美国新方法论是包装"读者不爱看
2. **翻译惯性**：英文怎么说就怎么翻，没人去交叉比对——Peter 说的 loop、Boris 说的 loop、Addy 说的 loop **其实根本是三个东西**，但中文转述时被合成了一个
3. **样本逻辑**：大部分写测评的人**没在 Claude Code 里用过 `/loop`**——他们看的是 Addy 的 Substack 长文，没去翻 Anthropic 的更新日志。一个是 PR 文案，一个是产品手册——前者好看，后者管用

◆ 英文世界主要媒体怎么解读

━━━━━━━━━━━━━━━━━━━━

英文圈情况比中文圈复杂一些，分两派——

**【A 派：捧场派】（O'Reilly、AlphaMatch、Firecrawl）**

O'Reilly Radar 直接和 Addy **联名发表**——这家"做技术书出版的"美国老牌媒体，对 Addy 这种"硅谷大厂工程主管"有天然的认同感。文章的核心论调：

> "Prompt engineering 关注**微观管理单次交互**…**Loop engineering 把开发者的注意力转向编写更广泛的软件流水线**。"

**这就是把 Loop Engineering 从"产品功能集"升格成"开发范式"的关键操作**——通过和 Prompt Engineering 对比，让读者下意识接受"我们正在见证编程范式的演化"。

AlphaMatch 那篇标题更夸张："**Loop Engineering: The Quiet Revolution in How We Work with AI**"（AI 工作方式的静默革命）。

**【B 派：警告派】（TechTalks 的"loopmaxxing"）**

但**英文圈里也有少数清醒声音**。bdtechtalks.com 那篇是其中最直白的：

> "**Loopmaxxing**（把 loop 这件事做到极致 / 走火入魔）：把无限循环周期应用到定义不清的问题上——和早期的 **tokenmaxxing**（往 prompt 里塞越多 token 越好的迷信）错误同构。让 loop 去'改进登录页的用户体验'——**剥夺了系统的具体退出条件**。"

💡【小白翻译】英文里 **-maxxing** 这个后缀来自健身圈和宅男文化（比如 "looksmaxxing" 指"把外貌优化到极致"），翻成中文最接近的是"走火入魔"。**loopmaxxing 就是"我什么问题都用 loop 解决，结果烧光预算"**。

这是**英文圈里我见到最锋利的批判**——但 TechTalks 是个相对小众的科技博客，**它的传播度和 O'Reilly Radar 不在一个量级**。

────────────────────

**英文圈的整体格局**：

| 媒体 | 立场 | 传播度 |
|---|---|---|
| O'Reilly Radar | 联名背书 | ⭐⭐⭐⭐⭐ |
| The New Stack | "Claude Code 之父抛弃 prompt" | ⭐⭐⭐⭐ |
| Cobus Greyling 系列 | "可执行的工程模式" | ⭐⭐⭐ |
| AlphaMatch | "静默革命" | ⭐⭐⭐ |
| **TechTalks (loopmaxxing)** | **唯一明确警告** | ⭐⭐ |

**结构上是 4:1 的不对称**——四个捧场的，一个警告的。这就是 AI 媒体生态的常态：**捧场的人有 100 个声道，警告的人只有 1 个**。

◆ 拆 Addy 的"五件套"——loop 到底在哪一件里？

━━━━━━━━━━━━━━━━━━━━

回到 Addy 自己的文章。他说一个 loop 需要五件事。我边讲边标注**每一件和 loop 本体的关系**：

**第一件：`/loop` 和 `/goal` 命令** —— **这就是 loop 本体**。`/loop 5m /code-review` 每五分钟跑一次代码审查（按时间触发）；`/goal "重构 auth"` 跑到目标达成为止（按目标触发）。`/goal` 设计最讲究的一点是**双 agent 架构**——一个干活，一个独立验收。Addy 这里有一句金句：**写代码的模型给自己批作业实在太宽容了**。

**第二件：git worktree** —— **不是 loop，是隔离机制**。以前两个 agent 在同一个工作目录 commit 互相覆盖；worktree 让你同时跑五个 agent，每个在自己的隔离副本里。Claude Code 直接支持 `--worktree`。`git worktree` 这功能 Git 2.5 就有，**2015 年的事**。

**第三件：SKILL.md** —— **不是 loop，是知识包**。新 agent 上手第一件事是踩坑：改了 `/config` 不知道 CI 会读老路径、用了 `any` 不知道项目禁了。SKILL.md 就是把"我们不这么做，因为去年那次事故"写进 markdown。本质就是 AGENTS.md / CLAUDE.md 的升级版——以前一份大文件，现在按"技能"切成多份按需加载。

**第四件：MCP** —— **不是 loop，是外部接口**。MCP 是 Anthropic 2024 年 11 月发布的协议，让 agent 能读 Linear 工单、查数据库、发 Slack。一个 agent 说"修好了，代码在这"和一个 loop 自己开 PR、自己关联工单、CI 绿了自己发 Slack 通知，区别就在这。

**第五件：sub-agents** —— **不是 loop，是分工**。回到那句"自己批自己作业太宽容"——所以你在 `.claude/agents/` 下专门配一个 verifier 子 agent，独立指令、独立 context、不看 maker 的思路、只看产出。

**附加件：STATE.md** —— **不是 loop，是记忆**。agent 会忘，repo 不会忘。跨 session 的状态写进 STATE.md 或者 Linear 看板，什么都行，只要它活在单次对话之外。

────────────────────

数一下——**五件套里只有第一件是 loop 本体，其它四件都是"让 loop 跑得稳的支持设施"**。

而第一件的 loop 本体是什么？是 `/loop` 和 `/goal` 两个 slash command——**Anthropic 官方文档对它们的描述总共不到 50 个字符**。

那 Addy 的"Loop Engineering"叙事问题在哪儿？**他把"两个 slash command + 四件相邻的产品功能"，打包改名叫一种新的工程范式**——让你以为 loop 是需要花周末学习的"方法论"，而不是一个你输完回车就能用的命令。

把这五件套换个名字描述，是这样的：

> 在 Claude Code / Codex 里：用 `/loop` 或 `/goal` 设定触发条件，用 git worktree 隔离并行任务，把项目规矩写进 SKILL.md，用 MCP 接外部系统，配一个独立的 verifier sub-agent 做验收，用 STATE.md 维护跨 session 状态。

**这就是 Claude Code 2.1.139 release notes**。把它起名叫"Loop Engineering"，效果是让你觉得自己需要"学习一种新工程范式"——但其实你需要的只是**打开 Claude Code 文档翻一翻**。

────────────────────

把三家措辞放一起对比，**包装的本质一目了然**：

| 谁说的 | 怎么说 |
|---|---|
| **Anthropic 官方文档** | "let the model self-pace" |
| **Addy Osmani** | "recursive goal where you define a purpose and the AI iterates until complete" |
| **O'Reilly Radar** | "Loop engineering shifts the developer's focus to authoring the broader software assembly line" |

同一个东西，从"让模型自己定节奏"到"递归目标直到完成"再到"编写软件流水线"——**词数翻 5 倍，难度感官翻 100 倍**。

Anthropic 自己懒得包装，Addy 替他们包装了——而且 Addy 不是 Anthropic 员工，他是 Google Chrome 工程主管，**所以包装出来的话语显得不像 Anthropic 公关稿**。

━━━━━━━━━━━━━━━━━━━━

◆ 五件套的年龄——`/loop` 自己刚满 27 天

━━━━━━━━━━━━━━━━━━━━

把五件套按发布时间排一下：

| 零件 | 来源 | 时间 |
|---|---|---|
| `git worktree` | Git 2.5 | 2015 |
| MCP | Anthropic | 2024-11 |
| Sub-agents（`.claude/agents/`）| Anthropic | 2025 年中 |
| Skills（SKILL.md）| Anthropic | 2025-10 |
| **`/loop`、`/goal`、`/batch`** | **Claude Code 2.1.139** | **2026-05-12** |

`git worktree` 11 岁，MCP 一岁半，Sub-agents 一岁，Skills 半岁多。最年轻的是 `/loop` 和 `/goal`——**到 Addy 写文章那天（6 月 8 日）刚满 27 天**。

5 月那波是临门一脚：之前你想搭这套东西，得自己写 bash 脚本串起来；5 月之后产品直接内置了。**Addy 是踩着这个时间点写的——五件套终于齐了，他赶紧命名抢窗口**。把陆陆续续攒了一年多的产品功能打个包、起个总称、宣告"新方法论诞生"——这就是 Loop Engineering 这篇文章干的事。

◆ 为啥要包装成方法论？

━━━━━━━━━━━━━━━━━━━━

你可能会问：既然就是产品功能集，Anthropic 自己写 release notes 不就行了？

行不通。

Anthropic 自己写"我们做了 `/goal` 命令"——这是产品手册，只有官方博客发，不会传播。**Addy 是 Google Chrome 工程主管，不是 Anthropic 员工**。他用"造词 + 方法论"的姿态写，看起来是"独立工程师对 AI 工程现状的观察总结"，这事儿才能被 Karpathy 转发、被 Latent Space 综述、被 Hacker News 推到首页。

**命名是传播。** 这一点 140 期那篇《Harness Engineering：同一件事，第三次被命名》（ https://mp.weixin.qq.com/s/SQGRI_DbFDOLjP904QuOPQ ）已经讲过了——Karpathy 一条推文的传播效果，比 100 篇 arXiv 论文加起来都强。

而 Addy 这次更进了一步：他不是在传播一项学术成果，他在传播一组**产品功能**。

读他的文章你会发现每一节金句的潜台词都是"快用我们这些新功能"：

- 写代码的模型给自己批作业太宽容了 → 快配 sub-agent
- agent 会忘，repo 不会忘 → 快用 STATE.md
- 别再 prompt agent，去设计那个 prompt agent 的系统 → 快用 `/goal`

这不一定是 Addy 故意的——他可能真心觉得这些功能改变了 agent 工程的形态。但客观效果是**变相帮 Anthropic 和 OpenAI 卖产品**——只不过他不是产品经理，他是工程主管，所以包装成"工程方法论"才像样。

━━━━━━━━━━━━━━━━━━━━

◆ 一个活体案例：Karpathy 的 autoresearch

━━━━━━━━━━━━━━━━━━━━

剥掉"Loop Engineering"这个壳子之后，这套东西本身有没有用？**有用。**

最干净的案例是 Andrej Karpathy 2026 年 3 月 7 日开源的 autoresearch。

这个项目只有约 630 行 Python，让 AI agent 自己跑 ML 实验——人睡觉时 agent 干活。

三个文件：`prepare.py`（数据预处理，agent 不动）、`train.py`（模型和训练循环，agent 只改这一个）、`program.md`（研究方向，人写）。

外层这么转：agent 读 `program.md` 和当前 `train.py`，形成假设、改一处、跑五分钟、看 validation 指标。改进就 git commit，没改进就回退。一小时约 12 个实验，一晚约 100 个。

**注意这个项目的结构——它其实是 Addy 五件套的"最小可用版"**：

| Addy 的五件套 | autoresearch 的对应 |
|---|---|
| `/loop` 调度 | 外层 while True + 五分钟计时 |
| worktree 隔离 | git commit / git revert |
| SKILL.md 知识包 | `program.md` 研究方向 |
| MCP 接外部 | 不需要——只跟 GPU 打交道 |
| verifier sub-agent | validation 指标数值判断 |
| STATE.md 跨 session 状态 | git history 本身 |

**Karpathy 用 630 行 Python 重新发明了 Addy 五件套的"最小可用版"**——用 git commit 当 worktree，用 validation 数值当 verifier，用 program.md 当 SKILL+STATE。

但成绩很硬：5 天 GitHub 25,000 stars，现在 8 万 7 千多 stars。**Shopify 创始人 Tobi Lutke 直接拿去复现，validation 提升 19%**，小模型超过手调大模型。其他用户在 16 块 GPU 集群上一晚跑 910 个实验、总成本 309 美元。

Karpathy 自己的总结：人的工作从"写训练代码"换成了"写研究方向"。

**这个案例的价值不在于"它实现了 Loop Engineering"，而在于它证明了——你不需要 Addy 推销的那五件 Anthropic 产品功能，git + python + 一个验证函数就够了。**

需要"产品端原生 loop 支持"的，是企业级、多人协作、跨系统的 agent 部署。你一个人在 Claude Code 里和我对话干活——这五件套你大部分都用不上。

━━━━━━━━━━━━━━━━━━━━

◆ 三个真正值得记住的坑

━━━━━━━━━━━━━━━━━━━━

剥掉营销话术之后，Addy 文章里真正有价值的内容是这一段——他自己警告的三个反模式。

**坑一：你信不过 verifier，loop 就跑了个寂寞**

没人盯的 loop 是没人盯地在犯错的 loop。

AutoGPT 在 2023 年的典型翻车是：search → save → verify → "再多查一点" → 重复 300 次、零产出、烧掉 80 美元。

号称的解法是 `/goal` 的双 agent 架构加硬预算。但有个嵌套问题：**你信不信得过那个 verifier？** 如果不信，你得自己看；你自己看，那 loop 跑了个啥？

**坑二：理解债**

loop 越快地交付你没写过的代码，"代码实际是啥"和"你脑子里以为的是啥"之间的鸿沟就越大。除非你真的去读它写的东西，否则顺滑的 loop 只会让这鸿沟变大得更快。

**坑三：认知投降**

loop 自己转的时候，你很容易就懒得有自己的判断了——loop 给啥你拿啥。

Addy 这句话很重：**设计 loop 这件事，带着判断去做是解药，为了逃避思考去做是助推剂——同一个动作，相反的结果。**

最后一句金句：**两个人搭一模一样的 loop，得到的结果可以完全相反。**

💡 翻译成人话

loop 不是工具，是放大器。**勤奋的人用 loop 放大勤奋，懒的人用 loop 放大懒。**

━━━━━━━━━━━━━━━━━━━━

◆ 这次是 Reflexion 第四次借尸还魂吗？

━━━━━━━━━━━━━━━━━━━━

如果你看过 140 期那篇《Harness Engineering：同一件事，第三次被命名》，你大概会问这个。

那篇说的是 Reflexion（2023）→ Context Engineering（2025）→ Harness Engineering（2026 年 2 月），同一件事被三批不同人独立发现、各自命名。

Loop Engineering 是另一个剧本——**不是同一件事被反复发现，是同一个作者把产品功能集主动包装成方法论**。

- Reflexion 是 agent 算法内部的反思机制，论文层面的事
- Loop Engineering 是 agent 产品功能集的包装，营销层面的事

140 期说"命名是传播，不是发明"。这次更进一步——**命名是包装，不是观察**。

把这个新词放回 Addy 自己搭的三层叙事：

| 层 | 这一层管什么 | 旧名字 |
|---|---|---|
| Context Engineering | 上下文窗口里放什么 | prompt + RAG |
| Harness Engineering | 单个 agent 跑在啥环境里 | system prompt + sandbox |
| **Loop Engineering（按时间）** | **定时触发 agent** | **cron + CI/CD** |
| **Loop Engineering（按目标）** | **递归 agent 直到完成** | **ReAct / AutoGPT** |

每一层对应的"旧东西"都早就存在。**Addy 真正的贡献是给"AI 工程"造了一套能写在岗位 JD 上的词汇表**——Context Engineer 招会写 prompt 搞 RAG 的；Harness Engineer 招会配 sandbox 写 AGENTS.md 的；Loop Engineer 招懂 CI/CD、懂任务编排的。

三个抽屉，三种 JD，三种立项理由。**词汇表是给组织看的，不是给技术看的**。

━━━━━━━━━━━━━━━━━━━━

◆ 结尾

━━━━━━━━━━━━━━━━━━━━

Boris 说"我的工作是写 loop"，Karpathy 说"人的工作从写训练代码换成了写研究方向"——杠杆点上移了，但人没被踢出去。

这件事是真的。

但所谓 Loop Engineering——它不是一种新工程范式，它是 Claude Code 2.1.139 的功能集导览，套了一个听起来很高阶的名字。**loop 本体就是 `/loop` 这个 30 字符的 slash command**——剩下四件套全是它的外围支持设施，被 Addy 打包一起包装成方法论。如果 Peter Steinberger 当时推文用的是 "design schedulers"，今天我们写的就是 Scheduler Engineering 了。

**最朴素的真相反而出现在 Anthropic 官方文档里**——`/loop` 的描述就一句话：**"let the model self-pace"**。让模型自己定节奏。没有递归目标、没有软件流水线、没有静默革命。Addy / O'Reilly / 媒体们的所有包装，都是在这句朴素的话上面层层堆叠形容词。

与其学习"Loop Engineering 这种新工程范式"，不如直接打开 Claude Code 文档翻一翻：`/loop`、`/goal`、`/batch`、`.claude/agents/`、SKILL.md。前者听起来你需要花周末读一本方法论的书，后者就是 release notes，半小时看完。

最后留 Addy 那句话——这句话本身是真的：

**搭你的 loop。但要像一个还打算继续当工程师的人那样搭，不要像一个只剩按"开始"按钮的人那样搭。**
