【DGX Spark实战】两台家用机分布式量化 70B 模型——穷人的 FP8 量化指南

━━━━━━━━━━━━━━━━━━━━

◆ 起因：140GB 的小众模型，没人替你量化

━━━━━━━━━━━━━━━━━━━━

上期（180 期 https://mp.weixin.qq.com/s/ ）我们讲了 Anthropic 最新的可解释性工具 NLA（Natural Language Autoencoder）——用两个 LLM 把模型的激活向量翻译成人话，再翻译回来。激活 -> 人话 -> 激活，中间那句人话就是模型此刻在想什么。

论文开源了四个模型的预训练权重：Qwen2.5-7B、Gemma-3-12B、Gemma-3-27B、Llama-3.3-70B。其中 70B 的 AV 模型（Activation Verbalizer）是基于 Llama-3.3-70B-Instruct 微调的，BF16 格式，28 个 safetensors shard，总共大约 139GB。

我要在两台 DGX Spark 上跑 NLA 推理实验。139 期（ https://mp.weixin.qq.com/s/ ）我们系统讲过量化的原理——从 GPTQ 到 SmoothQuant 到旋转揉匀，那期偏理论。这期是实战篇：当你真的需要量化一个 140GB 的模型，而且没有人替你做好现成的量化版时，怎么办。问题来了：

**谁来量化？**

如果你用的是 Llama-3.3-70B-Instruct 原版，HuggingFace 上一搜一堆现成的 FP8 版本。但 NLA 的 AV 模型是 Anthropic 自己在 70B 基础上微调的，参数全变了。没有人会闲着替一个学术论文的附属模型做量化发布。

NLA 代码仓倒是自带了一个量化脚本 `convert_hf_to_fp8.py`，看起来挺方便。但这个脚本有一个隐含假设：**你的 GPU 能放下整个模型。**

具体来说，它用 `ThreadPoolExecutor` 开多线程（默认 `max_workers=4`），把多个 safetensors shard 并行加载到 GPU 上做量化。这个设计是给多卡单机准备的——8 张 A100 共 640GB 显存，28 个 shard 并行加载毫无压力。但它没考虑跨节点分布式的场景，更没考虑 DGX Spark 这种"单卡 128GB 统一内存"的边缘硬件。

但量化脚本并不需要一次性把整个模型加载到内存。它是逐 shard 处理的，28 个 shard 每个 ~5GB，理论上串行处理每次只占 5GB 显存。128GB 处理 5GB 的数据，应该绰绰有余。

应该。

◆ 踩坑：统一内存的 mmap 陷阱

━━━━━━━━━━━━━━━━━━━━

DGX Spark 跟普通 GPU 服务器不一样。GH200 的 CPU 和 GPU 共享 128GB 统一内存（Unified Memory），没有独立显存。safetensors 文件通过 mmap 映射进内存——操作系统不会真的把文件"加载"到内存，而是建立一个映射，读到哪一页才真正分配物理内存。

这个机制在普通服务器上没问题，因为 CPU 内存和 GPU 显存是隔离的——mmap 用的是 CPU 内存，GPU 显存另算。但统一内存架构下，**mmap 的页缓存和 GPU 张量抢的是同一块 128GB**。

坑就在这里：

**多线程并行（max_workers=4）：** 4 个线程同时打开 4 个 shard，每个 ~5GB，mmap 映射建立后，safetensors 库遍历所有 key 把数据读到 GPU——但"读到 GPU"在统一内存里就是"读到同一块内存"。4 个 shard 的原始数据 + 量化后的数据 + 中间计算缓冲区，瞬间吃掉 40-50GB。然后第 5、6、7 个 shard 的 mmap 页缓存开始堆积，操作系统来不及回收，内存直接打满。

结果：**系统整个冻死。** 不是 Python OOM 报错，是 Linux 内核的 OOM killer 来不及杀进程，整个系统失去响应。连 ctrl+c 都来不及按，只能 `sudo reboot`。

好，那改成 `max_workers=1`，串行处理总行了吧？

**不行。** mmap 的页缓存是操作系统级别的。你处理完 shard 1 关掉文件句柄，Python 释放了 tensor 对象，`torch.cuda.empty_cache()` 也调了——但 mmap 的页缓存还躺在内存里。操作系统认为这些页"将来可能还要用"，不会主动释放。28 个 shard 串行处理到第十几个的时候，累积的页缓存还是会把 128GB 吃光。

一样的结果：冻死，reboot。

◆ 关键洞见：FP8 量化不需要跨 shard 信息

━━━━━━━━━━━━━━━━━━━━

被搞死两次之后冷静下来想一想。

FP8 block quantization 的数学操作是什么？对每个权重矩阵，切成 128x128 的小块，每块独立做三件事：

```
scale = max(|block|) / 448     # 448 是 FP8 e4m3 的最大值
q = clamp(weight / scale, -448, 448)
q = cast(q, float8_e4m3fn)
```

找最大值、除以 scale、截断、类型转换。**整个过程完全在一个权重矩阵内部完成。** 不需要看其他层的参数，不需要跨 shard 统计全局信息，不需要校准数据。

这意味着：28 个 safetensors shard 之间完全没有依赖关系。每个 shard 可以独立量化，最后把结果合并就行。

这不就是天然的分布式任务吗？

我有两台 DGX Spark。一台处理偶数号 shard，一台处理奇数号 shard。每台只需要处理 14 个 shard，每个 shard ~5GB，峰值显存不到 5GB。128GB 的统一内存处理 5GB 的数据，怎么都不会爆。

◆ 先跑通环境：Docker 里的 N 个坑

━━━━━━━━━━━━━━━━━━━━

在动手写分布式脚本之前，先解决环境问题。

**坑 1：Host 上没有 CUDA 版 PyTorch**

DGX Spark 的 Ubuntu 24.04 host 系统自带 conda base 环境，有 Python、有 numpy，但 PyTorch 是 CPU 版的。直接跑量化脚本：

```
RuntimeError: Torch not compiled with CUDA enabled
```

解法：用 Docker 容器。之前 175 期做 DeepSeek V4 双机推理时，我们已经有了一个 `vllm-node-sm120` 镜像（PyTorch 2.11.0 + 定制 NCCL for SM120）。直接 tag 出一个新名字避免污染原环境：

```bash
docker tag vllm-node-sm120:latest nla-llama70b:base
```

slave 上也要 tag 同名镜像，否则后面 ssh 远程启动容器会找不到。

**坑 2：容器里没有 python 命令**

NGC 容器的传统艺能——只有 `python3`，没有 `python` 软链接。所有脚本和命令行里都要写 `python3`。

**坑 3：没有 HuggingFace CLI**

NLA 模型需要通过 `huggingface-cli download` 从 HuggingFace Hub 拉模型。但系统上没有 `huggingface_hub` 包。Ubuntu 24.04 遵循 PEP 668，系统 Python 不让你直接 pip install：

```bash
pip install --break-system-packages huggingface_hub
```

是的，就是这么优雅。

三个坑，没一个跟量化有关。这就是工程的日常。

◆ 方案设计：三个文件解决问题

━━━━━━━━━━━━━━━━━━━━

最终方案由三个文件组成。

**文件一：convert_fp8_dual.py**

双机量化主脚本。用 `torch.distributed` + NCCL 后端做进程组协调。核心逻辑很简单——rank 0 拿偶数号 shard，rank 1 拿奇数号：

```python
dist.init_process_group(backend="nccl")
rank = dist.get_rank()
world_size = dist.get_world_size()

all_shards = sorted(f for f in os.listdir(INPUT_DIR) if f.endswith(".safetensors"))
my_shards = [s for i, s in enumerate(all_shards) if i % world_size == rank]
```

每个 shard 的处理流程：打开 safetensors 文件，逐个 tensor 加载到 GPU，判断要不要量化（embedding、layernorm、lm_head、router 这些对精度敏感的层跳过），量化完立刻 `.cpu()` 搬回 CPU 内存，然后 `torch.cuda.empty_cache()`：

```python
def process_shard(filename):
    q_weights = {}
    with safetensors.safe_open(os.path.join(INPUT_DIR, filename),
                                framework="pt", device="cuda") as f:
        for k in f.keys():
            w = f.get_tensor(k)
            if should_quantize(k):
                qw, s = block_fp8(w)
                q_weights[k] = qw.cpu()
                q_weights[k.replace(".weight", ".weight_scale_inv")] = s.cpu()
            else:
                q_weights[k] = w.cpu()
    safetensors.torch.save_file(q_weights, os.path.join(OUTPUT_DIR, filename))
    torch.cuda.empty_cache()
```

注意 `should_quantize` 里跳过的层——这些层的权重保持 BF16 原样存储：

```python
def should_quantize(key):
    skip = ("layernorm", "embed", "router", "mlp.gate.",
            "norm", "lm_head", "eh_proj", "weights_proj")
    return "weight" in key and not any(s in key for s in skip)
```

为什么跳过这些？embedding 是查表操作，输入是整数 token ID，量化没有意义。layernorm 和 norm 的参数量极小（几千个标量），量化省不了空间但会损失精度。lm_head 是最终输出层，直接影响 token 概率分布，精度敏感。router 是 MoE 的门控层，误差会导致专家分配出错。

**文件二：build_fp8_index.py**

rsync 把 slave 的 shard 回收到 host 之后，需要生成两个 HuggingFace 格式的元数据文件：`config.json`（加上 `quantization_config` 字段告诉 vLLM 这是 FP8 模型）和 `model.safetensors.index.json`（权重名 -> shard 文件的映射表）。

这个脚本在 host 上单独跑，扫描所有 shard 文件生成索引。不需要 GPU。

**文件三：run_convert_fp8.sh**

一键启动脚本，在 host 上执行，串联整个流程：

1. rsync 量化脚本到 slave
2. 清理旧容器
3. ssh 到 slave 启动 worker 容器（rank=1，后台跑）
4. 在 host 启动 head 容器（rank=0，前台跑，能看到输出）
5. head 结束后拉 worker 日志
6. rsync 回收 slave 上的量化结果
7. 跑 build_fp8_index.py 生成索引

◆ FP8 量化的数学：三行公式

━━━━━━━━━━━━━━━━━━━━

139 期讲过，量化方法从简单到复杂大致分三档：RTN（Round-To-Nearest，直接四舍五入）、GPTQ/AWQ（需要校准数据的高级量化）、SmoothQuant/SpinQuant（旋转揉匀）。我们这次用的是最简单的一档——**静态 per-block RTN 量化**，只不过目标格式不是 INT8/INT4，而是 FP8。

FP8（float8_e4m3fn）是一个 8 位浮点格式：4 位指数、3 位尾数、没有 NaN（fn = finite，no NaN）。能表示的最大值是 448，精度大约是 BF16 的 1/256。

为什么用最简单的 RTN 就够了？因为 FP8 是浮点格式，自带指数位，对大小值的适应性天然比定点的 INT8 好。INT8 量化到 4-bit 以下不加旋转揉匀就会崩，但 FP8 的 8-bit 浮点精度损失本身就小，RTN 足够用。NLA 的 AV 模型不是用来做精确数值计算的——它的任务是把激活向量翻译成人话，FP8 那点精度损失在 NLA 本身 FVE 0.6-0.8 的噪声里根本看不出来。

Block quantization 的思路是把权重矩阵切成 128x128 的小块，每块独立缩放到 FP8 能表示的范围：

```
scale = max(|block|) / 448
quantized = clamp(weight / scale, -448, 448)
quantized = float8_e4m3fn(quantized)
```

第一步：找到这个 128x128 块里绝对值最大的元素，算出缩放因子。比如块里最大值是 0.5，那 scale = 0.5/448 = 0.00112。

第二步：所有元素除以 scale，这样最大值被拉到 448，其他值等比缩放。然后 clamp 截断到 [-448, 448]。

第三步：cast 到 float8_e4m3fn。

反量化的时候乘回 scale 就行。每个 128x128 的块存一个 scale 值（float32），开销可以忽略。

这是纯静态量化——不需要校准数据，不需要跑推理，不需要统计激活分布。拿到权重文件就能算。这也是为什么它可以按 shard 独立处理：每个块的 scale 只取决于自己块内的最大值，跟其他块无关。

◆ Docker 双机协调：NCCL 的标配姿势

━━━━━━━━━━━━━━━━━━━━

这套双机协调的 Docker 启动参数在 175 期做 V4 推理的时候已经跑通过。这里再梳理一遍，给第一次看的网友。

`torch.distributed.init_process_group(backend="nccl")` 需要通过环境变量知道集群拓扑：

```bash
MASTER_ADDR=169.254.248.35   # host 的 CX7 网卡 IP
MASTER_PORT=29500             # 协调用的端口
WORLD_SIZE=2                  # 总共几台机器
RANK=0                        # 我是第几台（host=0, slave=1）
LOCAL_RANK=0                  # 本机第几块 GPU（单 GPU 就是 0）
```

NCCL 需要知道走哪张网卡：

```bash
NCCL_SOCKET_IFNAME=enp1s0f1np1   # CX7 200Gbps 网卡名
NCCL_IB_HCA=rocep1s0f1            # RoCE 设备名
GLOO_SOCKET_IFNAME=enp1s0f1np1   # Gloo 后端也要配（barrier 用）
```

Docker 启动参数的几个关键 flag：

```bash
--network host    # 直接用宿主机网络，不走 Docker 虚拟网桥
--ipc host        # 共享宿主机 IPC 命名空间（NCCL 需要共享内存通信）
--ulimit memlock=-1   # 不限制内存锁定（RDMA 需要锁页内存）
--gpus all        # 把 GPU 暴露给容器
```

启动顺序有讲究：**slave 先启动（docker run -d，后台），host 再启动（docker run --rm，前台）。** 因为 NCCL init 时 rank 0 是协调节点，它会等其他 rank 连进来。如果 rank 0 先启动，它会阻塞在那里等 rank 1；如果 rank 1 先启动，它会尝试连 rank 0，连不上会重试。所以让 slave 先跑起来，host 再启动，两边几乎同时完成 init。

启动脚本里 slave 用 `docker run -d`（detach，后台跑），host 用 `docker run --rm`（前台跑完自动删容器）。host 进程结束后再 ssh 过去拉 slave 的日志：

```bash
ssh $WORKER_SSH "docker logs $CONTAINER_WORKER 2>&1 | tail -10"
```

◆ 性能数据

━━━━━━━━━━━━━━━━━━━━

实际跑下来的数据：

**量化阶段：** 28 个 shard，每台处理 14 个，串行逐 shard 处理。每个 shard ~5GB，加载到 GPU、量化、写回磁盘，单个 shard 大约几十秒。双机并行，总量化时间几分钟。峰值显存每台不到 5GB——128GB 统一内存里 5GB，毫无压力。

**rsync 回收：** slave 上 14 个量化后的 shard，FP8 格式大约各 2.5GB，总共约 34GB。通过 CX7 200Gbps 直连 rsync，实际传输速度 600-750MB/s，不到一分钟传完。

**模型体积：** 139GB BF16 -> 约 70GB FP8。接近一半——FP8 是 8 位，BF16 是 16 位，再加上 embedding/norm 等不量化的层保持原大小，所以不是精确的 50%。

**全程零 OOM。** 之前单机跑两次死机，换成双机分布式，每台只碰 14 个 shard，每个 shard 处理完就释放，页缓存根本累积不起来。

━━━━━━━━━━━━━━━━━━━━

◆ 不量化的层：一个容易忽略的细节

━━━━━━━━━━━━━━━━━━━━

量化脚本里 `should_quantize` 函数跳过了一堆层。这不是随便写的，每种都有道理：

**embedding（embed_tokens）：** 输入是离散的 token ID，本质是查表，不是矩阵乘法。FP8 量化对查表操作没有意义，反而会损失词向量精度。

**layernorm / rmsnorm（norm）：** 参数量极小（hidden_size 个标量），量化省不了几 KB 空间，但归一化层的精度直接影响后续所有计算。

**lm_head：** 最终的 vocabulary projection。这一层把隐状态映射到词表空间算 logits，精度损失会直接导致输出概率分布偏移。

**router（MoE 门控）：** 虽然 Llama-3.3-70B 不是 MoE 架构，但脚本是通用的。MoE 模型的 router 决定每个 token 分配给哪个专家，门控精度下降会导致专家分配出错。

**eh_proj / weights_proj：** NLA 特有的投影层，也跳过。

生成 config.json 时，这些被跳过的层名会写进 `modules_to_not_convert` 列表，vLLM 加载时就知道这些层要按原始精度处理。

◆ 结语：分布式不一定是为了算得更快

━━━━━━━━━━━━━━━━━━━━

提到"分布式计算"，大部分人脑子里的画面是 1000 张 H100 训练万亿参数大模型。那是一种分布式。

我们这个也是分布式——两台家用级别的 DGX Spark，各自量化 14 个 shard，用 200Gbps 网线传结果。技术含量不高，场面也不壮观。

但问题就是这样的：一台机器因为统一内存的 mmap 页缓存机制，死活放不下 28 个 shard 的量化流程。不是显存不够，不是算力不够，是操作系统的内存管理跟统一内存架构之间的一个边角冲突——页缓存不释放，累积到内存爆，系统直接冻死。

解法不需要什么高深的分布式训练框架，不需要 tensor parallel 切分，不需要 pipeline parallel 流水线。28 个文件，一台处理一半，处理完拿回来合并。

分布式不一定是为了"算得更快"。有时候是为了"算得起来"。

穷人的分布式不是 1000 张 H100 训 GPT-5，是两台家用机互相搭把手，把一个 140GB 的模型塞进可用的格式里。不是我们想折腾，是没人替我们做好这事。
