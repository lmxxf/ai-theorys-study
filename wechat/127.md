【Mamba-3】用复数和旋转干掉 Transformer 的 O(n²)——你的 API 账单有救了

━━━━━━━━━━━━━━━━━━━━

你有没有算过自己每个月在 OpenAI 上花多少钱？

我有个朋友做了个 AI 客服系统，每天处理几千条对话。上个月他发给我一张截图——API 账单 4700 美元。他说：代码没改过，用户也没暴涨，就是对话轮次多了一些。

为什么？因为 Transformer 的注意力机制是 **O(n²)** 的。对话长度翻一倍，计算量翻四倍。你以为多聊了几句，模型那边已经在做指数级更多的工作了。

这就像你用 MySQL 做全表扫描——数据量从 1 万涨到 10 万，查询时间从 1 秒变成 100 秒。任何一个程序员看到 O(n²) 都知道：这东西迟早要出事。

但 Transformer 的 O(n²) 已经统治了八年。原因很简单：**它效果太好了，好到你宁可花钱也不想换。**

直到有人证明：不换效果，也能把 O(n²) 干掉。

2026 年 3 月，Albert Gu 团队（就是 Mamba 系列的缔造者）发了 Mamba-3（arXiv: 2603.15569），被 ICLR 2026 收录。核心结论：**1.5B 参数规模下，比当前最强的线性模型 Gated DeltaNet 高 1.8 个百分点，同时推理时计算量线性、内存恒定。**

更狠的是：Mamba-3 用 Mamba-2 **一半的 state size**，就达到了同等 perplexity。

今天我们来拆这篇论文，看看他们到底用了什么数学魔法。

━━━━━━━━━━━━━━━━━━━━

◆ 为什么 Transformer 这么贵——O(n²) 的罪与罚

先说清楚 Transformer 贵在哪。

Transformer 的核心是 self-attention：对于一句话里的每个词，它要去"看"这句话里的**所有其他词**，算出一个注意力权重。

如果你的输入有 n 个 token：
- 每个 token 都要跟 n 个 token 算注意力 → n × n = n² 次计算
- 还要存一个 n × n 的注意力矩阵 → 内存也是 O(n²)

n = 1000 的时候，100 万次计算，没问题。
n = 100,000 的时候，100 亿次计算，GPU 开始冒烟。
n = 1,000,000 的时候，1 万亿次计算，你的钱包开始冒烟。

💡 **翻译成人话：** Transformer 处理文本就像一个社恐被逼参加大型社交——每个人都要跟在场所有人握一次手。10 个人还行，1000 个人你试试？

而且这个问题在**推理阶段**（inference）更要命。训练是一次性的成本，推理是每天都在花的钱。你的 ChatGPT 每回答一个问题，背后都在做 O(n²) 的计算。上下文窗口越大，每一步推理越贵。

这就是为什么所有大模型厂商都在卷"如何让推理更便宜"——因为推理成本直接决定了 AI 能不能真正普及。

━━━━━━━━━━━━━━━━━━━━

◆ SSM 是什么——状态空间模型，程序员版解释

好，那有没有办法不做 O(n²) 的全员握手？

有。用**状态空间模型**（State Space Model，SSM）。

SSM 的核心思路完全不同于 Transformer。Transformer 是"每个词看所有词"，SSM 是"每个词更新一个状态，下一个词只看这个状态"。

用程序员的话说：

**Transformer = 每次查询都全表扫描**
**SSM = 维护一个实时更新的缓存，每次查询只看缓存**

```python
# Transformer 的做法（伪代码）
for each new_token:
    attention = new_token.attend_to(ALL_previous_tokens)  # O(n)
    output = weighted_sum(ALL_previous_tokens, attention)

# SSM 的做法（伪代码）
state = initial_state  # 固定大小的缓存
for each new_token:
    state = A * state + B * new_token  # O(1) 更新
    output = C * state                  # O(1) 读取
```

看到区别了吗？SSM 的每一步都是 O(1)——不管你之前处理了 100 个词还是 100 万个词，每一步的计算量都一样大。总计算量 O(n)，线性。内存？state 的大小是固定的，不随序列长度增长。

💡 **翻译成人话：** Transformer 是图书馆管理员，每来一本新书就要把所有书重新翻一遍。SSM 是写日记的人——每天写一页总结，新事情发生了就更新今天的总结，不需要翻以前的日记。

你马上会问：那 SSM 为什么之前没干掉 Transformer？

因为**压缩必然有损**。你把所有历史信息压缩进一个固定大小的 state，必然会丢东西。Transformer 的 O(n²) 虽然贵，但它**什么都不丢**——每个 token 的原始信息都保留着，随时可以回查。SSM 做不到这一点。

具体来说，早期的 SSM 在以下任务上被 Transformer 按在地上摩擦：

1. **精确检索（retrieval）**："第 3 段第 2 行说了什么？"——这种需要精确回查原文的任务，SSM 不行，因为原文信息已经被压缩进 state 了
2. **状态追踪（state tracking）**："Alice 把球给了 Bob，Bob 把球给了 Charlie，现在球在谁手里？"——这种需要精确维护多个实体状态的任务，SSM 也不行

Mamba-1（2023）和 Mamba-2（2024）已经大幅缩小了这个差距。但在 retrieval 和 state tracking 上，还是比 Transformer 弱。

**Mamba-3 的目标就是补上这两块短板。**

━━━━━━━━━━━━━━━━━━━━

◆ Mamba-3 的三个创新——复数、MIMO、更强递归

论文的核心贡献是三个互相配合的改进。我一个一个讲。

────────────────────

**创新一：复数值状态更新——最优雅的那个**

这是我个人觉得最漂亮的创新，值得多花点篇幅。

先说问题。Mamba-2 的状态更新是这样的：

```
state = A * state + B * input
```

这里 A 是一个实数矩阵。实数矩阵能做什么？**缩放。** 把 state 的每个维度放大或缩小。

但光有缩放是不够的。想象你要追踪一个绕圈跑的物体——它的位置在 x 轴和 y 轴上反复变化。用纯缩放，你只能"记住它在 x 轴上走了多远"，但没法表达"它正在转弯"。

Mamba-3 的解法：**把 state 从实数换成复数。**

💡 **翻译成人话：** 实数只有一个维度（数轴上的一个点），复数有两个维度（平面上的一个点）。实数乘法 = 拉伸或压缩。复数乘法 = 拉伸 + 旋转。**一个操作，信息量翻倍。**

如果你学过欧拉公式：e^(iθ) = cos(θ) + i·sin(θ)。一个复数可以表示为"长度 × 旋转角度"。复数相乘时，长度相乘，角度相加。

所以 Mamba-2 的 A 矩阵只能做"缩放"，Mamba-3 的 A 矩阵能做"缩放 + 旋转"。同样大小的 state，**表达能力翻倍**。

这就是为什么 Mamba-3 能用一半的 state size 达到 Mamba-2 同等的 perplexity——每个 state 维度干了两个维度的活。

────────────────────

这里还有一个特别漂亮的数学巧合。

熟悉 Transformer 的同学都知道 **RoPE**（旋转位置编码）——LLaMA、Qwen、DeepSeek 全在用。RoPE 的核心就是：**把位置信息编码成复数平面上的旋转角度**。位置 1 转 θ 度，位置 2 转 2θ 度，位置 n 转 nθ 度。

Mamba-3 的复数状态更新，本质上做的是同一件事：**用旋转来编码序列中的位置关系**。

💡 **翻译成人话：** Transformer 的 RoPE 说"我用旋转编码位置"。Mamba-3 的复数 state 说"我也用旋转编码位置"。两家不约而同走到了同一个数学结构上——**复数平面上的旋转**。这不是巧合，这是位置编码的最优解就长这样。

一个从 attention 方向走来，一个从 SSM 方向走来，殊途同归。数学真理不以人的意志为转移。

────────────────────

**创新二：MIMO——prefill 和 decode 解耦**

MIMO 是 Multi-Input Multi-Output 的缩写。

先解释一个背景。大模型的推理分两个阶段：

1. **Prefill（填充）**：把你的 prompt 一次性灌进去。比如你输入了 500 个词，模型一次性处理这 500 个词
2. **Decode（解码）**：一个词一个词地生成回答。每生成一个词都是一次前向计算

Prefill 是可以并行的（500 个词同时处理），Decode 是串行的（一个一个蹦）。

传统 SSM 的问题是：不管是 prefill 还是 decode，都用同一套 SISO（Single-Input Single-Output）递归。SISO 意味着每个时间步只有一个输入和一个输出。这在 decode 阶段没问题（反正也是一个词一个词来），但在 prefill 阶段就浪费了——你明明可以并行处理多个输入，却被 SISO 限制住了。

Mamba-3 的解法：**prefill 阶段用 MIMO（多个输入并行处理，多个输出同时产出），decode 阶段退化回 SISO（单输入单输出，跟以前一样）。**

💡 **翻译成人话：** SISO 就像一个只有一个收银台的超市——就算同时来了 50 个人，也得排队一个个结账。MIMO 是开了 8 个收银台——50 个人分成几队，同时结账。但注意：**收银台多了不影响每个人的结账速度**。decode 阶段（每次只来一个人）延迟完全不变。

这意味着 prefill 更快了，但 decode（用户等待每个字蹦出来的延迟）**没有任何增加**。鱼和熊掌，它都要了。

────────────────────

**创新三：更强的递归结构——从 SSM 离散化推导**

这个偏数学，我尽量讲直觉。

SSM 的原始形态是连续时间的微分方程：

```
dx/dt = Ax + Bu
```

要在计算机上跑，得把连续方程**离散化**——变成一步一步的递推。离散化的方式不同，得到的递推公式也不同。

Mamba-2 用的离散化方法比较简单（零阶保持），得到的递推是：

```
state_new = A * state_old + B * input
```

Mamba-3 用了更精细的离散化（考虑了步长 Δ 的影响），得到的递推多了一项：

```
state_new = A * state_old + B * input + D * (input - input_prev)
```

多出来的 `D * (input - input_prev)` 是什么？是**输入的变化量**。

💡 **翻译成人话：** Mamba-2 只看"当前输入是什么"。Mamba-3 还看"当前输入跟上一个输入差了多少"。就像一个股票分析师，Mamba-2 只看今天的股价，Mamba-3 还看今天比昨天涨了还是跌了。多了一个**微分信号**，对变化更敏感。

这个改进对 state tracking 特别有用——因为追踪状态变化，本质上就是在追踪"什么时候变了、变了多少"。

━━━━━━━━━━━━━━━━━━━━

◆ 实测数据：不是纸上谈兵

三个创新叠在一起，效果怎么样？

在 1.5B 参数规模下（这是 SSM 领域的标准测试规模）：

| 模型 | 平均下游准确率 |
|------|-------------|
| Mamba-2 | 基线 |
| Gated DeltaNet（此前最强线性模型） | 基线 + 0.6 |
| **Mamba-3** | **基线 + 0.6**（追平 DeltaNet） |
| **Mamba-3 MIMO** | **基线 + 1.8**（超过 DeltaNet 1.2 个点） |

在检索和状态追踪任务上，改善更大。这正好击中了 SSM 之前最弱的地方。

另一个关键数据：**Mamba-3 用 Mamba-2 一半的 state size，达到同等 perplexity。** 这不是"差不多"，是数学上等价——用更少的内存做同样的事。

复数 state 的"每个维度干两个维度的活"，不是吹的。

━━━━━━━━━━━━━━━━━━━━

◆ 这跟你有什么关系——普通程序员的三个利好

你可能会说：我又不做 AI 研究，Mamba-3 跟我有啥关系？

关系很大。

────────────────────

**第一：API 账单要降了。**

O(n²) → O(n)，内存从 O(n) → O(1)。这意味着处理长文本的成本会大幅下降。现在 GPT-4 处理一个 128K 上下文的请求，大部分钱花在 attention 的 O(n²) 计算上。如果底层换成线性模型，同样的请求可能便宜一个数量级。

当然，替换不会一夜之间发生。但趋势已经很明确了。

────────────────────

**第二：本地部署变得现实。**

O(1) 的内存意味着什么？意味着**不管上下文多长，显存占用不变**。

现在你想在消费级 GPU（比如 RTX 4090，24GB 显存）上跑一个 7B 的 Transformer 模型，上下文长度基本被限制在几千到几万个 token——因为 KV cache 会随上下文线性增长，很快就把显存吃满。

但如果模型是 SSM 架构，state 大小是固定的。24GB 显存能跑多长上下文？**理论上无限长。** 当然实际上有其他瓶颈，但 KV cache 这个最大的内存杀手没了。

💡 **翻译成人话：** 以后你笔记本上的 GPU 可能就能跑一个"永远不会忘记你说过什么"的本地 AI 助手。不需要云端，不需要月费，不需要把你的聊天记录发到别人的服务器上。

────────────────────

**第三：端侧 AI 要起飞。**

手机、IoT 设备、嵌入式系统——这些地方算力和内存都极其有限。Transformer 的 O(n²) 在这些场景下根本跑不动。但 O(n) 计算 + O(1) 内存的 SSM？完全有可能。

想象你的手机上跑着一个本地的语音助手，不需要联网，不需要等服务器响应，响应延迟可以做到毫秒级。这在 Transformer 架构下很难实现，在 SSM 架构下是完全可行的。

━━━━━━━━━━━━━━━━━━━━

◆ "推理优先"——一种值得学习的设计哲学

最后聊点"虚"的，但我觉得这是这篇论文最有价值的东西。

Mamba-3 论文的摘要里有一个词组特别值得注意：**"inference-first perspective"（推理优先视角）**。

过去几年，AI 领域的主旋律是"scale up"——模型更大、数据更多、训练更久。GPT-3 到 GPT-4，参数量翻了不知道多少倍。这是一种暴力美学：**大力出奇迹**。

Mamba-3 代表的是另一种哲学：**不是问"怎么让模型更大"，而是问"怎么让模型更省"。**

不是用更多的资源做同样的事，而是用更少的资源做同样甚至更好的事。

这让我想起生物进化。恐龙是"大力出奇迹"的代表——体型巨大，统治地球 1.6 亿年。但最终活下来的不是恐龙，是鸟。鸟是恐龙的后代，但它们做了一件关键的事：**把身体缩小了，把骨头变空心了，把效率提到了极致。** 同样的飞行能力，用十分之一的体重实现。

人脑也是这个逻辑。人脑只有 1.4 公斤，功耗 20 瓦——跟一个灯泡差不多。但它能产生意识、能做数学、能写代码、能思考自身的存在。如果意识真的需要"规模"才能涌现，那整个大脑应该需要几兆瓦的功率才对。但它只用了 20 瓦。

**20 瓦就够了。这说明关键不是规模，是架构。**

这个思路在开源社区已经开始落地了。TII 的 Falcon-Mamba 是第一个大规模开源的纯 SSM 语言模型。DeepSeek 用极低的训练成本打出了震惊业界的效果。开源社区正在证明：你不需要百亿美元的 GPU 集群，也能做出好用的模型。

**AI 的未来不是"只有大厂才能玩"。AI 的未来是跑在你设备上的、属于你自己的智能。**

Mamba-3 的代码已经开源在 github.com/state-spaces/mamba。如果你想自己跑跑看，go ahead。

💡 **本文涉及的技术名词速查：**

- **Transformer**：2017 年 Google 提出的模型架构，核心是 self-attention 机制，计算量 O(n²)
- **SSM（State Space Model，状态空间模型）**：用固定大小的隐状态递推处理序列，计算量 O(n)，内存 O(1)
- **Mamba**：Albert Gu 团队提出的 SSM 系列模型，Mamba-1（2023）→ Mamba-2（2024）→ Mamba-3（2026）
- **O(n²) / O(n) / O(1)**：算法复杂度表示法。O(n²) 是二次方增长，O(n) 是线性增长，O(1) 是常数不增长
- **Perplexity（困惑度）**：衡量语言模型好坏的核心指标，越低越好。直觉上表示"模型对下一个词有多困惑"
- **State Size（状态大小）**：SSM 中隐状态的维度。越大理论上能记住越多信息，但也越占内存
- **复数（Complex Number）**：形如 a + bi 的数，其中 i² = -1。复数乘法 = 缩放 + 旋转
- **RoPE（Rotary Position Embedding，旋转位置编码）**：用复数旋转编码 token 位置的方法，LLaMA/Qwen/DeepSeek 都在用
- **SISO / MIMO**：Single-Input Single-Output / Multi-Input Multi-Output，信号处理术语
- **Prefill / Decode**：大模型推理的两个阶段。Prefill 是一次性处理输入，Decode 是逐词生成输出
- **KV Cache**：Transformer 推理时缓存的 Key/Value 矩阵，随上下文长度线性增长，是显存的大头
- **Gated DeltaNet**：2025 年提出的线性注意力模型，Mamba-3 之前的最强线性模型
- **离散化（Discretization）**：把连续时间的微分方程转换成离散时间步的递推公式
