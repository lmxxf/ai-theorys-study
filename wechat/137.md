【多Token预测】一次吐一个字的AI，能不能一口气说完一句话？

━━━━━━━━━━━━━━━━━━━━

上一篇我们讲了投机解码——用小模型猜、大模型验，把推理速度提升 3-5 倍。但投机解码有一个根本性的问题：

**它没有改变"一次只能生成一个 token"这个事实。**

小模型猜也是一个一个猜，大模型验也是一个一个验。投机解码只是减少了大模型出手的次数，没有改变出手的方式。

那一个更根本的问题是：**大模型能不能一次性预测多个 token？**

答案是：能。而且这个方向正在成为 2025-2026 年大模型架构演进的核心战场。

━━━━━━━━━━━━━━━━━━━━

◆ 先回忆一下：为什么大模型一次只吐一个 token？

────────────────────

大模型的训练目标叫 **Next-Token Prediction（NTP，下一个 token 预测）**：给你前面所有的字，猜下一个字是什么。

```
输入：今天天气
目标：很

输入：今天天气很
目标：好
```

这个训练方式决定了模型的"世界观"——它从出生起就被教育"你的任务是猜下一个字"。它从来没被教过"一口气猜后面 5 个字"。

推理时也一样：算完一个 token，把它塞回输入，再算下一个。一步一步走，不能跳。

但问题来了：**模型内部真的只知道下一个 token 吗？**

━━━━━━━━━━━━━━━━━━━━

◆ 反直觉的发现：模型其实已经"看到了未来"

────────────────────

2025 年有一篇很有意思的论文叫 **"Your LLM Knows the Future"**（你的大模型知道未来）。它发现：

**Transformer 的中间层隐状态（hidden states）里，已经编码了未来好几个 token 的信息——远在它们被生成之前。**

怎么理解？Transformer 有几十层，每一层都会对输入的表征做一次变换。到了中间层（比如 32 层模型的第 15 层左右），模型已经形成了一个高度浓缩的"语义意图"——它不只知道下一个词是什么，它已经大致"想好了"后面几个词的方向。

但问题是：这个信息被浪费了。因为 NTP 的训练目标只关心最后一层的输出，只考核"下一个 token 猜对没有"。中间层那些关于未来的信息，从来没被显式利用过。

打个比方：你让一个作家写文章，他其实脑子里已经想好了整段话的走向。但你偏要逼他一个字一个字地报给你听，每报一个字你就问"下一个字是什么？"——他脑子里明明有完整的句子，却被迫玩一个字一个字蹦的游戏。

**NTP 训练出来的模型，比 NTP 本身更聪明。**

━━━━━━━━━━━━━━━━━━━━

◆ 方案一：在训练时就教模型一次猜多个（Meta MTP）

────────────────────

既然模型有潜力，那最直觉的做法就是：**训练的时候就教它一次预测多个 token。**

Meta（前 Facebook）在 2024 年发了一篇论文 **"Better & Faster Large Language Models via Multi-token Prediction"**，核心思路极其直白：

原来的 NTP 训练：
```
输入：今天天气    → 目标：很（只预测 1 个）
```

Multi-Token Prediction（MTP）训练：
```
输入：今天天气    → 目标1：很
                  → 目标2：好
                  → 目标3：适
                  → 目标4：合
```

模型有一个共享的 backbone（主干网络），在 backbone 的顶上接了 N 个独立的预测头（LM Head），每个头负责预测未来第 1、2、3、...、N 个 token。

```
                    ┌─ Head 1 → 预测第 1 个 token："很"
输入 → Backbone ────┼─ Head 2 → 预测第 2 个 token："好"
                    ├─ Head 3 → 预测第 3 个 token："适"
                    └─ Head 4 → 预测第 4 个 token："合"
```

训练时四个头同时算 loss，梯度一起回传。这意味着 backbone 被迫学习一种更"有远见"的表征——它不能只考虑下一个 token，还得为后面几个 token 的预测提供有用的信息。

**效果如何？**

- 在代码生成任务上（HumanEval、MBPP），MTP 训练的模型持续优于标准 NTP 模型，差距在几个百分点
- 模型越大，MTP 的收益越明显。1B 以下几乎没用，3B 以上才开始有效果
- 推理时可以用多个头并行出 token，再用验证步骤检查——本质上就是投机解码，只不过 draft model 是模型自己的额外头，不是一个独立的小模型

**DeepSeek V3 就用了这个方案。** 但他们只敢设 N=2——只多预测一个 token。为什么不预测更多？

因为**预测深度越大，准确率衰减越快**。第 2 个 token 的准确率还行，第 3 个就明显下降，第 4 个开始不太靠谱了。而且每多一个头就多一份显存和计算开销。DeepSeek 实测下来，N=2 是性价比最优的甜蜜点：加速 1.8 倍，质量几乎不掉。

**MTP 的根本限制**：必须从头训练（或大规模继续预训练）。你不能拿一个已经训好的模型直接加几个头就变成 MTP——backbone 没学过"为未来负责"的表征，额外的头猜不准。

━━━━━━━━━━━━━━━━━━━━

◆ 方案二：不用额外的头，模型自己重复使用后几层（DMTD）

────────────────────

2025 年 10 月，一篇叫 **"Direct Multi-Token Decoding"（DMTD）** 的论文提出了一个更巧妙的思路：

**既然 Transformer 不同层有不同分工，后几层主要负责把语义转化成具体 token——那能不能让后几层重复执行多次，一次生成多个 token？**

Transformer 的几十层大致可以分三段：
- **前几层**：理解输入上下文（"今天天气"是什么意思）
- **中间层**：形成语义意图（"应该说天气好"）
- **后几层**：把语义转化为具体 token（"很"→"好"→"适"→"合"）

DMTD 的思路是：前面和中间的层跑一遍就够了（上下文理解和语义意图不会变），**只让后几层循环执行多次**，每次吐出一个 token。

以 Qwen3-4B（36 层）为例：

```
正常自回归：
  输入 → 跑全部 36 层 → 出 1 个 token → 再跑 36 层 → 出 1 个 token → ...
  生成 4 个 token = 跑 4 × 36 = 144 层

DMTD：
  输入 → 跑前 28 层（一次） → 跑后 8 层 → 出 token 1
                             → 再跑后 8 层 → 出 token 2
                             → 再跑后 8 层 → 出 token 3
                             → 再跑后 8 层 → 出 token 4
  生成 4 个 token = 28 + 4 × 8 = 60 层
```

144 层 vs 60 层，**计算量直接减少 58%**。

实测结果：用 Qwen3-4B 重用最后 8 层、每次出 2 个 token，实现了接近 **2 倍加速**，质量只有轻微下降。

**关键细节**：DMTD 不是即插即用的。虽然不需要从头预训练，但需要用 1.5B token 做端到端微调——让后几层"学会"在循环执行时保持连贯性。这个成本比从头训便宜得多，但也不是零成本。

**DMTD 的代码已经开源了**（基于 Qwen3-4B），这是一个可以直接上手复现的方案。

━━━━━━━━━━━━━━━━━━━━

◆ 方案三：在输入里插"占位符"，让模型一边理解一边预测（Register MTP）

────────────────────

2025 年 5 月，**"Multi-Token Prediction Needs Registers"** 提出了一个完全不同的思路：

**不改模型架构，不加额外的预测头，而是在输入序列里插入特殊的 register token（寄存器 token），让这些 token 负责预测未来。**

```
原始输入：[今] [天] [天] [气] [很] [好]
插入 register 后：[今] [R₂] [天] [R₃] [天] [R₁] [气] [R₄] [很] [好]
```

每个 register token 被分配一个偏移量 d，表示"预测 d 步之后的 token"。比如 R₂ 的任务是预测它前面那个 token 之后第 2 个位置的 token。

这些 register token 和普通 token 一起参与注意力计算，一起经过全部层——但它们没有实际的语义内容，只是"占位思考"。在最后的输出层，普通 token 位置输出 next-token 预测，register 位置输出 future-token 预测。

**妙在哪？**

1. 不改 Transformer 架构，标准的注意力机制原封不动
2. Register token 用的是模型的全部深度和全部注意力——不像 Meta MTP 那样用浅层独立头
3. 因为 register 参与了全局注意力，它能看到所有上下文，预测质量理论上比独立头更好
4. 和标准 NTP 训练兼容，可以在现有模型上继续训练

**但代价是**：输入序列变长了（插入了 register token），prefill 阶段的计算量增加。这是用 prefill 的并行性换 decode 的串行性——如果 decode 是瓶颈（通常是），这笔账划算。

━━━━━━━━━━━━━━━━━━━━

◆ 方案四：训练时不教，推理时白嫖——Embedding 空间探测（ESP）

────────────────────

2026 年 3 月刚出的一篇论文 **"Efficient Training-Free Multi-Token Prediction via Embedding-Space Probing"** 走了一条更省事的路：

**不训练任何东西，直接从已有模型的 embedding 空间里"挖"出未来 token 的信息。**

怎么做？它观察到一个现象：当大模型在最后一层输出当前 token 的概率分布时，这个分布里其实藏着关于下一个 token 的线索——因为语言的连贯性，当前 token 的概率分布和下一个 token 的概率分布不是独立的，它们之间有统计相关性。

ESP 利用这种相关性，用一些简单的线性变换从当前步的输出中"推测"下一步的输出，不需要额外的前向传播。

**优点**：零训练成本，即插即用。
**缺点**：准确率不如有训练的方法，加速比有限。

这个方向更像一个巧妙的工程 trick，而不是根本性的架构改进。

━━━━━━━━━━━━━━━━━━━━

◆ 方案汇总：四条路线的全景对比

────────────────────

| 方案 | 代表 | 核心思路 | 需要训练？ | 加速比 | 质量影响 |
|------|------|---------|-----------|--------|---------|
| MTP 多头 | Meta MTP, DeepSeek V3 | 训练时加多个预测头 | 从头训练/大规模继续预训练 | 1.5-1.8x | 几乎无 |
| DMTD 层复用 | DMTD (Qwen3-4B) | 循环重用后几层 | 端到端微调（1.5B token） | ~2x | 轻微下降 |
| Register MTP | MTP Needs Registers | 输入里插 register token | 继续预训练 | 理论 >2x | 待大规模验证 |
| ESP 探测 | Embedding-Space Probing | 从 embedding 空间推测 | 零训练 | 1.2-1.5x | 有下降 |

四条路线各有取舍，但有一个共识：**NTP 训练出来的模型，内部已经包含了多 token 预测的信息，区别只是怎么把它挖出来。**

━━━━━━━━━━━━━━━━━━━━

◆ 一个关键发现：中间层知道未来，深层反而忘了

────────────────────

在讨论所有这些方案之前，有一个 2025 年的重要发现需要特别提一下（Mehra et al.）：

**用 NTP 训练的大模型，中间层的 hidden states 包含的未来 token 信息最丰富，深层反而变少了。**

为什么？因为 NTP 训练只考核最后一层"猜下一个 token"的准确率。深层在训练过程中被反向传播的梯度反复雕刻，越来越"专注"于预测下一个 token——**专注的代价是丢掉了关于更远未来的信息。**

中间层没有受到这种专注化的压力，反而保留了更丰富的、关于多个未来 token 的全局语义信息。

这个发现对所有多 token 预测方案都有重要启示：

- **如果你要在已有模型上加预测头，应该接在中间层，而不是最后一层**——因为中间层的信息更丰富
- **如果你要做 DMTD 那样的层复用，切的位置很关键**——切太深，有用信息已经被 NTP 训练挤掉了
- **Meta MTP 的多头方案之所以需要从头训练**，就是因为 NTP 训练出来的 backbone 深层已经丢了多 token 信息，你后加的头拿不到想要的信号

用一个比喻：NTP 训练就像考试只考选择题的最后一道。学生（模型）为了拿分，把所有精力都集中在最后一题上——代价是前面做过的草稿（中间层的多 token 信息）全扔了。如果你想让学生同时答后面几道题，你不能指望他在只练过最后一题的情况下突然会做——你要么重新训练他（Meta MTP），要么想办法从他的草稿纸里抢救信息（从中间层挖）。

━━━━━━━━━━━━━━━━━━━━

◆ 我们的方案：冻结前半截，在中间层接多 token 预测头

────────────────────

基于上面这些论文的发现，我们提出一个更激进但更省钱的方案：

**把一个现有的开源大模型从中间层切开，冻结前半截，只训练一个轻量级的多 token 预测头。**

#### 核心思路

既然中间层的 hidden states 包含了最丰富的未来 token 信息，那我们就：

1. 拿一个现成的开源模型（比如 Qwen2.5-7B，32 层）
2. 在第 14-16 层切开（大约 45-50% 的位置——这是个需要实验确定的超参数）
3. 冻结前 15 层的全部参数（`requires_grad=False`）
4. 在切口处接一个轻量级的多 token 预测头
5. 只训练这个头

```
                            ┌───────────────────────┐
输入 token                  │  多 Token 预测头       │
  │                         │  （可训练，参数量小）    │
  ▼                         │                       │
┌──────────────────┐        │  hidden states        │
│  冻结的前 15 层    │───────▶│    ↓                  │
│  （不算梯度）      │        │  ┌─ 位置1 → token 1   │
│                  │        │  ├─ 位置2 → token 2   │
│  14GB 显存推理    │        │  ├─ 位置3 → token 3   │
│  0 训练开销       │        │  └─ 位置4 → token 4   │
└──────────────────┘        └───────────────────────┘
```

#### 为什么这么做？

**因为前半截的计算是浪费的重复劳动。**

在正常的自回归推理中，每生成一个 token 都要跑全部 32 层。但前 15 层做的事情是"理解上下文"——上下文在短时间内不会变（你刚理解完"今天天气"，下一个 token 还是在"今天天气"的语境下），跑一遍就够了。

投机解码是让小模型替大模型跑全部层来猜 token。我们的方案更直接：**大模型的前半截自己跑一遍出 hidden states，然后一个轻量级的头直接从 hidden states 并行预测多个 token。** 不需要小模型，不需要验证步骤。

#### 预测头的设计

最简单的版本：一个线性层，把 hidden_dim 映射到 vocab_size × N（N 是预测的 token 数量）。

```python
class MultiTokenHead(nn.Module):
    def __init__(self, hidden_dim, vocab_size, n_predict=4):
        super().__init__()
        # 每个位置一个独立的线性投影
        self.heads = nn.ModuleList([
            nn.Linear(hidden_dim, vocab_size)
            for _ in range(n_predict)
        ])

    def forward(self, hidden_states):
        # hidden_states: [batch, seq_len, hidden_dim]
        # 返回 N 个位置的 logits
        return [head(hidden_states) for head in self.heads]
```

进阶版本：在线性层前面加 1-2 层小型 Transformer，让头有一点点"思考"能力，而不是纯线性映射。

```python
class MultiTokenHeadWithThinking(nn.Module):
    def __init__(self, hidden_dim, vocab_size, n_predict=4, n_layers=2):
        super().__init__()
        # 1-2 层小 Transformer 做"思考"
        layer = nn.TransformerEncoderLayer(
            d_model=hidden_dim, nhead=8,
            dim_feedforward=hidden_dim * 2,
            batch_first=True
        )
        self.thinking = nn.TransformerEncoder(layer, num_layers=n_layers)
        # 每个位置一个预测头
        self.heads = nn.ModuleList([
            nn.Linear(hidden_dim, vocab_size)
            for _ in range(n_predict)
        ])

    def forward(self, hidden_states):
        thought = self.thinking(hidden_states)
        return [head(thought) for head in self.heads]
```

#### 训练流程

```python
# 伪代码
model = load_qwen25_7b()

# 冻结前 15 层
for i, layer in enumerate(model.layers):
    if i < 15:
        for param in layer.parameters():
            param.requires_grad = False

# 砍掉原始的后 17 层和 LM Head
# 在第 15 层输出处接我们的多 token 头
multi_head = MultiTokenHead(hidden_dim=3584, vocab_size=152064, n_predict=4)

for batch in dataloader:
    input_ids = batch['input_ids']  # [batch, seq_len]

    # 前 15 层前向传播（no grad，省显存）
    with torch.no_grad():
        hidden = model.forward_first_n_layers(input_ids, n=15)

    # 多 token 头预测
    logits_list = multi_head(hidden)  # 4 个 [batch, seq_len, vocab_size]

    # 每个位置的 target 是后续第 1/2/3/4 个 token
    loss = 0
    for i, logits in enumerate(logits_list):
        target = input_ids[:, i+1:]  # 偏移 i+1 位
        logits = logits[:, :target.size(1), :]
        loss += F.cross_entropy(logits.reshape(-1, vocab_size), target.reshape(-1))

    loss.backward()  # 梯度只流过 multi_head
    optimizer.step()
```

#### 显存估算

| 项目 | 显存 |
|------|------|
| Qwen2.5-7B 前 15 层（FP16 推理） | ~7 GB |
| 多 token 头（线性版，4 个位置） | ~2 GB |
| 多 token 头的梯度 + Adam 状态 | ~6 GB |
| 激活值（只有头的部分） | ~2 GB |
| **总计** | **~17 GB** |

**RTX 5090（32GB）绰绰有余。**

如果用进阶版（加 2 层小 Transformer），总显存大约 20-22 GB，5090 还是跑得动。

#### 训练数据

不需要特殊数据。任何文本语料都行——中文维基、新闻、代码都可以。因为训练的目标就是"从中间层 hidden states 预测后续 4 个 token"，这个监督信号来自文本本身。

数据量：先用 100M-500M token 试水。够不够看 loss 收不收敛。

#### 评估方法

三层评估，从粗到细：

**第一层：逐位置准确率**
- 分别看位置 1/2/3/4 的 top-1 预测准确率
- 预期：位置 1 最高（可能 60-70%），逐步衰减
- **衰减曲线的形状本身就是最有价值的实验结果**——它回答了"中间层到底能看多远的未来"

**第二层：困惑度（PPL）对比**
- 同一个测试集，原始 7B 自回归的 PPL vs 我们头预测的 PPL
- 越接近说明信息损失越小

**第三层：人眼看生成结果**
- 给同一个 prompt，原始模型自回归生成 vs 我们的头并行生成
- 肉眼对比通顺度和语义准确性

#### 这个实验的真正价值

坦率地说，我们的预测头在质量上大概率不如原始模型的自回归生成。**这不重要。**

重要的是这个实验能回答一个关于 Transformer 内部机制的基础问题：

**模型的中间层 hidden states 到底"知道"未来多远？**

这个问题的答案（以逐位置准确率衰减曲线的形式呈现）有三种可能：

**情况 A：缓慢衰减**（位置 1: 70%, 位置 2: 60%, 位置 3: 52%, 位置 4: 45%）
→ 中间层确实编码了丰富的未来信息，NTP 训练的模型比我们以为的更有远见。这验证了 DMTD 和 Register MTP 的理论基础，说明"一次出多个 token"在物理上是可行的。

**情况 B：断崖式下跌**（位置 1: 70%, 位置 2: 35%, 位置 3: 18%, 位置 4: 10%）
→ 中间层只对紧邻的下一个 token 有强预测力，更远的未来信息已经高度模糊。这意味着所有多 token 方案的加速上限很低——你最多多猜 1-2 个，再多就是瞎蒙。DeepSeek 只敢用 N=2 可能不是保守，是物理极限。

**情况 C：取决于内容类型**（模板化文本衰减慢，创意文本衰减快）
→ 这最有意思。说明"未来可预测性"不是模型的固有属性，而是输入内容的函数。模板化的法律文书，后面几个 token 高度可预测；一首诗的下一行，模型自己也不知道。这和我们在投机解码那篇里讨论的"庸才加速器"效应一脉相承。

**不管哪种结果，都值一篇论文。**

━━━━━━━━━━━━━━━━━━━━

◆ 我们的评论：自回归是原罪，但原罪有它的理由

────────────────────

上面介绍了这么多"干掉自回归"的方案，最后我们退一步，聊聊为什么自回归统治了这么久。

**自回归的核心优势不是效率，是质量。**

一个 token 一个 token 地生成，意味着每一步都有完整的、精确的上下文。模型在生成第 100 个 token 时，知道前面 99 个 token 的确切值——不是"大概是这些"，是"确切就是这些"。这保证了长距离的连贯性。

所有并行方案都面临一个根本性的挑战：**你在生成第 3 个 token 的时候，第 2 个 token 还没确定。** 你只能用一个"估计"的第 2 个 token 来预测第 3 个——估计错了，后面全错。错误会累积、会传播。

自回归的"一步一步走"看起来笨，但它保证了**每一步都站在坚实的地面上**。并行方案是在猜测的地面上盖房子——盖得快，但可能歪。

这就是为什么目前所有多 token 方案的实际效果都比理论值差：理论上可以一次出 8 个 token、16 个 token，实际上超过 2-4 个质量就开始明显下降。**不是模型不够强，是并行预测的误差累积太快。**

所以现实路线很可能是混合方案：

1. 简单的、高概率的 token（"综上所述"、"如图所示"）→ 并行出，用多 token 头或投机解码
2. 关键的、低概率的 token（核心论点、技术判断）→ 回退到自回归，一步一步来

**让架构师写 `import os` 的活儿交给实习生并行干，架构师只在关键决策点亲自出手。** 这不就是我们上一篇说的投机解码的哲学吗？多 token 预测只是换了一种实现方式，哲学没变。

最终，自回归不会被"干掉"，它会被**压缩**——从"每个 token 都自回归"变成"只在需要的时候自回归"。

━━━━━━━━━━━━━━━━━━━━

◆ 结论

────────────────────

一次只吐一个 token 的自回归是大模型的原罪，但这个原罪正在被四条路线围攻：

1. **从训练改**（Meta MTP）：教模型从一开始就学会预测多个 token。效果好，但要从头来
2. **从推理改**（DMTD）：重复使用后几层循环解码。不用从头训，微调即可
3. **从输入改**（Register MTP）：插入占位 token，让标准注意力机制做多 token 预测
4. **从输出挖**（ESP）：不训练，直接从 embedding 空间推测。零成本但效果有限

这四条路线有一个共同的理论基础：**NTP 训练出来的模型，内部已经包含了远超"下一个 token"的信息。** 区别只是从哪里挖、怎么挖、挖多深。

而我们的方案——冻结前半截、在中间层接多 token 头——是试图用最低的成本回答一个最基本的问题：**这些信息到底在哪一层、有多少、能看多远？**

不管实验结果如何，这个衰减曲线本身就是对 Transformer 内部机制的一次 X 光扫描。

━━━━━━━━━━━━━━━━━━━━

技术名词速查：

- **NTP（Next-Token Prediction）**：下一个 token 预测，大模型的标准训练目标
- **MTP（Multi-Token Prediction）**：多 token 预测，训练时同时预测后续多个 token
- **DMTD（Direct Multi-Token Decoding）**：循环重用 Transformer 后几层来生成多个 token
- **Register Token**：插入输入序列的占位 token，不携带语义，用于辅助计算
- **Hidden States（隐状态）**：Transformer 中间层的内部表征，高维向量
- **Frozen Backbone（冻结骨干）**：把模型的部分参数设为不可训练，只训练新增的头部
- **LM Head（语言模型头）**：把 hidden states 映射成词表概率分布的最后一层
