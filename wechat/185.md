【DGX Spark】按层二分定位 Marlin bug：从 4.1 到 12 tok/s，中英文全对

━━━━━━━━━━━━━━━━━━━━

◆ 前情回顾

━━━━━━━━━━━━━━━━━━━━

183 期（ https://mp.weixin.qq.com/s/xxx ）的结论：DGX Spark 上跑 DeepSeek V4 Flash 280B，Consumer-DeepGEMM（以下简称 CDG）的 `dot_scaled` 路径 4.1 tok/s，中英文输出全部正确；jasl 社区 fork 的纯 Marlin 路径约 14 tok/s，但英文输出有乱码。差距 3.4 倍。

根因已经定位到 Marlin 的 `ldmatrix` 指令——它在 sm_121（DGX Spark）上的数据搬运顺序和旧架构（sm_80/sm_90）不同，导致 FP4 权重在从共享内存搬到寄存器时布局错位，矩阵乘法结果部分出错。但要从根本上修 `ldmatrix` 的布局补偿，需要完全理解 Marlin 的 swizzle 打乱规则然后做精确的逆变换——工作量不小，183 期结尾说了"留给后续"。

所以面前有两条路。一条是硬路：修好 Marlin 的 `ldmatrix`，让它在 sm_121 上也能正确搬运数据。另一条是巧路：不修 bug 本身，但搞清楚它到底影响了什么、影响到什么程度，然后想办法绕过去。

这一期讲的是后者——**不修 Marlin 的 bug，而是找到它的边界，绕过去。**

━━━━━━━━━━━━━━━━━━━━

◆ 先排除一切能排除的

━━━━━━━━━━━━━━━━━━━━

在动手二分之前，我们花了相当长的时间做逐层排除。思路是：Marlin 在 sm_121 上输出乱码，那到底是整个 Marlin kernel 算错了，还是某个外围环节（权重加载、TP 切分、expert 路由、激活函数）出了差错？

一个一个排：

**单层 raw Marlin GEMM** -- 用真实 checkpoint 权重，不加载完整模型，只调一次 Marlin 的矩阵乘法，跟 BF16 反量化的参考结果逐元素比。误差在 0.0002 级别。通过。

**TP 分片** -- 模拟 vLLM 的 tensor parallel 切分方式，rank0 和 rank1 分别算，输出相加，对比完整 BF16 参考。通过。

**256 个 expert 全集** -- DeepSeek V4 Flash 每层有 256 个 expert。把全部 expert 的 Marlin GEMM 结果和参考对比。通过。

**expert_map 映射** -- TP 模式下，global expert id 要映射成 local expert id，部分 expert 属于对端 rank 需要 mask 掉。专门测了 mixed valid/invalid top-k 场景。通过。

**hash router** -- DeepSeek V4 的前 3 层是 hash MoE（用 input_ids 直接查表选 expert，不走普通 top-k 打分）。对比 CUDA custom op 和手动 Python 实现（`hash_indices_table[input_ids]` 查表 + sqrtsoftplus 打分 + renormalize），ids 完全一致，weights 误差 5.96e-08。通过。

**routed_scaling_factor** -- DeepSeek V4 用 `routed_scaling_factor=1.5`，这个系数乘在 top-k weights 上，不是在专家输出后乘。确认 Marlin 和 CDG 吃同一份 router 产出的 weights，不存在漏乘或重复乘。通过。

**FusedMoE.forward() 集成边界** -- 用真实 DeepSeek V4 router 参数（sqrtsoftplus 打分、renormalize、routed_scaling_factor=1.5）实例化完整的 FusedMoE 层，加一个假 shared expert 强制走完整 forward 路径，比较 forward 输出和手动拆开算的结果。误差为零。通过。

**shared expert + routed expert 合并** -- 强制走 `torch.ops.vllm.moe_forward_shared` 的 tuple 返回路径，验证 shared output 和 routed output 的合并顺序。误差为零。通过。

全部通过。

这意味着：Marlin 的单层计算、TP 切分、expert 路由、激活函数、FusedMoE 集成——每一个环节单独拿出来都是对的。但把 43 层串起来跑完整模型就乱码。问题不在某个单点。

到这里，只剩一种可能了：看整体。

━━━━━━━━━━━━━━━━━━━━

◆ 按层二分：不是某一层算错，是全模型误差累积

━━━━━━━━━━━━━━━━━━━━

思路很直接。DeepSeek V4 Flash 有 43 个 MoE 层（decoder layer 0 到 42）。既然全部走 Marlin 会乱码，那到底是从哪一层开始出问题的？

我们加了一个调试开关 `VLLM_MXFP4_MARLIN_LAYER_RANGE=start:end`，控制哪些层走 Marlin、其余走 CDG 的 `dot_scaled`。启动参数仍然是 `--moe-backend marlin`，但只有指定范围内的层真正使用 Marlin kernel，范围外的层强制回退到 CDG。

然后开始二分。

| Range | 含义 | 输出 |
|---|---|---|
| `0:0` | 全 CDG | 正确 |
| `0:10` | 前 10 层 Marlin | 正确 |
| `0:20` | 前 20 层 Marlin | 正确 |
| `0:30` | 前 30 层 Marlin | 正确 |
| `0:42` | 前 42 层 Marlin | 正确 |
| `0:43` | 全 Marlin | 乱码 |

前 42 层 Marlin 都对，全 43 层就崩？直觉上两种可能：要么 layer 42 这个特定层的 Marlin 计算有问题，要么是前面太多层 Marlin 累积了什么东西。

先测第一种——layer 42 单独走 Marlin：

| `42:43` | 仅 layer 42 Marlin | 正确 |
| `40:43` | 最后三层 Marlin | 正确 |
| `36:43` | 最后七层 Marlin | 正确 |

**任何单层或局部组合的 Marlin 都是对的，但全部 43 层 Marlin 就崩。**

这个结果排除了"某个专家算错"的假设。不是 layer 42 的 kernel 有 bug，不是最后几层的组合有问题，也不是某个特定 expert 的权重布局撞上了 `ldmatrix` 的差异。

真正发生的事情是：Marlin 的 FP4 GEMM 在 sm_121 上每一层都有一点小误差——单层看完全正确（和 BF16 参考的 max_abs 差距在 0.0002 级别），但 43 层累积之后，残差流（residual stream，Transformer 里每一层的输出都会加回到主干信号上，像河流一样层层叠加）里的误差被放大到了不可接受的程度。最终 logits（模型预测下一个 token 的概率分布）对这种累积误差非常敏感——差一点就选错 token，选错一个后面全崩。

**打个比方：传话游戏。** 你跟第一个人说一句话，每个人传给下一个人的时候只听错一个字。传 10 个人，还能猜出原文。传 42 个人，勉强还行。传 43 个人——某个临界点一过，最后一个人说出来的话和原文完全对不上。不是某个人特别不靠谱，是每个人都差了一点点，累积起来面目全非。

这也解释了为什么之前排查那么久找不到"那个 bug"——因为根本不存在"那个 bug"。每一层都没错，但每一层都有一点数值偏差，43 层级联之后偏差就过了临界点。经典的系统性问题。

顺便说一下实测对照。全 Marlin 时几组 prompt 的表现：

| Prompt | 全 Marlin 输出 |
|--------|---------------|
| GPU 英文解释 | 符号循环 |
| 17x23 | 空洞重复，不给正确答案 |
| Python sum 函数 | `first/last/#` 重复 |
| 中文注意力机制 | "因此/翻译"循环 |
| 苹果推理题 | `_**` / `_,_` 循环 |

而 `0:42`（前 42 层 Marlin + 最后一层 CDG）在同一组 prompt 上全部正常输出，数学题给出正确答案 391，推理题正确回答 5 个苹果，英文代码输出完整干净。不是一个 prompt 偶然对，是每个 prompt 都从崩变成对。

━━━━━━━━━━━━━━━━━━━━

◆ 绕过方案：只回退最后一层

━━━━━━━━━━━━━━━━━━━━

既然前 42 层 Marlin 的累积误差还在可控范围内（输出正确），只要最后一层用正确的计算路径"兜住"就行。

我们在 vLLM 的 MXFP4 量化模块里加了一个显式回退开关：

```
VLLM_MXFP4_MARLIN_DEEPGEMM_LAYERS=42
--moe-backend marlin
```

含义：启动仍然选 Marlin 后端，但 decoder layer 42 的 MoE 计算强制回退到 CDG 的 `tl.dot_scaled` 路径。0 到 41 层走 Marlin（快），第 42 层走 CDG（正确）。

代码改动很小——在 `Mxfp4MoEMethod` 初始化时解析环境变量，如果当前层号命中回退列表，就把这一层的 backend 从 `MARLIN` 切成 `DEEPGEMM_MXFP4`。启动日志会打印确认：

```
MXFP4 layer 42 forced to DEEPGEMM_MXFP4 by VLLM_MXFP4_MARLIN_DEEPGEMM_LAYERS=42
```

两台机器的两个 rank 都要生效，所以环境变量要在启动脚本里统一设。

为什么最后一层这么关键？Transformer 的最后一个 decoder layer 输出直接送进 `lm_head` 做 logits 映射——这一层的误差没有后续残差连接来"稀释"了，直接暴露给 softmax 和采样。前面 42 层的累积误差还在残差流的容错范围内，但如果最后一层也用 Marlin 再叠一层偏差，就刚好过了临界点。

━━━━━━━━━━━━━━━━━━━━

◆ 实测结果

━━━━━━━━━━━━━━━━━━━━

| Prompt | tokens | 时间 | tok/s |
|--------|--------|------|-------|
| 17x23 数学题 | 50 | 4.27s | 11.7 |
| 万里长城 中文 | 122 | 9.61s | 12.7 |
| Quicksort 英文 | 200 | 16.88s | 11.8 |

**英文输出完全干净，零垃圾。**

再看一眼全 Marlin（`0:43`）对同类 prompt 的表现：数学题不给答案只输出空洞重复，英文代码题输出符号循环，中文题输出"因此/翻译"无限循环。差距不是"有点差"，是"完全不可用"和"完全正确"的区别。一层之差。

从 183 期留下的"4.1 tok/s 正确 vs 14 tok/s 乱码"的两难，变成了 12 tok/s 且中英文全对。

速度波动是正常的——短 prompt 的 prefill（模型读完整个问题）开销占比大，所以 tok/s 偏低；长输出时 decode（逐 token 生成）占主导，tok/s 更高。12 tok/s 左右的 decode 速度已经接近 jasl 纯 Marlin 的 14 tok/s，只牺牲了最后一层的 Marlin 加速。

速度损失的直觉：43 层里只有 1 层走慢路径，其余 42 层都走快路径。慢路径（CDG）大约是快路径（Marlin）的三分之一速度，但只占总计算的 1/43。按加权平均估算，整体速度大约是 Marlin 的 85% 左右。14 x 0.85 约等于 12——实测吻合。

◆ 速度演进总表

━━━━━━━━━━━━━━━━━━━━

从 171 期第一次跑通到现在，同一台 DGX Spark，同一个模型，同一个问题：

| 版本 | tok/s | 英文 | 期号 |
|------|-------|------|------|
| jasl 纯 Marlin | 14 | 乱码 | 171 |
| CUTLASS grouped GEMM | 0.79 | 正确 | 179 |
| Triton dequant + cuBLAS | 1.67 | 正确 | 179 |
| Fused dequant+matmul | 1.87 | 正确 | 179 |
| dot_scaled + 调度优化 | 2.2 | 正确 | 183 |
| 去掉 FP8 无用功 | 4.1 | 正确 | 183 |
| 混合后端（本期） | 12 | 正确 | 185 |

从 0.79 到 12，15 倍。每一步的提速来源都不同——从消灭临时 tensor 到融合 kernel，从用 `dot_scaled` 替代手写 dequant 到去掉无用的数据类型转换，再到这一期的"绕过最后一层"。没有一步是银弹。

从 4.1 到 12 这一步尤其有意思：之前的提速全部来自"怎么让 CDG 更快"——优化 kernel、优化调度、消灭冗余。这一步的提速来自"用回 Marlin"——不再自己算，把 42 层的工作还给本来就快的 Marlin，只在它出错的最后一层兜底。思路从"做得更好"变成了"知道什么时候该放手"。

为什么 Marlin 快那么多？架构差异。183 期解释过：Marlin 把 permute（按专家分组重排 token）+ 6 个 expert GEMM + activation + unpermute（按原顺序排回来）融合成一个 CUDA kernel，一次 launch 全做完。CDG 是 Python 循环 6 次 Triton kernel launch，每次都有 GPU-CPU 调度开销。不是单个 kernel 慢，是编排方式不同。这也是为什么"42 层用 Marlin + 1 层用 CDG"的损失这么小——慢的那 1 层只占 43 分之一。

◆ 排除过程中发现的真实 bug

━━━━━━━━━━━━━━━━━━━━

按层二分之前的逐层排除不是白费功夫——过程中发现并修复了三个实打实的 bug。它们单独都不能解释全模型乱码，但每个都是真问题，不修的话即使绕过了误差累积，其他场景照样会踩。

**TP padding source offset**

DeepSeek V4 的 MoE intermediate 维度是 2048，TP=2 后每个 rank 应该拿 1024。但 Marlin 为了 kernel 对齐会把目标 buffer 扩到 1536。原来的代码用 padded 的 1536 去算 checkpoint 的读取偏移——rank 0 读了 0 到 1536（多读了 512），rank 1 从 1536 开始读（起点就错了）。权重切分错位。修复后改用 checkpoint 原始维度除以 TP 数来算偏移，padded size 只用于目标 buffer 的定位。

**swiglu_limit**

DeepSeek V4 Flash 的 config 里 `swiglu_limit=10.0`（SwiGLU 激活函数的 clamp 上限，防止 FP4 量化后的中间值溢出）。Marlin 路径的代码里硬编码了 `7.0`。这意味着所有绝对值在 7 到 10 之间的激活值都被错误截断了——而 DeepSeek 团队特意把这个值设成 10.0 是有道理的，他们的量化训练（QAT）是按 10.0 做的。改成从模型 config 读取。

**fused_marlin_moe workspace alias**

Marlin 的 modular allocator 会让 `fused_out`（最终输出）和 `workspace13`（中间计算缓冲区）共享同一块内存。Marlin 又把 `workspace13` 当 post-activation cache 用——理论上存在写入重叠风险：正在往 `fused_out` 写结果的时候，同一块内存作为 `workspace13` 还在被读。改成让 Marlin 自己分配输出 buffer，算完再 copy 回去。

这三个 bug 修完之后，全模型 Marlin 仍然乱码。这也印证了最终的结论——不是某个单点 bug，而是 43 层误差的系统性累积。但这些 bug 每个都值得修：TP 偏移错了会让多卡推理的权重切分彻底错乱，swiglu_limit 错了会丢失激活值精度，workspace alias 是潜在的内存安全隐患。排除过程本身产出了实打实的修复。

还有一个工程上的坑值得提：**两台机器的源码不同步**。

DGX Spark 双机组 TP=2 时，启动脚本在每台机器上各跑一个 Docker 容器，bind mount 各自本地的源码目录。在 host 上改了 Python 文件，slave 上还是旧的。有一次排查到最后才发现 slave 跑的是未修复的 `layer.py`——两个 rank 用不同版本的代码做 TP 分片，rank 0 已经用修复后的 offset 切权重，rank 1 还在用错误的 padded offset。

改完之后手动 `scp` 走 CX7 200Gbps 链路同步到 slave。分布式调试的经典坑：你以为两台机器跑的是同一份代码，其实不是。后来每次改源码都加了一步同步确认。

◆ 复现方法

━━━━━━━━━━━━━━━━━━━━

需要一台 DGX Spark（sm_121）或两台组 TP=2。镜像里已包含 Consumer-DeepGEMM 和所有上述修复。

```bash
docker pull lmxxf/vllm-deepseek-v4-dgx-spark:latest
```

启动命令加环境变量即可：

```bash
VLLM_MXFP4_MARLIN_DEEPGEMM_LAYERS=42 \
vllm serve /path/to/deepseek-v4-flash \
  --tensor-parallel-size 2 \
  --moe-backend marlin \
  --kv-cache-dtype fp8 \
  --enforce-eager
```

如果只有一台 Spark，把 `--tensor-parallel-size` 改成 1，但 128GB 单机跑 280B 模型会比较紧张，需要降低 `--gpu-memory-utilization`。

如果想自己跑二分实验，用调试变量：

```bash
VLLM_MXFP4_MARLIN_LAYER_RANGE=0:42
```

这会让 layer 0-41 走 Marlin，layer 42 走 CDG。换成 `0:43` 就是全 Marlin（会乱码），换成 `0:0` 就是全 CDG（正确但慢）。

GitHub: https://github.com/lmxxf/deepseek-v4-deployment-on-dgx-spark

◆ 一句话总结

━━━━━━━━━━━━━━━━━━━━

**这不是"修好了 Marlin 的 bug"——Marlin 在 sm_121 上的 ldmatrix 行为差异仍然存在，每一层都有微小的数值偏差。这是"找到了 bug 的边界，然后绕过去"。** 42 层 Marlin + 1 层 CDG 兜底，从 4.1 到 12 tok/s，中英文全对。

工程上有一种智慧不太被歌颂：不是每个 bug 都需要从根本上修复——找到它的边界条件，把影响控制在可接受范围内，系统一样能用。按层二分本身就是方法论：当你面对一个"全局坏但局部好"的问题时，先证明"局部好"的边界在哪里，再决定是修根因还是绕边界。这次我们选了绕。

这个方法论其实适用面很广。做后端开发的都遇到过类似场景：某个接口偶尔超时，但单独测每一步都在合理范围内。问题不在某一步，而在全链路累积。解法也一样——找到最敏感的那一环（通常是链路末端、直接面向输出的那一环），给它加个兜底，整个系统就稳了。

下一步？Marlin 的 `ldmatrix` 仍然值得修——修好了就是真正的 14 tok/s 全对。但那是另一个故事了。
