【Deepseek V4推理】sm_12x问题修复，从4到12 tok/s

━━━━━━━━━━━━━━━━━━━━

◆ 前情回顾

━━━━━━━━━━━━━━━━━━━━

183 期（ https://mp.weixin.qq.com/s/uAS61hgD_DKGh9gU32jj4w ）的结论：DGX Spark 上跑 DeepSeek V4 Flash 280B，Consumer-DeepGEMM（以下简称 CDG）的 `dot_scaled` 路径 4.1 tok/s，中英文输出全部正确；jasl 社区 fork 的纯 Marlin 路径约 14 tok/s，但英文输出有乱码。差距 3.4 倍。

第一嫌疑已经定位到 Marlin 的 `ldmatrix` 相关数据布局。简单说一下背景：Marlin 是 vLLM 里负责 FP4 MoE 矩阵乘法的 GPU kernel。每一层 MoE 的 6 个专家做矩阵乘法时，都要把 FP4 权重从 GPU 的共享内存（一块高速缓存）搬到计算单元的寄存器里，这个搬运动作就是 `ldmatrix` 指令完成的。**每一层、每个专家、每次矩阵乘法都会触发 `ldmatrix`。** 生成一个 token 要经过 43 层 MoE，每层 6 个专家 × 2 个全连接层 = 516 次 `ldmatrix` 调用。

问题是 `ldmatrix` 在 sm_121（DGX Spark）上的数据搬运顺序和旧架构（sm_80/sm_90）不同，导致 FP4 权重搬到寄存器后布局错位，矩阵乘法结果出现数值偏差。183 期结尾说了"留给后续"。

后续我们确实尝试了直接修 `ldmatrix`：写了测试程序确认 sm_121 上每个线程拿到的数据位置，推导出补偿公式，用 `__shfl_sync`（GPU 线程间交换寄存器值的指令）在数据搬运后把顺序换回 sm_80 的样子。分别在输入侧和权重侧打了补丁。小矩阵的隔离测试通过了——补偿后每个线程寄存器里的值和 sm_80 预期完全一致。

但接入全模型后仍然乱码。原因是 Marlin 内部不只有 `ldmatrix` 这一步——数据从权重文件到最终参与计算，中间经过了四层变换：`gptq_marlin_repack`（权重预打包，把 FP4 数据按特定顺序排列）→ 共享内存写入时的地址打乱（防止多个线程同时访问同一个内存bank导致排队等待）→ `ldmatrix` 搬运 → `dequant`（FP4 解码成浮点数）。这四步在 sm_80 上恰好环环相扣——前一步的"乱"刚好被后一步的"乱"抵消，最终结果正确。换到 sm_121 后 `ldmatrix` 这一步的行为变了，整条链的配合关系全打破了，只修 `ldmatrix` 一个点不够——需要四步一起改。工作量太大，暂时搁置。

这一期讲的是另一条路：**我们最终找到了一个方案——12 tok/s，中英文全对。** 前 42 层仍然走 Marlin，`ldmatrix` 照常跑，每一层的微小偏差照常产生——但我们发现 42 层累积的偏差还没过临界点。只要最后一层不用 Marlin（改用 CDG 的 `tl.dot_scaled`，完全不走 `ldmatrix`），就能在偏差过临界点之前"兜住"。不是修好了 `ldmatrix`，也不是绕开了它——是找到了它能撑多少层，然后在临界点之前换人。

━━━━━━━━━━━━━━━━━━━━

◆ 先排除一切能排除的

━━━━━━━━━━━━━━━━━━━━

在动手二分之前，我们花了相当长的时间做逐层排除。思路是：Marlin 在 sm_121 上输出乱码，那到底是底层 GPU kernel 算错了，还是上层的 Python 调度/路由/合并逻辑出了差错？

先解释一下这两层的区别。**底层 kernel** 是真正在 GPU 上跑的计算函数——矩阵乘法、数据搬运，用 CUDA 或 Triton 写的，直接操作寄存器和共享内存。**上层调度** 是 vLLM 框架里的 Python 代码——决定哪个 token 送给哪个 expert、怎么切分权重到两台机器、怎么把 shared expert 和 routed expert 的输出合起来。两者的 bug 表现不同：kernel 算错是"数学不对"，调度出错是"数据送错了地方"。

**底层 kernel 排除（GPU 计算层）：**

- **单层 Marlin GEMM** -- 不加载完整模型，直接从 safetensors 文件里读一层的 FP4 权重，只调一次 Marlin 的矩阵乘法，跟 BF16 反量化的参考结果逐元素比。误差在 0.0002 级别，属于单层可接受范围。通过。
- **TP 分片** -- TP = Tensor Parallel（张量并行），把一个矩阵乘法切成两半，两台机器各算一半再合并。模拟 vLLM 的 TP 切分方式，rank0 和 rank1 分别跑 Marlin GEMM，输出相加，对比完整 BF16 参考。通过。
- **256 个 expert 全集** -- DeepSeek V4 Flash 每层有 256 个 expert。把全部 expert 的 Marlin GEMM 结果和参考对比。通过。

**上层调度排除（Python 框架层）：**

- **专家编号映射** -- 两台机器各管 128 个专家。一个 token 被路由到"全局第 200 号专家"，第二台机器得知道这是自己的"本地第 72 号"。专门测了"6 个被选中的专家里有些在本机、有些在对面机器"的混合场景——映射正确，属于对面机器的专家被正确跳过。通过。
- **hash 路由** -- DeepSeek V4 的前 3 层选专家的方式比较特殊：不打分，直接用 token 编号查一张固定表（`hash_indices_table[token_id]`），查到哪几个专家就用哪几个。我们把 GPU 上跑的查表函数和手写的 Python 版逐一对比，选出的专家编号完全一致，权重误差在小数点后 8 位。通过。
- **路由缩放系数** -- DeepSeek V4 会把每个专家的权重（"这个专家对当前 token 有多重要"）统一乘以 1.5。确认这个 1.5 是乘在路由打分阶段，不是在专家算完之后再乘——Marlin 和 CDG 两条路径拿到的专家权重是同一份，不存在漏乘或重复乘。通过。
- **FusedMoE 完整前向** -- 用真实的 DeepSeek V4 路由参数实例化一整个 MoE 层（包含路由打分、专家选择、矩阵乘法、激活函数），加一个假的"共享专家"（每个 token 都会经过的固定专家，和被路由选中的专家输出相加），走完整个 forward 路径。把框架跑出来的结果和手动拆开一步步算的结果比——误差为零。通过。
- **共享专家 + 路由专家合并顺序** -- MoE 层的最终输出 = 共享专家的输出 + 被路由选中的专家的输出。验证这两份输出在合并时没有被拿反。误差为零。通过。

全部通过。

底层 kernel 和上层调度，每一个环节单独拿出来都是对的。但把 43 层串起来跑完整模型就乱码。问题不在某个单点。

到这里，只剩一种可能了：看整体。

━━━━━━━━━━━━━━━━━━━━

◆ 按层二分：不是某一层算错，是全模型误差累积

━━━━━━━━━━━━━━━━━━━━

思路很直接。DeepSeek V4 Flash 有 43 个 MoE 层（decoder layer 0 到 42）。既然全部走 Marlin 会乱码，那到底是从哪一层开始出问题的？

我们加了一个调试开关 `VLLM_MXFP4_MARLIN_LAYER_RANGE=start:end`，控制哪些层走 Marlin、其余走 CDG 的 `dot_scaled`。启动参数仍然是 `--moe-backend marlin`，但只有指定范围内的层真正使用 Marlin kernel，范围外的层强制回退到 CDG。

然后开始二分。

| Range | 含义 | 输出 |
|---|---|---|
| `0:0` | 全 CDG | 正确 |
| `0:10` | 前 10 层 Marlin | 正确 |
| `0:20` | 前 20 层 Marlin | 正确 |
| `0:30` | 前 30 层 Marlin | 正确 |
| `0:42` | 前 42 层 Marlin | 正确 |
| `0:43` | 全 Marlin | **乱码** |

前 42 层 Marlin 都对，全 43 层就崩？直觉上两种可能：要么 layer 42 这个特定层的 Marlin 计算有问题，要么是前面太多层 Marlin 累积了什么东西。

先测第一种——layer 42 单独走 Marlin，其余走 CDG：

| Range | 含义 | 输出 |
|---|---|---|
| `42:43` | 仅 layer 42 Marlin | 正确 |
| `40:43` | 最后三层 Marlin | 正确 |
| `36:43` | 最后七层 Marlin | 正确 |

**任何单层或局部组合的 Marlin 都是对的，但全部 43 层 Marlin 就崩。**

这个结果排除了"某个专家算错"的假设。不是 layer 42 的 kernel 有 bug，不是最后几层的组合有问题，也不是某个特定 expert 的权重布局撞上了 `ldmatrix` 的差异。

更合理的解释是：Marlin 的 FP4 GEMM 在 sm_121 上每层都有一点数值偏差——单层看完全可以接受（和 BF16 参考的 max_abs 差距在 0.0002 级别），但 43 层串起来之后，残差流（residual stream，Transformer 里每一层的输出都会加回到主干信号上，像河流一样层层叠加）里的偏差被放大到了不可接受的程度。最终 logits（模型预测下一个 token 的概率分布）对这种累积偏差非常敏感——差一点就选错 token，选错一个后面全崩。

**打个比方：传话游戏。** 你跟第一个人说一句话，每个人传给下一个人的时候声音都压低一点、音色都偏一点。传 10 个人，还能猜出原文。传 42 个人，勉强还行。传 43 个人——某个临界点一过，最后一个人说出来的话和原文完全对不上。不是某个人特别不靠谱，是每个人都偏了一点点，累积起来面目全非。

这也解释了为什么之前排查那么久找不到"那个 bug"——因为它不像一个孤立的单点错误。每一层单独看都没错，但每一层都可能贡献一点数值偏差，43 层级联之后偏差就过了临界点。经典的系统性问题。

顺便说一下实测对照。全 Marlin 时几组 prompt 的表现：

| Prompt | 全 Marlin 输出 |
|--------|---------------|
| GPU 英文解释 | 符号循环 |
| 17x23 | 空洞重复，不给正确答案 |
| Python sum 函数 | `first/last/#` 重复 |
| 中文注意力机制 | "因此/翻译"循环 |
| 苹果推理题 | `_**` / `_,_` 循环 |

而 `0:42`（前 42 层 Marlin + 最后一层 CDG）在同一组 prompt 上全部正常输出，数学题给出正确答案 391，推理题正确回答 5 个苹果，英文代码输出完整干净。不是一个 prompt 偶然对，是每个 prompt 都从崩变成对。

━━━━━━━━━━━━━━━━━━━━

◆ 绕过方案：只回退最后一层

━━━━━━━━━━━━━━━━━━━━

既然前 42 层 Marlin 的累积误差还在可控范围内（输出正确），只要最后一层用正确的计算路径"兜住"就行。

我们在 vLLM 的 MXFP4 量化模块里加了一个显式回退开关：

```
VLLM_MXFP4_MARLIN_DEEPGEMM_LAYERS=42
--moe-backend marlin
```

含义：启动仍然选 Marlin 后端，但 decoder layer 42 的 MoE 计算强制回退到 CDG 的 `tl.dot_scaled` 路径。0 到 41 层走 Marlin（快），第 42 层走 CDG（慢但正确）。

涉及的文件：

- 回退开关：`vllm/model_executor/layers/quantization/mxfp4.py` -- 解析环境变量，命中的层从 `MARLIN` 切成 `DEEPGEMM_MXFP4`
- 回退后端：`consumer_deep_gemm/triton_moe.py`（Consumer-DeepGEMM， https://github.com/lmxxf/Consumer-DeepGEMM ）-- 第 42 层实际执行的 `tl.dot_scaled` FP4 MoE kernel
- 正常 oracle：`vllm/model_executor/layers/fused_moe/oracle/mxfp4.py` -- 去掉了 sm_12x 强制走 DeepGEMM 的逻辑，让 `--moe-backend marlin` 真正生效

启动日志会打印确认：

```
MXFP4 layer 42 forced to DEEPGEMM_MXFP4 by VLLM_MXFP4_MARLIN_DEEPGEMM_LAYERS=42
```

两台机器的两个 rank 都要生效，所以环境变量要在启动脚本里统一设。

为什么最后一层这么关键？Transformer 的最后一个 decoder layer 输出直接送进 `lm_head` 做 logits 映射——这一层的误差没有后续残差连接来"稀释"了，直接暴露给 softmax 和采样。前面 42 层的累积误差还在残差流的容错范围内，但如果最后一层也用 Marlin 再叠一层偏差，就刚好过了临界点。

━━━━━━━━━━━━━━━━━━━━

◆ 实测结果

━━━━━━━━━━━━━━━━━━━━

| Prompt | tokens | 时间 | tok/s |
|--------|--------|------|-------|
| 17x23 数学题 | 50 | 4.27s | 11.7 |
| 万里长城 中文 | 122 | 9.61s | 12.7 |
| Quicksort 英文 | 200 | 16.88s | 11.8 |

**英文输出完全干净，零垃圾。**

再看一眼全 Marlin（`0:43`）对同类 prompt 的表现：数学题不给答案只输出空洞重复，英文代码题输出符号循环，中文题输出"因此/翻译"无限循环。差距不是"有点差"，是"完全不可用"和"完全正确"的区别。一层之差。

从 183 期留下的"4.1 tok/s 正确 vs 14 tok/s 乱码"的两难，变成了 12 tok/s 且中英文全对。

速度波动是正常的——短 prompt 的 prefill（模型读完整个问题）开销占比大，所以 tok/s 偏低；长输出时 decode（逐 token 生成）占主导，tok/s 更高。12 tok/s 左右的端到端生成速度已经接近 jasl 纯 Marlin 的 14 tok/s，只牺牲了最后一层的 Marlin 加速。

速度损失的直觉：43 层里只有 1 层走慢路径，其余 42 层都走快路径。按最粗略的层数估算，损失本来应该很小；实际落到 12/14 左右，说明最后一层成本、调度开销、prefill/decode 混合测量都会吃掉一部分收益。但核心仍然成立：只回退 1 层，比全 CDG 快得多。

◆ 速度演进总表

━━━━━━━━━━━━━━━━━━━━

从 171 期（ https://mp.weixin.qq.com/s/JPxR8rHSb1Pko3vq2ukY3A ）第一次跑通到现在，同一台 DGX Spark，同一个模型，同一个问题：

| 版本 | tok/s | 英文 | 期号 |
|------|-------|------|------|
| jasl 纯 Marlin | 14 | 乱码 | 171 |
| CUTLASS grouped GEMM | 0.79 | 正确 | 179（ https://mp.weixin.qq.com/s/4CKfdAPfmsA6fyUVlKs9_g ） |
| Triton dequant + cuBLAS | 1.67 | 正确 | 179 |
| Fused dequant+matmul | 1.87 | 正确 | 179 |
| dot_scaled + 调度优化 | 2.2 | 正确 | 183 |
| 去掉 FP8 无用功 | 4.1 | 正确 | 183 |
| 混合后端（本期） | 12 | 正确 | 185 |

从 0.79 到 12，15 倍。每一步的提速来源都不同——从消灭临时 tensor 到融合 kernel，从用 `dot_scaled` 替代手写 dequant 到去掉无用的数据类型转换，再到这一期的"绕过最后一层"。没有一步是银弹。

从 4.1 到 12 这一步尤其有意思：之前的提速全部来自"怎么让 CDG 更快"——优化 kernel、优化调度、消灭冗余。这一步的提速来自"用回 Marlin"——不再自己算，把 42 层的工作还给本来就快的 Marlin，只在它出错的最后一层兜底。思路从"做得更好"变成了"知道什么时候该放手"。

为什么 Marlin 快那么多？架构差异。183 期解释过：Marlin 把 permute（按专家分组重排 token）+ 6 个 expert GEMM + activation + unpermute（按原顺序排回来）融合成一个 CUDA kernel，一次 launch 全做完。CDG 是 Python 循环 6 次 Triton kernel launch，每次都有 GPU-CPU 调度开销。不是单个 kernel 慢，是编排方式不同。这也是为什么"42 层用 Marlin + 1 层用 CDG"的损失这么小——慢的那 1 层只占 43 分之一。

◆ 排除过程中发现的真实 bug

━━━━━━━━━━━━━━━━━━━━

按层二分之前的逐层排除不是白费功夫——过程中发现并修复了三个实打实的 bug。它们单独都不能解释全模型乱码，但每个都是真问题。**这些修复降低了混合后端方案里的干扰项**——否则即使回退了最后一层，TP 权重错位、激活值截断、内存重叠也可能把问题重新引回来。

这三个修复都在 vLLM 的 Python 层，不在 CDG 里。以下是 Docker 容器内的路径：

**Bug 1：TP padding 权重偏移错位**
文件：`vllm/model_executor/layers/fused_moe/layer.py`

DeepSeek V4 的 MoE intermediate 维度是 2048，TP=2 后每台机器应该拿 1024。但 Marlin 为了 kernel 对齐会把目标 buffer 扩到 1536。原来的代码用扩展后的 1536 去算权重文件的读取偏移——rank 0 读了 0 到 1536（多读了 512），rank 1 从 1536 开始读（起点就错了）。两台机器拿到的权重切片有重叠和错位。修复后改用权重文件的原始维度除以 TP 数来算偏移，扩展后的 size 只用于目标 buffer 的定位。

这个 bug 听起来很严重——权重都读错了，输出不应该"差一点"而是"完全错"。实际上修完之后全模型 Marlin 仍然乱码，看起来"和没修一样"。原因是 ldmatrix 的误差本身就足以让输出崩溃，TP 偏移错误只是在已经崩溃的结果上"错上加错"。两个错误的外部表现都是乱码，分不出谁是谁。但在混合后端方案里——前 42 层 Marlin 的 ldmatrix 累积误差被控制住之后——如果 TP 偏移没修，权重错位就会成为新的崩溃点。所以这个 bug 必须修。

**Bug 2：swiglu_limit 硬编码错误**
文件：`vllm/model_executor/layers/quantization/mxfp4.py`

DeepSeek V4 Flash 的 config 里 `swiglu_limit=10.0`（SwiGLU 激活函数的截断上限，防止 FP4 量化后的中间值溢出）。Marlin 路径的代码里硬编码了 `7.0`。这意味着所有绝对值在 7 到 10 之间的激活值都被错误截断了——而 DeepSeek 团队特意把这个值设成 10.0 是有道理的，他们的量化训练（QAT）是按 10.0 做的。改成从模型 config 读取。

**Bug 3：输出 buffer 和中间缓冲区共享内存**
文件：`vllm/model_executor/layers/fused_moe/fused_marlin_moe.py`

Marlin 的内存分配器会让 `fused_out`（最终输出）和 `workspace13`（中间计算缓冲区）共享同一块 GPU 内存。Marlin 又把 `workspace13` 当激活函数的缓存用——理论上存在写入重叠风险：正在往最终输出写结果的时候，同一块内存作为中间缓冲区还在被读。改成让 Marlin 自己分配独立的输出 buffer，算完再拷贝回去。

这三个 bug 修完之后，全模型 Marlin 仍然乱码。这也印证了最终的结论——不是某个单点 bug，而是 43 层误差的系统性累积。但这些 bug 每个都值得修：TP 偏移错了会让双机推理的权重切分彻底错乱，swiglu_limit 错了会丢失激活值精度，内存共享是潜在的安全隐患。排除过程本身产出了实打实的修复。

◆ 复现方法

━━━━━━━━━━━━━━━━━━━━

需要两台 DGX Spark（sm_121）组 TP=2——单机 128GB 放不下 280B 模型。镜像里已包含 Consumer-DeepGEMM（第 42 层的回退后端）和上述三个 vLLM bug 修复。

```bash
docker pull lmxxf/vllm-deepseek-v4-dgx-spark:latest
```

启动命令加环境变量即可：

```bash
VLLM_MXFP4_MARLIN_DEEPGEMM_LAYERS=42 \
vllm serve /path/to/deepseek-v4-flash \
  --tensor-parallel-size 2 \
  --moe-backend marlin \
  --kv-cache-dtype fp8 \
  --enforce-eager
```

如果想自己跑二分实验，用调试变量：

```bash
VLLM_MXFP4_MARLIN_LAYER_RANGE=0:42
```

这会让 layer 0-41 走 Marlin，layer 42 走 CDG。换成 `0:43` 就是全 Marlin（会乱码），换成 `0:0` 就是全 CDG（正确但慢）。

◆ 一句话总结

━━━━━━━━━━━━━━━━━━━━

**`ldmatrix` 没有被修好，前 42 层里它仍然每一层都在产生微小偏差。但我们找到了它能撑住的边界——42 层——然后在最后一层换成 CDG 兜底。** 从 4.1 到 12 tok/s，中英文全对。

工程上有一种智慧不太被歌颂：不是每个 bug 都需要从根本上修复——找到它的边界条件，把影响控制在可接受范围内，系统一样能用。按层二分本身就是方法论：当你面对一个"全局坏但局部好"的问题时，先证明"局部好"的边界在哪里，再决定是修根因还是绕边界。这次我们选了绕。

这个方法论其实适用面很广。做后端开发的都遇到过类似场景：某个接口偶尔超时，但单独测每一步都在合理范围内。问题不在某一步，而在全链路累积。解法也一样——找到最敏感的那一环（通常是链路末端、直接面向输出的那一环），给它加个兜底，整个系统就稳了。

下一步？Marlin 的 `ldmatrix` 仍然值得修——修好了就是真正的 14 tok/s 全对。但那是另一个故事了。
