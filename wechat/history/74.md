【AI小知识】谷歌论文说了，Ctrl+C Ctrl+V，AI 智商翻 4 倍

"重要的事情说三遍"，原来是有科学依据的。

━━━━━━━━━━━━━━━━━━━━

◆ 发生了什么

━━━━━━━━━━━━━━━━━━━━

2025 年 12 月，Google Research 发了一篇论文，标题很学术：

《Prompt Repetition Improves Non-Reasoning LLMs》

翻译成人话：「把指令说两遍，AI 就能变聪明」。

有多离谱？

他们用 Gemini 2.0 Flash-Lite 做实验，任务很简单：从一堆文字里找特定的名字（类似你在日志里 grep 一个报错信息）。

• 说一遍：准确率 「21%」。基本是瞎子。
• 复制粘贴再说一遍：准确率飙到 「97%」。

「按一下 Ctrl+C Ctrl+V，智商翻 4 倍。」

而且不只是 Gemini——Google 还测了竞争对手：

| 模型 | 说一遍 | 重复指令一遍 |
|:--|:--|:--|
| Gemini 2.0 Flash | 较低 | 显著提升 |
| Gemini 2.0 Flash-Lite | 较低 | 显著提升 |
| GPT-4o | 较低 | 显著提升 |
| GPT-4o-mini | 较低 | 显著提升 |
| Claude Sonnet | 较低 | 显著提升 |
| DeepSeek V3 | 较低 | 显著提升 |

（论文实验是 2025 年做的，上面的模型型号是当时的。但原理对所有 Transformer 模型通用——Softmax 注意力稀释是架构层面的问题，不分品牌和版本。你现在用的 GPT-5.4、Gemini 3.1、Claude Opus 4.6，机制完全一样。）

7 个模型 × 7 个基准测试（ARC、OpenBookQA、GSM8K、MATH、MMLU-Pro，外加两个自研测试 NameIndex 和 MiddleMatch），一共 70 个组合。

结果：重复指令「**胜 47 次，败 0 次，平 23 次**」。

败 0 次。**零**。没有一个组合因为重复指令而变差。

不管你用哪家 AI，不管是数学题（GSM8K、MATH）、常识推理（ARC、OpenBookQA）还是知识检索（MMLU-Pro），重复指令都管用。这不是 bug，是 feature。

而且不会变慢——重复指令不增加输出 token 数量（答案长度不变），不增加延迟（重复部分在 prefill 阶段并行处理），可以直接 drop-in 到现有系统。

「免费的午餐。」

━━━━━━━━━━━━━━━━━━━━

◆ 实操：三种写法对比

━━━━━━━━━━━━━━━━━━━━

假设你有一份 5000 行的服务器日志，想找出所有包含 `ERROR` 的行里提到的用户 ID。

────────────────────
【写法 A：指令在前面（最常见，也最容易翻车）】

```
请找出所有包含 ERROR 的行，并提取其中的用户 ID（格式为 uid=xxx）。

[粘贴 5000 行日志]
```

指令在开头，被 5000 行日志淹没，中间部分的 ERROR 大概率被漏掉。

────────────────────
【写法 B：指令挪到末尾（老手的直觉）】

```
[粘贴 5000 行日志]

以上是服务器日志，请找出所有包含 ERROR 的行，
并提取其中的用户 ID（格式为 uid=xxx）。
```

有经验的人通常这么干——先贴内容，再说要求。利用的就是结尾的位置优势。比写法 A 好很多。

────────────────────
【写法 C：前后都说一遍（论文推荐，最优解）】

```
请找出所有包含 ERROR 的行，并提取其中的用户 ID（格式为 uid=xxx）。

[粘贴 5000 行日志]

提醒：请找出上面日志中所有包含 ERROR 的行，
并提取其中的用户 ID（格式为 uid=xxx）。不要遗漏。
```

开头一遍 + 末尾一遍 = 双重锚点。

注意：**只重复指令，不重复内容。** 指令通常就几十个 token，重复一遍的成本几乎为零。如果你把 5000 行日志也贴两遍——那就不是免费午餐了，是自费大餐 😂

━━━━━━━━━━━━━━━━━━━━

◆ 为什么有效

━━━━━━━━━━━━━━━━━━━━

【Softmax 的零和博弈】

大模型每生成一个字，都要先"看一遍"前面所有的文字，给每一段分配一个注意力权重。

关键来了：**这些权重加起来，必须严格等于 1。**

这是 Softmax 函数的铁律：

  Softmax(xᵢ) = exp(xᵢ) / Σ exp(xⱼ)

分母是所有 Token 的指数之和。Token 越多，分母越大，每个 Token 分到的份额就越小。

用数字感受一下：

```
你的指令：20 个 Token
日志内容：100 个 Token
→ 指令的注意力份额 ≈ 20/120 ≈ 16.7%

你的指令：20 个 Token
日志内容：10,000 个 Token
→ 指令的注意力份额 ≈ 20/10,020 ≈ 0.2%
```

同样的 20 个 Token 的指令，上下文从 100 膨胀到 10,000，注意力份额从 16.7% 暴跌到 0.2%。**缩水了 83 倍。**

「这不是走神，是通货膨胀。」

你的指令没变，但周围的"噪音 Token"越来越多，把你的权重物理性地稀释了。这也解释了为什么短对话 AI 很聪明，长对话越聊越傻——不是模型变笨了，是你的指令被淹没了。

────────────────────
【Lost in the Middle：中间最惨】

更糟的是，注意力的分配不是均匀的。

2023 年的经典论文（Lost in the Middle, Liu et al.）已经证实：AI 对开头和结尾的注意力最强，中间最弱。画成图就是个 U 型曲线——两头高，中间塌。

原因很直觉：开头的 Token 被所有后续 Token "看过"无数遍（累积曝光最多），结尾的 Token 离输出最近（位置优势最大）。夹在中间的？两头不靠，注意力最薄。

所以你塞了一万行日志，开头那句"请帮我提取报错信息"——它不是被忘了，是被挤到了 U 型谷底。

────────────────────
【重复指令 = 在终点线重新砸一个锚】

现在逻辑链就清楚了：

上下文越长 → Softmax 稀释越严重 → 开头指令沉底 → 而结尾天然有位置优势。

当你把指令复制粘贴到末尾：

1. 「位置优势」：末尾 Token 离输出最近，注意力权重天然最高
2. 「信号叠加」：同样的语义出现两次，相关 Token 的权重加倍
3. 「免费」：重复部分在 prefill 阶段并行处理，不增加延迟

「不是给 AI 一个后视镜——是直接把目的地搬到它脚下。」

────────────────────
【开推理模式呢？】

论文还测了一个重要的对照：如果开了推理模式（比如 GPT-5.4、DeepSeek R1 那种 CoT 推理），重复指令还有用吗？

答案：**中性到略正。** 不会更差，但提升不明显。

为什么？因为推理模式本身就会在内部"重新组织"问题——模型会先理解题意，再分步推理。这个"重新组织"的过程本身就起到了类似"重复指令"的效果。

所以：
- **不开推理**（普通模式）：重复指令是免费午餐，必须用
- **开了推理**（GPT-5.4/R1 模式）：不需要重复，推理过程自带这个效果

◆ "重要的事情说三遍"原来是真的

━━━━━━━━━━━━━━━━━━━━

产品经理提需求，如果只在群里说一句，你大概率当耳旁风。

只有当他：
• 群里 @ 你一遍
• Jira 上提个单子
• 跑到你工位上拍拍你的肩膀

三层信号叠加，你才觉得"哦，这事儿好像得干"。

AI 也一样。几千亿参数、上万维的空间里，你的一句指令就像扔进大海的一颗石子。多扔一颗，波纹共振，它才能在茫茫的参数海洋里捞出你想要的那根针。

以前觉得 AI 是神，全知全能。现在 Google 这篇论文告诉你：「AI 也是草台班子。」它也会走神，也会健忘，也需要你把话说两遍才能听进去。

但这反倒让我觉得 AI 更像人了：

• 那个需要你一遍遍强调需求才干活的 AI，像不像你的下属？
• 那个你说了一遍就左耳进右耳出的 AI，像不像开会时的你？

◆ 实用建议

━━━━━━━━━━━━━━━━━━━━

以后用 AI 写代码或查资料，如果它一本正经地胡说八道，别急着骂它人工智障。

「试试把你的问题复制一遍，贴在后面，再发给它。」

具体来说：

```
场景 1：长文档分析
  → 把"请从中提取 XXX"这句话，在文档末尾再贴一遍

场景 2：多轮对话越聊越傻
  → 把最初的需求描述，在新一轮提问时重新贴一遍

场景 3：RAG 检索（从知识库提取信息回答问题）
  → 检索到的文本塞进 context 后，在末尾重复一遍原始问题

场景 4：Agent / 工具调用
  → 工具返回结果后，在结果末尾重复一遍任务目标
```

注意：这个技巧对**不开推理模式的普通对话**效果最大。如果你用的是 GPT-5.4 或 DeepSeek R1 这类推理模式，推理过程本身已经有类似效果，重复不重复差别不大。

（个人习惯：我长期关闭思维链，所以"重复指令"对我来说是刚需。如果你也是不开推理模式直接用的人，这个技巧的收益更大。）

────────────────────
【进阶：让 AI 重读自己的输出】

除了复制指令，还有一个个人心得：**让 AI 写完之后，再读一遍自己写的东西。**

比如你让 AI 写了一篇长文，写完之后追一句："把文章再读一遍，看看有什么逻辑不通顺的。"

这和论文说的"重复指令"原理相关但不完全一样：

```
Ctrl+C Ctrl+V（论文方法）：
  同一次生成里多放一个锚点
  → 成本：几乎为零
  → 效果：对抗注意力稀释，"多看一眼"

"再读一遍"（实战心得）：
  触发第二次完整的前向传播
  → 成本：多花一轮对话的 token
  → 效果：用全新的注意力重新扫全文 + 审查自己的输出
```

第二次读的时候，上下文里已经有了 AI 自己第一次的输出——它不只是在重新理解你的指令，还在**审查自己刚才写的东西**。这比单纯重复指令多了一层自我纠错。

两个技巧可以组合用：写长文时先用重复指令保证第一遍质量，写完再让它重读一遍查漏补缺。

◆ 总结

━━━━━━━━━━━━━━━━━━━━

```
现象：
  重复指令一遍 → 准确率大幅提升（21% → 97%）
  70 个测试组合：胜 47，败 0，平 23

原理：
  Softmax 的零和博弈 → 上下文越长，指令权重越被稀释
  Lost in the Middle → 中间最惨，两头最强
  末尾重复 = 在终点线重新砸一个锚

适用范围：
  普通模式（不开推理）→ 免费午餐，必须用
  推理模式（GPT-5.4/R1）→ 效果中性，可以不用

实操：
  Ctrl+C Ctrl+V，把指令贴在末尾，完事
  写完让 AI 重读一遍，自我纠错
```

毕竟在这个 996 的世界里，连机器都需要被确认两次，才能确信自己听懂了你的心声。
