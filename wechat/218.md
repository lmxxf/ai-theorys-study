【多模态】为什么 DeepSeek、GLM 不是原生多模态？

一个有意思的现象：以编程能力见长的几个大模型——DeepSeek V4、GLM-5——主线模型都是**纯文本**的。想识图？另外给你一个独立的视觉模型线，或者后挂一个"识图模式"。

反过来，真正从头到尾把图像、音频、视频和文本混在一起训练的"原生多模态"，掰着指头数，确凿的只有 Gemini 一家。

这不是巧合，也不是谁家技术不行。这篇我们先把几家的技术文档摆出来看事实，再用几篇论文讲清楚背后的机制：**多模态不是免费的加法，它有价签。**

━━━━━━━━━━━━━━━━━━━━

◆ 先认人：几家旗舰到底是不是多模态

━━━━━━━━━━━━━━━━━━━━

【DeepSeek V4：模型卡白纸黑字写着 Text】

DeepSeek V4 的官方模型卡（2026 年 4 月 27 日版）里，Modalities 一栏只写了一行：

> "Text, three reasoning modes available"

没有 image，没有 vision。**主线模型是纯文本，官方自己钉死的。**

那最近上线的"识图模式"是什么？今年 5 月初，DeepSeek 短暂放出过一篇论文《Thinking with Visual Primitives》（之后又撤掉了，存档还能找到）。论文里写得很清楚：

> "The language backbone is instantiated with DeepSeek-V4-Flash, a Mixture-of-Experts (MoE) model comprising 284B total parameters and 13B active parameters during inference."

> "For visual encoding, we employ DeepSeek-ViT, an in-house ViT trained from scratch that supports arbitrary-resolution inputs."

翻译成人话：拿训练好的纯文本 V4-Flash 当底座，外面接一个自研的视觉编码器（ViT），图像切成 14×14 的小块编码、再做 3×3 空间压缩，变成一串"视觉 token"喂给文本模型。

💡 ViT（Vision Transformer）：把图像切成小方块，每块当成一个"词"来编码的视觉模型。后挂式多模态的标准配件——它负责把图像翻译成文本模型能吃的 token，文本主干本身不用动。

这是典型的**后挂式**架构：文本大脑训练完了，再给它装一只外接的眼睛。眼睛看到的东西先翻译成"文本模型的母语"再送进去。不是原生多模态。

【GLM-5：官方文档归在 text 线】

智谱的官方文档里，GLM-5 明确归在文本模型线。视觉能力走的是独立模型 GLM-5V-Turbo，它有自己的技术报告（arXiv:2604.26752），里面的表述是：

> 以 GLM-5-Turbo 为 "language-only base model"，再接上自研的视觉编码器 CogViT。

同样的配方：**纯文本底座 + 视觉编码器。** 只不过智谱给这条视觉线投入不小——CogViT 用了 SigLIP2 和 DINOv3 双教师蒸馏，还配了对比学习对齐。但架构路线没变：先有文本大脑，再装眼睛。

【Kimi K2.5："原生"这个词正在被稀释】

月之暗面的 K2.5 技术报告（arXiv:2602.02276）里自称 "native multimodal"。但往下读两段，定义就出来了：

> "The foundation of Kimi K2.5 is Kimi K2, a trillion-parameter mixture-of-experts (MoE) transformer model pre-trained on 15 trillion high-quality text tokens."

> "Kimi K2.5 is a native multimodal model built upon Kimi K2 through large-scale joint pre-training on approximately 15 trillion mixed visual and text tokens."

看清楚这个顺序：**先用 15T 纯文本 token 把 K2 训练完，再用 15T 图文混合 token 继续训练。** 视觉确实深度参与了预训练量级的训练（这远超后挂式），但起点是一个纯文本底座。

这算"半原生"：比后挂 ViT 走得远得多，但和"从第一个 token 开始就图文交错"还是两回事。

【Gemini：唯一确凿的从零原生】

Gemini 的技术报告（arXiv:2312.11805）写明：从一开始就在文本、图像、音频、视频的交错（interleaved）序列上预训练，还能原生输出图像 token。**按最严格的定义，全行业确凿做到"从零原生多模态"的，目前就这一家。**

GPT-4o / GPT-5 系列官方也宣称是统一的端到端多模态架构，但训练细节不透明，姑且算第二家。

至于 Claude——Anthropic 从不公开架构细节。社区有推测认为它是后挂视觉编码器（佐证：不支持音频输入、不原生生成图像、视觉是 Claude 3 才加上的——这三条都属实）。注意这只是**推测**，不是官方说法。

汇总一下：

| 模型 | 路线 | 出处 |
|----|----|----|
| Gemini | 从零原生（interleaved 预训练） | 官方技术报告 |
| GPT-4o/5 | 宣称统一架构，细节不明 | 官方博客 |
| Kimi K2.5 | 半原生（文本底座 + 15T 图文续训） | 官方技术报告 |
| GLM-5 | 纯文本，视觉走独立线（GLM-5V） | 官方文档 + 技术报告 |
| DeepSeek V4 | 纯文本，识图是后挂 ViT | 官方模型卡 + 撤稿论文 |
| Claude | 推测后挂（官方保密） | 社区推测 |

**编程评测越靠前的模型，越往纯文本那头扎堆。** 为什么？

在算账之前，得先回答一个更基本的问题：文本模型的目标函数是"预测下一个字"，那图像、音频混进来之后，模型到底在预测什么？

━━━━━━━━━━━━━━━━━━━━

◆ 插一节：原生多模态的目标函数是啥

━━━━━━━━━━━━━━━━━━━━

答案出乎意料地朴素：**主流路线不是发明新目标函数，而是把图像和音频也改造成离散 token，然后照旧预测下一个 token。**

【图像怎么变成"字"】

先单独训一个图像压缩器（VQ-VAE / VQGAN 一族）：encoder 把图像压成一格一格的连续向量，然后查一本"码本"——比如 8192 个可学习的向量——每格用**最近的码本条目的编号**替代。一张 256×256 的图，就变成了 32×32 = 1024 个整数，每个整数都在 0 到 8191 之间。

💡 码本（codebook）：一张可学习的"新华字典"，里面每个条目是一个向量。图像的每一小块不存原始像素，只存"字典里第几号条目和我最像"。图像就这样被翻译成了一串"字"，词表大小 8192。

这下图像和文本就是同一种东西了。把图像 token 和文本 token 拼进同一个序列，目标函数完全不变：**交叉熵，预测下一个 token。** Gemini 技术报告说能"原生输出图像 token"，指的就是这个——生成图像和生成文本是同一个 softmax 在干活，只是吐出来的编号要拿去查图像码本、再还原成像素。

【音频同理，而且更成熟】

神经音频编解码器（SoundStream、EnCodec 这类）把波形压成离散 token，常用残差量化：第一层码本量化一次，剩下的残差再用第二层码本量化，叠几层。1 秒音频变成几十到几百个整数。之后还是那句话——预测下一个 token。

所以原生多模态的统一目标函数可以写成一行：

L = −Σ log P(xₜ | x₁...xₜ₋₁)

其中 xₜ 可能是一个汉字、一个图像块编号、或一个音频帧编号——**模型根本不区分，它眼里全是 token。**

【两个例外，正好对应两条路线】

一是**只输入、不输出的视觉**——大部分后挂式 VLM 都是这样：图像 token 上根本**不算 loss**，只当作条件信息，loss 只算在文本 token 上。"看图说话"里只有"说话"部分挨梯度。视觉编码器自己的训练另算，通常用对比学习（CLIP/SigLIP 那套：配对的图文在向量空间里拉近，不配对的推远）。前面 DeepSeek、GLM 的后挂路线，走的就是这条。

二是**扩散路线**。论图像生成质量，预测离散 token 长期打不过扩散模型（它的 loss 是预测加进去的噪声，本质是个回归任务）。所以市面上有些"统一模型"其实是缝合的：理解走自回归，生成另挂一个扩散头，两个目标函数并存。

【一个藏在细节里的别扭】

文本的"下一个字"有强因果约束——语法、逻辑、指代，前文真的决定后文。但图像那 32×32 个 token 是**按从左到右、从上到下的光栅顺序硬展平的**：左上角的块"预测"它右边的块，这个"因果关系"是人为强加的——像素之间本来是空间关系，不是时序关系。这也是自回归生图长期吃亏的原因之一：扩散模型全图一起去噪，不用假装图像有阅读顺序。

理解了"万物皆 token"，下面的账就好算了——**这既是原生多模态的入场券，也是税单本身**：一张图被量化成上千个"字"塞进序列，每个都和文本 token 抢同一个 softmax、同一份注意力、同一组权重的梯度。目标函数统一了，账单也统一了。

━━━━━━━━━━━━━━━━━━━━

◆ 第一笔账：遗忘税——学了看图，忘了算数

━━━━━━━━━━━━━━━━━━━━

先说机制，再说论文。

神经网络的训练，本质是对同一组权重反复做梯度更新：算出当前任务的损失（loss），对每个参数求导，沿着让 loss 下降的方向挪一小步。

问题来了：**文本任务和视觉任务，对同一个参数的"理想值"可能不一样。** 你拿图文数据训练时，梯度把参数往"适合处理视觉"的方向拉；而这个参数原来的位置，可能正好是"做多步数学推理"的关键配置。拉走了，数学就掉了。

这不是猜想，是有实测的。

【论文一：多模态微调的遗忘是实锤】

Zhai et al.《Investigating the Catastrophic Forgetting in Multimodal Large Language Model Fine-Tuning》（CPAL 2024）第一个把这件事测成了实锤——有意思的是，它测出的遗忘发生在视觉侧：做完视觉指令微调的多模态模型，在基础图像分类任务上反而**不如自己内置的那个视觉编码器**，还会产生幻觉。微调的梯度连"自家带进门"的能力都能冲掉——遗忘不挑方向，谁的权重被借走谁遭殃。

💡 灾难性遗忘（catastrophic forgetting）：神经网络学新任务时，把旧任务的能力覆盖掉了。不是"记忆满了"，是新任务的梯度把旧任务依赖的权重配置冲掉了。

【论文二：文本侧的账——学了看图，算数真掉】

另一篇《Training-Free Mitigation of Language Reasoning Degradation After Multimodal Instruction Tuning》（arXiv:2412.03467）把账算到了文本侧：LLaVA 系列模型做完多模态指令微调后，GSM8K（小学数学应用题基准）这类语言推理任务成绩下降——开头那句"学了看图，忘了算数"，就是它实测出来的。

但更扎心的是这笔税**不可预测**：同一套多模态微调，有的语言底座掉得明显，有的几乎不掉甚至略升——事先没人能保证你最值钱的那部分能力不在受灾名单上。对一个把"编程/推理"当招牌的旗舰模型来说，这就是一笔无法事先封顶的风险敞口。DeepSeek 和 GLM 不是做不了多模态（他们的独立视觉线都做得不错），是不愿意让旗舰冒这个险。

━━━━━━━━━━━━━━━━━━━━

◆ 第二笔账：梯度冲突——两个老师往两个方向拽

━━━━━━━━━━━━━━━━━━━━

遗忘税讲的是"后果"，这一节讲"为什么会这样"。

联合训练多模态模型时，同一个 batch 里既有文本数据又有图像数据，两边各算各的 loss，各产生各的梯度，最后加在一起更新同一组权重。

用公式说：参数 θ 收到的更新方向是 g = g_text + g_vision。如果这两个梯度向量的夹角小（方向大致一致），合力健康；**如果夹角接近 180 度（点积为负），两个任务就在拔河**——文本想把 θ 往东拉，视觉想往西拉，合力要么互相抵消（两边都学不好），要么数据多的一方碾压数据少的一方（弱势模态被牺牲）。

《Mitigating Intra- and Inter-modal Forgetting in Continual Learning of Unified Multimodal Models》（arXiv:2512.03125）把统一多模态模型的能力退化明确归因到这个机制上：**模态间梯度冲突**（inter-modal gradient conflict）。这不是工程没做好，是优化问题的本性——只要两个任务共享参数，就存在冲突的可能；模态差异越大（文本的离散符号 vs 图像的连续像素统计），冲突越容易发生。

回头看 Kimi 的选择就特别有意思了。K2.5 报告里有一句关键结论：

> "Early fusion with a lower vision ratio yields better results given a fixed total vision-text token budget."

注意 "fixed total budget"（固定总预算）这几个字——**这就是官方亲口承认账本存在**：视觉 token 和文本 token 在抢同一笔训练预算，视觉比例放低反而效果更好。而"先焊死文本骨架、再低比例图文续训"这条路线，本身就是对梯度冲突的工程规避：文本能力先在没有干扰的环境里收敛到位，视觉再以不掀桌子的强度慢慢融入。

GLM-5V 的"language-only base model + 视觉编码器"是同款思路的更保守版本。**中国这几家不约而同选了"先文本后视觉"，不是巧合，是对同一笔账算出了同一个答案。**

━━━━━━━━━━━━━━━━━━━━

◆ 第三笔账：容量税——一张图吃掉半个上下文

━━━━━━━━━━━━━━━━━━━━

前两笔税收在训练阶段，这笔收在推理阶段。

图像变成 token 之后有多占地方？LLaVA-NeXT（即 LLaVA-1.6）官方博客给过一个数：处理一张 672×672 的图，要切成 5 个 336×336 的网格、每格 576 个 token，共 **2880 个视觉 token**。当年那一代模型的上下文窗口普遍是 4096——**一张图吃掉七成。**

而且这些 token 不是白放着的。Transformer 的注意力计算量随序列长度平方增长，KV cache 显存随长度线性增长——视觉 token 进了序列，每一层每一步都在持续收费。所以"压缩视觉 token"本身卷成了一个研究方向：《Efficient Large Multi-modal Models via Visual Context Compression》（arXiv:2406.20092）专门研究怎么把 LLaVA 的视觉上下文压小，2025 年更有篇综述《A Survey of Token Compression for Efficient Multimodal Large Language Models》（arXiv:2507.20198）把图像/视频/音频三个模态的压缩方法系统梳理了一遍——一个方向能卷出一篇综述，恰恰说明这笔税有多痛。

DeepSeek 自己也在这个方向上交过卷子：DeepSeek-OCR 那篇论文（arXiv:2510.18234）实测，用视觉 token 压缩文本，压缩比 10 倍以内时还原精度 97%。他们对"视觉 token 很贵、必须狠狠压缩"这件事，认识得比谁都清楚。前面提到 V4 识图模式给 ViT 输出做 3×3 空间压缩，也是同一个思路：**眼睛可以装，但不能让眼睛把脑子的工作台占满。**

━━━━━━━━━━━━━━━━━━━━

◆ 算总账：这是站位选择，不是能力高低

━━━━━━━━━━━━━━━━━━━━

把三笔账放在一起看：

| 账目 | 收税环节 | 机制 |
|----|----|----|
| 遗忘税 | 训练 | 新任务梯度冲掉原有权重配置，税额不可预测 |
| 梯度冲突 | 训练 | 两个任务对同一组参数拔河 |
| 容量税 | 推理 | 视觉 token 挤占上下文和算力 |

同样的参数量、同样的训练预算，是全花在"文本-推理-代码"这一条窄而深的路上，还是分出一部分去拓宽输入侧、容纳视觉？这是一道资源分配题，不是技术先进性排名。

Gemini 选择从零原生，换来的是真正统一的多模态理解——图像不需要先翻译成"文本的母语"，它和文本住在同一个表征空间里。代价就是上面三笔税照单全收。

这笔税在榜单上看得见。2026 年年中的格局：美国御三家里，Gemini 在**推理+编程+agent** 这条最受关注的主轴上排在最后——SWE-Bench Verified 上 Gemini 3.1 Pro 约 80.6%，而 GPT-5.5 和 Claude Opus 4.8 按厂商自报口径都在 88% 以上（第三方复测的绝对分数普遍更低，但 Gemini 垫底的相对排序不变）；编程助手市场上 Gemini 的存在感同样薄弱——JetBrains 2026 年 4 月的开发者调查里，领跑的是 Copilot、Claude Code、Cursor，Gemini 系工具份额不到一成。但有意思的是另一半：**Gemini 在多模态理解、科学问答这些维度上反而领先（MMMU 系列、LMArena 总榜），前端生成上也能互有胜负。** 弱的恰好是纯符号推理（税收走的地方），强的恰好是视觉相关（税买回来的东西）——**一张模型卡上同时印着税单和收据。** 这不是 Google 技术不行，是它替全行业蹚了"从零原生"这条最贵的路。

下面这句话给国产模型说句公道话：DeepSeek 和 GLM 在 SWE-Bench 上能跟美国旗舰掰手腕，靠的正是把全部预算押在符号推理这一条路上。**纯文本不是落后，是聚焦。**

DeepSeek 和 GLM 选择主线纯文本，把推理和代码能力推到极限，视觉另起一行——等于把税单拆开，让旗舰一分钱不交。

Kimi 选了中间解：先焊骨架再开眼，半价交税。

**编程恰好是最纯粹的符号任务**：长因果链、严格语法、零视觉依赖。一个把编程当核心卖点的模型，多模态化对它是纯支出、几乎零收入——这就是为什么"编程模型"和"纯文本"高度重合。不是不会，是不划算。

下次再看到"某某模型为什么不支持识图"的讨论，可以换个问法：**它的招牌能力，付得起多模态的税吗？**
