【AI可解释性】用JSON向AI提问效果更好？真的假的？

◆ 开篇：一个刷视频刷来的偏方

━━━━━━━━━━━━━━━━━━━━

前一阵刷视频，看到一些 up 主在推荐一个提示词偏方：**跟 AI 提问别写大白话，把问题组织成 JSON 格式发过去，回答质量会明显变好**。英文圈也有同款，Medium 上的标题起得更夸张——"ChatGPT 高手不想让你知道的技能：JSON Prompting"。理由通常是一句话："模型吃过海量代码和 API 文档，JSON 对它来说是高信号格式。"

我寻思着这事儿有没有道理。它闻起来有点玄学——但又不完全像玄学：写过 LLM 应用的程序员都知道，API 里确实有个正经功能叫 JSON mode / structured output（结构化输出），让模型保证输出合法 JSON。**偏方说的是输入侧（用 JSON 问），官方功能管的是输出侧（让模型答 JSON）**——两边都绕着同一个问题：格式这个东西，对模型来说到底意味着什么？

先交代一下本文的路线，省得读到一半迷路：**"JSON 输入会不会让回答更好"，据我所知没有人正经研究过**——但**输出侧**这两年被学术圈研究透了：从"强制输出格式会不会伤智商"的争论，一路走到 2026 年一篇预印本的发现——格式在模型内部竟然可以浓缩成**一根方向向量**。而输入和输出侧共用同一个东西：模型内部的"格式状态"。所以本文的打法是**绕道**：先把输出侧的机制搞清楚，最后回来对输入侧的偏方给一个有根据的猜测。

还要交代第二句：下面这几篇论文，每篇只负责自己那一小块证据，把它们串成一条能回应偏方的逻辑线，是我自己的拼法——故事是我编排的，证据是它们各自的。拼得成不成立，你看完自己判断。

下面出发，从工程实现讲到内部机制，最后回来看那个偏方。

◆ 第一节：JSON mode 的工程实现——掐着嘴巴逐 token 过滤

━━━━━━━━━━━━━━━━━━━━

先讲机制。主流的结构化输出实现，学名叫 grammar-constrained decoding（语法约束解码），代表作是 Outlines（arXiv 2307.09702）。

基本思路一句话就能说完：生成的每一步，把不符合 JSON 语法的 token 的 logits 置成 -inf，只在合法的 token 里采样。这一步谁都能想到，**难的不是掩码，是"哪些 token 合法"这个判断本身**，有两个坑：

第一个坑：**合法性取决于"现在写到语法的哪个位置"**。同一个 token `"` ，在等着写 key 的位置是合法的，在一个数字写到一半的位置就不合法。所以得有个东西记住"当前语法状态"——这就是有限状态机（FSM）：写完 `{` 进"期待 key"状态，写完 key 进"期待冒号"状态，一步步转移。纯 JSON 语法的核心状态并不复杂；碰上完整 JSON Schema 的嵌套对象、数组约束和正则，状态空间就会膨胀得多。

第二个坑（这个才要命）：**模型的 token 和语法的字符对不齐**。词表里的 token 是 BPE 切出来的多字符碎块——我翻了下 Qwen 的词表（15.1 万个 token），里面真实存在 `{"`、`":"`、`"},` 这样的碎块，最夸张的是 `},{"`：一个 token，横跨"关掉上一个对象 → 逗号 → 打开新对象 → 准备写 key"四个语法状态。判断这种 token 在某个状态下合不合法，得让它的每个字符在状态机上连走四步、步步不违规才行。而词表有十几万个 token，**每生成一步都对全词表做一遍这种模拟，慢到没法用**。

Outlines 的核心贡献就是解决第二个坑：**离线预处理**。把词表里每个 token，从状态机的每个状态出发都试走一遍，提前建好一张大表——"状态 S 下，合法的 token 集合是 {...}"。生成时不必再扫描全词表逐字符模拟，只需高效查表并做掩码，运行时开销大幅降低。所谓"把语法编译成 FSM"，真正让生成跑得快的是这张状态—token 转移索引表。

另外记住一个关键事实，后面要用：**这一切发生在输出层，模型本体一个参数都没动，模型甚至不知道自己被约束了**。

────────────────────

💡 这里藏着一个不显然的副作用

硬掩码会**扭曲模型本来的概率分布**。Grammar-Aligned Decoding（arXiv 2405.21047，NeurIPS 2024）指出：把不合法 token 掐掉再归一化，得到的分布并不等于"模型在所有合法输出上的真实条件分布"——模型本来沿着某条高概率路径走，被掩码硬拐进一条它自己觉得别扭的路径，后续每一步都是在"将错就错"。语法是合法了，但生成的内容可能已经不是模型的最优解。

────────────────────

打个比方：模型每一步"内心"都有一个完整的候选发言列表，约束解码就是站在旁边掐着它的嘴，凡是不符合语法的词一律不准出口。模型可能很想先说一句 "Sure, here's the JSON:"——不准，第一个字符必须是 `{`。

被掐着嘴说话，能不影响发挥吗？这正是接下来这场架吵的东西。

━━━━━━━━━━━━━━━━━━━━

◆ 第二节："格式税"之争——强制 JSON 到底伤不伤推理

━━━━━━━━━━━━━━━━━━━━

有了上面的机制背景，就能看懂这场争论了。

**正方**：Let Me Speak Freely（arXiv 2408.02442，EMNLP 2024 Industry track）。论文标题就是态度——"让我自由说话"。他们的实验结论是：强制 JSON / XML / YAML 格式输出，会显著伤害模型的推理能力，而且**约束越严格，退化越大**。直觉解释也说得通：推理类任务需要模型"先想后答"，而格式约束从第一个 token 就开始掐嘴，把"想"的空间挤没了。

**反方**：Outlines 背后的 dottxt 团队，Will Kurt 写了篇博客反驳（blog.dottxt.co，标题 "Say What You Mean"）。核心论点：你们观察到的退化，锅不在约束解码本身，而在**烂 prompt 和烂 schema**——比如 schema 设计得没给推理留字段、prompt 和格式要求打架。把 prompt 和 schema 写好，约束解码不但不伤，还常常更好。

**第三组证据**：JSONSchemaBench（arXiv 2501.10868）拿上万个真实世界的 JSON schema 系统评测了各家约束解码框架，比的不只是"语法合法率"，还有对下游任务质量的影响。各框架在效率和 schema 覆盖率上差异很大；但在它选取的下游任务里，约束解码整体没有收税，反而把表现最高提高了约 4%，连 GSM8K 这种结构很轻的任务也受益。它说明的不是"格式税永远不存在"，而是**不能从存在硬约束直接推出智商下降**：任务、prompt、schema 和实现方式都会改变结果。

老实说，两边抓到的是不同条件下的真现象。正方证明了格式限制确实可能让模型变笨；反方和 JSONSchemaBench 则说明，这不是硬约束必交的固定税率，prompt、schema 与实现足以让结果翻转。**这场争论真正缺的，是打开模型内部看一眼：格式处理到底发生在哪里、和内容处理是不是纠缠在一起。**

这就要先补一块背景知识。

━━━━━━━━━━━━━━━━━━━━

◆ 第三节：拒绝、人格、指令——一根向量就能开关一种行为

━━━━━━━━━━━━━━━━━━━━

可解释性研究这几年反复发现一件事：模型的很多**高层行为**，在内部激活空间里对应的不是一团乱麻，而是**一根方向向量**。

先给核心公式。Transformer 每层的输出都写进一条贯穿全网的"残差流"（residual stream），设某层某位置的激活为 h（一个几千维的向量），所谓 steering vector（转向向量）干预就是：

```text
h' = h + λ·v
```

v 是一根事先找到的方向向量（通常拿"有该行为的样本"和"无该行为的样本"的激活均值做差），λ 是强度系数。就这么一行加法，行为就变了。

这个家族不是一批平行的发现，是一条**时间上递进的线**，每一步都把胆子放大一圈：

**2024 年 6 月，开山之作：拒绝是一根轴。** refusal direction（拒绝方向，arXiv 2406.11717，NeurIPS 2024）：模型的"拒绝回答"行为由**一维子空间**中介——在 13 个开源模型（最大到 72B）上，把这一根方向从激活里减掉，模型就不再拒绝；加上去，无害问题也拒绝。（严谨起见：2026 年有工作 arXiv 2602.02132 提出反驳，认为某些设置下拒绝不止一根轴——"一根轴"是很好的一阶近似，别当铁律。）

**2024 年 10 月，工程化：指令也能变成轴。** instruction steering（arXiv 2410.12877，ICLR 2025，Microsoft）：把"请用 JSON 回答"这类**格式指令**提取成向量直接注入激活，不写进 prompt 也生效，而且可叠加（JSON + 小写同时注入）、可跨模型迁移。一根轴从"研究发现"变成了"工具箱"。

**2025 年 7 月，扩张到性格：人格也是轴。** persona vectors（人格向量，arXiv 2507.21509，Anthropic）：邪恶、谄媚、编造倾向，各自对应一根**可监控、可干预**的方向。我们之前也动手做过 persona steering 实验（ https://mp.weixin.qq.com/s/5EwUEXjFdD1ztjF8TWrvSA ），亲手拨过这根轴的人会对"人格是个向量"这句话有完全不同的体感。

注意这条线一路狂奔时埋下的一个**隐含假设**："找到能读出行为的方向"和"拿到能控制行为的方向"经常被当成同一件事——读出来的轴，拨它就灵。拒绝灵了，指令灵了，人格也灵了，于是这个假设很少被单独拿出来验。

直到 2026 年，有人回头查了这笔账。

━━━━━━━━━━━━━━━━━━━━

◆ 第四节：主菜——Perfect Detection, Failed Control

━━━━━━━━━━━━━━━━━━━━

来看这篇 2026 年 6 月的预印本（arXiv 2606.24952，注意：**预印本，未经同行评审**，结论强度自己打折）。它研究的格式行为特别接地气——先把这个困扰说清楚，自己搭过小模型管线的人都撞过。

场景是这样：你在本地跑一个小模型（论文用的是 Gemma-2-2B-it，26 层，残差流 2304 维），prompt 里明明白白写着"只输出 JSON，不要任何其他内容"。你期待拿到的是：

```text
{"answer": 42}
```

它实际给你的是：

````text
```json
{"answer": 42}
```
````

多出来的这层 `` ``` `` 叫 **markdown 代码围栏**——模型是被聊天数据训出来的，它"以为"你是个坐在聊天界面前的人类，贴心地给代码块加上排版标记，好让前端渲染出漂亮的高亮框。但你不是人类，你是一段 `json.loads(response)`——第一个字符是反引号不是 `{`，**直接抛异常**。于是你被迫在管线里加一段丢人的补丁：先用正则把 ```` ```json ```` 和结尾的 ``` 剥掉，再解析。每个用小模型干过活的人，代码库里都躺着这么几行剥壳正则。

而且这不是偶发抽风：论文实测，Gemma-2-2B-it 在这个设置下 **100% 套围栏**——prompt 里怎么求都没用。这就是我们最终要看的实验：**"套围栏还是给裸 JSON"这个包装决策，在模型内部是怎么做出的、能不能直接关掉。**

不过在上主菜之前得交代一句，省得你读原文时犯嘀咕：这篇论文的主角其实不是格式，是**幻觉**——作者研究的是"模型明明检测得到自己在编造，却控制不了"这条裂缝，格式在论文里只是个对照案例，用来证明"检测不等于控制"不是方法问题。而我们这篇文章的视角是反着借用的：**论文拿格式当陪衬讲幻觉的坏消息，我们拿幻觉当陪衬讲格式的好消息**。两个消息都值得听。先说坏消息——

**坏消息：幻觉看得见，推不动。** 作者给模型喂真假混合的实体（50 个真实 + 50 个编造的首都、科学名词之类），用线性探针读残差流，**从第 5 层起就能以 AUC = 1.000 完美分辨模型"知不知道这个实体是编的"**——检测这边，六种独立方法全部成功，这是 Perfect Detection。然后灾难开始了：沿着这根"看见幻觉"的方向去干预，想让模型拒绝编造——**数据驱动的方向推下去，编造反而变多**；换手工挑的方向，强度加到 α=15（相当于粗暴地拽），编造率才从约 70% 降到约 40%；SAE 特征放大 50 倍，无效；直接消融相关神经元，无效。这是 Failed Control。为什么推不动？作者算了一下："看见幻觉"的检测方向和"产生拒绝"的控制方向，cosine 相似度只有 0.12——**夹角 83°，接近正交**。你能看见它的窗户，和能推动它的门，根本不在一面墙上。

────────────────────

💡 0.12 算正交吗？为什么正交就推不动？强行推会怎么样？

先校准一下 0.12 这个数：在 2304 维空间里，两根随机方向的 cosine 期望是 0，典型波动量级约为 1/√2304 ≈ 0.02。所以 0.12 不是零——大约是随机波动量级的 6 倍，两根方向有微弱关联；但离"同一根轴"（cosine 接近 1）差着十万八千里。高维空间的基本盘是**几乎所有方向都互相正交**——错开是白送的默认态，重合才是需要专门解释的稀罕事。这句话记住，后面有用。

那为什么正交就推不动？下游每一层的神经元读激活的方式本质是内积：拿自己的权重向量 w 去量激活 h，得到 w·h。放到这个实验里：**d 是你注入的方向——"看见幻觉"的检测轴；w 是下游"拒绝电路"的读数权重——它对准的是拒绝轴**。你的干预是 h' = h + λ·d，拒绝电路读到的变化就是 λ×(w·d)。**如果 w 和 d 正交，内积为零——λ 加到天上去，拒绝电路读到的数字纹丝不动。** 探针能从检测轴读出"这实体是编的"，只说明信息在那儿存着；而决定"拒绝还是照编"的电路，权重对准的是另一个子空间。信息在场，不等于信息被用。

往深想一步，严谨地圈定范围：**这个近正交，是这个 2B 模型"明明检测到实体是假的、却拉不住输出"的几何表征之一**——第 5 层就能把真假实体线性分开，却照编不误，说明"检测到陌生实体"没有直接接上"拒绝输出"这根线。论文还找到更具体的输出机制：残差流里更强的**实体复制成分**压过了检测信号，而检测方向本身甚至会压低拒绝 token、抬高编造式开头。不是模型没有信号，而是信号既弱于主干流量，又没有接到正确的执行器上。至于"所有模型的幻觉是不是都因为这种断线"——这个诱人的推广，论文只测到 1B～9B 的几个模型，别跟着我一起想当然。

那强行推呢？cosine 0.12 不是零，线性投影上确实能在控制轴蹭到 λ×0.12 的分量；但真实模型后面还有非线性层和整套词表重排，不能把 α=15 才压到 40% 简单归因成"靠这点泄漏硬蹭"。实验能确定的是：要施加很强的扰动才换来部分改善，继续加力还会出现 token 循环等崩坏。真正有希望的路子是别逼一根线干两件事：**检测方向当传感器，控制方向当执行器**——运行时探针一亮灯"它可能在编"，就沿真正管拒绝的方向推一把，等于在模型外面焊一条"知→行"的反射弧。论文也试了把检测方向朝拒绝方向旋转 15°：在两类留出假实体上分别做到 73% 和 60% 拒绝，误拒真实问题 1.8%。不完美，但这条路确实接上了一部分。

────────────────────

**好消息：格式不一样。** 同一套流程跑"套不套 markdown 围栏"这个行为，一切都反过来了：

- **决策位置**：用 logit lens（逐层透镜）——把每层的中间激活直接投影到词表、看"这一层想说什么"——观察 `{` 和 `` ``` `` 这两个 token 的倾向分岔（`` ``` `` 在词表里是单个 token，代码语料里它出现得太频繁了）。分岔发生在**第 20-25 层，峰值第 24 层**。严格说，这只能证明裸 JSON 与围栏的竞争在最后几层才变得可直接读出，不能证明前 20 层完全没处理格式；但至少在输出兑现之前，包装选择表现得很晚。
- **方向的来历**（简陋得出奇）：围栏方向不需要跑模型收集激活，**直接从输出词表矩阵里减出来**——``d = normalize(W['{'] − W['```'])``，就是 `{` 和 `` ``` `` 两个 token 的 unembedding 向量之差。一行代码，对付着用——但下面 hint 块会讲，对付着居然够用，这件事本身有信息量。
- **干预效果**：在第 20-25 层沿这根方向注入，强度 α=3.5——**折算下来仅占该层激活范数的 0.6%**——结果：围栏消除 100%，JSON 合法 100%，答案正确率 100%。而且是陡峭的开关型响应：α 低于 2.5 围栏一个不少，超过 3.5 一个不剩，中间几乎没有过渡带。

0.6% 是什么概念？在一个 2304 维的向量上加了一个几乎察觉不到的小扰动，模型的"内容脑区"完全没被打扰，只有"包装决策"被精准改写了。

────────────────────

💡 为什么从词表矩阵里减两行就能得到一个"方向"？

看模型的最后一步在干什么。所有层算完得到最终激活 h（2304 维向量），变成"下一个 token 的概率"就一步矩阵乘：logits = W·h，其中 W 每个 token 一行。拆开看，token t 的分数 = W[t]·h——**h 和 W[t] 这一行做内积**。所以 W[t] 这行向量的几何含义是：残差流往这个方向指，token t 就更想被说出来——**词表矩阵的每一行，就是那个 token 的天线**。模型的最后一步毫无神秘感：拿 h 对着十几万根天线逐一做内积，谁的投影大谁上。

那两行相减呢？由内积的线性：``(W['{'] − W['```']) · h = logit('{') − logit('```')``。**这根差向量就是"两个 token 分数差"的梯度方向**——忽略归一化和后续层变换时，沿它推 h，`{` 相对围栏符号的优势涨得最快。想让模型"更想说 `{`、更不想说围栏"，不用学、不用采数据，输出字典里就藏着一个现成的线性把手。

还剩一个疑点：方向从输出端拿的，凭什么在第 24 层注入有效？因为残差流是加法结构——第 24 层加进去的分量会一路被抬着送到出口（除非后面的层专门抵消它）。而且证据现成：**logit lens 能工作本身**，就证明深层残差流已经"说着词表的语言"——第 24 层的激活直接投到 W 上就能读出合理预测，说明深层坐标系和输出端坐标系已经对齐了。

顺着这个疑点追问一层：既然方向从输出字典借、效果也终归要落在输出分数上，那它和直接在输出层加 logit bias（给 `{` 加分、围栏减分，API 里现成有）是一回事吗？**不完全一样，差在落地方式上。** bias 只改你点名的两格分数——**窄谱、绕过计算、单点钳制**；而往第 20-25 层的残差流里注入，落到出口时会让全词表按各自与这根方向的投影一起重新计分，中间还经过末端 attention/MLP 的加工，而且每个解码步都在续力——**广谱、经过计算、逐步在场**。理论上，与 markdown 围栏同向的其他 token 也会被联动压低；但论文没有测试 `~~~`、`<code>` 等变体，不能提前宣布一网打尽。粗略地说，一个更像按住手，一个更像在内部拨动倾向——这是机制解释，不是这篇论文做过的对照实验。

不过要把话说严格：从第 24 层到最终 logits，中间还隔着两层计算，真实的读出映射是 W 乘上后续层的雅可比，不是裸的 W。**直接加 d，等于假设那个雅可比约等于单位阵**——好在残差流的加法结构保证了直通路径贡献一个单位阵、修正只来自剩下两层的加工，所以这个假设在**末端几层**近似成立，裸 d 凑合能用。换到第 5 层就彻底不灵了：21 层变换累积下来，坐标系早拧走了，同一根 d 在那儿是乱码——这也是为什么读中间层表征的正经工作要给每层单独配雅可比透镜，比如 Anthropic 的 J-space 论文（245 期讲过： https://mp.weixin.qq.com/s/PleyNOp7zvV3x-j3DVqq9Q ）。所以干预点选在 20-25 层是两个条件的交集：格式决策恰好在那儿，**且**"从字典借方向"这个偷懒办法恰好在那儿还没失效。

一句话把这个 hint 块收平：这根方向是从输出端直接构造、又在出口附近注入的，所以它首先证明的是：**格式行为存在一个极易访问、功能上兼具读出与控制能力的线性把手**。一根不靠训练、不采激活的方向，推 0.6% 就让行为 100% 翻转且内容无损，这个控制把手确实非常突出；但仅凭这个实验，还不能断言模型把抽象的"格式概念"天然压缩成了唯一一根轴。

────────────────────

把坏消息和好消息并排放在一起，才是这篇论文真正想说的话。同一个模型、同一套方法，两种行为给出了两种世界：

- **幻觉**：检测方向和控制方向夹角 83°，几乎正交——你能看见它，但从看见的那个角度使劲，推不动它。知与行分家。
- **格式**：存在一根既能近乎完美读出格式、又能直接控制格式的方向——而且它简单到可以从词表矩阵里减出来。

用一个词概括：**格式在功能上呈现出"知行合一"**——同一根方向既读得出，也拨得动。但这里有个很重要的方法学钉子：论文后来用激活数据独立寻找格式检测方向，它和这根控制方向并不重合，cosine 同样接近高维随机水平。原因是**能检测一个行为的方向本来就不止一根，而是一个高维方向集合**；格式案例证明的是"控制方向同时也是好探针"，不是"所有检测方法都会找到同一根唯一轴"。反过来看幻觉，专门训练的检测探针 AUC = 1.000，真正能管拒绝的方向却只勉强读得出真假。于是更准确的结论是：**"找到一根能检测的方向 = 拿到控制权"不是定律；关键要看那根真正控制行为的方向，是否也读得到它应该响应的信号。**

到这儿，四块素材齐了：工程实现、格式税之争、一根轴家族、格式控制把手的直接证据。可以结账了。

━━━━━━━━━━━━━━━━━━━━

◆ 第五节：收束——格式和内容，至少存在可分离的控制接口

━━━━━━━━━━━━━━━━━━━━

把四节串起来，能得到一个挺干净的世界观：

**"说什么"通常是高维计算，"怎么包装"至少存在低维把手。** 在 Gemma-2-2B-it 这个案例里，JSON 外面套不套围栏，可以被一根方向干净地翻转；而真假实体涉及的检测与行动却散落在不同子空间。我们不能从一个 2B 模型的一种二元包装行为，直接推广成"所有格式都只有一根轴"；但这个对照已经足以说明：格式选择和内容计算至少可以相当程度地分离，像流水线末端一个可单独拨动的包装开关。这也让 instruction steering 能把格式指令做成可叠加向量这件事不再神秘：存在低维控制接口的行为，才容易搬运。

于是"怎么让模型输出 JSON"其实有两条路线：

- **硬约束**：FSM 掐嘴，作用在**输出层**。优点是数学上保证合法；代价是扭曲分布，模型内部想走的路被外力硬拐。
- **软引导**：拨轴，作用在**思考层**（残差流）。优点是顺着模型自己的表示去改，内容不受干扰；代价是没有 100% 的合法性保证，而且你得先有权限摸到激活——API 用户暂时享受不到。

回到"格式税"之争，这套内部视角给出了一个还算公道的说法：**格式税不是必然的，约束和模型原生倾向冲突时才容易收税。** 如果格式行为存在一个好用的低维控制接口，那么顺着它走（好 prompt、好 schema、或者直接 steering），内容可以几乎无损——这是 dottxt 一方和 JSONSchemaBench 支持的方向；而 FSM 在输出端逐 token 裁掉路径，碰上 schema 与模型原生分布严重冲突时，就可能把扭曲一路放大——这是 Let Me Speak Freely 一方观察到的现象。两边看到的都是真的，只是实验条件不同。

━━━━━━━━━━━━━━━━━━━━

◆ 尾声：回到开篇的偏方

━━━━━━━━━━━━━━━━━━━━

**用 JSON 提问，有合理内核，但别神化。** 用本文的视角看，一种可能是：输入侧的 JSON 给模型的结构化输出倾向送了一个免费的软信号——你没掐它的嘴、没动它的 logits，只是用输入形状把内部状态往"结构化模式"推了一把。还有一种更朴素、甚至可能更主要的解释：**JSON 先改造了提问的人**。它逼你给字段命名、拆开约束、消除指代、明确边界；效果变好，可能只是因为需求终于写清楚了。对**本身就该结构化的任务**（信息抽取、多字段填表、批量重复的流程化请求），这两种收益会叠加，效果好不奇怪；但对**需要自由推理和创意的任务**，把问题硬塞成键值对未必增加信息，反而可能诱导模型过早进入"填表模式"。up 主视频里不会告诉你的是后半句。

和开头交代过的一样，这个说法是**我的推断，不是哪篇论文的结论**——"JSON 提问 vs 回答质量"这个具体问题，据我所知还没有人正经测过（谁看到相关实验记得告诉我）。真要测，必须让自然语言版和 JSON 版包含完全相同的字段、顺序与约束，否则测到的不是 JSON 红利，只是"其中一组 prompt 写得更清楚"。四块拼图各自有证据：掐嘴会扭曲分布是证明过的，格式税在不同实验里会出现也会消失，格式行为在末端层存在一个强线性控制把手也是真的。但"输入侧 JSON = 给这个把手送软信号"这最后一跳，是我拼上去的。它合理，不代表它已经被验证。

下次你打开 JSON mode、或者往输入框里敲 JSON 的时候，可以想一下：合同（schema）是签在模型嘴上的，而它对格式的"想法"存放在倒数几层的一根方向向量里。这两个地方离得越近，你付的税就越少。
