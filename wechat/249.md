【论文解读】"必须输出 JSON"的时候，模型脑子里发生了什么——格式在 AI 内部是一根轴

━━━━━━━━━━━━━━━━━━━━

◆ 开篇：一个天天在用、但很少细想的功能

━━━━━━━━━━━━━━━━━━━━

写过 LLM 应用的程序员，大概率都干过这件事：在 API 里打开 JSON mode，或者传一个 schema 进去做 structured output（结构化输出），然后放心地拿返回值直接 `json.loads`。

它确实好用。但你有没有想过一个问题：**模型是"学会了"输出 JSON，还是"被摁着"输出 JSON？** 这两者的区别，比听上去大得多——它牵扯到一场持续两年的争论（"强制格式会不会伤智商"），也牵扯到 2026 年一篇很有意思的预印本：格式这个东西，在模型内部竟然可以浓缩成**一根方向向量**。

今天把这条线从工程实现讲到内部机制，走一遍。

━━━━━━━━━━━━━━━━━━━━

◆ 第一节：JSON mode 的工程实现——掐着嘴巴逐 token 过滤

━━━━━━━━━━━━━━━━━━━━

先讲机制。主流的结构化输出实现，学名叫 grammar-constrained decoding（语法约束解码），代表作是 Outlines（arXiv 2307.09702）。流程是这样的：

```text
输入：一个语法（比如 JSON schema 编译出的正则/CFG）
操作：把语法编译成有限状态机（FSM）
     生成每一步：
       1. 模型正常算出全词表的 logits（几万到十几万个 token 各一个分数）
       2. 查 FSM 当前状态：哪些 token 是"合法的下一步"
       3. 不合法的 token，logits 直接置 -inf
       4. 在剩下的合法 token 里正常采样
输出：一个保证语法合法的序列
```

关键点有两个。第一，**这一切发生在输出层，模型本体一个参数都没动**。第二，因为 FSM 的状态转移是预编译好的查表，每步的掩码开销近乎为零——这是 Outlines 的核心贡献，让约束解码快到可以生产环境用。

────────────────────

💡 这里藏着一个不显然的副作用

硬掩码会**扭曲模型本来的概率分布**。Grammar-Aligned Decoding（arXiv 2405.21047，NeurIPS 2024）指出：把不合法 token 掐掉再归一化，得到的分布并不等于"模型在所有合法输出上的真实条件分布"——模型本来沿着某条高概率路径走，被掩码硬拐进一条它自己觉得别扭的路径，后续每一步都是在"将错就错"。语法是合法了，但生成的内容可能已经不是模型的最优解。

────────────────────

打个比方：模型每一步"内心"都有一个完整的候选发言列表，约束解码就是站在旁边掐着它的嘴，凡是不符合语法的词一律不准出口。模型可能很想先说一句 "Sure, here's the JSON:"——不准，第一个字符必须是 `{`。

被掐着嘴说话，能不影响发挥吗？这正是接下来这场架吵的东西。

━━━━━━━━━━━━━━━━━━━━

◆ 第二节："格式税"之争——强制 JSON 到底伤不伤推理

━━━━━━━━━━━━━━━━━━━━

有了上面的机制背景，就能看懂这场争论了。

**正方**：Let Me Speak Freely（arXiv 2408.02442，EMNLP 2024 Industry track）。论文标题就是态度——"让我自由说话"。他们的实验结论是：强制 JSON / XML / YAML 格式输出，会显著伤害模型的推理能力，而且**约束越严格，退化越大**。直觉解释也说得通：推理类任务需要模型"先想后答"，而格式约束从第一个 token 就开始掐嘴，把"想"的空间挤没了。

**反方**：Outlines 背后的 dottxt 团队，Will Kurt 写了篇博客反驳（blog.dottxt.co，标题 "Say What You Mean"）。核心论点：你们观察到的退化，锅不在约束解码本身，而在**烂 prompt 和烂 schema**——比如 schema 设计得没给推理留字段、prompt 和格式要求打架。把 prompt 和 schema 写好，约束解码不但不伤，还常常更好。

**裁判视角**：JSONSchemaBench（arXiv 2501.10868）拿上万个真实世界的 JSON schema 系统评测了各家约束解码框架，比的不只是"语法合法率"，还有对下游任务质量的影响。结果大体上是：不同框架、不同任务差异很大，"格式税"不是一个恒定存在的东西，但也确实不是空穴来风。

老实说，两边都有道理。正方指出的现象是真的——你如果用过早期的 JSON mode，大概率也遇到过"套上格式之后答案明显变笨"的时刻，这不怪你的错觉。反方指出的归因问题也是真的——很多退化案例换个 schema 就消失了。**这场争论真正缺的，是打开模型内部看一眼：格式处理到底发生在哪里、和内容处理是不是纠缠在一起。**

这就要先补一块背景知识。

━━━━━━━━━━━━━━━━━━━━

◆ 第三节：背景——"行为 = 一根轴"这个家族

━━━━━━━━━━━━━━━━━━━━

可解释性研究这几年反复发现一件事：模型的很多**高层行为**，在内部激活空间里对应的不是一团乱麻，而是**一根方向向量**。

先给核心公式。Transformer 每层的输出都写进一条贯穿全网的"残差流"（residual stream），设某层某位置的激活为 h（一个几千维的向量），所谓 steering vector（转向向量）干预就是：

```text
h' = h + λ·v
```

v 是一根事先找到的方向向量（通常拿"有该行为的样本"和"无该行为的样本"的激活均值做差），λ 是强度系数。就这么一行加法，行为就变了。这个家族的代表作：

- **refusal direction（拒绝方向）**，arXiv 2406.11717，NeurIPS 2024。结论很惊人：模型的"拒绝回答"行为由**一维子空间**中介——在 13 个开源模型（最大到 72B）上，把这一根方向从激活里减掉，模型就不再拒绝；加上去，无害问题也拒绝。严谨起见补一句：2026 年有工作（arXiv 2602.02132）提出反驳，认为拒绝行为其实不止一根轴，一维假设在某些设置下会失效——"一根轴"是个非常好的一阶近似，但别当铁律。
- **persona vectors（人格向量）**，arXiv 2507.21509，Anthropic 2025。邪恶、谄媚、编造幻觉这类"性格倾向"，各自对应一根可以监控、可以干预的方向。我们之前也动手做过 persona steering 实验（往期链接待补），亲手拨过这根轴的人会对"人格是个向量"这句话有完全不同的体感。
- **instruction steering**，arXiv 2410.12877，ICLR 2025，Microsoft。这篇离今天的主题最近：**格式指令本身可以向量化**——把"请用 JSON 回答"这类指令提取成向量直接注入激活，不用写进 prompt 也能生效，而且可叠加（JSON + 小写同时注入）、可跨模型迁移。

看出趋势了吗？拒绝、人格、指令遵循……这些听起来很"高级"的行为，内部表示都出奇地低维。那么"输出什么格式"呢？

◆ 第四节：主菜——Perfect Detection, Failed Control

━━━━━━━━━━━━━━━━━━━━

来看这篇 2026 年 6 月的预印本（arXiv 2606.24952，注意：**预印本，未经同行评审**，结论强度自己打折）。它研究的格式行为特别接地气，就是每个用过小模型 JSON mode 的人都骂过的那个：**markdown 代码围栏**。

你要 JSON，模型给你的是：

````text
```json
{"answer": 42}
```
````

论文用的模型是 Gemma-2-2B-it，默认情况下 100% 会给 JSON 套上这层围栏——`json.loads` 直接爆炸，你还得自己写正则剥壳。作者问了三个问题：

**第一问：模型是什么时候决定套围栏的？** 用 logit lens（逐层透镜）——一种把每一层的中间激活直接投影到词表、看"这一层想说什么"的技术——逐层观察 "```" 这个 token 的倾向。结果：格式决策发生在**第 20-25 层**（模型共 26 层），峰值在第 24 层。也就是说，模型前 20 层都在处理"答案是什么"，最后几层才拍板"用什么包装"。

**第二问：能不能找到一根"围栏方向"？** 能。拿"套围栏的输出"和"裸 JSON 输出"的激活做均值差，得到一根方向向量，用它做线性探针检测"这次会不会套围栏"，检测近乎完美——这是标题里的 "Perfect Detection"。

**第三问：能不能沿这根方向干预，把围栏关掉？** 标题的后半句 "Failed Control" 说的是朴素做法会失败——但论文最终找到了成功的干预配方：在合适的层位沿这根方向做 steering，**注入量仅占激活范数的 0.6%**，就能 100% 消除围栏，同时 JSON 依然合法、答案内容不变。

0.6% 是什么概念？相当于在一个几千维向量上加了一个几乎察觉不到的小扰动，模型的"内容脑区"完全没被打扰，只有"包装决策"被精准改写了。

────────────────────

💡 全文最有味道的一个发现

论文对比了格式行为和其它行为（比如幻觉倾向）的一个关键差异：**"检测方向"和"控制方向"是不是同一根**。

- 幻觉这类行为：最能"读出"该行为的方向，和最能"改变"该行为的方向，是错开的（两者 cosine 相似度约 0.12，几乎正交）——你能看见它，但从看见的那个角度使劲，推不动它。
- 格式行为：两根方向**重合**。读出它的轴，就是控制它的轴。

用一个词概括：格式是模型内部 **"知行合一"** 的特例——它知道的地方，就是它做的地方。而"说真话还是编造"这种事，知与行分家。

────────────────────

到这儿，四块素材齐了：工程实现、格式税之争、一根轴家族、格式轴的直接证据。可以结账了。

◆ 第五节：收束——格式和内容，在模型内部是分开放的

━━━━━━━━━━━━━━━━━━━━

把四节串起来，能得到一个挺干净的世界观：

**"说什么"是高维的，"用什么格式说"是低维的。** 答案内容需要模型动用几乎全部表示空间去计算；而格式——JSON 还是自然语言、套不套围栏——在内部被压缩成了少数几根轴，在网络的后段（Gemma-2 的第 20-25 层）才介入，像流水线末端的包装工序。这也解释了 instruction steering 为什么能把格式指令做成可叠加、可迁移的向量：低维的东西才好搬运。

于是"怎么让模型输出 JSON"其实有两条路线：

- **硬约束**：FSM 掐嘴，作用在**输出层**。优点是数学上保证合法；代价是扭曲分布，模型内部想走的路被外力硬拐。
- **软引导**：拨轴，作用在**思考层**（残差流）。优点是顺着模型自己的表示去改，内容不受干扰；代价是没有 100% 的合法性保证，而且你得先有权限摸到激活——API 用户暂时享受不到。

回到"格式税"之争，这套内部视角给出了一个还算公道的裁决：**格式税不是必然的，约束加错了地方才收税。** 如果格式决策本来就是模型后段的一根轴，那么顺着这根轴走（好 prompt、好 schema、或者直接 steering），内容几乎无损——这是 dottxt 一方的道理；而 FSM 从第一个 token 就介入、且作用点和模型的格式表示完全错位，在 schema 和模型习惯冲突时就会把扭曲一路放大——这是 Let Me Speak Freely 一方观察到的现象。两边看到的都是真的，只是看的位置不同：一个看输出层的行为，一个（现在我们知道了）该去看第 24 层的那根轴。

下次你打开 JSON mode 的时候，可以想一下：合同（schema）是签在模型嘴上的，而它对格式的"想法"存放在倒数几层的一根方向向量里。这两个地方离得越近，你付的税就越少。
