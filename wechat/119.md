【鸿蒙AI+】在 OpenHarmony 开发板上跑 AgentScope 多智能体框架

——昨天跑通 LangChain，今天补上多智能体这块拼图

━━━━━━━━━━━━━━━━━━━━

◆ 为什么 LangChain 还不够

━━━━━━━━━━━━━━━━━━━━

昨天那篇，在 OH 上跑通了 LangChain。能调 DeepSeek API，能用 `HumanMessage` / `AIMessage` 做消息格式化，链式调用、prompt 模板都能跑。

但 LangChain 擅长的是**"链式调用"**——一步一步的流水线。你给它一个输入，它按顺序过几道工序，吐出一个输出。适合"提问→检索→生成→返回"这种线性场景。

多智能体场景不是这么回事。**多个 Agent 之间要自主对话、协调、分工。** Agent A 说完话，Agent B 要判断该不该接、接什么、还是该把话转给 Agent C。这不是流水线，是会议室。

LangChain 硬要做多智能体也能做（LangGraph），但不是它的基因。AgentScope（阿里开源）从第一天就是为多智能体设计的——消息广播、角色分配、对话流控制、Agent 间的自主协商，都是一等公民。

所以不是"替代"LangChain，是**补上多智能体这块拼图**。LangChain 管链式调用，AgentScope 管多体协同。两个框架在同一块板子上共存，各司其职。

打个比方：LangChain 是流水线车间——零件从这头进，从那头出，每一步固定。AgentScope 是项目组会议室——几个人坐一起讨论，谁先发言、谁反驳、谁拍板，不是写死的。

昨天搭好了车间，今天装会议室。

━━━━━━━━━━━━━━━━━━━━

◆ 依赖链分析：哪些要编，哪些白捡

━━━━━━━━━━━━━━━━━━━━

动手之前先算账。

AgentScope 1.0.16 的依赖树拉出来一看——好消息是，**昨天编 LangChain 时积攒的 20 多个包可以直接复用**。pydantic、pydantic-core、openai SDK、httpx、httpcore、jiter、uuid-utils……这些硬骨头昨天全编过了。今天不用重编。

就像你昨天装了厨房——灶台、燃气管、锅碗瓢盆全齐了。今天要做新菜，大部分家伙事直接拿来用，只需要添几样新调料。

**需要新编的原生扩展（4 个）：**

| 包 | 类型 | 作用 |
|:---|:---|:---|
| tiktoken 0.12.0 | Rust/PyO3 | token 计数（LangChain 时绕过了，AgentScope 要用） |
| cryptography 47.0.0 | Rust + C + OpenSSL | 加密/签名（JWT 认证等） |
| regex | C | 正则表达式加速 |
| cffi 1.17.1 | C + libffi | C 函数外部接口（cryptography 的 Python 绑定层依赖它） |

**可绕过或有 fallback 的：**

| 包 | 处理方式 |
|:---|:---|
| numpy | 写 stub——只有音频和 embedding 缓存用，核心 Agent 功能不需要 |
| aiohttp | 写 stub——dashscope SDK 顶层 import 了它，但同步 API 走 requests，不真正用 |
| rpds-py | 写 stub——jsonschema 需要的几个容器类，用 dict/set/tuple 模拟 |
| protobuf / sqlalchemy / wrapt | PyPI 上有纯 Python wheel，直接下载解压 |

**纯 Python 包（~35 个，直接丢进去）：**

agentscope 本体、anthropic、dashscope、starlette、uvicorn、pydantic-settings、python-socketio、shortuuid、jsonschema、PyJWT、opentelemetry 系列……全是 `.py` 文件，不挑平台。

算完账：**4 个要编的，3 个写 stub 糊弄的，35 个直接丢。** 昨天的基建投资，今天开始产生回报。

━━━━━━━━━━━━━━━━━━━━

◆ tiktoken：35 秒一次过

━━━━━━━━━━━━━━━━━━━━

tiktoken 是 OpenAI 出的 token 计数器，Rust 写的。昨天编 LangChain 时绕过了它（直接用 openai SDK 不走 langchain-openai），今天 AgentScope 要用，绕不过了。

tiktoken 用的构建工具和昨天的 pydantic-core 不一样（setuptools-rust 而不是 maturin），但底层还是 Rust + PyO3，套路相同：指定目标平台、指定 Python 版本、加上 `-l python3.11`。

35 秒编完，一次过。那个 `-l python3.11`——**OHOS musl 的 dlopen 不继承符号**，昨天踩了两次，今天已经内化成肌肉记忆了，每个 .so 都自动加。

━━━━━━━━━━━━━━━━━━━━

◆ cryptography：这次的硬骨头

━━━━━━━━━━━━━━━━━━━━

cryptography 是四个需要编的包里最复杂的。它不是单一 Rust 项目——是一个 **Rust workspace（8 个子 crate）+ CFFI 生成的 C 绑定层 + 依赖 OpenSSL** 的混合怪物。

编译过程踩了 5 个坑，全部围绕一个核心问题：**交叉编译时有两套工具链在同时工作，搞混了就完蛋。**

宿主机工具链（x86）负责跑 build script——生成代码、展开头文件、运行 CFFI 代码生成器。目标平台工具链（aarch64-ohos）负责编最终产物。这两套各自需要各自的编译器、头文件、库路径。

5 个坑拆开看都是"串台"的变体：

- 宿主机的 x86 gcc 混进了目标平台的编译——编出来的 .o 当然在 aarch64 上用不了
- 找到了宿主机 Python 3.10 的头文件，而不是目标平台 Python 3.11 的
- 宿主机 python 缺 cffi 模块——build script 要在宿主机上运行代码生成器
- 环境变量被 shell 语法截断——多行命令的 `&&` 把变量声明变成了独立命令
- PATH 被覆盖导致宿主机自己的编译器也找不到了

解法都是同一个思路：**用环境变量精确指定每一条路径，不让任何东西走默认值。** 宿主机的归宿主机，目标平台的归目标平台，一个萝卜一个坑。

33 秒编完。产物 6.0MB 的 .so。

**cryptography 的教训**：报错信息看似莫名其妙——"incompatible"、"file not found"、"module not found"——本质都是同一件事：**x86 的东西喂进了 aarch64 的流程，或者反过来。** 交叉编译 Rust+C 混合项目时，脑子里始终要分清"这一步在哪台机器上跑"。

━━━━━━━━━━━━━━━━━━━━

◆ regex 和 cffi：一行命令的事

━━━━━━━━━━━━━━━━━━━━

比起 cryptography，剩下两个 C 扩展简直是假期作业。regex 就 2 个 .c 文件，cffi 就 1 个 .c 文件加上之前编 CPython 时就编好的 libffi 静态库。一行编译命令，一次过，各几秒钟。

四个包的编译到此结束。tiktoken 35 秒，cryptography 33 秒（加踩坑时间大概 40 分钟），regex 和 cffi 各几秒。**真正花时间的不是编译本身，是搞清楚 cryptography 那五个坑。**

━━━━━━━━━━━━━━━━━━━━

◆ 推上板子：又一轮"跑一次缺一个"

━━━━━━━━━━━━━━━━━━━━

4 个编好的 `.so` + 35 个纯 Python 包打成增量包，hdc 推上板子，叠加在昨天的 LangChain 部署包之上。

先验证四个新编的 .so 能不能加载——cryptography、regex、cffi、tiktoken，全部一次通过。昨天的经验起作用了：每个都链了 libpython，没再出 symbol not found。

然后开始 `import agentscope`——

第一次：缺 sqlalchemy。这是个纯 Python 包（greenlet 是可选的 C 扩展，同步模式不需要），从 PyPI 下了 `py3-none-any` 的 wheel 解压推上去。
第二次：缺 multidict、yarl、frozenlist、propcache。这四个是 aiohttp 的依赖链，同样有纯 Python wheel。
第三次：缺 loguru。AgentScope 的日志库。
第四次：缺 bidict。python-socketio 的硬依赖。
第五次：缺 importlib-metadata。opentelemetry 需要它。

前后补了 10 个纯 Python 包。这种循环在移植工作里太正常了，不丢人。依赖树里的 import 是"按需加载"的——你不碰那个功能，它不报错。每往深走一步就冒出一个新的。好在全是纯 Python，从 PyPI 下载解压推上去就行，不用重新编译。

另外踩了一个 OH 特有的小坑：**OH 的 tar 不接受 `./` 前缀路径**，打包时用 `*` 而不是 `.`，否则解压报错。小事，但卡你十分钟。

补完纯 Python 包，碰到了真正有意思的坑。

━━━━━━━━━━━━━━━━━━━━

◆ aiohttp 3.6.2：跨越三年的不兼容

━━━━━━━━━━━━━━━━━━━━

AgentScope 依赖 dashscope SDK（阿里通义的 Python 客户端），dashscope 顶层 `import aiohttp`。

aiohttp 有 C 扩展（multidict/yarl 等），PyPI 上最后一个纯 Python wheel 是 3.6.2——2020 年发布的，Python 3.8 时代的产物。

推上去一跑，报错：asyncio 里找不到 coroutine 属性。Python 3.11 把 `@asyncio.coroutine` 装饰器删了，aiohttp 3.6.2 还在用。隔了三年，API 断代了。

但等一下——dashscope 真的**用** aiohttp 吗？看了一下 dashscope 的代码：同步 API 走 requests，异步 API 才走 aiohttp。AgentScope 调 DeepSeek 走的是 OpenAI 兼容接口（openai SDK），不走 dashscope 的异步通道。

**dashscope 只是在模块顶层 `import aiohttp`，但实际代码路径不经过它。**

这就是典型的**"启动时需要"和"运行时需要"的区别**。import 时 Python 会执行模块顶层代码，`import aiohttp` 失败整个 dashscope 就挂。但只要 import 不报错，后面实际调用时根本不走 aiohttp。

**解法：写 aiohttp stub。** 只提供 dashscope 导入时需要的符号（ClientSession、ClientTimeout 等），全是空壳，实际调用时抛异常。

同样的思路用在了另外两个地方：

**rpds stub**——jsonschema 依赖 rpds-py（Rust 扩展），需要几个不可变容器类。用 Python 内置的 dict、set、tuple 包一层就行，补上启动时会调用的几个方法。

**numpy stub**——AgentScope 有地方 `import numpy`，但只有音频处理和 embedding 缓存才真正用到。stub 在 import 时只打个 warning，不报错；真正调 numpy 函数时才抛异常。

**stub 思维的核心**：不是所有依赖都需要真正工作。很多 import 只是"声明式"的——模块顶层写了 `import xxx`，但实际执行路径根本不经过那段代码。**只要 import 不报错，就够了。**

这比把 aiohttp 从 3.6.2 移植到 Python 3.11、或者交叉编译 numpy 要省事一万倍。

━━━━━━━━━━━━━━━━━━━━

◆ 全链路打通

━━━━━━━━━━━━━━━━━━━━

补完所有依赖和 stub，写了个测试脚本：用 AgentScope 的 OpenAIChatModel 连 DeepSeek API，发一句"Hello from OH!"。

注意 AgentScope 1.0.16 的 API 变化：模型调用是 async 的，需要 asyncio.run() 包裹。

◆ 复盘

━━━━━━━━━━━━━━━━━━━━

从依赖分析到板子上跑通，AgentScope 移植比 LangChain 快得多。LangChain 花了两天（大半天改动态链接 + 半天编包），AgentScope 一天搞定。

原因很简单：**前一天的基建投资开始产生回报。**

第一，20 多个包直接复用。pydantic、pydantic-core、openai SDK、httpx 这些硬骨头昨天全编过了。AgentScope 额外需要的 4 个原生扩展里，tiktoken 和 regex 加起来不到 1 分钟，cffi 几秒钟。只有 cryptography 花了点时间踩坑。

第二，"OHOS musl dlopen 不继承符号"这个坑已经变成条件反射——每个 `.so` 都自动加 `-lpython3.11`，不用再排查一轮。昨天第一次碰花了大半天，第二次碰花了 10 分钟，今天编 4 个新包，一次都没再犯。**踩过的坑确实会变成直觉。**

第三，stub 思维。aiohttp、rpds、numpy 三个依赖如果正经编，每个都是一天的工程量。但分析完实际代码路径后发现，它们都是"import 时需要，运行时不走"的。**区分"启动时需要"和"运行时需要"，是移植工作里最重要的判断。** 该糊弄的时候果断糊弄，把精力留给真正要编的东西。

第四，cryptography 的教训值得记住：交叉编译 Rust+C 混合项目时，**宿主机工具链和目标平台工具链是两套独立的系统**。宿主机跑 build script 生成代码，目标平台编最终产物。它们各自需要各自的编译器、头文件、库路径。搞混了就是"x86 的目标文件喂进 aarch64 的链接器"——报错信息看着玄乎，本质就是串台。

回顾一下两天的数据对比：

| | LangChain（第一天） | AgentScope（第二天） |
|:---|:---|:---|
| 需要交叉编译的原生扩展 | 4 个（pydantic-core, jiter, uuid-utils, xxhash） | 4 个（tiktoken, cryptography, regex, cffi） |
| 可复用的已编包 | 0（从零开始） | ~20 个（LangChain 积累的） |
| 需要写 stub 的 | 0 | 3 个（aiohttp, rpds, numpy） |
| 纯 Python 包 | ~20 个 | ~35 个 |
| 总耗时 | ~2 天（含 CPython 动态化改造） | ~1 天（含踩坑） |
| 最耗时的环节 | 排查 dlopen segfault（大半天） | cryptography 5 个坑（~40 分钟） |

第一天是开荒，第二天是收割。差别不在个人能力，在于**基础设施的复利效应**。

最后一个感受：两天编完两个 AI 框架（LangChain + AgentScope），OpenHarmony 上的 Python AI 生态从零到可用，核心原因是 **Python 生态的 fallback 文化**。大量库的 C/Rust 扩展只是加速层，纯 Python 版本作为后备始终保留。这意味着在新平台上，你只需要编少数几个没有 fallback 的硬依赖（pydantic-core、cryptography），剩下的都能自动降级。生态的韧性不在于每个包都完美移植，在于降级路径足够多。

stub 也是 fallback 的一种——不是让模块"真正工作"，而是让 import 链不断裂。在嵌入式场景里，你不需要 numpy 的全部功能，你只需要 `import numpy` 不报错。这种"够用就行"的务实态度，反而比"要做就做完美"走得更远。

◆ 怎么在你的板子上跑

━━━━━━━━━━━━━━━━━━━━

仓库里有两个部署包，叠加使用：

1. clone 仓库
2. 用 hdc 推 LangChain 基础包到板子上，解压
3. 用 hdc 推 AgentScope 增量包，解压到 site-packages 目录
4. 下载 Mozilla CA 证书包推上去（板子上没有）
5. 跑 `import agentscope` 验证

具体命令见仓库 README。

LangChain 基础包 25MB，AgentScope 增量包 8.4MB。基础包含 CPython 3.11 + OpenSSL + LangChain 全依赖，增量包含 AgentScope 独有的依赖 + 四个新编的 .so + 三个 stub。

◆ 术语速查

━━━━━━━━━━━━━━━━━━━━

| 术语 | 解释 |
|:---|:---|
| AgentScope | 阿里开源的多智能体框架。Agent 之间能自主对话、协调、分工 |
| tiktoken | OpenAI 的 token 计数器。把文本拆成 token（模型实际处理的最小单位），Rust 写的 |
| cryptography | Python 生态最常用的加密库。底层是 Rust + OpenSSL，用于 HTTPS 证书验证、JWT 签名等 |
| cffi | C Foreign Function Interface，让 Python 调用 C 函数的桥梁 |
| stub | 桩模块。只提供 import 需要的符号（类名、函数名），实际功能是空的或直接报错 |
| dashscope | 阿里通义千问的 Python SDK。AgentScope 通过它调用通义系列模型 |
| fallback | 后备方案。C/Rust 扩展 import 失败时自动退回纯 Python 实现 |
