【鸿蒙AI+】在 OpenHarmony 开发板上跑 AgentScope 多智能体框架

——昨天跑通 LangChain，今天补上多智能体这块拼图

━━━━━━━━━━━━━━━━━━━━

◆ 为什么 LangChain 还不够

━━━━━━━━━━━━━━━━━━━━

昨天那篇，在 OH 上跑通了 LangChain。能调 DeepSeek API，能用 `HumanMessage` / `AIMessage` 做消息格式化，链式调用、prompt 模板都能跑。

但 LangChain 擅长的是**"链式调用"**——一步一步的流水线。你给它一个输入，它按顺序过几道工序，吐出一个输出。适合"提问→检索→生成→返回"这种线性场景。

多智能体场景不是这么回事。**多个 Agent 之间要自主对话、协调、分工。** Agent A 说完话，Agent B 要判断该不该接、接什么、还是该把话转给 Agent C。这不是流水线，是会议室。

LangChain 硬要做多智能体也能做（LangGraph），但不是它的基因。AgentScope（阿里开源）从第一天就是为多智能体设计的——消息广播、角色分配、对话流控制、Agent 间的自主协商，都是一等公民。

所以不是"替代"LangChain，是**补上多智能体这块拼图**。LangChain 管链式调用，AgentScope 管多体协同。两个框架在同一块板子上共存，各司其职。

打个比方：LangChain 是流水线车间——零件从这头进，从那头出，每一步固定。AgentScope 是项目组会议室——几个人坐一起讨论，谁先发言、谁反驳、谁拍板，不是写死的。

昨天搭好了车间，今天装会议室。

━━━━━━━━━━━━━━━━━━━━

◆ 依赖链分析：哪些要编，哪些白捡

━━━━━━━━━━━━━━━━━━━━

动手之前先算账。

AgentScope 1.0.16 的依赖树拉出来一看——好消息是，**昨天编 LangChain 时积攒的 20 多个包可以直接复用**。pydantic、pydantic-core、openai SDK、httpx、httpcore、jiter、uuid-utils……这些硬骨头昨天全编过了。今天不用重编。

就像你昨天装了厨房——灶台、燃气管、锅碗瓢盆全齐了。今天要做新菜，大部分家伙事直接拿来用，只需要添几样新调料。

| 包 | 类型 | 作用 |
|:---|:---|:---|
| tiktoken 0.12.0 | Rust/PyO3 | token 计数（LangChain 时绕过了，AgentScope 要用） |

◆ tiktoken：35 秒一次过

━━━━━━━━━━━━━━━━━━━━

tiktoken 是 OpenAI 出的 token 计数器，Rust 写的。昨天编 LangChain 时绕过了它（直接用 openai SDK 不走 langchain-openai），今天 AgentScope 要用，绕不过了。

tiktoken 用 setuptools-rust 构建（不是 maturin），但底层还是 Rust + PyO3。不走 `maturin build`，直接用 `cargo build`：

```bash
cd sources/tiktoken
RUSTFLAGS="-L <libpython路径> -l python3.11" \
PYO3_CROSS_PYTHON_VERSION=3.11 \
cargo build --target aarch64-unknown-linux-ohos --release --features python
```

产物是 `libtiktoken.so`，重命名为 `_tiktoken.cpython-311-aarch64-linux-ohos.so`，放到 `tiktoken/` 包目录。

35 秒编完，一次过。注意 `RUSTFLAGS` 里的 `-l python3.11`——**OHOS musl 的 dlopen 不继承符号**，这个坑昨天踩了两次，今天已经内化成肌肉记忆了，每个 `.so` 都自动加。

━━━━━━━━━━━━━━━━━━━━

◆ cryptography：这次的硬骨头

━━━━━━━━━━━━━━━━━━━━

cryptography 是四个需要编的包里最复杂的。它不是单一 Rust 项目——是一个 **Rust workspace（8 个子 crate）+ CFFI 生成的 C 绑定层 + 依赖 OpenSSL** 的混合怪物。

编译过程踩了 5 个坑，逐个来。

────────────────────

**坑 1：宿主机编译器混进来了**

```
_openssl.o is incompatible with aarch64linux
```

Rust 编译时会调 `cc` crate 编 C 代码。默认的 `cc` 是宿主机的 x86 gcc——编出来的 `.o` 当然在 aarch64 上用不了。

解法：用 `CC_aarch64_unknown_linux_ohos` 环境变量指定 OH 的交叉编译器。

────────────────────

**坑 2：找错了 Python 头文件**

```
pyconfig.h file not found
```

`cc` crate 找到的是宿主机 Python 3.10 的头文件。目标平台是 Python 3.11，头文件路径不一样。

解法：用 `CFLAGS_aarch64_unknown_linux_ohos` 加 `-I` 指向交叉编译的 Python 3.11 include 目录。

────────────────────

**坑 3：宿主机缺 cffi 模块**

```
No module named 'cffi'
```

cryptography 的 `build.rs` 会调宿主机的 python 运行 CFFI 代码生成器——这是在编译阶段，不是运行阶段。宿主机的 python 得装 cffi。

解法：`/tmp/python-host/bin/pip3.11 install cffi setuptools`。

────────────────────

**坑 4：OpenSSL 头文件展开失败**

```
Header expansion error
```

`OPENSSL_DIR` 环境变量被 shell 的 `&&` 截断了——写成多行命令时，`&&` 让前面的环境变量声明变成了独立命令。

解法：一行写完所有环境变量，不用 `&&` 分割。

────────────────────

**坑 5：linker cc not found**

PATH 被覆盖导致宿主机的 `/usr/bin/cc` 找不到了。不用 PATH 覆盖，改用环境变量精确指定。

────────────────────

**最终编译命令**长这样（省略了绝对路径）：

```bash
CC_aarch64_unknown_linux_ohos="<ohos-cc.sh>" \
CFLAGS_aarch64_unknown_linux_ohos="-I<OHOS Python 3.11 include>" \
OPENSSL_DIR=<openssl-shared路径> \
OPENSSL_STATIC=0 \
OPENSSL_NO_VENDOR=1 \
RUSTFLAGS="-L <libpython路径> -l python3.11" \
PYO3_CROSS_PYTHON_VERSION=3.11 \
PYO3_PYTHON=/tmp/python-host/bin/python3.11 \
maturin build --target aarch64-unknown-linux-ohos --release \
  --interpreter /tmp/python-host/bin/python3.11 --skip-auditwheel
```

33 秒编完。产物 6.0MB 的 `.so`。

**cryptography 的教训**：交叉编译 Rust+C 混合项目时，实际上有**两套工具链在同时工作**——宿主机工具链跑 build script（生成代码、展开头文件），目标平台工具链编最终产物。这两套不能混。`CC` 是宿主机的，`CC_aarch64_unknown_linux_ohos` 是目标平台的。搞混了就是"x86 的 .o 塞进 aarch64 的链接器"，报错信息看似莫名其妙，其实就是串台了。

━━━━━━━━━━━━━━━━━━━━

◆ regex 和 cffi：一行命令的事

━━━━━━━━━━━━━━━━━━━━

比起 cryptography，剩下两个 C 扩展简直是假期作业。

**regex**（2 个 .c 文件）：

```bash
$OHOS_CC -shared -fPIC -O2 \
  -I$PYTHON_INC -L$PYTHON_LIB -lpython3.11 \
  sources/regex-src/src/_regex.c sources/regex-src/src/_regex_unicode.c \
  -o regex/_regex.cpython-311-aarch64-linux-ohos.so
```

**cffi**（1 个 .c 文件 + libffi 静态库）：

```bash
$OHOS_CC -shared -fPIC -O2 \
  -DFFI_BUILDING=1 \
  -I$PYTHON_INC -I$LIBFFI_INC \
  -L$PYTHON_LIB -lpython3.11 -L$LIBFFI_LIB -lffi \
  cffi_src/src/c/_cffi_backend.c \
  -o _cffi_backend.cpython-311-aarch64-linux-ohos.so
```

都是一次过。libffi 是前天编 CPython 时就编好的静态库（`.a`），直接链进去，不需要额外 `.so`。

四个包的编译到此结束。tiktoken 35 秒，cryptography 33 秒（加踩坑时间大概 40 分钟），regex 和 cffi 各几秒。**真正花时间的不是编译本身，是搞清楚 cryptography 那五个坑。**

◆ 推上板子：又一轮"跑一次缺一个"

━━━━━━━━━━━━━━━━━━━━

4 个编好的 `.so` + 35 个纯 Python 包打成增量包，hdc 推上板子，叠加在昨天的 LangChain 部署包之上。

先验证编出来的 `.so` 能不能加载：

```
cryptography 47.0.0 ✅（Rust+OpenSSL 在 OHOS musl 上正常）
regex 2026.2.28 ✅（C 扩展正常）
cffi 1.17.1 ✅（C 扩展正常，libffi 静态链入）
tiktoken ✅（补了 tiktoken_ext/ 目录后正常）
```

四个 `.so` 全部加载成功。昨天的经验起作用了——每个都链了 `-lpython3.11`，没再出 symbol not found。

然后开始 `import agentscope`——

第一次：缺 sqlalchemy。这是个纯 Python 包（greenlet 是可选的 C 扩展，同步模式不需要），从 PyPI 下了 `py3-none-any` 的 wheel 解压推上去。
第二次：缺 multidict、yarl、frozenlist、propcache。这四个是 aiohttp 的依赖链，同样有纯 Python wheel。
第三次：缺 loguru。AgentScope 的日志库。
第四次：缺 bidict。python-socketio 的硬依赖。
第五次：缺 importlib-metadata。opentelemetry 需要它。

前后补了 10 个纯 Python 包。这种循环在移植工作里太正常了，不丢人。依赖树里的 import 是"按需加载"的——你不碰那个功能，它不报错。每往深走一步就冒出一个新的。好在全是纯 Python，从 PyPI 下载解压推上去就行，不用重新编译。

另外踩了一个 OH 特有的小坑：**OH 的 tar 不接受 `./` 前缀路径**。在 Linux 上 `tar czf xxx.tar.gz .` 打出来的包，路径是 `./agentscope/`、`./tiktoken/`……OH 上的 tar 解压时会报 `not under target`。换成 `tar czf xxx.tar.gz *` 打包（不带 `./` 前缀）就好了。小事，但卡你十分钟。

补完纯 Python 包，碰到了真正有意思的坑。

━━━━━━━━━━━━━━━━━━━━

◆ aiohttp 3.6.2：跨越三年的不兼容

━━━━━━━━━━━━━━━━━━━━

AgentScope 依赖 dashscope SDK（阿里通义的 Python 客户端），dashscope 顶层 `import aiohttp`。

aiohttp 有 C 扩展（multidict/yarl 等），PyPI 上最后一个纯 Python wheel 是 3.6.2——2020 年发布的，Python 3.8 时代的产物。

推上去一跑：

```
AttributeError: module 'asyncio' has no attribute 'coroutine'
```

Python 3.11 把 `@asyncio.coroutine` 装饰器删了。aiohttp 3.6.2 还在用。隔了三年，API 断代了。

但等一下——dashscope 真的**用** aiohttp 吗？看了一下 dashscope 的代码：同步 API 走 requests，异步 API 才走 aiohttp。AgentScope 调 DeepSeek 走的是 OpenAI 兼容接口（openai SDK），不走 dashscope 的异步通道。

**dashscope 只是在模块顶层 `import aiohttp`，但实际代码路径不经过它。**

这就是典型的**"启动时需要"和"运行时需要"的区别**。import 时 Python 会执行模块顶层代码，`import aiohttp` 失败整个 dashscope 就挂。但只要 import 不报错，后面实际调用时根本不走 aiohttp。

**解法：写 aiohttp stub。** 只提供 dashscope 导入时需要的符号（`ClientSession`、`ClientTimeout`、`WSMsgType` 等），全是空壳，实际调用时抛 `NotImplementedError`。

同样的思路用在了另外两个地方：

**rpds stub**——jsonschema 依赖 rpds-py（Rust 扩展），需要 `HashTrieMap`、`HashTrieSet`、`List` 三个容器类。用 Python 内置的 dict、set、tuple 包一层就行。关键是要加 `convert()` 类方法和 `update()` 方法——referencing 库启动时会调这两个。

**numpy stub**——AgentScope 有地方 `import numpy`，但只有音频处理和 embedding 缓存才真正用到。stub 在 import 时只打个 warning，不报错；真正调 numpy 函数时才抛异常。

**stub 思维的核心**：不是所有依赖都需要真正工作。很多 import 只是"声明式"的——模块顶层写了 `import xxx`，但实际执行路径根本不经过那段代码。**只要 import 不报错，就够了。**

这比把 aiohttp 从 3.6.2 移植到 Python 3.11、或者交叉编译 numpy 要省事一万倍。

◆ 复盘

━━━━━━━━━━━━━━━━━━━━

从依赖分析到板子上跑通，AgentScope 移植比 LangChain 快得多。LangChain 花了两天（大半天改动态链接 + 半天编包），AgentScope 一天搞定。

原因很简单：**前一天的基建投资开始产生回报。**

第一，20 多个包直接复用。pydantic、pydantic-core、openai SDK、httpx 这些硬骨头昨天全编过了。AgentScope 额外需要的 4 个原生扩展里，tiktoken 和 regex 加起来不到 1 分钟，cffi 几秒钟。只有 cryptography 花了点时间踩坑。

第二，"OHOS musl dlopen 不继承符号"这个坑已经变成条件反射——每个 `.so` 都自动加 `-lpython3.11`，不用再排查一轮。昨天第一次碰花了大半天，第二次碰花了 10 分钟，今天编 4 个新包，一次都没再犯。**踩过的坑确实会变成直觉。**

第三，stub 思维。aiohttp、rpds、numpy 三个依赖如果正经编，每个都是一天的工程量。但分析完实际代码路径后发现，它们都是"import 时需要，运行时不走"的。**区分"启动时需要"和"运行时需要"，是移植工作里最重要的判断。** 该糊弄的时候果断糊弄，把精力留给真正要编的东西。

第四，cryptography 的教训值得记住：交叉编译 Rust+C 混合项目时，**宿主机工具链和目标平台工具链是两套独立的系统**。宿主机跑 build script 生成代码，目标平台编最终产物。它们各自需要各自的编译器、头文件、库路径。搞混了就是"x86 的目标文件喂进 aarch64 的链接器"——报错信息看着玄乎，本质就是串台。

回顾一下两天的数据对比：

| | LangChain（第一天） | AgentScope（第二天） |
|:---|:---|:---|
| 需要交叉编译的原生扩展 | 4 个（pydantic-core, jiter, uuid-utils, xxhash） | 4 个（tiktoken, cryptography, regex, cffi） |
| 可复用的已编包 | 0（从零开始） | ~20 个（LangChain 积累的） |
| 需要写 stub 的 | 0 | 3 个（aiohttp, rpds, numpy） |
| 纯 Python 包 | ~20 个 | ~35 个 |
| 总耗时 | ~2 天（含 CPython 动态化改造） | ~1 天（含踩坑） |
| 最耗时的环节 | 排查 dlopen segfault（大半天） | cryptography 5 个坑（~40 分钟） |

第一天是开荒，第二天是收割。差别不在个人能力，在于**基础设施的复利效应**。

最后一个感受：两天编完两个 AI 框架（LangChain + AgentScope），OpenHarmony 上的 Python AI 生态从零到可用，核心原因是 **Python 生态的 fallback 文化**。大量库的 C/Rust 扩展只是加速层，纯 Python 版本作为后备始终保留。这意味着在新平台上，你只需要编少数几个没有 fallback 的硬依赖（pydantic-core、cryptography），剩下的都能自动降级。生态的韧性不在于每个包都完美移植，在于降级路径足够多。

stub 也是 fallback 的一种——不是让模块"真正工作"，而是让 import 链不断裂。在嵌入式场景里，你不需要 numpy 的全部功能，你只需要 `import numpy` 不报错。这种"够用就行"的务实态度，反而比"要做就做完美"走得更远。

```bash
hdc shell "cd /data/local/tmp && tar xzf langchain-ohos-deploy.tar.gz && rm langchain-ohos-deploy.tar.gz"

# 3. 推 AgentScope 增量包（叠加在基础包之上）
hdc file send build/agentscope-ohos-deps-v2.tar.gz /data/local/tmp/
hdc shell "cd /data/local/tmp/python-home/lib/python3.11/site-packages && tar xzf /data/local/tmp/agentscope-ohos-deps-v2.tar.gz && rm /data/local/tmp/agentscope-ohos-deps-v2.tar.gz"

# 4. 下载 CA 证书
wget -q https://curl.se/ca/cacert.pem
hdc file send cacert.pem /data/local/tmp/cacert.pem

# 5. 验证
hdc shell 'LD_LIBRARY_PATH=/data/local/tmp/lib \
  PYTHONHOME=/data/local/tmp/python-home \
  SSL_CERT_FILE=/data/local/tmp/cacert.pem \
  /data/local/tmp/python-home/bin/python3.11 -c "
import agentscope
print(f\"agentscope {agentscope.__version__} OK\")
"'
```

◆ 术语速查

━━━━━━━━━━━━━━━━━━━━

| 术语 | 解释 |
|:---|:---|
| AgentScope | 阿里开源的多智能体框架。Agent 之间能自主对话、协调、分工 |
| tiktoken | OpenAI 的 token 计数器。把文本拆成 token（模型实际处理的最小单位），Rust 写的 |
| cryptography | Python 生态最常用的加密库。底层是 Rust + OpenSSL，用于 HTTPS 证书验证、JWT 签名等 |
| cffi | C Foreign Function Interface，让 Python 调用 C 函数的桥梁。cryptography 的 Python 绑定层依赖它 |
| libffi | cffi 底层依赖的 C 库，处理不同 CPU 架构的函数调用约定 |
| setuptools-rust | Python 包的 Rust 扩展构建工具之一（另一个是 maturin）。tiktoken 用它 |
| cargo | Rust 的包管理器和构建工具，相当于 Python 的 pip + setuptools |
| cc crate | Rust 生态中用来编译 C 代码的库。cryptography 的 build script 用它编 OpenSSL 绑定 |
| CC_aarch64_unknown_linux_ohos | 环境变量，告诉 `cc` crate 交叉编译时用哪个 C 编译器 |
| PYO3_CROSS_PYTHON_VERSION | 环境变量，告诉 PyO3 目标平台的 Python 版本 |
| stub | 桩模块。只提供 import 需要的符号（类名、函数名），实际功能是空的或直接报错 |
| @asyncio.coroutine | Python 3.4 引入的协程装饰器，3.8 废弃，3.11 删除。aiohttp 3.6.2 还在用它 |
| dashscope | 阿里通义千问的 Python SDK。AgentScope 通过它调用通义系列模型 |
| rpds-py | Rust 实现的不可变数据结构（hash trie map/set）。jsonschema 依赖它 |
| workspace | Rust 中把多个 crate（包）组织在一起的方式。cryptography 有 8 个子 crate |
| fallback | 后备方案。C/Rust 扩展 import 失败时自动退回纯 Python 实现 |
