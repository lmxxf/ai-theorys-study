【Step 3.7 Flash】阶跃星辰的技术精读——和 DeepSeek V4 押的不是同一条路

━━━━━━━━━━━━━━━━━━━━

◆ 先说结论

━━━━━━━━━━━━━━━━━━━━

2026 年 5 月底，阶跃星辰（StepFun）发布了 Step 3.7 Flash——198B 总参数（196B 语言骨干 + 1.8B 视觉编码器 ViT），约 11B 激活参数/token，256K 上下文，288 个路由专家 + 1 个共享专家，Top-8 路由。

规模上和 DeepSeek V4 Flash（284B 总参数，13B 激活）是同一个重量级选手，两边都在打"参数量不到千亿俱乐部顶配、但性价比拉满"这张牌。但读完技术文档会发现一件更有意思的事：**这两个模型在几个核心架构决策上，押的是完全相反的赌注。**

V4 赌的是"未来上下文会越来越长，KV cache 存储会先爆掉"，把所有精力砸在 CSA + HCA 这套 KV cache 压缩机制上（第 166 期精读过，从 40.7GB 压到 228MB）。Step 3.7 Flash 赌的是"未来吞吐量和延迟才是硬指标"，注意力机制干脆放弃了压缩这条路，转而把 GQA 和投机解码（speculative decoding）焊死在一起。

这篇把 Step 3.7 Flash 的技术文档拆开讲，读的是官方博客原始 HTML、GitHub README、HuggingFace 的 config.json、以及它复用架构的 Step 3.5 Flash 论文（67 页），关键地方和 V4 做对比。

━━━━━━━━━━━━━━━━━━━━

◆ 1. 先弄清楚一件事：语言骨干不是全新设计

━━━━━━━━━━━━━━━━━━━━

config.json 里有个字段很关键：

```json
"text_config": {
  "model_type": "step3p5",
  "architectures": ["Step3p5ForCausalLM"]
}
```

**Step 3.7 的语言骨干，直接复用的是 Step 3.5 Flash 的架构类。** 3.7 相对 3.5 做的事情，本质是"给一个已经训好的纯文本模型加装原生视觉编码器"，而不是重新设计一遍语言模型。这意味着 Step 3.5 论文里对 attention、MoE、MTP、训练课程的描述，可以当作对 3.7 文本骨干的直接确认，不只是"同门师兄弟猜测"。

参数口径要小心区分——博客说的"198B / 11B 激活"是**多模态整体**口径（196B 语言 + 1.8B 视觉，激活量含视觉处理开销）；3.5 论文 Table 6 说的"198B / 13B 激活（含 MTP3）"是**纯语言模型**口径，还把 3 个 MTP 头也算进了总参数里。两个"198B"字面相同，统计范围不同，写文章、算显存的时候容易踩坑。

━━━━━━━━━━━━━━━━━━━━

◆ 2. 注意力机制：S3F1 vs CSA+HCA——两条相反的路

━━━━━━━━━━━━━━━━━━━━

**Step 3.7 怎么做的：**

45 层解码层，每 4 层一组，组内第 1 层是全注意力（full attention），后 3 层是滑动窗口注意力（sliding window attention，窗口固定 512 token）。这个排列比例是 3:1（滑窗:全量），论文里管它叫 **S3F1**。config.json 里体现为一个 `layer_types` 数组，全注意力层落在第 0、4、8……44 层（共 12 层），其余 33 层都是滑窗。

KV cache 用的是标准 GQA（Grouped Query Attention），**8 个 KV head，head_dim 128，没有任何低秩压缩**。更值得注意的是 query head 数量不对称：全注意力层 64 个 query head，滑窗层反而升到 96 个。论文管这个设计叫"free lunch"——滑窗场景下 query:KV 的比例已经到 12，计算仍然停留在 IO-bound 区间（显存带宽是瓶颈，不是算力），多加 query head 几乎不增加延迟，Table 7 给出的数字是：256K 上下文下，decode/prefill 延迟只多了 0%~8%。

RoPE 参数也按层区分：全注意力层 theta = 5,000,000、只旋转 128 维里的 64 维（partial rotary factor = 0.5）；滑窗层 theta = 10,000、旋转全部 128 维。基础训练只做到 128K，往 256K 扩展靠的是分段线性频率插值（config 里字段名叫 `yarn_only_types`，但类型实际是 `llama3` 式插值，不是严格意义上的 YaRN），而且**只用在全注意力层**——滑窗层窗口本来就只有 512，谈不上外推。

**为什么不学 V4 做 KV cache 压缩？** 论文原文说得很直接：他们评估过 linear attention / 类 MLA 的压缩路线，但这类方案的 state-update 机制会让投机解码的 draft tree 并行验证变复杂；标准 GQA 保留了完整的注意力语义，通过 KV masking 就能天然兼容并行验证。

────────────────────

💡 翻译成人话：V4 的算盘是"上下文会长到存不下，先把存储砍到骨头"；Step 3.7 的算盘是"上下文能装下就行，真正卡脖子的是吐字速度，所以宁可多存一点 KV cache，也要保证投机解码能跑起来"。**一个省的是显存，一个省的是时间。**

────────────────────

━━━━━━━━━━━━━━━━━━━━

◆ 3. Head-wise Gated Attention vs Attention Sink——同一个毛病，两种药方

━━━━━━━━━━━━━━━━━━━━

166 期讲过 V4 的 Attention Sink：标准 softmax 要求所有权重加起来必须等于 1，哪怕历史 token 全都不相关，模型也被迫"投票"。V4 的解法是在 softmax 分母里加一个可学习的 `exp(sink)` 项，相当于给模型一张"弃权票"，但这张票是**固定参数，和当前输入无关**。

Step 3.7 用的是另一套：**Head-wise Gated Attention**。每个注意力头有一个轻量的门控标量：

```
g_i = σ(w_gate^T x_i)
```

σ 是 sigmoid，w_gate 是这个 head 专属的一组可学习权重，x_i 是当前 token 的输入表示。这个门直接乘在这个 head 的输出上。数学上等价于给 softmax 分母引入一个"输入相关的弃权票"——不是固定给一张空白票，而是**每个 token 自己决定这张票投多少**。

论文自己做过消融实验（100B-A10B 规模的对照模型）：动态门控比固定 sink token 平均高 1.9 分（62.5 → 64.4）。

────────────────────

💡 类比一下：V4 的 sink 像单位食堂门口永远摆着一个意见箱，谁都能投，但箱子本身不看你是谁。Step 3.7 的门控像一个会看人下菜碟的保安——同样是"要不要把这一票算数"，但每个 token 走到门口，保安都会重新打量一眼。

────────────────────

━━━━━━━━━━━━━━━━━━━━

◆ 4. MoE 路由：都学了 DeepSeek 的作业，但补了一道题

━━━━━━━━━━━━━━━━━━━━

**基础参数：**

| | Step 3.7 Flash | DeepSeek V4 |
|---|---|---|
| 路由专家数 | 288 | 256（V4-Flash）/ 384（V4-Pro）|
| 共享专家数 | 1 | 1 |
| Top-k | 8 | 6 |
| 专家中间维度 | 1280 | 2048（Flash）/ 3072（Pro）|
| 路由打分函数 | sigmoid（×3.0 缩放）| sqrt(softplus(x)) |
| 门控计算精度 | 强制 FP32 | — |

打分函数不是这里的重点，真正有意思的是负载均衡策略。DeepSeek 早年提出的 loss-free load-balancing（不用辅助损失，纯靠 bias 项动态调整路由）已经成了行业标准做法，Step 3.7 也直接沿用了。但论文里指出一个 DeepSeek 方案没解决的问题：**loss-free 只保证全局 token 分配均衡，不保证 expert-parallel（EP）意义上、每个 GPU rank 拿到的负载均衡**——大规模分布式部署时，如果某个 rank 恰好揽下了一堆热门专家，就会拖慢整个推理批次（straggler 问题）。

所以 Step 3.7 在 loss-free 基础上又加了一层 **EP-group Balancing Loss**：把专家按 EP rank 分成 G 组，显式惩罚组间利用率不均。这个损失只在预训练阶段用，退火阶段就关掉了。算是站在 DeepSeek 的方案上，补了一个"工程部署"维度的洞。

**还有个细节可以对着看：** V4 前 3 层用的是固定的 Hash Routing（token ID 直接查表分配专家，不经过学习的路由网络）；Step 3.7 干脆更彻底——**前 3 层不是 MoE，是纯 dense FFN，压根不路由**。两家不约而同觉得"最底层不适合用学习出来的路由"，但给出的药方一个是"路由固定死"，一个是"直接不路由"。

**训练稳定性上的差异化素材：** Step 3.7 论文详细描述了两类 MoE 训练崩溃模式——"Expert Collapse Beyond Routing Collapse"（专家侧激活消失，哪怕路由统计看着健康）和"Localized Activation Blow-up"（深层少数专家激活范数爆炸）。缓解手段用的是 activation clipping（专家 FFN 内部限幅），config.json 里 `swiglu_limits` 数组只在倒数第 2、3 层非零——说明这个问题专挑深层 MoE 发作，头几层反而没事。

━━━━━━━━━━━━━━━━━━━━

◆ 5. 精度路线：V4 把量化焊进训练，Step 3.7 留到部署阶段

━━━━━━━━━━━━━━━━━━━━

V4 走的是 FP4 QAT（量化感知训练）——训练时就让模型习惯 FP4 的量化误差，99.9% 参数用 Muon 优化器更新，推理原生 FP4×FP8 混合精度（第 166 期详细拆过这套流程）。

Step 3.7 反过来：**整个训练过程全程 BF16 混合精度，全文没有一处提到训练阶段用了 FP8 或 FP4。** 优化器同样是 Muon（而不是 AdamW），但论文特别提到一个数值稳定性坑——Muon 里的 Newton-Schulz / Polar Express 正交化步骤对精度极其敏感，在 bf16 下偶尔会触发不可恢复的 loss spike，他们的解法是**只把这一步（状态和中间量）转成 float16**，训练的其余部分保持 BF16。

量化被推到了部署阶段，走的是 PTQ（训练后量化）路线：官方直接发了 BF16 / FP8 / NVFP4 / GGUF 四种权重仓库，NVFP4 版本靠 NVIDIA 的 modelopt 工具做后量化，配合 FP8 KV cache。README 给了一组具体的吞吐数据（GB200，张量并行 4，约 250 token 输入 / 1.6K token 输出的场景）：

| 并发数 | NVFP4 + MTP | NVFP4 无 MTP | 加速比 |
|---|---|---|---|
| 8 | 1309 tok/s | 1155 tok/s | 1.13× |
| 32 | 4391 tok/s | 3480 tok/s | 1.26× |
| 64 | 8229 tok/s | 5667 tok/s | 1.45× |

质量几乎没损失：GPQA Diamond avg@16，NVFP4+MTP 是 77.81%，不量化基线是 78.41%，官方自己说"在统计噪声范围内"。

（官方特别注明这组数字是"short-prompt, decode-heavy"场景，不代表长上下文 prefill 的表现——这个我照实转述，不替它过度解读。）

━━━━━━━━━━━━━━━━━━━━

◆ 6. MTP：V4 浅尝，Step 3.7 重注

━━━━━━━━━━━━━━━━━━━━

166 期讲过 V4 的 Multi-Token Prediction，depth = 1（训练时多预测下下一个 token 这一件事），DeepSeek 自己的消融实验显示这是收益/成本的甜蜜点，没有再往上加。

Step 3.7 走得更远，是 **MTP-3**（3 个预测头）。每个 MTP 头由 1 个滑窗注意力 block + 1 个 dense FFN 组成，训练时先专门优化第一个头，另外两个头在轻量后训练阶段从第一个头克隆出来再联合微调。目的很直接——服务上一节提到的投机解码：主模型正常生成的同时，3 个 MTP 头并行"抢跑"猜接下来几个 token，猜中就直接采纳，省掉真正的自回归步骤。这也是为什么它在 64 并发下能吃到 1.45 倍的吞吐提升。

V4 的 MTP 是"训练时的免费信号，推理时可以不用"；Step 3.7 的 MTP 是"推理时真正要吃满的加速引擎"。同一个技术名字，指向的是两种不同的产品目标。

━━━━━━━━━━━━━━━━━━━━

◆ 7. 视觉编码器：小 ViT + 工具调用，不是堆参数

━━━━━━━━━━━━━━━━━━━━

3.5 Flash 是纯文本模型（论文 Table 5 明确标注"Multi-modal: ✕"），视觉这块是 3.7 全新加的，只能从 config.json 里读：

```json
"vision_config": {
  "model_type": "perception_encoder",
  "image_size": 728, "patch_size": 14,
  "width": 1536, "layers": 47, "heads": 16
}
```

1.8B 参数的 ViT，接入方式是标准路线：ViT 编码 → projector（`understand_projector_stride: 2`，做下采样）→ 拼成 169 个图像 token 塞进文本序列，走的是类似 LLaVA 的接法，不是什么复杂的双塔架构。`model_type` 叫 `perception_encoder`，这个命名很可能是在指向 Meta 的 Perception Encoder（PE）架构——但这是我根据命名习惯的推断，官方文档没有一句话明说，供你自己核实。

1.8B 相对 196B 的语言骨干，是个很小的比例。博客对此的解释思路不是"我们要把 ViT 做大"，而是给模型配了 Visual Search 工具和 Python 工具，能对图像做裁剪（crop）、缩放（zoom）、画框这类操作来弥补小 ViT 在细节感知上的局限——**用工具调用换视觉参数量**，而不是死磕编码器本身的规模。至于预训练阶段视觉-语言对齐具体怎么做的、用了多少数据，博客和论文都没有披露，这块只能算空白。

━━━━━━━━━━━━━━━━━━━━

◆ 9. 开源程度：两种不同的诚意

━━━━━━━━━━━━━━━━━━━━

DeepSeek V4 的 GitHub 仓库直接放了 `model.py` 推理代码，架构细节读代码就能确认，166 期能精读到 `Compressor` 类这种实现层面的东西，靠的就是这份代码。

Step 3.7 的 GitHub 仓库（`stepfun-ai/Step-3.7-Flash`）**只有 README、LICENSE、assets，没有一行推理代码**。真正的模型实现文件（`modeling_step3p7.py`、`configuration_step3p7.py`）通过 HuggingFace 的 `auto_map` 字段指向，需要 `trust_remote_code=True` 才能加载——代码在，但不在 GitHub 这个更显眼的门面上，本文没来得及去 HuggingFace 仓库里逐行核实这份代码和 DeepSeek 那份的可读性差距。

权重本身开得很全（BF16/FP8/NVFP4/GGUF 四种格式，Apache 2.0 协议），配套还维护了一个 `llama.cpp` fork 和专属的 vLLM Docker 镜像。**权重开源的诚意不打折，但"架构透明度"这件事上，DeepSeek 摆在 GitHub 首页的 model.py，确实比藏在 HuggingFace auto_map 里的代码更方便外部审查。**

━━━━━━━━━━━━━━━━━━━━

◆ 一点自己的感受

━━━━━━━━━━━━━━━━━━━━

读完这两份文档，最大的感受不是"谁更强"，是**两个团队看到了同一个瓶颈，选了完全不同的下注方向**。

DeepSeek 看到的瓶颈是显存——上下文一旦拉长，KV cache 会先把你的机器撑爆，所以 CSA/HCA 两刀下去，压掉 99% 存储。阶跃看到的瓶颈是延迟——就算存得下，吐字慢用户也会跑，所以宁可留着标准 GQA 的存储开销，把资源砸在投机解码这条推理加速链路上（MTP-3 + 门控注意力 + EP-group 负载均衡，每一环都在为"吃满硬件吞吐"服务）。

这两个赌注不是谁对谁错的问题——大概率是各自的产品场景不一样：一个更在意长文档/长代码库场景能不能装得下，一个更在意 agent 高频调用工具时响应能不能跟得上。技术选型从来不是纯粹的"哪个更先进"，是"我更怕哪种失败模式"。

━━━━━━━━━━━━━━━━━━━━

技术名词速查

- **S3F1**：Step 3.7 Flash 的注意力层排列方式，每 4 层一组，1 层全注意力（Full Attention）+ 3 层滑动窗口注意力（Sliding Window Attention，窗口=512）
- **GQA（Grouped Query Attention）**：多个 query head 共享同一组 KV head 的注意力变体，介于"每个 head 独立 KV"（MHA）和"所有 head 共享 1 组 KV"（MQA）之间
- **投机解码（Speculative Decoding）**：用一个小的"草稿"机制先猜接下来几个 token，猜中就直接采纳，猜不中再退回正常逐字生成，用来加速自回归推理
- **Head-wise Gated Attention**：Step 3.7 的注意力门控机制，每个 head 根据当前输入算一个 sigmoid 标量门，动态调制该 head 的输出
- **MTP（Multi-Token Prediction）**：训练/推理时让模型同时预测未来多个 token（不只是下一个），Step 3.7 用了 3 个预测头（MTP-3）
- **QAT / PTQ**：量化感知训练（Quantization-Aware Training，训练时就模拟低精度误差）/ 训练后量化（Post-Training Quantization，训练完之后再量化）
- **Loss-free Load Balancing**：DeepSeek 提出的 MoE 负载均衡方法，不用额外的辅助损失函数，靠给专家动态加减 bias 项来调整路由倾向
