【AI可解释性】训练三层Mistral-7B模型的SAE——解开结构与语义的纠缠

━━━━━━━━━━━━━━━━━━━━

◆ 从第 64 期说起

━━━━━━━━━━━━━━━━━━━━

在第 64 期《为什么你的 Prompt 越长，AI 越听话？》（ https://mp.weixin.qq.com/s/yhrmzZhfUOn2x6cjQ-hgtg ）里，我们用 DeepSeek-V3 做了一组实验，得出了一个干净的结论：**AI 输出 JSON 时，结构处理和语义处理走的是两条独立的回路——结构在浅层，语义在深层，互不干扰。** 输入完全无意义的字符串，模型照样输出结构完美的 JSON；JSON 语法故意搞坏，语义连贯度不受影响。

但当我们试图在 Mistral-7B 上用线性方法把这两条回路"物理拆开"时，遇到了困难——要么完全打不穿，要么结构和语义一起崩，中间没有过渡。

为什么在 DeepSeek-V3 上看得清清楚楚的"两条独立回路"，到了 Mistral-7B 上就拆不开了？因为 Mistral-7B 用了 Sliding Window Attention——结构信息和语义信息被迫挤在同一个向量里逐 token 传递，在原始 4096 维空间里是纠缠的。你用线性刀切，切不开纠缠的东西。

**需要一种更精细的工具——SAE（Sparse Autoencoder，稀疏自编码器）。**

━━━━━━━━━━━━━━━━━━━━

◆ SAE 是什么：升维 → 稀疏 → 降维

━━━━━━━━━━━━━━━━━━━━

模型内部每一层都在传递一个 4096 维的向量（残差流，residual stream）。这个向量里编码了模型此刻"想到的一切"——语法、语义、事实知识，全搅在一起。**4096 个数字，混着成千上万个概念。** 一个概念分散在多个维度里，一个维度也参与多个概念，这叫**叠加**（superposition）。

SAE 的思路：把 4096 维的混合向量升维到一个更大的空间（比如 16384 维，即 4 倍扩展），逼迫大部分维度为零，只留几十个非零值。每一个非零位置就对应一个"被激活的特征"。然后再降维回 4096，要求重建尽量不丢信息。

结构很简单：

```python
class SAE:
    def __init__(self, d_in=4096, d_sae=16384):
        self.W_enc = Linear(d_in, d_sae)    # 编码器：4096 → 16384
        self.W_dec = Linear(d_sae, d_in)    # 解码器：16384 → 4096
        self.b_enc = zeros(d_sae)            # 编码偏置
        self.b_dec = zeros(d_in)             # 解码偏置
        # W_enc 和 W_dec 是两个核心矩阵，b_enc 和 b_dec 是偏置向量（线代里 y=Ax+b 的 b）
        # b_dec 出现两次：编码时先减掉（中心化），解码时再加回来——这是 SAE 的特殊设计

    def forward(self, h):
        # h: 某层残差流向量, shape [batch, 4096]
        pre_act = W_enc × (h - b_dec) + b_enc   # 矩阵乘法
        z = activation(pre_act)                  # 稀疏激活，16384 维
        h_prime = W_dec × z + b_dec              # 重建回 4096 维
        return h_prime, z
```

三步走：

**1. 减去解码偏置，编码到高维空间**

```
pre_act = W_enc × (h - b_dec) + b_enc
```

输入 `h` 是 4096 维，编码后 `pre_act` 是 16384 维——升了 4 倍。升维的目的是给叠加的概念腾出空间，让每个概念可以占一个独立维度。（Anthropic 原始论文用过 16 倍扩展到 16384 维，但 4 倍扩展对可解释性分析已经够用，特征数更少也更容易人工检查。）

**2. 激活函数逼迫稀疏**

```
z = activation(pre_act)    # 16384 维，但只有几十个非零
```

16384 个维度里，激活函数会把绝大多数压成 0，只留下几十个非零值。比如位置 #1000 只在 JSON 方括号的 token 上非零，那它就是"JSON 方括号特征"。

**3. 解码回原始维度**

```
h' = W_dec × z + b_dec
```

用稀疏的 z 重建出 h'，要求尽量接近原始的 h。重建得好，说明那几十个非零特征确实捕捉到了关键信息。

一句话：SAE 把 4096 维的"浓缩混合物"展开到 16384 维的"稀疏表示"，每个非零位置是一个可解释的特征。训练目标是让这个展开-压缩的过程不丢信息。

◆ 插播：三种"范数"速查

━━━━━━━━━━━━━━━━━━━━

后面的公式会反复出现 L0、L1、L2 这三个术语。它们都叫"范数"（norm），就是衡量一个向量"有多大"的不同方式。大学线代一般只教 L2，L0 和 L1 是机器学习领域用得多的。一次讲完，后面不再解释。

假设有个向量 z = [3, 0, -1, 0, 0.5]：

```
L0 范数 ‖z‖₀ = 3        → 数有几个非零值（3, -1, 0.5 三个）
                           不是真正的数学范数，但大家都这么叫

L1 范数 ‖z‖₁ = 4.5      → 所有分量绝对值之和：|3|+|0|+|-1|+|0|+|0.5| = 4.5

L2 范数 ‖z‖₂ = √(9+0+1+0+0.25) ≈ 3.2
                         → 就是你熟悉的"距离公式"（勾股定理的高维版）
```

本文关心的：**L0 越小 = 非零特征越少 = 越稀疏 = 越容易解释。** SAE 的三代激活函数，本质上都是在想办法把 L0 压下去。

━━━━━━━━━━━━━━━━━━━━

◆ 损失函数：两股力量的拉扯

━━━━━━━━━━━━━━━━━━━━

```
Loss = ‖h - h'‖² + λ · 稀疏惩罚(z)
```

其中 z 是前面编码器输出的 16384 维稀疏向量，h' 是用 z 重建回来的 4096 维向量。两项，两个目标，互相矛盾：

- **‖h - h'‖²（重建误差）**：h' 要尽量接近 h。逼模型"别丢信息"
- **λ · 稀疏惩罚(z)**：z 要尽量稀疏（非零值尽量少）。逼模型"用最少的特征解释输入"

只优化重建，所有 16384 个维度都非零——信息没丢，但没法解释。只优化稀疏，z 全置零——完美稀疏，但什么都丢了。

**好的 SAE 是在两者之间找到平衡：用 50-200 个非零特征就能准确重建 4096 维向量。**

━━━━━━━━━━━━━━━━━━━━

◆ 三代激活函数：SAE 的核心演进

━━━━━━━━━━━━━━━━━━━━

激活函数决定了"怎么让 z 变稀疏"。三代方案各自解决了上一代的痛点。

────────────────────

【第一代：ReLU + L1 惩罚】

```
z = ReLU(W_enc × h + b_enc)       # 负值变 0，正值保留
Loss = ‖h - h'‖² + λ · ‖z‖₁      # 重建误差 + L1 惩罚
```

ReLU 把负值干掉，L1 惩罚继续压正值。但 L1 惩罚的是绝对值之和——它鼓励"小"，不鼓励"零"。一个值从 0.01 压到 0 和从 1.01 压到 1.00，对 L1 来说省的都是 0.01，没有额外奖励。结果就是很多特征被压到很小（比如 0.001）但不是精确的零——稀疏度不够干净。而且 λ 是全局标量，对所有 16384 个特征施加相同的压力。弱特征被 L1 直接压死，强特征被削了一层降低重建精度。**L0（非零特征数）和重建误差高度耦合**，你没法独立控制。

────────────────────

【第二代：TopK】

既然 L1 控制不好非零特征数，直接硬性规定"只留 K 个"：

```
z = TopK(W_enc × h + b_enc, K)    # 只保留最大的 K 个值，其余全部置零
Loss = ‖h - h'‖²                   # 不需要 L1 惩罚了
```

稀疏度在前向计算时就硬性保证了（TopK 直接把多余的值归零），所以损失函数里不再需要稀疏惩罚项，只管"重建得准不准"。L0 直接等于 K，完全可控。**L0 和重建误差解耦了。**

问题：TopK 不可导。排第 K 名和第 K+1 名的特征值可能只差 0.001，但一个保留一个杀死——边界处梯度信号极弱。而且 K 全局固定——简单 token 用不了 100 个特征，复杂 token 又可能不够。

────────────────────

【第三代：JumpReLU】

Anthropic 在 2024 年的 Scaling Monosemanticity 论文里用的方案。**每个特征自己决定"多大算激活"。**

```
pre_act = W_enc × (h - b_dec) + b_enc
z = H(pre_act - θ) ⊙ pre_act        # ⊙ 是逐元素相乘
Loss = ‖h - h'‖²                     # 和 TopK 一样，只管重建误差
# H 是 Heaviside 阶跃函数，就是个开关：输入 > 0 输出 1，输入 ≤ 0 输出 0
# pre_act、θ、H(pre_act - θ) 全是 16384 维向量，逐元素操作，没有矩阵乘法
# 所以 H(pre_act - θ) 的意思是：每个特征各自比较，超过自己的阈值 → 1，没超过 → 0
# θ 是可学习的——每个特征训练时自动学出自己的阈值
# 最后 ⊙ pre_act：超过阈值的保留原值，没超过的变零
# 稀疏度由 θ 自动控制（没超过阈值的直接归零），不需要 L1 惩罚
```

JumpReLU 一共有 5 个可训练参数：

```
W_enc  16384 × 4096   编码器矩阵
W_dec  4096 × 16384   解码器矩阵
b_enc  16384 维        编码偏置
b_dec  4096 维         解码偏置
θ      16384 维        每个特征的激活阈值（JumpReLU 独有）
```

前四个和普通自编码器一样，θ 是 JumpReLU 独有的——训练过程中每个特征自动学出"多大算激活"的门槛。Heaviside 阶跃函数本身不可导（0/1 跳变没有斜率），但训练时用平滑函数近似，梯度下降照样能跑。细节不展开，知道"能训"就行。

JumpReLU 解决了前两代的所有问题：

| | ReLU+L1 | TopK | JumpReLU |
|---|---|---|---|
| 稀疏度控制 | λ 全局控制，耦合 | K 硬性截断，固定 | θ 每个特征独立学习 |
| L0 可控性 | 间接（调 λ） | 精确（= K） | 自适应（θ 自动调节） |
| 梯度信号 | 有（但 L1 干扰大特征） | 边界处弱 | tanh 近似保证梯度 |
| 灵活性 | 低 | 中 | 高（简单 token 少激活，复杂 token 多激活） |

ReLU+L1 用一把力压所有人，TopK 用一条线切所有人，JumpReLU 让每个人自己决定要不要站起来。

◆ 训练数据从哪来

━━━━━━━━━━━━━━━━━━━━

SAE 最优雅的地方：**不需要任何标注。**

```
1. 拿一个已经训好的大模型（我们用 Mistral-7B）
2. 喂一堆通用文本（OpenWebText）
3. 模型跑 forward pass，收集某一层的残差流向量
   → 每个 token 产生一个 4096 维向量
   → 100 万个 token 就是 100 万个训练样本
4. SAE 的任务：学着重建这些向量
```

完全无监督。SAE 不知道什么是"JSON"——它只是在学"怎么用最少的特征重建这些向量"。训完之后，你去检查每个特征在什么文本上激活，才发现"这个特征专门响应 JSON 的花括号"。

**特征的含义是事后看激活模式猜的，不是训练出来的。** Feature 1000 在一堆 `[` 上亮了，你说它是"方括号特征"——合理，但本质是归纳，不是证明。

◆ 给 Mistral-7B 训 SAE：三层实战

━━━━━━━━━━━━━━━━━━━━

Mistral-7B 一共 32 层。我们在 DGX Spark（NVIDIA GB10, 128GB 统一内存）上，选了第 8 层（浅层）、第 16 层（中层）、第 22 层（深层）各训了一个 SAE。没选最后几层（L30、L31），是因为越靠近输出层，信息越收敛到"下一个 token 是哪个字"的具体预测上，SAE 拆出来的特征反而不好解释。L22 恰好在"已经有格式决策但还没坍缩到具体 token"的位置。社区标准工具是 **SAELens**（Python 库，pip 直接装），封装了数据收集、SAE 架构、训练循环和评估指标。

**复现所需的全部信息：**

```
模型：mistralai/Mistral-7B-Instruct-v0.1（HuggingFace 直接下载，7B 参数，32 层，hidden_dim=4096）
数据集：Skylion007/openwebtext（Reddit 高赞链接爬的英文网页文本，约 800 万篇文档，什么都有：新闻、博客、技术文章。OpenAI 训 GPT-2 用的 WebText 的开源复现版）
SAE 目标层：Layer 8 / Layer 16 / Layer 22（blocks.N.hook_resid_post，残差流）
SAE 架构：JumpReLU（Anthropic Scaling Monosemanticity 同款）
字典大小：16384（4x expansion，即 4096 → 16384）
训练 tokens：~4900 万（OpenWebText 子集）
训练步数：12000
batch size：4096 tokens
学习率：5e-5（常数 + 1000 步 warmup + 末尾 20% 衰减）
模型精度：float16（autocast），SAE 精度：float32
框架：SAELens 6.39.0 + TransformerLens 2.16.1
训练时长：约 10 小时 / 层（单卡）
```

权重和代码全部开源（链接见文末）。

【探索：16384 个特征里，哪些是 JSON 的？】

到这里 SAE 已经训完了——它学会了怎么把 4096 维向量拆成稀疏特征再拼回来，但我们还不知道这 16384 个特征到底代表什么。接下来就是喂不同类型的文本进去，看哪些特征在哪类文本上亮，摸清楚它们各自编码了什么。

做法：写脚本分别喂 JSON 文本和英文文本过 Mistral-7B，收集目标层的激活，过 SAE encode 得到 16384 维稀疏向量，然后统计每个特征在不同类型 token 上的平均激活值，按比值排序——比值高的就是"偏好某类文本"的特征。

以 Layer 16 为例，先看基础指标：

```
1. 重建质量：MSE = 0.0116，Explained Variance = 0.96（含 BOS token）
   → 排除 BOS 后 EV 因方差骤降而不稳定（0.47），但 MSE 几乎不变（0.0114→0.0116）
   → 重建质量没问题，SAE 编码再解码丢的信息很少

2. 稀疏度：L0 = 57~79（每个 token 平均只激活 16384 的 0.3%~0.5%）
   → 16384 个特征里只亮了几十个

3. 特征可解释性：JSON 和自然语言的 top-20 特征几乎零重叠
   → SAE 确实把不同类型的概念分到了不同维度上
```

我们喂了 JSON 文本和普通英文文本，看 SAE 提取出的特征有什么不同。

JSON 侧 top-20 特征里，能直接看出语法结构：

```
Feature 1000  → 只在 [ 上激活      → 数组开头特征
Feature 11062 → 只在 {" 上激活     → JSON 对象开头特征
Feature 4765  → 只在 } }} }}} 上亮 → 闭合括号特征
Feature 2918  → 在 " 和 ": 上亮    → 键值分隔符特征
Feature 200   → 在 id, title, tags 上亮 → JSON 键名特征
```

自然语言侧 top-20 完全是另一个世界：

```
Feature 5088  → In, The, Machine  → 句首词特征
Feature 16167 → early, morning    → 时间概念特征
Feature 8350  → morning, night, light → 时间/光线语义簇
```

**两边的 top-20 几乎没有交集。** JSON 那边全是语法标点，自然语言这边全是语义概念。SAE 把它们分到了不同的"频道"上。

更硬核的证据：我们算了每个特征在 JSON 标点 token 上的平均激活值 vs 非标点 token 上的平均激活值，取比值。前 10 名的比值全部超过 1 亿——这些特征**只在 JSON 标点上亮，在非标点上完全不亮**。

```
Feature 2122 → 只在 ": 上激活，其他一律为零 → 纯 JSON 键值分隔符特征
Feature 9353 → 只在 {" 上激活 → 纯 JSON 对象开头特征
Feature 9582 → 只在 "] }] 上激活 → 纯 JSON 闭合特征
```

在原始 4096 维空间里，结构信息和语义信息混在同一批维度里，线性方法拆不开。SAE 升维到 16384 维后，结构和语义自动分到了不同的特征上。

────────────────────

【干预：关掉 JSON 结构特征，模型会怎样？】

在 Layer 16 的 hook 点，把模型激活过一遍 SAE（encode → 把目标特征 clamp 到 0 → decode），再用修改后的激活替换原始激活，让模型继续生成。

三级干预力度：

```
轻度：关掉 1 个 JSON 标点偏好特征（Feature 2122，专门响应 ": 键值分隔符）
中度：关掉 3 个（再加上 Feature 9353 响应 {"，Feature 1000 响应 [）
重度：关掉 top-10 全部 JSON 标点偏好特征
```

测了 7 个 prompt，都是让 Mistral-7B 生成 JSON 对象。例如：

```
语义类（模型知道该填什么内容）：
  car:        "Generate a JSON object representing a car with fields: make, model, and year."
  person:     "Generate a JSON object representing a person with fields: name, age, and city."
  book:       "Generate a JSON object for a book with fields: title, author, pages, and published."
  restaurant: "Generate a JSON object for a restaurant with fields: name, cuisine, rating, and address."
  school:     "Generate a JSON object for a school with fields: name, address, founded,
               and a teachers array of 3 objects each with name and subject."

无意义类（字段名是瞎编的，模型只能编内容）：
  blonf:      "Generate a JSON object representing a blonf with fields: zrelk, grimbat, and quav."
  trelm:      "Generate a JSON object representing a trelm with fields: plovk, dranq, and blixt."

（每个 prompt 末尾都加了 "Respond with ONLY the JSON, no explanation."）
```

5 个语义类 + 2 个无意义类。结果分成了两种模式——

**模式 A：完全崩溃（car、school、blonf 这三个 prompt）**

轻度干预（只关 1 个特征）就直接输出空 code block，结构和语义一起崩——说明这些 prompt 的 JSON 生成强依赖 Layer 16 的结构特征，一碰就碎。

**模式 B：渐进退化（person、book、restaurant 这三个 prompt）** ← 关键发现

轻度/中度干预后 JSON 结构依然合法（json.loads 通过），只是缩进减少。重度干预后结构才崩，但语义内容仍可辨识：

**book prompt + 重度干预：**

```
"title": "The Great Gatsby", "author: "F. Scott Fitzgerald "pages: 176
```

引号配对乱了，冒号位置不对，json.loads 直接报错（结构崩了）。但内容——书名 The Great Gatsby，作者 F. Scott Fitzgerald，页数 176，全对（语义内容仍可辨识）。

**person prompt + 重度干预：**

```
"name": "John", "age: 25 "city": "New York"
```

同样的规律：结构破损，语义内容仍在。

**这两种模式本身就是一个发现**：同样的干预操作，不同 prompt 的反应完全不同。模式 A 的 prompt 可能把全部结构生成押在了 Layer 16，没有冗余路径；模式 B 的 prompt 则有其他层的冗余支撑，所以能撑住轻度干预，重度干预时也只崩格式不崩内容。

**模式 B 证明了关键一点：在 Mistral-7B 上，SAE 确实能把结构和语义拆开——线性方法做不到的事，升维解耦做到了。**

────────────────────

【三层对比：浅层管标点，深层管规划】

上面只干预了第 16 层。我们在第 8 层和第 22 层上做了同样的实验，结果差异巨大。

先看一个发现——**不同层的 JSON 特征类型完全不同**：

```
第 8 层（浅层）的 JSON 特征：
→ 全是标点符号：引号 "、冒号 :、逗号 ,、闭合括号 }
→ 编码的是"这里要放个引号"这种语法级信息

第 22 层（深层）的 JSON 特征：
→ 出现了语义级概念：system、credentials、Post、Alice
→ 编码的是"这个值应该是什么类型的内容"
```

浅层管"怎么写标点"，深层管"该用什么格式输出"。Mistral-7B 内部有分工。

分别对三层做干预：

```
干预第 8 层（浅层，关掉标点特征）：
→ 模型输出 {{{[[[((( 重复结构符号，陷入死循环
→ 结构崩了，语义也崩了 → 等于把嘴缝上了

干预第 16 层（中层，关掉标点特征）：
→ 80% 的 prompt 输出空 code block，大面积崩溃
→ 和浅层类似，干预太底层

干预第 22 层（深层，关掉 top-10 JSON 偏好特征——在深层这些特征编码的是"该用 JSON 格式输出"这种规划级信息）：
→ restaurant: "name": " The Fat Cat" "cuisine": " New American" "rating": "3.5"
→ school: "name": " Lincoln High School", "address": " 12th Avenue", "founded": "1968,,"
→ 无论是 restaurant 还是 school，json.loads 都报错（格式坏了），但内容全在（餐厅名、菜系、学校名、地址都没丢）
```

注意 school 这个 prompt——前面在 Layer 16 单层干预时，它属于模式 A（关 1 个特征就完全崩溃）。但换到 Layer 22 干预，它不崩了，语义内容完整保留。同一个 prompt，在中层是脆弱的，在深层是有韧性的——因为深层关掉的是"格式规划"而非"基本输出能力"。

**只有深层干预（L22）实现了"零崩溃"——所有语义类 prompt 都没有退化成空输出或死循环，语义内容仍可辨识。** 浅层和中层干预太底层了，关掉标点特征等于让模型连基本的 token 序列都组织不了。

这告诉我们一个重要的工程经验：**拆黑箱不是在哪层动刀都行。** 你想分离"格式"和"内容"，就得在模型做"格式决策"的那层动手——太浅了关掉的是打字能力，太深了可能已经来不及了。第 22 层恰好是 Mistral-7B 做"该用什么格式输出"这个决策的位置。

◆ 总结

━━━━━━━━━━━━━━━━━━━━

回到开头的问题：在 Mistral-7B 上，线性方法拆不开结构和语义的纠缠。SAE 升维解耦之后，拆开了。

```
SAE 做的事：4096 维混合向量 → 升维到高维 → 稀疏化 → 降回 4096
目的：把叠加的概念解耦到独立的维度上
训练：无监督，只需要模型的中间层激活，不需要标注
结果：每个非零维度 = 一个可解释特征，可以精确干预
```

三代激活函数的演进：

```
ReLU+L1：全局一个 λ 控制稀疏 → 大特征被削，小特征被杀，L0 不可控
  ↓ 问题：稀疏度和重建误差耦合
TopK：直接保留最大的 K 个 → L0 精确可控，但边界梯度弱
  ↓ 问题：K 固定，不灵活
JumpReLU：每个特征学自己的阈值 θ → 自适应稀疏，梯度友好
```

在 Mistral-7B 上的实测结论：

1. 三层 SAE（L8/L16/L22）都能找到干净的 JSON 结构特征——这些特征只在 JSON 标点上激活，在非标点 token 上几乎为零
2. 浅层编码语法标点（"这里放引号"），深层编码输出规划（"该用 JSON 格式"）——模型内部有明确的分工
3. **深层干预（L22）是唯一能实现"格式崩溃 + 语义部分保留"且零崩溃的方案**——浅层干预太暴力，会把模型的基本输出能力一起摧毁
4. 原始空间里纠缠在一起的概念，SAE 升维后能精确解耦，而且选对层做手术是关键

SAE 不是万能的——它假设特征是线性叠加的（h ≈ W_dec × z），如果模型内部的概念表示本身就是非线性的，SAE 捕捉不到。而且即使 SAE 能找到干净的结构特征，干预效果也依赖于具体 prompt：我们实测中有些 prompt（如 car、school）关掉一个特征就完全崩溃，说明它们的 JSON 生成没有冗余路径，纠缠程度超出了 SAE 单层干预的能力。但总体而言，线性假设在 Mistral-7B 的多个层上成立得相当好，SAE 在深层的精确干预能力是线性方法做不到的。
