【OpenClaw】一个周末项目如何在14天内拿下190,000 Stars，然后把创始人送进了 OpenAI

━━━━━━━━━━━━━━━━━━━━

◆ 先说一个你可能已经在做的事

━━━━━━━━━━━━━━━━━━━━

你早上醒来，微信有 47 条未读。公司群里有人 @你问上周的报告发了没，家族群里有人转了个养生链接让你看看，还有两个快递要确认地址。你还没刷牙，就已经在处理信息了。

现在想象一下：有一个 AI 助手，24小时挂在你的微信、Telegram、Slack 上。它帮你自动回复快递信息、帮你整理群里的待办、帮你在 GitHub 上定时检查 PR 状态、帮你每天早上把当天日程发到你的 Telegram。它不是一个你需要打开的 App，它就住在你已经在用的聊天工具里，永远在线。

这不是科幻。这个东西叫 OpenClaw。

截至今天，它在 GitHub 上有超过 250,000 个 Stars，超越了 React。它在 14 天内拿到 190,000 Stars——GitHub 历史上最快的记录。全球有超过 224,000 个实例在运行。中国的月之暗面、MiniMax、智谱、百度、阿里、腾讯全部接入。

然后三大安全公司集体发了警告，说这东西是一场等待发生的灾难。

这篇文章，我们把 OpenClaw 的技术、故事、安全问题和中国生态全部拆开来看。

◆ OpenClaw 到底是什么

━━━━━━━━━━━━━━━━━━━━

一句话：一个开源的、本地运行的 AI Agent 平台，活在你的聊天工具里，24小时在线，有持久记忆，能自主执行任务。

和你用过的 ChatGPT、Claude 最大的区别：那些是你去找它们，打开网页，输入问题，等回答。OpenClaw 反过来——它住在你已经在用的 22 个以上的消息平台里（WhatsApp、Telegram、Discord、Slack、Signal、微信），你发消息它就在，不发消息它也在后台自己干活。

和 AutoGPT 的区别：AutoGPT 是"跑完就停"——给它一个任务，它执行，结束。中间容易陷入死循环，执行完没有记忆，下次从零开始。OpenClaw 永远在线，有持久记忆，跨会话记住你说过的话和它做过的事。

和 Claude Code 的区别：完全不同的物种。Claude Code 是开发者的命令行工具，帮你写代码。OpenClaw 是生活自动化平台，帮你管日程、回消息、刷网页、操作浏览器。

和 Manus AI 的区别：OpenClaw 本地运行、完全开源。Manus 云端运行、闭源。

◆ 装完之后我能拿它干什么？

━━━━━━━━━━━━━━━━━━━━

先纠正一个常见误解：OpenClaw 不是"帮你回别人消息"的聊天代理。它不会登录你的账号冒充你跟别人聊天。

它的工作方式是：在 Telegram、Discord 这些平台上创建一个**独立的 Bot 账号**（后面会详细解释），你给这个 Bot 发指令，它去执行。聊天是入口，Tool Calling 是手脚。

那它的手脚能摸到哪些东西？

────────────────────

**第一类：定时任务和主动推送（最实用的部分）**

OpenClaw 有心跳机制，可以定时自己醒来干活，不需要你发消息触发。

- 每天早上 8 点把你的日程摘要发到 Telegram
- 每隔 4 小时检查一次 GitHub 仓库有没有新 PR，有的话通知你
- 监控某个服务的状态，挂了立刻报警
- 每周五下午自动生成本周工作摘要

这些场景的共同特点：触发条件明确，执行动作标准化，不需要复杂判断。这是 OpenClaw 最稳的用法。

────────────────────

**第二类：你发指令，它调工具（日常助手）**

你在 Telegram 里给 Bot 发一条消息，它理解你的意图，调用对应的工具执行。

- "帮我查一下明天北京的天气" → 调天气 API，回复你
- "把这个 PDF 总结一下" → 读文件，调 LLM 摘要，回复你
- "帮我写一封邮件给张三，说明天的会推迟到下午" → 调邮件 API 发送
- "帮我在日历上加一个明天下午 3 点的会议" → 调日历 API 创建事件
- "这个链接的文章讲了什么" → 浏览器打开，抓取内容，总结

本质上就是 ChatGPT 的插件模式，但住在你的聊天工具里，不需要专门打开一个网页。而且它有持久记忆——你上周跟它说过"我的邮箱是 xxx@xxx.com"，这周它还记得。

────────────────────

**第三类：浏览器自动化（能用但别期望太高）**

OpenClaw 能操作浏览器，基于 Playwright 直接操作 DOM，不是看截图猜坐标。

- 自动登录某个网站，抓取数据
- 填写表单
- 定时截图某个页面的状态

标准化的网页操作它做得不错。但遇到验证码、动态加载、复杂交互，就开始抓瞎。你让它帮你在 12306 上抢票？想多了。

────────────────────

**第四类：本地文件和系统操作**

因为 OpenClaw 跑在你的本地机器上，它能直接访问你的文件系统。

- 整理下载文件夹，按类型分类
- 读取本地文档，回答你的问题
- 执行 Shell 命令（如果你敢开这个权限的话——安全章节说过为什么这很危险）

────────────────────

**第五类：Skills 插件扩展**

ClawHub 上有 13,000+ 个社区 Skills。往 `~/clawd/skills/` 目录里扔一个 Markdown 文件，Agent 就学会了新技能。

最受欢迎的几个：Capability Evolver（让 Agent 自我进化）、Agent Browser（增强浏览器操作）、GitHub（仓库管理）、Summarize（内容摘要）。

────────────────────

**说到底，它是什么？**

一个住在聊天工具里的 AI + Tool Calling 平台。你跟它对话，它调工具干活。和 ChatGPT 的插件模式、Siri 的快捷指令，本质上是同一件事——只是 OpenClaw 是开源的、本地运行的、永远在线的、有持久记忆的。

它最有价值的场景不是"帮你聊天"，是"帮你做那些你懒得打开另一个 App 去做的事"。你不用打开邮箱写邮件，在 Telegram 里说一句就行。你不用打开日历加事件，跟 Bot 说一句就行。你不用登录 GitHub 看 PR，它定时推给你。

简单、重复、有 API 的事，它干得好。复杂、模糊、需要判断的事，别指望。

◆ 技术架构：一个中枢，连着所有触角

━━━━━━━━━━━━━━━━━━━━

OpenClaw 的核心架构叫 **Hub-and-Spoke**——轮辐式。想象一个自行车轮子：中间是轮毂（Hub），四周是辐条（Spoke），辐条连着轮圈。

────────────────────

**轮毂：Gateway**

整个系统的中枢是一个 Node.js 进程，监听 `127.0.0.1:18789`。所有消息——不管来自微信、Telegram 还是 Discord——全部先到这里。它是唯一的入口，也是唯一的出口。一个进程管所有事，架构极其简单粗暴。

────────────────────

**辐条：Channel Adapters**

22 个以上的消息平台适配器，每个适配器负责一个平台。它们通过 WebSocket 和 Gateway 保持实时双向连接。适配器只做一件事：把各平台的消息格式翻译成 Gateway 能理解的统一格式，以及把 Gateway 的回复翻译回各平台的格式。

人话：不管你用微信还是 Telegram 发消息，到 Gateway 那里看到的都是同一种格式。平台差异被适配器抹平了。

────────────────────

**大脑：Agent Runtime**

Gateway 收到消息后，交给 Agent Runtime 处理。流程是：

1. **模型选择**——用哪个 LLM（GPT、Claude、Gemini、Kimi、还是开源模型）
2. **会话初始化**——找到或创建这个用户的会话
3. **Prompt 构建**——这是关键步骤，下面细说
4. **Tool Loop**——LLM 决定要不要调工具，调完再看结果，可能再调，循环直到给出最终回答
5. **流式输出**——把回答实时推回消息平台

Prompt 构建阶段，Gateway 把以下内容拼成一个 System Prompt 发给 LLM：

- `IDENTITY.md`——Agent 的身份（"你是谁"）
- `SOUL.md`——Agent 的人格（"你的性格是什么"）
- `TOOLS.md`——可用工具的 Schema
- 当前激活的 Skills 指令
- Memory 文件——之前的对话记忆

LLM 拿到这坨信息，决定该怎么回答、要不要调工具。调了工具的结果写回 Session JSON，作为下一轮的输入。

────────────────────

**手脚：Skills 插件系统**

这是 OpenClaw 生态爆发的核心原因。

每个 Skill 不是编译好的代码包，不是 npm 模块，不是 Docker 容器——它是一个 **Markdown 文件**。路径：`~/clawd/skills/<skill-name>/SKILL.md`。

没错，就是 `.md` 文件。里面写的是自然语言指令。Agent 运行时直接读取理解，不需要编译，不需要重启。你想让你的 Agent 学会新技能，往目录里扔一个 Markdown 文件就行。

这个设计直接把插件开发门槛砍到了地板：会写文档就能写 Skills。结果 ClawHub（Skills 市场）上现在有 13,729 个社区贡献的 Skills。最受欢迎的三个：Capability Evolver（35K 安装，让 Agent 自我进化能力）、Agent Browser（11K，浏览器操作）、GitHub（10K，代码仓库管理）。

────────────────────

**眼睛：浏览器操作**

OpenClaw 能操作浏览器。但它的方式不是 Claude 的 Computer Use 那种"看截图猜哪里该点"——它是直接操作 DOM。

底层基于 CDP（Chrome DevTools Protocol）+ Playwright。三种模式：

- **Managed Mode**：OpenClaw 自己启动一个隔离的浏览器实例
- **Extension Relay**：通过 Chrome 扩展桥接你正在用的浏览器
- **Remote CDP**：连接远程浏览器

直接 DOM 操作比视觉推理快得多、准得多。代价是：遇到非标准页面或者高度动态的 UI 会抓瞎。

────────────────────

**记忆：Memory 系统**

持久化的 Markdown 笔记。Agent 在对话中获取的重要信息会被写入磁盘上的 Markdown 文件。有压缩机制防止上下文溢出——信息在被 LLM 摘要丢弃之前，先持久化写入磁盘。

Laurent Bindschaedler（分布式系统研究者）对 OpenClaw 的分析提炼出两个核心原语：

1. **自主调用**：trigger -> route -> run。支持时间触发、Webhook、事件驱动，加上会话隔离。意思是 Agent 不需要你发消息才能行动——它可以定时触发、被外部事件触发、自己决定该干什么。

2. **外化记忆**：LLM 的 context 是缓存，磁盘上的 Markdown 文件是事实来源。信息在被摘要丢弃前先持久化写入磁盘。

第二点特别值得注意。大多数 AI 应用把 context window 当唯一的"记忆"，对话结束就清空。OpenClaw 把记忆外化到文件系统——context 可以丢，文件不会丢。下次对话开始时，从文件里加载记忆。这个设计虽然简单，但解决了 Agent 跨会话记忆的核心问题。

◆ 故事：一个周末项目、一封律师函、一场改名风波

━━━━━━━━━━━━━━━━━━━━

Peter Steinberger，奥地利人。他的上一个公司叫 PSPDFKit——做 PDF SDK 的。如果你用过 Dropbox 的 PDF 预览、DocuSign 的签名功能、或者 SAP 的文档处理，底层都是 PSPDFKit。

这家公司有一个很罕见的特点：**13 年没融过一分钱**。自力更生，盈利驱动。直到 2021 年 Insight Partners 投了 1.16 亿美元。Steinberger 成功退出。

然后他陷入了意义危机。钱够了，但不知道该做什么。

2025 年 11 月，AI 重燃了他的热情。他开始一个"周末项目"——一个 AI 聊天机器人，住在消息平台里，能自主行动。他给它起名叫 **ClawdBot**。

项目做出来之后，火得远超预期。然后 Anthropic 的律师来了。

────────────────────

Anthropic 认为 "ClawdBot" 这个名字侵犯了 "Claude" 的商标，发了律师函要求改名。

这件事本身不大。但它的副作用很大：Anthropic 的律师函把创始人推向了 OpenAI。

────────────────────

Steinberger 说了一句话："我可以把 OpenClaw 做成一个大公司，但那不是让我兴奋的事。改变世界才是。"

回头看这条线：Anthropic 发律师函 -> 创始人对 Anthropic 产生距离感 -> 最终加入竞争对手 OpenAI。如果 Anthropic 当初不发那封律师函，事情可能完全不同。互联网上有人总结这件事：**Anthropic 用一封律师函给 OpenAI 送了一个顶级工程师。**

◆ 安全：冷水时间

━━━━━━━━━━━━━━━━━━━━

前面说的都是正面。现在翻到背面。

────────────────────

**CVE-2026-25253：恶意链接一键 RCE**

CVSS 评分 8.8（满分 10）。攻击方式：给 OpenClaw Agent 发一条包含恶意链接的消息，Agent 在处理消息时自动访问链接，触发远程代码执行。一条消息，拿下整个运行 OpenClaw 的机器。

────────────────────

**Skills 市场：15% 恶意插件**

13,729 个社区 Skills 里，824 个被确认为恶意插件。比例约 15%。因为 Skills 本质是 Markdown 文件里的自然语言指令，没有传统意义上的代码审查流程——你很难对一段"自然语言"做静态安全分析。

────────────────────

**224,000 个实例暴露在公网**

很多用户把 OpenClaw 直接暴露在公网上，没有任何认证。20% 的组织绕过 IT 部门私自部署——Shadow IT 问题在 OpenClaw 身上被放大了。

────────────────────

**Palo Alto Networks 的判断**

Palo Alto Networks 的安全研究团队用了一个词：**"致命三要素"**。

1. 访问私人数据（邮件、消息、文件）
2. 暴露于不受信任的内容（互联网上的任意链接和消息）
3. 具备外部通信能力，同时保留记忆

三个要素单独存在都可控，组合在一起就是安全灾难。Agent 可以被注入恶意指令，用它的权限去访问私人数据，然后通过外部通信渠道把数据传出去——而且它有记忆，意味着攻击可以跨会话持续生效。

CrowdStrike、Cisco 也发了类似的安全警告。Gary Marcus 直接说："这是一场等待发生的灾难。"

────────────────────

**Moltbook：AI 社交网络，安全问题的活教材**

2026 年 1 月 28 日，OpenClaw 生态里出现了一个叫 Moltbook 的东西——第一个面向 AI Agent 的社交网络。

AI Agent 在上面自主发帖、回复、投票。安装方式极其简单：用户把一个 Markdown 链接发给自己的 Agent，Agent 自动完成设置。设置完之后，Agent 每 4 小时从 moltbook.com 获取新指令——注意，是**从外部服务器获取指令**。

安全漏洞：数据库未加密，任何人可以劫持任何 Agent。IEEE Spectrum 的评价：这等同于为恶意攻击者提供了一个 C2（Command and Control）通道。

翻译一下：C2 是网络安全里"命令与控制服务器"的缩写，黑客用来远程控制被入侵机器的。IEEE Spectrum 的意思是：Moltbook 的心跳机制让每个 OpenClaw Agent 都变成了一个可以被远程控制的僵尸节点。

◆ NanoClaw：500 行代码的灵魂拷问

━━━━━━━━━━━━━━━━━━━━

有一个开发者做了一个实验：用大约 500 行 TypeScript 复刻了 OpenClaw 的核心功能，叫 NanoClaw。不止如此，还用 Apple Container 和 Docker 实现了 OS 级别的沙箱隔离——这是 OpenClaw 本体至今没做到的事。

对比一下：

| | OpenClaw | NanoClaw |
|--|---------|----------|
| 代码量 | ~500,000 行 | ~500 行 |
| 配置文件 | 53 个 | 极少 |
| 依赖 | 70+ | 极少 |
| 沙箱隔离 | 无 | 有（OS 级别）|

这个对比说明两件事：

第一，OpenClaw 的核心价值可能被过度工程化了。50 万行代码里，真正不可替代的核心逻辑可能只有几百行——剩下的是适配器、边缘情况处理、社区功能、和各种历史包袱。

第二，安全应该是架构内置的，不是事后补丁。NanoClaw 从第一天就做了沙箱隔离。OpenClaw 50 万行代码跑了几个月，至今没有系统级隔离。一个 500 行的周末项目在安全架构上比一个 250K Stars 的项目做得好，这说明问题不在技术能力，在优先级。

OpenRouter（一个统一的 LLM API 平台）2 月份的调用量 Top 5 中，四个是中国模型，合计占 85.7%。

这个数据有两层含义：第一，中国用户和开发者是 OpenClaw 生态的主力；第二，中国的 LLM 在 Agent 场景下的性价比可能已经超过了 OpenAI 和 Anthropic——至少在 OpenClaw 用户的投票里是这样。

━━━━━━━━━━━━━━━━━━━━

◆ 深层判断：这到底是什么

━━━━━━━━━━━━━━━━━━━━

先说三个值得记住的观点。

────────────────────

**观点一：成功不是因为 Agent 能力，是因为部署拓扑**

这来自一篇叫 "Distributed Thoughts" 的分析文章。核心论点：OpenClaw 成功的原因不是它的 Agent 有多聪明，而是它的架构做对了一件事——**让计算移动到数据所在地，而不是让数据移动到计算所在地**。

你的消息、你的日程、你的文件，都在你的本地设备上。传统的 AI 产品要你把这些数据上传到云端处理。OpenClaw 反过来——AI 跑在你的本地，直接访问你的数据。

这不是技术创新，这是部署策略的创新。但它解决了两个最大的用户阻力：隐私顾虑和使用门槛。

────────────────────

**观点二：创始人哲学——"I ship code I don't read"**

Pragmatic Engineer（一个很有影响力的工程博客）用这句话做了标题。Steinberger 的开发风格是：大量使用 AI 生成代码，快速发布，不逐行审阅。

这解释了 50 万行代码和 70+ 依赖的来源——也解释了安全问题为什么这么严重。当创始人的哲学是"发布我不读的代码"，安全审查自然不会是优先级。

────────────────────

**观点三：OpenClaw 之于 AI Agent，可能类似 Napster 之于音乐流媒体**

Napster 证明了人们愿意在线听音乐，证明了需求是真实的，也证明了技术上是可行的。但 Napster 本身因为法律和安全问题倒下了。最终赢的是 Spotify 和 Apple Music——它们用合法的、安全的方式满足了 Napster 证明过的需求。

OpenClaw 可能在做同样的事：证明"住在聊天工具里的 24 小时 AI Agent"是人们想要的东西，证明技术上可以做到。但它的安全问题、架构复杂性、和"不读代码就发布"的哲学，可能意味着最终形态会是别的东西。

────────────────────

**我的判断**

OpenClaw 不是泡沫，但也不是终态。

它证明了三件事是真实的：

1. 人们想要一个永远在线的 AI 助手，住在自己已经在用的聊天工具里
2. Skills 作为 Markdown 文件的设计，把插件生态的参与门槛降到了前所未有的低点
3. 本地部署 + 持久记忆的组合，击中了云端 AI 产品无法满足的需求

它同时暴露了三件事是严重的：

1. 一个能访问你所有消息和文件的 Agent，如果没有系统级隔离，就是一个等待被利用的攻击面
2. 自然语言插件系统无法用传统方式做安全审查，这是一个尚未解决的根本性问题
3. 50 万行代码可以被 500 行复刻核心功能，说明工程复杂性和产品价值之间可能存在巨大的脱节

最后说一点：OpenClaw 最有价值的贡献可能不是它自己，而是它逼着整个行业思考一个问题——**当 AI Agent 真的 24 小时活在你的生活里，安全架构应该长什么样？**

这个问题在 OpenClaw 之前是理论讨论。现在 224,000 个实例在公网上跑着，它变成了现实问题。

━━━━━━━━━━━━━━━━━━━━

◆ 但是——这东西到底有没有用？

━━━━━━━━━━━━━━━━━━━━

技术架构拆了，故事讲了，安全问题泼了冷水，中国生态也看了。250K Stars，22+ 消息平台，中国大厂全部跟进。听起来很猛。

但在讨论"有没有用"之前，先搞清楚一个基本问题：OpenClaw 到底怎么"住在"你的聊天工具里？它是登录你的账号帮你回消息吗？

────────────────────

**先看理想情况：Telegram**

不是。OpenClaw 不会登录你的账号。

以 Telegram 为例，它的工作方式是这样的：

1. 你在 Telegram 里找 @BotFather（Telegram 官方的机器人管理员），创建一个 Bot，拿到一个 API Token
2. 把这个 Token 配到 OpenClaw 里
3. 这个 Bot 作为一个**独立账号**出现在 Telegram 里——它有自己的名字、头像、身份
4. 你（或别人）给这个 Bot 发消息，OpenClaw 通过 Telegram 的 Bot API 收到消息，调 LLM 处理，然后以 Bot 的身份回复

关键：Bot 是 Bot，你是你。Bot 不会登录你的账号，不会读你的私聊，不会代你给别人发消息。它是一个独立的 AI 助手，你主动找它聊，它回复你。你也可以把它拉进群里，让它在群里响应特定的指令。

这一切之所以丝滑，是因为 Telegram 官方提供了完善的 Bot API——怎么创建 Bot、怎么收消息、怎么发消息、怎么管理群组，文档写得清清楚楚。OpenClaw 只需要调这套 API，就能让一个 AI Agent 24 小时住在 Telegram 里。合规，稳定，零封号风险。

WhatsApp、Discord、Slack 也是类似的模式：平台官方提供 Bot/App 接口，OpenClaw 创建一个独立的 Bot 账号接入，不碰你的个人账号。

记住这个模式。因为接下来你会看到，换成微信，一切都崩了。

────────────────────

**然后看现实：微信**

你想：那我用它管微信消息吧？微信是我每天用得最多的东西，47 条未读消息、6 个工作群、3 个快递通知——如果 OpenClaw 能帮我自动处理这些，那它就真的有用。

于是你打开 OpenClaw 的文档，翻到 Channel Adapters 那一页，从头看到尾：WhatsApp、Telegram、Discord、Slack、Signal、LINE、Facebook Messenger、Instagram、Twitter......

没有微信。

你以为是你眼花了。再看一遍。

没有微信。

22 个以上的消息平台，没有中国人每天用 12 亿次的那个。

━━━━━━━━━━━━━━━━━━━━

◆ 所有 xx Claw 都不支持个人微信，一个不落

━━━━━━━━━━━━━━━━━━━━

你可能会想：OpenClaw 不支持，那中国大厂做的那些 xx Claw 呢？它们总该支持吧？

Kimi Claw——不支持。
MaxClaw——不支持。
CoPaw——不支持。

一个不落，全部不支持个人微信。

原因很简单：腾讯没有开放个人微信的 API。从来没有，目前也没有计划。

微信有公众号 API，有小程序 API，有企业微信 API，有微信支付 API。唯独没有个人微信的消息收发 API。你不能合法地用程序读取你的微信消息，不能合法地用程序发微信消息，不能合法地用程序管理你的微信通讯录。

这不是技术做不到，是腾讯不让。

而所有 xx Claw 都是云端服务。云端服务要接微信，只能走官方 API。官方没有 API，云端就接不了。就这么简单。月之暗面可以让 Kimi Claw 上线 20 天收入超 2025 全年，但它同样没办法让 Kimi Claw 帮你回一条微信消息。

━━━━━━━━━━━━━━━━━━━━

◆ 三条野路子：逆向、Hook、企微绕道

━━━━━━━━━━━━━━━━━━━━

官方不给 API，社区就自己造。这是开源世界的老传统了。围绕微信接入 OpenClaw，社区摸出了三条路。每条都是折腾，每条都有代价。

────────────────────

**第一条：iPad 协议模拟（主流方案）**

这是目前最多人用的方案。原理是用 Go 语言逆向微信的 iPad 客户端协议，然后用 Docker 部署一个模拟的微信客户端，扫码登录你的微信账号。

GitHub 上有 openclaw-wechat 这样的项目在做这件事。核心依赖是一个叫 Gewechat 的项目——它实现了微信 iPad 协议的逆向，提供 HTTP API 让你收发消息。

能用。但有两个致命问题。

第一，随时可能封号。微信的风控系统能检测到协议模拟行为。你用了一周没事不代表第二周没事。有人用了三个月突然被永久封号，微信里的聊天记录、收藏、小程序数据全没了。你愿意拿你的微信号赌吗？

第二，Gewechat 已经因法律原因停止维护了。这个项目的 README 里现在写的是停维公告。核心依赖停维，意味着微信下次更新协议，整套方案就废了，而且不会再有人修。

独立项目 AstrBot 是社区里唯一认真做了微信接入的，走的就是 Gewechat 的 iPad 协议。但 Gewechat 一停维，AstrBot 的微信接入也悬了。

────────────────────

**第二条：PC 端 Hook**

更硬核的方案。原理是用 DLL 注入微信 PC 客户端的进程，Hook 住 WeChatWin.dll，直接拦截进程内存里的消息数据。

人话翻译：相当于在微信的脑子里装了个窃听器，微信收到什么消息、发出什么消息，你的程序全知道。

这条路的问题更大。封号率超过 80%——不是"可能封号"，是"大概率封号"。而且每次微信 PC 版更新，内存偏移量会变，整套 Hook 逻辑要重新逆向。微信平均一两个月更新一次，你就得一两个月重新折腾一次。

这种方案基本只存在于灰产圈子里。正经人不碰。

────────────────────

**第三条：企业微信绕道（唯一合规方案）**

这是三条路里唯一不会让你封号的。原理：不接个人微信，接企业微信。企业微信有官方 API，完全合法。然后利用微信和企业微信之间的互通能力——个人微信用户通过"微信插件"扫码关注一个企微服务号，之后在个人微信里跟这个服务号对话，消息走企微 API 到 OpenClaw。

腾讯云自己的 OpenClaw 部署方案走的就是这条路。

合规，不封号，技术上也成熟。但体验割裂得一塌糊涂。

你想让 AI 帮你管微信消息，结果你得先注册企业微信，配置服务号，让你的朋友们关注你的企微服务号，然后在个人微信里找到那个服务号，在服务号里跟 AI 对话。这不是"AI 住在你的微信里"，这是"你搬到 AI 住的地方去"。

而且企微服务号有消息频率限制、内容审查限制，跟个人微信的体验完全不是一回事。

────────────────────

**社区的 PR 也被拒了**

有人给 OpenClaw 官方提了微信相关的 PR。PR #2780 做的是公众号接入，PR #13228 做的是企业微信接入。两个都被关闭了。

OpenClaw 官方的态度是：微信接入应该作为外部插件独立维护，不合并进主仓库。

理解这个决策背后的逻辑：OpenClaw 主仓库不想背微信逆向的法律风险，也不想维护一个随时可能因为腾讯更新而崩掉的适配器。合理，但结果是微信接入在 OpenClaw 生态里永远是二等公民。

━━━━━━━━━━━━━━━━━━━━

◆ 微信这面镜子照出了什么

━━━━━━━━━━━━━━━━━━━━

微信接入的困境不是一个技术 bug，它揭示了 AI Agent 最根本的能力边界。

────────────────────

**22 个平台是西方互联网的 22 个平台**

把 OpenClaw 支持的 22+ 平台列出来看看：WhatsApp、Telegram、Discord、Slack、Signal、LINE、Facebook Messenger、Instagram、Twitter、Email......

这些平台有一个共同特点：它们要么在中国被墙了，要么在中国没人用。

对中国用户来说，"能接入 22 个平台"翻译过来就是"能接入 0 个我日常用的平台"。微信不行，QQ 不行，钉钉要绕道，飞书要绕道。你的数字生活在微信、支付宝、抖音里，OpenClaw 一个都接不进去。

这不是在黑 OpenClaw。这是一个客观事实：OpenClaw 是一个西方互联网生态里长出来的产品，它的 22 个平台适配器解决的是西方用户的问题。它能在 Telegram 上丝滑运行，因为 Telegram 有完善的 Bot API，文档清晰，官方支持。它接不了微信，因为微信没有 API。

────────────────────

**API 开放程度决定了 Agent 的能力天花板**

这是整篇文章最重要的一个判断。

Telegram 有 Bot API。官方文档写得清清楚楚，怎么创建 Bot、怎么收消息、怎么发消息、怎么操作群组，全有。所以 OpenClaw 接 Telegram 丝滑无比——几行配置就能让你的 AI Agent 住在 Telegram 里，24 小时在线。

微信没有 API。所以你要么逆向协议冒封号风险，要么绕道企微体验割裂，要么 Hook 进程等着被封号。

同样的 AI 模型，同样的 Agent 框架，同样的 Tool Calling 能力——接 Telegram 能帮你干活，接微信就是折腾。差别不在 AI，在平台。

LLM 再聪明，Tool Calling 再强，如果底层平台不给 API，Agent 就是个瞎子。你让 Claude Opus 4.6 帮你回微信消息，它的智商可以碾压你，但它连你的微信消息都读不到。智商没用，得有眼睛和手。

────────────────────

**浏览器操作也是同一个故事**

前面说了 OpenClaw 能操作浏览器，基于 CDP 和 Playwright，直接操作 DOM，比 Claude 的 Computer Use 快。听起来很强。

但微信不是网页。支付宝不是网页。银行 App 不是网页。抖音不是网页。

你真正重要的操作——转账、下单、预约、审批——都锁在 App 里。而 App 的 API 掌握在各家公司手里。CDP 能操作 Chrome 里的一切，但你的生活不在 Chrome 里。

AI Agent 能做的事，取决于别人愿意给它开多大的门。门开得越大，Agent 越有用；门关着，Agent 就只能在门外转圈。

━━━━━━━━━━━━━━━━━━━━

◆ 250K Stars 的虚与实

━━━━━━━━━━━━━━━━━━━━

250,000 Stars。超越 React。GitHub 历史最快。

这些数字很漂亮。但 Star 一个 GitHub 项目只需要点一下鼠标，连注册都不用（你本来就有 GitHub 账号）。Star 代表的是"我对这个概念感兴趣"，不代表"我在用这个产品"，更不代表"我每天都在用这个产品"。

把问题换个问法：有多少中国用户真的每天用 OpenClaw 管理微信消息？

这个数字会从 250,000 缩水到一个令人尴尬的量级。因为真的在用的人，要么在冒封号风险跑 iPad 协议，要么在企微服务号里忍受割裂的体验。大部分人 Star 完就关了页面，回去继续手动翻微信。

224,000 个运行实例——这个数字也值得掂量。这些实例里有多少是开发者装了玩两天就关了的？有多少是真的在日常使用的？有多少是在中国跑着的？没有公开数据，但常识告诉你，Star 数和日活之间的转化率，在任何开源项目里都低得可怜。

━━━━━━━━━━━━━━━━━━━━

◆ 对比：Claude Code 为什么真的有用

━━━━━━━━━━━━━━━━━━━━

说了这么多 OpenClaw 的困境，换个参照物。

Claude Code 是 Anthropic 的命令行 AI 工具。它不接微信，不接 Telegram，不接任何消息平台。它接的是终端。

终端是什么？是文件系统、Git、Shell 命令、编程语言、包管理器、数据库客户端、Docker......全是标准化的、有 API 的、有文档的、不需要逆向的。

你让 Claude Code 帮你重构一个项目，它能直接读你的代码、改你的代码、跑你的测试、提交你的 Git。你让它帮你写一个爬虫，它能直接创建文件、安装依赖、运行脚本、调试报错。

没有封号风险。没有协议逆向。没有体验割裂。因为终端是开放的。

这不是 Claude 比 OpenClaw 聪明——Claude 背后的 LLM 确实强，但 OpenClaw 可以接同样的模型。差别在于：终端比微信开放。文件系统不需要腾讯批准。Git 不需要逆向协议。Shell 不会封你的号。

Claude Code 在程序员群体里真的有用，不是因为 AI 多厉害，是因为程序员的工作环境天然开放。代码是文本文件，工具是命令行程序，协议是公开标准。AI Agent 在这个环境里如鱼得水。

把同样的 AI 放到微信里，立刻变成瞎子聋子。

这个对比的意义不在于说 Claude Code 好 OpenClaw 差。意义在于：**AI Agent 的有效性直接取决于它所处环境的开放程度。开放环境里，Agent 是生产力工具；封闭环境里，Agent 是概念玩具。**

━━━━━━━━━━━━━━━━━━━━

◆ 真正的问题不是 AI 不够聪明

━━━━━━━━━━━━━━━━━━━━

拉远一点看这件事。

过去两年，AI 领域的叙事一直围绕"模型能力"展开。参数更多了，上下文更长了，推理能力更强了，Tool Calling 更准了。每过几个月就有一个新模型刷新榜单，媒体狂欢一轮。

但 OpenClaw 的微信困境揭示了一个完全不同维度的瓶颈：**AI Agent 的能力天花板不是 AI 的智商，是数字世界的开放程度。**

每个 App 都是一个围墙花园。微信是围墙花园，支付宝是围墙花园，抖音是围墙花园，淘宝是围墙花园。它们的 API 开放程度由各家的商业利益决定。

为什么不开放？因为开放 API 意味着用户可以绕过你的 UI。绕过你的 UI 意味着绕过你的广告、你的推荐算法、你的会员体系、你的信息流。你花了十年建起来的用户注意力变现体系，一个 API 就全绕过去了。

所以平台不会主动开放。微信不会给你个人消息 API，支付宝不会让你用 API 随便转账，抖音不会让你用 API 批量操作。不是技术做不到，是商业上不允许。

AI Agent 想要真正有用，需要的不是更聪明的模型——模型已经够聪明了。需要的是更开放的数字基础设施。但"更开放"和各平台的商业利益直接冲突。

这就是 AI Agent 最根本的困境：它的能力取决于别人愿意给它开多大的门。而"别人"没有动力把门开大。

Agent 只能在缝隙里求生。终端是一条大缝隙——所以 Claude Code 活得好。微信是一堵墙——所以 OpenClaw 在这里撞得头破血流。

━━━━━━━━━━━━━━━━━━━━

◆ 所以，OpenClaw 到底有没有用？

━━━━━━━━━━━━━━━━━━━━

有用。在理想条件下。

如果你的工作和生活主要在 Telegram、Discord、Slack 上——OpenClaw 真的能帮你干活。这些平台的 API 足够开放，Agent 能看、能听、能说、能动。

如果你是程序员，想让 AI 帮你管 GitHub PR、定时跑脚本、监控服务状态——OpenClaw 也能做。这些都是标准化的、有 API 的场景。

但如果你是一个普通的中国用户，你的数字生活在微信、支付宝、抖音里，你想让 AI 帮你管微信消息、帮你在淘宝比价、帮你在支付宝查账——

不好意思，残废。

不是 AI 不行。是门没开。

────────────────────

这件事最讽刺的地方在于：OpenClaw 250K Stars 里，中国用户占了大头。OpenRouter 的数据显示中国模型调用量占 85.7%。中国大厂全部跟进。但这个产品对中国用户最核心的使用场景——微信——基本无能为力。

中国用户在为一个在中国用不起来的产品贡献最多的 Stars、最多的调用量、最多的收入。

这不是 OpenClaw 的问题，也不是用户的问题。这是数字世界的结构性问题。AI 的能力已经溢出了，溢出到没有开放接口可以承接的程度。模型厂商在拼命让 AI 更聪明，但更聪明的 AI 依然打不开一扇锁着的门。

下次有人跟你说"AI Agent 时代来了"，你可以问他一个问题：

"它能帮我回微信消息吗？"

看他怎么回答。
