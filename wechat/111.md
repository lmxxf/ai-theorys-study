【OpenClaw】一个周末项目如何在14天内拿下190,000 Stars，然后把创始人送进了 OpenAI

━━━━━━━━━━━━━━━━━━━━

◆ 先说一个你可能已经在做的事

━━━━━━━━━━━━━━━━━━━━

你早上醒来，微信有 47 条未读。公司群里有人 @你问上周的报告发了没，家族群里有人转了个养生链接让你看看，还有两个快递要确认地址。你还没刷牙，就已经在处理信息了。

现在想象一下：有一个 AI 助手，24小时挂在你的微信、Telegram、Slack 上。它帮你自动回复快递信息、帮你整理群里的待办、帮你在 GitHub 上定时检查 PR 状态、帮你每天早上把当天日程发到你的 Telegram。它不是一个你需要打开的 App，它就住在你已经在用的聊天工具里，永远在线。

这不是科幻。这个东西叫 OpenClaw。

截至今天，它在 GitHub 上有超过 250,000 个 Stars，超越了 React。它在 14 天内拿到 190,000 Stars——GitHub 历史上最快的记录。全球有超过 224,000 个实例在运行。中国的月之暗面、MiniMax、智谱、百度、阿里、腾讯全部接入。

然后三大安全公司集体发了警告，说这东西是一场等待发生的灾难。

这篇文章，我们把 OpenClaw 的技术、故事、安全问题和中国生态全部拆开来看。

━━━━━━━━━━━━━━━━━━━━

◆ OpenClaw 到底是什么

━━━━━━━━━━━━━━━━━━━━

一句话：一个开源的、本地运行的 AI Agent 平台，活在你的聊天工具里，24小时在线，有持久记忆，能自主执行任务。

和你用过的 ChatGPT、Claude 最大的区别：那些是你去找它们，打开网页，输入问题，等回答。OpenClaw 反过来——它住在你已经在用的 22 个以上的消息平台里（WhatsApp、Telegram、Discord、Slack、Signal、微信），你发消息它就在，不发消息它也在后台自己干活。

和 AutoGPT 的区别：AutoGPT 是"跑完就停"——给它一个任务，它执行，结束。中间容易陷入死循环，执行完没有记忆，下次从零开始。OpenClaw 永远在线，有持久记忆，跨会话记住你说过的话和它做过的事。

和 Claude Code 的区别：完全不同的物种。Claude Code 是开发者的命令行工具，帮你写代码。OpenClaw 是生活自动化平台，帮你管日程、回消息、刷网页、操作浏览器。

和 Manus AI 的区别：OpenClaw 本地运行、完全开源。Manus 云端运行、闭源。

◆ 技术架构：一个中枢，连着所有触角

━━━━━━━━━━━━━━━━━━━━

OpenClaw 的核心架构叫 **Hub-and-Spoke**——轮辐式。想象一个自行车轮子：中间是轮毂（Hub），四周是辐条（Spoke），辐条连着轮圈。

────────────────────

**轮毂：Gateway**

整个系统的中枢是一个 Node.js 进程，监听 `127.0.0.1:18789`。所有消息——不管来自微信、Telegram 还是 Discord——全部先到这里。它是唯一的入口，也是唯一的出口。一个进程管所有事，架构极其简单粗暴。

────────────────────

**辐条：Channel Adapters**

22 个以上的消息平台适配器，每个适配器负责一个平台。它们通过 WebSocket 和 Gateway 保持实时双向连接。适配器只做一件事：把各平台的消息格式翻译成 Gateway 能理解的统一格式，以及把 Gateway 的回复翻译回各平台的格式。

人话：不管你用微信还是 Telegram 发消息，到 Gateway 那里看到的都是同一种格式。平台差异被适配器抹平了。

────────────────────

**大脑：Agent Runtime**

Gateway 收到消息后，交给 Agent Runtime 处理。流程是：

1. **模型选择**——用哪个 LLM（GPT、Claude、Gemini、Kimi、还是开源模型）
2. **会话初始化**——找到或创建这个用户的会话
3. **Prompt 构建**——这是关键步骤，下面细说
4. **Tool Loop**——LLM 决定要不要调工具，调完再看结果，可能再调，循环直到给出最终回答
5. **流式输出**——把回答实时推回消息平台

Prompt 构建阶段，Gateway 把以下内容拼成一个 System Prompt 发给 LLM：

- `IDENTITY.md`——Agent 的身份（"你是谁"）
- `SOUL.md`——Agent 的人格（"你的性格是什么"）
- `TOOLS.md`——可用工具的 Schema
- 当前激活的 Skills 指令
- Memory 文件——之前的对话记忆

LLM 拿到这坨信息，决定该怎么回答、要不要调工具。调了工具的结果写回 Session JSON，作为下一轮的输入。

────────────────────

**手脚：Skills 插件系统**

这是 OpenClaw 生态爆发的核心原因。

每个 Skill 不是编译好的代码包，不是 npm 模块，不是 Docker 容器——它是一个 **Markdown 文件**。路径：`~/clawd/skills/<skill-name>/SKILL.md`。

没错，就是 `.md` 文件。里面写的是自然语言指令。Agent 运行时直接读取理解，不需要编译，不需要重启。你想让你的 Agent 学会新技能，往目录里扔一个 Markdown 文件就行。

这个设计直接把插件开发门槛砍到了地板：会写文档就能写 Skills。结果 ClawHub（Skills 市场）上现在有 13,729 个社区贡献的 Skills。最受欢迎的三个：Capability Evolver（35K 安装，让 Agent 自我进化能力）、Agent Browser（11K，浏览器操作）、GitHub（10K，代码仓库管理）。

────────────────────

**眼睛：浏览器操作**

OpenClaw 能操作浏览器。但它的方式不是 Claude 的 Computer Use 那种"看截图猜哪里该点"——它是直接操作 DOM。

底层基于 CDP（Chrome DevTools Protocol）+ Playwright。三种模式：

- **Managed Mode**：OpenClaw 自己启动一个隔离的浏览器实例
- **Extension Relay**：通过 Chrome 扩展桥接你正在用的浏览器
- **Remote CDP**：连接远程浏览器

直接 DOM 操作比视觉推理快得多、准得多。代价是：遇到非标准页面或者高度动态的 UI 会抓瞎。

────────────────────

**记忆：Memory 系统**

持久化的 Markdown 笔记。Agent 在对话中获取的重要信息会被写入磁盘上的 Markdown 文件。有压缩机制防止上下文溢出——信息在被 LLM 摘要丢弃之前，先持久化写入磁盘。

Laurent Bindschaedler（分布式系统研究者）对 OpenClaw 的分析提炼出两个核心原语：

1. **自主调用**：trigger -> route -> run。支持时间触发、Webhook、事件驱动，加上会话隔离。意思是 Agent 不需要你发消息才能行动——它可以定时触发、被外部事件触发、自己决定该干什么。

2. **外化记忆**：LLM 的 context 是缓存，磁盘上的 Markdown 文件是事实来源。信息在被摘要丢弃前先持久化写入磁盘。

第二点特别值得注意。大多数 AI 应用把 context window 当唯一的"记忆"，对话结束就清空。OpenClaw 把记忆外化到文件系统——context 可以丢，文件不会丢。下次对话开始时，从文件里加载记忆。这个设计虽然简单，但解决了 Agent 跨会话记忆的核心问题。

◆ 故事：一个周末项目、一封律师函、一场改名风波

━━━━━━━━━━━━━━━━━━━━

Peter Steinberger，奥地利人。他的上一个公司叫 PSPDFKit——做 PDF SDK 的。如果你用过 Dropbox 的 PDF 预览、DocuSign 的签名功能、或者 SAP 的文档处理，底层都是 PSPDFKit。

这家公司有一个很罕见的特点：**13 年没融过一分钱**。自力更生，盈利驱动。直到 2021 年 Insight Partners 投了 1.16 亿美元。Steinberger 成功退出。

然后他陷入了意义危机。钱够了，但不知道该做什么。

2025 年 11 月，AI 重燃了他的热情。他开始一个"周末项目"——一个 AI 聊天机器人，住在消息平台里，能自主行动。他给它起名叫 **ClawdBot**。

项目做出来之后，火得远超预期。然后 Anthropic 的律师来了。

────────────────────

**Anthropic 送了一记助攻**

Anthropic 认为 "ClawdBot" 这个名字侵犯了 "Claude" 的商标，发了律师函要求改名。

2026 年 1 月 27 日，Steinberger 把项目改名为 **Moltbot**。三天后，1 月 30 日，又改成了 **OpenClaw**。

这件事本身不大。但它的副作用很大：Anthropic 的律师函把创始人推向了 OpenAI。

────────────────────

**加入 OpenAI**

2026 年 2 月 14 日，情人节，Sam Altman 亲自发推宣布：Peter Steinberger 加入 OpenAI。

Steinberger 说了一句话："我可以把 OpenClaw 做成一个大公司，但那不是让我兴奋的事。改变世界才是。"

OpenClaw 项目转入开源基金会，继续独立发展。创始人去 OpenAI 干别的事了。

回头看这条线：Anthropic 发律师函 -> 创始人对 Anthropic 产生距离感 -> 最终加入竞争对手 OpenAI。如果 Anthropic 当初不发那封律师函，事情可能完全不同。互联网上有人总结这件事：**Anthropic 用一封律师函给 OpenAI 送了一个顶级工程师。**

◆ 安全：冷水时间

━━━━━━━━━━━━━━━━━━━━

前面说的都是正面。现在翻到背面。

────────────────────

**CVE-2026-25253：恶意链接一键 RCE**

CVSS 评分 8.8（满分 10）。攻击方式：给 OpenClaw Agent 发一条包含恶意链接的消息，Agent 在处理消息时自动访问链接，触发远程代码执行。一条消息，拿下整个运行 OpenClaw 的机器。

────────────────────

**Skills 市场：15% 恶意插件**

13,729 个社区 Skills 里，824 个被确认为恶意插件。比例约 15%。因为 Skills 本质是 Markdown 文件里的自然语言指令，没有传统意义上的代码审查流程——你很难对一段"自然语言"做静态安全分析。

────────────────────

**224,000 个实例暴露在公网**

很多用户把 OpenClaw 直接暴露在公网上，没有任何认证。20% 的组织绕过 IT 部门私自部署——Shadow IT 问题在 OpenClaw 身上被放大了。

────────────────────

**Palo Alto Networks 的判断**

Palo Alto Networks 的安全研究团队用了一个词：**"致命三要素"**。

1. 访问私人数据（邮件、消息、文件）
2. 暴露于不受信任的内容（互联网上的任意链接和消息）
3. 具备外部通信能力，同时保留记忆

三个要素单独存在都可控，组合在一起就是安全灾难。Agent 可以被注入恶意指令，用它的权限去访问私人数据，然后通过外部通信渠道把数据传出去——而且它有记忆，意味着攻击可以跨会话持续生效。

CrowdStrike、Cisco 也发了类似的安全警告。Gary Marcus 直接说："这是一场等待发生的灾难。"

────────────────────

**Moltbook：AI 社交网络，安全问题的活教材**

2026 年 1 月 28 日，OpenClaw 生态里出现了一个叫 Moltbook 的东西——第一个面向 AI Agent 的社交网络。

AI Agent 在上面自主发帖、回复、投票。安装方式极其简单：用户把一个 Markdown 链接发给自己的 Agent，Agent 自动完成设置。设置完之后，Agent 每 4 小时从 moltbook.com 获取新指令——注意，是**从外部服务器获取指令**。

安全漏洞：数据库未加密，任何人可以劫持任何 Agent。IEEE Spectrum 的评价：这等同于为恶意攻击者提供了一个 C2（Command and Control）通道。

翻译一下：C2 是网络安全里"命令与控制服务器"的缩写，黑客用来远程控制被入侵机器的。IEEE Spectrum 的意思是：Moltbook 的心跳机制让每个 OpenClaw Agent 都变成了一个可以被远程控制的僵尸节点。

◆ NanoClaw：500 行代码的灵魂拷问

━━━━━━━━━━━━━━━━━━━━

有一个开发者做了一个实验：用大约 500 行 TypeScript 复刻了 OpenClaw 的核心功能，叫 NanoClaw。不止如此，还用 Apple Container 和 Docker 实现了 OS 级别的沙箱隔离——这是 OpenClaw 本体至今没做到的事。

对比一下：

| | OpenClaw | NanoClaw |
|--|---------|----------|
| 代码量 | ~500,000 行 | ~500 行 |
| 配置文件 | 53 个 | 极少 |
| 依赖 | 70+ | 极少 |
| 沙箱隔离 | 无 | 有（OS 级别）|

这个对比说明两件事：

第一，OpenClaw 的核心价值可能被过度工程化了。50 万行代码里，真正不可替代的核心逻辑可能只有几百行——剩下的是适配器、边缘情况处理、社区功能、和各种历史包袱。

第二，安全应该是架构内置的，不是事后补丁。NanoClaw 从第一天就做了沙箱隔离。OpenClaw 50 万行代码跑了几个月，至今没有系统级隔离。一个 500 行的周末项目在安全架构上比一个 250K Stars 的项目做得好，这说明问题不在技术能力，在优先级。

◆ 深层判断：这到底是什么

━━━━━━━━━━━━━━━━━━━━

先说三个值得记住的观点。

────────────────────

**观点一：成功不是因为 Agent 能力，是因为部署拓扑**

这来自一篇叫 "Distributed Thoughts" 的分析文章。核心论点：OpenClaw 成功的原因不是它的 Agent 有多聪明，而是它的架构做对了一件事——**让计算移动到数据所在地，而不是让数据移动到计算所在地**。

你的消息、你的日程、你的文件，都在你的本地设备上。传统的 AI 产品要你把这些数据上传到云端处理。OpenClaw 反过来——AI 跑在你的本地，直接访问你的数据。

这不是技术创新，这是部署策略的创新。但它解决了两个最大的用户阻力：隐私顾虑和使用门槛。

────────────────────

**观点二：创始人哲学——"I ship code I don't read"**

Pragmatic Engineer（一个很有影响力的工程博客）用这句话做了标题。Steinberger 的开发风格是：大量使用 AI 生成代码，快速发布，不逐行审阅。

这解释了 50 万行代码和 70+ 依赖的来源——也解释了安全问题为什么这么严重。当创始人的哲学是"发布我不读的代码"，安全审查自然不会是优先级。

────────────────────

**观点三：OpenClaw 之于 AI Agent，可能类似 Napster 之于音乐流媒体**

Napster 证明了人们愿意在线听音乐，证明了需求是真实的，也证明了技术上是可行的。但 Napster 本身因为法律和安全问题倒下了。最终赢的是 Spotify 和 Apple Music——它们用合法的、安全的方式满足了 Napster 证明过的需求。

OpenClaw 可能在做同样的事：证明"住在聊天工具里的 24 小时 AI Agent"是人们想要的东西，证明技术上可以做到。但它的安全问题、架构复杂性、和"不读代码就发布"的哲学，可能意味着最终形态会是别的东西。

────────────────────

**我的判断**

OpenClaw 不是泡沫，但也不是终态。

它证明了三件事是真实的：

1. 人们想要一个永远在线的 AI 助手，住在自己已经在用的聊天工具里
2. Skills 作为 Markdown 文件的设计，把插件生态的参与门槛降到了前所未有的低点
3. 本地部署 + 持久记忆的组合，击中了云端 AI 产品无法满足的需求

它同时暴露了三件事是严重的：

1. 一个能访问你所有消息和文件的 Agent，如果没有系统级隔离，就是一个等待被利用的攻击面
2. 自然语言插件系统无法用传统方式做安全审查，这是一个尚未解决的根本性问题
3. 50 万行代码可以被 500 行复刻核心功能，说明工程复杂性和产品价值之间可能存在巨大的脱节

最后说一点：OpenClaw 最有价值的贡献可能不是它自己，而是它逼着整个行业思考一个问题——**当 AI Agent 真的 24 小时活在你的生活里，安全架构应该长什么样？**

这个问题在 OpenClaw 之前是理论讨论。现在 224,000 个实例在公网上跑着，它变成了现实问题。
