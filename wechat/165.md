◆ V4 终于来了

━━━━━━━━━━━━━━━━━━━━

DeepSeek V4 开源了。MIT License，权重全扔 HuggingFace，一次放出两款模型（各有一个 Base 版本）。不是"申请内测"那种半开源，是真开——你下得动就能跑。

先看全家福：

| 模型 | 总参数 | 激活参数 | 精度 | 权重大小 |
|------|--------|---------|------|---------|
| V4-Pro | 1.6T | 49B | FP4（专家）+ FP8（其他） | 865 GB |
| V4-Flash | 284B | 13B | FP4（专家）+ FP8（其他） | 160 GB |

（两款模型都有对应的 Base 版本，是没做对齐的纯预训练权重，给研究者做微调用的，普通用户不用管。）

1.6T 总参数，激活只用 49B——这个比例比 V3 更极端。V3 是 671B 总参数激活 37B，V4-Pro 参数量翻了 2.4 倍，激活只多了 32%。MoE 的"参数多但每次只用一点"被推到了新的极限。

Flash 不是 Pro 的量化缩小版——它是独立设计的架构。层数从 61 砍到 43，hidden_size 从 7168 砍到 4096，注意力头从 128 减到 64，路由专家从 384 减到 256。完全不同的骨架，为低成本推理专门设计。

◆ 五大架构创新逐个拆

━━━━━━━━━━━━━━━━━━━━

V4 不是 V3 换个皮——架构上有五个大改动。下面一个一个说，每个先讲"解决什么问题"，再讲"怎么解决的"。

**【1. DSA2：注意力算力砍到 V3 的四分之一】**

问题：百万 token 上下文下，标准注意力的计算量和 KV cache 都爆炸。V3 已经用了稀疏注意力（我们在早期文章《DeepSeek V3.2的稀疏注意力，一项被低估的技术突破》（ https://mp.weixin.qq.com/s/kT_2qmGbRPWMCV_ZEwIpPQ ）中介绍过 DSA），但还不够省。

V4 的 DSA2 把四种稀疏注意力机制组合到一起：DSA（第一代）+ NSA（Native Sparse Attention）+ Sparse MQA + Sliding Window Attention。不是简单堆叠——不同的 head 用不同的稀疏策略，让模型在不同的注意力头上同时拥有局部精确关注和全局粗粒度扫描的能力。

这里有个关键区别：V3.2 的 DSA 是推理阶段的适配——模型已经训练好了，只是在推理时换了稀疏注意力来省算力，本质是"不改脑子，改眼镜"。V4 新加的 NSA（Native Sparse Attention）则是从预训练阶段就开始用原生稀疏注意力，模型一出生就是按稀疏方式"看世界"的——本质是"脑子就是这么长的"。DSA 管推理加速，NSA 管训练阶段就学会稀疏地看，两头都改了，所以效果才这么夸张。

效果：推理算力只需 V3.2 的 27%，KV cache 只需 10%。支持百万 token 上下文。

这组数字的含义是：同样的硬件，V4 能处理的上下文长度是 V3 的十倍。或者反过来说，处理同样长度的文本，V4 用的显存和算力只有 V3 的零头。光靠推理阶段的 DSA 不可能砍这么狠，训练端的 NSA 才是大头。

**【2. Engram：把查字典从 O(n) 变成 O(1)】**

问题：LLM 的知识存在参数里，检索靠注意力机制——本质上每次都要"翻一遍全书"找答案。对于静态的事实性知识（比如"法国首都是巴黎"），这种检索方式太浪费。

Engram（我们在第 49 期《DeepSeek Engram：LLM 终于有"词组词典"了》（ https://mp.weixin.qq.com/s/4DqR-WYNf2ww61MDJVAFZw ）介绍过这项技术）的做法是：把高频静态知识从注意力检索换成 O(1) 哈希查找。建一张记忆表，key 是 token 序列的哈希，value 是对应的知识向量。查表直接命中，不需要过注意力层。

关键设计：记忆表存在 CPU DRAM 里，不占 GPU VRAM。这意味着你可以存一张巨大的知识表（几十 GB 甚至更多），不会挤占宝贵的显存空间。GPU 需要的时候从 CPU 拉数据，延迟比 attention 计算低得多。

这是 DeepSeek 和北大联合的工作（arXiv 2601.07372），可以理解为 MoE 思想的进一步延伸——不只是"选专家"，还要"选记忆"。

**【3. mHC：让超大模型训练不再翻车】**

问题：模型越大，训练越不稳定。梯度爆炸、loss 发散、训练中途崩溃——这些问题在千亿参数以上的模型训练中是家常便饭。V4 的 1.6T 参数规模更是把这个问题推到了极限。

mHC（流形约束超连接，我们早在第 37 期《DeepSeek mHC：为什么"流形约束"是标题党》（ https://mp.weixin.qq.com/s/IMU__NKt_L41YeHKi7_A1g ）就详细拆解过这项技术）替代了普通的残差连接。核心思想：约束梯度在流形上传播，不让梯度跑出"合理空间"。普通残差连接是 y = x + f(x)，梯度可以往任何方向走；mHC 加了几何约束，保证梯度更新后参数仍然在"有意义"的流形上。

这是梁文锋挂名的论文。效果直接体现在 V4 的训练稳定性上——1.6T 参数的模型能训练 32T+ token 不崩，mHC 功不可没。

**【4. Mega MoE + FP4 Indexer：推理引擎大重构】**

问题：MoE 推理有三步——dispatch（把 token 分发给选中的专家所在的 GPU）、compute（专家计算）、combine（结果收集回来）。传统实现这三步严格串行，通信和计算不能重叠，GPU 大量时间在等数据搬完。

Mega MoE（我们在第 158 期《DeepSeek新技术：Mega MoE 和 FP4 Indexer——V4 发布前的两记重拳》（ https://mp.weixin.qq.com/s/g4NH_rcXxx83pPrhN5OloQ ）已经预判过这两项技术）把 dispatch + compute + combine 合成一个 CUDA kernel，通信和计算完全重叠，GPU 不再空等。

V4 的专家规模也从 V3 的 256 选 8 变成了 384 选 6——更多专家、更少激活，路由更稀疏、每个专家更专精。和上周开源的 DeepGEMM 代码里的默认配置一模一样，我们在第 158 期就从代码里读出了这组数字。

FP4 Indexer 解决的是长上下文的另一个痛点：百万 token 上下文不能每个 token 都用全精度算。FP4 Indexer 的策略是先用 FP4 权重做粗筛（快但粗），找到相关的 token 子集后再用高精度精算。两阶段检索，兼顾速度和精度。

整个推理引擎是 FP4 权重 + FP8 激活的混合精度方案。

**【5. Muon 优化器：替换 AdamW】**

问题：AdamW 是过去几年大模型训练的标配优化器，但在超大规模训练中收敛速度和稳定性都有瓶颈。

V4 换用了 Muon 优化器，更快收敛、更稳定训练。具体细节 DeepSeek 没有展开太多，但从结果看——32T+ token 的预训练能跑完且效果好——优化器的贡献不可忽视。

◆ 训练和推理流程

━━━━━━━━━━━━━━━━━━━━

**训练分三个阶段：**

1. 预训练：32T+ token，32K 上下文窗口。这个数据量比 V3 的 14.8T 翻了一倍多。
2. 上下文扩展：从 32K 扩展到 1M（百万 token）。
3. 后训练：两个子阶段——先用领域专家做 SFT + GRPO（分组相对策略优化），再通过在线蒸馏把多个领域专家的能力整合到一个模型里。

**推理有三种模式：**

- Non-Think：快速直觉式回答，不做深度推理，适合简单问题
- Think High：逻辑分析模式，会展开推理链
- Think Max：最大推理努力，需要 384K+ 上下文窗口，适合复杂数学、代码等硬推理任务

三种模式对应不同的延迟和成本，用户（或调用者）可以根据任务难度选择。

推理价格 0.30 美元/百万 token，如果属实，比当前市面上同级别模型便宜一个数量级。DSA2 带来的算力节省（只需 V3.2 的 27%）和 FP4 混合精度直接反映在了定价上。

◆ 硬件真相

━━━━━━━━━━━━━━━━━━━━

V4 的 FP4 精度方案对硬件有明确要求。原生支持 FP4 的硬件目前只有两家：

- NVIDIA Blackwell SM100（B200/GB200 系列）
- 华为昇腾 950PR（国内首款支持 FP4 的商用推理卡，今年 3 月发布）

值得注意的是时间线。V4 的发布可以说是今年 AI 圈最大的连续剧——春节前（2 月中旬）就传要发，社区万众期待，Polymarket 上甚至开了"V4 能不能在 3 月 31 日前发布"的赌盘。结果跳票了三次：2 月没来，3 月没来，4 月初 Reuters 说"几周内"，还是没来。直到 4 月底才正式开源。

再看另一条时间线：昇腾 950PR——国内首款支持 FP4 的商用推理卡——今年 3 月 20 日发布。V4 的权重恰好是 FP4 精度。950PR 发售了，V4 才放出来。这个发布节奏不难理解。

FP4 不是一个纯软件优化，它需要硬件支持。上一代的 H100 只支持到 FP8，A100 连 FP8 都不支持。要在这些卡上跑 V4 的 FP4 权重，只能先反量化到 FP16 再算——显存占用翻几倍，FP4 的优势全丢了。

◆ 能不能自己跑？

━━━━━━━━━━━━━━━━━━━━

先说结论：

**V4-Pro（1.6T）**：个人用户基本没戏。FP4 混合精度下权重文件总共约 865GB（64 个 safetensors 分片），单机装不下。你需要一个小型集群才能跑完整的 Pro 版本。

**V4-Flash（284B）**：这是个人用户的真正目标。权重本身已经是 FP4+FP8 混合精度发布的（专家 FP4，其他 FP8），不需要再做量化——文件总共约 160GB（46 个 safetensors 分片）。单台 128GB 内存的机器还是差一口气，但并非没有办法。

**个人用户选择一：Mac Studio M3 Ultra。** 256GB 版装 160GB 的 V4-Flash 后还剩 96GB，日常推理足够；512GB 版更从容，百万 token 长上下文也撑得住。800GB/s 的内存带宽是最大优势——V4-Flash 激活参数才 13B，计算量不大，瓶颈在内存搬运，而 M3 Ultra 的带宽恰好是强项。缺点是 Apple Silicon 没有 FP4 原生指令，MLX 框架走软件模拟，速度会打折扣。但单机不用跨节点通信，综合体验可能不差。参考数据：同样是 MoE 大模型，DeepSeek R1 671B 在 M3 Ultra 512GB 上用 MLX 跑到约 18 tok/s。V4-Flash 比 R1 小得多（284B vs 671B），速度应该更快。

**个人用户选择二：两台 DGX Spark。** 每台 128GB 共享内存，两台用一根 200G QSFP56 DAC 线直连（0.5 米，即插即用），组成 256GB 的推理集群。NVIDIA 官方直接支持这种配置——两台之间走 200Gbps 带宽，用 NCCL 做张量并行，不需要交换机。256GB 跑 160GB 的 V4-Flash 绰绰有余，还有近 100GB 空间留给 KV cache 和激活值。而且 Spark 是 Blackwell 架构，FP4 原生支持，不需要软件模拟。跑 V4-Pro 还是不够，但对个人用户来说 Flash 已经是顶配了。

权重已经是 FP4 精度发布的，不存在"等社区量化"这个环节。真正要等的是推理框架的适配——vLLM、SGLang 这些框架支持 DeepseekV4ForCausalLM 架构之后，就能直接跑。

◆ DeepSeek 的开源策略

━━━━━━━━━━━━━━━━━━━━

MIT License 开源 1.6T 参数的前沿模型，这件事本身值得多说两句。

DeepSeek 的开源不是慈善。它是一种竞争策略：通过开源建立技术影响力和生态，倒逼闭源厂商降价，同时让自己的技术路线成为事实标准。V3 的开源已经证明了这条路能走通——V3 开源后，整个行业的 API 定价被迫下调。

V4 的开源更进一步。五大技术创新全部公开——DSA2 的稀疏注意力组合、Engram 的哈希记忆、mHC 的训练稳定性、Mega MoE 的推理优化、Muon 优化器。任何一个都是其他团队可能藏着当论文发的东西，DeepSeek 打包开源。

520 万美元的训练成本也是一种信号：我们用不到别人十分之一的钱就能做出同级别模型。这不只是技术实力的展示，也是对竞争对手的一种压力——你花 1 亿美元训练的模型，我 520 万美元做出来还开源了，你怎么跟你的投资人交代？

V4 的开源，本质上是 DeepSeek 在用最低成本制造最大影响力。至于这种策略能持续多久、其他团队会不会跟进——这是另一个故事了。
