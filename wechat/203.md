【端侧多模态】把多模态大模型塞进移动设备——两个文件，一只眼睛一张嘴

━━━━━━━━━━━━━━━━━━━━

◆ 起因

━━━━━━━━━━━━━━━━━━━━

上一期（ https://mp.weixin.qq.com/s/B2oGgfGG_2a6sEo0jtBiUQ ）算了三笔税，解释了"为什么 DeepSeek、GLM 选择主线纯文本"。但市面上仍然有大量"看图说话"的多模态模型在跑——它们走的都是后挂式：纯文本底座外接一只眼睛。这条路真正落到工程上是什么样的？我手头正好有个嵌入式项目，从这里说起最直观。

项目是在 RK3588 的 NPU 上跑多模态大模型。下载 Qwen3-VL-2B 的时候发现要下两个文件——一个语言模型（2.3GB），一个视觉编码器（812MB）。两个模型分开下、分开加载、分开推理，最后串起来才能"看图说话"。

板端运行的日志长这样：

```
LLM Model loaded in  2892.80 ms
ImgEnc Model loaded in  1008.81 ms
ImgEnc Model inference took  2749.71 ms
```

先加载语言模型，再加载视觉编码器，图片编码完了喂给语言模型——两步走。

这里有个容易忽略的事实先点出来：**HuggingFace 上发布的标准 Qwen3-VL-2B 其实是一个整体**——视觉编码器、投影层、语言模型一起打包在一份 safetensors 里，用 `AutoModelForVision2Seq.from_pretrained` 一行就能加载。LLaVA、MiniCPM-V、InternVL 在服务端的发布形态都是这样，vLLM 也是当成一个整体喂进去。

**"拆成两个文件"是端侧独有的工程现实。** 那么两个问题自然就来了：

- **架构上**：这种"拼接式"多模态凭什么把两个独立训练好的模型接起来，就能"看图说话"？
- **部署上**：为什么服务端一个文件就够，端侧偏偏要拆成两个？

这篇就回答这两件事。先说架构，再说端侧。

后挂式架构有个共同特点先记住：**视觉是只进不出的**——图像 token 只当输入条件，不算 loss、也不生成。"看图说话"四个字里，模型真正被训练的只有"说话"。眼睛是另外预训练好的，怎么训的下面会讲。

━━━━━━━━━━━━━━━━━━━━

◆ 多模态模型的"拼接"架构

━━━━━━━━━━━━━━━━━━━━

打开任何一个主流多模态大模型——Qwen3-VL、MiniCPM-V、LLaVA、InternVL——里面都是三个部件：

**1. 视觉编码器（Vision Encoder）**

把图片变成一组向量的模型。输入一张 448×448 的图片（约 20 万个像素），输出**784 个 1024 维**的向量（对应图被切成 28×28 个小块，每块编码成一个向量）。这些向量编码了图片的视觉特征——边缘、纹理、形状、局部结构、整体布局。具体的数字推导和训练细节下面会讲。

Qwen3-VL-2B 用的是 **SigLIP2-Large（约 300M 参数）**，一种基于 ViT（Vision Transformer）的图片编码器。SigLIP2 也是当前主流多模态模型几乎一致的选择——LLaVA、InternVL、MiniCPM-V 系列都用它或它的前身 SigLIP。

**2. 投影层（Projector / Merger）**

把视觉编码器输出的向量"翻译"成语言模型能理解的格式。Qwen3-VL-2B 里视觉编码器的输出维度是 1024 维，语言模型期望的输入维度是 2048 维——投影层就是做这个维度转换的。

Qwen3-VL 的投影层叫 Merger，结构非常简单：一个 LayerNorm + 两个线性层（MLP）。

**3. 语言模型（LLM）**

就是我们熟悉的大语言模型——接收一串 token 向量，逐个预测下一个 token。Qwen3-VL-2B 的底座是 Qwen3-2B。

────────────────────

三者的关系：

```
图片 → 视觉编码器 → 视觉向量 → 投影层 → "视觉 token" → 拼到文字 token 前面 → 语言模型 → 输出文字
```

打个比方：**视觉编码器是眼睛，投影层是翻译，语言模型是大脑。**

眼睛看到图片，用自己的"语言"描述了看到的东西。但大脑说的是另一种"语言"——它只认文字 token。翻译的工作就是把眼睛的描述转换成大脑能理解的格式。转换完了，大脑就像收到了一段特殊的"文字"，照常生成回答。

**这就是为什么可以分成两个文件**——视觉编码器和语言模型是两个独立训练出来的模型，中间只靠一个很小的投影层连接。在 RK3588 上部署时，视觉编码器跑 RKNN Runtime（NPU 加速矩阵运算），语言模型跑 RKLLM Runtime（NPU 加速 token 生成），各走各的推理流程。

━━━━━━━━━━━━━━━━━━━━

◆ 视觉编码器怎么训出来的

━━━━━━━━━━━━━━━━━━━━

视觉编码器的全部任务就一件事：**把图片压成一组定长的高维向量**。具体到 Qwen3-VL-2B 用的 SigLIP2-Large：

- 输入：一张图（实际部署常用 448×448，可变）
- 切 patch：每 16×16 像素一个 patch
- 输出：每个 patch 编码为一个 **1024 维向量**（这些数字直接来自 Qwen3-VL-2B 的 config.json，不是猜的）

448×448 切完是 28×28 = **784 个 patch**，每个 1024 维。所以编码器吐给下游的是一个 `[784, 1024]` 的矩阵——整张图就是这 784 个高维位置点。

**关键性质**：这些向量里**没有任何"这是猫"、"这是球"这类离散标签**。它们只是高维空间里的位置——所有"猫耳朵"在那个空间里都聚成一团，所有"球的反光"也聚成一团，但**没人告诉模型这一团叫耳朵那一团叫球**。SigLIP2 不知道猫和球。它只是个**视障的测量员**——能精确告诉你"这块区域有圆形物体、毛茸茸纹理、暖色调"，但从没见过分类标签。

那这个测量员是怎么训出来的？

────────────────────

**【一】CLIP：用对比学习训出"图文语义对齐"**

CLIP（OpenAI, 2021）是这条路的开山祖师。思路非常朴素：

> 同时训练一个图片编码器和一个文本编码器。让"一张猫的图"和"a cat"这两个完全不同模态的输入，编码后落到高维空间里**几乎同一个位置**。

具体做法：

1. 拿一个 batch，N 张图 + N 段对应的描述（OpenAI 用了 4 亿对图文，数据集叫 WIT-400M，闭源）
2. 图片编码器输出 N 个向量（每张图压成**一个**全局向量，通过 CLS token 或全局池化），文本编码器输出 N 个向量
3. 算 N×N 相似度矩阵——第 i 行第 j 列 = 图 i 和文 j 的余弦相似度
4. **对角线**才是正确配对，**非对角线**全是错误配对
5. 损失函数：**对每一行做 softmax + 交叉熵**——让正确配对的相似度最高
6. 反向传播，更新两个编码器的权重

打个比方：把 N 张图和 N 段文字混在一起，要求模型**给每张图选出对应的文字**——它做不到就罚，做到了就奖。几亿次这样的训练之后，模型学会了把语义相似的图文塞到同一个区域。

**CLIP 的两个核心局限**：

- **batch 越大越好**：softmax 是全局归一化，N×N 矩阵里负样本越多，对比信号越强。CLIP 用了 **batch size = 32768** 训练——一台普通 GPU 装不下，必须 256 张 V100 同步训 12 天。训练贵得惊人。
- **整张图压成一个向量**：编码器只学到"袋子级"的语义——这袋子里大概有什么。**没学到 patch 之间的关系**，也不会读细节。

────────────────────

**【二】SigLIP：一个数学小改动，省一大堆 GPU**

SigLIP（Google, 2023）只改了一处：**把 softmax 换成 sigmoid**。

```
CLIP：  每行 N 个样本做 softmax → 概率和为 1 → N 个样本互相竞争
SigLIP：每对图文独立做 sigmoid → 每对独立判断 → 样本间不耦合
```

具体讲：CLIP 是"在 N 个文字里挑最像这张图的"（多选一）；SigLIP 是"这对图文，是不是配对？"（二分类）。每对独立判断，标签是 1（正确配对，对角线上）或 -1（错误配对，非对角线）。

数学上的直接收益：**每对样本的梯度不再依赖整个 batch 里的其他样本**。所以可以用更小的 batch size 达到相同效果——SigLIP 用 16k 甚至更小的 batch，仍然能超过 CLIP 的零样本分类准确率。**训练成本降一半甚至更多**。

工程细节：sigmoid loss 有个隐患——负样本远多于正样本（一个 batch 里 N 个正样本 vs N²−N 个负样本）。SigLIP 论文做了温度（temperature）和偏置（bias）的精心调参解决类别失衡，细节不展开。

**结论**：SigLIP 出来后，CLIP 系几乎被一夜淘汰。所有后挂式 VLM 的视觉塔默认改用 SigLIP。

────────────────────

**【三】SigLIP2：从"只会对比"到"边对比边生成、边理解局部"**

SigLIP2（Google, 2025）才是 Qwen3-VL 用的版本。它不只是把 SigLIP 训得更大——**训练目标本身被扩展了**。原来只有对比学习这一个 loss，SigLIP2 同时跑四种：

| 名字 | 干什么 | 解决什么 |
|---|---|---|
| **Sigmoid 对比损失** | 沿用 SigLIP 的图文对齐 | 全局语义对齐 |
| **LocCa**（Location-aware Captioning） | 一个 decoder 头看视觉特征，**生成图片的描述文字**；同时学"指认物体在图里的位置" | 让编码器学**精细局部内容**，不再只懂袋子级语义 |
| **SILC**（Self-distillation） | 一个学生网络看图的局部裁剪，要复现教师网络看整张图的输出 | 学**局部-全局一致性** |
| **TIPS**（Masked Prediction） | 把 50% 的 patch 遮住，让模型猜遮住的内容 | 学 patch 之间的**空间结构**和上下文 |

四种 loss 加权混在一起一起训。**直观理解**：CLIP/SigLIP 的眼睛只会"看一眼判断像不像"，SigLIP2 的眼睛额外学会了"描述局部"、"补全遮挡"、"维持整体一致"——三项能力都直接指向"理解空间结构"。

训练数据也升级到 **WebLI**（Google 内部数据集，10B 图片 + 12B 文本描述，覆盖 109 种语言）——比 CLIP 的 400M 大 25 倍。

还加了 **NaFlex 多分辨率训练**：传统模型只能吃固定大小（比如 256×256），NaFlex 让 SigLIP2 在多种分辨率下都能跑得好——这是 Qwen3-VL 能用 448 而不是被锁死在 256 的前提。

**结果**：在分类、检索、零样本任务上都比 SigLIP 好，**而且尤其在 OCR、DocVQA 这类需要看清局部细节的任务上提升明显**——LocCa 和 TIPS 训出来的局部表征终于派上用场。

────────────────────

**【四】视觉编码器训完之后**

训完得到一个 ViT 模型，能把任意图片映射成一组 patch 向量（Qwen3-VL-2B 这里是 784 个 1024 维向量）。语义相似的图（两张猫）在高维空间里相邻，不同的（猫 vs 汽车）相隔很远。**这就是后挂式 VLM 拿来即用的"眼睛"**。

**但视觉编码器只学了"袋子级"的图文对齐**——它从没见过 LLM、不知道 token 是什么、也不会生成文字。它给下游的就是 784 个无标签的高维位置点。

────────────────────

**【五】眼利的读者会问：那"猫在玩球"和"球在玩猫"它分得清吗？**

**分不清**。

这就是 CLIP 系著名的 **bag-of-words 问题**：模型把句子当成一袋词，不在乎语序和关系。Winoground 基准（NeurIPS 2022）做过一组对照实验——两张图（A：猫在凳子上 / B：凳子在猫上）+ 两段对应文字描述，要模型给图配对。

- CLIP 的 image score（图配文字对错率）**接近 25%**，跟瞎猜一个水平
- 人类：90%+

为什么？因为对比学习的训练目标只是"在 batch 里挑出正确配对"——batch 里其他几万张图根本不会出现"球在玩猫"这种反常组合，所以**只匹配关键词就够拿满分**。模型没动力学组合关系，自然就不学。

至于图里嵌入的文字（招牌、文档、UI 截图），CLIP 几乎看不见——一张写满字的图在它眼里大致是"白色矩形 + 一些黑色噪点"。这是 2023 年之前的 VLM 在 DocVQA、TextVQA 上垮到一塌糊涂的根本原因。

SigLIP2 因为加了 LocCa 和 TIPS，**在 OCR 和细节定位上比 CLIP 强很多**——能看见招牌上的字了。但组合理解（"谁玩谁"）这种**关系级**的事情仍然是它的短板，SigLIP2 论文里也没特别强调 Winoground 的提升。

**根因**：组合理解和读字这两件事，**根本就不是视觉编码器该学的**——它只负责测量。理解"猫在玩球"是 LLM 在第二阶段视觉指令微调时学出来的。眼睛只测量，**大脑负责讲故事**。

这又一次回到那条铁律：**后挂式多模态必须解冻 LLM**。否则就是 2022 年那种弱智看图模型——能识别物体，但读不懂场景，到 2026 年一毛钱用都没有 😒。

━━━━━━━━━━━━━━━━━━━━

◆ 投影层怎么训的

━━━━━━━━━━━━━━━━━━━━

视觉编码器有了，语言模型也有了（比如 Qwen3-2B，用海量文本预训练好的）。两个模型各说各的"语言"——前面提过维度对不上，需要投影层做翻译。

**Qwen3-VL 的投影层（Merger）干两件事**：

- **维度对齐**：1024 维 → 2048 维（线性投影）
- **空间合并**：把 2×2 的 patch 合成 1 个视觉 token（`spatial_merge_size=2`，写在 config 里）。所以 SigLIP2 输出的 28×28=784 个 patch，经过 Merger 合并后变成 14×14=**196 个视觉 token**送给 LLM

为什么要合并？两个原因。一是省 LLM 的上下文——**196 只有 784 的 1/4**，attention 计算量 O(N²) 从 ~61 万降到 ~3.8 万，**降到 1/16**。二是相邻 patch 的语义本来就高度冗余（相邻 16 像素的区域内容几乎一样），合并后信息损失很小。

**训练策略：冻住两头，只训中间。**

视觉编码器的参数不动，语言模型的参数也不动。只训练投影层那几千万个参数——让投影层学会"怎么翻译"。

────────────────────

【训练数据】

用的是"看图说话"数据——每条数据是一张图片 + 一段对这张图片的文字描述。比如：

- 图片：一只橘猫趴在键盘上
- 文字：`A orange cat lying on a computer keyboard`

几百万到几千万条这样的配对。

────────────────────

【训练过程】

```
1. 图片过视觉编码器 → 得到视觉向量（参数冻结，不更新）
2. 视觉向量过投影层 → 得到"视觉 token"
3. "视觉 token" + 文字 token 拼在一起，喂给语言模型
4. 语言模型预测下一个 token（参数冻结，不更新）
5. 算损失：标准语言模型交叉熵（预测下一个 token 的概率 vs 真实下一个 token）
6. 梯度只回传到投影层，更新投影层参数
```

**损失函数就是最普通的语言模型交叉熵损失（next-token prediction）**——没有任何花活。投影层的目标就是：让翻译出来的"视觉 token"足够好，好到语言模型像读到了一段高质量的文字描述一样，能准确地预测后续文字。

────────────────────

【投影层有多小？】

以 Qwen3-VL-2B 为例，三个部件的参数量大致是：

```
视觉编码器（SigLIP2-Large）：约 3 亿参数
语言模型（Qwen3-2B）       ：约 17 亿参数
投影层（Merger）           ：约 2600 万参数
```

投影层只占总参数的 **1.3%**。结构就是一个 LayerNorm + 两个线性层（MLP）——几个权重矩阵，几十 MB。

**这意味着：把两个大模型（总计 20 亿参数）连接起来的"翻译官"，只有 2600 万参数。** 整个多模态能力的"接口协议"就编码在这 1% 多一点里面。

────────────────────

当然，实际训练多模态模型通常不止训投影层。比如 LLaVA 的两阶段方案：

- **第一阶段**：只训投影层（上面说的过程），用大规模图文配对数据
- **第二阶段**：解冻语言模型（甚至视觉编码器），用高质量的指令微调数据做全参数微调

但核心思路不变——**先用投影层把两个模型"接上"，再精调整体**。投影层是多模态能力的起点。

────────────────────

**所以严格讲，端侧那个 2.3GB 的 `.rkllm` 文件，里面的不是 HuggingFace 上原版 Qwen3-2B，而是经过视觉指令微调后的版本——同一个底座，不同的权重。** 投影层只能做"维度对齐 + 浅层翻译"，让视觉 token 长得像文字 token；但"看到一张猫的图回答它是什么颜色"这种推理能力，必须在 LLM 权重里训出来——它原本只见过纯文本，没学过怎么用视觉 token 答题。

第二阶段的训练数据是高质量的视觉指令微调集——多轮 VQA、看图问答、OCR、空间推理——几百万到几千万条，量级从几千 GPU 小时起。早期 LLaVA-1.0、MiniGPT-4 这种"只训投影层、LLM 一动不动"的纯后挂方案 2024 年之后基本被淘汰了——便宜，但只能做到"看图说话"基础水平，复杂推理、细粒度 OCR、空间关系全都不行。

**投影层是接口协议，LLM 也要学会怎么用这个协议**——两边都得动。

那能不能想办法保留 LLM 不动？理论上有几条折中路线：

- **Q-Former**（BLIP-2 用的）：在投影层和 LLM 之间塞一个更复杂的"翻译模块"（几亿参数），让翻译模块替 LLM 多担一些活
- **LoRA 注入**：LLM 主权重不动，只在旁边挂一组小适配器去训——物理上 LLM 文件没动，但有效权重还是变了
- **不接 LLM**：CLIP/SigLIP 直接做图文检索、分类、匹配，连"生成"都不要，对 LLM 零依赖

但只要目标是"让 LLM 生成关于图像的高质量文本"——尤其是细粒度推理（"猫的左爪在哪"、"读出招牌上的字"、"图里有几只动物"）——上面这些方案在 VQA、MMMU、DocVQA 这类榜单上**通常会比全量微调版本低 15-30 个点**。原因是 LLM 的推理路径是按"纯文本因果"刻进权重的，它不会自动把这种推理路径搬到陌生类型的 token 上——投影层只能改变视觉 token 的"形状"，改不了 LLM 的"思维方式"。

**所以工业级产品基本都接受了同一个事实：想做好多模态生成，LLM 权重必须跟着动**。买不起 GPU 的研究组可能还在用纯不动方案，但能上榜单的几乎没有。

━━━━━━━━━━━━━━━━━━━━

◆ Qwen3-VL 的特色：DeepStack

━━━━━━━━━━━━━━━━━━━━

大多数多模态模型（LLaVA 一族）的投影层做的事很直接：**取视觉编码器最后一层的输出，投影一下，拼到语言模型的输入层。**

```
ViT 最后一层输出 → 投影层 → LLM 输入层
```

Qwen3-VL 做了一个更深度的融合，叫 **DeepStack**（原始论文来自微软+复旦，NeurIPS 2024）。

**不只是最后一层**——ViT 是堆叠多层的结构（Qwen3-VL-2B 用的 SigLIP2-Large 是 24 层），每一层捕捉的信息不同：

- **浅层**：边缘、纹理、颜色——底层视觉特征
- **中层**：物体部件、局部结构——中层语义
- **深层**：物体类别、场景语义——高层语义

普通投影层只用最后一层的输出，浅层和中层的信息全丢了。**DeepStack 把 ViT 中间多个层的特征也抽出来，各自过投影，以残差方式注入语言模型的前几层。** Qwen3-VL-2B 的配置文件里写得明白：`deepstack_visual_indexes=[5,11,17]`——从 ViT 的第 6、12、18 层各抽一份（注意这是 2B 这个规模搭配 24 层 ViT 的具体配置，更大的 VL 模型用的层号会不一样）。

```
ViT 第 6 层特征  → 投影 ↘
ViT 第 12 层特征 → 投影 → 注入 LLM 的前几层（残差相加）
ViT 第 18 层特征 → 投影 ↗
ViT 最后一层输出 → 投影 → LLM 输入层（正常路径）
```

普通投影层像是翻译只转述了对方最后一句总结。DeepStack 像是翻译把对方讲话过程中不同阶段的笔记——草图、细节、结论——全都递了进去。注意笔记都递到大脑加工的**早期阶段**，让它从一开始就同时拿到粗细两套信息，而不是只凭一句总结开工。

DeepStack 原始论文在 TextVQA、DocVQA 这类高分辨率细粒度任务上提升显著；Qwen3-VL 官方也把它列为本代的核心架构更新之一，称其强化了 OCR、小物体这类细粒度视觉理解。

━━━━━━━━━━━━━━━━━━━━

◆ 为什么"拼接"架构在服务端也是主流

━━━━━━━━━━━━━━━━━━━━

先回答第一个问题：为什么不从头训一个"天生就能看图"的大模型？

**答案是成本和迭代速度。** 上一期讲过从零原生的代价：Gemini 替全行业蹚了那条最贵的路，三笔税照单全收。所以绝大多数厂商选了便宜的方案——后挂式。

【复用 + 迭代】

训一个好的视觉编码器（SigLIP2）要几亿图文配对数据 + 几千 GPU 小时。训一个好的语言模型（Qwen3-2B）要几万亿 token 文本 + 几万 GPU 小时。**这两个模型已经有人花了巨大代价训好了。** 只需要训一个 2600 万参数的投影层——几百万数据、几百 GPU 小时——就能把它们组装成多模态模型。

升级也方便：视觉编码器从 CLIP 升级到 SigLIP 再到 SigLIP2，或者语言底座从 Qwen2 换到 Qwen3，**换零件之后重训投影层、再做视觉指令微调**，整体框架不变。Qwen-VL 系列就是这么迭代过来的——从早期的 Qwen-VL（自研 ViT + Qwen-7B）到 Qwen2-VL、Qwen2.5-VL，再到 2025 年的 Qwen3-VL，每一代都是"换底座 + 重新走两阶段训练流程"。

注意，这里的"语言模型不用动"指的是**不用重新做几万亿 token 的纯文本预训练**——那是几万 GPU 小时的工程。但下游的视觉指令微调还是要做，几千 GPU 小时起。所以省的不是微调成本，省的是**预训练成本**——而预训练成本恰好是大头。

如果是端到端的从零原生多模态，每次换零件都得从头训文本+视觉联合预训练——上一期讲过的三笔税要重交一遍。这就是大多数厂商选后挂式的原因。

**注意一点**：上面说的"拼接"是**训练阶段**的拼接——三个部件分别训练、再用投影层缝合。**但发布到 HuggingFace 时，它是一个完整的 safetensors 仓库**，加载、推理、量化都按一个整体处理。"拼接"在服务端只是工程师的视角，不是文件系统的视角。

那为什么到了端侧，文件系统也得跟着拼？

━━━━━━━━━━━━━━━━━━━━

◆ 端侧的现实：从一个文件到两个文件

━━━━━━━━━━━━━━━━━━━━

回到 RK3588 项目。HuggingFace 上下下来的 Qwen3-VL-2B 是一个完整模型仓库，但要在板子上跑起来，必须先用 Rockchip 的工具链转换成两份独立的部署文件：

| 文件 | 大小 | 内容 | Runtime |
|------|------|------|---------|
| `qwen3-vl-2b-instruct_w8a8_rk3588.rkllm` | 2.3 GB | 语言模型（W8A8 量化） | RKLLM Runtime |
| `qwen3-vl-2b_vision_rk3588.rknn` | 812 MB | 视觉编码器 + 投影层（FP16） | RKNN Runtime |

为什么必须拆？五个原因，第一条是最直观的。

【原因一：模态稀疏——大多数对话压根用不到视觉】

这一条是端侧多模态和服务端最大的差异，先讲。

手机用户的真实使用场景里，**绝大多数对话是纯文本的**：聊天、写邮件、查天气、问知识。只有少数时候才会发张图让它看。

如果绑成一个文件、一个 runtime：即使只输入文字，视觉编码器那 812MB 也得**常驻内存**（否则反复加载/卸载，几百毫秒的 I/O 延迟用户立刻能感觉到）。视觉模块占的内存、NPU 资源、带宽预算，纯文本时全部在空转。

**拆成两个文件之后**：
- 纯文本对话 → 只加载 `.rkllm`，视觉模块根本不进内存，省 812MB
- 第一次发图片 → 才把 `.rknn` 加载进来跑一遍，结果作为视觉 token 喂给 LLM，跑完视觉编码器又可以被释放
- 进入图片选择界面 → 可以**预加载**视觉模块（用户还在挑图的几秒里悄悄加载，挑完正好就绪）
- 系统内存压力高 → 主动驱逐视觉模块，纯文本对话不受影响

**按需付费的内存模型，这是端侧的命门**。云端不在乎——H100 80GB 内存，整个模型 BF16 塞进去也才十几 GB，视觉模块常驻是常态。端侧 16GB RAM 里能省 800MB，是用户能直接体感到的差异（"为什么我聊天 app 占内存 1GB，但发了张图突然变 1.8GB"——就是这个机制）。

**服务端可以无视模态稀疏性，端侧必须把它当一等公民。** 这是端侧多模态和"正经大模型"最根本的差异。

【原因二：两套工具链，分别处理不同的算子】

RKNN-Toolkit2 是面向 CNN/Transformer 这类**前馈视觉模型**编译的，它把整张计算图静态展开成 NPU 指令，一次推理跑完。RKLLM-Toolkit 是面向**自回归语言模型**编译的，它要管 KV cache、token 循环采样、分块 prefill 这些 LLM 特有的东西。两个 Runtime 在底层调用的 NPU 指令集合不同，优化策略也不同——**强行塞进一个 runtime 不是做不到，是做了也不好用**。

MNN、NCNN、TensorRT-LLM-edge 走的是同一条路：视觉和 LLM 各用一套部署管线。这不是 Rockchip 的特色，是端侧的通用模式。

【原因三：量化策略必须不同】

语言模型对量化更敏感的部位是 attention 和 KV cache，普遍用 **W8A8 或 W4A16**（权重 4-8 bit、激活 8-16 bit），能在精度损失可控的前提下把 LLM 压到 RAM 里。

视觉编码器是前馈结构、计算密集型，对量化更敏感的是浅层卷积/patch embedding，常用 **FP16 或 W8A16**——量化太狠会直接影响特征质量，连带影响"看图说话"的准确度。

服务端不在乎，统一 BF16 跑就完事了。端侧 RAM 和带宽都是硬上限，**两个部件必须用最适合自己的量化策略**——这就要求它们能被独立编译、独立量化。

【原因四：推理流程根本不同】

视觉编码器是**并行**的：一张图片一次性过 ViT 24 层，2.7 秒（从日志里能看到）。

语言模型是**串行**的：每生成一个 token 就要跑一遍整个 LLM 的所有层，再把这个 token 喂回去生成下一个。一句话答案要循环几十次。

并行和串行的硬件调度策略完全不一样——NPU 上视觉部分能吃满 batch、语言部分要打满 latency。**用同一个 runtime 同时调度这两种工作负载是反人性的设计**。

【原因五：内存隔离】

板子上一共 16GB 内存（RK3588 高配版），系统/应用要占走一部分，留给模型的不超过 10GB。视觉编码器 812MB + 语言模型 2.3GB + 中间激活 + KV cache，已经塞得紧巴巴。

如果两个部件共享同一个 runtime 的内存池，一个吃多了另一个就饿死。**拆成两个进程/两份内存预算，能保证视觉和 LLM 都有自己的下限**——这在嵌入式上不是优化，是生存。

────────────────────

把五条捏一起：**端侧拆文件不是"架构选择"，是模态稀疏 + 工具链 + 硬件约束逼出来的工程现实。** 服务端有 80GB H100，BF16 整个塞进去就完事；端侧 16GB RAM 想跑 2B 模型，必须把每一份资源都按模态、按计算特性拆开来管。

这也解释了为什么 RKNN 的发布形态里，视觉编码器和投影层是**打包在同一个 `.rknn` 文件**里的——它们俩都是前馈、都是图像侧的算子、都用同一种量化、推理时严格串行。**真正的边界不在"哪些部件来自不同训练"，而在"哪些部件适合用同一套 runtime 跑"。**

────────────────────

【端侧能跑多大？】

Qwen3-VL-2B 这个 2.0B 总参数是端侧能跑的甜区。**RK3588 / RKLLM 工具链目前官方支持的多模态模型基本就是 Qwen3-VL-2B 和老一些的 MiniCPM-V-2_6**——同样的拼接思路放到 7B、72B 上就溢出板子内存了（语言模型本身就装不下，更别说再加视觉模块）。**端侧多模态目前的甜区基本锁死在 1B-3B**，再大就只能上 Jetson Orin 这种带 64GB 内存的高配开发套件，或者干脆走云端 API。

云端的 Qwen3-VL-32B、Qwen3-VL-235B 不存在这个问题——但那些已经不在"端侧多模态"的讨论范围里了。

━━━━━━━━━━━━━━━━━━━━

◆ 小结：服务端和端侧看到的是同一只大象的两面

━━━━━━━━━━━━━━━━━━━━

后挂式多模态的"拼接"，在不同视角下意味着不同的东西。

**训练视角**：视觉编码器、投影层、语言模型分别训练，再用投影层（只占 1.3% 参数）缝合，第二阶段再做视觉指令微调把 LLM 调到位。复用强模型省预训练成本，换零件不用推倒重来。

**服务端部署视角**：拼接是工程师的认知边界，不是文件系统的边界。HuggingFace 上发布的就是一个完整的 safetensors 仓库，加载、推理、量化都按一个模型处理。

**端侧部署视角**：拼接被迫显性化为两个文件、两套 runtime——因为视觉和 LLM 在算子集合、量化策略、推理模式、内存预算上都对不上号。**不拆不行**。

所以"为什么多模态模型要拆成两个文件"这个问题，准确的回答是：**它在云上从来不拆，是端侧硬件把它逼着拆了**。
