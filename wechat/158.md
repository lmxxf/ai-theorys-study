◆ 先理解 MoE 推理的瓶颈在哪

━━━━━━━━━━━━━━━━━━━━

在讲 Mega MoE 之前，先搞清楚 MoE（Mixture of Experts，混合专家）推理的流程。DeepSeek-V3 有 671B 参数，但每次推理只激活其中一小部分专家——V3 原始架构是 256 个路由专家选 8 个，而这次 DeepGEMM 代码里的默认配置已经变成了 384 个专家选 6 个（大概率是 V4 的规模）。不管具体数字怎么变，MoE 省算力的核心不变：参数很多，但每次只用一点。

这里有两个关键数字：**top-k 决定"每个 token 用几个专家"，EP 决定"专家分几张卡放"。** 以新配置为例，top-6 意味着每个 token 只找 384 个专家里最相关的 6 个；EP=8 意味着 384 个专家平均分到 8 张 GPU 上，每张卡放 48 个。问题来了：一个 token 选中的 6 个专家大概率不在同一张卡上，所以必须跨 GPU 搬数据。

传统 MoE 推理分三步，**严格串行**：

```
第一步：Dispatch（分发）
  把 token 通过 NVLink 发到目标专家所在的 GPU
  ↓ 等发完
第二步：Expert Compute（专家计算）
  专家执行两段线性层 + SwiGLU 激活
  ↓ 等算完
第三步：Combine（回收）
  把计算结果通过 NVLink 发回原来的 GPU，累加
```

注意"等发完""等算完"——**每一步都要等上一步完全结束才能开始**。GPU 要么在算，要么在等数据——不能同时干两件事。

这就是瓶颈：**通信和计算无法重叠，GPU 有大量时间在发呆。**

━━━━━━━━━━━━━━━━━━━━

◆ Mega MoE：把三步合成一步

━━━━━━━━━━━━━━━━━━━━

Mega MoE 的核心思想极其直接：**把 dispatch、compute、combine 编译进同一个 CUDA kernel 里，让通信和计算在硬件级别同时执行。**

```
传统 MoE：
  [Dispatch 通信] → 等 → [Expert 计算] → 等 → [Combine 通信]
  ════════════════════════════════════════════════════════════
  GPU空闲            GPU工作             GPU空闲

Mega MoE：
  [Dispatch + Expert计算 + Combine 交织执行]
  ════════════════════════════════════════
  GPU 全程工作，通信在后台同时进行
```

怎么做到的？三个关键技术：

────────────────────

【技术一：信号单元（Signal Unit）】

传统做法：等所有专家都算完，再统一把结果发回去。

Mega MoE 的做法：每个专家每算完一小块 token（block_m 个），就**立刻**通过 NVLink 把这一小块的结果发回去，不等别的专家。

具体机制：每个专家分配一个原子计数器（信号单元）。专家的 Down 层（linear2）算完一个 block_m 的 token 后，原子递增信号值。Combine 的发送逻辑在后台轮询这个信号——信号一到，立刻发送，不等其他 block。

**结果就是：专家还在算第 3 块 token 的时候，第 1 块的结果已经在 NVLink 上飞回去了。**

────────────────────

【技术二：对称内存（Symmetric Memory）】

传统跨 GPU 通信要走 send/recv 握手——"我要发了""好的收到""开始传""传完了"。四步握手，每步都有延迟。

Mega MoE 用了 PyTorch 2.9 新引入的对称内存：**所有 GPU 用相同的虚拟地址访问彼此的显存**。就像共享内存一样，不需要握手，直接读写。

```python
buffer = deep_gemm.get_symm_buffer_for_mega_moe(
    group, num_experts, num_max_tokens_per_rank,
    num_topk, hidden, intermediate_hidden
)
```

一个 buffer 搞定所有 GPU 之间的通信空间。底层是 NVLink RDMA，Hopper/Blackwell 架构原生支持。

────────────────────

【技术三：FP8 激活 × FP4 权重】

Mega MoE 不只是把三步合成一步，还顺手把精度压了一级：

- 激活值（token 数据）：FP8（8 bit 浮点）
- 专家权重：**FP4**（4 bit 浮点）

FP4 是什么？跟 INT4 不一样——INT4 是整数，间距均匀；FP4 是浮点，有指数位（E2M1：1 位符号 + 2 位指数 + 1 位尾数），间距不均匀，天然适应权重分布。**关键是 Blackwell 架构的 Tensor Core 原生支持 FP4 运算，不需要先反量化到 FP16 再算**——INT4 做不到这一点。

之前我们在第 56 期讲过 DeepSeek 用 FP8 怎么节省成本。这次又往下砍了一半——权重从 FP8 压到 FP4，每个专家的权重存储直接减半。几百个专家的权重总量是 MoE 的大头，这一刀砍下去效果显著。

────────────────────

【Mega MoE 的 API 长什么样】

最终暴露给用户的接口极其简洁：

```python
# 1. 把专家的两段线性层权重（FP4）重排成 Mega MoE kernel 需要的内存布局
transformed_l1, transformed_l2 = deep_gemm.transform_weights_for_mega_moe(
    l1_weights, l2_weights
)

# 2. 一次调用完成 dispatch + compute + combine
y = torch.empty((num_tokens, hidden), dtype=torch.bfloat16, device='cuda')
deep_gemm.fp8_fp4_mega_moe(
    y, transformed_l1, transformed_l2, buffer,
    activation_clamp=..., fast_math=...
)
```

**原来需要分三步写的代码，现在一个函数搞定。** 底下是 12000 多行的 CUDA kernel，但你不用管。

测试参数用的是代码里的默认配置：384 个专家，top-6 路由，hidden = 7168，EP = 8——比 V3 的 256 专家 top-8 更大，大概率就是 V4 的规模。官方性能数据还没放出来（README 写着 "will be posted later"），但从技术原理看，理论上限是接近消除通信等待的全部空泡。

◆ FP4 Indexer：给长上下文装一个快速筛选器

━━━━━━━━━━━━━━━━━━━━

第二个新技术解决的是另一个问题：**长上下文推理太慢**。

标准注意力机制要对所有历史 token 做注意力计算——上下文越长，计算量线性增长。DeepSeek V3.2 引入了稀疏注意力机制（DeepSeek Sparse Attention, DSA），核心组件就是 Lightning Indexer（闪电索引器）——先粗筛再精算。DSA 的原理在 V3.2 论文（2025 年底）里就公开了，线上也一直在跑，但底层 CUDA kernel 之前没开源——这次 DeepGEMM 更新才把代码放出来，顺便从 FP8 升级到了 FP4：

```
传统注意力（暴力扫描）：
  Query → 对全部 10 万个历史 token 做注意力 → 输出
  代价：O(n)，n = 上下文长度

Lightning Indexer（先筛后算）：
  Query → Indexer 快速打分，选出最相关的 2048 个 token → 只对这 2048 个做精确注意力 → 输出
  代价：打分 O(n) 但极轻量 + 精确注意力 O(2048)
```

**Indexer 的工作就是"粗筛"——不需要精确的注意力值，只需要排出一个 top-k 排名。**

之前的 Indexer 用 FP8 精度做打分。这次更新把它压到了 **FP4**：

```python
# 旧版：FP8 精度打分
fp8_paged_mqa_logits(...)

# 新版：FP8 Query × FP4 Key 混合精度打分
fp8_fp4_paged_mqa_logits(...)
```

────────────────────

【为什么 Indexer 特别适合低精度】

Indexer 的计算目标是**排序**，不是精确数值。你只需要知道"第 3721 个 token 比第 8846 个 token 更相关"，不需要知道具体分数是 0.8721 还是 0.8719。

**排序对精度的容忍度远高于精确计算。** FP4 的绝对误差比 FP8 大约 10 倍，但对 top-k 排序的影响微乎其微——只要相对顺序不变，分数差多少无所谓。

这就是为什么 FP4 用在 Indexer 而不是主体计算——**对的地方用对的精度**。

────────────────────

【FP4 Key Cache 的真正收益】

Key cache 从 FP8 压到 FP4，每个 token 的存储量再减半。这意味着**同等显存内可以缓存更多历史 token**。

之前能缓存 10 万 token 的显存，现在能缓存 20 万。上下文窗口直接翻倍，而 Indexer 的打分速度几乎不变（因为 FP4 计算在 Blackwell Tensor Core 上是原生指令）。

**更长的上下文 + 几乎相同的延迟 = 更强的长文本能力。** 这大概率是 V4 在长上下文上的杀手锏之一。

◆ 其他更新：一句话总结

━━━━━━━━━━━━━━━━━━━━

这次更新还有几个值得一提的东西：

- **PDL（Programmatic Dependent Launch）**：传统流程是 kernel A 跑完 → 信号回 CPU → CPU 提交 kernel B，中间 GPU 在等 CPU 转一圈。PDL 让 kernel A 自己在快完事时直接通知 GPU"下一个可以启动了"，省掉 GPU→CPU→GPU 的往返。Hopper/Blackwell 架构原生支持。
- **Swap A/B 优化**：Tensor Core 做矩阵乘法时 M 维度每次至少要凑满 64 行（block_m=64）。decode 阶段 batch size=2 时，原始乘法是 `[2, 7168] × [7168, intermediate]`，M=2，64 行的 block 里只有 2 行是真数据，剩下 62 行空转。Swap A/B 转置后变成 `[intermediate, 7168] × [7168, 2]`，M=intermediate（几万），轻松填满 block_m=64，小维度 2 跑到 N 方向去了——实测提升 8.1%。
- **DeepEPv2 Layout**：配合 Mega MoE 的新内存布局，让 GEMM kernel 直接按 EP（专家并行，即跨 GPU 搬 token 的通信层）的 buffer 格式来算，省掉算完再转格式的额外拷贝。

◆ 这次更新说明了什么

━━━━━━━━━━━━━━━━━━━━

回顾我们聊过的 DeepSeek 技术栈：

| 期数 | 技术 | 解决的问题 |
|------|------|-----------|
| 第 49 期 | Engram（N-gram Embedding） | 给词组也配上向量，不用现场算 |
| 第 56 期 | FP8 GEMM | 用低精度省算力 |
| 第 107 期 | DualPath | 通信和计算分双路 |
| 第 126 期 | mHC / Attention Residuals | 残差连接优化 |
| **本期** | **Mega MoE** | **通信和计算合成一个 kernel** |
| **本期** | **FP4 Indexer** | **长上下文筛选用更低精度** |

你能看到一条清晰的路线：**精度越压越低（FP16 → FP8 → FP4），kernel 越融越大（分步 → 双路 → mega），通信越藏越深（显式等待 → 重叠 → 对称内存）。**

每一步都不是算法创新——没有新的注意力机制，没有新的训练方法。全是工程优化：**把同样的数学运算，用更少的比特、更少的等待、更少的 kernel launch 来完成。**

DeepSeek 一直在告诉你一件事：**AI 竞赛的下一个战场不是模型架构，是推理效率。** 谁能用更少的算力跑出同样的效果，谁就能把价格打下来，把上下文拉上去，把延迟压下去。

从这些已经开源的零件里，V4 的推理引擎大致能猜出来了：

- **MoE 层**：Mega MoE 单 kernel 融合 dispatch+compute+combine，权重 FP4，激活 FP8，通信通过对称内存完全隐藏在计算背后
- **注意力层**：DSA 稀疏注意力，Lightning Indexer 用 FP4 粗筛 top-2048，主体注意力只对这 2048 个 token 做精确计算，Key Cache FP4
- **kernel 调度**：PDL 让相邻 kernel 流水线重叠，DeepEPv2 Layout 省掉格式转换的拷贝
- **长上下文**：FP4 Key Cache 让同等显存缓存量翻倍，配合稀疏注意力把长文本延迟压下来

一句话：**FP4 权重 + FP8 激活 + 稀疏注意力 + 单 kernel MoE + 通信全隐藏。** 不是某一项技术多牛，是所有优化焊在一起形成的系统级效率。

V4 来不来不重要。技术栈已经摊在 GitHub 上了。
