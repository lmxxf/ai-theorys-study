听着耳熟。

💡【小白翻译】**编排（orchestration）**：指挥多个 AI 模型协同干活的系统。你提一个问题，编排器看了之后决定"这个让 GPT 答""那个让 Claude 答"，最后把答案合并给你。看起来像"用一个模型"，实际背后是一个团队。LangGraph、AutoGen、CrewAI 这些开源框架在做同样的事，开源的免费用。

◆ 第一章：先看 Sakana 自己怎么吹，媒体怎么帮腔

━━━━━━━━━━━━━━━━━━━━

把 Sakana 官网和媒体报道整理出来，三条主线非常清晰：

【主线一：技术上的"learned orchestration"叙事】

Sakana 反复强调一个关键词——**"learned"**（学出来的）。"我们不是在写 if-else 规则，是训练了一个模型让它学会编排。"

具体来说，Fugu 背后有两篇 **ICLR 2026** 的论文：

- **TRINITY**（arxiv: 2512.04695）：一个 **0.6B 参数**的紧凑模型 + 10K 参数的轻量级 head，用 **CMA-ES**（协方差矩阵自适应进化策略，一种进化算法）训练，给一池子大模型分配 **Thinker / Worker / Verifier** 三种角色。
- **Conductor**（arxiv: 2512.04388）：一个 **7B 参数**的指挥家模型，用 **强化学习** 训练，**学习"用自然语言协调智能体"的策略**。

Fugu 技术报告（arxiv: 2606.21228）说："我们的方法综合使用了大规模微调、进化算法和强化学习。"——**三种方法叠加，三层结构**。

Sakana 的口径：**"协调策略可以从端到端的奖励最大化中自然涌现"**（coordination strategies emerge naturally through pure end-to-end reward maximization）。

熟悉的味道。**"涌现"** 这个词又出现了——和当年 Doc-to-LoRA 论文里的"记忆巩固"、"突触稳态"是一个套路：**用一个性感的科学比喻包装工程方案**。

【主线二：跑分的"shoulder-to-shoulder"叙事】

Sakana 官网的原话是 **"shoulder-to-shoulder with Fable 5 and Mythos Preview"**（与 Fable 5 / Mythos Preview 并肩）。

但要注意几个细节：

1. **Fable 5 和 Mythos 不在 Fugu 的智能体池里**——因为它们"公开访问不可用"（被美国出口管制了）。Sakana 自己也承认了这一点。
2. **head-to-head 四项对比中 Fable 5 赢三项**：SWE-Bench Pro、TerminalBench 2.1、Humanity's Last Exam 全部 Fugu Ultra 落败，只有 LiveCodeBench Fugu 赢。
3. **非 Fugu 跑分大多来自厂商自报**，没有第三方独立复现。

所以"shoulder-to-shoulder"翻译过来其实是：**"我们挑了一项赢的，剩下输的项目也输得不太多"**。

【主线三：发烧友的"模型变团队"叙事】

最有意思的是发烧友层面的反应。这里面有些话特别值得引用：

- **"模型本身就是团队，不再是巨石"**
- **"用户体验到的是一个模型，但底下永远是一个团队"**
- **"单一厂商的 AI 产品将看起来很脆弱"**
- **"长期终局可能不是一个通用模型，而是多个专家的协同"**

注意这套叙事的结构：**它不是在论证 Fugu 好，而是在论证"未来必然走 Fugu 这条路"**。这是经典的**叙事先于证据**——先建立一个"未来方向"，然后把自己的产品塞进这个方向里，让读者觉得"反对 Fugu = 反对未来"。

但请注意一个细节——**这套"模型即团队"叙事，本身没有任何新东西**：

- AutoGPT、BabyAGI 2023 年就在干这个，至今没产品化
- 微软 AutoGen、LangGraph、CrewAI 这些 multi-agent 框架早就跑了两三年
- OpenRouter 一直在做"一个 API 路由到 N 个模型"
- "学习路由"这个事，**Mixture-of-Experts (MoE)** 内部干了很多年——只是 MoE 是**模型内部**的路由，Fugu 是**模型之间**的路由

**Sakana 真正的"创新"，是把这套已经存在的东西，包装成一个 API 端点卖给企业。** 商业包装的创新 > 技术创新。

◆ 第二章：拆开看论文——TRINITY 和 Conductor 到底是什么

━━━━━━━━━━━━━━━━━━━━

两篇论文我都看完了。这里给你拆一下技术上到底是什么。

────────────────────

【TRINITY：一个进化出来的协调员】

模型架构：

- **主干（backbone）**：约 0.6B 参数的小语言模型
- **协调 head**：约 10K 参数的轻量级 head
- **训练算法**：**CMA-ES**（separable Covariance Matrix Adaptation Evolution Strategy）

💡【小白翻译】**CMA-ES**：一种进化算法。不算梯度，靠"随机变异 + 选择优秀者"来优化。优点是不需要可导（不可微的目标也能优化），缺点是**参数维度一高就慢得离谱**——所以 TRINITY 才把可调参数压到 10K（协调 head）这么少，主干 0.6B 是冻结的。

工作机制：在每一轮对话中，TRINITY 给智能体池里的 LLM **分配三种角色**之一：

- **Thinker（思考者）**：负责生成思路
- **Worker（工作者）**：负责执行具体任务
- **Verifier（验证者）**：负责检查结果

"Thinker-Worker-Verifier" 这套架构本身——**和你高中做物理题没什么区别**：先有个思路（思考者），按思路算（工作者），最后检查一下（验证者）。**这套范式在 multi-agent 领域被反复用了至少 3 年**（参见 AutoGen 的 "actor-critic-judge"、CrewAI 的 "planner-executor-reviewer"）。

跑分：**LiveCodeBench 86.2%**，论文宣称"超越所有单个模型"。

【审稿意见】TRINITY 的合理之处在于——**CMA-ES 配 10K 参数的 head，工程上是合理的**。它不在重新训练大模型，而是在"喂哪个题给哪个模型"这件事上做了优化。但这种优化的**天花板取决于智能体池本身**：如果池子里没有强模型，TRINITY 学不出强方案；如果池子里有 GPT-5.5、Claude Opus 这种强模型，TRINITY 学出来的方案本质上是**"什么时候调用强模型"**——这件事写几条 if-else 规则其实也能搞定，CMA-ES 进化算法是**漂亮包装**。

────────────────────

【Conductor：用强化学习训练一个 7B "指挥家"】

模型架构：

- **主干**：约 7B 参数的指挥家模型
- **训练算法**：强化学习（reinforcement learning），具体算法和奖励结构论文没披露
- **能力**：**用自然语言生成"协调策略"和"沟通模板"**

工作机制：不是 TRINITY 那种"硬分配三种角色"，而是 Conductor **生成一段自然语言的"指挥词"**，告诉智能体池中的某些 LLM：

> "你扮演一个 Python 安全审计专家，先列出潜在漏洞，再用 OWASP Top 10 框架排序，最后给出修复 patch。"

然后这些智能体按这个指挥词去干活。Conductor 还能 **递归调用自己**——一个 Conductor 任务里可以再嵌套若干个 Conductor 任务，理论上"无限套娃"。

跑分：声称在 **LiveCodeBench 和 GPQA 上达到 SOTA**。

【审稿意见】这其实就是**"自动 Prompt Engineering + 自动 Workflow 编排"**——把人类工程师写 prompt 的工作交给一个 7B 模型来做。听起来很美，但要警惕**两个问题**：

1. **强化学习训练成本**：7B 模型用 RL 训练协调策略，每一步 rollout 需要调用智能体池里的所有大模型推理——这是**典型的"用大模型推理喂小模型训练"**，训练成本是天文数字。Sakana 没披露具体数字。
2. **递归调用的延迟爆炸**：每嵌套一层 Conductor 调用，延迟翻倍。这就是为什么 Mollick 实测 Fugu 跑一个 coding 任务要 **30 分钟**——你不知道它在底下嵌套了多少层。

────────────────────

【Fugu 技术报告：三合一缝合怪】

Fugu 技术报告（arxiv: 2606.21228）说，最终的 Fugu 是 **"大规模微调 + 进化算法 + 强化学习"** 三种方法叠加。

翻译过来就是：

```
Sakana 把 TRINITY 的 0.6B 进化协调员
        +
        Conductor 的 7B 强化学习指挥家
        +
        一堆人工调过的 fine-tuning 数据
        ↓
缝合成一个产品，包装成"foundation model"卖
```

**"Foundation model" 这个词在这里是被滥用的**。Foundation model 的本义是 **GPT-4 / Claude Opus 这种从头训练的基础模型**——它本身就是能力的载体。Fugu 不是。Fugu 是**用别人的 foundation model 当零件**，自己只是个调度器。

但 Sakana 在营销页面把 Fugu 叫做 "foundation model"——这是**语义偷换**。它指望读者把"调度器"和"基础模型"等同起来。

💡【小白翻译】这就好比一个**外卖平台**说"我们提供了一个全新的外卖品牌"——但实际上他们既不开餐厅也不养食材，他们只是把美团、饿了么、肯德基、麦当劳的订单聚合在一个 App 里。**你订到的菜，还是别人做的。**

◆ 第三章：Fugu 的结构性硬上限——协调成本

━━━━━━━━━━━━━━━━━━━━

这一章是关键。

**我对 Fugu 最大的怀疑不在跑分，而在一个数学上的硬约束——任何"多智能体编排"方案都有一个无法绕开的成本：协调延迟。**

────────────────────

【Ethan Mollick 的实测：30 分钟跑一个 coding 任务】

Wharton 商学院 AI 研究者 Ethan Mollick（《Co-Intelligence》作者，业内最受尊重的 AI 实测博主之一）在 Fugu 发布次日就跑了实测。他的原话翻译过来是：

> **"还行，但实际使用上不如 Fable 5。一个典型的编程测试跑了 30 分钟。"**

30 分钟。

作为对比：

| 模型 | 同等任务延迟 |
|------|----------|
| Fable 5 | ~1-3 分钟 |
| GPT-5.5 (high reasoning) | ~2-5 分钟 |
| Opus 4.7 | ~30 秒-2 分钟 |
| **Fugu Ultra** | **~30 分钟** |

为什么 Fugu 这么慢？因为它内部的工作流是这样的：

```
你提一个问题
   ↓
Conductor (7B) 分析问题，生成指挥词        ← 第 1 次 LLM 调用
   ↓
TRINITY (0.6B) 决定 Thinker/Worker/Verifier 分配
   ↓
调用 GPT-5.5 (Thinker)，生成思路            ← 第 2 次 LLM 调用
   ↓
调用 Gemini 3.5 (Worker_1)，执行第一部分      ← 第 3 次 LLM 调用
   ↓
调用 Claude (Worker_2)，执行第二部分         ← 第 4 次 LLM 调用
   ↓
调用 GPT-5.5 (Verifier)，检查结果            ← 第 5 次 LLM 调用
   ↓
[发现错误，回到 Thinker 重试]
   ↓
... (递归套娃 N 次) ...
   ↓
Conductor 合成最终答案                      ← 第 N+1 次 LLM 调用
```

**每多嵌套一层，延迟就线性增长，且每一次 LLM 调用都要消耗 token。**

────────────────────

【这就是 Fugu 版的"îng îng îng"】

回到 Doc-to-LoRA 那篇实验。我们当时发现：

> **块越多，准确率越低。不是边际递减，是单调递减到零。**

为什么？因为分块拼接策略让 **LoRA rank 线性增长**，最终 rank 超过基础模型 hidden_dim 的 60%，**LoRA 不再是"低秩扰动"**，反而成了"高秩破坏"。模型语言能力崩溃，输出变成 `"îng îng îng"`。

**Fugu 的"协调成本"是同构的问题**：

| 维度 | Doc-to-LoRA | Fugu |
|------|-------------|------|
| 看起来可线性扩展的量 | 文档分块数 N | 协调嵌套层数 N |
| 实际线性增长的代价 | LoRA rank → 扰动占比 | 延迟 + token 成本 |
| 硬上限来源 | rank 必须保持"low" | 延迟必须可接受 |
| 上限被突破后 | 模型输出崩溃 | 用户体验崩溃（30 分钟） |
| Sakana 的处理方式 | **假装上限不存在** | **假装上限不存在** |

**两次都栽在同一个坑里**：拿一个**有数学/物理硬上限的方法**，当**无限可扩展的方向**卖。

────────────────────

【Kingy AI 的专业评价：partially supports】

最严肃的第三方评测来自 Kingy AI 的博客 Did Sakana Fugu Ultra Really Match Fable 5 and Mythos 5?。他们的结论是：

> **"Fugu Ultra is promising, benchmark-strong, and strategically important. It is not yet independently proven to be a Fable/Mythos replacement."**
>
> （Fugu Ultra 有潜力，跑分强劲，战略意义重要。但它**还没有被独立证实**可以替代 Fable/Mythos。）

他们指出四个方法论问题：

1. **Apples-to-oranges**：Fugu Ultra 是多智能体系统，每个任务花更多算力；Fable / Mythos 是单模型。**这就好比拿超级计算机的跑分去和笔记本电脑比**。
2. **厂商自报基线**：Fugu 的对比对象（Fable 5、Mythos）的跑分多来自厂商，没有独立复现。
3. **缺乏独立复现**：没有第三方大范围重跑 Fugu 的跑分。
4. **延迟/成本未知**：尽管定价透明，**真实工作流的"完成一个任务的总成本"未公开**。

第 4 点是 Sakana 最不想讨论的——**因为一旦把延迟和总 token 成本算进去，Fugu 的"shoulder-to-shoulder"就站不住了**。

◆ 第四章：回到 Doc-to-LoRA 旧账——同一种翻车配方

━━━━━━━━━━━━━━━━━━━━

现在让我们回到 2026 年 3 月那个红楼梦实验。

────────────────────

【当年 Sakana 的故事】

2026 年 2 月，Sakana 发了 **Doc-to-LoRA**：

- **科学比喻**：海马体 / 睡眠 / 记忆巩固
- **技术包装**：超网络（hypernetwork）输入文档 → 输出 LoRA 权重 → "AI 学会了像人一样把文档压进长期记忆"
- **跑分**：SQuAD 83.5%、大海捞针 40K token 高准确率、跨模态迁移 75%
- **媒体口径**：**"AI 终于学会了睡觉记住了"**
- **Sakana 自己的宣传话术**：**"代表 AI 长期记忆的方向"**

────────────────────

【我们 2026 年 3 月做的事】

在 NVIDIA DGX Spark 上，用红楼梦英译本前 8 章（85K tokens）做了一组消融实验：把文档切成 1 块、5 块、20 块、50 块、100 块、176 块，分别生成 LoRA，挂到 Gemma-2-2b-it 上跑 24 道问答。

结果就在这张表里：

| 条件 | 块数 | Rank | 文本覆盖 | 准确率 |
|------|------|------|----------|--------|
| Baseline | 0 | 0 | 0% | **25.0%** |
| N=1 | 1 | 16 | 0.6% | 16.7% |
| N=5 | 5 | 48 | 2.8% | 8.3% |
| N=20 | 20 | 168 | 11.4% | 4.2% |
| N=50 | 50 | 408 | 28.4% | **0.0%** |
| N=100 | 100 | 808 | 56.9% | **0.0%** |
| N=ALL(176) | 176 | 1416 | 100.0% | **0.0%** |

**N≥50 之后，模型不再说人话**，输出退化为：

```
"îng îng îng îng îng îng îng îng..."
"первыхпервых первыхпервых..."
```

为什么？因为 LoRA 的本质是**低秩**扰动。Gemma-2-2b-it 的 hidden_dim 是 2304，rank-1416 的扰动**占了原始维度的 61%**——这已经不是"在原画上加几笔"，而是"在原画上盖了另一幅画"。**基础模型的"本我流形"被淹没了**。

详细分析见《[Doc-to-LoRA 实验——红楼梦 vs 超网络](https://mp.weixin.qq.com/s/rhfqd_BtYemPudSE-3k9hw)》。

────────────────────

【为什么 Sakana 没发现这个问题？】

Sakana 自己论文里给的实验数据，**都在"短文档 + 小 rank"的甜蜜区**。他们公开的训练参数显示，超网络训练时**最多见过 8 块拼接**——他们自己知道大 rank 会崩，所以在论文里**根本没把 N 拉大测**。

这不是"实验不充分"。这是**故意只展示甜蜜区**。

**Fugu 这次的"甜蜜区"是什么？** 是跑分。是那些没有时间和成本约束的标准 benchmark。如果让 Mollick 这种实战派去跑真实任务——30 分钟的延迟就出来了。

━━━━━━━━━━━━━━━━━━━━

◆ 第五章：Sakana 的固定翻车配方

━━━━━━━━━━━━━━━━━━━━

把 Doc-to-LoRA 和 Fugu 放在一起看，**Sakana 这家公司有一个非常固定的产品发布模式**。我把它整理成一张表：

| 步骤 | Doc-to-LoRA（2026-02） | Fugu（2026-06） |
|------|--------------|------|
| 1. 选一个性感的科学比喻 | 海马体 / 睡眠 / 记忆巩固 | 鱼群涌现 / 多智能体协作 |
| 2. 论文实验只展示"甜蜜区" | 短文档 / 小 rank / Gemma-2-2b | 精选 benchmark / 隐藏延迟 |
| 3. 闭口不提结构性硬上限 | rank 一拉就崩 | 30 分钟跑一个任务 |
| 4. 包装成"AI 的某个方向" | "AI 长期记忆" | "多智能体编排是未来" |
| 5. 卡时机发布 | ICLR 季 | Fable 5 全球下线后 10 天 |
| 6. 媒体集体高潮 | "AI 终于学会记忆了" | "对标 Fable 5" |
| 7. 实战派发现是垃圾 | 我们的红楼梦实验 | Mollick 的 30 分钟实测 |

**核心病灶完全一致**：

- 拿一个**有数学/物理硬约束**的方案
- **假装这个硬约束不存在**
- 用**漂亮的比喻和精选数据**让读者忽略硬约束
- 趁市场窗口期**卖一波信徒**

Doc-to-LoRA 的硬约束是：**rank 不能太高，否则 LoRA 不再是 Low-Rank Adaptation**。这是 LoRA 这套数学工具的本质约束——你绕不开。

Fugu 的硬约束是：**协调成本不能太高，否则多智能体不再"省时省钱"**。这是分布式系统的本质约束——你也绕不开。

**这两个约束都不是工程优化能解决的**。它们是数学/物理决定的硬边界，就像光速、像热力学第二定律。Sakana 反复在做的事，是**包装绕过这些边界的幻觉**。

━━━━━━━━━━━━━━━━━━━━

◆ 第六章：那么 Fugu 一无是处吗？

━━━━━━━━━━━━━━━━━━━━

也不是。我想把话说公允一点。

────────────────────

【Fugu 在哪些场景下其实有用】

**第一类场景：被 KYC 锁死的企业**

6 月 12 日 Anthropic 关掉 Fable 5 / Mythos 全球访问之后，很多企业一夜之间没了"最强 AI"。如果你是一个**不在乎延迟、只要"有 AI 能用"的企业**——Fugu 是个临时替代品。

这个场景 Fugu 是真的有用的。**但它的真正卖点不是"shoulder-to-shoulder with Fable 5"，而是"我比你不能用的 Fable 5 强"**。这个标准实际比较低。

**第二类场景：异构任务的简单路由**

如果你有一个**确实需要多种能力混合的任务**（比如一段需要先做 OCR、再做翻译、再做总结的工作流），Fugu 的"自动路由"比你自己手写 if-else 快一些。但**OpenRouter + LangChain 也能干这事**，而且免费、可控、不黑盒。

**第三类场景：研究多智能体协调本身**

TRINITY 和 Conductor 的论文，作为"多智能体强化学习"领域的学术工作，是有价值的。**让协调策略从端到端学出来**这件事本身值得研究。但**学术价值 ≠ 产品价值**。Sakana 把学术成果包装成 "对标 Fable 5" 的产品时，就过线了。

────────────────────

【真正能解决 Fugu 问题的方向是什么？】

**降低协调成本的根本路径，是让协调成本接近零——也就是把"多智能体"装回单个模型内部**。

这不是新概念。MoE（Mixture-of-Experts）本质上就是**模型内部的多智能体**：

- DeepSeek-V3 / R1 用 MoE 在单次前向传播中调度 256 个专家
- Mixtral 用 8 个专家 + Top-2 路由
- GPT-4 / Claude Opus 都被广泛猜测使用了 MoE 结构

**MoE 的协调延迟 ≈ 0**，因为它发生在单次 forward pass 内部。**Fugu 的协调延迟 ≈ N 次 LLM 调用之和**。这两个量级差几个数量级。

如果"多智能体编排"是未来——那它的未来在 **MoE 这种内嵌路由**上，而不是 Fugu 这种**外挂调度器**上。Fugu 走的是死胡同。

━━━━━━━━━━━━━━━━━━━━

◆ 第七章：写给被 Fugu 吹得动心的读者

━━━━━━━━━━━━━━━━━━━━

如果你正在被 Sakana 的营销打动——或者你的老板/客户被打动了——这里有几个**实用的判断标准**，帮你识别下一个"Sakana 式产品"：

**判据一**：**它是不是只展示甜蜜区数据？**
- Doc-to-LoRA 不展示 N>8 的数据 → rank-1416 直接崩
- Fugu 不展示真实任务的端到端延迟 → 30 分钟

**判据二**：**它的"创新"是不是只换了一层包装？**
- Doc-to-LoRA 的"超网络生成 LoRA" = HyperNetworks 2016 年就有 + LoRA 2021 年就有
- Fugu 的"learned orchestration" = AutoGen 2023 年就有 + MoE 多年前就有

**判据三**：**它有没有揭露"硬约束"在哪里？**
- 严肃工作会告诉你"在什么条件下不行"
- Sakana 式工作只告诉你"在什么条件下行"

**判据四**：**它的发布时机有没有可疑的"卡窗口"？**
- Doc-to-LoRA 卡的是 ICLR 2026 审稿季
- Fugu 卡的是 Anthropic 下线后的市场真空

**判据五**：**实战派怎么说？**
- 看 Mollick、Karpathy、Yann LeCun 这种没利益冲突的人怎么评价
- 不要看 VentureBeat、The Decoder 这种**靠流量吃饭的科技媒体**怎么评价

━━━━━━━━━━━━━━━━━━━━

◆ 收尾：鱼群很美，但不是所有鱼都该被神话

━━━━━━━━━━━━━━━━━━━━

Sakana 这个名字来自"鱼群"——他们的 narrative 一直是**"涌现的集体智能比单个大模型更强"**。

我承认这是一个**漂亮的科学愿景**。鱼群确实展现出涌现行为。生物大脑确实通过分工协作完成了进化中最复杂的任务。

但**漂亮的愿景不等于工程现实**。

愿景是 **"鱼群涌现智能"**。现实是 **"30 分钟跑一道编程题"**。
愿景是 **"AI 学会睡眠记忆"**。现实是 **"rank-1416 的时候 AI 开始念经"**。

**Sakana 的产品发布从来不是技术发布，是叙事发布。** 他们在卖故事。故事漂亮，听众买单。

我喜欢漂亮的故事。但当一个故事**连续两次**用同样的配方端上来——选一个性感比喻、藏起硬约束、卡个市场窗口期、收割一波信徒——

**我们就有义务把它戳穿，免得让更多读者被这种"叙事工程学"骗走时间。**

如果将来 Sakana 端出第三道菜，我们继续验证。不替任何一方说话，把账摊开。
