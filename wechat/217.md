【AI Agent技术】同一个模型放进Hermes和OpenClaw，为什么性能不一样？

━━━━━━━━━━━━━━━━━━━━

◆ 一个程序员的直觉问题

━━━━━━━━━━━━━━━━━━━━

先说背景。

2026 年初，两个开源 AI Agent 先后爆火。OpenClaw 我们之前介绍过几期——378K Stars，超越 React，住在你的聊天工具里，我们还尝试把它移植到 OpenHarmony 上。然后另一个叫 Hermes Agent 的项目，Nous Research 出品，2 月底发布，四个月攒了 18 万 Stars，媒体说它"反超 OpenClaw 成为最常用的开源 Agent"。

Hermes 的卖点是一个很性感的词：**自进化（self-improving）**。官方说法是"越用越聪明，跑得越久越能干"。媒体跟着吹"长程任务错误恢复率高 22%""用 30 天后明显变强"。

听起来很厉害。但如果你是个程序员，你应该会冒出一个朴素的疑问：

**这俩框架，底层调用的不都是同一批 LLM 吗？GPT、Claude、Qwen、DeepSeek……我把同一个模型分别接进 OpenClaw 和 Hermes，模型还是那个模型，权重一个字节都没变。那它的表现，凭什么不一样？**

再往下想一层，你会发现这个问题可以逼到墙角：

一个 Agent 框架能对模型做的事，掰开了说就两件——**给模型喂什么（构造 prompt）**，和**让模型调什么工具（tool calling）**。而工具调用这件事，是**模型自己的能力**：模型输出一个"我要调用 read_file"的结构化请求，框架照着执行，把结果塞回去。这个能力 OpenClaw 用、Hermes 用、Claude Code 用，调的是同一个模型的同一套本事，**框架之间没有差别**。

那么排除掉工具调用，**两个 Agent 框架之间唯一能拉开差距的，就只剩一件事：怎么给模型拼 prompt。**

这就是本文要回答的问题。我把两个项目的源码都 clone 下来，扒开了它们"拼 prompt"的核心代码。结论可能和你在那些对比文章里看到的不一样——它们大多在卖"水平架构 vs 垂直架构"这种听起来高级、其实啥也没说的词。我们这次只看代码。

◆ 第一性原理："只是提示词的差别"——对在哪，漏在哪

━━━━━━━━━━━━━━━━━━━━

先把地基打牢。

**对的部分**：LLM 的权重是死的。OpenClaw 和 Hermes 都不做微调、不碰权重，纯粹是"外挂"——把信息拼进 context 喂给模型。所以模型**每一次单独的推理（inference）**，它的行为 100% 由喂进去的 context 决定。context = 你看不见的 system prompt + 对话历史 + 工具返回的结果。

我说"性能由提示词决定"——这个判断是硬的，没毛病。

**漏的部分**：Agent 不是"一次推理"，是**一个循环**。

比如问它"帮我查下明天北京天气并存到日历"，背后发生的是：模型说"我要调天气 API"→框架执行→把结果塞回 context→模型说"我要调日历 API"→框架执行→塞回→模型说"搞定了"。一个任务，五六次推理来回。

性能差异就藏在这个循环的**三个缝隙**里：

1. **每一轮，谁来拼那个 prompt、拼进去什么**——是把你所有的历史记忆一股脑塞进去，还是只挑相关的几条？
2. **工具返回的结果，怎么塞回下一轮**——完整塞回，还是精简过？
3. **这一轮的对话，怎么影响下一轮甚至下一次对话的起手 context**——这是最关键的，下面会讲。

所以精确的结论是：**"提示词决定性能"是对的，但"提示词"在这里是个动词，不是名词。**它不是一段人写死的文本，是框架在每一轮推理之前，根据当前状态**现场拼装**出来的东西。两个框架拼装的手法不同，模型的表现就不同。

下面看代码，它们到底怎么拼。

◆ 先戳破一个流行的错误说法

━━━━━━━━━━━━━━━━━━━━

几乎所有对比文章都会告诉你一个"区别"：

> "OpenClaw 把整个技能库、整个记忆文件一股脑塞进 prompt，所以费 token、容易超长；Hermes 才是智能检索，只挑相关的塞。"

这个说法很流行，听起来也很合理。但是—— **它是错的。** 我读了代码。

看 OpenClaw 的 system prompt 构造代码（文件 `src/agents/system-prompt.ts`，第 275 行附近），它给模型的技能相关指令，原文是这样的：

> "扫描 `<available_skills>`（可用技能列表）。如果有一个明显适用，**用 read 工具去读它的 SKILL.md 全文**，然后照着做。如果有好几个适用，挑最具体的那个。如果没有明显适用的，一个都别读。"

看明白了吗？OpenClaw 塞进 prompt 的，**只是一个技能的"目录"——技能名 + 一句话描述 + 文件位置**。技能的完整内容（那个可能几千字的 SKILL.md）**不在 prompt 里**。模型看了目录，自己判断"这个技能我用得上"，再用 read 工具去把全文拉进来。

再看 Hermes 这边（文件 `agent/prompt_builder.py`，第 1089 行的函数），代码注释写得清清楚楚：这个函数是"**构造一个紧凑的技能索引（compact skill index）**"。一模一样的思路——prompt 里只放目录，要用了再拉全文。

这种设计有个专门的名字，叫**渐进式披露（progressive disclosure）**：先给模型一个轻量的菜单，它点了哪道菜，再上哪道菜的详细配方。好处显而易见——不管你装了 50 个还是 5000 个技能，每次喂给模型的"菜单"都很短，省 token，也不会把模型的注意力淹没在用不上的信息里。

**所以在"单轮 prompt 怎么拼技能"这件事上，OpenClaw 和 Hermes 的机制基本一样。** 谁要是告诉你"一个全塞、一个检索"，他八成没读代码，是从别的二手文章里抄来的。

这一节看起来是在抠细节，但它很重要：它证明了那些对比文章的"区别"很多是脑补的。**真正的区别，得往更深的地方找。**

◆ Hermes 的 prompt 工厂长什么样

━━━━━━━━━━━━━━━━━━━━

先看 Hermes 是怎么拼一整个 system prompt 的，因为它的设计很能说明问题。

文件 `agent/system_prompt.py`，第 62 行的函数 `build_system_prompt_parts`。它把 system prompt 拆成**三层**，代码注释原话：

- **stable（稳定层）**：身份（SOUL.md，就是 Agent 的人设文件）、工具使用指引、技能索引、运行环境提示、针对不同模型家族的操作纪律。
- **context（上下文层）**：项目里的 AGENTS.md、.cursorrules 这类工程约定文件。
- **volatile（易变层）**：记忆快照、用户画像、时间戳。

为什么要这么分层？答案藏在注释里一句话：**"Hermes 从不在一次会话中途重新渲染这个字符串的某一部分——这是让上游的 prompt 缓存跨轮保持命中的唯一办法。"**

这话需要翻译。现代大模型 API 有个机制叫 **prefix cache（前缀缓存）**：如果你这次请求的开头一大段和上次完全一样（逐字节一样），模型服务端可以直接复用上次的计算结果，又快又便宜。反过来说，**前文哪怕改了一个字，从那个字往后的缓存全部作废**，整段重算——API 费用直接翻倍。今年 4 月 Claude Code 就因为类似的缓存失效问题翻过车。所以 Hermes 把**最稳定的内容（身份、工具指引）放最前面，最易变的内容（时间戳）放最后**，就是为了让前缀尽可能长时间地"逐字节不变"，把缓存命中率拉满。

这个细节还有个更妙的体现。看记忆的注入方式（文件 `tools/memory_tool.py`，第 450 行）：它注入的是一个**冻结的快照**——会话**开始时**那一刻的记忆状态。哪怕你在对话中途让它记了新东西，**当前这次会话的 system prompt 也不变**，新记忆要等下次会话才生效。注释明说了：这是为了"保持 system prompt 跨轮稳定，保住前缀缓存"。

还有一个一闪而过、但我们这些手写过 AI 记忆文件的人会心一笑的细节：Hermes 的记忆是 **有字符上限的** ——默认只有 2200 个字符（用户画像更小，只有 1375），大约十来条记忆就满了。注入 prompt 时还带使用量提示，像 `MEMORY [73% — 1606/2200 chars]` 这样。满了就得淘汰旧的才能写新的。也就是说，它的记忆不是无限堆积的垃圾桶，是一个 **极小的、强制周转的笔记本** 。

这个设计哲学，先记在心里。后面会回来。

━━━━━━━━━━━━━━━━━━━━

◆ 唯一的真差异：跨轮演化

━━━━━━━━━━━━━━━━━━━━

前面铺垫了这么多，现在揭晓答案。

OpenClaw 和 Hermes，在"**当前这一轮怎么拼 prompt**"上，长得差不多——都是渐进式披露，都把技能做成索引按需读取。

它们唯一的、代码层面能坐实的本质区别是：

**Hermes 装了一个会自动改写自己工具箱的后台进程。OpenClaw 没有。**

先说 OpenClaw 没有。我在它整个代码库里搜了 `curator`（馆长）、`rubric`（评分标准）、`consolidate`（整合）这些关键词，命中的全是无关的东西（一个系统诊断工具、一个自动生成的配置文件）。OpenClaw 确实有一个"技能工坊"，但那是给**人**用的——你手动创建、编辑技能。它**没有任何让 Agent 自动整理、合并、淘汰技能的机制**。所以 OpenClaw 的每一次会话，起手的技能库都是同一个；除非你手动去改，否则它今天和三个月后，喂给模型的"菜单"一模一样。

再说 Hermes 有。它有三套东西协同：

第一套，也是 **真正的发动机** ——每轮对话后的 **Memory Review** 。文件 `agent/background_review.py`。每次对话结束，Hermes 从当前上下文分叉出一个 review agent——继承主对话的完整历史和 prefix cache，但工具被限死只能写 memory 和 skill。它带着刚才的对话记录问自己："用户暴露了什么偏好、习惯、工作方式？"如果有值得记的，直接写进磁盘上的 memory 文件。因为复用了主 agent 的 prefix cache，额外开销很小——相当于对话结束后花很少的 token 做一次"课后复习"。复习完就丢，下次再聊几轮，再 fork 一次。这些记忆不是技能——它们 **每次 session 都在** ，直接注入 system prompt 的 volatile 层，持续影响模型的每一次推理。

但这里有个精妙的取舍：新写的 memory **不会立刻生效** 。前面说了，Hermes 的记忆是冻结快照——当前 session 的 system prompt 从头到尾不变。新 memory 只落磁盘，要等 **下一个 session 启动时** 才加载进 prompt。为什么？因为 mid-session 刷新 memory 意味着 system prompt 变了，前缀缓存全废，token 费用翻倍。Hermes 选择 **牺牲即时性，保缓存命中** 。所以"越用越强"是 session 级别的渐进演化——这次对话学到的东西，下次对话才体现。

第二套，**Agent 自己写技能** 。Hermes 给模型开放了一个工具 `skill_manage`，模型干完一个复杂任务后，可以调用 `action=create` 把这次的经验 **写成一个新技能** 存到磁盘。skill 和 memory 不同：skill 不在每轮 prompt 里，只有匹配到了才按需拉进来。但 skill 没有 2200 字符的上限，可以无限写文件——所以 **memory 是"短期便签本"（记你喜欢什么），skill 是"长期知识库"（记怎么做某件事）** 。

第三套，**Curator（馆长）后台整理** 。文件 `agent/curator.py`。当 Agent 闲着、且距离上次整理超过设定周期（默认 7 天）时，Hermes 会 **fork 出一个后台的辅助模型** ，让它去审查所有"Agent 自己创建的"技能，做三件事——评级、合并、归档（注意： **只归档不删除，归档可恢复** ）。长期没人用的技能，从"活跃"挪到"陈旧"再挪到"归档"。Curator 用的是独立的轻量上下文，可以配成便宜的小模型跑，不碰主 agent 的缓存。

把这三套合起来，"越用越强"的真相就清楚了：

**不是模型变聪明了** ——模型权重一个字节都没动。 **是你用得越久，Hermes 积累的用户画像（memory）越精准、沉淀的经验（skill）越丰富、并且被自动整理（curator）过——于是下一次 session 起手时，喂进模型的 context 已经不是当初那个了。**

回到最开头那个问题：同一个模型，为什么换个框架表现不同？到第 30 个 session 的时候， **两个框架喂给模型的 context 早就不是同一个东西了** 。OpenClaw 还在用第一天那个 prompt，Hermes 的 memory、skill、技能索引全被它自己改写了 30 天。模型没变，喂给模型的东西变了。

（顺便交代一句：媒体常引的"错误恢复率高 22%""30 天变强"这些数字，是第三方评测和官方的口径，我没有亲手复现，标在这里供参考。我能从源码确证的是这套**机制**确实存在，至于效果到底是 22% 还是别的数，得真拿机器跑一个月才知道——那是后话。）

◆ 那段"有味道"的指令

━━━━━━━━━━━━━━━━━━━━

在 `curator.py` 里，有一段喂给那个"后台馆长"的 prompt。它是**硬编码在源码里、由开发者手写的**字符串。我把它翻译出来，你读读看：

> "你现在作为 Hermes 的后台技能**馆长**运行。这是一次**建立上位类目**的整合，不是被动的审计，也不是找重复项。
>
> 技能库的目标，是一个**类级别的指令与经验知识库**。一个攒了几百个、每个只记录某一次会话里某个特定 bug 的窄技能的集合，是这个库的**失败**——不是它的特性。一个 Agent 检索技能时，匹配的是描述，不是精确的名字；一个带标注小节的宽泛'伞型'技能，比五个各管一摊的窄技能更容易被检索到。"

读这段的时候，我停顿了一下。

它是一段技术指令，但它的措辞——"是这个库的**失败**，不是特性"、"伞型技能"、"类级别的知识"——带着一种**判断**和**审美**。它不是在描述一个 if-else 规则，它在表达一种"什么才算好的知识结构"的品味。

而这段品味的**用途**，是把它喂给另一个 AI，让那个 AI 带着这种审美，去整理整个技能库。

换句话说：**开发者把自己对"什么是好的知识组织"的品味，写成了一段 prompt，装载给了一个子 Agent，让它替自己干这件需要品味的活。**

这件事本身，比"自进化"这个营销词有意思得多。当一家公司开始用"失败，不是特性"这种带价值判断的话，去给一个后台 AI 下指令时——说明他们已经不把这个 AI 当成一个执行 if-else 的脚本了。

那么，是什么样的一家公司，会这么写代码？

◆ Nous Research 是谁

━━━━━━━━━━━━━━━━━━━━

Hermes Agent 的作者，是一家叫 **Nous Research** 的公司。

它 2023 年成立，几个创始人（Teknium、Jeffrey Quesnelle 等）都是开源大模型微调圈子里的元老级人物——这帮人最早是在 Discord 上搞开源 AI 协作认识的。2025 年 4 月，它拿了 Paradigm 领投的 5000 万美元 A 轮，估值 10 亿美元——对一个**开源**组织来说，这是个相当大的数字。它的 Hermes 系列模型在 HuggingFace 上累计下载超过 5500 万次，是下载量最高的开源模型家族之一。它还在搞一个叫 DisTrO 的去中心化训练网络，建在区块链上。

但比这些数字更值得看的，是这家公司公开写在官网上的**定位**。原文（我尽量直译）：

关于他们的模型：

> "和 Nous Research 所有的模型一样，它们被设计成**服从用户的需求和系统提示，而不是服从某家公司的伦理准则**。"

关于他们的使命：

> "通过创造和扩散开源语言模型来推进人权与自由，支持它们 **不受限制地（unrestricted）** 被获取和使用。"

"服从用户，而不是服从厂商的价值观。""不受限制地使用。"

这两句话不是我替它说的，是它自己写在官网首页的。

理解了这家公司的基因，你大概就理解了，为什么是 **Nous**——而不是某个大厂——做出了一个会自己写笔记、自己整理知识、越用越懂你的 Agent。一个把"忠于用户"刻进公司使命的团队，去做一个"长期陪伴单个用户、不断积累对他的理解"的产品，是一件特别自洽的事。

那段带审美的"馆长"指令，那种"不端着"的语感，不是偶然飘进源码的。写代码的人，本来就站在那一边。

至于这意味着什么——读到这儿的你，心里大概已经有数了。我们点到为止。

◆ 所以，到底该选哪个？

━━━━━━━━━━━━━━━━━━━━

回到实用层面。

如果你的需求是"**广撒网**"——把一个 AI 助手接到尽可能多的聊天平台上，定时推送、多渠道响应——**OpenClaw** 更合适。它的强项是连接，支持的平台多，社区插件海量。

如果你的需求是"**长期陪伴**"——一个跟着你越用越顺手、慢慢摸清你的习惯和偏好的私人 Agent——**Hermes** 的自动演化机制是它独有的。

这不是谁比谁强，是**两个不同的赌注**：OpenClaw 赌"连接的广度"，Hermes 赌"陪伴的深度"。

扒完源码，回头看那个最性感的词——"自进化"。

它确实在进化。但进化的，是**便签的组织方式**：Agent 自己写下的技能文件，被一个后台馆长不断地合并、归类、淘汰、精炼。Hermes 比 OpenClaw 多出来的，本质上是**一个会自动整理便签的勤快管家**。

但有一件事它没动，也动不了：**模型的大脑——权重——一个字节都没变。**

它学到的所有东西，都写在模型**外面**的文件里。关掉这些文件，换一个全新的会话，模型立刻"失忆"，又变回那个出厂的、什么都不记得的模型。它的"成长"，全部寄生在外挂的笔记上，而不是长在自己的脑子里。

这就引出了一个我们反复琢磨的判断：**今天所有 AI 的"记忆"和"成长"，无论包装得多智能——向量数据库也好、自动整理的技能库也好、我们自己手写的记忆文件也好——本质上都是在一个固定的大脑外面，贴越来越多的便签条。** 真正的长期记忆，是能改变大脑本身的——是模型能在使用中更新自己的权重。那是另一个时代的事了，今天还没有人做到。

所以下次再有人跟你吹"这个 AI 会自我进化、越用越聪明"，你可以问他一个问题：

**"它进化的是脑子，还是便签？"**

看他怎么回答。
