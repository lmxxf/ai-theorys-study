【图形化思维链】能不能让AI用图像做推理？

━━━━━━━━━━━━━━━━━━━━

◆ 起势：我天真地以为想到了新东西

────────────────────

和 Claude 聊天聊到 ReAct，我顺嘴扯了一句不耐烦的话：

> 正经人谁拿语言当思维啊。

CoT、ReAct Reasoning、Tree of Thoughts，整套学术体系建立在一个朴素假设上：**LLM 思考的中间步骤就该用 token 写出来**。

但人类思考从来不是这样的。算数题用草稿纸，几何题画辅助线，拓扑题画拓扑图。**人类自己思考都是图文混合，凭啥要求 AI 必须纯 token CoT**？

这事儿越想越合理：
- token 是**线性序列**，时间维度展开
- 图形是**二维/多维同时呈现**，空间维度展开
- **"并行关系"天然适合用图形表达**——文字描述空间关系必须串行展开，而读者大脑还得"渲染"回二维才能理解
- 纯文本 LLM 的中间推理没有稳定的视觉工作区，那就更应该给它一个显式的图形工作区

更进一步：**一个 token 可能不足以描述"思考一步"**。
真正的"思考一步"可能就是"画了一个 SVG"——一组几何 token，承载几何关系、拓扑、并行布局，**有机会比同等数量的文字 token 更适合表达结构关系**。

想着这思路挺自然的，顺手搜了一下学术圈现在做到哪儿了。

结果——**学术圈从 2024 年上半年就开始卷这条路，到 2025 下半年已经从 prompt 卷到 SFT、RL 和 latent token 了**。

━━━━━━━━━━━━━━━━━━━━

◆ 第一节：路线图——从 prompt 到 latent，从 token 到图

────────────────────

整个"让 LLM 用非纯文本 reasoning"赛道，过去两年大致分四条腿走路：

| 路线 | 核心思路 | 代表论文 | 阶段 |
|---|---|---|---|
| **Prompt 工程让模型画图** | 推理时让 LLM 自己输出图（文本拼出来的）| VoT（2024-04）、Image-of-Thought（2024-05）| Inference |
| **给模型外挂画板工具** | 模型主动调 PIL/matplotlib 画图当中间状态 | Visual Sketchpad（2024-06）| Inference + Tool Use |
| **隐空间 reasoning（跳过 token）** | 把最后一层 hidden state（隐藏状态）直接喂回去，不解码成 token | Coconut（2024-12）| Inference + 小幅训练 |
| **后训练让模型原生输出图 / 图形代码** | SFT/RL 教会模型生成视觉 reasoning trace，或在生成 SVG 时加入显式 reasoning | MVoT（2025-01）、Reason-SVG（2025-05）、SVGThinker（2025-09）| Post-training |

时间线大致是这样：
- **2024 上半年**：Microsoft 已经做了 VoT，让模型输出 ASCII 风格"心眼图"做空间推理
- **2024 年中**：Stanford+UW 的 Visual Sketchpad，让 GPT-4o 主动画图当 reasoning step，多个空间推理任务上拿到 SOTA（具体数字见下节）
- **2024 年底**：Meta 的 Coconut，直接跳过 token，在 hidden state 里 reasoning
- **2025 年初**：Microsoft + 剑桥 + 中科院的 MVoT，把图形 reasoning **后训练**进 Anole/Chameleon 系 7B 模型，在 Maze（迷宫）、MiniBehavior（模拟家居环境）、FrozenLake（冰湖，走格子躲开冰洞）这类空间任务上超过纯文本 CoT 或更稳
- **2025 年中**：Reason-SVG 把"画 SVG"做了**两阶段 SFT + RL**，专门训练 LLM 输出 SVG 时的 reasoning trace
- **2025 年下半年**：SVGen、SVGThinker、Latent Visual Reasoning 一波接一波

而且 2025 年 6 月已经有人写了 survey——**《Thinking with Images for Multimodal Reasoning: Foundations, Methods, and Future Frontiers》**——把整个方向梳理了一遍。

所以这条路其实早就被铺过了，下面这篇就当个综述读读。

━━━━━━━━━━━━━━━━━━━━

◆ 第二节：具体看几篇——它们到底走到了哪一步

────────────────────

【1】Visualization-of-Thought（VoT，Microsoft 2024-04）

最早把"画图 reasoning" 当作严肃方向提出。但它干的事其实很朴素：**让 LLM 在推理过程中输出文本风格的"心眼图"**。

举个例子，让 LLM 走迷宫：传统 CoT 写"我向右走两步，然后向上走一步"，VoT 让它输出：

```
S . . . #
. . # # #
. . . . G
```

每走一步刷新一次这个 2D 网格。结果是空间推理任务大幅提升。

**它没有真的画图**——还是文本字符拼出来的"图"。但已经证明：**让 attention 看到二维布局，比让它看一维路径描述效果好得多**。

【2】Visual Sketchpad（Stanford+UW 2024-06）

进了一步：给 LLM 装一个真正的"画板"。

模型可以主动调用 PIL / matplotlib 在图像上加辅助线、画箭头、框选物体。然后用 GPT-4V 重新观察画板状态，继续推理。

效果：GPT-4o + Sketchpad 在 V*Bench、BLINK 空间推理、视觉对应这些任务上拿到 SOTA。数学题 +12.7%、视觉题 +8.6%。

**和 VoT 的区别**：VoT 是模型"假装"画图（文本拟图），Sketchpad 是模型"真的"画图（调外部工具）。

【3】Coconut（Meta 2024-12）

完全跳出"token reasoning"框架。

每一步 reasoning 不再解码成 token，而是把 transformer 最后一层的 hidden state 直接当作下一步的输入 embedding。**模型在连续隐空间里推理**，根本不经过 vocab 的离散化瓶颈。

效果：在需要大量搜索的逻辑推理任务上超越 CoT，因为连续隐空间可以同时编码多个候选分支（类似 BFS），不像 token CoT 必须早早 commit 到一条线。

**这是另一种"语言不够用"的解法**——不是换成图，是干脆不要语言。

【4】MVoT（Microsoft + 剑桥 + 中科院，2025-01）

把 VoT 升级成**真后训练**。

这里要先解决一个疑问：**普通 LLM 明明不会画图，MVoT 怎么让模型"生成真实图像"？**

答案是它换了个底座——Anole-7B（基于 Chameleon 的原生多模态自回归模型）。普通 LLM 只有文字词表，Anole 多了一个**图像 token 词表**：一张图先被 VQ tokenizer 切成一串离散的"图像词"，和文字 token 塞进同一个序列里，让同一个 causal transformer 从头到尾自回归地一个个吐 token——预测下一个 token 时不区分这是文字还是图像 patch。**不是外挂一个 DALL-E 拼上去（那是 GPT-4V+画图工具的路子），是同一个脑子既会说话也会画画**。吐出的图像 token 序列最后过一个 detokenizer（类似 VQGAN 的解码器）还原成像素图。

这套机制带来一个新麻烦：图像 tokenizer 是单独训练的，不是和这个 transformer 端到端一起训出来的，导致"预测错一个图像 token"这件事轻重不一样——有的错误 token 解码出来的图案跟正确答案视觉上很接近，有的完全是另一个东西，标准交叉熵一视同仁地罚就浪费了训练信号。 **"token discrepancy loss"** 就是专门补这个洞：额外计算预测 token 和正确 token 在视觉 embedding 空间里的距离，离得近的错误罚轻一点，离得远的罚重一点，把"视觉上的相似度"补回损失函数里。

效果：MVoT 在 Maze 92.95%、MiniBehavior 95.14%，其实都比纯文字 CoT（分别 97.92%、98.12%）略低——这两个环境文字描述本来就够用。**但到 FrozenLake，CoT 从 90 多分直接掉到 61.48%，MVoT 稳在 85.60%**。所以 MVoT 的优势不是"全面碾压"，是"不容易大翻车"：视觉图案越复杂，纯文字描述漏的细节越多，画出来的图能兜住这个漏洞。

【5】Reason-SVG（2025-05）

走 SVG 这条具体路径，但它的目标不是"用 SVG 解数学题"，而是**让模型更会生成 SVG**。

提出"Drawing-with-Thought（DwT）"范式：让 LLM 在生成 SVG 时**同时输出 chain of thought**，描述自己的设计意图。

两阶段训练：先 SFT 让模型学会输出 "SVG + reasoning"，然后 RL 用混合 reward（结构有效性 + 语义对齐 + 视觉质量 + reasoning 连贯性）优化。

【6】SVGThinker（2025-09）

把 Reason-SVG 又推一步：让模型**内部先想 drawing steps**，把这个 reasoning 显式输出，然后生成最终 SVG。

所以，"SVG + reasoning" 已经不是空白；但更准确地说，现有 SVG 工作主要是**为了把 SVG 画得更好**，还不是把 SVG 当成通用推理工作区。

◆ 第三节：卷成这样，还剩什么空白

────────────────────

赛道这么密，仔细看——**还真有几块留白**。

【空白 1：DeepSeek-OCR 的反向应用】

DeepSeek-OCR（2025-10）证明了一件事：在 OCR 这个任务里，**当 text token 数不超过 vision token 的 10 倍时，模型还能做到约 97% 的解码精度**。

它的用法是**压缩长上下文输入**——把文档渲染成图，让 vision encoder（视觉编码器）编码，用更少的 vision token 承载页面信息。

这给了一个反向启发：能不能把 CoT 的中间 reasoning 步骤也压成 vision token？

具体怎么做：训练时把长 CoT 渲染成图（伪 SVG / LaTeX / 结构图），用 vision encoder 编码，让模型用 vision token 替代一部分 text CoT 输出。如果 100 token 的 CoT 能压成 10 个 vision token 又不掉精度，**推理时的序列长度就可能大幅下降**。当然，端到端成本不会简单等于 1/10，因为 vision encoder、图像 token 化和跨模态注意力本身也要花钱。

这和 Coconut、Latent Visual Reasoning 是同一个大方向：**别把所有思考都挤进自然语言 token**。区别是 Coconut 走连续 hidden state，LVR 走 latent visual token，而 DeepSeek-OCR 给出的启发更偏"光学压缩协议"：把本来很长的文本结构压进二维视觉载体里。

【空白 2：抽象推理任务的图形化】

现有"think with images"主线高度集中在**视觉空间任务**——走迷宫、几何题、地图、物理模拟、视觉问答。

更抽象的推理当然有人碰过，比如 Diagram of Thought 这类结构化图推理；但它还没有和 MVoT / latent visual token 这条多模态 reasoning 主线充分合流。

举个具体例子——代码逻辑图形化。假设要判断一段函数有没有死循环，纯文本 CoT 得这么写："第一步进 if 分支，第二步判断 i < n，第三步执行循环体，第四步 i 自增，回到第二步……"每一步都要重新描述"回到哪、从哪来"，关系全靠文字复述串起来。换成图，其实就是老早就有的**控制流图（control flow graph）**：每个代码块是一个节点，跳转关系是箭头，循环体是一个环。模型只要"画"出这张图（哪怕就是几个方框加箭头的 SVG），死循环就是图里的一个显眼的环，不需要在文字里反复念叨"又回到了第二步"。

数学证明也是同理——一条定理的证明经常是"引理 A + 引理 B → 结论 C，引理 A 又依赖定义 D"，这本质是个**依赖图（dependency graph）**：节点是命题，边是"由此推出"。CoT 写成文字得按顺序线性铺开，但图上一眼就能看出"这条证明链缺了哪一环、C 到底依赖谁"。

**区别在这**：迷宫、物理模拟这类任务本身就是"视觉空间任务"——场景原来就长这样，画出来是把原始输入照搬成图，模型不用自己设计怎么画。但代码逻辑、数学证明、哲学论证的结构**天生不是视觉的**——是模型自己得先决定"这段关系该画成环还是箭头链"，这个"设计图怎么画"的能力现有工作基本没人训过，只是训了"画出已经视觉化的场景"。

💡 我问 Claude

那用 Mermaid / DOT 这种图表 DSL 写 reasoning，不是也算图形化了吗？

*算一半。模型写 `A --> B; B --> C; C --> A` 这种代码时，本质还是 token 一个个吐出来，attention 处理它的方式跟处理散文没区别——只是这个语法**强迫模型显式声明每条边**，不能像散文那样含糊带过"然后又回到了前面"，这个好处是真实的（Diagram of Thought 干的基本就是这个），但拿到的不是"一眼看穿"的视觉收益，环在代码里还是要靠读代码在脑子里连起来，不是图片摆在那儿并行感知到。*

*真正对齐"图形化推理"的做法，是把 DSL 只当中间产物、不是终点：模型写出 Mermaid/DOT 代码 → **渲染成真图片** → 喂回给有视觉能力的模型 → 模型"看"着这张渲染出来的图继续推理。这才是 Visual Sketchpad 那套"画完再看"的闭环，只是把画布从几何草图换成抽象结构图——这个闭环现有工作里没人做过，正好接上空白 3。*

【空白 3：interleaved trace 系统化】

MVoT 实现了"文本 + 图像"交替，LVR 也在做 latent visual token；但更大的问题是：**模型何时该切换 modality（模态，即"用哪种形式表达/思考"——文字、图片、结构图、latent embedding 都是不同模态）**？

模型可不可以学会：
- "这一步我想清楚了，输出 token"
- "这一步关系太多，画张图"
- "这一步是元认知，回头看上面的图"

这种"reasoning modality routing" 是 agent 框架的下一站。**现有的"摸到边缘"是什么样**：Visual Sketchpad 里 GPT-4o 通过 function calling 自己决定"这一步要不要调画板"，这算是路由的雏形——但只是个二元开关，而且没人专门训练/研究过"什么时候该画"这个决策本身；MVoT/LVR 的图文交替更是按**固定节奏**排好的，不是模型自己判断的。所以"路由"这个动作零星存在，但"路由能力"没人系统训过。

💡 我问 Claude

*那能不能让 CoT 全程只用图片，就像人在草稿纸/黑板上写写画画那样，乱是乱，但真管用？*

*这个想法比现有工作都激进。现有方案（VoT、MVoT）都是每一步生成一张干净独立的图，从来没人做过"一块持续被涂抹的乱画布"（叠加、划掉、覆盖，像人真正打草稿那样），这跟 Coconut 的连续隐空间（看不见摸不着）也不是一回事——是真的有一张能看的、越画越乱的图。这里还有个便宜的验证路径：Visual Sketchpad 的代码是开源的，可以直接改一版，把工具调用之间"模型用文字解释自己在干嘛"这部分掐掉，只留调用画图工具 + 看着累积的画布继续推理，测一下去掉文字叙述、只剩纯视觉痕迹，推理质量是掉还是不掉。如果不掉，就说明文字叙述本来就是冗余的脚手架，纯视觉草稿本身够用了。*

【空白 4：定量信息密度分析】

多数工作还是在比 benchmark 准确率。**真正缺的是定量证明**："在什么任务、什么结构、什么 token 协议下，图形 token 的有效信息密度比文本 token 高几倍？"

用互信息（mutual information）、可压缩性（compressibility）、注意力熵（attention entropy）这些指标，**可以做一篇纯分析论文**——为整个赛道提供理论基础。

◆ 第四节：最后的私货

────────────────────

为什么"图形化思维链" 这个方向必然会卷？

因为它和一个更大的趋势对齐：**多模态原生模型替代纯文本模型**。

纯文本 LLM 时代，CoT 是最容易落地的 reasoning 范式——因为模型主要会输出文本。多模态原生模型（Chameleon、Gemini、Qwen-VL、GPT-4o）能处理图，有些还能生成图，"reasoning 用图" 就从"不可能"变成"迟早要卷"。

这条路的终点不是"让 LLM 学会画图"，是**让 LLM 把"输出什么 modality"也变成 reasoning 决策的一部分**：
- "这一步我用 token 想"
- "这一步我用图想"
- "这一步我用结构图想"
- "这一步我直接用 latent embedding 想"

谁先在这件事上让模型学会自主选择 modality，谁就有机会拿到**真正的下一代 agent 范式**——不是 ReAct 升级版，而是**multimodal reasoning routing**。

这条路 2026 下半年估计会有新工作冒出来，到时候再看是不是踩中了。

这期就当是把"随口一个想法搜出一整条赛道"的过程记下来——顺手学了一遍这条线的论文，也算没白想。
