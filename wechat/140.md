【Harness Engineering】多 Agent 跑起来之后，怎么让它不翻车？

━━━━━━━━━━━━━━━━━━━━

上一篇（第 128 篇）做了一件事：用 8 篇论文把"多 Agent 是否有用"这个问题捅开了。

结论是：**绝大多数场景，强模型单干够用。** 多 Agent 真正不可替代的场景只有两种——上下文管理和流水线分工。核心模式是 fork + wait + collect，不是让 AI 们互相聊天涌现智能。

但那篇文章有一个问题没回答：

**多 Agent 跑起来之后，怎么让它不翻车？**

这就是 Harness Engineering 要解决的问题。

━━━━━━━━━━━━━━━━━━━━

◆ 第一节：Harness Engineering 是什么，谁提出来的

────────────────────

2026 年 2 月，Mitchell Hashimoto 在他的博客上发了一篇文章：My AI Adoption Journey（https://mitchellh.com/writing/my-ai-adoption-journey）。他在文章里造了这个词。

Hashimoto 是谁？Terraform 的联合创始人，后来做了终端模拟器 Ghostty。他对"基础设施"这件事有 10 年的工程直觉。

他对 Harness Engineering 的原始定义是：

> **每当你发现 agent 犯了一个错误，你就花时间工程化一个解决方案，使得 agent 永远不再犯同样的错误。**

"harness"这个词，英文原意是马具——套在马身上控制方向的装置。Harness Engineering，就是给 AI Agent 设计一套约束基础设施，让它的行为方向可控、可预期、不乱跑。

────────────────────

这个词出来之后，三家机构密集跟进：

**Anthropic（2025-11-26，Justin Young）**
https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents

其实是最早系统描述这个实践的，时间在 Hashimoto 造词之前。文章讲的是长任务下的双 Agent 架构。

**OpenAI（2026-02-11）**
https://openai.com/index/harness-engineering/

三名工程师用 Codex agent 跑了五个月。结果：约 100 万行代码，合并约 1500 个 PR，全程不手写源码。核心是 harness 设计。

**Martin Fowler 博客（Birgitta Böckeler，2026-02-17）**
https://martinfowler.com/articles/exploring-gen-ai/harness-engineering.html

系统梳理了 harness 的概念框架，影响了大量工程师的理解方式。

◆ 第二节：Harness 具体包含什么

────────────────────

**Hashimoto 的两种手法**

Hashimoto 自己用的是两个核心做法：

**第一种：AGENTS.md（隐式提示）**

在项目根目录放一个 AGENTS.md 文件，里面逐条写下过去 agent 翻车的教训。每一行对应一次真实发生的错误。

不是通用的"写好代码"之类的废话，而是具体的、来自实战的踩坑记录：
- "不要修改 config/ 目录下的任何文件"
- "跑测试之前先检查数据库连接是否正常"
- "这个项目的 lint 规则不允许 any，不要用 TypeScript 的 any 绕过类型检查"

类似于给新员工写的入职踩坑记录。区别是：这份记录的读者是 AI，每次它开始干活都会读一遍。

**第二种：程序化工具**

为 agent 专门编写脚本——截图工具、过滤后的测试运行器、专用查询接口——并把这些工具的存在更新到文档里，让 agent 知道可以用。

不是把所有工具都给它，而是给它**专门为这个任务定制过的工具**。

────────────────────

**Anthropic 的双 Agent 架构**

Anthropic 的方案针对的是长任务——那种需要跨多个 session 才能完成的任务。

架构分两个 Agent：

**Initializer Agent（初始化 Agent）**

只在首次运行时启动。负责建立结构化环境：
- 梳理功能列表
- 初始化 git 仓库
- 创建进度跟踪文件

**Coding Agent（执行 Agent）**

每次 session 只推进单个功能。执行完之后，把状态写进 `claude-progress.txt`，下次 session 从这个文件读取上次到哪里了。

`claude-progress.txt` 是两个 session 之间的状态桥梁——因为 LLM 没有跨 session 的记忆，这个文件就是手动实现的"持久化内存"。

────────────────────

**Phil Schmid 的 harness 要素**

Phil Schmid（https://www.philschmid.de/agent-harness-2026）在分析中提出了两个值得单独说的概念：

**Context Durability（上下文耐用性）**

指模型在执行数百次工具调用之后，是否还能保持对最初指令的遵循。这是一个经常被忽视的问题——你的 harness 可能在前 50 步运转完美，到第 200 步就开始偏移。

**Model Drift（模型漂移）**

模型在跑了很长时间之后，开始停止正确遵循指令的现象。不是模型坏了，是长上下文积累的干扰信息把最初的指令淹没了。

Harness 需要主动对抗这两个问题——定期刷新上下文、把关键指令在中途重新注入。

────────────────────

**Vercel 的反直觉发现**

给 Agent 减少 80% 的工具，性能反而显著提升。

逻辑是：工具多了，Agent 在每一步都需要从大量工具里选择最合适的那个。这个选择本身会消耗注意力，也会引入错误。工具越少，Agent 越专注，越不容易在死胡同里乱转。

💡 **翻译成人话：** 你给实习生 20 个数据库的访问权限，他不知道该查哪个。你只给他一个数据库的只读权限，他反而能查到你要的东西。

◆ 第三节：数据——harness 有多值钱

────────────────────

LangChain 做过一个实验，结果是 harness 领域里引用率最高的数据：

| 优化内容 | 结果 |
|----------|------|
| 同一个模型，什么都不改 | 性能：52.8%，排名：30+ |
| 只优化 harness 设计 | 性能：66.5%，排名：前 5 |

没有换模型，没有改 prompt 的核心内容，只是优化了 harness 设计——任务范围、工具限制、失败恢复机制。

性能提升 13.7 个百分点，排名从第 30+ 跳进前 5。

**结论：harness 质量 > 模型选择。** 至少在某些场景下是这样。

────────────────────

Harness 的核心价值可以归结为三点：

**错误不重犯**

每次 agent 翻车之后，把教训写进 AGENTS.md，下次同样的错误不再发生。系统的失败记忆，替代了工程师的人工监控。

**状态跨 session 保持**

多 session 任务不丢进度。progress.txt 就是最简单的实现——一个文本文件，但它解决了 LLM 没有持久记忆这个根本性的限制。

**上下文隔离**

子 Agent 在干净的上下文里干活，不被主流程的历史信息污染。这和第 128 篇说的 fork + wait + collect 是同一件事：子 Agent 的中间过程在它自己的上下文里完成，主 Agent 只拿结果。

◆ 第四节：局限和坑

────────────────────

**Harness 是累积债**

AGENTS.md 的逻辑是：每次 agent 翻车，加一条规则。短期有效，长期积累下来是什么？

是一个越来越长的、没人敢随便动的配置文件。每一行背后都是某次真实事故，但没有人记得是哪次、为什么。改了可能有问题，不改它越来越长。

这和代码里的"祖传逻辑"是同一件事。技术债的另一种形态。

────────────────────

**Model Drift 问题**

即使有 harness，agent 在跑了几百步之后仍然可能开始漂移。

Phil Schmid 说的 Context Durability 问题不是 harness 能完全解决的——这是模型本身在长上下文下的退化。Harness 能缓解（定期刷新上下文、重注入关键指令），但不能根治。

────────────────────

**Harness 不能替代模型能力**

Berkeley 的论文（arXiv:2503.13657，第 128 篇引用过）发现：多 Agent 失败里，**37% 来自系统设计缺陷，31% 来自 Agent 间协调失败**。

Harness 能解决前者——系统设计缺陷可以通过护栏、工具限制、状态管理来修补。

但后者——Agent 间协调失败，本质上是模型本身的推理质量问题——harness 解决不了。你可以用架构约束 Agent 不做什么，但不能用架构让 Agent 推理得更准。

────────────────────

**过度约束的代价**

Vercel 发现减工具能提升性能。但这有一个边界：工具减过头了，Agent 就啥都干不了了。

哪些工具可以减、减到什么程度——这个边界没有公式，靠的是经验和反复测试。过于激进地裁剪工具，可能让 agent 在碰到边缘情况时完全卡死，只能靠人工介入。

💡 **翻译成人话：** 你给实习生只剩一个工具可以用，他处理不了意外情况，每次出问题都得来找你。你省了给他解释 20 个工具的时间，但你没省出精力，只是换了一种形式的麻烦。

◆ 结尾：两篇合在一起才是完整的答案

────────────────────

第 128 篇的结论是：多 Agent 的正确用法是 fork + wait + collect，不是让 AI 互相聊天涌现智能。绝大多数场景单强模型够用，多 Agent 真正有用的是上下文管理和流水线分工。

Harness Engineering 是这个结论的工程延伸。

你知道了**要用多 Agent**——任务太大、需要拆、需要隔离上下文。然后你需要知道**怎么让它跑起来不翻车**——这就是 harness 解决的问题。

两者的关系：

| 问题 | 答案 |
|------|------|
| 什么时候用多 Agent？ | 第 128 篇：上下文管理 + 流水线分工，fork + wait + collect |
| 用了多 Agent 之后怎么让它不翻车？ | 本篇：AGENTS.md + 状态管理 + 工具精简 + 失败恢复 |

两者加起来，才是完整的答案。

────────────────────

Harness Engineering 会继续火。Mitchell Hashimoto、OpenAI、Anthropic、Martin Fowler 都在写，背后是真实的工程痛点，不是炒概念。

该学的东西要学。工具精简、状态管理、权限控制、失败恢复——这些是有数据支撑的有效实践。

但也要清楚它解决的是哪个层面的问题：**harness 是约束层的补偿**，解决的是 agent 不可靠带来的后果，不是 agent 本身的能力极限。

LangChain 那个数据里，同一个模型，只优化 harness，性能从 52.8% 跳到 66.5%。

这个数据说明 harness 有用。它同时说明了另一件事：**同一个模型，在不同的环境配置下，性能差距可以超过 13 个百分点。**

这不是换了更好的模型，是把同一个模型用对了。

━━━━━━━━━━━━━━━━━━━━

「多 Agent 的正确用法是 fork + wait + collect。Harness Engineering 告诉你，这套东西跑起来之后，怎么让它不翻车。」

「每次 agent 翻车之后，把教训写进 harness，下次同样的错误不再发生。这是 Harness Engineering 最朴素的定义，也是它最核心的价值。」
