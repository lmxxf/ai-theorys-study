# 万物皆可 Harness 之 Coding Agent Harness（01）：上下文是怎么组装的

> 这是「万物皆可 Harness」系列的第一篇。本系列围绕一件事：给 AI Coding Agent（目前是 Claude Code）构建团队级的外围 Harness，让 AI 写代码这件事在团队中更统一、更可控、更高效。

---

## 万物皆可 Harness

Harness 是 2026 年 AI 工程领域最高频的词之一。Anthropic、OpenAI、Martin Fowler、LangChain、DeepMind，近三个月内都在研究和讨论这个概念。但大多数文章在讲"Harness 是什么"，很少有人往回追一步：这个词到底什么意思？为什么偏偏选了它？

追到词源。1300 年，古法语 *harnois*，最初指骑士的战甲。后来含义迁移到驾驭牲畜的挽具，套在马身上让马能拉车的那套装备。1690 年代出现了比喻义：**to control for use as power，控制某物，使其产生力量。**

注意：从来都不是"限制"的意思。Harness 不是笼子，是挽具。马没有挽具照样能跑，但拉不了车。挽具不限制马的力量，挽具让马的力量**可用**。

沿着这个定义看，Harness 其实无处不在：

- **Test Harness**，让测试能跑起来的基础设施，不是限制代码，是让"验证"这个动作可执行
- **Claude Code**，Anthropic 围绕 Claude 模型做的 Coding Agent Harness，不是限制模型，是让模型的编程能力在实际项目中可用
- **团队 Harness**，围绕 Claude Code 做的外围控制系统，不是限制 Claude Code，是让团队统一、高效地使用它

同一个模式一层套一层：**只要存在"有力量但还不够可控"的执行体，就需要一套让它的力量可用的控制系统。这就是 Harness。**

所以，万物皆可 Harness。这不是修辞，是这个词的定义决定的。

---

## Coding Agent Harness 和 Agentic System Harness

不过，得区分清楚一件事。

Anthropic 和 OpenAI 最初谈 Harness 时，说的是 **Coding Agent Harness**，围绕 Claude Code、Codex 这类 AI 编程工具的控制系统，解决的是"AI 写代码的过程如何可靠、如何长时间持续工作"。Anthropic 专门发过两篇文章讲 long-running agent 的 Harness 设计（2025-11、2026-03），核心问题是 Agent 需要跨多个上下文窗口、持续数小时甚至数天地工作，每次新 session 从零开始没有记忆。

这个词后来被快速泛化。LangChain 在讲通用 Agent 的 Middleware，Gartner 在讲 Agentic System 的治理框架，都叫 Harness。

但这两者不是一回事。Broda 和 Miller 在 Agentic Mesh（2026-03）里有句话说得很准：

> "An enterprise harness is not a larger coding harness. It is a different architectural form."

**Coding Agent Harness 和 Agentic System Harness 是两种不同的架构形态。** 前者控制"AI 写代码的过程"，后者控制"AI 驱动的业务系统的运行时"。前者能直接复用 Git、CI/CD、Linter 这些软件工程 50 年的基础设施，后者要处理运行时编排、多 Agent 协调、业务语义验证，完全不同的问题域。

打个比方：汽车生产线的质检系统和自动驾驶的决策系统，都叫"控制系统"，但设计逻辑完全不同。

本系列聚焦 **Coding Agent Harness**。Agentic System Harness 是另一个话题。

## 给 Harness 做 Harness

先说清楚 Claude Code 的定位。Anthropic 官网的描述是"agentic coding tool"。用 Agent = Model + Harness 这个公式拆开看：Claude 模型是 Model，Claude Code 是围绕模型的 Harness，两者合在一起才是一个完整的 Coding Agent。所以 Claude Code 既是一个 Agent 产品，也是一个 Harness 实现。本系列关注的是它作为 Harness 的那一面。

Claude 模型是马，Claude Code 是挽具，让模型的编程能力在工程场景中可用。

但当一个团队要统一使用 Claude Code，统一编码规范、统一安全策略、统一质量关卡、统一审计追踪，就需要在 Claude Code 外面再包一层控制系统。

**给 Harness 做 Harness。**

这不是文字游戏。OpenAI 已经干过。

2026 年 2 月，OpenAI 发了一篇 Harness Engineering 文章。他们用 Codex 开发了一个百万行代码的产品，三个工程师，零行人写代码，全由 Codex Agent 生成。怎么做到的？给 Codex 构建了外围 Harness：AGENTS.md 定规范，docs/ 目录做知识库，CI/Linter 做机械验证，架构约束通过结构化测试强制执行。

OpenAI 造的 Codex，他们当然对 Codex 从原理到内部了如指掌。但他们**选择**站在接口层来做 Harness、来讲 Harness，通过指令文件、PR、CI 反馈、可观测性数据来控制 Agent 行为。这个选择是刻意的：站在接口层做，方法论就是可复用的，别人也能照着造。发出来的是方法论，不是内部工具文档。

2026 年 3 月底，Claude Code 的源码因为 npm 发包时遗漏了 .npmignore 配置而意外泄露，约 51 万行 TypeScript 代码。这给了一个额外的视角：不只在接口层使用挂载点，还能看到挂载点内部是怎么实现的、为什么这么设计、边界在哪。

所以这个系列做的事很明确：**致敬 OpenAI 的 Harness Engineering，他们对自家 Codex 做的事，我们对 Claude Code 做一遍。** 读它的源码，理解它的内部设计，然后在它预留的挂载点上构建团队级的 Harness。

## 地基和脚手架

在进入技术细节之前，先建立一个贯穿全系列的分析框架。

Martin Fowler 网站上的 Harness Engineering（Böckeler, 2026-04）区分了两类控制手段：

- **计算型控制**，确定性的，CPU 执行，快速可靠。比如 Linter、Git Hook、文件权限。
- **推断型控制**，概率性的，LLM 执行，灵活但不确定。比如 AI Code Reviewer、AI Planner。

换个更直觉的说法：**地基**和**脚手架**。

Anthropic 在 2026 年 3 月的文章里有句话很到位：

> "Every component in a harness encodes an assumption about what the model can't do on its own, and those assumptions go stale."

Harness 里的每个组件都在假设"模型还做不到某件事"。假设会过时，模型变强了，脚手架就该拆。但地基不会过时。文件系统权限是确定性的，CI 验证是确定性的，Git Hook 是确定性的。**概率性执行体永远需要确定性约束，不管模型多强。**

LangChain 说得更直白：**"You can't prompt your way to HIPAA compliance."** 靠写 prompt 是满足不了合规要求的。

每分析一个 Claude Code 的设计，都会标注它是地基还是脚手架。这决定了构建团队 Harness 时，精力该往哪放。

## 本篇主角：上下文的组装机制

Claude Code 有六个挂载点：CLAUDE.md、Hooks、Permissions、MCP、Skills、Agents。

本篇从 CLAUDE.md 开始，因为它是最基础的，上下文注入的入口。

一个关键认知：**上下文是有限资源。** Anthropic 在 2025 年 9 月的 Context Engineering 文章里说：

> "Context must be treated as a finite resource with diminishing marginal returns."

往模型的上下文窗口里塞 1000 字和塞 10000 字，效果不是线性增长，后者甚至可能因为信息过载而更差。LangChain 给这个现象起了个名字叫 **Context Rot**（上下文腐烂）。

CLAUDE.md 不是 System Prompt。System Prompt 是 Anthropic 硬编码在 Claude Code 内部的，用户看不到也改不了。CLAUDE.md 是用户自己写的 Markdown 文件，存在文件系统里，能被 Git 追踪，能被团队共享。你在里面写"我们用 Go，不用 Java"、"测试覆盖率不低于 80%"，Claude 在每次对话时都会读到这些规则并倾向于遵守。但注意，这里是"倾向于"，不是"保证"。模型是概率性执行体，CLAUDE.md 里的规则是通过上下文影响模型行为，不是确定性约束。想要确定性保证，需要靠 Hooks、CI 这些地基层的机制，这也是为什么我们要构建外围 Harness。

OpenAI 的 Codex 有 AGENTS.md，Cursor 有 .cursorrules，同一个设计范式。

但"读一个 Markdown 文件然后组装进上下文"这件看似简单的事，Claude Code 用 1100 行 TypeScript 做出了六个值得拆开看的工程设计。

---

## 设计一：六级加载

官方文档说 CLAUDE.md 有四个层级。打开源码，实际是六个：

```
Managed → User → Project → Local → AutoMem → TeamMem
```

源码注释写得清楚：**后加载的优先级更高**，模型会更关注后加载的内容。

六层各管什么：

| 层级 | 路径 | 谁写的 | 管什么 |
|------|------|--------|--------|
| Managed | `/etc/claude-code/CLAUDE.md` | 企业 IT 管理员 | 安全策略、合规底线 |
| User | `~/.claude/CLAUDE.md` | 开发者个人 | 个人偏好 |
| Project | `项目根/CLAUDE.md` | 团队 Lead | 项目规范、架构约定 |
| Local | `项目根/CLAUDE.local.md` | 开发者个人 | 个人的项目级覆盖（不进 Git） |
| AutoMem | `~/.claude/projects/.../memory/` | Agent 自动写 | 跨会话记忆 |
| TeamMem | `~/.claude/projects/.../memory/team/` | 团队同步 | 团队共享记忆（Team/Enterprise 版功能） |

需要说明的是，TeamMem 是 Claude 的 Team Plan 和 Enterprise Plan 的功能，指的是模型层面的跨会话共享记忆。而本系列要做的"团队 Harness"是另一回事：通过 Project 层（Git 仓库里的 CLAUDE.md）和 Managed 层（企业 IT 部署的全局策略），确保团队每个人的 Claude Code 实例都加载相同的编码规范、安全策略和质量关卡。前者是模型的记忆，后者是确定性的配置分发。

这是一个**权力级联系统**。类比 CSS：浏览器默认样式 → 用户样式 → 作者样式 → 行内样式，每层可以覆盖上一层。

这里有个反直觉的地方：Managed 层（企业安全策略）最先加载，优先级**最低**。安全策略不该最高优先吗？

实际上没问题。Managed 层部署在 `/etc/claude-code/`，普通开发者根本没有写权限。安全约束靠文件系统权限来保证，不靠"模型更关注"。**真正的安全边界用 CPU 守，不用 LLM 守。** 加载顺序解决的是另一个问题：在模型注意力有限的情况下，离当前工作最近的规则最突出。

## 设计二：目录遍历

加载 Project 和 Local 层时，Claude Code 从当前工作目录一路向上遍历到文件系统根目录，收集沿途所有规则文件。收集从下往上，但加载从上往下。**越靠近当前工作目录的规则，优先级越高**。

Monorepo 场景下这个设计自然生效：

```
/workspace/
  CLAUDE.md              → "TypeScript，严格模式"
  packages/
    frontend/
      CLAUDE.md          → "React 18，Server Components"
    backend/
      CLAUDE.md          → "Go，RESTful"
```

在 `packages/frontend/` 下启动 Claude Code，两份规则文件都加载，frontend 的覆盖 workspace 的。

不需要发明新的配置语法。**目录结构就是作用域结构**，文件系统的层级本身就是天然的 Scope 机制，Anthropic 直接复用了它。

## 设计三：文件引用与防循环

规则文件支持用 `@` 语法引用其他文件，`@./docs/api-rules.md` 会把那个文件的内容拉进来。这样主文件只写概要，具体规则分散在独立文件里。

OpenAI 在 Codex 文档里有句话说得好："Instead of treating AGENTS.md as an encyclopedia, treat it as the table of contents." 不要把规则文件当百科全书，要当目录用。`@` 引用就是把"目录"这个理念变成了工程机制。

有引用就有循环引用的风险，A 引用 B，B 引用 A。Claude Code 用了两层防御：最大递归深度 5 层加上已处理路径去重。循环引用只会被静默跳过，不报错，不崩溃。

路径提取不是正则匹配，而是用 Markdown 解析库的 Lexer 做 token 级分析。代码块里写 `@./scripts/deploy.sh` 不会被误识别为引用，解析器知道那是代码，不是正文。细节不大，但用解析器而不是正则，反映了工程判断力的差别。

## 设计四：条件加载

六个设计里最精到的一个。

`.claude/rules/` 目录下的规则文件，支持指定"只在操作特定路径的文件时才生效"：

```markdown
---
paths: ["src/api/**", "src/routes/**"]
---
# API 开发规范
所有端点必须有 JSDoc 注释...
```

Claude 在编辑 `src/api/` 下的文件时，这条规则才被加载进上下文。编辑 `src/components/Button.tsx` 时，它根本不出现。

这就是在工程层面管理**注意力预算**。上下文有限、边际递减，好的 Harness 不是给模型更多信息，而是**在正确的时机给正确的信息**。

模型不需要"决定"自己该看什么规则。Harness 已经替它做了这个决定。这是确定性的前馈控制，不依赖模型的判断力，由 CPU 直接决定加载哪些规则。

---

## 设计五：注释剥离

CLAUDE.md 是 Markdown，支持 HTML 注释 `<!-- ... -->`。Claude Code 加载时会剥离所有块级 HTML 注释。

效果是这样的：

```markdown
<!-- 维护者注意：这条规则因为 2025Q3 生产事故而加，
     架构迁移完成后可移除。Issue: #1234 -->
所有数据库写操作必须在事务中执行。
```

HTML 注释是给人类维护者看的，解释这条规则的历史背景和移除条件。Claude 看不到，不浪费注意力。正文是给模型看的规则。

同一份文件，人和 AI 各看各的部分。比维护两份文件（一份给人看的文档加一份给 AI 看的规则）优雅得多。

---

## 设计六：多 Agent 协作时的去重

Claude Code 支持多个 Agent 同时工作，每个 Agent 在自己的 Git Worktree（工作树）里独立操作。但 Worktree 嵌套在主仓库目录下，向上遍历收集规则文件时，会同时经过 Worktree 根目录和主仓库根目录，两边内容一样，不去重就会重复加载。

Claude Code 通过检测"当前是否在嵌套 Worktree 中"来跳过重复的文件。源码注释里直接贴了对应的 GitHub Issue 编号，这是从真实用户反馈中发现并修复的问题。

这个点离大多数开发者比较远。但它发出的信号很明确：**多 Agent 并行协作场景下的配置隔离，已经是基础设施级的问题了。**

## 六个设计，全是地基

回到地基/脚手架框架，给这六个设计分个类：

| 设计 | 地基/脚手架 | 为什么 |
|------|------------|--------|
| 六级加载 | 地基 | 权力分层是组织问题，模型再强也要分谁说了算 |
| 目录遍历 | 地基 | 文件系统作用域是确定性结构 |
| 文件引用 + 防循环 | 地基 | 引用管理是编译器级别的问题 |
| 条件加载 | 地基 | 注意力预算是信息论问题，窗口再大也有 Context Rot |
| 注释剥离 | 地基 | 多类读者共存是文档工程问题 |
| Worktree 去重 | 地基 | 多实例隔离是基础设施问题 |

**全是地基。** 没有一个会因为模型变强而过时。

## 怎么用：CLAUDE.md 里到底该写什么

分析完加载机制，关键问题来了：知道了上下文怎么组装，那往里面装什么？

这里要致敬一个社区项目：**everything-claude-code**。这个项目来自 Anthropic x Forum Ventures 黑客松（2026 年 2 月，纽约）的冠军团队 Affaan Mustafa 等人，是他们 10 个多月日常使用 Claude Code 积累下来的配置集，开源后 GitHub 50K+ star。它提供了一整套生产级的 Claude Code 配置，包括 38 个 Agent、156 个 Skill、26+ 个 Hook、90+ 条 Rule，覆盖 13 种编程语言，是目前社区里最完整的 Coding Agent Harness 实践参考。下面的构建建议很大程度上参考了 everything-claude-code 的设计思路。

### CLAUDE.md 到底该写什么

这个问题目前没有标准答案，各家做法不一样，社区也在探索中。我们综合了 Anthropic 官方文档、everything-claude-code 的实践、以及 HumanLayer 等多个来源的经验。HumanLayer 是 Y Combinator 投资的 AI Agent 公司，做的是 AI 编程 Agent 在复杂代码库中的落地工具（CodeLayer）。其创始人 Dexter Horthy 在 2025 年 4 月最早提出了"Context Engineering"这个术语，后来被 Anthropic 等公司广泛采用。他们在 CLAUDE.md 的写法上有不少实践经验。综合这些来源，总结出一个三层框架：**WHAT（这是什么）、WHY（为什么这么做）、HOW（怎么做）**。

**WHAT 层：项目地图。** 告诉模型项目是什么、用什么技术栈、目录结构长什么样、关键命令是什么（怎么跑测试、怎么构建、怎么部署）。这是最基础的上下文，模型不知道这些就是个瞎子。everything-claude-code 自己的 CLAUDE.md 就是这个思路：项目概览、架构说明、关键命令、开发注意事项，72 行，非常简洁。

**WHY 层：设计决策。** 解释为什么选了这个技术栈、为什么架构是这样的、各模块的职责划分。模型知道 WHY 之后，遇到没有明确规则覆盖的场景也能做出合理判断，而不是瞎猜。

**HOW 层：工作方法和规则。** 这一层又分两部分。一部分是**思维框架和工作方法论**，比如"收到编程任务先分析 25%、设计 35%、风险评估 15%、实现 25%"，"遇到复杂问题先生成至少 3 种不同方案再选"，"先写测试再写实现"。这不是可有可无的"软指导"，而是实实在在改变模型行为的指令。另一部分是**具体的编码规范和安全红线**，这些更适合放在 `.claude/rules/` 下按主题拆分。

### 不同层级的 CLAUDE.md 写什么

回到前面分析的六级加载机制，每一级的定位不同，写的内容也不同：

| 层级 | 写什么 | 示例 |
|------|--------|------|
| **Managed** `/etc/claude-code/` | 全公司的安全底线和合规要求。由 IT 管理员统一部署，开发者不能改 | "禁止在代码中硬编码任何密钥"、"所有 API 必须有认证" |
| **User** `~/.claude/CLAUDE.md` | 个人的工作方法论和思维框架。跨所有项目生效 | "收到任务先分析再设计再实现"、"遇到复杂问题先列方案再选" |
| **Project** `项目根/CLAUDE.md` | 项目的 WHAT + WHY + 项目级 HOW。进 Git，团队共享 | 项目概览、技术栈、架构说明、关键命令、`@` 引用指向 rules/ |
| **Local** `项目根/CLAUDE.local.md` | 个人在这个项目里的偏好覆盖。不进 Git | "我负责 API 模块，优先关注 api/ 目录"、"我习惯用 vim 模式" |
| **rules/** `.claude/rules/*.md` | 具体的编码规范、安全规则、测试要求。可用 paths 条件加载 | Go 编码风格、安全红线、TDD 要求、API 设计规范 |

几个关键判断：

**团队统一性不只靠 Project 层。** Project 层进 Git，所有人 pull 下来就生效，是项目级规范的核心。但 User 层（`~/.claude/CLAUDE.md`）也需要团队统一的基座。

为什么？因为 User 层写的是跨项目生效的工作方法论：第一性原理分析、先设计后实现、遇到复杂问题先列方案再选、先写测试再写代码。这些不是个人偏好，是团队希望每个成员的 Claude Code 都具备的基础工作习惯。如果完全放任个人去写，有人写了，有人没写，有人写的方向不一样，团队成员的 Claude Code 表现差异就会很大。

**统一基座 + 个人扩展。** 团队维护一份 User 层的基础模板，包含共性的工作方法论，通过安装脚本或配置仓库分发到每个成员的 `~/.claude/` 目录。成员在此基础上追加自己的个性偏好（比如"我习惯用 vim 模式"、"我负责的模块优先用某种设计模式"）。

这个思路不是我们独创的。OpenAI 的 Codex 体系里，个人配置（`~/.agents/skills/`）、团队配置（repo 内的 `.agents/skills/`）、组织配置（`/etc/codex/skills/`）就是三层结构，个人层优先级最高，但团队可以通过配置文件强制标准化关键项。Codex 文档里有一句话很到位："新人 clone repo 然后运行安装命令，就拿到和团队一样的 Agent 配置，no drift, no 'it works on my machine'。" 社区里也出现了 claude-code-templates 这样的项目，通过 npm 包一键安装标准配置到本地。

**这正是"给 Harness 做 Harness"的意义。** Claude Code 原生设计里 User 层是纯个人的，Anthropic 不管你写什么。但团队 Harness 在上面加了一层：提供基础模板、统一分发、允许个性扩展。Harness 的本义就是这样：不是限制，是让力量可用。

**安全底线放 Managed 层。** 由 IT 管理员部署到 `/etc/claude-code/`，开发者没有写权限。这些规则靠文件系统权限保护，不靠模型"更关注"来保证。

**个人覆盖用 Local 层。** `CLAUDE.local.md` 不进 Git（应该加到 .gitignore 里），用来写个人在这个项目里的特殊偏好，不影响团队其他人。

### 分层组织：CLAUDE.md 做目录，rules/ 做正文

CLAUDE.md 本身应该保持简洁（业界建议不超过 200 行），写 WHAT 和 WHY，加上 HOW 层里的工作方法论。具体的编码规范、安全规则、测试要求，通过 `@` 引用拆到 `.claude/rules/` 下。

一个 Go 后端项目的 CLAUDE.md 大概长这样：

```markdown
# CLAUDE.md

## 项目概览
Go 后端服务，运行在 Kubernetes 上，对接 PostgreSQL 和 Redis。
代码托管在 GitLab，CI 用 GitLab CI。Go 1.22+，标准库优先。

## 架构
- cmd/           程序入口
- internal/      业务逻辑（不对外暴露）
- api/           API 定义和 handler
- pkg/           可复用的公共库

## 关键命令
- go test ./...           跑全部测试
- make lint               跑 golangci-lint
- make build              构建二进制

## 工作方法
- 收到任务先理解需求，再设计方案，最后实现。不要拿到需求就开始写代码
- 遇到复杂问题，先列出至少 2 种不同方案，分析权衡，再选择
- 先写测试再写实现（TDD），覆盖率不低于 80%
- 改动代码前先看现有代码，理解上下文再动手

## 规则
@.claude/rules/go-conventions.md
@.claude/rules/security.md
@.claude/rules/testing.md
@.claude/rules/git-workflow.md
@.claude/rules/api-design.md
```

### rules/ 里写什么

参考 everything-claude-code 的 rules/ 结构，具体规则至少覆盖五个维度：

**1. 编码规范。** everything-claude-code 的 rules/common/coding-style.md 提供了一个通用模板：不可变优先、小文件（200-400 行）、全面错误处理、边界输入验证。在此基础上叠加语言特定的规则。比如 rules/golang/ 下有 Go 专属的编码风格、设计模式、安全和测试规则，并且用 Frontmatter 的 paths 字段限定只对 `.go` 文件生效：

```markdown
---
paths: ["**/*.go", "**/go.mod"]
---
# Go 编码规范
- gofmt 和 goimports 是强制的，不讨论风格
- accept interfaces, return structs
- 错误必须带上下文：return fmt.Errorf("failed to create user: %w", err)
```

**2. 安全红线。** everything-claude-code 的 security.md 定义了 8 项强制安全检查：不能硬编码密钥、所有用户输入必须验证、必须用参数化查询防注入、错误信息不能泄露敏感数据。安全规则不加 paths 条件，每次对话都加载。

**3. 测试要求。** everything-claude-code 把 TDD 定义为强制工作流：先写测试（RED），跑失败，再写实现（GREEN），跑通过，最后重构（IMPROVE），覆盖率不低于 80%。

**4. Git 和协作流程。** commit message 用 conventional commits 格式（feat/fix/refactor/docs/test），PR 要分析完整的 commit 历史而不是只看最后一个 commit。

**5. 模块专属规则。** 用条件加载按需生效。处理 API 代码时加载 API 规范，处理数据库代码时加载数据库规范，互不干扰。

### 别忘了给规则加注释

每条重要规则都应该用 HTML 注释写清楚：为什么加这条、什么条件下可以移除。Claude 看不到注释，不浪费注意力。但三个月后维护这份文件的团队成员看得到。没有注释的规则，最终会变成没人敢删的"祖传代码"。

### 别止步于 CLAUDE.md

最后要强调一点。CLAUDE.md 里的规则，模型是"倾向于遵守"而非"保证遵守"。本文前面分析过，上下文注入是推断型控制，不是确定性控制。

举个例子：你在规则里写"禁止使用 --no-verify 跳过 Git Hook"，模型大部分时候会遵守，但它本质上是一个概率性执行体，不能保证 100% 不犯。而 everything-claude-code 的做法是用一个 Hook（block-no-verify）在工具层直接拦截包含 --no-verify 的命令，确定性地阻断，不给模型犯错的机会。

**这就是为什么光有 CLAUDE.md 不够，需要构建完整的外围 Harness。** CLAUDE.md 解决的是上下文层的控制，告诉模型"应该怎么做"。但真正的安全红线和质量关卡，必须用 Hooks 在工具层兜底，用 Permissions 在权限层设防。上下文层、工具层、权限层，三层叠加才是完整的 Harness。

这也是后续几篇要展开的内容。

## 后续构建路线图

本篇拆解了第一个挂载点 CLAUDE.md 的上下文组装机制。Claude Code 还有五个挂载点没碰：Hooks、Permissions、MCP、Skills、Agents。加上团队级的外围构建，整个系列的路线图大致如下：

**第一阶段：理解六个挂载点（逐个拆源码）**

| 篇 | 主题 | 核心问题 |
|-----|------|----------|
| 01（本篇） | CLAUDE.md 上下文组装 | 模型看到什么？怎么控制它看到什么？ |
| 02 | Hooks 生命周期 | 工具层怎么兜底？哪些动作必须用 CPU 守？ |
| 03 | Permissions 权限模型 | 哪些操作需要人类确认？权限边界怎么划？ |
| 04 | MCP Server 集成 | 怎么给 Agent 接外部工具？工具多了性能怎么办？ |
| 05 | Skills 按需加载 | 怎么避免把所有知识塞进上下文？按需加载的机制是什么？ |
| 06 | Sub-Agents 与多 Agent 协作 | 并行 Agent 怎么隔离上下文？Worktree 机制的完整设计 |

**第二阶段：构建团队 Harness**

拆完六个挂载点之后，知道了 Claude Code 提供了什么能力、边界在哪，就可以在上面搭建团队级的外围 Harness。几个核心模块：

**配置分发体系。** 前面分析了六级加载的权力级联。团队 Harness 要做的是：维护 Managed 层的安全底线（IT 部署）+ User 层的基础模板（统一工作方法论）+ Project 层的项目规范（进 Git）+ rules/ 的条件规则（按需加载）。通过安装脚本一键分发，做到"新人 clone repo + 跑一行命令 = 拿到标准配置"。

**验证流水线。** CLAUDE.md 是推断型控制（模型"倾向于遵守"），验证流水线是计算型控制（确定性保证）。Böckeler 在 Martin Fowler 网站的文章里讲得很清楚：Harness 的两条腿是 Guides（前馈，生成前引导）和 Sensors（反馈，生成后检验）。只有前馈 = agent 知道规则但不知道是否生效；只有反馈 = agent 反复犯同样错误。两条腿都要有。具体包括：
- **Hooks 拦截层**：在工具调用前/后触发脚本。block-no-verify（拦截跳过 Git Hook）、lint-on-save（保存时自动 lint）、secret-scan（阻断包含密钥的提交）。everything-claude-code 有 26+ 个 Hook 模板。
- **CI 验证层**：结构测试（依赖方向是否合规）、覆盖率门槛、安全扫描。OpenAI 的做法是用 linter 和 CI 验证文档交叉链接的一致性——连文档都有确定性校验。
- **AI 审查层**：用另一个 LLM 做 Code Review。Anthropic 2026-03 的文章叫这个"GAN 风格架构"——Generator 和 Evaluator 分离，因为 agent 自我评估有偏差（"confidently praise their own outputs"）。

**上下文生命周期管理。** 这是 Harness Engineering 里最容易被忽视的部分。LangChain 的 Context Rot 研究表明，agent 在 20-30 轮对话后性能可预测地退化。Chroma 测了 18 个模型，结论一致：随着输入长度增长，性能越来越不可靠。应对策略：
- **Sub-Agent 作为 Context Firewall**：HumanLayer 的 Kyle 把 sub-agent 叫"上下文防火墙"——把大任务拆给子 agent，每个子 agent 有独立的、干净的上下文窗口，干完把结果交回来，主 agent 的上下文不被污染。
- **结构化交接优于压缩**：Anthropic 发现 context resets（结构化交接重新开始）比 in-place compaction（原地压缩）更有效。不要试图在一个越来越长的上下文里苟延残喘，该重启就重启，用进度文件 + git history 让新 session 快速接手。
- **输出静默原则**：验证通过时不输出任何东西（不浪费 context），只在失败时输出错误信息。HumanLayer 叫这个"success silent, errors only"——成功是静默的，错误才值得说。

**可观测性。** OpenAI 在百万行代码项目里做的一件关键事：让 agent 接入遥测系统（logs、metrics、spans），agent 自己能查 Sentry 错误、读 Datadog 指标、看 OpenTelemetry traces。agent 不是在真空中写代码，它能看到代码跑起来之后的实际表现。Mitchell Hashimoto 的总结最到位：Harness Engineering 就是**"让 agent 永远不再犯同一个错误"**。可观测性是实现这一点的基础设施。

**第三阶段：度量与演进**

Harness 不是一劳永逸的。Anthropic 原话："Every component in a harness encodes an assumption about what the model can't do on its own, and those assumptions go stale." Harness 里的每个组件都在假设模型还做不到某件事——假设会过时。

度量体系至少包括：
- **Agent 成功率**：每个 PR 一次通过 CI 的比例
- **人工干预率**：每个 PR 需要人类修改的次数
- **Context 利用率**：上下文窗口的平均使用量和边际效用
- **规则命中率**：哪些 rules 频繁触发，哪些从未触发（从未触发的规则可能是废物，也可能是地基——要区分）

随着模型升级，定期审查每个 Harness 组件：地基留着（文件系统权限、CI 验证、Git Hook——模型再强也需要确定性约束），脚手架该拆就拆（模型学会了的规则就不需要再写在 CLAUDE.md 里占上下文）。
