微信公众号文章格式规范（Markdown 版）

━━━━━━━━━━━━━━━━━━━━

◆ 文件格式

- 文章用 .md（Markdown），支持加粗、表格等语法
- 封面图用 .svg，转 .png 发布
- 命名：93.md / cover93.svg / cover93.png（数字递增）

━━━━━━━━━━━━━━━━━━━━

◆ 排版符号

- **加粗** 用于强调重点句子和金句
- 【小节标题】用中文方括号（◆ 下面的子内容）
- 💡 名词解释/人话翻译（给小白看的解释用这个标志）
- ━━━ 粗分隔线（章节之间）
- ──── 细分隔线（小节之间）

━━━━━━━━━━━━━━━━━━━━

◆ 标题层级

【关键词】文章主标题（文章开头第一行）
◆ 一级标题（大节）
【二级标题】（小节，在 ◆ 下面的子内容）
- 列表项（用 Markdown 列表 `-`）

注意：▸ 不再使用，统一用 ◆ 作为大节标题

━━━━━━━━━━━━━━━━━━━━

◆ 表格格式

用 Markdown 表格：

| 列1 | 列2 | 列3 |
|-----|-----|-----|
| 内容 | 内容 | 内容 |

表格中的角色名统一用 **中文（eng）** 格式，例如：专家（expert）、苏格拉底（socratic）

◆ Markdown 注意事项

- 段落之间必须空一行，否则不换行
- 金句独立成段，每句之间空一行
- **加粗 `**` 内侧紧贴引号/间隔号会解析失败**（实测 commonmark 库验证，VSCode 预览同款规则）：
  - ❌ 失败：`**"引号加粗"**`（`**` 贴着引号 " "）、`**·吧**`（贴间隔号 ·）、`"动土"**——`（`**` 左贴引号右贴破折号）
  - ✅ 正常：`汉字**加粗**汉字`（紧贴汉字没问题）、`**加粗**：`、`，**加粗**，`（紧贴普通标点逗号冒号没问题）
  - 原因：CommonMark 的 flanking 规则把引号/间隔号算作标点，导致 `**` 定界符判定失效。**和"贴不贴汉字"无关，关键看 `**` 内侧紧贴的是不是引号/间隔号这类特殊标点。**
  - 解法：把引号挪到加粗外面，或加粗内容不含引号，或 `**` 两侧补空格。
- 公式/代码用 backtick 包裹：`E = mc²`
- 数学公式中的下标用 Unicode 下标字符，不用 `_t` 写法：
  - 时间步：sₜ、xₜ、sₜ₋₁、xₜ₋₁
  - 编号：B₁、B₂、x₁、x₂
  - 转置：kᵀ、vᵀ（用 Unicode 上标 ᵀ）
  - 希腊字母直接用 Unicode：α、β、θ、Δ
  - 常用 Unicode 数学符号：· (乘) ² (平方) ½ (二分之一) → ≈ ≠ ≤ ≥
- 代码块中的伪代码（Python 风格变量名如 state、new_token）不用 Unicode 下标，保持程序员可读性
- 微信公众号编辑器对 Markdown 支持有限，发布前用 Markdown 转微信工具预览

◆ 风格要点

- 语言接地气，少装逼
- 技术名词要翻译成人话（用 💡 标记）
- **正文不要黄脸/手势表情 emoji（😏😂🔧 之类）**——公众号正文保持稳重。唯一例外：💡 灯泡是 hint 块（"打个比方"/名词解释）的固定格式标记，保留
- 段落短一点，别堆大段
- 重点用 **加粗** 强调
- 数据和对比用 Markdown 表格呈现
- 第一次出现的专业术语必须解释，别假设读者看过之前的文章

【别堆砌英文词】
- 章节标题 / 小节标题 / hint 块第一次出现的英文术语必须带中文锚点：`saturation events` → `Saturation Events（饱和事件）`，`entropy neuron` → `"熵神经元"（entropy neuron，给整体分布"踩刹车"的细胞）`
- 正文里反复用英文做精确指代可以，但读者第一次见时必须有中文 + 一句话解释
- 文末"技术名词速查"做 backup，所有术语再来一次中英对照定义
- 判据：扫一遍标题，全英文的小节标题对小白不友好，需要补中文

【别引论文图号 / 表号】
- "Figure 5" / "Table 3" 这种引用对读者无意义——读者看不到图，引了反而显得在炫论文细节
- 除非公众号里**自己重新画了图**，否则一律改成自然语言描述："论文里有一张二维散点图（横轴 X、纵轴 Y）"、"统计验证用的是 Kendall's τ"
- 只有一种例外保留图号：核对到具体图位置有学术严谨度价值（例如"跨模型证据放在 Appendix 而不是主图"这种校准）

【💡 hint 块标题分工】
- **"打个比方"** —— 用类比/比喻解释抽象概念
- **"我自己容易搞混的问题"** —— 学习者读到这里冒出来的疑问，问答形式直接解释（不是真正的比喻，别用"打个比方"）
- 不用 **"翻译成人话"** 这种说法

【避免的口头禅/用词】
- **"拆 / 拆开 / 拆过 / 拆解"** —— 作者烦这个词，别用。改用"分析 / 讲 / 聊 / 剖 / 说清楚"等替代（如"我们专门分析过""改这个流程"）

【参考资料区规则】
- 文末"参考资料"只放**外部论文/资料**（带英文标题 + arXiv 号，读者好搜）
- **自家公众号往期链接不进参考资料附录**——在正文里引到的地方直接带链接即可
- 链接前后留空格、带 https:// 前缀，避免和中文括号粘连导致微信识别不出

◆ 审稿与修订（写完到发之前的必经流程）

【铁律 1：所有论文引用必须事实核查，不能信训练数据记忆】

公众号引论文有一票易错点：arxiv ID 编错 / venue 编错（ICLR vs TMLR）/ 数字编错 / 神经元编号编错 / 把多篇论文的发现张冠李戴 / 凭直觉补出"反向孪生神经元"这种论文里压根没有的内容。

**每篇引到的论文，派子代理去 web 上拉原文核对一遍。** 给子代理一份具体的 10 条核查清单（arxiv ID / 作者 / 实验设计 / 关键数字 / 关键引文 / 实验对象 / 图号），不能宽泛地说"看看准不准"。子代理回来后逐条按"OK / 需调整 / 查不到"列。

判据：扫一遍正文，所有数字 / 引文 / 编号 / venue 都过过手——不是"看起来像对的"，是"web 上实证过"。

【铁律 2：自家公众号往期链接默认会编错，必须问 Zero 要】

公众号文章 URL 的 hash 是不可预测的。我每次都会无意识"编"一个看起来像样的链接（hash 字母数字随机看着像）——但和真链接对不上。

引往期文章时：要么用纯文字"199 期"，要么直接问 Zero 要真链接，**不能凭印象写**。

【铁律 3：长英文引文删，论文细节别炫】

公众号读者不读英文。Gurnee 论文里那段超长英文引文是给同行看的，公众号读者会跳过。改成中文概述 + 一两个英文术语点缀。

论文图号 / 表号也别引（"Figure 5"、"Table 3"）——读者看不到图。除非公众号里自己画了新图，或者图号本身有学术校准价值（"跨模型证据在 Appendix 不在主图"这种）。

【铁律 4：跑题的论文整段删，不要心疼篇幅】

判据：每节的论文 / 实验 / 数据点是不是在加强本期的核心论点？

- 如果某篇论文虽然有趣但跟主线没关系（234 期 Anthropic 押韵实验跑题到"规划"，跟"末层职能"不直接相关）——**整节删**
- 如果某个实验"看起来反直觉"，但仔细想其实是 trivially 显然的（234 期 Lioubashevski 因果干预，表面是开关跳变，仔细想就是覆盖下一层 hidden state）——**删，或软化为"论文里还试过 X，但实验对照不够干净，本期不展开"**
- 如果某段技术细节需要先讲 3 个新概念才能讲清楚（234 期 MechLens Mistral 异类讲 GQA + SWA + MHA），而结论跟主线关系不大——**删**

宁可短而集中，不要长而失焦。

【铁律 5：hint 块不能腰斩论证句】

💡 hint 块经常是为了"顺便解释一个名词"插进去的——但插的位置不对会把读者从主线拖走。

判据：hint 块插入后，**正文还能不能直接接续上一段？**

- ✗ 不能：腰斩——把 hint 块挪到段落末尾或下一节开头
- ✓ 能：hint 块和主线并行，没问题

另外两类 💡 标题分工：
- "打个比方" —— 用类比 / 比喻解释抽象概念
- "我自己容易搞混的问题" —— 学习者读到这里冒出来的疑问，问答形式直接解释

【铁律 6：数字解读不能想当然】

234 期我把 Lioubashevski "stricter Kendall's τ > 0 的样本占比 67.4%" 误读成"严格递增的样本占比"——后者是更严格的统计，论文没报告。读者也不会发现这个错，但学术上不诚实。

数字（特别是统计量）不能凭"看起来像什么"翻译——必须读原文公式 / 定义 / 算法。**所有非常规统计量（不是 mean / std / accuracy 这种）都派子代理核查它的精确定义。**

【铁律 7：通读阶段四件事】

第一稿写完后通读，按顺序检查：

1. **逻辑跳了吗** —— 每节的论文是不是在加深主线？跨节衔接是不是自然？
2. **格式割裂了吗** —— hint 块插的位置对吗？长括号注解破坏节奏吗？小节标题数太多了吗（超过 4 个就该合并）？
3. **英文堆砌了吗** —— 章节标题 / 小节标题 / hint 块 / 表格列名第一次出现的英文术语有没有中文锚点？
4. **跟读者站位对吗** —— "234 期"这种内部代号改成"本期"；"我们之前"指代要明确；学术腔（"本表用 Table 6"）改成口语

【铁律 8：温度高时不审稿】

亢奋下笔容易过断言、复读、塞太多。**重要文章冷一晚次日审，不是次日更聪明，是温度差让昨日稿子成为可观察对象。**

◆ 封面图要点

- 直白，一眼能看懂
- 别搞太抽象的
- 主标题要大、醒目
- 配色跟主题相关
- 别画复杂 logo，画不好就用简化图形或文字代替

【SVG 中文字体配置】

cairosvg 转 PNG 时需要指定本地中文字体，否则中文显示为方块。

在 SVG 开头加：

```xml
<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 800 600">
  <style>
    text { font-family: 'Noto Sans CJK SC', sans-serif; }
  </style>
  ...
</svg>
```

系统可用的中文字体（fc-list :lang=zh family）：
- Noto Sans CJK SC（简体中文，推荐）
- Noto Serif CJK SC（简体中文，衬线）
- Noto Sans CJK TC（繁体中文）
- AR PL UKai TW MBE（台湾楷书）

转换命令：`cairosvg cover77.svg -o cover77.png -d 150`
