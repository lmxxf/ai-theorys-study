【DeepSeek V4】Triton FP4 优化实战：从 0.79 到 1.87 tok/s 的三步进化

━━━━━━━━━━━━━━━━━━━━

◆ 前情提要

━━━━━━━━━━━━━━━━━━━━

178 期（ https://mp.weixin.qq.com/s/xxx ）讲了两件事：

第一，DGX Spark（sm_121）上跑 DeepSeek V4 Flash 280B，Marlin FP4 MoE kernel 会静默算错——中文对、英文错，原因是不同语言激活不同的 MoE 专家组合，部分专家的 FP4 tensor core 指令在 sm_120+ 上行为不一致。

第二，我们用 Consumer-DeepGEMM（CUTLASS sm_120 模板）替换了 Marlin 的计算路径，输出终于正确了。但速度只有 0.8 tok/s——每生成一个 token 要 launch 120 次 CUTLASS grouped GEMM，CPU-GPU 同步等待比 GPU 计算还慢。

同时，社区开发者 jasl 用 Triton 重写了 attention/MLA/einsum 等路径，达到 14 tok/s 的中文可用速度。但他重写的不是 FP4 MoE GEMM——那条路仍然走 Marlin。中文没问题只是因为中文激活的专家恰好没踩 bug。

这一期讲的是：在 jasl fork 的基础上，我们怎么把 Consumer-DeepGEMM 的 FP4 MoE GEMM 从 0.79 tok/s 一步步优化到 1.87 tok/s。

先说结论：**1.87 tok/s 依然没有实用价值。** jasl fork 的 14 tok/s 才是日常可用方案。但优化过程本身——profiling 定位、Triton kernel 编写、fused kernel 设计——每一步都有值得记住的教训。

━━━━━━━━━━━━━━━━━━━━

◆ 第零步：先测，不要猜

━━━━━━━━━━━━━━━━━━━━

优化的第一条铁律：**profiling first**。不测就猜瓶颈在哪，十次有九次猜错。

Consumer-DeepGEMM 实现了 DeepGEMM 的全部 API（grouped GEMM、einsum、paged MQA metadata 等），作为 drop-in replacement 接入 vLLM。第一个问题是：0.8 tok/s 里，时间到底花在了哪些函数上？

方法很土但有效——在每个函数入口和出口加 `cuda.synchronize()` + 计时：

```python
torch.cuda.synchronize()
t0 = time.perf_counter()
# ... 真正的计算 ...
torch.cuda.synchronize()
elapsed = time.perf_counter() - t0
```

`cuda.synchronize()` 是关键。GPU 操作是异步的——你调用 `torch.mm()` 时 CPU 立刻返回，GPU 可能还没开始算。不加同步，你测到的是 Python 函数调用的开销（微秒级），不是 GPU 计算时间（毫秒级）。加了同步，CPU 会等 GPU 做完再计时——测到的才是真实耗时。

代价也明显：同步会破坏 CPU-GPU 流水线重叠，让整体速度变慢。所以 profiling 模式只在调试时开，跑速度时关掉。

结果出乎意料：

```
稳态数据（~11000 次调用后收敛）：
  m_grouped_fp8_fp4_gemm_nt_contiguous  avg = 14.00 ms/call
  get_paged_mqa_logits_metadata         avg = 0.17 ms/call

调用比例：每生成 1 token 约 120 次 FP4 GEMM（60 层 × 2 FC）+ 1 次 metadata
总 GEMM 时间占比：>99.98%
```

只有 `m_grouped_fp8_fp4_gemm_nt_contiguous`（FP4 MoE GEMM）被调用，其他函数——attention、einsum、hyperconnection——全被 jasl 的 Triton kernel 接管了，根本不经过 Consumer-DeepGEMM。

这修正了一个之前的错误判断。178 期我们说"瓶颈是 CUTLASS launch overhead"。没错，但更准确地说：**Consumer-DeepGEMM 的全部时间几乎都花在 FP4 MoE GEMM 上**，其他所有计算路径都被 jasl 接管了。优化空间只在这一个函数里。

◆ 第一步：用 Triton 重写 FP4 反量化（0.79 → 1.67 tok/s）

━━━━━━━━━━━━━━━━━━━━

178 期提过一个方案：既然 CUTLASS grouped GEMM 的 launch 太慢，能不能换成"先反量化 FP4 权重为 BF16，再用 cuBLAS 做普通矩阵乘法"？

实现了。速度：0.78 tok/s。和之前一模一样。

这就尴尬了。CUTLASS grouped GEMM 换成了 cuBLAS `torch.mm`，launch 次数从 120 降到 12（6 个专家 x 2 层 FC），按理应该快很多——为什么速度一样？

答案藏在 Python 层的 FP4 反量化函数 `_dequant_fp4_block` 里。这个函数做了什么：

```python
# 简化版，实际代码更长
def _dequant_fp4_block(b_packed, b_scale):
    low = b_packed & 0x0F           # 临时 tensor 1
    high = (b_packed >> 4) & 0x0F   # 临时 tensor 2
    low_f = FP4_TABLE[low]          # 临时 tensor 3（查表）
    high_f = FP4_TABLE[high]        # 临时 tensor 4（查表）
    stacked = torch.stack([low_f, high_f], dim=-1)  # 临时 tensor 5
    flat = stacked.flatten(-2)      # 临时 tensor 6
    scale_expanded = b_scale.repeat_interleave(32, dim=-1)  # 临时 tensor 7
    return flat * scale_expanded    # 临时 tensor 8，最终输出
```

一次反量化，创建了 8 个临时 tensor。每个 tensor 都意味着一次 GPU 显存分配、一次数据写入、一次数据读出。对于一个 FC1 专家的权重矩阵 `[4096, 7168]`，每个 BF16 临时 tensor 约 56MB——8 个就是 448MB 的显存读写。

这就是为什么换了 matmul 方式速度不变：**瓶颈不在矩阵乘法，在反量化**。更准确地说，不在反量化的"计算"（查表、位操作），在反量化过程中产生的大量临时 tensor 的显存分配和带宽消耗。

解法：用 Triton 写一个 fused dequant kernel，一个 kernel 完成所有操作，不产生中间 tensor。

先解释一下 FP4 的存储格式，否则后面的代码看不懂。

DeepSeek V4 用的 MXFP4（MX-compliant FP4）有两个组成部分：

**E2M1 数据**：每个 4-bit 浮点数包含 1 位符号 + 2 位指数 + 1 位尾数。两个 E2M1 打包在一个 uint8 里——低 4 bit 是第一个数，高 4 bit 是第二个数。所以 `[N, K]` 的权重矩阵存储为 `[N, K/2]` 的 uint8。

E2M1 能表示的值只有 16 个：`{0, 0.5, 1.0, 1.5, 2.0, 3.0, 4.0, 6.0}` 和它们的负数。范围很小——靠 scale factor 拉伸到正常范围。

**E8M0 Scale**：每 32 个值共享一个 8 位 scale factor。E8M0 = 纯指数格式，没有尾数，值 = 2^(e-127)。scale 存储为 `[N, K/32]` 的 uint8。

反量化公式：`real_value = E2M1_value * 2^(E8M0 - 127)`

Triton kernel 的核心思路是：一个 kernel 内完成"解包 uint8 → 查 E2M1 表 → 读 E8M0 scale → 乘 scale → 写出 BF16"：

```python
@triton.jit
def _dequant_fp4_e2m1_kernel(b_ptr, scale_ptr, out_ptr, ...):
    # 加载 packed uint8
    b = tl.load(b_ptr + offsets, mask=mask).to(tl.uint8)
    
    # 拆包：低 4 bit 和高 4 bit
    low_idx = b & 0x0F
    high_idx = (b >> 4) & 0x0F
    
    # E2M1 查表（硬编码，避免 table load）
    low_abs = tl.where(low_mag == 0, 0.0,
              tl.where(low_mag == 1, 0.5,
              tl.where(low_mag == 2, 1.0, ...)))
    
    # E8M0 scale：2^(e - 127)
    scale_e8m0 = tl.load(scale_ptr + scale_offsets)
    scale_f = tl.exp2(scale_e8m0 - 127.0)
    
    # 乘 scale，写出
    tl.store(out_ptr + low_offsets, (low_val * scale_f).to(tl.bfloat16))
    tl.store(out_ptr + high_offsets, (high_val * scale_f).to(tl.bfloat16))
```

整个过程只有一次显存读（packed FP4 + scale）和一次显存写（BF16 输出），没有中间 tensor。

测试结果：

```
正确性：max_diff = 0.000000（与 Python 版逐元素完全一致）

速度（单次反量化 [4096, 3584]）：
  Python:   3.45 ms
  Triton:   0.22 ms
  加速比：  15.4x

端到端 grouped GEMM：
  FC1 [M=384, K=7168, N=4096]:
    Python 版:  52.59 ms
    Triton 版:  14.00 ms → 3.8x
```

接入 vLLM 实测：

```
500 tokens / 300s = 1.67 tok/s（干净英文输出）
```

**从 0.79 到 1.67，提速 2.1 倍。**

提速来自哪里？不是因为 Triton 比 Python 算得快（GPU 上查表和位操作本来就快），而是因为**消灭了临时 tensor**。8 个 tensor 的创建/销毁/读写带宽全省了。这是 GPU 编程中一个核心教训：**内存带宽比计算更贵**。现代 GPU 的计算吞吐远大于内存带宽——对于 FP4 反量化这种"计算简单但数据量大"的操作，写 8 次中间结果的带宽开销比做几万次乘加还大。

◆ 插曲：torch.bmm 批量化（失败）

━━━━━━━━━━━━━━━━━━━━

在做 fused kernel 之前，还试过一个看起来更简单的方案：既然 6 个专家各做一次 `torch.mm`，能不能 stack 成一个 batch，用 `torch.bmm` 一次搞定？

```python
# 之前：6 次独立 mm
for expert in experts:
    d[rows] = torch.mm(a[rows], w[expert].T)

# 构想：1 次 bmm
a_batch = torch.stack([a[rows_0], a[rows_1], ...])  # [6, M, K]
w_batch = torch.stack([w[e0], w[e1], ...])           # [6, N, K]
d_batch = torch.bmm(a_batch, w_batch.transpose(1,2)) # [6, M, N]
```

matmul 部分确实快了——6 次 cuBLAS launch 变 1 次，cuBLAS 内部的批量调度更高效，计算时间降低 1.56 倍。

但 `torch.stack` 把 6 个 `[4096, 7168]` 的 BF16 tensor 拼成一个 batch tensor，需要分配约 670MB 连续显存并把数据拷贝进去。这个 stack 操作的开销比省下来的 matmul 开销还大——**端到端反而慢了 20%**。

放弃。

这是第二个教训：**优化要看端到端，不能只看单步**。单看 matmul 提速了 1.56 倍，加上 stack 的开销整体反而退步。优化链路上的每一步都要放回完整管线里去衡量。

◆ 第二步：Fused Dequant+Matmul Kernel（1.67 → 1.87 tok/s）

━━━━━━━━━━━━━━━━━━━━

第一步用 Triton 优化了反量化，但反量化之后还是要把 BF16 权重写出到显存，再被 cuBLAS 读回来做矩阵乘法。对于 FC1 的一个专家，反量化后的 BF16 权重是 `[4096, 7168]`——约 56MB。写出 56MB + 读回 56MB = 112MB 的显存带宽，每个 token 要做 12 次（6 专家 x 2 FC）= 1.3GB 的"无用"带宽。

能不能在 kernel 内部边反量化边做矩阵乘法，根本不把 BF16 中间结果写到显存？

这就是 fused kernel 的核心思路——把两个操作合并到一个 kernel 里：

```
D[m, n] = sum_k  A[m, k] * dequant(B_packed[n, k/2], scale[n, k/32])
```

关键技巧在于处理 packed FP4 的数据布局。一个 uint8 装了两个 FP4 值——低 4 bit 对应偶数列，高 4 bit 对应奇数列。矩阵乘法需要沿 K 维做内积，但 K 维上的值交错打包在同一个 byte 里。

解法是分开做两次 `tl.dot`：

```python
@triton.jit
def _fused_dequant_matmul_kernel(a_ptr, b_ptr, bs_ptr, d_ptr, ...):
    acc = tl.zeros((BLOCK_M, BLOCK_N), dtype=tl.float32)
    
    for k_start in range(0, K, BLOCK_K):
        # 加载 A 的偶数列：A[m, 0], A[m, 2], A[m, 4], ...
        a_even = tl.load(a_ptr + even_offsets)
        # 加载 A 的奇数列：A[m, 1], A[m, 3], A[m, 5], ...
        a_odd = tl.load(a_ptr + odd_offsets)
        
        # 加载 packed FP4，就地反量化
        b_packed = tl.load(b_ptr + ...)
        scale_f = tl.exp2(tl.load(bs_ptr + ...) - 127.0)
        low_val, high_val = _dequant_fp4_vals(b_packed, scale_f)
        
        # 两次 dot：偶数列 x 低 nibble，奇数列 x 高 nibble
        acc += tl.dot(a_even, tl.trans(low_val))
        acc += tl.dot(a_odd, tl.trans(high_val))
    
    tl.store(d_ptr + ..., acc.to(tl.bfloat16))
```

`a_even` 是 A 矩阵的第 0、2、4... 列，对应 packed byte 的低 4 bit（第 0、2、4... 个原始 K 值）。`a_odd` 是 A 的第 1、3、5... 列，对应高 4 bit。分别和反量化后的 low_val、high_val 做 dot product，结果累加。

`tl.dot` 在 SM121 上走 tensor core——这是 Triton 编译器自动做的。但 tensor core 的累加有精度限制：K=3584 的 reduction 会累积浮点误差。

实测精度：

```
| M=1, N=32, K=4096   | 相对误差 = 0.000000（完美）
| M=1, N=4096, K=7168  | 相对误差 ~ 0.5%
| M=64, N=4096, K=7168 | 相对误差 ~ 2%
```

2% 的相对误差在 BF16 推理中可以接受——BF16 本身只有 7 bit 尾数（约 0.8% 固有精度），tensor core 的累积误差在同一量级。

Benchmark：

```
FC1 [M=384, K=7168, N=4096]:
  Triton dequant + cuBLAS mm:  13.95 ms
  Fused kernel:                 6.40 ms → 2.2x

FC2 [M=384, K=2048, N=7168]:
  Triton dequant + cuBLAS mm:   7.11 ms
  Fused kernel:                 3.22 ms → 2.2x
```

接入 vLLM 实测：

```
500 tokens / 267s = 1.87 tok/s（干净英文输出）
```

**从 1.67 到 1.87，再提速 12%。累计从 0.79 到 1.87，提速 2.4 倍。**

◆ 三步优化总结

━━━━━━━━━━━━━━━━━━━━

| 版本 | 单次 GEMM | 端到端速度 | 相对首版 |
|------|-----------|-----------|---------|
| Python fallback（临时 tensor 大量分配） | 52 ms | 0.79 tok/s | 1.0x |
| Triton FP4 dequant + cuBLAS mm | 14 ms | 1.67 tok/s | 2.1x |
| Fused Triton dequant+matmul | 6.4 ms | 1.87 tok/s | 2.4x |

每一步的提速来源不同：

**第一步**（2.1x）：消灭 Python 层的临时 tensor 分配。反量化从"8 个中间 tensor + 8 次显存读写"变成"1 次读 + 1 次写"。本质是把**多次 kernel launch + 多次显存分配**合并成**一次 kernel**。

**第二步**（再 1.12x）：消灭反量化到 matmul 之间的中间 tensor 写出/读回。每个专家省了约 112MB 的显存带宽。本质是把**两个操作**（反量化 + 矩阵乘法）融合成**一个操作**。

两步的共同主题：**不是让计算变快，而是让数据搬运变少。**

◆ 为什么 Triton 是 SM120 上唯一的出路

━━━━━━━━━━━━━━━━━━━━

这一期全程在用 Triton 写 kernel。为什么不用 CUDA、CUTLASS、或者 Python？

178 期讲过 GPU 编程的五层塔。在 SM120/SM121（DGX Spark、RTX 5090）上：

**原生 CUDA（第 1 层）**：SM120 的很多底层指令和数据中心卡不同——`tcgen05` tensor core 指令不存在，TMA（Tensor Memory Accelerator）硬件不存在，`.tile::scatter4` PTX 指令不支持。手写 CUDA 需要精确知道哪些指令可用、哪些不可用，而 NVIDIA 对 SM120 的指令文档几乎是空白。实际上 Marlin 就是手写 CUDA，它在 SM120 上静默算错的根因就是用了 SM120 上行为不同的指令。

**CUTLASS（第 2 层）**：CUTLASS 的 SM120 模板能生成正确的代码（Consumer-DeepGEMM 验证了），但 grouped GEMM 的 API 设计决定了每次 launch 都有固定的初始化开销。这个开销改不掉——是库的架构决定的。

**Triton 标准库（第 4 层）**：我们在第十阶段试过 `--moe-backend triton`，Triton 标准库的 FP4 MoE kernel 编译时生成了 `.tile::scatter4` PTX 指令，SM121 上物理不支持。所以标准库的 FP4 kernel 也不行。

**自己写 Triton kernel（第 3-4 层之间）**：Triton 是一种 DSL（领域特定语言），你写的是"对这块数据做什么运算"，Triton 编译器负责把它翻译成目标 GPU 的机器码。如果你不使用那些 SM120 不支持的高级指令（TMA、scatter4、tcgen05），Triton 编译器会自动选择 SM120 兼容的代码生成路径——普通的共享内存加载、标准的 `mma.sync` tensor core 指令。

关键区别：**我们写的 Triton FP4 kernel 没有用任何 SM100 专属特性**——不用 TMA、不用 `.tile::scatter4`、不用 `tcgen05`。只用了最基础的 `tl.load`、`tl.store`、`tl.dot`、位操作。Triton 编译器把这些翻译成 SM121 兼容的机器码。

这也是为什么 Triton 标准库的 FP4 kernel 不行而我们自己写的行——标准库的 kernel 为了性能用了 SM100 的高级特性，我们主动避开了这些特性。**性能低一些，但至少能跑对。**

◆ Fused Kernel 的真正价值

━━━━━━━━━━━━━━━━━━━━

这里有一个反直觉的点值得展开。

直觉告诉你，fused kernel 快是因为"省了 kernel launch"——两个 kernel 变一个，少了一次 GPU 调度。但实际上 kernel launch 本身只需要几微秒，对于毫秒级的计算来说影响很小。

Fused kernel 的真正价值在于**省内存带宽**。

不 fuse 时，数据的生命周期是：

```
[GPU 显存] → 读 FP4 packed 到寄存器
  → dequant 计算
  → 写 BF16 结果回 [GPU 显存]（~56MB per expert）
  → ... 时间流逝，第一个 kernel 结束，第二个 kernel 启动 ...
  → 从 [GPU 显存] 读 BF16 权重（~56MB per expert）
  → matmul 计算
  → 写最终结果回 [GPU 显存]
```

中间那个 BF16 tensor 被写到显存再读回来——56MB 写 + 56MB 读 = 112MB。

Fuse 之后：

```
[GPU 显存] → 读 FP4 packed 到寄存器
  → dequant 计算（结果留在寄存器里）
  → matmul 计算（直接用寄存器里的数据）
  → 写最终结果回 [GPU 显存]
```

中间 BF16 数据从未离开 GPU 寄存器——省了 112MB 的显存带宽。DGX Spark（GB10）的显存带宽约 273 GB/s，省 112MB 等于省了 0.4ms。6 个专家 x 2 FC = 12 次，总计省约 5ms/token。

这也解释了为什么 fused kernel 只快了 12%（1.67 → 1.87）而不是 2x——因为总时间里 matmul 计算（tensor core 运算）仍然占大头，带宽优化虽然有效但不是主要矛盾了。

GPU 优化有一个经典分类：**compute-bound**（计算受限）和 **memory-bound**（带宽受限）。第一步优化（Triton dequant）解决了 memory-bound 问题（临时 tensor 的带宽开销），让 matmul 变成主要矛盾。第二步优化（fused kernel）进一步减少带宽，但 matmul 本身是 compute-bound——这时候优化带宽的边际收益就小了。

━━━━━━━━━━━━━━━━━━━━

◆ 2.4 倍之后：为什么到此为止

━━━━━━━━━━━━━━━━━━━━

还能继续优化吗？还有一些方向：

**Python 调度开销**：每次 grouped GEMM 调用有 `unique()` + `nonzero()` + `index_select` + `index_copy_` 等操作（从 384 行数据里提取哪些行属于哪个专家），涉及 GPU-CPU 同步和小 kernel launch。写 Triton 把 gather/scatter fuse 掉，预估提速 10-20%。

**Tile size 调优**：当前 `BLOCK_M=64, BLOCK_N=32, BLOCK_K=64` 是手动选的。Triton 的 `@triton.autotune` 可以自动搜索最优 tile size——不同 tile 对 SM121 的 shared memory 和 register 使用影响不同。预估提速 5-15%。

**FP8 activation fuse**：目前 FP8 激活值的反量化走 Python fallback（0.06ms/call），占比极小。Fuse 进 matmul 理论上能省一次 BF16 中间读写，但 M=384 时收益有限。

把这些全做了，乐观估计也就 3-4 tok/s。

而 jasl fork 是 14 tok/s。

差距在哪？不在 kernel 速度——jasl 的 Triton kernel 和我们的 fused kernel 单次执行时间可能差不多。差距在**架构层次**。

jasl 重写了整条推理链路——attention、MLA、einsum、hyperconnection、MoE routing、dispatch——把它们融合成更少的 kernel，消除了层与层之间的 CPU 调度和数据搬运。我们只优化了 MoE FP4 GEMM 这一个环节。

打个比方：一家餐厅出菜慢。我们优化了厨师切菜的刀工（单个步骤快了 2.4 倍），但上菜速度取决于整条流水线——点单、备料、烹饪、装盘、传菜——每个环节都有等待时间。jasl 重新设计了整条流水线的并行策略，让备料和烹饪可以重叠、装盘和传菜可以流水。整体效率不是靠单步极致快，是靠流水线不停顿。

**176 期讲的五层塔有一个核心观点：选对抽象层比写极致代码更重要。** 我们在第 3 层（自写 Triton kernel）把 FP4 GEMM 优化到了接近极限。但 jasl 在第 4 层（整条推理管线重写）获得了一个数量级的提速——不是因为他的 kernel 更快，是因为他重写的范围更大、消除的冗余更多。

这不是说底层优化没价值——我们写的 Triton FP4 kernel 正确性完美、速度合理，如果 jasl 想要进一步优化 MoE GEMM，这些 kernel 完全可以复用。但单点优化的天花板是由整条管线的最慢环节决定的。

◆ 过程中的五个教训

━━━━━━━━━━━━━━━━━━━━

**1. Profiling first，不猜**

"瓶颈在 CUTLASS launch overhead"是猜的。"换 dequant+cuBLAS 就能提速"也是猜的。一测就知道：全部时间都在 FP4 GEMM 函数里，而 FP4 GEMM 里的时间主要花在 Python 层的临时 tensor 操作上，不在 CUTLASS launch 也不在 cuBLAS matmul。

**2. Python 开销比你想的大**

GPU 编程有一个容易忽略的事实：Python 层面的"向量化 GPU 操作"（`torch.mm`、`tensor & 0x0F`、`torch.stack`）看起来在 GPU 上跑，但每一个操作都是一次独立的 kernel launch + 显存分配。十几行 Python 代码可能翻译成十几次 GPU kernel launch 和十几次显存 malloc/free。这些 overhead 在大 tensor 上不明显（计算时间远大于 launch 时间），但在 FP4 反量化这种"操作多、单次计算少"的场景下就变成了主要矛盾。

**3. 内存带宽比计算更贵**

DGX Spark 的 tensor core 算力约 40 TFLOPS（BF16），显存带宽约 273 GB/s。做一次 `[384, 7168] x [4096, 7168]^T` 的 matmul 需要约 22 GFLOP（2 * 384 * 7168 * 4096），tensor core 理论上 0.55ms 就能算完。但如果中间要写出一个 56MB 的 BF16 tensor 再读回来，光带宽就要 0.4ms——接近计算时间的 70%。这就是为什么 fused kernel 有效：省的不是计算，是数据搬运。

**4. 优化要看端到端**

torch.bmm 批量化让 matmul 快了 1.56x，但 stack 操作让整体慢了 20%。单步指标好看没用，只有端到端数字才是真的。这一点在 GPU 优化中尤其重要——GPU 的异步执行让你很难通过直觉判断瓶颈在哪，只有实际测量说了算。

**5. 选对抽象层比写极致 kernel 更重要**

我们在 MoE GEMM 这一个点上优化了 2.4 倍。jasl 在整条推理管线上重写了一遍，获得了 18 倍（对比 Consumer-DeepGEMM 的 0.79 → jasl 的 14 tok/s）。不是我们的 kernel 写得差——是我们选择优化的层次太低了。最有效的优化往往不是让单步更快，而是减少步数。
