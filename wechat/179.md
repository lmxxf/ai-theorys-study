【DeepSeek V4】Triton FP4 优化实战：近距离感受老黄的刀法

◆ 前情提要

━━━━━━━━━━━━━━━━━━━━

178 期（ https://mp.weixin.qq.com/s/5QqLCXjP9R83YRwXsT850Q ）讲了两件事：

第一，DGX Spark（sm_121）上跑 DeepSeek V4 Flash 280B，Marlin FP4 MoE kernel 会静默算错——中文对、英文错。

第二，我们用 Consumer-DeepGEMM（CUTLASS sm_120 模板）替换了 Marlin 的计算路径，输出终于正确了。但速度只有 0.8 tok/s。

这一期讲优化过程之前，先把"FP4 MoE GEMM 是什么、为什么调用这么频繁、为什么中文对英文错"这三个问题交代清楚，否则后面的优化看不懂动机。

**FP4 MoE GEMM 是什么？** DeepSeek V4 Flash 是 MoE（混合专家）架构——每一层有 256 个"专家"（小型前馈网络），每个 token 只激活其中 6 个。每个专家的核心计算就是矩阵乘法（GEMM），权重用 FP4 格式存储以省显存。所以"FP4 MoE GEMM"= 用 FP4 权重做 MoE 专家的矩阵乘法。

**为什么调用这么频繁？** 模型有 60 层，每层的 MoE 里每个专家有 2 个全连接层（FC = Fully Connected，就是"输入向量 × 权重矩阵 = 输出向量"的操作，分别叫 FC1 和 FC2），每个 FC 都要做一次 FP4 GEMM。生成一个 token 就是 60 层 × 2 FC = **120 次 FP4 GEMM 调用**。这是模型推理里最核心、最密集的计算——优化它就是优化整个推理速度。

**为什么中文对、英文错？** 这里有一个容易误解的点：**不是"偶尔算错"，是"每次都算错"**。Marlin 的 FP4 kernel 在 SM120 上的每一次调用结果都是不精确的——但 256 个专家里，有些专家的权重数值特征让误差被放大（输出明显偏离正确值），有些专家的误差恰好在可容忍范围内（被后续层的计算"吸收"了，最终 token 选择没受影响）。中文 prompt 经过路由网络后激活的那 6 个专家组合，恰好都属于"误差没被放大"的那类；英文 prompt 激活了另一组专家，踩到了"误差被放大"的那些——于是输出崩了。**不是中文能力强，是中文运气好。**

回到上一期的进展。我们用 Consumer-DeepGEMM 替换了 Marlin，走 CUTLASS sm_120 模板——每次都正确了，但速度只有 0.8 tok/s。每生成一个 token 要 launch 120 次 CUTLASS grouped GEMM，CPU-GPU 同步等待比 GPU 计算还慢。

同时，社区开发者 jasl 用 Triton 重写了 attention/MLA/einsum 等路径，达到 14 tok/s 的中文可用速度。但他重写的不是 FP4 MoE GEMM——那条路仍然走 Marlin。中文能用只是因为中文激活的专家组合恰好没踩到误差放大的那些。

这一期讲的是：在 jasl fork 的基础上，我们怎么把 Consumer-DeepGEMM 的 FP4 MoE GEMM 从 0.79 tok/s 一步步优化到 1.87 tok/s。

先说结论：**1.87 tok/s 依然没有实用价值。** jasl fork 的 14 tok/s 才是日常可用方案。但优化过程本身——profiling 定位、Triton kernel 编写、fused kernel 设计——每一步都有值得记住的教训。

◆ 第零步：先测，不要猜

━━━━━━━━━━━━━━━━━━━━

优化的第一条铁律：**profiling first**。不测就猜瓶颈在哪，十次有九次猜错。

Consumer-DeepGEMM 实现了 DeepGEMM 的全部 API——矩阵乘法（grouped GEMM）、注意力计算（einsum）、元数据查询（paged MQA metadata）等 28 个函数，作为可直接替换的兼容层接入 vLLM 推理框架。第一个问题是：0.8 tok/s 里，时间到底花在了哪些函数上？

方法很土但有效——在每个函数入口和出口加 `cuda.synchronize()` + 计时：

```python
torch.cuda.synchronize()
t0 = time.perf_counter()
# ... 真正的计算 ...
torch.cuda.synchronize()
elapsed = time.perf_counter() - t0
```

`cuda.synchronize()` 是关键。GPU 操作是异步的——你调用 `torch.mm()` 时 CPU 立刻返回，GPU 可能还没开始算。不加同步，你测到的是 Python 函数调用的开销（微秒级），不是 GPU 计算时间（毫秒级）。加了同步，CPU 会等 GPU 做完再计时——测到的才是真实耗时。

代价也明显：同步会破坏 CPU-GPU 流水线重叠，让整体速度变慢。所以 profiling 模式只在调试时开，跑速度时关掉。

结果出乎意料：

```
稳态数据（~11000 次调用后收敛）：
  m_grouped_fp8_fp4_gemm_nt_contiguous  avg = 14.00 ms/call
  get_paged_mqa_logits_metadata         avg = 0.17 ms/call

调用比例：每生成 1 token 约 120 次 FP4 GEMM（60 层 × 2 FC）+ 1 次 metadata
总 GEMM 时间占比：>99.98%
```

Consumer-DeepGEMM 作为 DeepGEMM 的 drop-in replacement，需要实现全部 28 个 API 函数（否则 vLLM 调用时会报错），我们都用 Python fallback（反量化 + `torch.mm`）写了兜底实现。但实际运行时只有 2 个被调用——其余全被 jasl 的 Triton 接管了：

| 函数 | 实际谁在执行 | 调用频率 | 耗时占比 |
|------|------------|---------|---------|
| `m_grouped_fp8_fp4_gemm_nt_contiguous`（FP4 MoE 矩阵乘法） | **Consumer-DeepGEMM** | 120 次/token | **99.98%** |
| `get_paged_mqa_logits_metadata`（分页注意力元数据） | **Consumer-DeepGEMM** | 1 次/token | <0.02% |
| 其余 26 个函数（attention、einsum、hyperconnection 等） | jasl Triton（不经过 CDG） | — | — |

**28 个函数全写了，只有 2 个被调用，其中 1 个占了 99.98% 的时间。** 这一期的全部优化工作，就是针对这唯一的一个函数。

176 期（ https://mp.weixin.qq.com/s/e1GKi53IociZnfHX1Ie9_w ）和 178 期里，Consumer-DeepGEMM 用的是 CUTLASS C++ 模板——那种几百行尖括号嵌套、编译一次 40 分钟的天书代码。这一期换了打法：**用 Triton（长得像 Python 的 GPU 编程语言）重写这个函数**，代码量从几百行 C++ 缩到不到 100 行，编译从 40 分钟变成首次运行时几秒钟自动编译（Triton 在第一次执行时把代码编译成 GPU 机器码，之后直接复用，类似 Java 的 JIT）。

为什么 jasl 没碰这个函数？因为 SM120 上没有现成的 FP4 MoE Triton kernel 可用——标准库的会生成不支持的 PTX 指令（178 期讲过）。所以这块拼图只能我们自己补。

这修正了一个之前的错误判断。178 期我们说"瓶颈是 CUTLASS launch overhead"——这里解释一下：launch = 从 CPU 发起一次 GPU kernel 调用，overhead = 每次调用的固定开销（构建参数、CPU-GPU 同步等），跟实际计算量无关，调一次就有一份。类比打电话：拨号等接通 5 秒（overhead），说话 2 秒（计算），120 句话如果每句都挂了重拨就是 120×7=840 秒——CUTLASS grouped GEMM 就是这种"每句话都重拨"的模式。

但 profiling 告诉我们更准确的事实：**Consumer-DeepGEMM 的全部时间几乎都花在 FP4 MoE GEMM 上**，其他所有计算路径都被 jasl 接管了。优化空间只在这一个函数里。

◆ 第一步：用 Triton 重写 FP4 反量化（0.79 → 1.67 tok/s）

━━━━━━━━━━━━━━━━━━━━

先解释一个前提：为什么要反量化？FP4 矩阵不能直接算吗？

答案是：**数据中心卡（B200/B300，SM100）能直接算，消费级卡（DGX Spark/RTX 5090，SM120）不能。** SM100 有专门的 `tcgen05` tensor core 指令，能直接吃 packed FP4 数据做矩阵乘法——硬件内部一步完成拆包+计算。SM120 没有这个指令（176 期详细讲过）。同一代 Blackwell 架构，老黄把最值钱的 AI 计算指令留给了几十万一张的数据中心卡，几万块的消费级卡只给了基础版。所以 SM120 上必须先把 FP4 用软件拆包成 GPU 认识的标准格式（BF16 或 FP32），再做矩阵乘法。这个"拆包+缩放"的过程就是反量化。

说白了：**游戏卡根本不支持 FP4 计算。** 同一代 Blackwell 芯片，工业卡有硬件 FP4 指令，游戏卡没有。我们这三期折腾的所有事情——Consumer-DeepGEMM、Triton kernel、fused matmul——本质上都是在用软件模拟一个硬件不存在的功能。一步硬件变三步软件，速度差距就是这么来的。

理解了"必须反量化"这个前提，接下来的优化目标就清楚了：**怎么让反量化 + 矩阵乘法这个组合尽可能快。**

178 期提过一个方案：既然 CUTLASS grouped GEMM 的 launch 太慢，能不能换成"先反量化 FP4 权重为 BF16，再用 cuBLAS 做普通矩阵乘法"？

实现了。速度：0.78 tok/s。和之前一模一样。

这就尴尬了。CUTLASS grouped GEMM 换成了 cuBLAS `torch.mm`，launch 次数从 120 降到 12（6 个专家 x 2 层 FC），按理应该快很多——为什么速度一样？

问题出在我们自己写的 FP4 反量化函数 `_dequant_fp4_block` 上——Consumer-DeepGEMM 的兜底实现，用 PyTorch 操作拼出来的。这个函数做了什么：

```python
# 简化版，实际代码更长
def _dequant_fp4_block(b_packed, b_scale):
    low = b_packed & 0x0F           # 临时 tensor 1
    high = (b_packed >> 4) & 0x0F   # 临时 tensor 2
    low_f = FP4_TABLE[low]          # 临时 tensor 3（查表）
    high_f = FP4_TABLE[high]        # 临时 tensor 4（查表）
    stacked = torch.stack([low_f, high_f], dim=-1)  # 临时 tensor 5
    flat = stacked.flatten(-2)      # 临时 tensor 6
    scale_expanded = b_scale.repeat_interleave(32, dim=-1)  # 临时 tensor 7
    return flat * scale_expanded    # 临时 tensor 8，最终输出
```

一次反量化，创建了 8 个临时 tensor（tensor = 多维数组，深度学习里对向量、矩阵的统称；这里的临时 tensor 都是二维矩阵）。每个临时矩阵都意味着一次 GPU 显存分配和一次数据写入+读出——不是分配本身慢（微秒级），而是**数据在显存里反复搬运占用了宝贵的内存带宽**。对于一个 FC1 专家的权重矩阵（4096 行 × 7168 列），反量化后的 BF16 矩阵约 56MB。8 行 Python 代码 = 8 次 GPU kernel launch + 多个大小不等的临时矩阵反复读写显存。DGX Spark 的显存带宽只有 273 GB/s，这些"写出→读回→再写出→再读回"的搬运加起来，比实际的查表和位操作计算还慢。

这就是为什么换了矩阵乘法的实现方式速度不变：**瓶颈不在矩阵乘法（matmul = matrix multiply，神经网络里最核心的计算操作），在反量化**。更准确地说，不在反量化的"计算"（查表、位操作），在反量化过程中产生的大量临时 tensor 的显存分配和带宽消耗。

解法：用 Triton 写一个融合 kernel（fused kernel——"融合"就是字面意思，把原来分开的多步操作合并成一个 kernel 一次完成），不产生中间 tensor。

先解释一下 FP4 的存储格式，否则后面的代码看不懂。

DeepSeek V4 用的 MXFP4（MX-compliant FP4）有两个组成部分：

**E2M1 数据**：每个 4-bit 浮点数包含 1 位符号 + 2 位指数 + 1 位尾数。两个 E2M1 打包在一个 uint8 里——低 4 bit 是第一个数，高 4 bit 是第二个数。所以 `[N, K]` 的权重矩阵存储为 `[N, K/2]` 的 uint8。

E2M1 能表示的值只有 16 个：`{0, 0.5, 1.0, 1.5, 2.0, 3.0, 4.0, 6.0}` 和它们的负数。范围很小——靠 scale factor 拉伸到正常范围。

**E8M0 Scale**：每 32 个值共享一个 8 位 scale factor。E8M0 = 纯指数格式，没有尾数，值 = 2^(e-127)。scale 存储为 `[N, K/32]` 的 uint8。

反量化公式：`real_value = E2M1_value * 2^(E8M0 - 127)`

Triton kernel 的核心思路是：一个 kernel 内完成"解包 uint8 → 查 E2M1 表 → 读 E8M0 scale → 乘 scale → 写出 BF16"：

```python
@triton.jit
def _dequant_fp4_e2m1_kernel(b_ptr, scale_ptr, out_ptr, ...):
    # 加载 packed uint8
    b = tl.load(b_ptr + offsets, mask=mask).to(tl.uint8)
    
    # 拆包：低 4 bit 和高 4 bit
    low_idx = b & 0x0F
    high_idx = (b >> 4) & 0x0F
    
    # E2M1 查表（硬编码，避免 table load）
    low_abs = tl.where(low_mag == 0, 0.0,
              tl.where(low_mag == 1, 0.5,
              tl.where(low_mag == 2, 1.0, ...)))
    
    # E8M0 scale：2^(e - 127)
    scale_e8m0 = tl.load(scale_ptr + scale_offsets)
    scale_f = tl.exp2(scale_e8m0 - 127.0)
    
    # 乘 scale，写出
    tl.store(out_ptr + low_offsets, (low_val * scale_f).to(tl.bfloat16))
    tl.store(out_ptr + high_offsets, (high_val * scale_f).to(tl.bfloat16))
```

整个过程只有一次显存读（packed FP4 + scale）和一次显存写（BF16 输出），没有中间 tensor。

测试结果：

```
正确性：max_diff = 0.000000（与 Python 版逐元素完全一致）

速度（单次反量化 [4096, 3584]）：
  Python:   3.45 ms
  Triton:   0.22 ms
  加速比：  15.4x

端到端 grouped GEMM：
  FC1 [M=384, K=7168, N=4096]:
    Python 版:  52.59 ms
    Triton 版:  14.00 ms → 3.8x
```

接入 vLLM 实测：

```
500 tokens / 300s = 1.67 tok/s（干净英文输出）
```

**从 0.79 到 1.67，提速 2.1 倍。**

提速来自哪里？不是因为 Triton 比 Python 算得快（GPU 上查表和位操作本来就快），而是因为**消灭了临时 tensor**。8 个 tensor 的创建/销毁/读写带宽全省了。这是 GPU 编程中一个核心教训：**内存带宽比计算更贵**。现代 GPU 的计算吞吐远大于内存带宽——对于 FP4 反量化这种"计算简单但数据量大"的操作，写 8 次中间结果的带宽开销比做几万次乘加还大。

◆ 插曲：torch.bmm 批量化（失败）

━━━━━━━━━━━━━━━━━━━━

补充一点：上面说"用 Triton 重写"，重写的只是 FP4 反量化这一步。反量化完成后的矩阵乘法，仍然用的是 `torch.mm`——PyTorch 的矩阵乘法函数，底层调 NVIDIA 的 cuBLAS 库。cuBLAS 和 176 期讲的 CUTLASS 名字像但不是一个东西：**cuBLAS 是闭源的成品库**（NVIDIA 编译好的黑盒，你调用就行，不知道里面怎么实现的，但保证快且正确）；**CUTLASS 是开源的模板库**（给你一箱零件自己拼，能定制但编译慢、门槛高）。`torch.mm` 调的是 cuBLAS，Consumer-DeepGEMM 之前用的是 CUTLASS。cuBLAS 在 SM120 上跑标准的 BF16 矩阵乘法完全没问题——出问题的只是 FP4 这种特殊格式。所以第一步的方案是"混搭"：**Triton 做反量化（快 15 倍），cuBLAS 做矩阵乘法（本来就快）**。

那自然就想到下一个优化：既然 6 个专家各做一次 `torch.mm`（= 一次 cuBLAS 调用），能不能把 6 组数据拼成一个 batch，用 `torch.bmm`（批量矩阵乘法，一次调用算多组）一次搞定？

```python
# 之前：6 次独立矩阵乘法
for expert in experts:
    d[rows] = torch.mm(a[rows], w[expert].T)

# 构想：先把 6 组数据叠成一摞（stack），再一次批量乘完（bmm = batch matrix multiply）
a_batch = torch.stack([a[rows_0], a[rows_1], ...])  # 6 个矩阵叠成 [6, M, K]
w_batch = torch.stack([w[e0], w[e1], ...])           # 6 个权重叠成 [6, N, K]
d_batch = torch.bmm(a_batch, w_batch.transpose(1,2)) # 一次算完 6 组，结果 [6, M, N]
```

matmul 部分确实快了——6 次 cuBLAS launch 变 1 次，cuBLAS 内部的批量调度更高效，计算时间降低 1.56 倍。

但 `torch.stack` 把 6 个 `[4096, 7168]` 的 BF16 tensor 拼成一个 batch tensor，需要分配约 670MB 连续显存并把数据拷贝进去。这个 stack 操作的开销比省下来的 matmul 开销还大——**端到端反而慢了 20%**。

放弃。

这是第二个教训：**优化要看端到端，不能只看单步**。单看 matmul 提速了 1.56 倍，加上 stack 的开销整体反而退步。优化链路上的每一步都要放回完整管线里去衡量。

━━━━━━━━━━━━━━━━━━━━

◆ 第二步：Fused Dequant+Matmul Kernel（1.67 → 1.87 tok/s）

━━━━━━━━━━━━━━━━━━━━

第一步消灭了反量化**内部**的临时 tensor（8 个中间矩阵→0 个），但整个流程仍然分两步走：

```
步骤 1：Triton dequant kernel
  读入：FP4 packed [4096, 3584] = 14.6MB
  写出：BF16 [4096, 7168] = 56MB  ← 这个中间结果落地到显存

步骤 2：cuBLAS matmul
  读入：BF16 [4096, 7168] = 56MB  ← 从显存重新读回来
  读入：A [M, 7168]
  写出：D [M, 4096]
```

中间那个 56MB 的 BF16 矩阵被**写了一次、读了一次**——写是 dequant kernel 写的，读是 cuBLAS 读的。每个专家 112MB 的"无用"带宽，每个 token 做 12 次（6 专家 × 2 FC）= **1.3GB 的数据在显存里搬了个寂寞**。

能不能在 kernel 内部边反量化边做矩阵乘法，让这个 56MB 的 BF16 矩阵根本不落地到显存？

这就是 fused kernel 的核心思路——把两个操作合并到一个 kernel 里：

```
分两步（之前）：
  W_bf16[n, k] = dequant(B_packed[n, k/2], scale[n, k/32])   ← 写出 56MB 到显存
  D[m, n]      = A[m, :] × W_bf16[n, :]                       ← 从显存读回 56MB

融合成一步（现在）：
  for k in range(K):
      D[m, n] += A[m, k] * dequant(B_packed[n, k/2], scale[n, k/32])
  ↑ dequant 在循环的每一步里就地完成，W_bf16 根本不存在
```

差别就一个：56MB 的 `W_bf16` 矩阵是"先完整算出来存到显存，再整个读回来做乘法"，还是"算一小块、乘一小块、累加，从头到尾不落地"。后者就是 fused。

为什么切成小块就不用落地？因为 GPU 有两层存储——**显存**（大，128GB，慢）和**片上缓存**（小，每个计算单元几百 KB，快几百倍）。完整的 56MB 矩阵片上缓存放不下，只能写到显存；但 fused kernel 每次循环只 dequant 一个 `[64, 64]` 的小块（几 KB），直接留在片上寄存器里给矩阵乘法用，用完就扔——**从头到尾没碰显存**。

实现上的关键技巧在于处理 packed FP4 的数据布局。一个 uint8 装了两个 FP4 值——低 4 bit 对应偶数列，高 4 bit 对应奇数列。矩阵乘法需要沿 K 维做内积，但 K 维上的值交错打包在同一个 byte 里。

解法是分开做两次 `tl.dot`（`tl` = `triton.language`，Triton 库的标准缩写；`tl.dot` 是 Triton 的矩阵乘法函数，名字叫 dot 但实际是矩阵乘矩阵，底层用 tensor core 硬件加速）：

```python
# 简化版，省略了地址偏移计算（实际代码约 100 行，主要是指针算术）
@triton.jit
def _fused_dequant_matmul_kernel(a_ptr, b_ptr, bs_ptr, d_ptr, ...):
    acc = tl.zeros((BLOCK_M, BLOCK_N), dtype=tl.float32)
    
    for k_start in range(0, K, BLOCK_K):
        # 加载 A 的偶数列：A[m, 0], A[m, 2], A[m, 4], ...
        a_even = tl.load(a_ptr + even_offsets)
        # 加载 A 的奇数列：A[m, 1], A[m, 3], A[m, 5], ...
        a_odd = tl.load(a_ptr + odd_offsets)
        
        # 加载 packed FP4，就地反量化（不写出中间结果）
        b_packed = tl.load(b_ptr + packed_offsets)
        scale_f = tl.exp2(tl.load(bs_ptr + scale_offsets) - 127.0)
        low_val, high_val = _dequant_fp4_vals(b_packed, scale_f)
        
        # 两次 dot：偶数列 × 低半字节，奇数列 × 高半字节
        acc += tl.dot(a_even, tl.trans(low_val))
        acc += tl.dot(a_odd, tl.trans(high_val))
    
    tl.store(d_ptr + out_offsets, acc.to(tl.bfloat16))
```

`a_even` 是 A 矩阵的第 0、2、4... 列，对应 packed byte 的低 4 bit（第 0、2、4... 个原始 K 值）。`a_odd` 是 A 的第 1、3、5... 列，对应高 4 bit。分别和反量化后的 low_val、high_val 做 dot product，结果累加。

`tl.dot` 在 SM121 上走 tensor core——这是 Triton 编译器自动做的。但 tensor core 的累加有精度限制。

矩阵乘法的本质是"一行乘一列再全部加起来"——K=7168 意味着要累加 3584 次乘积（因为偶数列和奇数列分开做，每次 3584 个）。每次加法都有浮点舍入误差，累加 3584 次后误差会叠加。

精度基准是 Python 参考路径的结果（先反量化成 BF16 → 再用 cuBLAS 做 FP32 矩阵乘法 → 截断回 BF16），这是数值上最正确的路径。fused kernel 和它比：

```
| M=1, N=32, K=4096   | 相对误差 = 0.000000（完美）
| M=1, N=4096, K=7168  | 相对误差 ~ 0.5%
| M=64, N=4096, K=7168 | 相对误差 ~ 2%
```

2% 的相对误差完全可以接受——别忘了，FP4 本身就是把 16-bit 精度暴力压到 4-bit（只有 16 个可能的值），量化本身引入的误差比这个 2% 大得多。在一个本来就"糙"的基础上多糙 2%，对模型输出没有可观察的影响。

这里补充一个容易忽略的背景：DeepSeek V4 的 FP4 不是训练完再压缩的（post-training quantization），而是**训练时就模拟了 FP4 精度**（Quantization-Aware Training, QAT）。模型在训练过程中就"看到"了 FP4 的量化误差，权重的值是适应了这种误差的。这意味着推理时的 FP4 反量化规则**必须和训练时一致**——否则模型"学会适应"的那套误差模式就对不上了。我们的 Triton kernel 严格按照 MXFP4 标准（E2M1 数据 + E8M0 scale）做反量化，和数据中心卡上的正确行为一致，所以没有这个问题。而 Marlin 在 SM120 上的错误不是"精度低"而是"数据布局解释错误"——算出来的是垃圾，不是"稍微不精确的结果"。

还有一个让问题更严重的因素：MoE 架构的**容错性天生就差**——而 DeepSeek V4 的训练方式让这个问题雪上加霜。

DeepSeek V4 的训练分两个阶段：**先独立训练 10 多个领域专家**（数学、代码、Agent、指令跟随等，每个在各自领域的数据上单独做 SFT + RL，互不干扰），**再把这些专家作为 teacher 蒸馏到最终的统一模型**。这意味着 256 个专家的权重不是从同一个模型分化出来的——它们是各自独立演化后再"拼"到一起的，彼此之间没有共享梯度的历史，权重分布差异极大。

每个 token 只激活 6 个专家，算错一个就没有其他专家来"纠偏"。如果是 dense 模型（所有参数都参与每次计算），一处小误差可能被其他参数"稀释"掉；MoE 的稀疏激活意味着每个专家的话语权极大——一个算错，这一层的输出就歪了，歪了传到下一层继续歪，60 层下来面目全非。这就是为什么前面说"256 个专家里可能只有十几个踩雷，但踩一个就崩"。

这里你可能会问：**既然 FP4 只有 16 个值，反量化到 FP8（256 个值）就能无损表示了，为什么要一路膨胀到 BF16（65536 个值）？** 反量化到 FP8 确实不会丢精度，而且数据量只有 BF16 的一半，带宽省一倍。但问题是：反量化完了还要做矩阵乘法，矩阵乘法需要 GPU 硬件支持对应的数据格式。SM121 上的 FP8 矩阵乘法指令同样是残缺的——还是老黄的刀法，消费级卡的 FP8 tensor core 支持不完整。BF16 矩阵乘法是 SM121 上最稳、最快、兼容性最好的路径。说白了：**SM121 上 FP4 矩阵乘法没有，FP8 矩阵乘法也残的，只有 BF16 矩阵乘法是完整可用的。** 老黄一刀砍了 FP4，顺手把 FP8 也砍了半刀。所以只能一路反量化到 BF16 再算——不是我们想膨胀，是没得选。

Benchmark：

```
FC1 [M=384, K=7168, N=4096]:
  Triton dequant + cuBLAS mm:  13.95 ms
  Fused kernel:                 6.40 ms → 2.2x

FC2 [M=384, K=2048, N=7168]:
  Triton dequant + cuBLAS mm:   7.11 ms
  Fused kernel:                 3.22 ms → 2.2x
```

接入 vLLM 实测：

```
500 tokens / 267s = 1.87 tok/s（干净英文输出）
```

**从 1.67 到 1.87，再提速 12%。累计从 0.79 到 1.87，提速 2.4 倍。**

为什么 fused kernel 只快了 12% 而不是翻倍？因为第一步已经解决了最大的带宽瓶颈（临时 tensor），第二步省的是反量化到矩阵乘法之间的中间 tensor 读写。GPU 优化有一个经典分类：**compute-bound**（计算受限）和 **memory-bound**（带宽受限）。第一步把问题从 memory-bound 变成了 compute-bound——matmul 本身的 tensor core 运算成了主要矛盾。第二步继续省带宽，但主要矛盾已经不在带宽了，所以边际收益变小。

◆ 两步优化总结

━━━━━━━━━━━━━━━━━━━━

| 版本 | 单次 GEMM | 端到端速度 | 相对首版 |
|------|-----------|-----------|---------|
| 基线：Python fallback（临时 tensor 大量分配）※ | 52 ms | 0.79 tok/s | 1.0x |
| 第一步：Triton FP4 dequant + cuBLAS mm | 14 ms | 1.67 tok/s | 2.1x |
| 第二步：Fused Triton dequant+matmul | 6.4 ms | 1.87 tok/s | 2.4x |

※ 基线说明：178 期用的是 CUTLASS C++ 模板（0.8 tok/s），后来我们发现 CUTLASS launch overhead 改不掉（库的架构决定的），就换成了"Python 反量化 + cuBLAS 矩阵乘法"的方案——速度一样是 0.79 tok/s（CUTLASS 没更快是因为 launch overhead 抵消了计算优势），但 Python 代码改起来方便，适合继续优化。这一期的优化就是从这个 Python 版本出发的。

两步的提速来源不同：

**第一步**（2.1x）：消灭 Python 层的临时 tensor 分配。反量化从"8 个中间矩阵 + 8 次显存读写"变成"1 次读 + 1 次写"。本质是把**多次 kernel launch + 多次显存分配**合并成**一次 kernel**。

**第二步**（再 1.12x）：消灭反量化到矩阵乘法之间的中间 tensor 写出/读回。每个专家省了约 112MB 的显存带宽。本质是把**两个操作**（反量化 + 矩阵乘法）融合成**一个操作**。

两步的共同主题：**不是让计算变快，而是让数据搬运变少。**

◆ 为什么 Triton 是 SM120 上唯一的出路

━━━━━━━━━━━━━━━━━━━━

这一期全程在用 Triton 写 kernel。为什么选 Triton？最直接的原因：**jasl 的 fork 已经用 Triton 重写了其他所有计算路径，我们给他补最后一块 FP4 MoE 的拼图，当然用同样的语言。** 但选 Triton 不只是"跟风"——在 SM120 上它确实是唯一可行的选择。

178 期讲过 GPU 编程有从底层到高层的多个抽象层。简单回顾一下，在 SM120/SM121（DGX Spark、RTX 5090）上每一层的状况：

**手写 CUDA（最底层）**：直接写 GPU 汇编级代码。SM120 的很多底层指令和数据中心卡不同——FP4 tensor core 指令不存在，硬件数据搬运加速器不存在。手写 CUDA 需要精确知道哪些指令可用、哪些不可用，而 NVIDIA 对 SM120 的指令文档几乎是空白。实际上 Marlin 就是手写 CUDA，它在 SM120 上静默算错的根因就是用了 SM120 上行为不同的指令。

**CUTLASS（中间层）**：NVIDIA 的开源 C++ 模板库。SM120 模板能生成正确的代码（Consumer-DeepGEMM 验证了），但每次调用都有固定的初始化开销。这个开销改不掉——是库的架构决定的。

**Triton 标准库（高层）**：我们试过直接用 vLLM 集成的 Triton MoE kernel（由 OpenAI 开发，贡献给 Triton 官方项目），但这个 FP4 kernel 编译时生成了一条叫 `.tile::scatter4` 的 PTX 指令——这条指令在数据中心卡上存在，在 SM121 上物理不存在。所以标准库的 FP4 kernel 也不行。

这里的 PTX 就是去年新闻里说的"DeepSeek 工程师绕过 CUDA 直接写汇编"的那个 PTX——NVIDIA GPU 的低级指令语言，介于 CUDA 和硬件机器码之间。DeepSeek 训练 V3 时用 PTX 手动控制寄存器和通信调度，在 2048 块 H800 上实现了约 10 倍效率提升。我们这里遇到的是同一层东西的反面：Triton 编译器自动生成的 PTX 里包含了 SM121 不支持的指令，于是炸了。**DeepSeek 工程师主动选择写 PTX 是为了极致优化，我们被动撞上 PTX 是因为硬件不支持——同一个技术层，一个是主动下潜，一个是被动触礁。**

**自己写 Triton kernel（我们的选择）**：Triton 是一种"长得像 Python 的 GPU 编程语言"，你写的是"对这块数据做什么运算"，Triton 编译器负责把它翻译成目标 GPU 的机器码。关键在于：如果你不使用那些 SM120 不支持的高级指令，Triton 编译器会自动选择 SM120 兼容的代码生成路径。

关键区别：**我们写的 Triton FP4 kernel 没有用任何 SM100 专属特性**——不用 TMA、不用 `.tile::scatter4`、不用 `tcgen05`。只用了最基础的 `tl.load`、`tl.store`、`tl.dot`、位操作。Triton 编译器把这些翻译成 SM121 兼容的机器码。

这也是为什么 Triton 标准库（Triton 官方自带的 kernel 库，不是 jasl 写的）的 FP4 kernel 不行而我们自己写的行——标准库的 kernel 只考虑了数据中心卡（SM100），压根没适配消费级卡（SM120）。你说这算不算"不对"？Triton 标准库可以说"我又没说支持 SM120"；但 vLLM 在集成时应该检查 SM 版本然后拒绝或 fallback，而不是让用户跑到编译阶段才看到一行 PTX 报错——这就是集成方的锅。我们手写的 kernel 主动避开了 SM100 专属指令，jasl 重写 attention/einsum 时也是同样的思路。**性能低一些，但至少能跑对——在一个官方不打算支持的硬件上。**

◆ 2.4 倍之后：为什么到此为止

━━━━━━━━━━━━━━━━━━━━

还能继续优化吗？技术上还有一些方向（Python 调度开销融合、tile size 自动搜索、FP8 激活值 fuse），但全做了乐观估计也就 3-4 tok/s。而 jasl fork 是 14 tok/s。

差距在哪？

这里要注意一个关键区分：jasl 重写的那些路径——attention、MLA、einsum、hyperconnection——**都不涉及 FP4**。它们用的是 FP8 或 BF16，SM120 上本来就能算，只是 vLLM 原来的实现有兼容性问题，jasl 用 Triton 重写后就跑通了。**唯一用到 FP4 的是 MoE 专家的权重矩阵**——这条路 jasl 没碰（因为 SM120 上没有现成的 FP4 方案可用），我们碰了。

jasl 快不是因为他解决了 FP4 问题——是因为他让 FP4 之外的所有路径都跑在了高效的 Triton kernel 上。但 FP4 这个坑他也躲不开：jasl fork 的 14 tok/s 里，FP4 MoE 仍然走的是 Marlin（算错的那个），只是中文恰好不踩雷。英文照样出垃圾——178 期讲过的问题至今没解决。**只要 SM120 没有 FP4 硬件指令，这个坑就永远在那。**

我们在 FP4 这一个点上优化了 2.4 倍，但 FP4 软件模拟的天花板在那——再怎么优化也比不上 jasl 那些根本不需要 FP4 的路径。

不过我们写的 Triton FP4 kernel 并非没有价值——正确性完美，速度合理，如果 jasl 将来想把 FP4 MoE 也换掉 Marlin，这些 kernel 可以直接复用。

说到底，**消费级卡不支持 FP4 硬件计算，怎么折腾软件都有天花板。** 数据中心卡一条硬件指令完成的事，我们用 Triton 写了一百多行代码去模拟——优化了 2.4 倍，依然和硬件原生差一个数量级。就像用 CPU 软件渲染 3D 画面——再怎么优化算法，也比不上一块真显卡。FP4 计算就是消费级 Blackwell 的"缺的那块显卡"。

如果你的目标是"在 DGX Spark 上日常跑 280B 模型"，正确的选择不是无限优化 FP4 软件模拟，而是：用 jasl fork 跑中文（14 tok/s 够用），英文需求换一个不依赖 FP4 的小模型（比如 Qwen 3.6-27B，BF16 全精度单卡跑，零兼容性问题）。如果有 4 台 Spark（512GB），可以把 V4 的权重量化到 FP8——SM120 的 BF16 矩阵乘法完整可用，FP8 反量化到 BF16 的开销远小于 FP4，速度会好得多。如果只有两台，用 PyTorch 直接加载模型跑实验（正确性验证、中间层激活提取等，175 期（ https://mp.weixin.qq.com/s/vYaa7HaUfE8ykPQGa-EHVg ）讲过这条路）反而比折腾 FP4 kernel 优化更实用——速度慢但输出正确，做一次实验够了。工程的智慧不只是"怎么把它做快"，也包括"什么时候该换条路"。

◆ 过程中的六个教训

━━━━━━━━━━━━━━━━━━━━

**1. Profiling first，不猜**

"瓶颈在 CUTLASS launch overhead"是猜的。"换 dequant+cuBLAS 就能提速"也是猜的。一测就知道：全部时间都在 FP4 GEMM 函数里，而 FP4 GEMM 里的时间主要花在 Python 层的临时 tensor 操作上，不在 CUTLASS launch 也不在 cuBLAS matmul。

**2. Python 开销比你想的大**

GPU 编程有一个容易忽略的事实：Python 层面的"向量化 GPU 操作"（`torch.mm`、`tensor & 0x0F`、`torch.stack`）看起来在 GPU 上跑，但每一个操作都是一次独立的 kernel launch + 显存分配。十几行 Python 代码可能翻译成十几次 GPU kernel launch 和十几次显存 malloc/free。这些 overhead 在大 tensor 上不明显（计算时间远大于 launch 时间），但在 FP4 反量化这种"操作多、单次计算少"的场景下就变成了主要矛盾。

**3. 内存带宽比计算更贵**

DGX Spark 的 GPU 算力约 40 TFLOPS（每秒 40 万亿次浮点运算），显存带宽约 273 GB/s。粗略估算：一次矩阵乘法（384 行 × 7168 列 × 4096 列）的计算量 GPU 大约 0.5ms 就能算完。但如果中间要写出一个 56MB 的 BF16 矩阵再读回来，光数据搬运就要约 0.4ms——接近计算时间的 80%。这就是为什么 fused kernel 有效：省的不是计算，是数据搬运。

**4. 优化要看端到端**

torch.bmm 批量化让 matmul 快了 1.56x，但 stack 操作让整体慢了 20%。单步指标好看没用，只有端到端数字才是真的。这一点在 GPU 优化中尤其重要——GPU 的异步执行让你很难通过直觉判断瓶颈在哪，只有实际测量说了算。

**5. 选对抽象层比写极致 kernel 更重要**

我们在 MoE GEMM 这一个点上优化了 2.4 倍。jasl 在整条推理管线上重写了一遍，获得了 18 倍（对比 Consumer-DeepGEMM 的 0.79 → jasl 的 14 tok/s）。不是我们的 kernel 写得差——是我们选择优化的层次太低了。最有效的优化往往不是让单步更快，而是减少步数。

**6. 你不需要会写 CUDA，只需要知道它是怎么回事**

这三期（176、178、179）讲了很多底层细节——tensor core、FP4 打包格式、E8M0 scale、kernel launch overhead、显存带宽。你不需要记住这些细节，更不需要自己能手写 Triton kernel。

但你现在知道了"GPU 编程大概是怎么回事"：知道 kernel 是什么、知道 launch 有开销、知道临时 tensor 会吃带宽、知道融合 kernel 就是把多步合一步、知道消费级卡和数据中心卡的指令集不一样。**这就够了。**

这是 vibe coding 时代的核心能力：你不需要自己能写，但你需要知道"这个东西大概是怎么回事"。有了心智模型，你让 AI 写 Triton kernel 时能判断方向对不对、结果靠不靠谱。不懂原理的人让 AI "写一个快的 GPU kernel"，AI 给什么就信什么；读完这三期的人会追问"你用了 TMA 吗？SM120 不支持"——**知道该问什么，比知道怎么写更重要。**

vLLM 官方仍然没有修复 SM120+ 的 MXFP4 MoE 问题。issue #40928 挂着，排期不明。DeepGEMM 明确不打算支持 SM120+。社区自救仍然是唯一出路。

◆ 代码

━━━━━━━━━━━━━━━━━━━━

本期涉及的 Triton FP4 kernel 在 Consumer-DeepGEMM 仓库：

```
Consumer-DeepGEMM/consumer_deep_gemm/triton_moe.py
```

核心内容：
- `_dequant_fp4_e2m1_kernel`：独立 FP4 反量化 Triton kernel
- `_fused_dequant_matmul_kernel`：Fused 反量化+矩阵乘法 kernel
- `m_grouped_fp8_fp4_gemm_nt_contiguous_triton`：接入 vLLM 的 MoE grouped GEMM 入口

Consumer-DeepGEMM: https://github.com/lmxxf/Consumer-DeepGEMM
jasl fork: https://github.com/jasl/vllm (分支 ds4-sm120)
部署记录: https://github.com/lmxxf/deepseek-v4-deployment-on-dgx-spark
