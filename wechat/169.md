一个 token 的完整旅程——DeepSeek V4 预测"中国的首都是"的全流程

━━━━━━━━━━━━━━━━━━━━

◆ 今天干什么

━━━━━━━━━━━━━━━━━━━━

上一期我们讲了 DeepSeek V4 注意力的演化史——从 MHA 一路走到 CSA/HCA。那篇是"为什么这么设计"。

今天反过来：**不讲为什么，只讲怎么跑。**

我们拿一句"中国的首都是"作为输入，假设 tokenizer 把它切成 5 个 token，然后一步一步走完 V4-Pro 的整条前向传播路径——从文字变成 token ID，到 embedding，到 61 层 Transformer，到最后 softmax 采样出"北京"。

每一步都标维度变化。一个矩阵乘法都不漏。

先把 V4-Pro 的关键参数摆出来，后面随时查：

```text
词表大小:              129,280
隐藏维度 (hidden_size): 7,168
Transformer 层数:       61
注意力头数:             128
KV 头数:                1（MLA，只存 1 份 512 维 latent）
Q 低秩:                 q_lora_rank = 1,536
KV latent 维度:         head_dim = 512
RoPE 维度:              qk_rope_head_dim = 64
O 低秩:                 o_lora_rank = 1,024，o_groups = 16
滑动窗口:               128
Hyper-Connections:      hc_mult = 4（维护 4 份副本）
MoE 专家总数:           384
每 token 激活专家:       6（top-6）
共享专家:               1
专家中间维度:            3,072
MoE 打分函数:           sqrtsoftplus = sqrt(softplus(x))
前 3 层路由:             hash-based（token ID 直接映射）
其余 58 层路由:          score-based（打分选 top-6）
MTP 层:                 1（compress_ratio = 0，纯滑动窗口）
```

压缩注意力配置（61 层主干 + 1 层 MTP）：

```text
层 0:   ratio=128  HCA
层 1:   ratio=128  HCA
层 2:   ratio=4    CSA（带 Indexer）
层 3:   ratio=128  HCA
层 4:   ratio=4    CSA（带 Indexer）
...（CSA 和 HCA 交替）
层 60:  ratio=4    CSA（带 Indexer）
MTP:    ratio=0    纯滑动窗口
```

HCA = Highly Compressed Attention，128:1 粗压缩，全看。
CSA = Compressed Sparse Attention，4:1 细压缩，Indexer 选 top-1024。

好了，开始走。

━━━━━━━━━━━━━━━━━━━━

◆ Step 0: Tokenize——文字变 token ID

━━━━━━━━━━━━━━━━━━━━

输入："中国的首都是"

Tokenizer（V4 用的是 BPE 类 tokenizer，词表 129,280）把这句话切成 5 个 token。假设切法如下：

```text
"中国" → token_id = 9412
"的"   → token_id = 1756
"首"   → token_id = 38901
"都"   → token_id = 12044
"是"   → token_id = 2564
```

（具体 ID 是假设的，不同版本 tokenizer 可能不同。重点是切成 5 个 token。）

输出：一个整数数组 `[9412, 1756, 38901, 12044, 2564]`，形状 `[1, 5]`（batch=1, seq_len=5）。

这一步纯查表，没有任何神经网络计算。

━━━━━━━━━━━━━━━━━━━━

◆ Step 1: Embedding——token ID 变向量

━━━━━━━━━━━━━━━━━━━━

模型有一张 embedding 表，形状 `[129280, 7168]`——词表里每个 token 对应一行 7168 维向量。

```text
输入:  token_ids [1, 5]
查表:  embedding_table [129280, 7168]
输出:  h [1, 5, 7168]
```

5 个 token ID 各自去 embedding 表里查到自己那一行，得到 5 个 7168 维向量。

这一步也是纯查表（不是矩阵乘法，是 index select）。

此时 `h` 的形状是 `[1, 5, 7168]`——batch=1，序列长度=5，每个 token 是 7168 维。

━━━━━━━━━━━━━━━━━━━━

◆ Step 2: Hyper-Connections 初始化——1 份变 4 份

━━━━━━━━━━━━━━━━━━━━

V4 不用标准的残差连接（`output = input + layer(input)`）。它用 **Hyper-Connections (HC)**，同时维护 **4 份 hidden state 副本**。

HC 的初始化很简单：把 embedding 输出复制 4 份，然后拼起来。

```text
输入:  h [1, 5, 7168]
操作:  复制 4 份，沿最后一维拼接
输出:  h_hc [1, 5, 7168 × 4] = [1, 5, 28672]
```

这 4 份不是完全相同的复制——初始化时会乘上一组学习的系数。但在第一层之前，4 份的内容基本相同。

后面 61 层 Transformer，每一层都在这 4 份上操作：先合并成 1 份送入 attention/FFN，再把输出展开回 4 份。4 份之间通过学习的权重矩阵交换信息，比简单残差更灵活。

为什么要 4 份？直觉上，标准残差连接只有"原样保留"和"加上变化"两条路。HC 给了 4 条路——模型可以在 4 条路上分别存不同粒度的信息（比如一份偏语义、一份偏位置、一份偏局部上下文），然后在需要时灵活合并。这就是 HC 比残差连接更强的原因。

━━━━━━━━━━━━━━━━━━━━

◆ Step 3 ~ Step 63: 61 层 Transformer

━━━━━━━━━━━━━━━━━━━━

从这里开始，5 个 token 要穿过 61 层 Transformer。每一层内部的流程完全一样，只是参数不同。

每一层做两件大事：**注意力** 和 **MoE FFN**。每件大事前后都有 Hyper-Connections 的合并/展开操作。

一层的完整流程：

```text
[hc_pre]    4 份 → 1 份（送入注意力）
[RMSNorm]   归一化
[Attention]  注意力计算
[hc_post]   1 份 → 4 份（和残差组合）
[hc_pre]    4 份 → 1 份（送入 MoE）
[RMSNorm]   归一化
[MoE FFN]   混合专家前馈
[hc_post]   1 份 → 4 份（和残差组合）
```

下面拆开讲每一步。

━━━━━━━━━━━━━━━━━━━━

◆ 3.1 hc_pre: 4 份合并为 1 份

━━━━━━━━━━━━━━━━━━━━

进注意力之前，要把 4 份 hidden state 合并成 1 份。

```text
输入:  h_hc [1, 5, 28672]   （4 份 × 7168）
操作:  把 28672 拆成 [4, 7168]，用学习的权重向量 α [4] 加权求和
       α 是通过 Sinkhorn 正则化约束的（双随机矩阵，保证信息不丢不膨）
输出:  h [1, 5, 7168]
```

4 份变 1 份。这个 1 份是 4 份的加权混合，权重是训练学出来的。

同时，原始的 `h_hc [1, 5, 28672]` 保留着——后面 hc_post 还要用它做残差。

━━━━━━━━━━━━━━━━━━━━

◆ 3.2 RMSNorm

━━━━━━━━━━━━━━━━━━━━

```text
输入:  h [1, 5, 7168]
操作:  RMSNorm（均方根归一化，eps=1e-6）
       RMS = sqrt(mean(h²) + eps)
       h = h / RMS × γ        （γ 是可学习的缩放参数，7168 维）
输出:  h [1, 5, 7168]          （形状不变，数值归一化了）
```

RMSNorm 比 LayerNorm 简单——不减均值，只除均方根。效果差不多，速度快。

━━━━━━━━━━━━━━━━━━━━

◆ 3.3 注意力——这是重头戏

━━━━━━━━━━━━━━━━━━━━

注意力是整个 Transformer 最复杂的部分。V4 的注意力比标准注意力多了好几层嵌套：Q 低秩投影、MLA 的 KV latent、RoPE、Compressor、Indexer、分组低秩 O 投影。

一步一步来。

**3.3.1 Q 的低秩投影**

标准注意力直接把 hidden state 投影成 Q：`Q = h × W_q`，其中 W_q 的形状是 `[7168, 128×head_dim_q]`。

V4 的 Q 有低秩压缩。不是一步到位，而是先压后展：

```text
第一步：压缩
  h [1, 5, 7168] × W_q_down [7168, 1536] → q_latent [1, 5, 1536]

第二步：RMSNorm
  q_latent [1, 5, 1536] → q_latent [1, 5, 1536]   （归一化）

第三步：展开
  q_latent [1, 5, 1536] × W_q_up [1536, 128×(head_dim_per_head + 64)]
  → q_full [1, 5, 128 × (某个维度)]
```

这里需要解释一下维度。V4 的 128 个 Q 头，每个头的 Q 向量分两部分：

```text
- 非 RoPE 部分：用于和 KV latent 做点积（维度由 head_dim=512 决定）
- RoPE 部分：64 维，加上旋转位置编码
```

但因为 KV 只有 1 份 512 维 latent（num_key_value_heads=1），Q 的非 RoPE 部分实际上是 512 维（和 latent 对齐）。所以每个 Q 头的维度 = 512 + 64 = 576。

```text
W_q_up: [1536, 128 × 576] = [1536, 73728]
q_full: [1, 5, 73728] → reshape → [1, 5, 128, 576]
```

拆开：

```text
q_nope [1, 5, 128, 512]   ← 非位置编码部分
q_rope [1, 5, 128, 64]    ← 位置编码部分
```

**为什么 Q 要低秩压缩？** 因为从 7168 维直接投影到 73728 维（128 头 × 576 维）需要一个巨大的矩阵 `[7168, 73728]`，约 5.3 亿参数。用两步低秩分解（7168→1536→73728），参数量降到 7168×1536 + 1536×73728 ≈ 1.24 亿，省了 76%。

**3.3.2 KV 的投影**

V4 的 KV 是 MLA 风格——只投影出 1 份 latent + 1 份 RoPE key：

```text
h [1, 5, 7168] × W_kv [7168, 512 + 64] → kv [1, 5, 576]
```

拆开：

```text
kv_latent [1, 5, 1, 512]     ← 1 份 KV latent（num_kv_heads=1）
k_rope    [1, 5, 1, 64]      ← 1 份 RoPE key
```

注意：V4 的 MLA 和 V3 不同——V3 有 kv_lora_rank（先压再展），V4 直接投影到 512 维，没有 latent 的二次压缩。更简单。

**3.3.3 RoPE（旋转位置编码）**

Q 和 K 的 RoPE 部分（各 64 维）要加上位置信息。

```text
q_rope [1, 5, 128, 64] × RoPE(position_ids, theta=10000, YaRN scaling)
→ q_rope [1, 5, 128, 64]

k_rope [1, 5, 1, 64] × RoPE(position_ids, theta=10000, YaRN scaling)
→ k_rope [1, 5, 1, 64]
```

RoPE 把位置信息编码成旋转角度，乘到 Q 和 K 上。位置越远，旋转角度差越大，点积越小——这就是"距离越远相关性越低"的位置先验。

YaRN 是长上下文的 RoPE 扩展方案，factor=16，让模型能处理 1M token 的超长上下文（base 是 65536 位置，×16 = 1048576）。

**3.3.4 关键判断：Compressor 能用吗？**

到这里，每个 token 都有了 Q 和 KV。接下来该做注意力计算了。但 V4 的注意力不是直接 Q×K——要先过 Compressor。

Compressor 需要把多个 token 的 KV 压成一条。但我们只有 5 个 token。

```text
HCA 层（ratio=128）：需要 128 个 token 才能凑满一个块。5 < 128，凑不满。
CSA 层（ratio=4）：  需要 4 个 token 才能凑满一个块。5 个 token 能凑 1 个块（4 个），剩 1 个。
```

**5 个 token 太短了。** HCA 层完全凑不满一个压缩块。CSA 层勉强凑 1 个块，但剩下的 1 个 token 凑不满第二个块。

凑不满的怎么办？**全走滑动窗口。**

滑动窗口大小是 128。我们只有 5 个 token，5 < 128，所以全部 5 个 token 都在滑动窗口范围内——**每个 token 直接用原始 KV 做标准注意力，不压缩**。

这意味着对于"中国的首都是"这种短输入：

```text
HCA 层：全走滑动窗口（5 < 128，没有任何压缩块）
CSA 层：可能有 1 个压缩块（token 1~4 压成 1 条），但 token 5 仍在滑动窗口内
         实际上由于 5 个 token 都在滑动窗口范围内，Indexer 也没什么可选的
```

**所以对短输入来说，V4 的注意力退化成了标准的因果注意力。** CSA/HCA/Indexer 这些花哨的机制在短输入时几乎不起作用——它们是为长上下文设计的。短输入时 V4 和传统 Transformer 没区别。

这很合理。5 个 token 的上下文不需要"压缩"——全看就行了，也看得起。

**3.3.5 注意力计算**

既然全走滑动窗口，注意力计算就是标准的因果注意力（causal attention）：

```text
Q: [1, 5, 128, 576]  拆成 q_nope [1, 5, 128, 512] 和 q_rope [1, 5, 128, 64]
K: [1, 5, 1, 576]    拆成 kv_latent [1, 5, 1, 512] 和 k_rope [1, 5, 1, 64]
V: 就是 kv_latent [1, 5, 1, 512]（MLA：K 和 V 共享 latent）
```

先算相似度分数。分数由两部分组成——非 RoPE 部分和 RoPE 部分：

```text
score_nope = q_nope × kv_latentᵀ
  [1, 128, 5, 512] × [1, 1, 512, 5] → [1, 128, 5, 5]
  （128 个 Q 头，每个都和同 1 份 KV latent 做点积）

score_rope = q_rope × k_ropeᵀ
  [1, 128, 5, 64] × [1, 1, 64, 5] → [1, 128, 5, 5]
  （128 个 Q 头的 RoPE 部分，和 1 份 k_rope 做点积）

score = (score_nope + score_rope) / √(512 + 64) = score / √576
  → [1, 128, 5, 5]
```

加上因果掩码（causal mask）——每个 token 只能看自己和前面的 token：

```text
         t1   t2   t3   t4   t5
t1       ✓    -    -    -    -
t2       ✓    ✓    -    -    -
t3       ✓    ✓    ✓    -    -
t4       ✓    ✓    ✓    ✓    -
t5       ✓    ✓    ✓    ✓    ✓
```

`-` 的位置设为负无穷，softmax 后变成 0。

```text
attn_weights = softmax(score + causal_mask)
  → [1, 128, 5, 5]   每行加起来 = 1
```

加权求和得到注意力输出：

```text
attn_output = attn_weights × kv_latent
  [1, 128, 5, 5] × [1, 1, 5, 512] → [1, 128, 5, 512]
```

128 个头，每个头输出 512 维。

**3.3.6 O 的分组低秩投影**

标准 Transformer 用一个大矩阵 W_o 把多头输出拼接后投影回 hidden_size：

```text
标准：[1, 5, 128×512] × W_o [65536, 7168] → [1, 5, 7168]
```

V4 用分组低秩投影：128 个头分成 16 组，每组 8 个头。每组先压到 o_lora_rank=1024 维，再展开回 7168 维。但 16 组共享同一个展开矩阵。

```text
第一步：128 头分 16 组，每组 8 头 × 512 维 = 4096 维
  attn_output reshape → [1, 5, 16, 4096]

第二步：每组压缩
  每组 [1, 5, 4096] × W_o_down_group [4096, 1024] → [1, 5, 1024]
  16 组各有自己的 W_o_down（16 个不同的 [4096, 1024] 矩阵）
  输出 [1, 5, 16, 1024]

第三步：16 组加起来
  [1, 5, 16, 1024] → sum(dim=2) → [1, 5, 1024]

第四步：展开回 hidden_size
  [1, 5, 1024] × W_o_up [1024, 7168] → [1, 5, 7168]
```

注意力输出：`[1, 5, 7168]`。

**为什么 O 也要低秩？** 和 Q 的低秩压缩同理——省参数。128 头 × 512 维 = 65536 维直接投影到 7168 维，矩阵太大。分组低秩把 65536×7168 ≈ 4.7 亿参数压到 16×4096×1024 + 1024×7168 ≈ 7400 万参数。

━━━━━━━━━━━━━━━━━━━━

◆ 3.4 hc_post: 1 份变回 4 份

━━━━━━━━━━━━━━━━━━━━

注意力输出是 1 份 `[1, 5, 7168]`，要展开回 4 份，和之前保留的残差组合：

```text
输入:
  attn_out [1, 5, 7168]          ← 注意力输出（1 份）
  h_hc_residual [1, 5, 28672]    ← hc_pre 之前保留的 4 份残差

操作:
  1. attn_out 通过学习的权重扩展成 4 份: [1, 5, 7168] → [1, 5, 28672]
  2. 和残差加权组合: h_hc = α × h_hc_residual + β × expanded_attn_out
     （α、β 是学习的系数，受 Sinkhorn 约束）

输出:  h_hc [1, 5, 28672]       （更新后的 4 份 hidden state）
```

这就是 Hyper-Connections 比标准残差连接灵活的地方：标准残差是 `h = h + attn_out`（一条路），HC 是 4 条路各自有不同的混合系数。

━━━━━━━━━━━━━━━━━━━━

◆ 3.5 hc_pre + RMSNorm（第二次，送入 MoE）

━━━━━━━━━━━━━━━━━━━━

和 3.1、3.2 完全相同的流程：

```text
h_hc [1, 5, 28672] → hc_pre → h [1, 5, 7168] → RMSNorm → h [1, 5, 7168]
```

4 份合并成 1 份，归一化，准备送入 MoE FFN。

━━━━━━━━━━━━━━━━━━━━

◆ 3.6 MoE FFN——384 个专家选 6 个

━━━━━━━━━━━━━━━━━━━━

V4-Pro 的所有 61 层都是 MoE（Mixture of Experts）。没有 dense 层。

MoE 做的事：**每个 token 从 384 个专家里选 6 个来处理，再加上 1 个所有 token 共享的专家。**

但前 3 层和后 58 层的选法不同。

**3.6.1 前 3 层：hash-based routing**

前 3 层用 hash 路由——不打分，直接用 token ID 决定激活哪些专家。

```text
expert_indices = hash(token_id) % 384  →  选出 6 个专家
```

具体的 hash 映射是预定义的查找表。每个 token ID 对应固定的 6 个专家——同一个 token 不管出现在哪个位置、什么上下文里，前 3 层激活的专家都一样。

为什么前 3 层用 hash？因为前几层的 hidden state 还没有足够的语义信息（刚从 embedding 出来），用打分来选专家效果不好——分数噪声太大。Hash 路由虽然粗暴，但保证了负载均衡（每个专家被访问的概率相同），避免了"训练初期所有 token 都涌向同一个专家"的崩溃问题。

**3.6.2 后 58 层：score-based routing**

第 4 层开始用正经的打分路由：

```text
第一步：算分数
  h [1, 5, 7168] × W_gate [7168, 384] → scores [1, 5, 384]

第二步：打分函数 sqrtsoftplus
  scores = sqrt(softplus(scores))
  其中 softplus(x) = log(1 + exp(x))
  sqrtsoftplus 比 softmax 更平滑，避免 winner-take-all

第三步：选 top-6
  从 384 个分数里选最大的 6 个
  top6_indices [1, 5, 6]     ← 选中的专家编号
  top6_scores  [1, 5, 6]     ← 对应的分数

第四步：归一化权重
  weights = top6_scores / sum(top6_scores)   ← 6 个权重加起来 = 1
```

**3.6.3 专家内部：SwiGLU FFN**

每个被选中的专家是一个标准的 SwiGLU FFN：

```text
输入:  x [1, 1, 7168]   （一个 token 的 hidden state）

gate = x × W1 [7168, 3072]  → [1, 1, 3072]    ← gate 分支
up   = x × W3 [7168, 3072]  → [1, 1, 3072]    ← up 分支
hidden = SiLU(gate) * up                        ← 逐元素相乘
hidden = clamp(hidden, -10.0, 10.0)             ← swiglu_limit 截断
down = hidden × W2 [3072, 7168] → [1, 1, 7168] ← 投影回来
```

SwiGLU 的核心是那个 `SiLU(gate) * up`——gate 分支过激活函数做"开关"，up 分支提供"内容"，逐元素相乘。哪些维度被 gate "打开"了，对应维度的 up 值就通过；gate 关闭的维度输出接近 0。

`swiglu_limit = 10.0` 是一个工程保护——把中间值截断到 `[-10, 10]`，防止 FP8/FP4 量化时溢出。

**专家权重 FP4 量化**：每个专家的 W1、W2、W3 都以 FP4 存储和计算。384 个专家 × 3 个矩阵 × 7168 × 3072 = 超过 250 亿参数，FP4 比 FP16 省 4 倍存储。

**3.6.4 共享专家**

除了 6 个路由专家，还有 1 个共享专家。它的结构和路由专家完全一样（SwiGLU，7168→3072→7168），但**每个 token 都会过它**，不需要路由。

```text
shared_output = SharedExpert(h)   ← 每个 token 都算
```

**3.6.5 合并**

```text
moe_output = shared_output + Σ(weight_i × expert_i_output)   （i = 1..6）
  → [1, 5, 7168]
```

6 个路由专家的加权和 + 共享专家的输出 = MoE 层的最终输出。

━━━━━━━━━━━━━━━━━━━━

◆ 3.7 hc_post（第二次）

━━━━━━━━━━━━━━━━━━━━

和 3.4 一样，MoE 输出从 1 份展开回 4 份，和残差组合：

```text
moe_out [1, 5, 7168] → 展开 + 残差组合 → h_hc [1, 5, 28672]
```

**一层 Transformer 到此结束。**

━━━━━━━━━━━━━━━━━━━━

◆ 重复 61 次

━━━━━━━━━━━━━━━━━━━━

上面 3.1 到 3.7 的全部流程，重复 61 次。

每一层的区别：
- 注意力类型交替：层 0、1 是 HCA（ratio=128），层 2 是 CSA（ratio=4），层 3 是 HCA……交替进行
- 对于 5 个 token 的短输入，HCA 和 CSA 都退化为滑动窗口，实际行为相同
- 前 3 层 MoE 用 hash routing，后 58 层用 score routing
- 每一层有自己独立的权重参数

61 层跑完，`h_hc` 的形状还是 `[1, 5, 28672]`。

━━━━━━━━━━━━━━━━━━━━

◆ Step 64: hc_head——4 份合并回 1 份

━━━━━━━━━━━━━━━━━━━━

61 层全部跑完，最终输出还是 4 份。现在要合并回 1 份，准备做 token 预测。

```text
输入:  h_hc [1, 5, 28672]
操作:  hc_head — 用 sigmoid 门控加权合并
       gate = sigmoid(h_hc × W_gate)      ← 学习哪些维度重要
       h = Σ(gate_i × h_hc_i)             ← 4 份加权合并
输出:  h [1, 5, 7168]
```

这一步和 hc_pre 不同——hc_pre 用的是线性加权，hc_head 用 sigmoid 做非线性门控，表达力更强。这是因为 hc_pre 只是层内中间步骤（要过 61 遍，速度敏感），hc_head 只跑一次（最终输出，精度敏感）。

━━━━━━━━━━━━━━━━━━━━

◆ Step 65: RMSNorm → lm_head → logits

━━━━━━━━━━━━━━━━━━━━

```text
第一步：最终 RMSNorm
  h [1, 5, 7168] → RMSNorm → h [1, 5, 7168]

第二步：lm_head 投影到词表
  h [1, 5, 7168] × W_lm_head [7168, 129280] → logits [1, 5, 129280]
```

注意 V4 的 `tie_word_embeddings = false`——lm_head 的权重和 embedding 表不共享。这意味着模型有两张 `[129280, 7168]` 的大表：一张用于输入（embedding），一张用于输出（lm_head）。不共享的好处是输入和输出可以学到不同的表示空间。

logits `[1, 5, 129280]` 的含义：对于输入的 5 个位置，每个位置都给出"下一个 token 最可能是谁"的 129280 个分数。

**我们只关心最后一个位置**（第 5 个 token "是"的位置），因为我们要预测的是"是"后面跟什么。

```text
logits_last = logits[0, 4, :]   → [129280]
```

━━━━━━━━━━━━━━━━━━━━

◆ Step 66: softmax → 采样 → "北京"

━━━━━━━━━━━━━━━━━━━━

```text
第一步：temperature 缩放
  logits_last = logits_last / temperature    （temperature 通常 0.6~1.0）

第二步：softmax
  probs = softmax(logits_last)   → [129280]
  每个位置的值 ∈ (0, 1)，所有值加起来 = 1
  这就是"每个 token 被选中的概率"

第三步：采样
  如果 temperature=0（贪心）：直接选概率最大的
  如果 temperature>0：按概率分布随机采样（可能用 top-p/top-k 截断）
```

对于"中国的首都是"这个输入，"北京"对应的 token 概率几乎肯定是最高的——不管用什么采样策略，输出都是"北京"。

```text
输出:  token_id = 58832   →  "北京"
```

**一个 token 的完整旅程到此结束。**

━━━━━━━━━━━━━━━━━━━━

◆ 补充：MTP（Multi-Token Prediction）

━━━━━━━━━━━━━━━━━━━━

V4 在 61 层主干之后还有一个 MTP 层（num_nextn_predict_layers = 1），用于同时预测"下一个 token"和"下下一个 token"。

```text
MTP 层的配置:
  compress_ratio = 0     ← 不压缩，纯滑动窗口
  滑动窗口 = 128
  共享主模型的 embedding 和 lm_head 权重
```

MTP 的作用主要在训练阶段——它给模型额外的梯度信号："不仅要预测下一个 token 对，还要预测下下一个 token 对"。这迫使中间层学到更有前瞻性的表示。

推理时 MTP 可以用于 speculative decoding（投机解码）——先用 MTP 头快速猜测多个 token，再用主模型验证。猜对了就省了多步解码。但对于单次预测"中国的首都是→北京"这种场景，MTP 不影响主路径的输出。

MTP 层的 compress_ratio=0 意味着它不做任何 KV 压缩，只看最近 128 个 token 的原始 KV。这很合理——MTP 做的是末尾预测，不需要回看百万上下文。前面 61 层已经把长上下文信息揉进 hidden state 了。

━━━━━━━━━━━━━━━━━━━━

◆ 如果输入不是 5 个 token 而是 5 万个呢？

━━━━━━━━━━━━━━━━━━━━

上面全程走的是短输入，CSA/HCA 几乎没用上。如果输入是 50000 个 token（比如一本小说），情况就完全不同了：

```text
HCA 层（ratio=128）：
  50000 / 128 = 390 个压缩块（每块 128 token 压成 1 条 latent）
  + 最近 128 个 token 走滑动窗口
  注意力看 390 + 128 = 518 条 KV
  （不带 Indexer，全看）

CSA 层（ratio=4）：
  50000 / 4 = 12500 个压缩块（每块 4 token 压成 1 条 latent）
  Indexer 从 12500 条里选 top-1024
  + 最近 128 个 token 走滑动窗口
  注意力看 1024 + 128 = 1152 条 KV

标准注意力（没有压缩）：
  注意力看 50000 条 KV
```

差距出来了：

```text
标准注意力: 50000 条
HCA:       518 条    （省 99%）
CSA:       1152 条   （省 97.7%）
```

**这就是 V4 的长上下文优势。** 不是"看得少"，而是"先压缩再看"——HCA 用 128:1 的粗摘要扫全局，CSA 用 4:1 的细摘要选重点，滑动窗口保住最近的上下文。三种粒度交替，每过两层模型既扫了全局又精挑了细节。

━━━━━━━━━━━━━━━━━━━━

◆ 多想一步

━━━━━━━━━━━━━━━━━━━━

今天走完了一个 token 从输入到输出的每一步，维度变化、矩阵乘法，一个不漏。但走完全程之后，最让我意外的不是某个具体模块，而是一个反直觉的事实：

**对于 5 个 token 的短输入，V4-Pro 的 1.6T 参数、CSA/HCA、Lightning Indexer、Compressor——这些花了整篇论文讲的创新，全都没用上。**

5 个 token 走完 61 层，注意力全是滑动窗口内的标准因果注意力，和 2017 年的原版 Transformer 没有本质区别。MoE 前 3 层是 hash routing——token ID 决定一切，连打分都不用。Hyper-Connections 的 4 份副本在工作，但对 5 个 token 的微小上下文来说，和标准残差的效果差距也微乎其微。

**V4 所有的架构创新，几乎都是为长上下文准备的。**

短输入时模型"假装"自己是一个普通 Transformer。长上下文来了，那些机制才像开关一样一个一个亮起来：Compressor 开始压缩、Indexer 开始选 top-k、HCA 和 CSA 交替扫描和精选。

这给了一个有趣的工程启示：**好的架构设计不是"时刻都在发力"，而是"需要时才介入，不需要时自动退化为最简单的形式"。** CSA/HCA 在短输入时退化为标准注意力，MoE 在前几层退化为 hash 路由——这些不是设计缺陷，是设计意图。

从反面想：如果一个架构创新在任何输入上都大幅改变行为——不管输入是 5 个 token 还是 5 万个——那它大概率引入了不必要的归纳偏置。好的设计应该像 V4 这样：**机制的强度和问题的复杂度成正比。简单问题不需要复杂机制，让它静静地退化就好。**
