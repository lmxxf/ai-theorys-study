【DeepSeek V4 推理】一次反转：Marlin 不是算错了，而是算得太“正确”

━━━━━━━━━━━━━━━━━━━━

◆ 前情回顾

━━━━━━━━━━━━━━━━━━━━

185 期（那篇“sm_12x 问题修复，从 4 到 12 tok/s”）讲过一个现象：

DGX Spark 上跑 DeepSeek V4 Flash 280B，纯 Consumer-DeepGEMM（以下简称 CDG）能正确输出，但速度只有 4 tok/s 左右；纯 Marlin 能跑到 14 tok/s 左右，但英文输出乱码；最后我们用了一个混合方案：

```bash
VLLM_MXFP4_MARLIN_LAYER_RANGE=0:42
```

也就是：

- decoder layer 0-41 走 Marlin；
- decoder layer 42 走 CDG；
- 最终速度 12 tok/s 左右；
- 中英文输出正常。

当时的解释是：Marlin 在 sm_121 上每层有微小误差，前 42 层还没过临界点，最后一层如果继续用 Marlin，就把误差推过临界点；换成 CDG 就能兜住。

这个解释能解释现象，但它不是最终真相。

后来我们继续抓真实激活、做 mixed/all-marlin 双配置对比，发现 185 期的判断里有一个关键假设错了：

**前 42 层并没有出现 mixed 和 all-marlin 的分叉。**

也就是说，不存在“前 42 层 Marlin 一点点污染 residual stream，最后一层压垮骆驼”的证据。

真正的分叉点非常干净：

**layer 42。**

最后一层。

同一份输入，同一份路由，同一份专家权重，只是 kernel 不同：

- CDG 输出正常；
- Marlin 输出乱码。

而且更反直觉的是：

**按高精度公式算，Marlin 更接近参考答案；CDG 反而更远。**

这期就讲这个反转。

━━━━━━━━━━━━━━━━━━━━

◆ 先把假信号清掉

━━━━━━━━━━━━━━━━━━━━

最早那次 dump 里，我们看到 layer 42 的输出 `absmax=9344`，比正常层大几百倍。当时第一反应很自然：最后一层爆了。

后来复查才发现，这是 vLLM 的 warmup/profile dummy batch，不是真实 prompt。

真实请求从第 3 次每层调用开始。我们后来在 dump hook 里加了：

```bash
VLLM_MXFP4_DUMP_SKIP=3
```

跳过前三次 warmup/profile，只抓真实请求。

真实 prompt 是：

```text
Write a quicksort function in Python and explain how it works.
```

真实 prefill 的 shape 是：

```text
x.shape = (14, 4096)
```

14 行互不相同，符合 prompt token 数。

这时 layer 42 的数值不再爆炸：

```text
x_absmax   ≈ 2.55
out_absmax ≈ 8.8
```

所以“layer 42 数值爆炸”这条线撤回。

工程排障里很容易犯这种错误：你抓到了一个很夸张的数，就以为它是真凶。其实它可能只是启动阶段的假输入。模型服务框架会做 warmup、profiling、CUDA graph 捕获，这些都会制造看起来很像真实推理的张量。

所以后面所有结论都只基于跳过 warmup 之后的真实 dump。

━━━━━━━━━━━━━━━━━━━━

◆ 真正的分叉：0-41 完全一致，42 才分叉

━━━━━━━━━━━━━━━━━━━━

我们重跑了两次服务。

第一次 mixed：

```bash
VLLM_MXFP4_MARLIN_LAYER_RANGE=0:42
```

含义：0-41 Marlin，42 CDG。

第二次 all-marlin：

```bash
VLLM_MXFP4_MARLIN_LAYER_RANGE=0:43
```

含义：0-42 全 Marlin。

同一个 prompt，温度 0，抓每一层 MoE 的：

- 输入 `x`
- 路由专家编号 `topk_ids`
- 路由权重 `topk_weights`
- 输出 `out`
- 实际后端类型：`MarlinExperts` 还是 `DeepGemmFP4Experts`

然后逐层比较 mixed 和 all-marlin。

结果非常干净：

```text
layer 0-41:
  x_max_diff   = 0
  out_max_diff = 0

layer 42:
  x_max_diff        = 0
  topk_ids_diff     = 0
  topk_weights_diff = 0
  out_max_diff      = 0.16210938
  out_p999_diff     = 0.109375
  out_mean_diff     ≈ 0.02455
  out_cos           ≈ 0.99939
```

这说明什么？

第一，前 42 层没有分叉。

不是“前面 Marlin 慢慢污染，最后一层兜底”。

第二，layer 42 的输入完全一样。

不是路由错了，不是专家编号错了，不是输入已经不同了。

第三，唯一变量就是最后一层 MoE 的 backend。

同一份输入，CDG 算出一种输出，Marlin 算出另一种输出。差异看起来不大，最大也就 0.16，但它发生在最后一个 decoder layer，马上就要进 `lm_head` 变成 logits。

Transformer 最后一步对这种差异非常敏感。

因为生成模型不是做回归题，输出不是“差一点也行”。它最后要在 vocabulary 上选下一个 token。两个 logits 只要差一点，argmax 就可能换。第一个 token 换了，后面的上下文就全换，乱码循环就开始了。

这也是为什么你看张量余弦相似度 `0.99939`，觉得“这不是挺接近吗”；但看生成结果，就是“正常英文”和“符号循环”的差别。

语言模型的采样边界就是这么薄。

━━━━━━━━━━━━━━━━━━━━

◆ 先排除一个很像真凶的假真凶：expert parallel

━━━━━━━━━━━━━━━━━━━━

中间有个坑，值得单独写。

DeepSeek 官方推理代码里，MoE 专家是按 expert parallel 切的：

- rank0 管 0-127 号专家；
- rank1 管 128-255 号专家；
- token 路由到谁，只有负责那个专家的 rank 计算；
- 最后 all-reduce 合并。

所以我们第一反应是：会不会 Marlin 的 `expert_map` 没传对？比如 rank0 错误地把 178 号专家也算了，或者没有跳过远端专家。

这看起来非常可疑。

因为真实 dump 里 rank0 的 `topk_ids` 里确实有很多大于 127 的专家：

```text
[178, 15, 14, 162, 137, 91]
[184, 135, 25, 185, 11, 126]
...
```

如果 rank0 只该算 0-127，那这些专家都应该跳过。

但查 vLLM 启动参数后发现，我们这次服务没有开：

```bash
--enable-expert-parallel
```

实际运行方式不是 expert parallel，而是 tensor parallel：

- TP=2；
- 每个 rank 都有 256 个专家；
- 但每个专家的 intermediate 维度切成两半；
- rank0 算前半片；
- rank1 算后半片；
- 两边输出 shape 都是 `(tokens, 4096)`；
- 最后 all-reduce 相加。

这点很关键。

如果你按 expert parallel 去复现，会错得很离谱。

我们写了离线 replay 脚本验证三种语义：

1. EP rank0：只加载 0-127 号专家。
2. 全专家 full FFN：加载 256 专家，不切 intermediate。
3. TP rank0：加载 256 专家，但只取 intermediate 前半片。

只有第 3 种能完全复现 all-marlin dump：

```text
marlin rank0 dump vs marlin_replay_tp0:
  max  = 0
  mean = 0
```

这说明 dump 没错，权重没错，TP 切片也没错。

Marlin 在服务里到底算了什么，我们已经能离线 0 误差复现。

到这里，问题不再是“是不是框架调错了”。

问题变成：

**Marlin 和 CDG 到底是不是在算同一个数学对象？**

答案：不是。

━━━━━━━━━━━━━━━━━━━━

◆ 关键反转：Marlin 更接近高精度公式

━━━━━━━━━━━━━━━━━━━━

我们做了一个高精度 reference。

具体做法：

1. 从 safetensors 里读 layer42 的 FP4 权重和 E8M0 scale。
2. 手写解量化：

```text
FP4 weight -> float32 weight
scale      -> float32 power-of-two scale
```

3. 按 TP rank0 切片：

```text
w1/w3: 取 intermediate 前半片
w2:    取对应列片
```

4. 按 DeepSeek V4 的专家公式算：

```python
gate = x @ w1.T
up   = x @ w3.T

gate = clamp(gate, max=10.0)
up   = clamp(up, min=-10.0, max=10.0)

h = silu(gate) * up
out = h @ w2.T
```

再乘上 topk 权重，把 6 个专家加起来。

这就是 W4A16 语义：权重是 FP4，激活是 BF16/FP32 算。

然后分别对比 CDG dump 和 Marlin dump：

```text
Marlin vs ref_tp0:
  max  = 0.03125
  mean = 0.00259
  cos  = 0.999990

CDG vs ref_tp0:
  max  = 0.1660
  mean = 0.02441
  cos  = 0.999395
```

反直觉的地方来了：

**Marlin 更接近高精度公式。**

如果你把“正确”定义成“尽量接近 BF16 激活的高精度公式”，那 Marlin 是更正确的。

但现实输出刚好相反：

- Marlin 乱码；
- CDG 正常。

这说明我们之前把“正确”定义错了。

DeepSeek V4 Flash 这个模型部署时真正期待的，不是 W4A16 语义。

它期待的是 W4A8。

━━━━━━━━━━━━━━━━━━━━

◆ W4A16 和 W4A8 差在哪？

━━━━━━━━━━━━━━━━━━━━

先把术语拆开。

W4A16：

```text
Weight 4-bit
Activation 16-bit
```

也就是权重用 FP4，激活用 BF16。

W4A8：

```text
Weight 4-bit
Activation 8-bit
```

权重还是 FP4，但激活先量化成 FP8，再做矩阵乘法。

这不是“精度高一点/低一点”这么简单。

它改变的是模型实际运行时的函数。

假设一层 MoE 的第一段矩阵乘法是：

```text
y = x @ W
```

W4A16 算的是：

```text
y = bf16(x) @ fp4(W)
```

W4A8 算的是：

```text
x8 = quant_fp8_per_token_group(x, block=128)
y  = x8 @ fp4(W)
```

中间多了一步：

```text
x -> FP8 block quantization
```

DeepGEMM FP4 backend 代码里写得很直白：

```text
Uses FP8 activations and MXFP4 weights
```

Oracle 里也写了：

```text
DeepGEMM FP4 uses FP8 per-token-group activation quantization
with block 128
```

并且不是只有第一段 FC1 这样。

SwiGLU 之后，第二段 FC2 的输入也会再量化成 FP8：

```text
FC1:  FP8 activation × FP4 weight
SwiGLU
FC2:  FP8 activation × FP4 weight
```

这就是 CDG/DeepGEMM 的运行语义。

而 Marlin 这条路径，本质是：

```text
BF16 activation × FP4 weight
```

也就是 W4A16。

所以 Marlin 更接近“高精度公式”一点都不奇怪。

它本来就在少做一次激活量化。

问题是：模型最终能不能正确输出，不取决于你是不是更接近一个理想公式，而取决于你是不是复现了模型被量化、校准、部署时的运行语义。

这就是这次 bug 最反直觉的地方：

**不是 Marlin 算错了。**

**是 Marlin 算得太像 W4A16 了。**

但这个模型在这条部署路径上需要的是 W4A8。

━━━━━━━━━━━━━━━━━━━━

◆ 为什么前 42 层没事，最后一层有事？

━━━━━━━━━━━━━━━━━━━━

这里不要再回到“误差累积”的旧解释。

新的事实是：

```text
0-41 层 mixed/all-marlin 完全一致
```

因为 0-41 两边都用 Marlin。

它们当然一致。

这不是说 0-41 层的 Marlin 语义一定和 CDG 完全等价，而是说在这次 mixed/all-marlin 对照里，生成结果分叉不是从 0-41 开始的。真正改变路径的那个变量，只出现在 layer42。

真正的问题是：为什么 layer42 一个 0.16 级别的输出差异，就能把生成带崩？

原因是 layer42 是最后一个 decoder layer。

Transformer 的最后流程大概是：

```text
hidden state
  -> 最后一层 decoder
  -> final norm
  -> lm_head
  -> logits
  -> 选下一个 token
```

前面的层即使有小偏差，后面还有几十层的非线性变换、残差叠加、归一化在继续处理它。

最后一层没有这个缓冲。

它的输出直接决定 logits。

如果某两个候选 token 的 logits 很接近：

```text
token A: 12.031
token B: 12.028
```

那 hidden state 里 0.1 级别的某些通道变化，完全可能把 A/B 顺序翻过来。

一旦第一个 token 翻错，自回归链条就变了：

```text
正确路径：Write -> a -> function -> ...
错误路径：</ -> _ -> , -> </ -> ...
```

后面每一步都基于前一步的输出继续采样。错一个 token 不是“错一个字”，而是换了一条时间线。

这就是为什么张量差异看起来“小”，输出差异却是“正常英文”和“符号循环”。

LLM 的最后一层不是普通中间层。

它是高维语义空间坍缩到 token 的最后一道门。

━━━━━━━━━━━━━━━━━━━━

◆ 185 期错在哪，对在哪？

━━━━━━━━━━━━━━━━━━━━

185 期的工程结论是对的：

```text
0-41 Marlin + 42 CDG
```

这仍然是当前最小正确规避。

185 期的机制解释错了一半。

当时说：

> 前 42 层 Marlin 误差累积，最后一层 CDG 兜住。

现在修正为：

> 前 42 层在 mixed 和 all-marlin 里都是 Marlin，所以两边完全一致；真正的差异只发生在 layer42。layer42 用 CDG 时走 W4A8 语义，用 Marlin 时走 W4A16 语义。最终层对 W4A8/W4A16 的语义漂移极敏感，所以全 Marlin 乱码。

这两个解释外部行为一样：

- 只换最后一层就好；
- 全 Marlin 就坏；
- 混合方案速度接近纯 Marlin。

但内部机制完全不同。

旧解释是“Marlin 有小错误，最后一层别再错了”。

新解释是“Marlin 算的是另一种合法语义，但不是这个模型最后一层需要的语义”。

这差别很大。

因为如果旧解释成立，修法是继续追 Marlin 的数值 bug、ldmatrix、repack、共享内存布局。

如果新解释成立，修法是：

**让 Marlin 支持 W4A8 activation quantization 语义。**

也就是：

```text
FC1 输入：FP8 per-token-group quantization, block=128
SwiGLU 后：再 FP8 per-token-group quantization, block=128
FC2 输入：用 FP8 activation
```

这不是修一个 `ldmatrix` 就能解决的。

这是 kernel 语义层面的差异。

━━━━━━━━━━━━━━━━━━━━

◆ 为什么这件事对工程很重要？

━━━━━━━━━━━━━━━━━━━━

很多人排量化模型 bug，会默认一个框架：

```text
高精度 reference = 正确
谁离 reference 近 = 谁正确
```

这在普通场景里大体没错。

但对量化模型，尤其是经过特定部署路径校准的模型，这个框架会误导你。

因为模型实际学到/适配的不是一个抽象数学公式，而是一条具体的数值管线。

这个管线包括：

- 权重量化格式；
- scale 格式；
- activation quantization；
- block size；
- clamp limit；
- TP 切片方式；
- reduction 顺序；
- kernel 内部 accumulation 精度；
- 最后输出 cast。

这些都属于模型运行时函数的一部分。

你把其中一个环节“提高精度”，不一定更正确。

就像训练时一直给一个人戴着某种颜色滤镜看世界，他已经按那个滤镜校准了判断。你突然把滤镜摘掉，物理上更真实，但他的判断系统反而失准。

这次 Marlin 就是这样。

它不是低级 bug。

它是另一套数值世界。

在前面 42 层，这套世界还能过。

到最后一层，过不了了。

━━━━━━━━━━━━━━━━━━━━

◆ 当前工程结论

━━━━━━━━━━━━━━━━━━━━

现在的最小可用方案不变：

```bash
VLLM_MXFP4_MARLIN_LAYER_RANGE=0:42
```

或者更明确：

```bash
VLLM_MXFP4_MARLIN_DEEPGEMM_LAYERS=42
```

含义：

- 0-41 层走 Marlin；
- layer42 走 CDG/DeepGEMM；
- 保持 12 tok/s 左右；
- 中英文正确。

如果要做真正的全 Marlin 正确，不是继续盯着“Marlin 为什么不贴 BF16 reference”。

它已经很贴了。

要做的是：

```text
Marlin W4A8
```

也就是给 Marlin 加上和 DeepGEMM FP4 backend 一致的 FP8 activation quantization：

1. FC1 前，对 `x` 做 FP8 per-token-group quantization，block=128。
2. FC1 后做 clamp + SwiGLU。
3. SwiGLU 输出再做 FP8 per-token-group quantization，block=128。
4. FC2 用 FP8 activation × FP4 weight。
5. 保持 TP 半片语义和最终 all-reduce 不变。

这才是完整对齐 CDG 的方向。

否则纯 W4A16 Marlin 就算写得再精确，也可能继续在最后一层翻车。

◆ 复现实验文件

━━━━━━━━━━━━━━━━━━━━

这次新增了一个离线回放脚本：

```text
test/replay_layer42_ep_ref.py
```

它做三件事：

1. 读取 mixed 和 all-marlin 的 layer42 dump。
2. 从 safetensors 读真实 layer42 专家权重，按 TP rank0 切片。
3. 同时比较：
   - CDG dump；
   - Marlin dump；
   - 离线 Marlin replay；
   - 高精度 TP0 reference；
   - 错误的 EP/全 FFN reference。

关键输出：

```text
marlin rank0 dump vs marlin_replay_tp0:
  max=0
  mean=0

mixed(CDG rank0 dump) vs ref_tp0:
  max=0.16601562
  mean=0.02440922

marlin rank0 dump vs ref_tp0:
  max=0.03125000
  mean=0.00258975
```

这个输出把问题钉死了：

- Marlin dump 可以被离线 Marlin 零误差复现；
- TP 切片语义正确；
- Marlin 更接近 W4A16 reference；
- CDG 和 Marlin 的差异来自 W4A8/W4A16 语义，而不是普通 kernel bug。

◆ 一句话总结

**这次 bug 的最终形状不是“Marlin 算错了”，而是“Marlin 算的是 W4A16，DeepGEMM/CDG 算的是 W4A8；DeepSeek V4 Flash 的最后一层需要 W4A8 语义”。**

185 期的混合方案仍然对，但理由要改。

最后一层 CDG 不是在“纠正前 42 层 Marlin 的错误”，而是在把最后一次 MoE 拉回模型实际部署时的量化语义。

这就是为什么只换一层就能从乱码变正常。

有时候工程排障最难的不是找到“哪里错了”，而是承认：

你以为的正确参考，本来就不是系统真正运行的参考。
