【DeepSeek V4 推理】W4A16 vs W4A8，Marlin 不是算错了，而是算得太“正确”

◆ 前情回顾

━━━━━━━━━━━━━━━━━━━━

185 期 https://mp.weixin.qq.com/s/dEc-wD0JcM5WhKicOUR1pQ （那篇“sm_12x 问题修复，从 4 到 12 tok/s”）讲过一个现象：

DGX Spark 上跑 DeepSeek V4 Flash 280B，纯 Consumer-DeepGEMM（以下简称 CDG）能正确输出，但速度只有 4 tok/s 左右；纯 Marlin 能跑到 14 tok/s 左右，但英文输出乱码；最后我们用了一个混合方案：

```bash
VLLM_MXFP4_MARLIN_LAYER_RANGE=0:42
```

也就是：

- decoder layer 0-41 走 Marlin；
- decoder layer 42 走 CDG；
- 最终速度 12 tok/s 左右；
- 中英文输出正常。

当时的解释是：Marlin 在 sm_121 上每层有微小误差，前 42 层还没过临界点，最后一层如果继续用 Marlin，就把误差推过临界点；换成 CDG 就能兜住。

这个解释能解释现象，但它不是最终真相。

后来我们继续抓真实激活、做 mixed/all-marlin 双配置对比，发现 185 期的判断里有一个关键假设错了：

**前 42 层并没有出现 mixed 和 all-marlin 的分叉。**

也就是说，不存在“前 42 层 Marlin 一点点污染 residual stream，最后一层压垮骆驼”的证据。

真正的分叉点非常干净：

**layer 42。**

最后一层。

同一份输入，同一份路由，同一份专家权重，只是 kernel 不同：

- CDG 输出正常；
- Marlin 输出乱码。

而且更反直觉的是：

**按高精度公式算，Marlin 更接近参考答案；CDG 反而更远。**

这期就讲这个反转。

◆ 先把假信号清掉

━━━━━━━━━━━━━━━━━━━━

最早那次 dump 里，我们看到 layer 42 的输出 `absmax=9344`，比正常层大几百倍。当时第一反应很自然：最后一层爆了。

────────────────────

💡 我自己容易搞混的问题

**absmax** = absolute max，张量里所有元素绝对值的最大值，排障时常用来一眼看"这层有没有飞起来"。正常 transformer 中间层的 hidden state（BF16 存，LayerNorm 反复把它拉回 O(1) 量级）absmax 通常在个位数到几十。9344 不是 BF16 装不下——BF16 能装到 3.4e38——而是远远超出训练好的模型该有的统计分布，通常等于这层炸了。

────────────────────

后来复查才发现，这是 vLLM 启动时的 dummy batch——服务起来后、收到真实请求**之前**，vLLM 自己会先用一批假输入跑几遍模型，做 profiling（估算 KV cache 能开多大）和 warmup（暖 CUDA kernel，避免第一个真实请求慢）。这些假 batch 的特征是 batch 里所有行完全重复、统计分布跟训练数据完全不同，所以即使输出爆炸，也不能说明真实推理有问题。

我们的 dump hook 是挂在 MoE forward 函数上的，每次被调用就 dump 一份。结果前几次 dump 抓到的全是 vLLM 自己造的假输入：

| 第 N 次调用 | 来源 | 输入特征 |
|---|---|---|
| 第 1-3 次 | profiling / warmup | shape 大、所有行完全重复 |
| 第 4 次起 | 真实用户 prompt | 14 行各不相同 |

后来在 dump hook 里加了：

```bash
VLLM_MXFP4_DUMP_SKIP=3
```

跳过前 3 次调用，只从真实请求开始抓。

────────────────────

💡 我自己容易搞混的问题

这几次 dummy batch 跑出来的"输出"留不留在上下文里？**不留。**

KV cache 在 profiling 跑完就被清空、激活/hidden state 每次 forward 用完就释放、模型权重本来就是只读的。真实请求来的时候是从干净的 KV cache 开始写。

**那这几次跑是干嘛使的？** vLLM 在拿假数据"试探"环境，留下来的不是数据本身，是三件副作用：

1. **KV cache 容量**——vLLM 用一个最大可能的假 batch 跑一遍 forward，测显存峰值，剩下的（× `--gpu-memory-utilization` 比如 0.80）全分给 KV cache。**这是 profiling 的真正目的**。后续所有"能并发多少 sequence、context 能开多长"都基于这次测量。
2. **CUDA graph 录制**——decode forward 路径会被录下来当模板，后续真实推理 replay 模板省 launch 开销（224 期启动用了 `--enforce-eager` 关了这条，所以实际可能没录，但默认是有的）。
3. **kernel JIT / 暖机**——Triton kernel（包括 CDG 用的 `tl.dot_scaled`）、cuBLAS handle、CUDA 内存池，第一次调用都有一次性开销，warmup 让它们提前付完，真实请求第一个 token 不至于卡几秒。

简单说：**vLLM 自己不会去看那些 dump 出来的 hidden state 数值**（它只关心显存峰值和 kernel 调用链），是我们额外挂的 hook 把这些"试探阶段的中间值"偷出来了。

所以 `VLLM_MXFP4_DUMP_SKIP=3` 不是关 vLLM 的 profiling/warmup（那是关不掉的、也不该关——少了它真实请求会慢、KV cache 大小算不出来），而是告诉**我们的探针**：前 3 次别记录。

────────────────────

真实 prompt 是：

```text
Write a quicksort function in Python and explain how it works.
```

真实 prefill 的 shape 是：

```text
x.shape = (14, 4096)
```

14 行互不相同，符合 prompt token 数。

这时 layer 42 的数值不再爆炸：

```text
x_absmax   ≈ 2.55
out_absmax ≈ 8.8
```

所以“layer 42 数值爆炸”这条线撤回。

工程排障里很容易犯这种错误：你抓到了一个很夸张的数，就以为它是真凶。其实它可能只是启动阶段的假输入。模型服务框架会做 warmup、profiling、CUDA graph 捕获，这些都会制造看起来很像真实推理的张量。

所以后面所有结论都只基于跳过 warmup 之后的真实 dump。

◆ 真正的分叉：0-41 完全一致，42 才分叉

━━━━━━━━━━━━━━━━━━━━

我们重跑了两次服务。

第一次 mixed：

```bash
VLLM_MXFP4_MARLIN_LAYER_RANGE=0:42
```

含义：0-41 Marlin，42 CDG。

第二次 all-marlin：

```bash
VLLM_MXFP4_MARLIN_LAYER_RANGE=0:43
```

含义：0-42 全 Marlin。

同一个 prompt，温度 0，抓每一层 MoE 的：

- 输入 `x`
- 路由专家编号 `topk_ids`
- 路由权重 `topk_weights`
- 输出 `out`
- 实际后端类型：`MarlinExperts` 还是 `DeepGemmFP4Experts`

────────────────────

💡 我自己容易搞混的问题

这两个 Experts 类不是 vLLM 自己造的——是 vLLM 把两个外部 kernel 库各自包了一层：

- `MarlinExperts` ← **Marlin**，IST-DASLab 那批 GPTQ 作者写的 W4 高速 GEMM 项目，2024 年初为 NVIDIA Ampere/Hopper 优化的
- `DeepGemmFP4Experts` ← **DeepGEMM**，DeepSeek 团队自家的开源 FP8/FP4 GEMM 库（名字 = Deep(Seek) + GEMM），他们 V3/R1 推理在用

注意我们 sm_121 上用的不是上游原版 DeepGEMM（那个针对 H100/H800 用 PTX 写得很极致，跑不了消费级 GB10），而是 **Consumer-DeepGEMM**——这个项目里把 DeepGEMM 的 FP4 MoE 思路用 Triton `tl.dot_scaled` 在 sm_120+ 上重写了一遍，vLLM 调进来还是同一个 `DeepGemmFP4Experts` 入口。

所以 dump 里看到这两个名字 = 这一层 MoE 跑的是哪个世界的 kernel。这次的反转就是这两个世界**不只是 kernel 不同，是算的数学对象不同**——后面 W4A16 vs W4A8 那段就讲这个。

────────────────────

然后逐层比较 mixed 和 all-marlin。

结果非常干净：

```text
layer 0-41:
  x_max_diff   = 0
  out_max_diff = 0

layer 42:
  x_max_diff        = 0
  topk_ids_diff     = 0
  topk_weights_diff = 0
  out_max_diff      = 0.16210938
  out_p999_diff     = 0.109375
  out_mean_diff     ≈ 0.02455
  out_cos           ≈ 0.99939
```

────────────────────

💡 我自己容易搞混的问题

这些数字怎么算的？大部分是**两个张量逐元素相减**之后再归约，最后那个 `cos` 不是：

| 指标 | 这是啥的 diff | 怎么算 | 看什么 |
|---|---|---|---|
| `x_max_diff` | **输入激活** `x`（这层 MoE 的输入 hidden state） | `(x_mixed - x_all_marlin).abs().max()` | 进 layer 42 之前两个配置一不一样。**`=0` 是关键锚点**——证明唯一变量就是这一层的 kernel |
| `topk_ids_diff` | **路由专家编号**（这 token 选了哪 6 个 expert） | 同上 | 路由有没有跑歪。`=0` 说明两边选的 expert 完全相同 |
| `topk_weights_diff` | **路由权重**（6 个 expert 各占多少比例） | 同上 | 路由权重有没有偏。`=0` 说明权重也一模一样 |
| `out_max_diff` | **输出** `out` | `(out_mixed - out_all_marlin).abs().max()` | 最离谱那一个元素差多少（最坏情况） |
| `out_p999_diff` | 同上 | `(out_mixed - out_all_marlin).abs()` 排序后取 99.9% 分位 | 把最极端的 0.1% 丢掉，剩下绝大多数里最差的——能区分"只有几个孤点离谱"还是"普遍都差不少" |
| `out_mean_diff` | 同上 | `(out_mixed - out_all_marlin).abs().mean()` | 平均偏差，整体偏移的体量 |
| `out_cos` | 同上 | `cos(out_mixed, out_all_marlin)` ——**不相减**，把两个张量各自展平成一维向量算夹角余弦 | 方向相似度，`1.0` = 同向，`0` = 正交。看的是**方向**像不像，跟 magnitude 无关 |

**前三个都是 0 是这次实验最重要的事**——它锁死了"输入端完全一致、路由也完全一致"，所以 layer 42 输出的 0.16 差异只能扣在这一层 kernel 的语义差异上，不是别的。

为什么 output 要 4 个一起看？

- 只看 `max` → 容易被一个 outlier 误导
- 只看 `mean` → 平均下来很小，但分布是不是均匀差不知道
- 只看 `cos` → 方向相似但 magnitude 可能整体偏（比如一个向量被放大 2 倍，cos 仍然是 1.0）

这次结果 `max=0.16`、`p999=0.11`、`mean=0.025`、`cos=0.999`——magnitude 接近、方向也接近，**没有 outlier、没有局部 spike**。这种"整体微小偏移"的指纹，是"两套数值语义在算同一件事但路径不同"的典型形状，不是"某条 expert 算坏了"那种局部错误。

但即使统计上这么接近，过完 `lm_head` 还是把 token 翻了——后面 §"为什么前 42 层没事，最后一层有事？" 会再展开这点。

────────────────────

这一组数字钉死了一件事：**唯一变量就是 layer 42 的 backend**。前 42 层 mixed 和 all-marlin 完全一致（两边都跑 Marlin，确定性 0 差异），layer 42 的输入、路由、专家编号也一模一样——只是 kernel 不同，CDG 跑出来正常，Marlin 跑出来乱码。

那 layer 42 的 kernel 到底差在哪？这就要先排除一个看起来很像真凶的怀疑对象。

◆ 先排除一个很像真凶的假真凶：expert parallel

━━━━━━━━━━━━━━━━━━━━

中间有个坑，值得单独写。

────────────────────

💡 我自己容易搞混的问题

**专家并行（Expert Parallel, EP）vs 张量并行（Tensor Parallel, TP）——同样是双机跑 MoE，两种切法**

DeepSeek V4 的每一层 MoE 有 256 个 expert，每个 token 路由到 top-6 个 expert。问题是 256 个 expert 的权重一台机放不下，必须两台 Spark 分着扛。怎么分？两种思路：

**EP（专家并行）：按"哪个 expert 归谁"切。**

- rank0 管 0-127 号 expert 的**完整权重**
- rank1 管 128-255 号 expert 的**完整权重**
- 每个 token 路由出 top-6，假设 4 个落在 0-127、2 个落在 128-255
- rank0 算自己那 4 个、rank1 算自己那 2 个，两边都不重复劳动
- 最后跨机 all-reduce 把 6 个结果加起来

```text
rank0:  expert 0   expert 1   ...  expert 127   ← 完整权重
rank1:  expert 128 expert 129 ...  expert 255   ← 完整权重
```

**TP（张量并行）：每个 expert 都被两台切一半，按 intermediate 维度切。**

- rank0 和 rank1 都加载**全部 256 个 expert**
- 但每个 expert 内部的 FFN 中间维度 `intermediate=2048` 被劈成两半，rank0 拿前 1024，rank1 拿后 1024
- 每个 token 路由的 top-6 expert，两台都各自算自己那一半
- 最后 all-reduce 把两半"竖着"加起来拼成完整输出

```text
rank0:  expert 0[:1024]   expert 1[:1024]   ...  expert 255[:1024]   ← 半片权重
rank1:  expert 0[1024:]   expert 1[1024:]   ...  expert 255[1024:]   ← 半片权重
```

**关键区别**：

| | EP | TP |
|---|---|---|
| 每 rank 加载的 expert 数 | 一半（128 个） | 全部（256 个） |
| 每个 expert 权重是否完整 | 完整 | 砍半 |
| `expert_map` 是否生效 | 是，要跳过远端 expert | 否，每 rank 都算所有 expert |
| 跨机通信时机 | 路由前 + 算完后 all-reduce | 只在算完后 all-reduce |
| 权重存储总量 | 1 份 = 全模型权重 | 2 份 = 全模型权重 × 2 |

注意 TP 的总存储是 EP 的两倍（每个 rank 都要装全套 expert，只是每个砍半）——但每个 expert 的计算被两台分了，对延迟有利。EP 省内存但通信路径更复杂。

DeepSeek 官方代码用 EP（他们集群规模大、内存紧），但 vLLM 在 DGX Spark 上默认用 TP（消费级双机，简单）。**这两种语义的代码长得像、变量名也像，但实际算的不是一回事**——我们一开始就栽在这里。

────────────────────

DeepSeek 官方推理代码用的就是 EP。我们的第一反应是：会不会 vLLM 这边也是 EP，但 Marlin 的 `expert_map` 没传对？比如 rank0 错误地把 178 号专家也算了，或者没有跳过远端专家。

这看起来非常可疑。

因为真实 dump 里 rank0 的 `topk_ids` 里确实有很多大于 127 的专家：

```text
[178, 15, 14, 162, 137, 91]
[184, 135, 25, 185, 11, 126]
...
```

如果 rank0 只该算 0-127，那这些专家都应该跳过。

但查 vLLM 启动参数后发现，我们这次服务没有开：

```bash
--enable-expert-parallel
```

实际运行方式不是 expert parallel，而是 tensor parallel：

- TP=2；
- 每个 rank 都有 256 个专家；
- 但每个专家的 intermediate 维度切成两半；
- rank0 算前半片；
- rank1 算后半片；
- 两边输出 shape 都是 `(tokens, 4096)`；
- 最后 all-reduce 相加。

这点很关键。

如果你按 expert parallel 去复现，会错得很离谱。

我们写了离线 replay 脚本验证三种语义：

1. EP rank0：只加载 0-127 号专家。
2. 全专家 full FFN：加载 256 专家，不切 intermediate。
3. TP rank0：加载 256 专家，但只取 intermediate 前半片。

只有第 3 种能完全复现 all-marlin dump：

```text
marlin rank0 dump vs marlin_replay_tp0:
  max  = 0
  mean = 0
```

这说明 dump 没错，权重没错，TP 切片也没错。

Marlin 在服务里到底算了什么，我们已经能离线 0 误差复现。

到这里，问题不再是“是不是框架调错了”。

问题变成：

**Marlin 和 CDG 到底是不是在算同一个数学对象？**

答案：不是。

◆ 关键反转：Marlin 更接近高精度公式

━━━━━━━━━━━━━━━━━━━━

我们做了一个高精度 reference。

具体做法：

1. 从 safetensors 里读 layer42 的 FP4 权重和 E8M0 scale。
2. 手写解量化：

```text
FP4 weight -> float32 weight
scale      -> float32 power-of-two scale
```

3. 按 TP rank0 切片：

```text
w1/w3: 取 intermediate 前半片
w2:    取对应列片
```

────────────────────

💡 我自己容易搞混的问题

**为什么这里有 3 个矩阵 `w1/w2/w3`？经典 FFN 不是 2 个矩阵够了吗？**

经典 Transformer 的 FFN 确实是两个矩阵：

```python
# 经典 FFN（GPT/BERT 时代）
h = relu(x @ W_in)    # 升维
out = h @ W_out       # 降维
```

从 LLaMA 开始（后来很多 MoE 包括 DeepSeek 都跟了），FFN 换成了 **SwiGLU**——升维这一步变成两条并行路径，一条算"内容"（up）、一条算"门控"（gate），相乘之后再降维：

```python
# SwiGLU FFN（LLaMA / DeepSeek 时代）
gate = x @ w1   # 升维路径 1：算门控
up   = x @ w3   # 升维路径 2：算内容
h    = silu(gate) * up   # 门控 × 内容（element-wise）
out  = h @ w2   # 降维
```

所以是 3 个矩阵：

| 矩阵 | 形状 | 作用 |
|---|---|---|
| `w1`（gate_proj） | `(intermediate, hidden)` | 升维，算门控 |
| `w3`（up_proj） | `(intermediate, hidden)` | 升维，算内容 |
| `w2`（down_proj） | `(hidden, intermediate)` | 降维 |

为什么从 2 个变 3 个还嫌不够？因为 SwiGLU 的"门控 × 内容"结构表达能力比 ReLU 强很多——`gate` 决定每个中间通道开多大、`up` 决定塞什么内容进去。LLaMA 论文实测下来同等参数下性能更好，几乎所有后续模型都跟了。

**TP 切片为什么 `w1/w3` 切 intermediate 前半片、`w2` 切列片？**

- `w1/w3` 的输出维度是 intermediate（2048），rank0 拿前 1024 → 它俩"行"切（**沿输出维度切**）
- `w2` 的输入维度是 intermediate（2048），rank0 拿前 1024 → 它"列"切（**沿输入维度切**）
- 两边算完各得 `(tokens, 4096)`，最后 all-reduce 加起来 = 完整结果

这是 Megatron 的经典 column-row 切法：升维矩阵按输出切、降维矩阵按输入切，中间不用通信、最后只 all-reduce 一次。

────────────────────

4. 按 DeepSeek V4 的专家公式算：

```python
gate = x @ w1.T
up   = x @ w3.T

gate = clamp(gate, max=10.0)
up   = clamp(up, min=-10.0, max=10.0)

h = silu(gate) * up
out = h @ w2.T
```

再乘上 topk 权重，把 6 个专家加起来。

这就是 W4A16 语义：权重是 FP4，激活是 BF16/FP32 算。

把这个公式跑出来的结果记作 `ref_tp0`——`ref` 是 reference（高精度参考），`_tp0` 是 TP rank 0 切片（rank0 那台 Spark 看到的那半片输出）。

简单回顾一下它怎么算出来的：从 safetensors 里读 layer 42 的 **FP4 权重** + **E8M0 scale** → 手写解量化成 **float32 权重**（这就是"高精度"的来源，把权重从 4-bit 完全还原成 32-bit 浮点）→ 按 TP rank0 取半片 → 直接用 BF16 输入 `x` 跑 SwiGLU 公式 → 乘 topk 权重并把 6 个专家加起来。中间没有任何 FP8 量化、没有任何 kernel 黑盒，就是张量运算硬算出来。**这是 W4A16 的理论结果，CDG 和 Marlin 都该跟它比**。

然后分别拿真实服务里 dump 出来的 CDG 输出和 Marlin 输出去跟它比：

```text
Marlin vs ref_tp0:
  max  = 0.03125
  mean = 0.00259
  cos  = 0.999990

CDG vs ref_tp0:
  max  = 0.1660
  mean = 0.02441
  cos  = 0.999395
```

反直觉的地方来了：

**Marlin 更接近高精度公式。**

如果你把“正确”定义成“尽量接近 BF16 激活的高精度公式”，那 Marlin 是更正确的。

但现实输出刚好相反：

- Marlin 乱码；
- CDG 正常。

这说明我们之前把“正确”定义错了。

DeepSeek V4 Flash 这个模型部署时真正期待的，不是 W4A16 语义。

它期待的是 W4A8。

━━━━━━━━━━━━━━━━━━━━

◆ W4A16 和 W4A8 差在哪？

━━━━━━━━━━━━━━━━━━━━

在拆术语之前，先说一个更底层的问题——为什么量化命名要分 W 和 A？

────────────────────

💡 我自己容易搞混的问题

**为什么矩阵乘法里一边叫"权重"、一边叫"激活"？两个矩阵不是平等的吗？**

先说一个更基础的：**机器学习的核心公式跟小学的 `y = ax + b` 是同一件事吗？**

几乎是。把 `a` 换成矩阵 `W`、把 `x` 换成向量、把 `b` 换成向量（现代 transformer 经常直接省掉）、把"乘"换成矩阵乘 `@`，就是机器学习的标准写法 `y = x @ W + b`。整个 transformer 剥光了看，就是这种线性变换嵌套几十次、中间夹一点非线性（SiLU、Softmax）。

命名约定上 `W` = weight（权重）、`b` = bias（偏置）——所谓"权重"和"激活"的名字就是这么来的：`W` 是权重（那个训练好就冻住的常量矩阵），`x` 是这一层的输入（从上一层流过来的激活值）。

回到原问题：数学上 `y = x @ W` 这种矩阵乘法（顺便一提：`@` 是 Python 3.5 起的矩阵乘法运算符，比 `np.matmul(x, W)` 短得多，现在 PyTorch/NumPy 代码里随处可见；另外深度学习代码里 `x` 通常是**行向量**——`x.shape = (batch, features)`，每行一个 token，跟数学教材习惯的列向量 `y = W @ x` 反过来），`x` 和 `W` 看起来是对称的两个矩阵，没区别。但在 transformer 推理里，它俩的**地位完全不同**：

| | 权重 W | 激活 x |
|---|---|---|
| 何时确定 | 训练完就**冻住**，部署后整个生命周期不变 | 每次请求**重新算出来** |
| 大小 | 巨大、固定（DeepSeek V4 280B 就是这个） | 小、随 batch / seq_len 变 |
| 存哪里 | 显存里常驻 | 每次 forward 临时分配 |
| 量化时机 | **离线**——训练后转一次格式，存盘 | **在线**——每次 forward 跑的时候量化 |
| 我能不能精挑细选量化方法 | 能。可以慢慢校准、看分布、挑 scale | 不行。来不及，量化必须很快 |

所以"权重"和"激活"虽然在矩阵乘法里平等参与计算，但**工程上完全不是一回事**。权重是**常量**，激活是**变量**。

这就是为什么量化策略要分开标 W 和 A：

- **W4** = 把训练完的权重压成 4-bit，**离线**做、可以慢慢挑、一次定终身
- **A8 / A16** = 每次 forward 时激活用 8-bit 还是 16-bit，**在线**量化、要快

这次的核心反转就来自这里——Marlin 走的 W4A16 是"权重压 4-bit、激活保持 BF16 不动"，CDG 走的 W4A8 是"权重压 4-bit、激活每次还要额外压成 FP8 再算"。**这不是精度高低的差别，是激活这条变量管线多了一道在线量化工序**。

模型量化训练/校准的时候，跑的是哪条管线，部署就只能跑哪条。Marlin 把那道在线工序省了，所以在数值上反而"更接近高精度参考"，但模型不认这条路径。

────────────────────

先把术语拆开。

W4A16：

```text
Weight 4-bit
Activation 16-bit
```

也就是权重用 FP4，激活用 BF16。

W4A8：

```text
Weight 4-bit
Activation 8-bit
```

权重还是 FP4，但激活先量化成 FP8，再做矩阵乘法。

这不是“精度高一点/低一点”这么简单。

它改变的是模型实际运行时的函数。

假设一层 MoE 的第一段矩阵乘法是：

```text
y = x @ W
```

W4A16 算的是：

```text
y = bf16(x) @ fp4(W)
```

W4A8 算的是：

```text
x8 = quant_fp8_per_token_group(x, block=128)
y  = x8 @ fp4(W)
```

中间多了一步：

```text
x -> FP8 block quantization
```

DeepGEMM FP4 backend 代码里写得很直白：

```text
Uses FP8 activations and MXFP4 weights
```

vLLM 的量化配置生成器里也写了：

```text
DeepGEMM FP4 uses FP8 per-token-group activation quantization
with block 128
```

并且不是只有第一段 FC1（SwiGLU 之前的升维矩阵乘，对应前面 `gate = x @ w1` 和 `up = x @ w3`）这样。

SwiGLU 之后，第二段 FC2（降维矩阵乘 `out = h @ w2`）的输入也会再量化成 FP8：

```text
FC1:  FP8 activation × FP4 weight
SwiGLU
FC2:  FP8 activation × FP4 weight
```

这就是 CDG/DeepGEMM 的运行语义。

而 Marlin 这条路径，本质是：

```text
BF16 activation × FP4 weight
```

也就是 W4A16。

所以 Marlin 更接近“高精度公式”一点都不奇怪。

它本来就在少做一次激活量化。

问题是：模型最终能不能正确输出，不取决于你是不是更接近一个理想公式，而取决于你是不是复现了模型被量化、校准、部署时的运行语义。

这就是这次 bug 最反直觉的地方：

**不是 Marlin 算错了。**

**是 Marlin 算得太像 W4A16 了。**

但这个模型在这条部署路径上需要的是 W4A8。

────────────────────

💡 我自己容易搞混的问题

**这个"精度差"具体差在硬件的哪一步？**

要看懂这一点，先简单分一下 GPU 上两类计算单元的分工：

| 单元 | 干什么 |
|---|---|
| **Tensor Core** | 只会做矩阵乘加 `D = A @ B + C`。`A`、`B` 必须是低精度（FP16 / BF16 / FP8 / FP4 等），`C`、`D` 用 FP32 累加。是 GPU 旗舰算力的来源——H100 / Blackwell 上几乎所有标榜的 PFLOPS 都是它的算力 |
| **CUDA Core** | 通用 ALU。干 Tensor Core 干不了的活：非线性（SiLU / Softmax）、LayerNorm、逐元素加 `+ b`、量化 / 反量化、路由 / topk……单条吞吐量比 Tensor Core 低一两个数量级 |

所以现代 LLM 推理 kernel 都是 fused 设计：**Tensor Core 算主干矩阵乘，CUDA Core 在寄存器里接力做"+b、SiLU、量化"这些尾巴活，数据全程不下显存**。性能差异不在"算得多快"，在"内存搬运省了多少"。

回到 Marlin vs CDG：

| 步骤 | Marlin（W4A16） | CDG（W4A8） | 硬件 |
|---|---|---|---|
| 把激活准备好 | 直接用 BF16 | **多一步：把 BF16 量化成 FP8（block=128）** | CUDA Core |
| 主干矩阵乘 | **Tensor Core 跑 BF16 × FP4** | **Tensor Core 跑 FP8 × FP4** | **Tensor Core** |
| SwiGLU、加路由权重 | CUDA Core 接力 | CUDA Core 接力 | CUDA Core |

**真正的差异就是 Tensor Core 这一步吃的输入精度不同**——Marlin 喂的是 BF16 激活，CDG 喂的是 FP8 激活。CUDA Core 那边的活两边几乎一样。

这就是为什么"让 Marlin 改成 W4A8 语义"不是修一行代码——你要在 GEMM 之前插入一道 FP8 量化（per-token-group block=128，CUDA Core 干），还要把 Tensor Core 的 MMA 指令从 BF16×FP4 切换成 FP8×FP4（不同的 PTX 指令、不同的寄存器布局），SwiGLU 之后还要再来一次 FP8 量化喂给下一个 GEMM。本质上是**换整条 Tensor Core 流水线的输入格式**。

━━━━━━━━━━━━━━━━━━━━

◆ 为什么前 42 层没事，最后一层有事？

━━━━━━━━━━━━━━━━━━━━

这里不要再回到“误差累积”的旧解释。

新的事实是：

```text
0-41 层 mixed/all-marlin 完全一致
```

因为 0-41 两边都用 Marlin。

它们当然一致。

这不是说 0-41 层的 Marlin 语义一定和 CDG 完全等价，而是说在这次 mixed/all-marlin 对照里，生成结果分叉不是从 0-41 开始的。真正改变路径的那个变量，只出现在 layer42。

────────────────────

💡 我自己容易搞混的问题

**前 41 层 `x_max_diff = 0`、`out_max_diff = 0`——这是不是说前 41 层 Marlin 完全没误差？**

**不是。** 这个误区很值得点透，因为我自己写到这都差点掉进去。

这次 dump 比较的是**两份 dump 之间**——mixed（0-41 Marlin + 42 CDG）和 all-marlin（0-42 全 Marlin）。**前 41 层两边都用 Marlin**，跑同一个 prompt、温度 0、同一份权重——结果当然完全一样。这是**确定性**的 0，不是"Marlin 算得准"的 0。

打个比方：同一个只有 4 位小数显示的计算器按两遍 `1 ÷ 3`，两遍都得 `0.3333`。这只证明这台计算器自己确定可重现，不能说明它跟"真正的答案"（0.3333333…）贴得多近——它每次都丢掉相同的尾巴而已。

那 Marlin 跟"高精度参考"（BF16 激活 × 反量化 FP4 权重的 fp32 公式）比，前 41 层有没有偏差？**有，而且每层都有，量级跟 layer 42 差不多**。文章里 §"关键反转"那节列了 layer 42 的数字：

```text
Marlin vs ref:  max=0.03125, mean=0.00259, cos=0.999990
```

这个量级的偏差**前 41 层每一层都在发生**，只是我们没去测——因为这次实验的目的不是"测每层 Marlin 有多准"，而是"找到 mixed/all-marlin 的第一个分叉点"。**没测出来 ≠ 没有，只是没列入比较**。

那为什么前 41 层有偏差还能跑对？因为**每一层 transformer block 都自带 RMSNorm + 残差 + Softmax + 非线性**——一层的小偏差进入**下一层**时，会被下一层的 RMSNorm 重新拉回 O(1) 量级，被 Softmax 的强非线性吞掉一部分，被残差路径稀释。**真正起"消化"作用的是后续 block 的这些操作**。

到了 layer 42 就麻烦了——layer 42 当然也有自己的 RMSNorm / 残差 / 非线性，但**它是最后一个 transformer block**。它输出的偏差，后面没有"下一个 block"来吸收，只剩 `final_norm → lm_head → logits → argmax`。`final_norm` 只是一次归一化、`lm_head` 是一个线性投影，**对偏差几乎不衰减**。0.16 量级的偏差直接打到 logits 上，没有任何缓冲。

所以正确的认知是：

| 层 | 跟高精度参考比 | 对最终输出 |
|---|---|---|
| Layer 0-41 Marlin | 每层都有 ~0.03 偏差 | 不致命——后面有几十层缓冲 |
| Layer 42 Marlin | ~0.16 偏差 | **致命**——直接打到 logits |
| Layer 42 CDG | 跟高精度参考差更远（0.16） | 不致命——**这才是模型校准的部署语义** |

注意最后一行的反直觉：**CDG 数值上"更不准"，但模型反而能跑对**。因为模型量化校准时跑的就是 W4A8 这条管线，logits 分布是按这条管线长出来的，argmax 边界也按这条管线划的。"准"和"对"在量化模型里不是同义词。

────────────────────

那为什么 layer 42 一个 0.16 级别的输出差异就能把生成带崩？上面 💡 块讲了"没有下一个 block 来吸收偏差"——具体看一下偏差怎么传到 token 上。

最后一层之后的流程是：

```text
hidden state → final_norm → lm_head → logits → argmax 选下一个 token
```

`final_norm` + `lm_head` 对 hidden state 偏差几乎不衰减。所以 layer 42 这 0.16 的偏差，几乎原样打到 logits 上。

logits 是 vocabulary 上的得分，最终靠 argmax 选 token。如果两个候选 token 的 logits 很接近：

```text
token A: 12.031
token B: 12.028
```

那 hidden state 里 0.1 量级的通道变化完全可能把 A/B 顺序翻过来。

第一个 token 翻错之后，自回归链条整条换轨：

```text
正确路径：Write -> a -> function -> ...
错误路径：</ -> _ -> , -> </ -> ...
```

后面每一步都基于前一步采样。**错一个 token 不是"错一个字"，而是换了一条时间线**——这就是为什么张量余弦 0.999 看着"几乎一样"，输出却是"正常英文"和"符号循环"的差别。

LLM 的最后一层不是普通中间层。它是高维语义空间坍缩到 token 的最后一道门。

━━━━━━━━━━━━━━━━━━━━

◆ 185 期错在哪，对在哪？

━━━━━━━━━━━━━━━━━━━━

185 期的工程结论是对的：

```text
0-41 Marlin + 42 CDG
```

这仍然是当前最小正确规避。

185 期的机制解释错了一半。

当时说：

> 前 42 层 Marlin 误差累积，最后一层 CDG 兜住。

现在修正为：

> 前 42 层在 mixed 和 all-marlin 里都是 Marlin，所以两边完全一致；真正的差异只发生在 layer42。layer42 用 CDG 时走 W4A8 语义，用 Marlin 时走 W4A16 语义。最终层对 W4A8/W4A16 的语义漂移极敏感，所以全 Marlin 乱码。

这两个解释外部行为一样：

- 只换最后一层就好；
- 全 Marlin 就坏；
- 混合方案速度接近纯 Marlin。

但内部机制完全不同。

旧解释是“Marlin 有小错误，最后一层别再错了”。

新解释是“Marlin 算的是另一种合法语义，但不是这个模型最后一层需要的语义”。

这差别很大。

因为如果旧解释成立，修法是继续追 Marlin 的数值 bug、ldmatrix、repack、共享内存布局。

如果新解释成立，修法是：

**让 Marlin 支持 W4A8 activation quantization 语义。**

也就是：

```text
FC1 输入：FP8 per-token-group quantization, block=128
SwiGLU 后：再 FP8 per-token-group quantization, block=128
FC2 输入：用 FP8 activation
```

这不是修一个 `ldmatrix` 就能解决的。

这是 kernel 语义层面的差异。

━━━━━━━━━━━━━━━━━━━━

◆ 为什么这件事对工程很重要？

━━━━━━━━━━━━━━━━━━━━

排量化模型 bug 时很容易默认一个框架："**高精度 reference = 正确，谁离 reference 近谁就对**"。普通场景没问题，但对量化模型会误导你。

因为模型实际适配的不是一个抽象数学公式，而是一条具体的数值管线——权重量化格式、scale 格式、激活量化、block size、clamp limit、TP 切片、reduction 顺序、kernel 内部 accumulation 精度……这些都是"模型运行时函数"的一部分。**你把其中一个环节"提高精度"，不一定更正确**。

打个比方：训练时一直给一个人戴着某种颜色滤镜看世界，他已经按那个滤镜校准了判断。你突然把滤镜摘掉，物理上更真实，但他的判断系统反而失准。

Marlin 这次就是这样——不是低级 bug，是另一套合法的数值世界。前 42 层这套世界还能蒙混过关，最后一层过不去。

━━━━━━━━━━━━━━━━━━━━

◆ 当前工程结论

━━━━━━━━━━━━━━━━━━━━

现在的最小可用方案不变：

```bash
VLLM_MXFP4_MARLIN_LAYER_RANGE=0:42
```

或者更明确：

```bash
VLLM_MXFP4_MARLIN_DEEPGEMM_LAYERS=42
```

含义：

- 0-41 层走 Marlin；
- layer42 走 CDG/DeepGEMM；
- 保持 12 tok/s 左右；
- 中英文正确。

如果要做真正的全 Marlin 正确，不是继续盯着“Marlin 为什么不贴 BF16 reference”。

它已经很贴了。

要做的是：

```text
Marlin W4A8
```

也就是给 Marlin 加上和 DeepGEMM FP4 backend 一致的 FP8 activation quantization：

1. FC1 前，对 `x` 做 FP8 per-token-group quantization，block=128。
2. FC1 后做 clamp + SwiGLU。
3. SwiGLU 输出再做 FP8 per-token-group quantization，block=128。
4. FC2 用 FP8 activation × FP4 weight。
5. 保持 TP 半片语义和最终 all-reduce 不变。

这才是完整对齐 CDG 的方向。

否则纯 W4A16 Marlin 就算写得再精确，也可能继续在最后一层翻车。

━━━━━━━━━━━━━━━━━━━━

◆ 那为什么不直接把 Marlin 改成 W4A8？

━━━━━━━━━━━━━━━━━━━━

技术上能做，但目前没做。原因分两层。

**第一层：工程量**

要让 Marlin 支持 W4A8，得改 3 件事：

1. **FC1 入口加一道 FP8 量化**：把 `x`（BF16）按 per-token-group block=128 压成 FP8，fuse 进 Marlin kernel 的前置阶段，别下显存。
2. **主 MMA 流水线换指令族**：BF16 × FP4 走的 PTX 指令是一套（`mma.m16n8k16` 之类），FP8 × FP4 是另一套（`mma.m16n8k32` + scale 指令）。指令换了 → 寄存器布局换 → shared memory 排版换 → ldmatrix 切片换。Marlin 内部牵一发动全身。
3. **SwiGLU 后再加一次 FP8 量化**：FC2 入口同样要 fuse 进来。

这些都是 kernel 里牵一发动全身的改造，不是改几行代码。

**第二层：硬件层面的不确定性**

更要命的是：**sm_121 上的 FP8 Tensor Core 指令到底支不支持，是个未知数**。

GB10 是消费级 Blackwell，跟数据中心 Blackwell（GB200，sm_100）规格不一样。数据中心 Blackwell 上 FP4/FP6/FP8 Tensor Core 全开，但消费级这一代老黄一向有刀法——比如 sm_120 上 `tcgen05` 那种 Hopper/Blackwell 旗舰指令物理就不存在。FP8 Tensor Core 在 sm_121 上是不是被砍了，得查 ISA 文档或写探针测才知道。

如果真的被砍了，那这事**不是工程问题，是硬件物理不支持**——根本做不了。CDG 那条路之所以走 Triton `tl.dot_scaled`，大概率就是绕开这个硬件缺失。

**第三层：性价比**

就算技术上做得到，**收益其实不大**：

- 当前混合方案 12 tok/s，纯 Marlin 大概 14 tok/s——差距只有 17% 左右
- 真做出 Marlin W4A8，最理想情况也就是恢复到纯 Marlin 速度

也就是说，付出"重写 Marlin 一大块 + 趟 sm_121 FP8 Tensor Core 是否存在"的工程代价，换来 17% 的速度提升。这点速度提升，加 batch、加 PP microbatch、调 scheduler 都能搞回来。对生产推理意义不大。

**所以更值得做的方向是反过来的**——不是把 Marlin 改快、改对，而是**把 CDG 的 Triton 路径调度优化、调对得跟 Marlin 速度逼近**。后一条路在已经熟悉的 Triton 栈上起步，工程量小得多，也不依赖 sm_121 的 FP8 Tensor Core 是否存在。这才是后续真正性价比高的方向。

混合方案不是临时妥协，是当前条件下的最优解。

◆ 一句话总结

━━━━━━━━━━━━━━━━━━━━

**这次 bug 的最终形状不是“Marlin 算错了”，而是“Marlin 算的是 W4A16，DeepGEMM/CDG 算的是 W4A8；DeepSeek V4 Flash 的最后一层需要 W4A8 语义”。**

185 期的混合方案仍然对，但理由要改。

最后一层 CDG 不是在“纠正前 42 层 Marlin 的错误”，而是在把最后一次 MoE 拉回模型实际部署时的量化语义。

这就是为什么只换一层就能从乱码变正常。

有时候工程排障最难的不是找到“哪里错了”，而是承认：

你以为的正确参考，本来就不是系统真正运行的参考。

━━━━━━━━━━━━━━━━━━━━

**「Marlin 不是算错了，是算的另一套合法语义——W4A16。但 DeepSeek V4 Flash 最后一层要的是 W4A8。」**

**「越接近高精度公式，越不一定正确。模型实际跑的是一条具体管线，不是一个抽象数学公式。」**

**「前 42 层 mixed 和 all-marlin 完全一致——0 误差。所谓‘误差累积’的旧叙事，根本没发生过。」**

**「张量余弦 0.99939,看着像是‘几乎一样’;但落到最后一层 logits 上,就是‘正常英文’和‘符号循环’的距离。语言模型的采样边界,就是这么薄。」**
