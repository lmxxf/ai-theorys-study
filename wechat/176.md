◆ 前情提要

━━━━━━━━━━━━━━━━━━━━

先解释一个词：**kernel**（核函数）。在 GPU 编程里，所有"从 CPU 发起、在 GPU 上执行的函数"统一叫 kernel。不管是一行 `torch.mm` 矩阵乘法（底层也是调一个 kernel），还是几百行手写的 CUDA 代码，只要是跑在 GPU 上的计算函数，都叫 kernel。你可能在别处见过"算子"这个词——意思差不多，但"算子"是深度学习框架层面的叫法，容易和数学里的算子混淆，本文统一用 kernel。这个词后面会反复出现。

注意：kernel 是由 CPU 发起的，不是 GPU 自己调自己。你写 `torch.mm` 然后 `torch.add`，实际上是 CPU 依次对 GPU 说"跑乘法 kernel"→"跑加法 kernel"，每次都要从 CPU 发一条指令过去。GPU kernel 调 GPU kernel 在技术上可行但几乎没人用。后面说的"融合 kernel"就是把多步操作合并成一个 kernel，减少 CPU→GPU 的发起次数。

175 期我们用纯 PyTorch 在 DGX Spark 上搭了 DeepSeek V4 Flash 280B 的实验平台——替换了 6 个 TileLang kernel，跑通了中间层激活值提取。

文章最后留了一个问题：纯 PyTorch fallback 比 TileLang 原版慢 27 倍，根因是"没有融合 kernel，同一块数据在显存里被读写三次"。

这一期就来讲：**GPU 编程到底有哪些层次？从 PyTorch 的一行 `torch.mm` 到 NVIDIA 底层的手写 CUDA kernel，中间经历了什么？**

不是综述——是我们在这个项目里实际用过、踩过坑的东西。每一层都有真实代码。

━━━━━━━━━━━━━━━━━━━━

━━━━━━━━━━━━━━━━━━━━


◆ 第 5 层：纯 PyTorch —— 人话

━━━━━━━━━━━━━━━━━━━━

这是我们 175 期实验平台的全部 kernel 代码。DeepSeek V4 官方推理代码里定义了 6 个 kernel 函数（`fp4_gemm`、`fp8_gemm`、`sparse_attn` 等），我们的 `kernel_sm121.py` 按照同样的函数名和接口逐个重写。举 `fp4_gemm` 的例子：

DeepSeek V4 Flash 280B 的权重用 FP4 存储（4-bit，省空间——280B 参数压到 148GB 才能塞进两台 128GB 机器），推理时每个 token 产生的激活值用 FP8（8-bit，精度够用）。所以每一层的核心计算就是 FP8 激活值 × FP4 权重——不是我们选的，是模型发布时就这么设计的。

具体来说，输入是 FP8 激活值 `a[M,K]` 和 FP4 权重 `b[N,K/2]`（两个 4-bit 权重挤在一个字节里），输出 BF16 矩阵 `[M,N]`：

```python
# 第一步：FP4 权重解包
# 一个字节里塞了两个 4-bit 权重，用位运算拆开
raw = b.view(torch.uint8)          # 把原始数据当成字节数组
low  = raw & 0x0F                  # 取低 4 位 → 第一个权重的编号
high = (raw >> 4) & 0x0F           # 取高 4 位 → 第二个权重的编号
# 用编号去查表，得到真正的浮点数值（4-bit 只有 16 种可能值）
b_fp32 = torch.stack([FP4_TABLE[low], FP4_TABLE[high]], dim=-1).flatten(-2)

# 第二步：乘以缩放因子还原真实数值
# 每 32 个权重共享一个 scale（量化时除以 scale 压缩，现在乘回来）
b_deq = b_fp32.view(N, n_blocks, 32) * b_scale.unsqueeze(-1)

# 第三步：矩阵乘法，得到最终输出
out = torch.mm(a_deq, b_deq.t())
```

三步：拆字节→查表还原→矩阵乘法。`torch.mm` 是矩阵乘法，`view` 是改变数组形状，`&` 和 `>>` 是位运算。一个会写 Python 的人配合注释应该能看懂。

**优点**：好写、好调试、好理解。一天写完 6 个 kernel，290 行。

**缺点**：慢。三步操作 = GPU 被叫三次，每次都要从显存读数据、算完、写回。175 期讲过——就像跑三趟厨房。

慢多少？我们加了 profiler 实测：

```text
fp4_gemm:    2121 次调用, 1.327 秒 (37.5%)
fp8_gemm:     322 次调用, 1.603 秒 (45.3%)
act_quant:   2443 次调用, 0.461 秒 (13.0%)
sparse_attn:   43 次调用, 0.039 秒 ( 1.1%)
```

fp8_gemm 平均每次 5 毫秒。同样的矩阵尺寸（M=12, N=1024, K=4096），用 PyTorch 内置的 `torch._scaled_mm`（直接调 GPU tensor core）只要 0.01 毫秒——**差 500 倍**。

但纯 PyTorch 不是没有优化空间。我们把 `sparse_attn` 从 Python for 循环改成 `torch.einsum`，占比从"大瓶颈"降到 1.1%——**不需要写任何 kernel，只需要把 Python 循环换成向量化操作**。这在 GPU 编程里叫"向量化"，是性价比最高的优化手段。

我们的实验平台最终就用了这个慢版本。目的是提取中间层激活值做研究，跑一次 prefill 5.5 秒拿到数据就行，不需要像推理服务那样一秒吐几十个 token。重点是正确性，不是速度——慢点儿就慢点儿了。

💡 翻译成人话：纯 PyTorch 就像自然语言写作——人人都能写。你可以写出完全正确的程序，只是它可能跑得像蜗牛。大多数时候，把 for 循环换成向量化操作就够了；真的不够用，再往下爬。

◆ 第 4 层：Triton —— 用 Python 写 GPU Kernel

━━━━━━━━━━━━━━━━━━━━

Triton 是 OpenAI 开源的 GPU kernel 编译器。用 Python 语法写，编译成 GPU 机器码。

PyTorch 和 Triton 的本质区别：**PyTorch 是调用现有 kernel，Triton 是写新 kernel。** PyTorch 只能从"菜单"上点——`torch.mm` 是一个现成 kernel，`torch.add` 是另一个，你没法把两步合成一步。Triton 让你自己写 GPU 内部的计算逻辑，想怎么组合怎么组合。

上面纯 PyTorch 版的 fp4_gemm 分三步走显存，我们尝试用 Triton 重写它。核心思路：**把"解包 FP4 → 乘 scale → 矩阵乘法"三步融合成一个 kernel**，数据在 GPU 寄存器里流转，不落显存。

```python
@triton.jit  # 这个装饰器告诉 Triton："把下面的函数编译成 GPU 机器码"
def _fp4_gemm_kernel(a_ptr, a_scale_ptr, b_ptr, b_scale_ptr, out_ptr,
                     M, N, K: tl.constexpr, ...):
    # GPU 同时启动几千个线程跑同一份代码，每个线程负责输出矩阵的一小块
    # program_id 就是"我是第几号工人"——通过它算出自己负责哪一块
    pid_m = tl.program_id(0)   # 我负责输出矩阵的第几行块
    pid_n = tl.program_id(1)   # 我负责输出矩阵的第几列块

    # 累加器：这一小块的计算结果，先存在寄存器里（最快的存储）
    acc = tl.zeros((BLOCK_M, BLOCK_N), dtype=tl.float32)

    # 沿着 K 维度分块循环——矩阵乘法的核心：逐块累加
    for k0 in range(0, K, BLOCK_K):
        # 从显存读 FP8 激活值到寄存器
        a_even = tl.load(a_ptr + ...)
        a_odd  = tl.load(a_ptr + ...)

        # 从显存读 packed FP4 权重，直接在寄存器里解包（不写回显存！）
        b_packed = tl.load(b_ptr + ...)
        b_low  = _decode_fp4_nibble(b_packed & 0x0F)     # 低 4 位 → 第一个权重
        b_high = _decode_fp4_nibble((b_packed >> 4) & 0x0F)  # 高 4 位 → 第二个权重

        # 乘 scale 还原真实值，然后做矩阵乘法——全在寄存器里完成
        # 这就是"融合"的意义：解包、乘scale、矩阵乘法一气呵成，数据不落显存
        acc += tl.dot(a_even, tl.trans(b_low * bs_even)) * a_s[:, None]
        acc += tl.dot(a_odd, tl.trans(b_high * bs_odd)) * a_s[:, None]

    # 最后才把结果写回显存——整个过程只读一次、写一次
    tl.store(out_ptr + ..., acc)
```

几个关键概念：

**program_id**：GPU 启动几千个线程同时跑同一份代码，每个线程通过 `program_id` 知道自己负责输出矩阵的哪一块。就像流水线工人——你负责第 3 排第 7 列，我负责第 5 排第 2 列。

**tl.load / tl.store**：从显存读数据到寄存器 / 从寄存器写回显存。这是 GPU 编程里最贵的操作——寄存器访问 ~1 纳秒，显存访问 ~100 纳秒，差 100 倍。融合 kernel 的本质就是**减少 load/store 次数**。

**tl.dot**：矩阵乘法的核心操作。Triton 会把它编译成 GPU 的 tensor core 指令——专门为矩阵乘法设计的硬件单元。

**FP4 解包用算术而非查表**：PyTorch 版用查找表（16 个值的数组，用编号直接取），但查表要访问显存或共享内存——在 kernel 内部反而比直接算更麻烦更慢。4-bit 浮点数只有 1 位符号 + 2 位指数 + 1 位尾数，几步位运算就能算出来，全在寄存器里完成：

```python
@triton.jit
def _decode_fp4_nibble(nibble):
    sign = (nibble >> 3) & 1
    e = (nibble & 0x07) >> 1   # 2-bit 指数
    m = nibble & 1             # 1-bit 尾数
    val_norm = (2 + m) * tl.exp2((e - 1)) * 0.5
    val_sub = m * 0.5
    val = tl.where(e == 0, val_sub, val_norm)
    return tl.where(sign == 1, -val, val)
```

16 个可能值全部与 FP4 查找表精确匹配——验证通过。

**实测结果：对短 prompt（M=1~12）没效果。** M 是要同时计算的 token 数——短 prompt 只有几个 token，矩阵很小。这时 Triton kernel 的启动开销（CPU 编译+发起，~5 毫秒）远大于 GPU 上的计算本身（~0.02 毫秒）。就像你请了一个大厨来煎一个鸡蛋——大厨穿围裙、磨刀、热锅的准备时间比煎蛋本身长得多。

💡 翻译成人话：Triton 让你用 Python 写 GPU kernel——不用学 C++，不用管线程同步，不用手动分配共享内存。代价是：启动开销比原生 CUDA 大，小矩阵不划算。适合中等规模以上的计算密集任务。

Triton 还有一个优势：**跨硬件移植性好**。OpenAI 设计 Triton 的目标就是让所有人都能写 kernel，所以编译器自动适配不同 GPU 架构。TileLang、DeepGEMM、CUTLASS 在 sm_120+ 上全趴了，只有 Triton 能跑——这也是为什么社区给 vLLM 做 sm_120+ 支持时，fallback 方案一律选 Triton。

◆ 第 3 层：TileLang —— DeepSeek 的选择

━━━━━━━━━━━━━━━━━━━━

TileLang 是微软开源的 GPU kernel DSL（领域特定语言），和 Triton 是同一层的平行替代品——都是"用 Python 写 GPU kernel"，只是编译器后端不同。DeepSeek V4 的官方推理代码选了 TileLang 写全部 kernel，不用 Triton，不用 CUDA。

看一个 TileLang 版的 fp8_gemm 长什么样：

```python
@tilelang.jit(pass_configs=pass_configs)  # 编译成 GPU 机器码（和 Triton 的 @triton.jit 同理）
def fp8_gemm_kernel(N, block_size=128, ...):
    @T.prim_func
    def fp8_gemm_kernel_(
        A: T.Tensor[(M, K), "float8_e4m3"],       # FP8 激活值
        B: T.Tensor[(N, K), "float8_e4m3"],       # FP8 权重（这是 fp8_gemm，两边都是 FP8）
        A_scale: T.Tensor[(M, K // block_size), "float8_e8m0fnu"],  # 缩放因子
        ...
    ):
        # 手动分配"共享内存"——GPU 线程块内部共享的高速缓存
        # 这是 TileLang 和 Triton 的关键区别：Triton 自动管，TileLang 要你手动管
        A_shared = T.alloc_shared((block_M, block_K), "float8_e4m3")
        B_shared = T.alloc_shared((block_N, block_K), "float8_e4m3")

        # 沿 K 维度分块循环（和 Triton 版一样的逻辑）
        for ko in T.serial(K // block_K):
            T.copy(A[...], A_shared)   # 从显存搬一块到共享内存
            T.copy(B[...], B_shared)   # 从显存搬一块到共享内存
            T.gemm(A_shared, B_shared, C_local)  # 在共享内存里做矩阵乘法
```

和 Triton 对比：

| 维度 | Triton | TileLang |
|------|--------|----------|
| 语言 | Python + Triton IR | Python + MLIR |
| 内存管理 | 自动（编译器决定） | **手动**（`T.alloc_shared`） |
| 编译器 | OpenAI Triton → PTX | 微软 TVM/MLIR → PTX |
| 适合谁 | 快速原型、中等复杂度 | 追求极致性能、愿意手调 |
| sm_120+ 支持 | ✅ | ❌（v0.1.8 只支持 sm_75/90/100）|

TileLang 要求你手动管理共享内存（GPU 上线程块内部共享的高速缓存），这意味着更多的控制权，也意味着更多的心智负担。

💡 翻译成人话：Triton 和 TileLang 的关系类似自动挡和手动挡。Triton 是自动挡——编译器帮你管内存分配，你只管写计算逻辑，上手快，大多数场景够用。TileLang 是手动挡——你自己决定数据放在哪级缓存、什么时候搬运，能榨出更极致的性能，但写错了就翻车。

DeepSeek 选 TileLang 而不是 Triton，不是因为 Triton 性能不行，而是 DeepSeek 有专门的 kernel 工程团队，他们在自家数据中心的 GPU（sm_90/sm_100）上用手动挡能做到更极致的优化。代价是移植性差——TileLang 只支持数据中心级 GPU（sm_90/sm_100），整个 sm_120+（RTX 5090、RTX 6000 Pro、DGX Spark）都不在支持列表里。哪怕你花几万块买了 96GB 显存的专业卡，想直接跑 DeepSeek V4 官方的 TileLang kernel 也跑不了。不只是 TileLang——DeepGEMM（DeepSeek 的高性能矩阵乘法库）同样不支持，PyTorch 的支持也还在做。社区已经在催了：DeepGEMM issue #236、PyTorch issue #157549、vLLM issue #41063，全是要求支持 sm_120+ 的。我们被迫退到 Triton 和纯 PyTorch。

◆ 第 2 层：CUTLASS —— NVIDIA 的 C++ 模板库

━━━━━━━━━━━━━━━━━━━━

CUTLASS 是 NVIDIA 官方的 CUDA 模板库——用 C++ 写，提供矩阵乘法的高性能实现。

我们在 vLLM 推理部署（171 期）的后续优化里碰到了 CUTLASS。171 期跑通了推理服务，但后来发现长输出有垃圾——根因是 Marlin（vLLM 的 FP4 MoE kernel）在 sm_120+ 上静默算错。为了修这个问题，我们做了一个 Consumer-DeepGEMM 项目，试图用 CUTLASS 的 sm_120+ 模板替换 Marlin。

CUTLASS 代码长这样（从我们的 Consumer-DeepGEMM 项目里截取）：

```cpp
// ===== 第一部分：声明"我要算什么" =====

// 输入 A：FP8 激活值（推理时每个 token 产生的中间结果）
using ElementA = cutlass::mx_float8_t<cutlass::float_e4m3_t>;
using LayoutATag = cutlass::layout::RowMajor;    // 行优先存储

// 输入 B：FP4 权重（模型参数，4-bit 压缩存储）
using ElementB = cutlass::mx_float4_t<cutlass::float_e2m1_t>;
using LayoutBTag = cutlass::layout::ColumnMajor; // 列优先存储

// 输出 D：BF16（计算结果）
using ElementD = cutlass::bfloat16_t;

// 在哪跑、用什么硬件指令
using ArchTag = cutlass::arch::Sm120;            // 目标架构：sm_120+
using OperatorClass = cutlass::arch::OpClassBlockScaledTensorOp;  // 用 block-scaled tensor core
using ThreadBlockShape = Shape<_128, _128, _128>;  // 每个线程块处理 128×128×128 的数据块

// ===== 第二部分：让 CUTLASS 模板系统自动生成 kernel =====

// 把上面的类型参数喂给 CUTLASS 的 "Builder"，它自动决定：
// - 数据怎么从显存搬到共享内存
// - 共享内存怎么分配
// - 线程之间怎么协作
// - 结果怎么写回
using CollectiveMainloop = typename cutlass::gemm::collective::CollectiveBuilder<
    ArchTag, OperatorClass,
    ElementA, LayoutATag, 16,      // A 的对齐要求
    ElementB, LayoutBTag, 128,     // B 的对齐要求
    float,                         // 累加精度（中间结果用 float 防精度丢失）
    ThreadBlockShape, ClusterShape,
    ...>::CollectiveOp;

// ===== 第三部分：组装成可运行的 kernel =====

using GroupedGemm = cutlass::gemm::device::GemmUniversalAdapter<
    cutlass::gemm::kernel::GemmUniversal<
        GroupProblemShape,          // 支持多组矩阵（MoE 的 256 个专家）
        CollectiveMainloop,         // 上面自动生成的计算逻辑
        CollectiveEpilogue>>;       // 输出处理（类型转换等）

// ===== 使用：构造参数 → 初始化 → 运行 =====
GroupedGemm gemm;
auto status = gemm.initialize(arguments, workspace);  // 分配工作空间
status = gemm.run(stream);                            // 在 GPU 上执行
```

注意到了吗？**这里看不到任何循环、任何 load/store、任何算术运算。** 你只是在声明类型——"我要用 FP8 乘以 FP4，输出 BF16，在 sm_120+ 上跑，线程块大小是 128×128×128，支持多组矩阵"。然后 CUTLASS 的模板系统根据这些类型参数，在编译期展开出几千行优化过的 CUDA 代码。

这就是 C++ 模板元编程——你写的不是代码，是"关于代码的代码"。编译器替你写真正的 kernel。

💡 翻译成人话：如果 Triton 是"告诉 GPU 怎么算"，CUTLASS 是"告诉编译器你要算什么，让编译器去想怎么算"。像是你告诉建筑公司"我要一栋 3 层楼、钢筋混凝土、抗 8 级地震"，具体怎么配钢筋、怎么浇混凝土是工程师的事。

我们用 CUTLASS 做到了什么？

1. 类型探针通过——CUTLASS 接受 FP8×FP4→BF16 on sm_120+ ✅
2. `can_implement()` 通过——对真实 tensor shape 返回 True ✅
3. 实际 launch 通过——`initialize() + run()` 不报错 ✅
4. 数值正确性验证通过——171 期 vLLM 推理的长输出垃圾消失 ✅

**代价：编译时间**。每次改一个模板参数，ARM64 上重新编译要 40 分钟。12 次编译 = 8 小时的等待。

**代价：运行速度**。我们选了 CUTLASS 的 block-scaled MX kernel 替换 Marlin，修正了计算结果，但推理速度从 ~13 tok/s 降到了 ~0.8 tok/s——每次 CUTLASS launch 都有大量 CPU↔GPU 同步开销，280B MoE 模型每生成一个 token 要 launch 120 次。正确但慢，后续优化空间还很大。

**为什么不能照着 Marlin 重写一个？** 171 期用的 jasl/vllm fork 是给 sm_120+ 做通用适配的，不是专门针对 DeepSeek V4——它用 Triton 重写了 DeepGEMM 的 hyperconnection kernel，解决了启动报错，但 V4 特有的 FP4 MoE 路径不在它的范围内，仍然走 vLLM 内置的 Marlin kernel（vLLM 的 FP4 量化加速模块，用原生 CUDA 手写的——就是我们五层塔最底下的第 1 层）。Marlin 在 sm_120+ 上静默算错——不是代码逻辑有 bug，是它用的 FP4 矩阵乘法硬件指令在 sm_120+ 上行为和数据中心 GPU 不一致。你照着 Marlin 的逻辑重写一个 Triton 版也没用，同样的指令同样算错。必须换一条完全不同的计算路径，走不同的硬件指令。

这也是五层塔越往下移植性越差的具体体现：**用原生 CUDA 写的 kernel，默认都是给数据中心 GPU（sm_90/sm_100）写的。** 除非作者特意测试并支持了消费级显卡，否则拿到 sm_120+ 上大概率跑不了或者算错。Triton 不存在这个问题——编译器自动适配目标硬件，同一份代码换 GPU 不用改。

【插一段：sm_100 和 sm_120+ 到底差在哪】

很多人以为"都叫 Blackwell，应该兼容吧"——不是的。Blackwell 实际上分两个家族：

- **sm_100 家族（数据中心）**：B100、B200、GB200、B300 等——给云厂商和超算中心用的
- **sm_120 家族（个人电脑）**：RTX 5090（sm_120）、DGX Spark（sm_121）、RTX PRO 6000（sm_122）

同一个家族内部基本兼容，但两个家族之间 GPU 芯片内部的 tensor core 指令集不完全一样：

- sm_100 有 `tcgen05` 指令（第 5 代 tensor core 指令生成器）——专门用于高效调度 FP4/FP8 矩阵乘法
- sm_120+ **没有** `tcgen05`——要做同样的 FP4 矩阵乘法，得走另一套指令路径（`mma.sync` + block-scaled MX），数据布局要求也不同。CUTLASS 有对应的 sm_120 模板，这就是我们 Consumer-DeepGEMM 的计算路径

这不是 NVLink、InfiniBand 那种卡和卡之间通信的区别（那是"高速公路不同"），而是**卡内部的计算引擎型号不同**（"发动机不同"）。Marlin 用了 sm_100 专属的 tensor core 指令，拿到 sm_120+ 上这些指令要么不存在、要么行为不一致——算出来的数就是错的。

NVIDIA 为什么要这么分？因为数据中心 GPU 和消费级 GPU 的设计目标不同：数据中心追求大规模并行训练/推理的极致吞吐，消费级追求图形渲染+单卡推理的性价比。tensor core 的指令集设计、内存控制器、线程调度器都有差异。它们共享 Blackwell 这个"品牌名"和大部分微架构，但底层硬件指令不完全兼容——对写 PyTorch 的人没影响（PyTorch 帮你挡了），对写原生 CUDA kernel 的人就是地雷。"Blackwell"这个名字迷惑性极大：你以为买了 Blackwell 架构的卡就能跑 Blackwell 的 kernel，实际上数据中心 Blackwell 和个人电脑 Blackwell 是两种芯片。

━━━━━━━━━━━━━━━━━━━━

◆ 第 1 层：原生 CUDA —— 手搓汇编

━━━━━━━━━━━━━━━━━━━━

最底层。直接写 `__global__ void kernel(...)`，手动管理线程、共享内存、寄存器分配、同步屏障、内存对齐……

我们项目里没有直接写原生 CUDA kernel（谢天谢地），但 DeepGEMM 里有。

DeepGEMM 是 DeepSeek 自己开源的 GEMM 库——比 CUTLASS 更底层。它直接用 cuTE（CUTLASS 的底层张量抽象层）手写 CUDA kernel。下面这段来自 DeepGEMM 源码，是给 sm_90（Hopper 数据中心卡）写的 BF16 矩阵乘法：

```cpp
// 来源：DeepGEMM/csrc/sm90_bf16_gemm.cu（DeepSeek 开源）
// 这是一个给数据中心 GPU 手写的矩阵乘法 kernel

// 模板参数：编译时确定线程块处理多大的数据块
template <uint32_t BLOCK_M, uint32_t BLOCK_N, uint32_t BLOCK_K_, ...>

// CUTLASS_GLOBAL = __global__，表示这是 GPU kernel 入口
// __launch_bounds__ 告诉编译器：每个线程块最多用这么多线程，帮编译器优化寄存器分配
CUTLASS_GLOBAL __launch_bounds__(kNumTMAThreads + kNumMathThreads, 1)
void sm90_bf16_gemm_impl(
    int* grouped_layout,    // MoE 专家分组信息
    uint32_t shape_m, ...   // 矩阵尺寸

    // TmaDescriptor：Hopper 架构专属的硬件级张量搬运指令
    // TMA = Tensor Memory Accelerator，比 tl.load 更底层
    // 让硬件自动把数据从显存搬到共享内存，CPU/GPU 代码都不用管搬运细节
    // 这就是 sm_100 家族有而 sm_120+ 没有的东西之一
    const __grid_constant__ cute::TmaDescriptor tensor_map_a,
    const __grid_constant__ cute::TmaDescriptor tensor_map_b,
    ...
```

看到了吧——`TmaDescriptor` 这种东西是数据中心 GPU 专属的硬件特性，sm_120+ 压根没有这个硬件单元。这就是为什么 DeepGEMM 不可能简单"移植"到消费级卡上。

严格来说，在原生 CUDA 下面还有一层 **PTX**（Parallel Thread Execution）——NVIDIA GPU 的中间汇编语言，比 CUDA C++ 再底层一步。之前自媒体反复炒作"DeepSeek 用 CUDA 汇编写 AI"，说的就是 PTX。但 DeepGEMM 主体仍然是 CUDA C++ + cuTE 写的，只是在少数极致优化的地方用了 PTX 内联汇编（`asm volatile("...")`）。"用了几行内联汇编做极致优化"被自媒体夸大成了"用汇编写 AI"——就像说"Linux 内核是用汇编写的"一样离谱（Linux 99% 是 C，只有启动代码和少数硬件交互用汇编）。

不管是 CUDA C++ 还是 PTX 内联汇编，DeepGEMM 都只给数据中心 GPU 写了。它的 `hyperconnection.hpp` 有一行硬编码：

```cpp
// csrc/apis/hyperconnection.hpp:56
assert(arch == sm_90 || arch == sm_100);  // Unsupported architecture
```

171 期我们撞上的 `Unsupported architecture` 就是这行。

💡 翻译成人话：原生 CUDA 就像手搓汇编——理论上你可以控制一切，实际上你会被线程同步、内存对齐、寄存器溢出、bank conflict 这些问题折磨到怀疑人生。除非你是 NVIDIA 的工程师或者 DeepSeek 的 kernel 团队，否则不要碰这一层。

◆ 对照表：五层塔一览

━━━━━━━━━━━━━━━━━━━━

| 层 | 代表 | 语言 | 谁在用 | 性能 | 开发速度 | 移植性 |
|----|------|------|--------|------|---------|--------|
| 5 | **纯 PyTorch** | Python | 我们（fallback） | 最慢（1×） | 最快（1 天） | 最好 |

◆ 实测教训：Triton 不是银弹

━━━━━━━━━━━━━━━━━━━━

今天最大的教训是：**写了 Triton 融合 kernel 不等于性能提升。**

我们给 fp4_gemm 和 fp8_gemm 都写了 Triton 版本。编译通过，正确性验证通过，在容器里跑通了。然后实测——12 token 的 prefill 从 5.5 秒变成 5.2 秒。几乎没差。

为什么？

因为我们的实验场景是 **bandwidth bound（带宽瓶颈）**，不是 compute bound（计算瓶颈）。

区分很重要：

- **Compute bound**：GPU 的计算单元来不及算。大矩阵乘法（M=1024, N=4096, K=4096）就是这种——计算量 O(M×N×K)，GPU 要算几十亿次乘加。融合 kernel 能减少中间数据的显存读写，让计算单元不用等数据。
- **Bandwidth bound**：GPU 的计算单元闲着没事干，全部时间花在等数据从显存搬过来。小矩阵乘法（M=1, N=2048, K=4096）就是这种——数据量 O(N×K) 不变，计算量 O(M×N×K) 极小。

我们的 MoE 每个专家平均只分到 1-2 个 token（M=1~2），fp8_gemm 的注意力层也只有 M=12。GPU 几乎全部时间花在从显存读权重（N×K 个元素），算乘法的时间可以忽略。

这种情况下，融合 kernel 省掉的是"中间结果的读写"——但中间结果只有 M×K 个元素，M=1 时只有一行，省了也几乎感觉不到。

我们还试了另一个方向：**把 FP8 权重预先 dequant 成 BF16 缓存起来**，运行时直接用——结果更慢。因为 BF16 每个元素 2 字节，FP8 每个元素 1 字节。bandwidth bound 场景下，读 2 字节的 BF16 比读 1 字节的 FP8 慢两倍。你省了 dequant 的计算时间，但多花了一倍的搬运时间。

```text
优化前：读 FP8 (1 字节) → dequant → 算   = 读 1 字节 + 少量计算
优化后：读 BF16 (2 字节) → 算              = 读 2 字节
结论：更慢
```

💡 翻译成人话：如果你的快递员每趟只能搬一箱，瓶颈是搬运次数。你把箱子里的东西预先组装好（省了拆包时间），但箱子变成两倍大——快递员每趟搬的更慢了。正确做法是减小箱子（用更紧凑的格式），而不是预先拆包。

**这个教训值得记住：优化之前先用 profiler 量化瓶颈。** 我们加了 7 行 profiler 代码，立刻看到 fp8_gemm 322 次调用占 45%，每次 5 毫秒。然后用 `torch._scaled_mm` benchmark 确认同样的矩阵只要 0.01 毫秒——瓶颈一目了然是 Triton 的 launch overhead，不是计算本身。

◆ 什么时候该用哪一层

━━━━━━━━━━━━━━━━━━━━

| 场景 | 建议 |
|------|------|
| 做实验、验证正确性 | **第 5 层 PyTorch**。先跑通再优化。 |
| for 循环是瓶颈 | 还是 **PyTorch**。用 einsum/gather/向量化操作替换循环，性价比最高。 |
| 需要融合 kernel（多步操作合一） | **第 4 层 Triton**。Python 语法写 kernel，一两天能出活。 |
| 需要极致性能 + 固定矩阵尺寸 | **第 2-3 层 CUTLASS / TileLang**。编译慢，但运行时最快。 |
| 需要压榨新硬件特性（TMA、warp specialization） | **第 1 层原生 CUDA**。除非你是 kernel 工程师，否则别碰。 |

大多数 AI 研究者只需要第 5 层。大多数 AI 工程师只需要到第 4 层。到第 2 层以下的人，写的代码是给前面所有人用的——他们是 DeepGEMM、FlashAttention、vLLM 这些框架的作者。

◆ 跋

━━━━━━━━━━━━━━━━━━━━

从 PyTorch 到 CUDA，五层塔不是一条"越往下越好"的路。每一层存在的理由是不同的：

- PyTorch 存在是因为**正确性比速度重要**——你得先跑通再谈优化。
- Triton 存在是因为**大多数人不应该写 C++**——Python 语法就能写出不错的 kernel。
- TileLang 存在是因为**手动挡能榨出最后一滴性能**——你比编译器更了解数据应该怎么摆，但只有大厂的 kernel 团队才值得用。
- CUTLASS 存在是因为**模板元编程能在编译期生成几千行优化代码**——你声明"算什么"，编译器决定"怎么算"。
- 原生 CUDA 存在是因为**总有模板覆盖不到的角落**——DeepGEMM 的 Hyperconnection kernel 就是一个。

我们的项目从第 5 层开始，被硬件限制逼到第 4 层，在 vLLM 项目里爬到了第 2 层，从头到尾避开了第 1 层。

**最终让实验平台跑起来的，是第 5 层的 290 行纯 PyTorch。** Triton 写了 250 行，实测对短 prompt 没效果。Profile 数据告诉我们下一步真正该优化的是 `act_quant`（28.5%）和 `hc_sinkhorn`（27.6%）——这两个用纯 PyTorch 写的函数，才是长 prompt 下的瓶颈。

性能优化的第一步永远不是"用更底层的语言重写"，而是"量化瓶颈在哪"。
