◆ 上一轮：Triton 融合 + sparse_attn 向量化

━━━━━━━━━━━━━━━━━━━━

175 期发出来之后先做了一轮优化（没单独写文章），针对前两个方向：

**fp4_gemm / fp8_gemm：把分步计算合并成一次 GPU 操作。** 175 期的做法是"先把压缩格式的权重还原成普通浮点数，再做矩阵乘法"——同一块数据在显存里被读写三次（还原后写回显存→矩阵乘法读出来→结果写回显存）。改成 Triton（一种 GPU 编程语言，比 CUDA 好写，编译器自动做很多优化）之后，解包 + 缩放 + 矩阵乘法在一次 GPU 操作里完成，中间数据留在芯片寄存器里不落显存。

FP4 的解包值得多说一句。模型权重用 FP4（4-bit 浮点）存储，两个 FP4 值打包在一个字节里——字节的高 4 位和低 4 位各一个（这半个字节在业界叫 nibble——byte 谐音 bite 咬一大口，nibble 是小咬一口。网上 CUDA 相关的文章基本都直接用这个词，知道就好）。解包就是把字节拆成两个 nibble，再把每个 nibble 还原成浮点数。175 期用查表法（16 个可能值建一张表），Triton 版改成算术解码——sign 1 位 + 指数 2 位 + 尾数 1 位，直接算出浮点值，不查表。

**sparse_attn：从逐个处理改成批量处理。** DeepSeek V4 的稀疏注意力不是对所有历史 token 做 attention，而是只挑最相关的 top-k 个位置。175 期的实现是 Python for 循环——每个 token 单独挑、单独算，GPU 大部分时间在等 Python 发指令。改成向量化后，所有 token 需要的历史位置一次性挑出来，矩阵乘法批量算完，GPU 一直在干活。

不过这一轮也踩了两个坑：

1. **短 prompt 几乎没提速**——Triton 每次启动一个 GPU 操作有固定开销（约 5ms），12 个 token 的实际计算只要 0.02ms，启动开销是计算的 250 倍。短 prompt 下 PyTorch 自带的矩阵乘法反而更快
2. **把权重提前还原成大格式反而更慢**——FP4 每个值占 0.5 字节，还原成 BF16 变成 2 字节，数据量翻 4 倍。GPU 的瓶颈是显存读写带宽，数据越大读得越慢

结论：**瓶颈在显存带宽，不在计算量。** Triton 融合的价值在长 prompt（矩阵更大，计算占比上升），短 prompt 反而不如 PyTorch 原生接口。

用 971 token 的长 prompt 跑，生成速度从 1.8s/tok 提到了 0.3s/tok，快了 6 倍。**这就是本期优化的起点。**

◆ 认知反哺：PTX 指令告诉我们硬件能做什么

━━━━━━━━━━━━━━━━━━━━

这一轮的关键不是"会写 Triton"——上一轮已经写了 Triton 融合 kernel。关键是 182/183 期在 vLLM 推理那边做 PTX 指令调研时发现的一个事实：

**sm_121 的 FP4 MMA 硬件指令是完整存在的。**

具体来说，`mma.sync.aligned.block_scale` 指令在 sm_120/121 上都有，支持 FP4（e2m1）和 FP8（e4m3/e5m2）的混合精度矩阵乘法。Triton 的 `tl.dot_scaled("e4m3", "e2m1")` 能直接用。

175 期的 fp4_gemm 是这么做的：

```text
int8 → FP4_TABLE 查表解包成 f32 → 再做 tl.dot(f32, f32)
```

三步分离，数据在显存里读写三次。当时的认知是"sm_121 没有原生 FP4 MMA 支持，只能手动解包"。

182/183 期翻 PTX 手册翻了出来——FP4 MMA 指令全部存在。Triton 的 `tl.dot_scaled("e4m3", "e2m1")` 接口能用，而且速度实打实地快。

不过要说清楚一件事：Triton 3.6.0 在 sm_121 上调用 `dot_scaled` 时，底层并没有真正走 FP4 tensor core 的原生数据通路——它在 kernel 内部把 FP4 数据解包成 bf16（bfloat16），然后走 bf16 的 tensor core 做矩阵乘法。这个过程叫 "bf16 emulation"（Triton issue #7550）。听起来像是降级，但关键在于：**整个解包 + 缩放 + bf16 matmul 全部在一次 kernel 调用内完成，中间数据留在芯片寄存器里不落显存。** 而 175 期的手动方案是分三步走显存的。所以即使底层用的是 bf16 计算，融合本身就够快了。

💡 翻译成人话：之前以为这条路不通所以绕了远路。后来在另一个项目里查了硬件手册发现路是通的。回来一试，果然快了好几倍。做 vLLM 推理那几期不是白折腾——认知反哺了实验平台。

◆ 改动 1：fp4_gemm 用 tl.dot_scaled 替代手动 FP4 解包

━━━━━━━━━━━━━━━━━━━━

动手之前先写了个小测试（test_native_dot.py）确认 sm_121 上哪些能用。FP8 native dot 没用——Triton 3.6.0 在 sm_121 上 `tl.dot(fp8, fp8)` 和手动 cast 到 f32 后再 dot 速度差不多，表现更像普通 f32 路径，没看到 FP8 MMA 带来的收益。但 FP8×FP4 的 `tl.dot_scaled("e4m3","e2m1")`（FP8 激活值 × FP4 权重，正好是 V4 MoE 专家层的数据类型组合）是真的快：虽然底层还是 bf16 emulation，不是原生 FP4 tensor core 路径，但解包、缩放、矩阵乘法融合在一个 kernel 里，避免了中间结果落显存。对比手动解包后再 matmul，全档位 6-7x：

| M | dot_scaled | 手动解包 | 提速 |
|---|-----------|---------|------|
| 1（生成） | 0.089ms | 0.568ms | 6.4x |
| 12（短 prompt） | 0.089ms | 0.518ms | 5.8x |
| 64 | 0.281ms | 1.879ms | 6.7x |
| 384 | 1.511ms | 10.34ms | 6.8x |
| 1024（长 prompt） | 4.009ms | 27.49ms | 6.9x |

从 M=1（单 token 生成）到 M=1024（长 prompt prefill），提速倍数稳定。

────────────────────

### 新旧 kernel 对比

旧 kernel（175 期）是手动 FP4 decode + even/odd split + 两次 f32 dot：

```python
# 旧：50 多行 Triton JIT 代码
low_nibble = b_packed & 0x0F
low_sign = (low_nibble >> 3) & 1
# ... 16 行手动 FP4 E2M1 解码 ...
acc += tl.dot(a_even, tl.trans(b_low * bs_even), ...) * a_s[:, None]
acc += tl.dot(a_odd,  tl.trans(b_high * bs_odd), ...) * a_s[:, None]
```

新 kernel 用 `tl.dot_scaled` 一次搞定：

```python
# 新：核心就一行
acc += tl.dot_scaled(a_chunk, a_sc, "e4m3", b_chunk, b_sc, "e2m1")
```

Triton 的 `dot_scaled` 接收 FP8 的 A 矩阵块和 FP4 的 B 矩阵块，外加各自的 E8M0 block scale，在一次 kernel 调用里完成解包+缩放+矩阵乘法。数据在寄存器里流转完所有步骤再写回显存。

接口写起来简单，但适配到 fp4_gemm 的函数签名时踩了两个坑。

────────────────────

### 坑 1：A scale 的 block size 不匹配

模型的 A scale 是每 128 个 FP8 值共享一个缩放因子。但 `tl.dot_scaled` 的工作单位是 32——每次处理 32 个值，需要对应的缩放因子。

最直觉的做法是把每个缩放因子复制 4 份（128 / 32 = 4），让每 32 个值都能找到自己的缩放因子。但这里有个隐蔽的性能陷阱：**fp4_gemm 在一次 forward pass 里被调用 7 万多次**（43 层 x 256 个专家 x 每个专家多个矩阵乘法）。每次调用都分配一个临时 tensor 做 repeat_interleave，7 万次的临时 tensor 分配开销抵消了 kernel 本身的提速。

解法是在 Triton kernel 内部用整数除法复用原始 scale：

```python
a_scale_idx = k0 // 128  # 每 4 个 chunk 共享同一个 scale
a_sc = tl.load(a_scale_ptr + m_offs[:, None] * as_stride_m + a_scale_idx * as_stride_k, ...)
```

不分配临时 tensor，在 kernel 里靠地址计算直接读对应的 scale。

────────────────────

### 坑 2：B 权重转置撑爆统一内存

`tl.dot_scaled` 要求权重矩阵按"K 维度连续"的方式排列在内存里。但模型存的是"N 维度连续"——相当于矩阵存的方向和 `dot_scaled` 要读的方向是反的，需要转置。

第一版直接在 fp4_gemm 里做 `b.view(torch.uint8).t().contiguous()`——把权重当字节看（`.view`），转置（`.t()`），然后在内存里重新排列成连续的（`.contiguous()`，这一步会分配新内存）。问题是——这和 A scale 一样——7 万次调用，每次都分配一个转置后的临时 tensor。虽然单个 tensor 不大，但 PyTorch 的 CUDA 内存分配器有碎片化问题，累积起来足以把 128GB 统一内存撑爆死机。

第二版想做全局缓存：第一次调用时转置并缓存，后续直接读缓存。但 768 个专家权重（256 个专家 x 3 个矩阵）的转置副本加起来又是几十 GB，两份（原始+转置）同时存在直接 OOM。

最终解法：**不转置，改读法。** 矩阵在内存里是一维的字节串，"行优先"还是"列优先"只是你用什么步长去跳着读。Triton kernel 的 `tl.load` 可以通过 stride 参数控制"每跳一行跳多远、每跳一列跳多远"。把步长参数交换一下，同一块内存，不搬不动，kernel 读出来的效果就等价于转置后的矩阵。

```python
b_u8 = b.view(torch.uint8)  # 不转置，原样传进 kernel
# kernel 里这样读：n 方向跳 K//2 步，k 方向跳 1 步
# 效果等价于读转置后的矩阵，但内存里什么都没动
b_chunk = tl.load(b_ptr + n_offs[None, :] * b_stride_n + kp_offs[:, None] * b_stride_k, ...)
```

零额外内存，零临时分配。175 期"在一个碗的厨房里做饭"的教训又用上了——统一内存架构下，任何多余的一碗汤都可能是压死骆驼的最后一根稻草。

◆ 改动 2：Sinkhorn 迭代次数

━━━━━━━━━━━━━━━━━━━━

DeepSeek V4 的 Hyper-Connection 机制里有一个 Sinkhorn 归一化步骤——把一个 4×4 的矩阵反复做行列归一化，让每行每列的和都等于 1。原版迭代 20 次。但 4×4 这么小的矩阵，测下来 7 次就基本收敛了（误差 1.7%），最终取 5 次，精度换速度。一行代码：

```python
sinkhorn_iters = min(sinkhorn_iters, 5)
```

长 prompt 下收益明显——hc_sinkhorn 的 kernel 时间占比从 27.6% 降到 12.5%。代价是激活范数有可见漂移；做"同一问题，两种 prompt"这种对比实验可以接受，做严格数值复现实验应切回 7 次。

双机 TP=2，971 token prompt 的 profile 对比：

| 指标 | 优化前 | 优化后 | 提速 |
|------|--------------|--------|------|
| Prefill（971 tok） | 14.6s（66.6 tok/s） | 9.0s（108 tok/s） | 1.62x |

fp4_gemm 从 21.2% 降到 18.5%，hc_sinkhorn 从 27.6% 降到 12.5%。新的最大瓶颈是 fp8_gemm（25.3%）——这个我们暂时没改，因为 FP8 native dot 在 sm_121 上没有实际提速（Triton 3.6.0 的 `tl.dot(fp8, fp8)` 内部还是 cast 成 f32，没分发到 FP8 MMA）。

◆ 补记：RTX 5090 的 FP4 ptxas bug 和我们无关

━━━━━━━━━━━━━━━━━━━━

写完这篇之后看到一篇文章，有人在 RTX 5090（sm_120a）上逆向分析了 NVIDIA ptxas 编译器的 FP4 矩阵乘法 codegen，发现全 1.0 的矩阵乘法理论输出 32 实际输出 2.0，一半数据被静默丢弃，数据格式按定点数而不是标准 e2m1 浮点来解码。非常扎实的工作。

我们一度怀疑 DGX Spark（sm_121）也受影响——毕竟同属 sm_12x 家族。写了探针验证，结论是：**不影响。** sm_120a（a = accelerated，RTX 5090 专属，功能最全）才有 `kind::f8f6f4`（裸 FP4 mma.sync），那个 bug 也只在这条指令上。sm_121 属于 sm_120f（f = family，通用版，功能是 sm_120a 的子集），根本没有这条指令，PTX 编译器直接拒绝。sm_121 的 FP4 只能走 `kind::mxf8f6f4.block_scale`（带 block scale 的 MX 格式），Triton 的 `tl.dot_scaled` 用的正是这条路径。
