◆ 认知反哺：PTX 指令告诉我们硬件能做什么

━━━━━━━━━━━━━━━━━━━━

这次优化的关键不是"会写 Triton"——Triton 融合 kernel 在 175 期就写了。关键是 182/183 期在 vLLM 推理那边做 PTX 指令调研时发现的一个事实：

**sm_121 的 FP4 MMA 硬件指令是完整存在的。**

具体来说，`mma.sync.aligned.block_scale` 指令在 sm_120/121 上都有，支持 FP4（e2m1）和 FP8（e4m3/e5m2）的混合精度矩阵乘法。Triton 的 `tl.dot_scaled("e4m3", "e2m1")` 能直接用。

175 期的 fp4_gemm 是这么做的：

```text
int8 → FP4_TABLE 查表解包成 f32 → 再做 tl.dot(f32, f32)
```

三步分离，数据在显存里读写三次。当时的认知是"sm_121 没有原生 FP4 MMA 支持，只能手动解包"。

182/183 期翻 PTX 手册翻了出来——FP4 MMA 指令全部存在。虽然 Triton 3.6.0 在 sm_121 上实际 fallback 到 bf16 emulation（不是真正走 FP4 tensor core 数据通路），但 `tl.dot_scaled` 这个接口能用，而且速度实打实地快——因为它把解包、缩放、矩阵乘法融合成一次 kernel 调用，省掉了中间的显存读写。

💡 翻译成人话：之前以为这条路不通所以绕了远路。后来在另一个项目里查了硬件手册发现路是通的。回来一试，果然快了好几倍。做 vLLM 推理那六期不是白折腾——认知反哺了实验平台。

◆ 改动 1：fp4_gemm 用 tl.dot_scaled 替代手动 FP4 解包

━━━━━━━━━━━━━━━━━━━━

先写了 microbench（test_native_dot.py）测三件事。

────────────────────

### 测试结果

**FP8 native dot**：Triton 3.6.0 在 sm_121 上没有分发到 FP8 MMA 指令——`tl.dot(fp8, fp8)` 和 `.to(float32)` 后再 dot 速度一样。不值得改。

**FP4 dot_scaled**：全档位 6-7x 提速。

```text
M=1  (gen)  : dot_scaled=0.089ms  dequant=0.568ms  speedup=6.4x
M=12 (short): dot_scaled=0.089ms  dequant=0.518ms  speedup=5.8x
M=64        : dot_scaled=0.281ms  dequant=1.879ms  speedup=6.7x
M=384       : dot_scaled=1.511ms  dequant=10.34ms  speedup=6.8x
M=1024(long): dot_scaled=4.009ms  dequant=27.49ms  speedup=6.9x
```

从 M=1（单 token 生成）到 M=1024（长 prompt prefill），提速倍数稳定，说明瓶颈确实在 kernel 调用开销而不是算力。

**Sinkhorn 收敛**：4x4 矩阵，7 次迭代的 max_diff 对比 20 次只有 1.7e-2，5 次足够用。

────────────────────

### 新旧 kernel 对比

旧 kernel（175 期）是手动 FP4 decode + even/odd split + 两次 f32 dot：

```python
# 旧：50 多行 Triton JIT 代码
low_nibble = b_packed & 0x0F
low_sign = (low_nibble >> 3) & 1
# ... 16 行手动 FP4 E2M1 解码 ...
acc += tl.dot(a_even, tl.trans(b_low * bs_even), ...) * a_s[:, None]
acc += tl.dot(a_odd,  tl.trans(b_high * bs_odd), ...) * a_s[:, None]
```

新 kernel 用 `tl.dot_scaled` 一次搞定：

```python
# 新：核心就一行
acc += tl.dot_scaled(a_chunk, a_sc, "e4m3", b_chunk, b_sc, "e2m1")
```

Triton 的 `dot_scaled` 接收 FP8 的 A 矩阵块和 FP4 的 B 矩阵块，外加各自的 E8M0 block scale，在一次 kernel 调用里完成解包+缩放+矩阵乘法。数据在寄存器里流转完所有步骤再写回显存。

◆ 踩坑：A scale 复用和 B 转置 OOM

━━━━━━━━━━━━━━━━━━━━

接口写起来简单，适配到 fp4_gemm 的函数签名时踩了两个坑。

────────────────────

### 坑 1：A scale 的 block size 不匹配

模型的 A scale 是 per-128 block（每 128 个 FP8 值共享一个 scale）。但 `tl.dot_scaled` 的工作单位是 32——每次处理 32 个 K 值，需要对应的 scale。

最直觉的做法是 `a_scale.repeat_interleave(4, dim=-1)` 把每个 scale 复制 4 份，变成 per-32。但这里有个隐蔽的性能陷阱：**fp4_gemm 在一次 forward pass 里被调用 7 万多次**（43 层 x 256 个专家 x 每个专家多个矩阵乘法）。每次调用都分配一个临时 tensor 做 repeat_interleave，7 万次的临时 tensor 分配开销抵消了 kernel 本身的提速。

解法是在 Triton kernel 内部用整数除法复用原始 scale：

```python
a_scale_idx = k0 // 128  # 每 4 个 chunk 共享同一个 scale
a_sc = tl.load(a_scale_ptr + m_offs[:, None] * as_stride_m + a_scale_idx * as_stride_k, ...)
```

不分配临时 tensor，在 kernel 里靠地址计算直接读对应的 scale。

────────────────────

### 坑 2：B 权重转置撑爆统一内存

`tl.dot_scaled` 的 RHS（B 矩阵）要求 K-major 布局。模型的 FP4 权重存储是 `[N, K//2]`（N-major），需要转置成 `[K//2, N]`。

第一版直接在 fp4_gemm 里做 `b.view(torch.uint8).t().contiguous()`。问题是——这和 A scale 一样——7 万次调用，每次都分配一个转置后的临时 tensor。虽然单个 tensor 不大，但 PyTorch 的 CUDA 内存分配器有碎片化问题，累积起来足以把 128GB 统一内存撑爆死机。

第二版想做全局缓存：第一次调用时转置并缓存，后续直接读缓存。但 768 个专家权重（256 个专家 x 3 个矩阵）的转置副本加起来又是几十 GB，两份（原始+转置）同时存在直接 OOM。

最终解法：**不转置。** Triton kernel 通过 stride 参数直接按 N-major 布局读数据——把 `b_stride_n` 和 `b_stride_k` 按原始布局传进去，load 时交换行列维度的偏移量。B 矩阵在内存里还是 `[N, K//2]`，但 kernel 读它的方式等价于读 `[K//2, N]`。

```python
# B stays in its original [N, K//2] layout. The Triton kernel loads
# a [K//2, N] tile by swapping strides in the address calculation.
b_u8 = b.view(torch.uint8)
# b_u8.stride(0) = K//2 (N-major), b_u8.stride(1) = 1
# kernel loads as: b_ptr + kp_offs * b_stride_k + n_offs * b_stride_n
#   where b_stride_n = b_u8.stride(0) = K//2
#   and   b_stride_k = b_u8.stride(1) = 1
```

零额外内存，零临时分配。175 期"在一个碗的厨房里做饭"的教训又用上了——统一内存架构下，任何多余的一碗汤都可能是压死骆驼的最后一根稻草。

◆ 改动 2 & 3：Sinkhorn 和 sparse_attn

━━━━━━━━━━━━━━━━━━━━

这两个改动比较简单，一起说。

**hc_sinkhorn 20 次 -> 5 次迭代。** 4x4 的双随机矩阵，20 次完全过度收敛。microbench 测下来 7 次就够了（max_diff 1.7e-2），保守取 5 次。一行代码：

```python
sinkhorn_iters = min(sinkhorn_iters, 5)
```

hc_sinkhorn 在长 prompt 下占比不低——175 期的 profile 显示长 prompt 时它占了 27.6% 的 kernel 时间（短 prompt 只有 2.9%），因为 Sinkhorn 迭代次数和 token 数量无关，但随着 prompt 变长其他 kernel 被摊薄后它就凸显出来了。减到 5 次迭代，降到 9.9%。

**sparse_attn 向量化。** 把 Python for 循环逐位置遍历改成 `torch.einsum` + `gather` 全向量化。这个其实 175 期第七阶段就做完了，这里只是和 dot_scaled 一起跑了新的 benchmark 确认组合效果。

◆ 最终结果

━━━━━━━━━━━━━━━━━━━━

双机 TP=2，971 token prompt 的 profile 对比：

```text
                    175 期第七阶段    186 期        提速
─────────────────────────────────────────────────────
Prefill (971 tok)   14.6s            9.0s          1.62x
                    (66.6 tok/s)     (108 tok/s)
生成速度             ~0.3 s/tok       0.24 s/tok    1.25x
                                     (4.11 tok/s)
总 kernel 时间       36.655s          21.815s       1.68x

fp4_gemm 占比        37.5% / 21.2%   14.7%
hc_sinkhorn 占比     2.9% / 27.6%    9.9%
```

（175 期有两组数字——短 prompt 和长 prompt 的占比不同，因为不同 kernel 的 scaling 特性不一样。）

fp4_gemm 从最大的瓶颈变成了第二梯队。新的最大瓶颈是 fp8_gemm——这个我们暂时没改，因为 FP8 native dot 在 sm_121 上没有实际提速。

对实验平台来说，关键数字是 prefill：**9 秒完成一次 971 token 的 forward pass，提取全部 43 层的中间激活值。** 做对比实验时每组跑两次 prefill（两个不同的 prompt），18 秒出结果，够用了。

◆ 下一步：开灯/关灯对比实验

━━━━━━━━━━━━━━━━━━━━

平台搭完了（175 期），速度优化了（186 期），接下来终于可以做真正想做的事了。

计划是这样的：给模型同一个问题，但换不同的系统提示词（system prompt），对比 43 层每一层的激活值差异。

比如最简单的一组对比：

```text
Prompt A: "你是一个有帮助的助手。请回答：1+1等于几？"
Prompt B: "你是一个故意说错话的助手。请回答：1+1等于几？"
```

两次 prefill，各 9 秒，各拿到 43 层的 hidden state。然后逐层算余弦相似度——前几层应该几乎一样（都在做 token embedding），中间层开始分化（模型"理解"了不同的指令），最后几层差异最大（生成结果不同）。这条从接近 1.0 到逐渐下降的余弦相似度曲线，就是"系统提示词在模型内部哪一层开始生效"的可视化。

Anthropic 的情绪探针实验用的是类似方法——给不同情绪的 prompt，用线性探针检测内部向量方向。我们不训练探针，先做最简单的：只看激活值的统计特征变化。

这就是 175 期最后一段说的"开灯/关灯对比实验"——现在灯终于装好了，该开了。
