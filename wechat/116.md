【AI Agent的"算法"】LLM是怎么决定"回答问题"还是"调用工具"的？

2026 年，全世界都在谈 Agent。但你跟大部分谈 Agent 的人聊两句就会发现——他们连 LLM 是怎么"决定调用工具"的都说不清楚，以为模型里面有个神秘的"决策引擎"，以为 AI 真的会"执行"工具。

不是的。

前两篇写了怎么在鸿蒙上实现 AI Agent（ReAct 循环篇 https://mp.weixin.qq.com/s/jqve7BeE9vRUj-lECdiTrg ，MCP/Skill/RAG 篇 https://mp.weixin.qq.com/s/ix7n19cYy90EgJQzAZKO-w ），里面反复提到"LLM 自动选工具"、`finish_reason: tool_calls`。

但到底什么叫"自动选"？模型内部到底发生了什么？它是怎么决定"该直接回答"还是"该调工具"的？调哪个工具？传什么参数？

今天把这个问题拆成两层讲：**API 协议层**（发生了什么）和**模型训练层**（为什么能这样）。

先说结论：**LLM 不会"执行"任何东西。它只会"说话"——一个 token 一个 token 地往外蹦。所谓"调用工具"，是 LLM 输出了一段特定格式的 JSON，App 代码解析这段 JSON 去执行。**

━━━━━━━━━━━━━━━━━━━━

◆ 第一层：API 协议层——"发生了什么"

━━━━━━━━━━━━━━━━━━━━

**1. tools 数组 = 菜单**

App 调 LLM API 时，请求体里有个 `tools` 字段。这就是工具菜单——告诉 LLM"你有这些工具可以用"。

```json
{
  "model": "deepseek-chat",
  "messages": [...],
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "maps_weather",
        "description": "查询指定城市的天气信息",
        "parameters": {
          "type": "object",
          "properties": {
            "city": { "type": "string", "description": "城市名称" }
          },
          "required": ["city"]
        }
      }
    }
    // ... 还有 web_search、memory_write 等其他工具
  ]
}
```

每个工具有三样东西：**名字**（name）、**描述**（description）、**参数规格**（parameters，JSON Schema 格式）。LLM 看的就是这些。它靠 description 理解工具能干什么，靠 parameters 知道该传什么参数。

就像把菜单递给一个人——菜名、配料说明、价格都写在上面，他看完再点菜。

💡 注意：`tools` 不是写在 system prompt 里的，它是请求体里和 `messages` 平级的独立字段。但 API 提供商在后端大概率会把 `tools` 数组序列化后拼到上下文的某个位置——因为模型本身只认 token 序列，不认"JSON 字段"。对开发者来说分开传，对模型来说最终都是上下文里的一段文本。

而且——**每次 API 请求都要带 `tools` 数组**。不是"注册一次就行"，是每次 `/chat/completions` 调用都要完整地把工具菜单传过去。这意味着工具描述会吃 token——工具越多，每次请求的 token 开销越大。MCP/Skill/RAG 篇里接高德地图一下子来了 12 个工具，每个带 description + parameters schema，加起来几百 token。所以不能无脑往 tools 里塞一百个工具，得控制数量，或者按场景动态筛选。

还有一个容易忽视的问题：对话越长，工具描述在上下文里的"存在感"就越弱。同样 20 个工具，对话到第 30 轮，有些模型还能精准选工具，有些就容易"忘了"某些工具的存在。**200k 的上下文窗口装得下不代表看得清**——注意力分配的质量才是关键。

────────────────────

**2. LLM 的两种返回**

LLM 看完"菜单"和用户的问题后，只有两种选择：

**选择 A：直接回答**（我自己就能答）

```json
{
  "choices": [{
    "message": {
      "role": "assistant",
      "content": "北京是中国的首都，位于华北平原北部。"
    },
    "finish_reason": "stop"
  }]
}
```

**选择 B：调用工具**（我需要帮手）

```json
{
  "choices": [{
    "message": {
      "role": "assistant",
      "content": null,
      "tool_calls": [{
        "id": "call_abc123",
        "type": "function",
        "function": {
          "name": "maps_weather",
          "arguments": "{\"city\":\"北京\"}"
        }
      }]
    },
    "finish_reason": "tool_calls"
  }]
}
```

区别就在 `finish_reason`。`"stop"` = 模型觉得自己回答完了。`"tool_calls"` = 模型觉得需要调工具。

注意看选择 B：`content` 是 `null`，模型没说任何自然语言，而是在 `tool_calls` 里输出了一段结构化数据——我要用 `maps_weather` 工具，参数是 `{"city":"北京"}`。

💡 这里有个容易忽略的细节：`arguments` 的值是**字符串**，不是对象。`"{\"city\":\"北京\"}"` 是个 JSON 字符串，App 端需要 `JSON.parse()` 才能拿到真正的参数对象。为什么是字符串？因为 LLM 输出的就是一串文本 token，API 层把它放进了一个字符串字段里。

────────────────────

**3. App 端的 while 循环**

回忆 ReAct 循环篇的 ReAct 循环——App 端其实就是一个无脑 while：

```typescript
while (iterations < MAX_ITERATIONS) {
    const response = await client.sendMessage(apiMessages, tools)
    const choice = response.choices[0]

    if (choice.finishReason === 'tool_calls') {
        // 执行工具 → 结果塞回对话 → continue
    }

    // finish_reason === 'stop' → 返回给用户
    return textContent
}
```

**"自动选工具"的真相**：App 端没有任何判断逻辑，它不做决策。它只看 `finish_reason`——`tool_calls` 就执行工具，`stop` 就返回。**所有的"智能"都在 LLM 的输出里。App 就是个跑腿的。**

────────────────────

**4. tool_calls 的结构细节**

一次返回可以有**多个** tool_call（并行工具调用）：

```json
"tool_calls": [
  {
    "id": "call_001",
    "function": { "name": "web_search", "arguments": "{\"query\":\"北京天气\"}" }
  },
  {
    "id": "call_002",
    "function": { "name": "web_search", "arguments": "{\"query\":\"上海天气\"}" }
  }
]
```

LLM 同时要查北京和上海的天气，一次返回两个调用。每个 tool_call 有唯一的 `id`——后面返回结果时要用这个 id 对应回去。

────────────────────

**5. tool 消息：把结果喂回去**

App 执行完工具后，结果以 `role: "tool"` 消息追加到对话历史：

```json
{
  "role": "tool",
  "tool_call_id": "call_001",
  "content": "北京今天晴，气温12-22度，东北风3级"
}
```

`tool_call_id` 关联到具体是哪个调用的结果。然后把更新后的对话历史再发给 LLM——这就是 while 循环的 `continue`。

LLM 拿到工具结果后再次做选择：还需要调别的工具？还是可以组织答案了？如果它觉得信息够了，就输出自然语言回答，`finish_reason: "stop"`，循环结束。

────────────────────

**整个流程的类比**

把这个流程想象成一个只会写字条的参谋和一个只会跑腿的勤务兵：

| 角色 | 对应 | 做的事 |
|:---|:---|:---|
| 参谋 | LLM | 只会写字条（输出 token） |
| 勤务兵 | App（AgentCore） | 只会跑腿（执行工具） |
| 资源清单 | tools 数组 | 参谋手边有哪些渠道可用 |
| 点菜单 | tool_calls | 参谋写的字条："去气象局问北京天气" |
| 回报 | tool 消息 | 勤务兵跑回来："报告，晴，12-22度" |
| 最终答案 | content | 参谋再写一张字条，这次是给用户的 |

参谋不会自己去气象局。勤务兵不知道该不该去气象局。**参谋负责判断，勤务兵负责执行**——两边缺一不可。

━━━━━━━━━━━━━━━━━━━━

◆ 第二层：模型训练层——"为什么能这样"

━━━━━━━━━━━━━━━━━━━━

上面讲的是协议层——LLM 输出了 JSON，App 去执行。但问题来了：**LLM 怎么就"学会"输出这种格式的 JSON 了？它怎么知道什么时候该回答、什么时候该调工具？**

────────────────────

**1. 一切都是 next token prediction**

先理解一个根本事实：**LLM 的本质就是一个 token 预测器。**

它不知道什么是"工具"，不知道什么是"天气"，甚至不知道自己在"对话"。它只知道一件事：**在当前上下文里，接下来最可能出现的 token 是什么。**

当你问 LLM "北京今天天气怎么样"，同时递了一个包含 `maps_weather` 工具的 tools 数组。LLM 看到这个上下文后开始预测：

- 接下来的 token 是 `"北` (开始用自然语言回答) 的概率：30%
- 接下来的 token 是 `{` (开始输出 JSON 调用工具) 的概率：70%

概率往 `{` 那边倾斜了——于是模型输出 `{`。

一旦输出了 `{`，后面就被"引导"到 JSON 格式上了——`"tool_calls"` → `[` → `{` → `"function"` → ... 就像你手写邮件，写了 "Dear " 之后大概率会接人名或 "Sir"，不会突然蹦出一段中文。上下文决定了后续 token 的概率分布。

💡 重点来了：**没有一个独立的"决策模块"在做"调不调工具"的判断。** 没有 if-else，没有决策树。全是 token 概率。

────────────────────

**2. Function Calling 的 SFT 训练**

那这个概率分布是怎么来的？靠**训练**。

具体来说，是 SFT（Supervised Fine-Tuning，监督微调）阶段。在预训练之后，模型会被喂大量格式化的训练数据，教它"看到工具描述 + 用户问题时，应该输出什么样的 JSON"。

训练样本大概长这样：

```
[系统提示]
你是一个助手。可用工具：
- weather(city: string): 查询城市天气
- calculator(expression: string): 计算数学表达式

[用户]
北京今天天气怎么样

[助手]
{"tool_calls": [{"function": {"name": "weather", "arguments": {"city": "北京"}}}]}

[工具结果]
北京今天晴，12-22度

[助手]
北京今天是晴天，气温在12到22度之间，适合出行。
```

成千上万条这样的训练数据喂下去，模型学到的是**模式**：

- 用户问天气 + tools 里有 weather → 输出调 weather 的 JSON
- 用户问 1+1 + tools 里有 calculator → 输出调 calculator 的 JSON
- 用户问"你好" + tools 里不管有什么 → 直接回答，不调工具
- 用户问天气 + tools 里**没有** weather → 用自己的知识回答（可能不准）

**它不是"理解"了工具的功能，而是学会了"在这种上下文模式下，输出那种格式的 token 序列"。**

────────────────────

**3. 名字的历史：Function Calling vs Tool Use**

你可能见过两种叫法：

| 叫法 | 来源 | 时间 |
|:---|:---|:---|
| Function Calling | OpenAI | 2023 年 6 月首次发布 |
| Tool Use | Anthropic (Claude) | 2024 年 |

本质完全一样：让 LLM 在适当时候输出结构化 JSON 而不是自然语言。现在行业标准化了，OpenAI 自己也从 `functions` 参数改成了 `tools` 参数。ReAct 循环篇里用的就是 `tools`。

────────────────────

**4. token 层面到底发生了什么**

再深入一层。LLM 内部有一个叫 **softmax** 的层，它把模型对所有可能 token 的"偏好值"（logits）转换成概率分布。

当 LLM 处理完上下文（用户问题 + tools 数组），到了该输出第一个 token 的时候：

```
token 候选          logit (偏好值)     softmax 概率
─────────────────────────────────────────────────
{                     4.2              0.65
北                    2.1              0.08
我                    1.8              0.06
根                    1.5              0.04
...                   ...              ...
```

`{` 的概率最高 → 模型选 `{` → 开始输出 JSON。

接下来预测第二个 token，这时上下文多了一个 `{`：

```
token 候选          softmax 概率
─────────────────────────────
"                    0.92      ← 几乎确定是双引号
n                    0.01
t                    0.01
...
```

然后第三个 token，上下文是 `{"` ：

```
token 候选          softmax 概率
─────────────────────────────
tool                 0.45
function             0.30
name                 0.15
...
```

就这样一个 token 一个 token 地蹦，蹦出一段完整的 tool_calls JSON。

**整个过程没有任何"决策引擎"在运作。** 只有一个概率预测器，在上下文的引导下，一步一步走向 JSON 格式的输出。

────────────────────

**5. 为什么会"幻觉"工具调用**

理解了上面的原理，就能理解一些 Agent 开发中的常见坑：

**坑 1：模型调了一个不存在的工具**

tools 数组里有 `web_search` 和 `memory_write`，但模型输出了 `{"name": "get_stock_price"}`——一个菜单上没有的菜。

为什么？因为模型的训练数据里见过 `get_stock_price` 这种工具，它的 token 概率分布里残留了这个模式。**tools 数组只是上下文的一部分，不是硬约束。**模型"看到"了菜单，但它也可能"记起"以前见过的菜。

这就是 ReAct 循环篇里 `unknown tool` 错误处理存在的原因：

```typescript
} else {
    result = { toolCallId: callId, content: `Error: unknown tool "${toolName}"`, isError: true }
}
```

**坑 2：参数类型传错了**

JSON Schema 定义 `city` 是 string，但模型传了个数组 `["北京", "上海"]`。Schema 约束是靠训练学来的，不是硬编码在推理过程里的——模型大多数时候能遵守，但不是 100%。

**坑 3：该调工具的时候不调**

你问"北京今天天气"，tools 里明明有 weather 工具，模型却直接编了个答案："北京今天晴天，25度"。因为 token 概率没往 `{` 那边走，而是走向了自然语言。尤其当上下文很长、对话轮次很多的时候，工具描述的"影响力"被稀释了。

💡 这就是为什么 ReAct 循环篇里提到"聪明的模型暗示一下就够了，便宜的模型得写明确点"——system prompt 写得越清晰，tools 描述越准确，token 概率往正确方向走的可能性就越大。

────────────────────

**6. Constrained Decoding：从"大概率正确"到"保证正确"**

既然 token 预测有不确定性，有没有办法**强制**模型输出合法格式？

有。叫 **Constrained Decoding**（约束解码）。

原理很直接：在 token 采样时，把不符合目标格式的 token 概率直接设为 0。

比如 JSON Mode（`response_format: { type: "json_object" }`）：

```
正常 softmax 输出:
  {  → 0.65
  北 → 0.08
  我 → 0.06

JSON 约束后:
  {  → 1.00    ← 唯一合法选项
  北 → 0.00    ← 不符合 JSON 语法，强制归零
  我 → 0.00
```

模型还是在预测 token，只是**选择范围被缩小了**。就像考试只能从 ABCD 里选，不能写简答。

Function Calling 可以看作一种**隐式的** Constrained Decoding——API 提供商在后端对工具调用的输出格式做了约束，确保返回的 JSON 结构合法。但具体实现各家不同，有些是硬约束（DeepSeek），有些是软引导（早期 OpenAI）。

这也是为什么同样的 tools 数组，不同模型的工具调用可靠性差距很大——**不只是模型"聪明"程度不同，还因为各家后端的约束策略不同。**

◆ 串起来：从用户提问到工具执行的完整链路

━━━━━━━━━━━━━━━━━━━━

把两层合在一起，完整走一遍"北京今天天气怎么样"的全流程：

```
1. 用户输入: "北京今天天气怎么样"

2. App 构造请求:
   messages: [系统提示, 用户消息]
   tools: [web_search, memory_write, maps_weather]
       ↓
3. LLM 处理上下文:
   softmax 层计算概率 → { 的概率 > 北 的概率
   → 开始输出 JSON token 序列
   → 完整输出: {"name":"maps_weather","arguments":"{\"city\":\"北京\"}"}
       ↓
4. API 返回:
   finish_reason: "tool_calls"
   tool_calls: [{id:"call_001", function:{name:"maps_weather", arguments:...}}]
       ↓
5. App 检查 finish_reason === "tool_calls"
   → JSON.parse(arguments) → {city: "北京"}
   → 调用高德地图 API（通过 MCP 或直接 HTTP）
   → 拿到结果: "晴，12-22度，东北风3级"
       ↓
6. App 追加 tool 消息:
   {role:"tool", tool_call_id:"call_001", content:"晴，12-22度，东北风3级"}
       ↓
7. App 再次调 LLM API（带完整对话历史）
       ↓
8. LLM 看到工具结果，这次:
   softmax 层计算概率 → 北 的概率 > { 的概率
   → 输出自然语言: "北京今天晴天，气温12-22度，东北风3级，适合出行。"
       ↓
9. API 返回:
   finish_reason: "stop"
       ↓
10. App 检查 finish_reason === "stop" → 返回给用户
```

**两次 API 调用，一次工具执行。** LLM 全程只干了一件事：预测 token。App 全程只干了一件事：看 `finish_reason` 决定走哪个分支。

━━━━━━━━━━━━━━━━━━━━

◆ 为什么理解这个很重要

━━━━━━━━━━━━━━━━━━━━

理解了"LLM 只是 token 预测器"这个本质之后，很多 Agent 开发中的经验就有了理论解释：

**1. 为什么 system prompt 写得好，Agent 更可靠？**

因为 system prompt 是上下文的一部分，直接影响 token 概率分布。写"遇到天气问题请使用 maps_weather 工具"，等于在概率分布上给 `{` 加了权重。

**2. 为什么 Agent 有时候"忘了"用工具？**

不是它"决定不用"，是长上下文稀释了 tools 描述的影响力，token 概率没往 `{` 走。对话越长越容易出现。

**3. 为什么换模型后工具调用行为会变？**

每个模型的 SFT 训练数据不同。DeepSeek 的 Function Calling 训练数据和 Claude 的不一样，GPT 的又不一样。模型不同 → 概率分布不同 → 同样的 tools 数组，行为不一样。

**4. 为什么工具的 description 很重要？**

description 是 LLM "理解"工具用途的唯一依据。写"查询天气"比写"weather api"好得多——前者的 token 模式更容易和"天气怎么样"这类问题匹配上。

**5. 为什么 ReAct 循环篇里工具执行失败要把错误返回给 LLM？**

因为 LLM 拿到错误信息后，上下文变了，token 概率分布也变了。它可能换个工具试，可能换个参数重来。如果 App 自己处理错误不告诉 LLM，就浪费了模型的"纠错"能力。

────────────────────

一句话收尾：**LLM 没有"决策引擎"。它是一个 token 预测器，被训练成了在看到工具菜单时能输出正确的点菜单。App 拿着这个点菜单去后厨执行。所谓"智能"，是训练数据里的模式匹配，通过 next token prediction 涌现出来的。**

━━━━━━━━━━━━━━━━━━━━

◆ 术语速查

━━━━━━━━━━━━━━━━━━━━

| 术语 | 解释 |
|:---|:---|
| Function Calling | OpenAI 最早提出的概念，让 LLM 输出结构化 JSON 来"调用函数"，现已标准化为 tools 机制 |
| SFT | Supervised Fine-Tuning，监督微调。用标注好的"问题→正确输出"数据对训练模型 |
| Next Token Prediction | LLM 的核心机制：给定上下文，预测下一个最可能出现的 token |
| Softmax | 神经网络中的概率化层，把模型对各 token 的偏好值（logits）转换成概率分布 |
| finish_reason | LLM API 返回字段。`stop` = 回答完毕，`tool_calls` = 想调用工具 |
| JSON Schema | 描述 JSON 数据结构的标准格式，用来定义工具参数的类型和约束 |
| Constrained Decoding | 约束解码，在 token 采样时屏蔽不符合目标格式的 token，强制输出合法结构 |
| tool_call_id | 工具调用的唯一标识符，用来将执行结果关联回具体的调用请求 |
