【DeepSeek V4推理】张量并行改流水线并行：能跑，但不是更快

━━━━━━━━━━━━━━━━━━━━

◆ 前情回顾

━━━━━━━━━━━━━━━━━━━━

昨天的 173 期（ https://mp.weixin.qq.com/s/Gis8yrgPSQJPsWfCrDJb1w 序号很靠前，原因是五一长假前就写出来了，结果因为假期中突然发现了英文乱码的bug就忙着debug导致这篇一直没发出来 ）讲的是 TP（Tensor Parallel，张量并行）：同一个 Transformer layer 里的矩阵乘法怎么横向切到两台 DGX Spark 上，哪些地方是 Column Parallel，哪些地方是 Row Parallel，AllReduce 到底发生在哪。

173 期的核心结论是：TP 不是“把模型随便分成两半”，而是每一层都在两台机器上同时算，只是权重张量按维度切开。算完某些 Row Parallel 矩阵后，两台机器必须通过 NCCL AllReduce 把部分和合起来。也就是说，TP 的计算并行度很直接，但跨机器通信也很频繁。

这期就接着昨天的问题往下走：既然 TP 每一层都要跨机器通信，那能不能换成 PP（Pipeline Parallel，流水线并行）？让第一台机器跑前半层，第二台机器跑后半层，少做每层 AllReduce，会不会更快？

185 期（ https://mp.weixin.qq.com/s/dEc-wD0JcM5WhKicOUR1pQ ）把 DGX Spark 双机上的 DeepSeek V4 Flash 推理速度从 4 tok/s 拉到了 12 tok/s：前 42 层 MoE 走 Marlin，最后一层回退到 Consumer-DeepGEMM 的 `tl.dot_scaled` 路径。

这个方案的关键不是“Marlin 被修好了”，而是我们找到了它在 sm_121 上还能稳定撑住的边界。0 到 41 层走 Marlin，layer 42 走 Consumer-DeepGEMM，输出正确，速度接近纯 Marlin。

当时的并行方式是 TP=2，也就是 Tensor Parallel，张量并行。两台 DGX Spark 各持有每一层权重的一半，每一层都一起算，然后把结果合并。

在这个基础上，这次想试另一个方向：

**能不能把 TP=2 改成 PP=2？**

也就是 Pipeline Parallel，流水线并行。让第一台机器跑前半层，第二台机器跑后半层。直觉上看，这样可以少做很多跨机器 all-reduce，可能更适合两台机器之间的网络。

结果先说：

**能跑。**

**需要一个很小的程序修改。**

**但单请求生成速度没有变快，实测大概 10-11 tok/s，比 TP=2 的 12 tok/s 略慢。**

这期记录的就是这次 TP 改 PP 的完整过程。它很适合作为一个反直觉案例：通信少了，不等于端到端一定更快；模型能声明 `SupportsPP`，也不等于所有执行路径都自然稳定。

━━━━━━━━━━━━━━━━━━━━

◆ TP 和 PP 到底差在哪

━━━━━━━━━━━━━━━━━━━━

先把概念说清楚。

TP=2 是把每一层里的大矩阵横着切开：

```text
每一层：

rank0 算一半矩阵
rank1 算另一半矩阵
两边结果合并
进入下一层
```

对 DeepSeek V4 这种 43 层 MoE 模型来说，TP=2 的特点是：

- 每台机器都有 43 层
- 每一层的权重张量只拿一半
- 每一层都需要跨机器通信
- 单个 token 经过每一层时，两台机器同时干活

PP=2 是把模型按层竖着切开：

```text
rank0：layers 0-21
rank1：layers 22-42

token 先经过 rank0 的前半个模型
再把 hidden states 传给 rank1
rank1 跑后半个模型
最后输出 logits
```

PP=2 的特点是：

- 每台机器只放一部分层
- 但每层的权重是完整的，不再做张量切分
- 跨机器通信从“每层都合并一次”变成“阶段之间传 hidden states”
- 单请求 decode 时，前半模型和后半模型有明显的串行关系

很多人第一眼会以为 PP=2 更省显存，因为每台机器只放一半层。这个直觉只对了一半。

TP=2 是“每层都在，但每层只有半个张量”；PP=2 是“只有半数层，但每层是完整张量”。两者算下来，每台机器承载的模型权重都接近半个模型。

这次实测的 PP=2 显存也验证了这一点：

| rank | 模型权重 | KV cache |
|------|----------|----------|
| PP rank0 | 约 74.8 GiB | 约 21-23 GiB |
| PP rank1 | 约 71.9 GiB | 约 20-24 GiB |

所以 PP=2 不是“显存魔法”。它真正改变的是计算和通信的形状。

TP 像两个人一起切每一道菜，每道菜切完都要合盘。PP 像一个人做前菜，另一个人做后菜，中间递一次盘子。盘子递得少了，但一道菜从开始到上桌，不一定更快。

━━━━━━━━━━━━━━━━━━━━

◆ 第一次试跑：能加载，但第二个请求卡死

━━━━━━━━━━━━━━━━━━━━

一开始的启动方式很自然：

```bash
vllm serve /root/.cache/huggingface/deepseek-v4-flash \
  --tensor-parallel-size 1 \
  --pipeline-parallel-size 2 \
  --distributed-executor-backend ray \
  --gpu-memory-utilization 0.80 \
  --kv-cache-dtype fp8 \
  --max-model-len 4096 \
  --enforce-eager \
  --moe-backend marlin
```

意思是：

- TP 从 2 改成 1
- PP 从 1 改成 2
- 分布式后端仍然用 Ray
- MoE 后端仍然用 Marlin
- 最后一层仍然通过环境变量回退到 Consumer-DeepGEMM

DeepSeek V4 Flash 这边本身不是完全不支持 PP。模型类里已经有 `SupportsPP`，vLLM 能把 43 层切成两个 pipeline stage。日志里能看到层切分大致是：

```text
PP rank0: 22 layers
PP rank1: 21 layers
```

模型能加载，worker 能起来，第一个请求也能返回。

然后第二个请求卡死。

不是输出慢一点，是服务整体进入一种很不健康的状态：请求长时间不返回，`/health` 也开始超时。这个现象和 CUDA kernel 算错不一样，更像执行图或调度层卡住。

关键日志是这一句：

```text
Using RayPPCommunicator (which wraps vLLM _PP GroupCoordinator) for Ray Compiled Graph communication.
```

默认 Ray executor 在 PP 场景下走的是 Ray compiled graph + `RayPPCommunicator` 这条路径。它的设计目标是把分布式执行图编译好，减少调度开销。但在这次双机 PP=2 + eager + DeepSeek V4 的组合里，它第一次请求能过，第二次请求复用执行图时会挂住。

这不是 DeepSeek V4 的权重问题，也不是 Marlin 的数值问题。它发生在请求调度和 pipeline 通信层。

结论：**默认 Ray PP 路径在这个组合下不可用。**

━━━━━━━━━━━━━━━━━━━━

◆ 第一个修复：换 Ray Executor V2

━━━━━━━━━━━━━━━━━━━━

解决这个卡死，不需要改模型代码。vLLM 里还有一条新的 Ray executor 路径，可以通过环境变量打开：

```bash
VLLM_USE_RAY_V2_EXECUTOR_BACKEND=1
```

打开之后，日志里能看到执行方式变了：

```text
Asynchronous scheduling is enabled
RayWorkerProc
Worker_PP0
Worker_PP1
```

这条路径不再走前面那个会卡住的 Ray compiled graph PP 通信方式。换完以后，第二个请求不再卡死，服务能连续返回。

到这里，PP=2 的第一个大坑解决了。

但还没有结束。

请求确实返回了，可输出是坏的：开头可能像是正常答案，后面开始出现乱码、重复、符号循环。这种现象非常熟悉——185 期全 Marlin 43 层时就是这个味道。

也就是说，PP 调度层现在稳定了，但数值路径又回到了老问题：最后一层回退没有真正生效。

━━━━━━━━━━━━━━━━━━━━

◆ 第二个坑：最后一层回退没打中

━━━━━━━━━━━━━━━━━━━━

185 期的混合方案依赖一个环境变量：

```bash
VLLM_MXFP4_MARLIN_DEEPGEMM_LAYERS=42
```

含义是：layer 42 不走 Marlin，强制走 Consumer-DeepGEMM。

TP=2 时它是生效的。启动日志会打印：

```text
MXFP4 layer 42 forced to DEEPGEMM_MXFP4 by VLLM_MXFP4_MARLIN_DEEPGEMM_LAYERS=42
```

但 PP=2 后，这个打印一开始没出现。

原因很小，也很典型：我们之前在 `mxfp4.py` 里用正则从 layer name 里解析层号，正则写得偏窄。

旧写法只匹配这种名字：

```python
\.layers\.(\d+)\.ffn\.experts$
```

但 PP 路径里实际传进来的名字可能是：

```text
model.layers.42.ffn.experts
```

也可能在不同封装层里更早停在：

```text
model.layers.42.ffn
```

于是环境变量明明传进容器了，layer 42 也确实在 PP rank1 上，但解析层号失败，回退逻辑没命中。最后一层继续走 Marlin，全模型又回到 43 层 Marlin 的坏路径，输出自然会乱码。

修复就是把正则放宽：

```python
old:
\.layers\.(\d+)\.ffn\.experts$

new:
(?:^|\.)layers\.(\d+)\.ffn(?:\.experts)?$
```

同时加了一个调试开关：

```bash
VLLM_MXFP4_DEBUG_LAYER_NAMES=1
```

用来打印每一层拿到的 `layer_name`、解析出的 `layer_idx`、当前后端和环境变量。确认 PP rank1 上的 `model.layers.42.ffn.experts` 被正确识别后，日志终于出现：

```text
MXFP4 layer 42 forced to DEEPGEMM_MXFP4 by VLLM_MXFP4_MARLIN_DEEPGEMM_LAYERS=42
```

到这里，PP=2 的第二个坑解决。

所以“改成 PP 需要改程序吗？”答案是：

**vLLM 的模型并行主体不用改。DeepSeekV4ForCausalLM 本来支持 PP。**

**但我们自己的 mixed backend 回退逻辑需要小改。**

具体来说，就是 `mxfp4.py` 里 layer index 的解析要兼容 PP 路径下的 layer name。否则 layer 42 回退打不中，输出会坏。

━━━━━━━━━━━━━━━━━━━━

◆ 实测速度

━━━━━━━━━━━━━━━━━━━━

稳定后做了几组请求。

| Prompt | 输出 tokens | 时间 | tok/s | 结果 |
|--------|-------------|------|-------|------|
| `17 * 23 = ? 只回答结果。` | 2 | 0.27s | 7.37 | `391` |
| Python 小代码 | 28 | 2.62s | 10.70 | 正确 |
| 200 字介绍万里长城 | 72 | 6.72s | 10.71 | 中文正常 |
| English quicksort | 180 | 16.04s | 11.22 | 英文正常 |

还有一个细节：第一次请求会很慢，曾经测到 28 秒左右。这不是稳态 decode 速度，而是服务刚启动后的 warmup、CUDA graph/worker 初始化、缓存准备等开销混在一起。连续请求之后，同样的数学题可以 0.34 秒左右返回。

所以更公平的结论是：

**PP=2 稳态生成速度大概 10-11 tok/s。**

对比 185 期的 TP=2 混合后端：

| 方案 | 并行方式 | 后端 | 单请求生成速度 | 输出 |
|------|----------|------|----------------|------|
| 185 期 | TP=2 | 42 层 Marlin + layer42 Consumer-DeepGEMM | 约 12 tok/s | 正确 |
| 本期 | PP=2 | 42 层 Marlin + layer42 Consumer-DeepGEMM | 约 10-11 tok/s | 正确 |

也就是说，PP=2 确实能正常返回，中英文和代码都正常；但它没有超过 TP=2。

这和最初的直觉有点相反。

━━━━━━━━━━━━━━━━━━━━

◆ 为什么通信少了，反而没更快

━━━━━━━━━━━━━━━━━━━━

TP=2 的缺点很明显：每一层都要跨机器合并结果。

对 43 层模型来说，这听起来很贵。PP=2 把跨机器通信变成 stage 之间传 hidden states，通信次数看起来少很多。那为什么没更快？

核心原因是：

**单请求 decode 场景下，PP 的流水线填不满。**

流水线并行真正擅长的是大 batch 或高并发。比如同时有很多请求，每个请求都被切成 microbatch，rank0 跑第一个 microbatch 的前半段时，rank1 可以跑上一个 microbatch 的后半段。这样两段 pipeline 都有活干。

但我们现在测的是最常见的交互式单请求 decode：

```text
生成第 N 个 token：

rank0 跑 layers 0-21
把 hidden states 传给 rank1
rank1 跑 layers 22-42
输出 token

生成第 N+1 个 token：

rank0 再跑 layers 0-21
再传给 rank1
rank1 再跑 layers 22-42
输出 token
```

这里 rank0 和 rank1 不是一直同时满载。一个 token 的完整路径有明显前后依赖。rank1 必须等 rank0 的 hidden states，rank0 跑完以后也不能直接完成这个 token 的后半段。

这就是 pipeline bubble，流水线气泡。

TP=2 虽然每层都有 all-reduce，但每一层里两台机器同时算同一个 token 的同一层。它的通信更频繁，但并行度更直接。

这里还有一个 DGX Spark 特有的观察。

DGX Spark 很适合做“大显存桌面机器”，但它不是 H100/H200 那种高带宽 HBM 服务器。GB10 走的是统一内存，容量大，带宽相对数据中心 GPU 慢很多。换句话说，它可能是目前最慢的一类“大显存/大内存推理主机”：能把 280B FP4 模型塞进去，但每个 token 都在被内存带宽拖着走。

这会改变我们对“跨机通信”的直觉。

PP 主要省的是网线通信。TP 每一层都要跨机器 all-reduce，PP 把这种频繁通信变成阶段之间传 hidden states。按直觉，网线慢，少走网线就该更快。

但在 DGX Spark 上，情况反过来了：**网线虽慢，内存更慢。**

两台 Spark 之间的 CX7/RoCE 200Gbps 当然远不如 HBM，但 DGX Spark 的本地统一内存也不是 HBM。对这个模型来说，很多时间本来就在等本地内存把权重喂给计算单元。于是 TP 的 AllReduce 虽然有成本，却没有大到完全主导；PP 省掉一部分网线通信后，反而暴露出单 token 的 stage 串行和 pipeline bubble。省下来的通信，不够抵消流水线气泡。

所以 PP 想明显赢，前提应该是：**网线瓶颈远大于本地内存瓶颈。** 也就是跨机器通信慢到离谱，而每台机器自己读内存/算前向相对没那么慢。只有这种情况下，少走网线才可能压过 PP 的串行代价。

举个猜测场景：如果是多台 Mac 做大模型推理，机器之间不是 CX7/RoCE，而是通过 USB/普通以太网之类的链路拼起来，那网络可能真的比本地统一内存慢得多。那种环境下，PP 可能比 TP 更有意义——因为 TP 每层 all-reduce 会被低速链路拖死，PP 少通信的收益才足够大。

但 DGX Spark 双机不是这个形态。它的本地内存已经慢，网线相对没有慢到足以让 PP 翻盘。这也是这次实验最有意思的地方之一：双机推理的瓶颈不是一句“网络慢”就能解释完，要看网络慢到什么程度，也要看本地内存到底有多慢。

所以这次结果可以理解为：

- PP 减少了每层 all-reduce
- 但引入了单 token 的 stage 串行
- DGX Spark 的本地统一内存本身就慢，CX7/RoCE 相对它没有慢到完全主导
- PP 只有在“网络瓶颈远大于内存瓶颈”的机器组合上才更可能明显收益
- Ray V2 executor 还有调度开销
- `--enforce-eager` 下没有吃到更多图优化
- 两台 DGX Spark 之间虽然是 CX7/RoCE，但 stage handoff 仍然不是免费的

最后合在一起，PP=2 没有超过 TP=2。

这不是 PP 不好，而是这个测量场景不适合它展示优势。

PP 的优势更可能出现在：

- batch 更大
- 多请求并发更高
- pipeline microbatch 能排满
- 每个 stage 的计算时间更均衡
- executor 对 PP 的调度更成熟

现在这次测试只是单请求交互式生成。这个场景下，TP=2 仍然更合适。

━━━━━━━━━━━━━━━━━━━━

◆ 这次到底修了什么

━━━━━━━━━━━━━━━━━━━━

这次从 TP=2 改 PP=2，表面上只是启动参数从：

```bash
--tensor-parallel-size 2
--pipeline-parallel-size 1
```

改成：

```bash
--tensor-parallel-size 1
--pipeline-parallel-size 2
```

但真正让它可用的是三个条件同时满足：

**第一，使用 Ray Executor V2。**

```bash
VLLM_USE_RAY_V2_EXECUTOR_BACKEND=1
```

否则默认 Ray PP compiled graph 路径第一个请求可能能过，第二个请求会卡死。

**第二，保留 layer 42 Consumer-DeepGEMM 回退。**

```bash
VLLM_MXFP4_MARLIN_DEEPGEMM_LAYERS=42
```

否则全 43 层 Marlin 在 sm_121 上仍然会乱码。这一点和 185 期完全一致。

**第三，修正 layer name 解析。**

PP 路径下 layer name 的形态和 TP 路径不完全一样。旧正则太窄，导致 layer 42 回退打不中。修成：

```python
(?:^|\.)layers\.(\d+)\.ffn(?:\.experts)?$
```

之后才真正命中 `model.layers.42.ffn.experts`。

所以这次不是“大改程序支持 PP”，而是“PP 主体可用，但混合后端补丁要兼容 PP 路径”。

这个区别很重要。

如果是模型本身不支持 PP，那要改的是模型 forward、layer partition、KV cache、logits processor 等一整套东西。

但这次不是。DeepSeek V4 的 PP 结构已经在 vLLM 里。我们碰到的问题分别在：

- Ray PP executor 默认路径
- 自定义 MXFP4 mixed backend 的 layer 识别
- sm_121 上 Marlin 最后一层数值边界

它们都在“部署组合和补丁适配”这一层，不在模型结构本身。

━━━━━━━━━━━━━━━━━━━━

◆ 结论

━━━━━━━━━━━━━━━━━━━━

这次实验的最终结论很朴素：

**DeepSeek V4 Flash 在双 DGX Spark 上可以从 TP=2 改成 PP=2。**

**要稳定运行，需要打开 Ray Executor V2。**

**要输出正确，需要保留 layer 42 Consumer-DeepGEMM 回退，并修正 PP 路径下的 layer name 解析。**

**单请求生成速度约 10-11 tok/s，没有超过 TP=2 的 12 tok/s。**

但要补一个前提：**PP 不是这类模型生产级部署里的主流选择。**

生产环境如果有足够多的数据中心 GPU 和高速互联，通常会优先用 TP（Tensor Parallel）、EP（Expert Parallel）、DP（Data Parallel）这些更容易吃满硬件的并行方式。它们的目标是吞吐、并发、调度效率和集群利用率。PP 在大模型训练里很常见，但在这种在线推理场景里，尤其是单请求 decode，很容易被流水线气泡拖住。

这次之所以还要试 PP，本质上是因为 DGX Spark 这种机器很特殊：它像一台家用/桌面级的大内存主机，显存意义上的容量够大，但带宽和互联都不是生产级 H100 集群那种形态。家用电脑、多台桌面机器、Mac 这类设备做大模型推理时，经常不是为了“最优部署”，而是为了“能不能把模型拆开塞进去”。在这种语境下，PP 才会变成一个值得测试的方案。

所以 PP=2 目前不是生产默认方案。它更像一个基线：证明这条路能跑，证明 vLLM + DeepSeek V4 + 双机 PP 的组合可以稳定返回，也给后续高并发、batch、pipeline microbatch 调优留了入口。

如果目标是现在就获得最快的交互式单请求速度，还是 185 期的 TP=2 混合后端更合适。

如果目标是研究“在多请求并发下，两台 Spark 怎么把流水线吃满”，那 PP=2 值得继续挖。

这次最有价值的地方，不是 10 tok/s 这个数字，而是把问题边界分清了：

- 卡死是 Ray executor 路径问题
- 乱码是 layer 42 回退没命中
- 不够快是单请求 pipeline bubble
- 不需要重写 DeepSeek V4 的 PP 支持

工程里经常就是这样。一个看起来像“并行策略”的问题，拆开以后其实是调度、数值、命名、启动参数四件事叠在一起。

把它们一件件剥开，系统才会重新变得可理解。
