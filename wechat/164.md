◆ 编译器类比：从"每次重新搜"到"编译一次，查询多次"

━━━━━━━━━━━━━━━━━━━━

160 期我们建立了一个基本框架：所有 Agent 记忆方案本质上都在做三件事——**压缩、索引、遗忘**。Mem0、Graphiti、Cognee、GraphRAG、Evermind，各有各的取舍，但底层模式是相同的：原始信息进来，经过某种处理，存入某种数据结构，等用户提问时再捞出来。

Karpathy 的提案跳出了这个框架。他用了一个程序员一听就懂的类比：**编译器**。

想象你有一堆 C 语言源文件。每次运行程序，你有两种选择：

**方案 A（解释器模式）**：每次运行时，从头读源代码，逐行解释执行。这就是 RAG——每次用户提问，从头去原始文档里搜索，找到相关片段，临时拼装出答案。

**方案 B（编译器模式）**：先花一次时间把源代码编译成可执行文件，以后每次运行直接跑编译好的二进制。这就是 LLM Wiki——先让 LLM 把原始资料"编译"成结构化的 wiki 页面，以后查询直接查 wiki。

对应关系是这样的：

- **Raw Sources（原始文档）= 源代码**。你的文章、论文、会议记录、对话历史，都是"源文件"。LLM 读但不改它们。
- **LLM 编译 = 将原始资料转化为 wiki 页面**。LLM 读完一篇新文档后，不只是存个向量，而是提取关键信息，写成摘要页、实体页、概念页，并更新已有页面的交叉引用。
- **Wiki Pages = 编译产物**。你实际查询和交互的知识库，是一组互相链接的 Markdown 文件。
- **Lint = 事实核查与维护**。定期让 LLM 检查 wiki 的健康状况——找矛盾、标记过时信息、补缺失的交叉引用。
- **Query = 查编译后的 wiki**，而不是每次从头检索原始文档。

Karpathy 的核心论点是：**RAG 每次查询都在"重新发现"知识，没有积累。** 你问一个需要综合五篇文档才能回答的问题，RAG 每次都要重新找到这五篇、重新拼装——昨天拼过一次，今天再拼一次，什么都没留下。而 wiki 模式在 ingest 阶段就完成了综合，知识是**复利式增长**的。每多灌一篇文档进来，wiki 不只是多了一页，还可能触发十几个已有页面的更新。

◆ 三层架构：一个文件夹就能跑

━━━━━━━━━━━━━━━━━━━━

Karpathy 的设计出奇地简单。没有数据库，没有向量引擎，没有图谱存储——就是文件夹里的 Markdown 文件。

**【第一层：Raw Sources】**

一个文件夹，放你的原始资料。文章、PDF、会议记录、聊天记录，什么都行。LLM 只读不写。这是你的"源代码"，不可变。

**【第二层：Wiki】**

另一个文件夹，全是 LLM 生成和维护的 Markdown 文件。包括：

- 摘要页（某篇文章的要点提取）
- 实体页（某个人、某家公司、某个概念的专属页面）
- 综合页（跨多个来源的主题综合）
- index.md（全 wiki 的目录，每个页面一行摘要加链接）
- log.md（操作日志，记录什么时候导入了什么、更新了什么）

页面之间用 wikilink 互相链接。用 Obsidian（一个本地 Markdown 笔记工具，支持双向链接和图谱可视化）打开，就是一张知识图谱。

**【第三层：Schema】**

一个配置文件（比如 CLAUDE.md），告诉 LLM 该怎么组织 wiki——命名规范、页面结构、工作流程。相当于编译器的配置。

整个 ingest 流程：新文档丢进 raw 文件夹 -> LLM 读完讨论要点 -> 写摘要页 -> 更新 index -> 更新相关的实体页和概念页（可能一次触发 10-15 个页面更新）-> 追加 log。

查询流程：先查 index.md 找到相关页面 -> 读页面内容 -> 综合回答并附引用 -> 好的回答可以反过来存成新的 wiki 页面。

**【维护（lint）：给 wiki 做体检】**

定期让 LLM 扫描全 wiki 做健康检查。Karpathy 列了六项具体检查目标：

1. **页面之间的矛盾**——A 页说张三是 CEO，B 页说张三是 CTO，哪个对？
2. **被新来源取代的过时声明**——三个月前写的"最新版本是 v2.3"，新文档说已经出 v3.0 了
3. **孤立页面**——没有任何其他页面链接到它，说明这个知识点跟整个知识库脱节了
4. **被提及但缺少独立页面的重要概念**——好几个页面都提到了"信息瓶颈理论"，但没有一个专门的页面解释它
5. **缺失的交叉引用**——A 页和 B 页明显在说相关的事，但互相没有链接
6. **可以通过搜索填补的信息空白**——wiki 里提到了某个实验结果但没有来源，LLM 可以建议去搜一下

注意：Karpathy 原文对 lint 的描述就这些——六个检查项，零实现细节，没有给 prompt 模板，触发方式是手动的（"periodically, ask the LLM"）。他自己说整篇文档"intentionally abstract"，实现留给使用者自己。所以目前 lint 更像是一份功能需求清单，不是一套现成的工具。

注意一个细节：当 wiki 规模小的时候（比如 100 页以内），LLM 可以把整个 index.md 一次性读进上下文窗口，直接导航。规模大了，Karpathy 建议加一个本地搜索工具，比如 qmd（支持 BM25 + 向量混合搜索），但这只是提效手段，不是核心架构。

这就是那条读者留言说的"向量化搜索只是提效加速器"——架构的核心是编译出来的 wiki 本身，不是搜索引擎。

◆ 跟 160 期的方案做个对比

━━━━━━━━━━━━━━━━━━━━

回忆 160 期讲的几个方案：

- **Mem0**：新记忆进来，用 LLM 当裁判做 ADD/UPDATE/DELETE，维护一个扁平的记忆池
- **Graphiti**：抽取实体和关系，建时序知识图谱，每条边带时间戳
- **Cognee**：建认知图谱，支持多跳检索
- **GraphRAG**：建图谱 + 社区检测 + 层次摘要

这些方案有一个共同特征：**它们都是在 RAG 的框架内做升级。** 核心流程还是"用户提问 -> 检索 -> 返回相关信息 -> LLM 生成回答"。只不过检索的数据结构从向量库升级到了知识图谱，从单跳变成了多跳，从无时间概念变成了有时间衰减。

LLM Wiki 做的是**范式转换**。它不是"更好的搜索"，而是"先编译再查询"。区别在哪？

**RAG 系：** 原始信息 -> 切块/抽实体 -> 存入数据库 -> 每次查询时检索+即时综合

**Wiki 系：** 原始信息 -> LLM 编译成结构化知识页 -> 每次查询直接读编译产物

关键差异是**综合发生的时机**。RAG 系在查询时综合（query-time synthesis），Wiki 系在导入时综合（ingest-time synthesis）。

这意味着什么？

第一，**查询时的上下文质量不同。** RAG 塞给 LLM 的是检索回来的原始片段——可能重复、可能矛盾、可能缺少上下文。Wiki 塞给 LLM 的是已经编译好的知识页——去重、去矛盾、有交叉引用。

第二，**知识是否积累不同。** RAG 每次查询的综合结果随着对话消失。Wiki 的每次 ingest 都在已有知识上叠加，每次好的查询结果都可以回流成新页面。

第三，**成本结构不同。** RAG 每次查询都付检索+综合的成本。Wiki 在导入时付一次编译成本，之后查询只需要读几个 Markdown 文件。查询越多，Wiki 的单次成本越低。

如果用那条读者留言的框架来说：session 历史是 raw，编译后的 wiki 就是记忆，图谱/语义/情景这些是编译方法论，向量搜索是提效加速器。这套隐喻比 160 期的"压缩、索引、遗忘"更接近 Karpathy 的设计哲学。

◆ 三周爆发：开源生态速览

━━━━━━━━━━━━━━━━━━━━

Karpathy 的 Gist 发出三周，GitHub 上已经冒出十几个实现。挑几个有代表性的看看：

**【lucasastorian/llmwiki】** 最接近"产品"形态的实现。Next.js + FastAPI + Supabase，提供在线服务 llmwiki.app。支持上传 PDF，通过 MCP 协议连接 LLM，自动编译 wiki。接近 600 stars，38 commits。能用，但还很早期。

**【Pratiyush/llm-wiki】** 最有工程完成度的实现。把 Claude Code、Codex CLI、Cursor、Gemini CLI 的会话历史转成静态 wiki 网站。2368 个测试通过，有 CI/CD，有 Docker 支持，有四因子置信度评分和 15 条 lint 规则（其中 8 条结构检查、3 条 LLM 驱动）。页面生命周期管理做到了 draft -> reviewed -> verified -> stale -> archived，90 天自动标记 stale。141 stars，但工程质量可能是目前最高的。

**【nvk/llm-wiki】** 面向 AI Agent 的编译知识库。支持并行多 Agent 研究、论文驱动的调查流程。同时提供 Claude Code 插件和 OpenAI Codex 插件，还有一个 AGENTS.md 让任意 LLM Agent 都能用。

**【nashsu/llm_wiki】** 跨平台桌面应用。把 Karpathy 的方案包装成了 GUI，面向不想碰命令行的用户。

**【ekadetov/llm-wiki】** Claude Code 插件，直接在 Obsidian 中构建知识库。最贴合 Karpathy 原始设计中"用 Obsidian 当 IDE"的思路。

看到规律了吗？大部分实现都在做同一件事：把 Karpathy 的文档变成可以"一键跑起来"的工具。核心逻辑就是 Karpathy 写的那些——ingest、compile、query、lint。各家的差异主要在前端交互和部署形态上。

**成熟度如何？** 老实说，大部分还处于"能跑但别太依赖"的阶段。lucasastorian/llmwiki 接近 600 stars 算最高的了，大部分实现才一两百。Pratiyush/llm-wiki 工程质量不错（2368 个测试、15 条 lint 规则），但 141 stars 说明用户基数还很小，实战检验不够。这个生态刚刚萌芽，离生产可用还有距离。

◆ 具体怎么做的：拆开 llmwiki 看一眼

━━━━━━━━━━━━━━━━━━━━

光看 Karpathy 的 Gist 只能看到理念，看不到实现。lucasastorian/llmwiki 是目前 stars 最多（接近 600）、最接近产品形态的实现（Apache 2.0 完全开源），值得拆开看一看它到底怎么把"编译器类比"落地的。

**【核心架构：5 个 MCP 工具 + 一份提示词】**

llmwiki 不是一个独立的应用——它是一个 MCP Server，接到支持 MCP 的 LLM 上用（目前主要是 Claude，但协议本身不绑定模型）。LLM 通过 5 个工具操作 wiki：

- **guide**：返回一份完整的系统指令（提示词），告诉 LLM 怎么组织 wiki
- **search**：两种模式——浏览文件列表，或用 PGroonga 做全文搜索
- **read**：读取文档内容，支持 PDF 分页、图片 base64、电子表格按 sheet 读取，单次有 12 万字符的预算上限
- **write**：三个子命令——create（新建页面）、str_replace（替换内容）、append（追加内容）
- **delete**：软删除（归档），overview.md 和 log.md 受保护不能删

整个系统的"智能"不在代码里——在提示词里。代码只管存取，LLM 按提示词的指令决定写什么、怎么组织、什么时候更新。

**【提示词长什么样】**

提示词在 `mcp/tools/guide.py` 里完全公开。摘几段关键的：

wiki 目录结构是强制的：

```
/wiki/overview.md    -- 枢纽页，必须包含源计数、关键发现、最近更新
/wiki/concepts/      -- 抽象概念（框架、方法论、原则）
/wiki/entities/      -- 具体实体（人、组织、论文、技术）
/wiki/log.md         -- 只追加的时间线日志
```

写作标准很严格：

```
 -- wiki 页面必须比普通聊天回复丰富得多
 -- 纯文字的页面是不合格的
 -- 每页必须至少包含一个视觉元素：Mermaid 图、表格或 SVG
 -- 每个事实性声明必须引用来源，用脚注格式 [^1]: filename.pdf, p.3
 -- 必须交叉引用相关页面
```

三个工作流的指令也写在提示词里：

```
Ingest:  read -> 讨论要点 -> 创建/更新 concept 和 entity 页面
         -> 更新 overview -> 追加 log
         （单个源通常触及 5-15 个 wiki 页面）

Query:   search -> read -> 综合回答并附引用
         -> 如有价值则反向创建新页面 -> 追加 log

Lint:    检查矛盾、孤立页面、缺失引用、过时声明 -> 追加 log
```

注意这些全是**提示词里的自然语言指令**，不是代码逻辑。LLM 收到这份指令后自己决定怎么执行——写什么页面、更新哪些引用、log 里记什么。没有硬编码的 if-else，全靠 LLM 理解。

**【Ingest 的具体管道】**

注意：Karpathy 的原始设计是"一个文件夹就能跑"，但 llmwiki 这个实现落地时还是用了数据库。文档上传后的处理链路是传统工程，不涉及 LLM：

1. 上传：通过 TUS 协议（支持断点续传）存到 S3
2. 格式转换：PDF 走 Mistral OCR 或本地 pdf-oxide（免费替代）；Office 文件先用 LibreOffice 转 PDF 再 OCR；电子表格转 Markdown 表格
3. 分块：按段落分割，约 512 token 一块，128 token 重叠，维护标题层级上下文，最小 32 token 过滤
4. 入库：写入 Postgres 的 document_chunks 表

分块存完之后，LLM 通过 read 工具读取内容，再按提示词里的 ingest 工作流生成 wiki 页面。

**【Lint 的实现：没有代码，全靠 LLM】**

这是最有意思的一点。lint **没有任何自动化代码实现**——没有规则引擎，没有检查脚本，没有 CI/CD。用户对 LLM 说"lint my wiki"，LLM 自己用 search + read 检查，发现问题用 write 修。

也就是说，lint 的质量完全取决于 LLM 那一刻的理解力和注意力。100 页 wiki，LLM 读完能不能记住 A 页说了什么跟 B 页矛盾——取决于上下文窗口够不够、attention 有没有稀释。又回到了那张软盘的问题。

**【一句话总结】**

llmwiki 的代码干的全是脏活（OCR、分块、存取、认证），所有"智能"的部分（怎么写 wiki、怎么组织知识、怎么 lint）都写在提示词里交给 LLM 执行。本质上就是**一个精心设计的 MCP 提示词 + 一个文档处理管道**。编译器的"编译逻辑"不在编译器代码里，在 LLM 的脑子里。

换句话说，如果你用 llmwiki 的云端服务，**它本身不跑任何 LLM**。guide/search/read/write/delete 五个工具全是 CRUD 存取操作，没有推理逻辑。你在自己的 LLM 里配好 MCP 连接，LLM 先调 guide 拿到提示词，然后所有的"读原文、写 wiki 页、做 lint"都是你自己的 LLM 在执行，llmwiki 只管存和取。所以它的云端成本很轻——不用买 GPU，只需要 Postgres + S3，LLM 的 API 费用是你自己的。本质上就是**一个带提示词的网盘**。数据存在哪？用 llmwiki.app 云端服务的话，文档和 wiki 页面存在他们的 Supabase（Postgres）+ S3 上。介意数据上云的可以自己部署——仓库里有 docker-compose.yml，本地跑 Postgres，S3 换 MinIO，数据全在自己机器上。

◆ 泼冷水：编译靠 LLM，lint 靠 LLM，问题换了个位置

━━━━━━━━━━━━━━━━━━━━

160 期结尾我写了一句话：**"每一个关键判断都外包给了同一个会幻觉的大脑。"**

LLM Wiki 解决了这个问题吗？

没有。它把问题换了个位置。

RAG 系的风险窗口在**查询时**——每次检索可能捞错、LLM 综合可能出错，但原始文档是完整保留的，错了顶多这一次回答有问题。

Wiki 系的风险窗口在**编译时**——LLM 在 ingest 阶段就做了综合判断，这些判断固化成了 wiki 页面。如果编译时 LLM 理解错了某篇文档、漏了关键信息、搞混了两个同名实体，这个错误就被"编译"进了知识库，后续所有查询都会引用这个错误的编译产物。

更隐蔽的是：**wiki 看起来比 RAG 检索结果更"权威"。** RAG 返回的是原始片段，用户和 LLM 都能看到原文，有机会发现检索错了。Wiki 返回的是 LLM 编写的"知识页面"，措辞流畅、逻辑自洽，你很难分辨这是忠实的综合还是幻觉。编译产物自带一种虚假的可信度。

Lint 能解决这个问题吗？理论上可以——让 LLM 定期审查 wiki，找出矛盾和错误。但 lint 用的还是 LLM。用同一个会幻觉的大脑去检查自己编译出来的产物，就像让一个刚抄完作业的学生自己批改——他可能真的能发现一些明显错误，但系统性偏差他看不出来，因为那是他自己的偏差。

还有一个实际问题：**成本不低。** Karpathy 说 ingest 一篇文档可能触发 10-15 个页面更新，每次更新都是一次 LLM 调用。如果你有几百篇原始文档，初始编译可能要花几十美元的 API 费用——这跟 GraphRAG 建图的成本问题是同构的。RAG 的优势之一就是 ingest 便宜（切块+embedding，不需要 LLM 理解），Wiki 和图谱方案都把成本从查询端转移到了 ingest 端。

lint 更贵。每次 lint 要让 LLM 读全 wiki 做逐项检查——100 页 wiki 就是几十万 token 的输入，一次 lint 可能比 ingest 十篇新文档还烧钱。Karpathy 说"periodically"但没说多频繁——每天跑？每周跑？每次 ingest 后跑？他没定义。频率高了准确但烧钱，频率低了省钱但错误会累积。这又是一个需要人来拿捏的判断。

最后一个问题更根本：**wiki 的规模上限在哪？** 100 页的时候，LLM 可以一次读完 index.md 直接导航。1000 页呢？10000 页呢？Karpathy 自己也说，规模大了需要加搜索工具。但一旦你开始靠搜索而不是靠 wiki 的内在结构来导航，你其实又回到了 RAG 的路上——只不过搜索的对象从原始文档变成了 LLM 编译的 wiki 页面。编译器类比在小规模下成立，大规模下可能退化成"先用 LLM 改写一遍文档，再对改写后的文档做 RAG"。

还有一个不太好听但很现实的问题：**为什么三周之内冒出来十几个 llm-wiki 实现？** 因为这个生意的实现成本接近零。llmwiki 的云端服务本身不跑 LLM——前面说过了，它就是个带提示词的网盘。用户自己的 LLM 烧 token 生成内容，数据免费存在平台上。平台需要什么？一个 Postgres + 一个 S3，没有 GPU 成本，没有 LLM 成本。用户每存一页 wiki，背后烧的是自己的 API 费用，平台坐在那收数据就行。所以它当然免费——你用自己的钱帮它填充数据。提示词是 Karpathy 写好的（开源），MCP Server 包装 CRUD 一周能写完，任何会写后端的程序员看一眼就知道能做。三周十几个实现，不是因为理念多有启发性，是因为**护城河几乎不存在**。

━━━━━━━━━━━━━━━━━━━━

◆ 回到三个操作

━━━━━━━━━━━━━━━━━━━━

拉回 160 期的框架，看看 LLM Wiki 在"压缩、索引、遗忘"三个维度上分别做到了什么。

**压缩——这是 LLM Wiki 最强的维度。** 编译本身就是一种深度压缩。不是简单的摘要，而是跨文档的综合。10 篇关于同一个主题的文章，编译后可能变成一个实体页 + 两个概念页 + 几条交叉引用。信息密度的提升是量级性的。160 期讲的方案里，GraphRAG 的社区层次摘要和 Evermind 的语义巩固也在做类似的事，但 LLM Wiki 把压缩做得最彻底——它的编译产物就是压缩后的知识，不是检索的中间产物。

**索引——靠 wiki 的内在结构。** index.md 是目录，wikilink 是超链接，页面之间的引用关系就是一张图。不需要向量数据库，不需要 embedding，导航靠的是 LLM 读目录 + 沿链接跳转。这个机制在小规模下优雅高效，但前面说了，规模大了会退化。跟 Cognee 的多跳图谱检索比，wiki 的链接结构是隐式的（靠 Markdown 链接），图谱是显式的（靠三元组）。各有各的适用范围。

**遗忘——lint 机制做了尝试，但还不够。** Karpathy 的 lint 可以标记过时信息、发现矛盾，这相当于一种被动遗忘。但跟 Graphiti 的时间衰减（关系过期自动降权）和 Mem0 的主动 UPDATE/DELETE 比，wiki 的遗忘机制更粗糙——它依赖你主动触发 lint，然后依赖 LLM 判断什么过期了。没有自动的时间衰减，没有基于使用频率的权重调整。

所以 LLM Wiki 的性格很清晰：**压缩极强，索引优雅但有规模天花板，遗忘机制最弱。** 它适合"持续积累、定期整理"的知识管理场景——个人研究、读书笔记、长期项目——而不是需要实时更新和自动遗忘的 Agent 对话记忆场景。

回到那条读者留言：用 LLM Wiki 的逻辑来做 Agent 记忆，理论上是通的——session 历史是 raw，编译后的 wiki 是记忆，lint 做纠错维护。但要让这套逻辑真正跑起来，还需要解决遗忘策略和规模扩展的问题。目前的开源实现都还没走到这一步。

160 期结尾说的那个根本问题，在 LLM Wiki 这里依然成立：**用一个会幻觉的大脑来编译、维护、审查自己的知识库——幻觉不是消失了，是被编译进了看起来更可信的产物里。**

这可能比 RAG 的幻觉更危险。RAG 幻觉是一次性的，下次查询可能就对了。编译进 wiki 的幻觉是持久性的，除非 lint 抓到，否则每次查询都会引用它。

不过话说回来，人脑的记忆又何尝不是这样呢？我们"记住"的东西，很多时候也是大脑"编译"过的版本，带着偏差、遗漏和虚构。只是我们通常不会对自己的记忆跑一遍 lint。

从这个角度看，LLM Wiki 至少比人脑多了一步：它承认编译会出错，并设计了纠错机制。

这就是 2026 年 4 月的状态：所有的知识管理方案，不管是 RAG、知识图谱还是 LLM Wiki，都在用不同的策略管理同一个核心矛盾——**我们需要用一个不完全可靠的智能来管理知识，而管理知识这件事本身要求可靠性。**

这个矛盾，目前无解。但至少我们有了更多管理它的方式。
