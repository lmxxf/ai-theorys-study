【端侧多模态】把多模态大模型塞进 RK3588——两个文件、两套 runtime、一只眼睛一张嘴

━━━━━━━━━━━━━━━━━━━━

◆ 起因

━━━━━━━━━━━━━━━━━━━━

上一期算了三笔税，解释了"为什么 DeepSeek、GLM 选择主线纯文本"。但市面上仍然有大量"看图说话"的多模态模型在跑——它们走的都是后挂式：纯文本底座外接一只眼睛。这条路真正落到工程上是什么样的？我手头正好有个嵌入式项目，从这里说起最直观。

项目是在 RK3588 的 NPU 上跑多模态大模型。下载 Qwen3-VL-2B 的时候发现要下两个文件——一个语言模型（2.3GB），一个视觉编码器（812MB）。两个模型分开下、分开加载、分开推理，最后串起来才能"看图说话"。

板端运行的日志长这样：

```
LLM Model loaded in  2892.80 ms
ImgEnc Model loaded in  1008.81 ms
ImgEnc Model inference took  2749.71 ms
```

先加载语言模型，再加载视觉编码器，图片编码完了喂给语言模型——两步走。

这里有个容易忽略的事实先点出来：**HuggingFace 上发布的标准 Qwen3-VL-2B 其实是一个整体**——视觉编码器、投影层、语言模型一起打包在一份 safetensors 里，用 `AutoModelForVision2Seq.from_pretrained` 一行就能加载。LLaVA、MiniCPM-V、InternVL 在服务端的发布形态都是这样，vLLM 也是当成一个整体喂进去。

**"拆成两个文件"是端侧独有的工程现实。** 那么两个问题自然就来了：

- **架构上**：这种"拼接式"多模态凭什么把两个独立训练好的模型接起来，就能"看图说话"？
- **端侧上**：为什么服务端一个文件就够，端侧偏偏要拆成两个？



这篇就回答这两件事。先说架构，再说端侧。

后挂式架构有个共同特点先记住：**视觉是只进不出的**——图像 token 只当输入条件，不算 loss、也不生成。"看图说话"四个字里，模型真正被训练的只有"说话"。眼睛是用对比学习另外训出来的，下面会讲。

━━━━━━━━━━━━━━━━━━━━

◆ 多模态模型的"拼接"架构

━━━━━━━━━━━━━━━━━━━━

打开任何一个主流多模态大模型——Qwen3-VL、MiniCPM-V、LLaVA、InternVL——里面都是三个部件：

**1. 视觉编码器（Vision Encoder）**

💡 翻译成人话：把图片变成一组向量的模型。输入一张 448×448 的图片（几十万个像素），输出几百个向量（每个向量几百到一千多维）。这些向量包含了图片的语义信息——"这是一只猫"、"背景是草地"之类的。

Qwen3-VL-2B 用的是 **SigLIP2-Large（约 300M 参数）**，一种基于 ViT（Vision Transformer）的图片编码器。MiniCPM-V 4.6 用的是 SigLIP2-400M，同一个系列、不同规模。

**2. 投影层（Projector / Merger）**

💡 翻译成人话：把视觉编码器输出的向量"翻译"成语言模型能理解的格式。视觉编码器的输出维度可能是 1152 维，但语言模型期望的输入维度是 1536 维——投影层就是做这个维度转换的。

Qwen3-VL 的投影层叫 Merger，结构非常简单：一个 LayerNorm + 两个线性层（MLP）。

**3. 语言模型（LLM）**

💡 翻译成人话：就是我们熟悉的大语言模型——接收一串 token 向量，逐个预测下一个 token。Qwen3-VL-2B 用的是 Qwen3-2B，MiniCPM-V 4.6 用的是 Qwen3.5-0.8B。

────────────────────

三者的关系：

```
图片 → 视觉编码器 → 视觉向量 → 投影层 → "视觉 token" → 拼到文字 token 前面 → 语言模型 → 输出文字
```

打个比方：**视觉编码器是眼睛，投影层是翻译，语言模型是大脑。**

眼睛看到图片，用自己的"语言"描述了看到的东西。但大脑说的是另一种"语言"——它只认文字 token。翻译的工作就是把眼睛的描述转换成大脑能理解的格式。转换完了，大脑就像收到了一段特殊的"文字"，照常生成回答。

**这就是为什么可以分成两个文件**——视觉编码器和语言模型是两个独立训练出来的模型，中间只靠一个很小的投影层连接。在 RK3588 上部署时，视觉编码器跑 RKNN Runtime（NPU 加速矩阵运算），语言模型跑 RKLLM Runtime（NPU 加速 token 生成），各走各的推理流程。

━━━━━━━━━━━━━━━━━━━━

◆ 视觉编码器怎么训的——对比学习

━━━━━━━━━━━━━━━━━━━━

SigLIP2 不是凭空冒出来的，它的前身是 CLIP 和 SigLIP。训练方法的核心思路叫**对比学习（Contrastive Learning）**。

💡 翻译成人话：给模型看大量的"图片 + 文字描述"配对数据（几亿对），让模型学会——看到一张猫的图片，它的向量应该和"一只猫"这段文字的向量靠近；看到一张狗的图片，它的向量应该和"一只猫"的向量远离。

────────────────────

【CLIP 的做法：softmax 对比损失】

CLIP（OpenAI, 2021）同时训练一个图片编码器和一个文本编码器。一个 batch 里有 N 张图片和 N 段文字，算出 N×N 的相似度矩阵，对角线上的才是正确配对。

损失函数用 **softmax + 交叉熵**：对每一行（固定一张图片，和所有 N 段文字比较），做 softmax 归一化，然后算交叉熵——让正确配对的概率尽量高。

问题：softmax 是全局归一化——每个样本的损失依赖这个 batch 里所有其他样本。**batch 越大效果越好**，所以 CLIP 要用 32768 的巨大 batch size，训练成本很高。

────────────────────

【SigLIP 的改进：sigmoid 对比损失】

SigLIP（Google, 2023）改了一招：**把 softmax 换成 sigmoid。**

💡 翻译成人话：不再让 N 个样本互相竞争"谁的概率最高"。而是对每一对（图片ᵢ, 文字ⱼ）独立判断——"这对是不是正确配对？"输出一个 0 到 1 之间的概率。对角线上的配对标签是 1（正确），其他位置标签是 -1（错误）。

```
CLIP：  一行 N 个样本做 softmax → 概率和为 1 → 样本间竞争
SigLIP：每对独立做 sigmoid → 每对独立判断 → 样本间不耦合
```

**好处**：每对样本的梯度不再依赖其他样本，可以用更小的 batch size 达到相同效果，训练更高效。论文实测，SigLIP 在相同计算预算下，分类准确率和零样本性能都优于 CLIP。

**SigLIP2**（Google, 2025）在 SigLIP 基础上改进了训练策略（多分辨率、多任务、自蒸馏等），是目前最强的开源视觉编码器之一。Qwen3-VL 和 MiniCPM-V 4.6 都选了它。

────────────────────

**训练完得到什么？**

一个 ViT 模型，能把任意图片映射成一组固定格式的向量。语义相似的图片（比如两张不同角度的猫），向量在高维空间里距离很近。语义不同的图片（猫和汽车），向量距离很远。

**注意：这个阶段视觉编码器完全不知道什么是"语言模型"。** 它只学了"图片和文字的语义对齐"——知道什么图片对应什么文字描述，但不会生成文字。

━━━━━━━━━━━━━━━━━━━━

◆ 投影层怎么训的

━━━━━━━━━━━━━━━━━━━━

视觉编码器有了，语言模型也有了（比如 Qwen3-2B，用海量文本预训练好的）。两个模型各说各的"语言"——视觉编码器输出 1152 维向量，语言模型期望 1536 维 token embedding。怎么把它们接起来？

**训练策略：冻住两头，只训中间。**

💡 翻译成人话：视觉编码器的参数不动，语言模型的参数也不动。只训练投影层那几千万个参数——让投影层学会"怎么翻译"。

────────────────────

【训练数据】

用的是"看图说话"数据——每条数据是一张图片 + 一段对这张图片的文字描述。比如：

- 图片：一只橘猫趴在键盘上
- 文字：`A orange cat lying on a computer keyboard`

几百万到几千万条这样的配对。

────────────────────

【训练过程】

```
1. 图片过视觉编码器 → 得到视觉向量（参数冻结，不更新）
2. 视觉向量过投影层 → 得到"视觉 token"
3. "视觉 token" + 文字 token 拼在一起，喂给语言模型
4. 语言模型预测下一个 token（参数冻结，不更新）
5. 算损失：标准语言模型交叉熵（预测下一个 token 的概率 vs 真实下一个 token）
6. 梯度只回传到投影层，更新投影层参数
```

**损失函数就是最普通的语言模型交叉熵损失（next-token prediction）**——没有任何花活。投影层的目标就是：让翻译出来的"视觉 token"足够好，好到语言模型像读到了一段高质量的文字描述一样，能准确地预测后续文字。

────────────────────

【投影层有多小？】

以 MiniCPM-V 4.6 为例，我实际分析了模型权重：

```
model.vision_tower：5.22 亿参数（SigLIP2-400M）
model.language_model：7.52 亿参数（Qwen3.5-0.8B）
model.merger（投影层）：2597 万参数
```

投影层只占总参数的 **2%**。结构就是一个 LayerNorm + 两个线性层（MLP）——六个权重矩阵，49.5 MB。

**这意味着：把两个大模型（总计十几亿参数）连接起来的"翻译官"，只有 2600 万参数。** 整个多模态能力的"接口协议"就编码在这 2% 里面。

────────────────────

当然，实际训练多模态模型通常不止训投影层。比如 LLaVA 的两阶段方案：

- **第一阶段**：只训投影层（上面说的过程），用大规模图文配对数据
- **第二阶段**：解冻语言模型（甚至视觉编码器），用高质量的指令微调数据做全参数微调

但核心思路不变——**先用投影层把两个模型"接上"，再精调整体**。投影层是多模态能力的起点。

━━━━━━━━━━━━━━━━━━━━

◆ Qwen3-VL 的特色：DeepStack

━━━━━━━━━━━━━━━━━━━━

大多数多模态模型（LLaVA 一族）的投影层做的事很直接：**取视觉编码器最后一层的输出，投影一下，拼到语言模型的输入层。**

```
ViT 最后一层输出 → 投影层 → LLM 输入层
```

Qwen3-VL 做了一个更深度的融合，叫 **DeepStack**（原始论文来自微软+复旦，NeurIPS 2024）。

**不只是最后一层**——ViT 是堆叠多层的结构（Qwen3-VL-2B 用的 SigLIP2-Large 是 24 层），每一层捕捉的信息不同：

- **浅层**：边缘、纹理、颜色——底层视觉特征
- **中层**：物体部件、局部结构——中层语义
- **深层**：物体类别、场景语义——高层语义

普通投影层只用最后一层的输出，浅层和中层的信息全丢了。**DeepStack 把 ViT 中间多个层的特征也抽出来，各自过投影，以残差方式注入语言模型的前几层。** Qwen3-VL-2B 的配置文件里写得明白：`deepstack_visual_indexes=[5,11,17]`——从 ViT 的第 6、12、18 层各抽一份（注意这是 2B 这个规模搭配 24 层 ViT 的具体配置，更大的 VL 模型用的层号会不一样）。

```
ViT 第 6 层特征  → 投影 ↘
ViT 第 12 层特征 → 投影 → 注入 LLM 的前几层（残差相加）
ViT 第 18 层特征 → 投影 ↗
ViT 最后一层输出 → 投影 → LLM 输入层（正常路径）
```

💡 翻译成人话：普通投影层像是翻译只转述了对方最后一句总结。DeepStack 像是翻译把对方讲话过程中不同阶段的笔记——草图、细节、结论——全都递了进去。注意笔记都递到大脑加工的**早期阶段**，让它从一开始就同时拿到粗细两套信息，而不是只凭一句总结开工。

DeepStack 原始论文在 TextVQA、DocVQA 这类高分辨率细粒度任务上提升显著；Qwen3-VL 官方也把它列为三大架构更新之一（另两个是 Interleaved-MRoPE 和文本-时间戳对齐），称其强化了 OCR、小物体这类细粒度视觉理解。

━━━━━━━━━━━━━━━━━━━━

◆ 为什么"拼接"架构在服务端也是主流

━━━━━━━━━━━━━━━━━━━━

先回答第一个问题：为什么不从头训一个"天生就能看图"的大模型？

**答案是成本和迭代速度。** 上一期讲过从零原生的代价：Gemini 替全行业蹚了那条最贵的路，三笔税照单全收。所以绝大多数厂商选了便宜的方案——后挂式。

【复用 + 迭代】

训一个好的视觉编码器（SigLIP2）要几亿图文配对数据 + 几千 GPU 小时。训一个好的语言模型（Qwen3-2B）要几万亿 token 文本 + 几万 GPU 小时。**这两个模型已经有人花了巨大代价训好了。** 只需要训一个 2600 万参数的投影层——几百万数据、几百 GPU 小时——就能把它们组装成多模态模型。

升级也方便：视觉编码器从 CLIP 升级到 SigLIP 再到 SigLIP2——重新训投影层，语言模型不用动。语言模型从 Qwen2 换到 Qwen3.5——重新训投影层，视觉编码器不用动。

MiniCPM-V 系列就是这么迭代的：

| 版本 | 视觉编码器 | 语言模型 | 时间 |
|------|------------|----------|------|
| MiniCPM-V 2.6 | SigLIP-400M | Qwen2-7B | 2024-08 |
| MiniCPM-V 4.5 | SigLIP2-400M | Qwen3-8B | 2025-08 |
| MiniCPM-V 4.6 | SigLIP2-400M | Qwen3.5-0.8B | 2026-05 |

视觉编码器升级了一次，语言模型升级了两次，连参数量都能从 7B 瘦身到 0.8B——每次都是"换零件 + 重新训投影层"。端到端训练的模型，每次升级就意味着从头来。

**但注意**：这里说的"拼接"是**训练阶段**的拼接——三个部件分别训练，再用投影层缝合。**发布到 HuggingFace 时，它仍然是一个完整的模型仓库**：视觉编码器、投影层、语言模型一起打包在 safetensors 里，加载时一个调用就搞定。所谓"拼接架构"在服务端是工程师的视角，不是文件系统的视角。

那为什么到了端侧，文件系统也得跟着拼？

◆ 端侧的现实：从一个文件到两个文件

━━━━━━━━━━━━━━━━━━━━

回到 RK3588 项目。HuggingFace 上下下来的 Qwen3-VL-2B 是一个完整模型仓库，但要在板子上跑起来，必须先用 Rockchip 的工具链转换成两份独立的部署文件：

| 文件 | 大小 | 内容 | Runtime |
|------|------|------|---------|
| `qwen3-vl-2b-instruct_w8a8_rk3588.rkllm` | 2.3 GB | 语言模型（W8A8 量化） | RKLLM Runtime |
| `qwen3-vl-2b_vision_rk3588.rknn` | 812 MB | 视觉编码器 + 投影层（FP16） | RKNN Runtime |

为什么必须拆？四个原因，一个比一个硬。

【原因一：两套工具链，分别处理不同的算子】

RKNN-Toolkit2 是面向 CNN/Transformer 这类**前馈视觉模型**编译的，它把整张计算图静态展开成 NPU 指令，一次推理跑完。RKLLM-Toolkit 是面向**自回归语言模型**编译的，它要管 KV cache、token 循环采样、分块 prefill 这些 LLM 特有的东西。两个 Runtime 在底层调用的 NPU 指令集合不同，优化策略也不同——**强行塞进一个 runtime 不是做不到，是做了也不好用**。

MNN、NCNN、TensorRT-LLM-edge 走的是同一条路：视觉和 LLM 各用一套部署管线。这不是 Rockchip 的特色，是端侧的通用模式。

【原因二：量化策略必须不同】

语言模型对量化更敏感的部位是 attention 和 KV cache，普遍用 **W8A8 或 W4A16**（权重 4-8 bit、激活 8-16 bit），能在精度损失可控的前提下把 LLM 压到 RAM 里。

视觉编码器是前馈结构、计算密集型，对量化更敏感的是浅层卷积/patch embedding，常用 **FP16 或 W8A16**——量化太狠会直接影响特征质量，连带影响"看图说话"的准确度。

服务端不在乎，统一 BF16 跑就完事了。端侧 RAM 和带宽都是硬上限，**两个部件必须用最适合自己的量化策略**——这就要求它们能被独立编译、独立量化。

【原因三：推理流程根本不同】

视觉编码器是**并行**的：一张图片一次性过 ViT 24 层，2.7 秒（从日志里能看到）。

语言模型是**串行**的：每生成一个 token 就要跑一遍整个 LLM 的所有层，再把这个 token 喂回去生成下一个。一句话答案要循环几十次。

并行和串行的硬件调度策略完全不一样——NPU 上视觉部分能吃满 batch、语言部分要打满 latency。**用同一个 runtime 同时调度这两种工作负载是反人性的设计**。

【原因四：内存隔离】

板子上一共 16GB 内存（RK3588 高配版），系统/应用要占走一部分，留给模型的不超过 10GB。视觉编码器 812MB + 语言模型 2.3GB + 中间激活 + KV cache，已经塞得紧巴巴。

如果两个部件共享同一个 runtime 的内存池，一个吃多了另一个就饿死。**拆成两个进程/两份内存预算，能保证视觉和 LLM 都有自己的下限**——这在嵌入式上不是优化，是生存。

────────────────────

把四条捏一起：**端侧拆文件不是"架构选择"，是工具链 + 硬件约束逼出来的工程现实。** 服务端有 80GB H100，BF16 整个塞进去就完事；端侧 16GB RAM 想跑 2B 模型，必须把每一份资源都按模态、按计算特性拆开来管。

这也解释了为什么 RKNN 的发布形态里，视觉编码器和投影层是**打包在同一个 `.rknn` 文件**里的——它们俩都是前馈、都是图像侧的算子、都用同一种量化、推理时严格串行。**真正的边界不在"哪些部件来自不同训练"，而在"哪些部件适合用同一套 runtime 跑"。**

━━━━━━━━━━━━━━━━━━━━

◆ 用实际数据说明

━━━━━━━━━━━━━━━━━━━━

两个我实际接触过的端侧多模态模型，参数分布对比：

| 部件 | Qwen3-VL-2B | MiniCPM-V 4.6 |
|------|-------------|---------------|
| 视觉编码器 | SigLIP2-Large（~300M） | SigLIP2-400M（522M） |
| 投影层 | Merger MLP（~26M） | Merger MLP（26M） |
| 语言模型 | Qwen3-2B（~1.7B） | Qwen3.5-0.8B（752M） |
| **总参数** | **~2.0B** | **~1.3B** |
| **投影层占比** | **~1.3%** | **~2.0%** |

两个模型的投影层参数占比都在 **1-2%** 之间——几千万参数就把十几亿参数的两个模型连起来了。这也是为什么端侧能跑：2B 总参数对 RK3588 来说还在能力范围内，但同样的拼接思路放到 7B、72B 上就溢出板子内存了。**端侧多模态目前的甜区基本就锁死在 1B-3B**，再大就只能上 Jetson Orin 这种带 64GB 内存的高配开发套件。

━━━━━━━━━━━━━━━━━━━━

◆ 小结：服务端和端侧看到的是同一只大象的两面

━━━━━━━━━━━━━━━━━━━━

后挂式多模态的"拼接"，在不同视角下意味着不同的东西。

**训练视角**：视觉编码器、投影层、语言模型分别训练，再用投影层（只占 2% 参数）缝合。复用强模型省成本，换零件重训投影层就能迭代，工程上的最优解。

**服务端部署视角**：拼接是工程师的认知边界，不是文件系统的边界。HuggingFace 上发布的就是一个完整的 safetensors 仓库，加载、推理、量化都按一个模型处理。

**端侧部署视角**：拼接被迫显性化为两个文件、两套 runtime——因为视觉和 LLM 在算子集合、量化策略、推理模式、内存预算上都对不上号。**不拆不行**。

所以"为什么多模态模型要拆成两个文件"这个问题，准确的回答是：**它在云上从来不拆，是端侧硬件把它逼着拆了**。

技术名词速查：

- **ViT**（Vision Transformer）：把图片切成小块（patch），当作"token"输入 Transformer 的图片编码器。2020 年 Google 提出，现在是视觉编码器的主流架构。
- **SigLIP / SigLIP2**：Google 的视觉-语言对比学习模型，用 sigmoid 损失替代 CLIP 的 softmax 损失，训练效率更高。
- **对比学习**（Contrastive Learning）：让模型学会区分"什么和什么相似、什么和什么不同"的训练方法。核心是拉近正样本对、推开负样本对。
- **投影层**（Projector / Merger）：连接视觉编码器和语言模型的小型网络，做维度转换和特征对齐。
- **冻结**（Freeze）：训练时固定某个模块的参数不更新。"冻住视觉编码器"意味着只让梯度流过投影层，不改变视觉编码器的权重。
- **交叉熵损失**（Cross-Entropy Loss）：语言模型最常用的损失函数。模型预测下一个 token 的概率分布，和真实的下一个 token 比较，概率越接近损失越小。
- **DeepStack**：多层视觉特征注入策略（微软+复旦提出，NeurIPS 2024，Qwen3-VL 采用），把 ViT 中间多层的特征以残差方式注入 LLM 的前几层，实现更深度的视觉-语言融合。
