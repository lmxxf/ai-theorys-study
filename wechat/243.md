◆ 开篇：sparse_attn 这个名字有点骗人

━━━━━━━━━━━━━━━━━━━━

第一眼看到 `sparse_attn` 这个名字，我以为它就是"稀疏 attention 的那个 kernel"——一个原子操作，进去 Q/K/V 出来 O，跟标准的 `scaled_dot_product_attention` 在 API 层面应该差不多。

把代码看完才发现不是。它是**好几步组合在一起的一个流程**：

1. 用 Indexer 给的 index 从 KV 总表里抓出当前 query 真正要看的那些位置
2. Q 和这些抓出来的 K 算点积、乘 scale
3. 把没选到（Indexer 返回 -1）的位置 mask 成 -inf
4. 给末尾拼一根 attention sink（注意力"下水口"，让没处安放的权重有地方流走——第二章 Step 4 专门讲）
5. softmax，然后只取前 topk 个权重去乘 V，sink 那一列的概率被丢掉

这五步任何一步漏了 attention 的语义都不对，但单看每一步又都不"sparse"——稀疏体现在第 1 步：Indexer 把候选 K 从全表 T 个砍到 topk 个，后面四步还是普通 attention，只是底盘小了。

**所以 sparse_attn 不是一刀切下去的稀疏 kernel，而是在 Indexer 砍小的底盘上跑了一次标准 attention，再加一根 sink 当压力阀**。我重新看了一遍源码才搞清楚这件事。今天把这个流程说清楚，顺带说一下为什么我这一版 sparse_attn 要做 query 分块——和 242 期 Indexer 分块的原因不同源。

参考代码：仓库里的 `kernel_sm121.py`，`sparse_attn` 函数。

💡 **打个比方**：Indexer 是"图书管理员把这一节课要用的 32 本书从 10 万本里挑出来推到你桌上"；sparse_attn 是"你坐下来一本一本翻、挑句子、把摘抄拼成一段笔记"。Indexer 决定看什么，sparse_attn 决定怎么用——两步分得很清。

◆ 第一章：输入输出对一遍维度

━━━━━━━━━━━━━━━━━━━━

先把账面摆清楚。`sparse_attn` 的签名：

```python
def sparse_attn(q, kv, attn_sink, topk_idxs, softmax_scale) -> o:
    ...
```

| 参数 | 形状 | 含义 |
|---|---|---|
| `q` | `[b, s, h, d]` | b 个 batch、s 个 query 位置、h 个头、每头 d 维 |
| `kv` | `[b, T, d]` | 压缩后的 KV 总表，T = S/ratio（CSA 的 4:1 压缩） |
| `attn_sink` | `[h]` | 每个 head 一个学到的标量，扮演"虚拟 key"，下面会讲 |
| `topk_idxs` | `[b, s, topk]` | Indexer 给出的索引，每个 query 位置看哪 topk 个 KV |
| `softmax_scale` | 标量 | 通常是 1/√d，标准 attention 的那个缩放 |
| 输出 `o` | `[b, s, h, d]` | 和 `q` 同形——注意这是**单头的结果**，h 个头还没合并，也还没过 O 投影，不是这一层最终的激活值 |

V4 实际跑起来时一组典型数字：`b=1`、`s` 是当前 prompt 长度（17832 这个数字其实很保守——V4 支持的上下文上限是 100 万 token，17832 连 2% 都不到）、`h=64`、`d=512`、`T ≈ s/4`、`topk=512`。每一层都过一遍这个函数，43 层 attention 串起来。

注意几件事：

- **kv 是 1 头的**：第三维就是 `[b, T, d]`，没有 h 这个轴。这是 DeepSeek 从 V2 起就在做的 MLA latent——V4 Flash 这个实验平台是 64 个 Q 头共享同一份 KV latent，存的时候只存一份，算 attention 时所有头都查这份。
- **这份 kv 和 242 期 Indexer 用的 kv 不是同一份**：Attention 和它内部的 Indexer 各自有一个独立的 `Compressor` 模块、独立的权重、独立的 `kv_cache`——Attention 自己的是 512 维（滑动窗口 raw kv + 压缩 kv 拼一起），Indexer 自己的是 128 维（只存压缩部分，不需要滑动窗口，因为最近邻的 token 走另一条路径直接强制纳入，不用 Indexer 挑）。两份都是从同一个隐藏状态 `x` 各自压缩出来的，topk_idxs 是拿 Indexer 那份 128 维小 kv 算出来的，但真正 gather 去做 attention 的，是这里这份 512 维的大 kv——**每一个 CSA 层要同时背两份 KV cache，不是共用一份省显存**。
- **topk_idxs 是 int64**：每个值要么是 `[0, T)` 之间的合法位置，要么是 `-1` 占位（Indexer 没凑够 topk 个时填的，242 期专门讲过）。
- **attn_sink 只有一个 h 维度**：每个 head 一个标量，没有 d 维。这意味着 sink 不参与"value 的混合"，它只是 attention 概率里那一根多出来的"逃生通道"——下面专门讲。

输出 `o` 和 `q` 同形——这一刀走完，每个 query 位置每个头都拿到了一份 d 维的混合 value。送回去给后面的 O 投影，再回到 4096 维高速公路。

◆ 第二章：五步流程，一帧一帧说

━━━━━━━━━━━━━━━━━━━━

这一节只想清楚一件事：**给定当前已经攒好的 KV 缓存，面对下一个 query token，这五步怎么算**。先只盯着这一个 query 位置，不去管背后是逐个算还是一大批一起算——批量的事本章末尾和第三章再说，这里先不提。

进入五步之前，先把符号钉死——本章全程用 V4 Flash 的真实数字（还是那次 S=17832 的例子），看到字母随时回来对：

| 符号 | 含义 | 本例取值 |
|---|---|---|
| `h` | head 数 | 64 |
| `d` | 每个 head 的维度 | 512 |
| `T` | 当前 KV 缓存的行数（压缩后） | ≈4458（S=17832 ÷ 4） |
| `t` / `topk` | 这个 query 挑出来看的行数 | 512 |
| `kv` | KV 缓存本体，形状 `[T, d]` | 4458 行 × 512 维 |
| `q` | 当前这一个 query，形状 `[h, d]` | 64 头 × 512 维 |
| `o` | 输出，形状 `[h, d]`，和 q 同形 | 64 头 × 512 维 |

【Step 1】**按 index 抓 KV**——这个 query 自己的 `topk_idxs`，从 KV 缓存的 T 行里点出 topk 行：

```text
kv_gathered = gather(kv, topk_idxs)   形状 [topk, d]
```

这一步的几何意义就是：这个 query 面对的 KV 缓存有 4458 行，但它只挑出其中 512 行。**稀疏的"稀"就在这里**——后面所有计算的底盘从 T 缩到 topk。

"行"要单独说一下，特别容易误会。KV 缓存里的 4458 行**不是 4458 个原始 token**，是 **4458 个压缩块**——V4 的 CSA 层规矩是：一个 token 刚进来时以原始、全精度的形态在滑动窗口（最近 128 个）里待着；一旦它**老到滑出这 128 的窗口**，就会被 CSA 打包进"4 合 1"的压缩块，从此以后**再也不以单独 token 的身份出现**，永远以那个 4 合 1 的压缩块参与后面所有层的计算。所以这个 query 挑出的 512 行，**语义覆盖上相当于约 2000 个原始 token**（512 块 × 每块 4 个 token），但**计算量上只是 512 次点积**，不是 2000 次——这正是 CSA 用压缩换来的好处：用 1 条的算力，兑现 4 条的信息覆盖。压缩块是**打包好就不再拆开**的原子单位，Indexer 挑它、sparse_attn 算它，都当一个 key，谁也不会再把它还原回 4 个 token。

注意 `topk_idxs` 里有 `-1` 占位，gather 前要 `clamp(min=0)`，否则负数索引会拿到错的行——但拿到的内容不重要，下一步会被 mask 掉。

【Step 2】**Q · K · scale**——这一步和标准 attention 一模一样：

```text
scores = einsum('hd, td -> ht', q, kv_gathered) · softmax_scale
       形状 [h, topk]
```

读法：这个 query，每个 head，对自己挑出来的 topk 个候选 K 各算一个点积分数。h 头互相独立，topk 行互相独立。说得再直白点：**盯着一个 head 看，`scores` 就是一条 512 长的打分数组**——512 个候选、每个一分；64 个 head 就是 64 条这样的数组并排放着。而 `topk_idxs` 只有一条 512 长的索引数组，64 个 head 共用同一条——挑哪些位置是 Indexer 定的，跟 head 无关，所以下一步 mask 时一条掩码要广播给全部 64 个 head。

【Step 3】**把假位置 mask 掉**——如果这个 query 的 `topk_idxs` 里有 `-1` 占位（没凑够 topk 个真实候选），对应的 scores 设为 `-inf`：

```text
valid = (topk_idxs >= 0)        广播到 [h, topk]
scores[!valid] = -inf
```

走到 softmax 时这些位置直接变 0。**这一步是 sparse attention 和 dense attention 唯一不"对称"的地方**——Indexer 决定了哪些位置参与计算、哪些位置直接出局，掩码是它的副产品。

💡 常识科普

看到这里你可能会想：既然 `-1` 位置反正是废的，为什么不直接把这些位置从数组里剔掉、把 512 缩成"每个 query 各自实际有效的那个长度"，后面所有计算都省下来？逻辑上完全成立，代价是**每个 query 的有效数不一样，形状就变得不规则了**——GPU 硬件是围绕"整批一起算规则形状"设计的，一旦每个 query 需要不同长度，就得先扫一遍数每个人有几个、算前缀和、再启动第二个 kernel 重排、下游还得再同步一次每个 query 该算多长——**每一次同步都要让所有线程停下来对齐，而 GPU 最怕的就是这个**。

对比一下就懂了：**mask 掉几十万个位置，硬件层面就是"顺便做一次布尔比较+把某个位置写成 -inf"，是搭着 gather/Q·K 那趟流水线免费捎带的，几乎不占额外时间**；而"重排成不规则形状"则要打断流水线两三次，每次停顿都浪费掉几千个线程周期。这是 GPU 时代和 CPU 时代最反直觉的一条价值观倒置：**CPU 上 cycle 是稀缺资源、能省则省；GPU 上 cycle 廉价、协调昂贵，宁可多做几十万次无用功，也别让线程等**。

所以 mask 不是"懒得优化"，是 GPU 硬件模型下唯一对得起性能的写法。整个深度学习圈里 mask 无处不在（padding mask、causal mask、attention mask 全是这个套路），不是大家没想到"不规则"这条路，是**都试过、都放弃了**——规则化 + 掩码是十几年 GPU 时代磨出来的教条，不是设计上的偷懒。

什么时候会凑不够 512 个真实候选？因果约束逼出来的——**这个 query 只能往前看，看不到它之后的历史**，它面前摆着多少条 KV 缓存，由它自己在序列里的位置决定。比如序列第 100 个位置，往前只有约 25 条压缩 KV 可看（100/4），肯定凑不够 512；第 10000 个位置，往前已经有约 2500 条，还是不够；等到第 2048 个位置之后，往前的量才追上 512，才不再需要 `-1` 填充。

【Step 4】**拼上 sink，softmax**——这一步是 sparse_attn 不"标准"的地方：

💡 常识科普

sink 本义是"水槽、下水口"，attention sink 直译就是**注意力的下水口**。这个词来自一个真实观察：研究者发现 LLM 会把大量注意力权重莫名其妙地堆在第一个 token（通常是没有语义的句首符）上。原因是 softmax 的硬约束——权重必须加起来等于 1，可有些 query 真的谁都不想看，这份"不想看"没法表达，多余的权重只能倒在一个无害的地方，模型自己学会了拿第一个 token 当垃圾桶。这个自发现象被命名为 attention sink。V4 做的是把它转正：与其浪费一个真 token 当垃圾桶，不如每个 head 直接配一个可学习的标量当专职下水口——就是下面代码里的 `attn_sink`。

```text
sink = attn_sink                    [h] → 广播成 [h, 1]
all_scores = concat([scores, sink], dim=-1)    [h, topk+1]
weights = softmax(all_scores, dim=-1)
```

scores 末尾被拼了一根 sink 列，然后整个 `topk+1` 长度一起过 softmax。sink 这一列得到的概率，被 softmax 当作"我不打算把权重分给真实位置"的份额。

【Step 5】**只取前 topk 个权重，乘 V**——sink 那一列已经完成它的使命，输出时不去碰它：

```text
attn_weights = weights[..., :topk]                    [h, topk]
o = einsum('ht, td -> hd', attn_weights, kv_gathered)
```

注意一个细节：`attn_weights` 的每一行加起来**不再等于 1**——因为 sink 那一列吞掉的那部分概率没回来。如果 attention 觉得"topk 个真实候选都不太行"，sink 那一列就吃掉很多概率，输出 `o` 的模长会变小。**这就是 attention sink 的作用：给 softmax 一个泄压阀，不强迫它把权重平均分给一堆并不重要的真实位置**。

170 期《Deepseek 的流形约束——一个 token 从球面滑到单纯形的旅程》（ https://mp.weixin.qq.com/s/ozog4BkollQAc7wzUkME0g ）讲过：softmax 把任意实数映射到概率单纯形上，权重必须加起来等于 1。没有 sink 的稀疏 attention 等于硬性要求"必须从这 topk 个里挑"——如果 topk 个里没一个特别相关，softmax 只能被迫均分（落到单纯形的中心），输出就贴近 value 的"平均值"，糊了。**sink 那一列等价于把单纯形多开了一维出口**——"我都不喜欢"这个选项是合法的。

💡 **打个比方**：超市试吃柜台 10 道菜让你打分。强制规则是"必须从这 10 道里挑一道带回家、不能拒绝"——你只能矮子里拔将军。加个 sink 等于柜台多了一个"我空手走"的选项——10 道菜都不行你就走。**走的这部分概率被丢掉，不混进你今天的菜篮子**。

为什么这一根 sink 不会被训练学成 0？因为它是一个可学习的标量，从训练数据里凝结出来的——当某些 query 真的没有合适候选时，sink 的存在能让 loss 下降；学术界研究过这件事（Xiao et al. *Efficient Streaming Language Models with Attention Sinks*, arXiv:2309.17453），attention sink 学到的往往是"无语义的 anchor"——它本身不代表任何 token 的意思，只是 softmax 系统在数学上需要一个"泄压口"。

到这里五步走完，这一个 query 位置拿到了自己的输出 `o`，形状 `[h, d]`，结束。

以上全程只站在"一个 query、面对它自己的 KV 缓存"这个视角——没提过 prefill，也没必要提：不管是逐个 token 生成时的解码，还是长 prompt 一次性喂进去的 prefill，每一个 query 位置要做的都是同一套五步。区别只在于要不要一次性给一大批 query 位置都算这五步——**prefill 本质上就是把这一整套逻辑，对一大批 query 位置并行算完，等价于把这些位置挨个按 KV 缓存的方式迭代一遍，只是批量做，不是真的一个一个来**。

能这么批量并行，是因为"依赖前面"这件事的真实含义容易被误解——**query 位置 s 依赖的是前面位置的 K、V（已经算好的、彼此独立的投影结果），不是依赖前面位置算完的 attention 输出**。K、V 本身是当前层输入激活值的直接投影，每个位置各投影各的，互相零依赖，所以哪怕是 17832 个位置，也能一次性摊平算完，没有谁必须等谁。真正躲不掉的串行只发生在**层与层之间**——第二层要用第一层**全部位置**的输出当输入，所以 43 层必须一层接一层往下走；但每一层内部，还是这套"KV 已经现成、所有位置一次并行摊开"的逻辑，跟第一层同构。第三章要讲的内存问题，根子就在"层内部批量摊开"这五个字上，不是这五步本身变复杂了。

顺带说一句这个"批量并行"给正常程序员的反直觉——**很多讲 attention 的教程会突然扔出一个"causal mask"，说"要把当前位置往后的都 mask 成 -inf"，然后就没了**。作为一个正常程序员看到这里第一反应一定是懵的：预测第 s 个位置的时候，for 循环从 0 到 s、只喂前 s 个 token 不就得了？为什么要**先算 S 个位置，然后把每个位置未来那部分擦掉**？听起来又蠢又浪费。

答案还是那句：**GPU 不允许你"少算"，只允许你"多算再作废"**。要并行训练/推理，就得一次性把 S 个位置的 attention 全算完；可数学上第 s 个位置又只该看到 [0..s-1]——**唯一能同时满足"一次算完"和"每人只看前面"的写法，就是全算 + 把未来 mask 成 -inf**。causal mask 不是什么优化小技巧，是 **"Transformer 能并行训练"这件事的物理前提** ——没有它，几十万亿 token 的预训练根本没法完成。跟 sparse_attn 里 -1 位置的 mask、跟 padding 的 mask 全是一个套路：**GPU 世界里所有"我不想看这个"最后都写成"算了但作废"**。这条价值观倒置不写破，读者会一直觉得"这些教程里到处都是 mask，肯定藏着什么高深道理"——其实没什么道理，就一句"GPU 就长这样"而已。

◆ 第三章：旧版为什么会 OOM——三个张量同时铺平

━━━━━━━━━━━━━━━━━━━━

上面这五步在 `s` 不大的时候完全够用。问题是 V4 真实跑起来时，prefill 阶段 `s` 直接就是整段 prompt——17832 都是常见数字。

我第一版 sparse_attn 是"全 `s` 一次性跑完"，这五步里三个中间张量同时活在显存里：

| 张量 | 形状 | b=1, s=17832, h=64, d=512, topk=512 单层占用 |
|---|---|---|
| `kv_gathered` | `[b, s, topk, d]` | 17832 · 512 · 512 · 2B（fp16）≈ **8.7 GB** |
| `scores` | `[b, s, h, topk]` | 17832 · 64 · 512 · 4B（fp32）≈ **2.2 GB** |
| `weights`（topk+1）| `[b, s, h, topk+1]` | 同量级 **2.2 GB** |

**单层就 13 GB 上下**。再考虑这五步都在浮点运算时还要再开几个临时中间值，单层峰值翻倍是常事。

单看 13 GB 好像不多，但要放进真实的账本里看：DGX Spark 单台 128 GB 统一内存，280B 的 FP4 权重双机对半分之后**单台常驻 82 GB**，21 层 CSA 每层还背两份 KV cache，真正留给中间张量的余量只有二三十 GB——13 GB 的峰值再加上临时副本，一跑就 OOM。

这就是为什么 sparse_attn 必须分块。

━━━━━━━━━━━━━━━━━━━━

◆ 第四章：query 分块——`SPARSE_ATTN_CHUNK = 512`

━━━━━━━━━━━━━━━━━━━━

分块思路非常朴素：

- **输出 `o` 整张铺好** —— `o = torch.empty_like(q)`，形状 `[b, s, h, d]` 一次性分配，约 1.1 GB（fp16）。这部分必须留着，它就是最终结果。
- **中间张量按 chunk 算** —— 每次只取 `q[:, s_start:s_end]` 和 `topk_idxs[:, s_start:s_end]`，把第二章的五步走一遍，得到 `o_chunk`，写回 `o[:, s_start:s_end]`。
- **下一个 chunk 复用同一组中间张量** —— Python 的引用 + PyTorch 的内存池让上一轮的 `kv_gathered / scores / weights` 自然被回收。

环境变量 `SPARSE_ATTN_CHUNK` 默认 512，chunk 内的中间张量变成：

| 张量 | 形状（s=512）| 占用 |
|---|---|---|
| `kv_gathered` | `[1, 512, 512, 512]` | ≈ 256 MB |
| `scores` | `[1, 512, 64, 512]` | ≈ 64 MB |
| `weights` | `[1, 512, 64, 513]` | ≈ 64 MB |

单 chunk 峰值在 **400 MB 量级**，相对之前一层 13 GB 是 30 倍的压缩。跑 17832 长度的 prompt，外层循环走 35 次，每次结束自然回收，峰值不再随 `s` 增长。

代码点睛（`kernel_sm121.py` 的 `sparse_attn` 函数节选）：

```python
o = torch.empty_like(q)
for s_start in range(0, s, chunk_size):
    s_end = min(s_start + chunk_size, s)
    q_chunk = q[:, s_start:s_end].float()
    idx_chunk = topk_idxs[:, s_start:s_end]
    # ... 五步 ...
    o[:, s_start:s_end] = einsum('bsht,bstd->bshd', attn_weights, kv_gathered)
    del q_chunk, idx_chunk, kv_gathered, scores, weights, attn_weights
```

最后那一行 `del` 是良心活儿——不写 PyTorch 也会在循环下一轮回收，但显式 `del` 让显存峰值的"屋顶"扁一点，对 DGX Spark 这种打满了 unified memory 的机器友好。

━━━━━━━━━━━━━━━━━━━━

【为什么 chunk = 512？】

没有多深的道理——两头的约束都很松：往大了切，单 chunk 中间张量线性涨，只要峰值别再把内存顶爆就行；往小了切，Python 循环次数变多，开销也就多一点。512 是在这个宽松区间里随手取的一档，让单 chunk 峰值落在几百 MB 量级，够用了。**没做过精细调参，所以干脆做成环境变量**——真在别的机器上跑出问题，调一调就行。

━━━━━━━━━━━━━━━━━━━━

◆ 第五章：和 Indexer 的对偶——内存问题不同源

━━━━━━━━━━━━━━━━━━━━

242 期 Indexer 也要分块。第一眼看上去两个算子"都因为 prompt 太长所以分块"——但仔细看，**它俩的内存问题完全不同源**。

| 算子 | 真正吃显存的中间张量 | 复杂度 |
|---|---|---|
| Indexer（242 期）| `index_scores` | **O(b · s · T)** ≈ O(S²/ratio) |
| sparse_attn | `kv_gathered`、`scores`、`weights` | **O(b · s · topk · d) + O(b · s · h · topk)** |

Indexer 的内存爆炸是因为它要算"每个 query 位置对每个候选 K 块的相关度"——s × T 是 O(S²/ratio) 的方阵（compressor 把 T 压到 S/4，但二次性还在）。S=17832 时这个矩阵就要 17832² / 4 ≈ 8000 万个 float，单层 100 多 MB；32 头一展开就 GB 级别。**Indexer 的二次性来自"全表筛选"这个动作本身**，不分块没法做。

sparse_attn 不一样。它的 `scores` 只有 `[b, s, h, topk]`——**对 s 是一次线性，对 topk 是另一次线性，没有二次项**。理论上 17832 × 64 × 512 这个张量只有 2 GB 量级，不至于爆炸。

那为什么 sparse_attn 在 17832 长度 prompt 上还是炸了（这不是推演，是先炸后修的实录）？公平地说，层与层是串行的，一层算完中间张量就释放，理论上不会跨层累积——问题出在**余量本来就没剩多少**：

1. **它有三个 O(S · topk) 量级的张量同时活着**（`kv_gathered`、`scores`、`weights`），不是一个——第三章算过，加起来单层 13 GB 上下，再算上 einsum 过程中的临时副本，峰值还要往上走
2. **地板已经很高**——就是第三章末尾那本账：权重 82 GB + 双份 KV cache 垫底，余量只有二三十 GB，13 GB+ 的峰值贴着天花板
3. **反复的大块分配制造碎片**——43 层每层都要"分配 13 GB → 释放 → 下一层再分配"，跑几层之后内存池的碎片化让"名义上够"变成"实际上分配不出来"

所以 Indexer 是"二次复杂度"的爆炸，sparse_attn 是"一次复杂度 × 大常数，撞上余量太小"的爆炸。**前者是数学问题，后者是工程问题**——前者就算余量再大，S 一涨数学上也兜不住；后者分块只是把峰值从"贴着天花板"打回到"舒服的区间"，机器富裕的话不分块也未必炸。

━━━━━━━━━━━━━━━━━━━━

【顺带的洞见：Indexer 和 sparse_attn 是 V4 长上下文的两根支柱】

一个负责"挑"（O(S²/ratio) 筛选 + 分块），一个负责"算"（O(S · topk · d) 加权 + 分块）。前者把候选从全表砍到 topk，后者在 topk 个候选上跑一次普通 attention。两个算子串起来，让 V4 的 attention 总开销从 dense 的 O(S² · d · h) 砍到 O(S · topk · d · h)——**当 topk ≪ T 时，省掉的是一个量级**。

168 期《MHA、MLA、DSA、CSA/HCA——从 V1 到 V4》（ https://mp.weixin.qq.com/s/YqcTnIrGKEYZ2n-sMAelKg ）讲 MHA → MLA → DSA → CSA/HCA 这条路时，说"注意力的演化战场永远在 KV 上"——MLA 砍 KV 维度，CSA 砍 KV 条数，到了 V4 的 DSA + sparse_attn，砍的是"每个 query 真正看的 KV 条数"。Indexer 砍候选量，sparse_attn 在砍完的底盘上算 attention。**两个算子组合起来才完成"稀疏"这件事**——单看 sparse_attn 自己，它其实是一段"普通 attention + sink 泄压阀"，稀疏的语义全在 Indexer 给的 topk_idxs 里。

━━━━━━━━━━━━━━━━━━━━

◆ 第六章：流形视角的一个小复习

━━━━━━━━━━━━━━━━━━━━

借 170 期"流形约束"的语言再走一遍这条流程。先把两个几何名词落地：

💡 常识科普

**单纯形（simplex）**：把 100% 的权重分给 N 个候选，所有可能的分法摊出来的那个形状。3 个候选时就是一个三角形——角落 = 全押一家（比如 (1,0,0)），正中心 = 三家均分，边上和内部是各种中间分法。2 个候选是线段，4 个是四面体，512 个候选就是它的 511 维版本，画不出来但图景不变。softmax 吐出的那组权重，永远落在这个形状内部的某一点。

**凸包（convex hull）**：把 512 个 V 向量看成空间里的 512 个点，用保鲜膜把它们整体裹一圈，膜围出来的那块区域。数学性质：只要权重非负、加起来等于 1，"加权平均"的结果就必然落在凸包内部——**跑不出去**。

有了这两个词，sparse_attn 的几何故事就三句话：

- **没有 sink 时，输出被锁死在凸包里**。softmax 权重和恒等于 1，所以输出 o 永远是 512 个 V 的某种加权平均、永远在凸包内——哪怕 512 个候选全都不相关，o 也必须硬凑一个"它们的混合物"出来，看起来跟正常输出一样理直气壮，实际是垃圾的平均值。
- **拼上 sink，等于给这套分配加了第 513 个选项**——一个只参与分权重、不贡献 V 的虚位。sink 分走概率 p 之后，真实候选的权重只剩 1-p，输出变成"(1-p) × 某种混合物"——**整体朝原点方向缩短了**。
- **于是输出的模长本身变成了一个信号**。一个 query 越是"无所适从"，sink 吃掉的 p 越大，输出向量越短——下游拿到一个"声音很小"的向量，自然不会被它带偏。没有 sink 的话，这个"我其实不确定"的信息根本没有通道表达出来。

顺带一句 Q·K 那头的几何：169 期《一个 token 的旅程——Deepseek V4 全流程总复习》（ https://mp.weixin.qq.com/s/mnbaXhQmQjDAyGD1dDdq5Q ）讲过的那两道无参 RMSNorm 把 Q 和 K 都钉在固定半径的球面上，所以打分用的点积等价于余弦相似度——分数的量纲从一开始就是干净的。

━━━━━━━━━━━━━━━━━━━━

◆ 第七章：复盘——这一节我学到了什么

━━━━━━━━━━━━━━━━━━━━

写到这里把这趟体感落一下：

1. **sparse_attn 是个组合名词，不是原子算子**——一个 gather、一个 Q·K、一个 mask、一个 sink concat、一个 softmax、一个 V 加权。任何一步不一样，attention 的语义就不一样。
2. **attention sink 不是 V4 新发明**——2023 年就有论文研究流式语言模型里的 sink 现象（Xiao et al.），V4 把它当成长上下文 + 稀疏选择下的标准配件：稀疏越狠，"我都不喜欢"这个出口越重要。
3. **稀疏的语义全在 Indexer 给的 topk_idxs 里**——sparse_attn 自己只是在小底盘上跑了一次普通 attention。Indexer 负责"看哪儿"，sparse_attn 负责"怎么用"。两个算子的责任边界很清楚。
4. **OOM 的根因不是稀疏不够稀，是 PyTorch 一次性铺平**——`[b, s, topk, d]` 这个张量光看维度不吓人，但 s=17832 一乘就是 13 GB，撞上"权重 82 GB + 双份 KV cache"垫高的地板，余量顶不住。分块是工程解，不是算法解。
5. **chunk size 两头的约束都很松**，随手取一档、做成环境变量就够了——别的机器出问题调一调就行，不值得精细调参。

下一期讲 **mHC Sinkhorn**——V4 把 mHC（multi-stream Hyper-Connection）混合矩阵约束在 Birkhoff 多面体上的那一步。170 期挖了个坑，Sinkhorn 这一步骤的几何被一句话带过，那期之后我一直想找机会展开。244 期补上。
