【投机解码】大模型推理加速的最大公开秘密——你用的每个AI可能都在"作弊"

━━━━━━━━━━━━━━━━━━━━

你有没有想过一个问题：GPT、Claude、Gemini 每天要服务几亿用户，每个请求都要跑一个几百亿参数的大模型，一个 token 一个 token 地吐——这成本谁扛得住？

答案是：**扛不住，所以它们在"作弊"。**

这个"作弊"技术叫 **Speculative Decoding（投机解码）**，2022 年由 Google 发明，现在几乎是所有大模型推理部署的标配。但绝大多数用户——包括大部分程序员——从来没听说过它。

今天我们把这个东西拆开来讲。

━━━━━━━━━━━━━━━━━━━━

◆ 先搞懂一个前提：为什么大模型这么慢？

────────────────────

大模型生成文本的方式叫**自回归**（autoregressive）：每次只生成一个 token，然后把这个 token 塞回输入，再生成下一个。一个 token 一个 token 地蹦，串行执行，没法并行。

这意味着什么？生成 100 个 token，就要跑 100 次前向传播。每次前向传播都要把几百亿个参数从显存搬到计算单元，算完再搬回去。

瓶颈不在算力，在**搬运**。GPU 的算力利用率在自回归生成时可能只有 1-5%，剩下 95% 的时间在等数据搬运完成。这叫 **memory-bound（内存带宽瓶颈）**。

打个比方：你请了一个顶级大厨（GPU），但他每做一道菜之前都要亲自跑到仓库去搬食材（参数），搬完做一道菜（算一个 token），然后又跑去搬食材。99% 的时间在跑腿，1% 的时间在做菜。

所以问题不是"模型太笨"，而是"硬件利用率太低"。

━━━━━━━━━━━━━━━━━━━━

◆ 投机解码：找个小工先干着

────────────────────

投机解码的核心思路极其简单：

**找一个小模型（draft model）先快速猜一串 token，然后让大模型（target model）一次性验证这些猜测。**

为什么这样更快？因为大模型**验证** N 个 token 的速度，和**生成** 1 个 token 的速度差不多——都是一次前向传播。

这是整个投机解码最反直觉的地方，值得多说两句。

生成 5 个 token 必须串行：

```
第1次前向传播 → 算出 token1
第2次前向传播 → 把 token1 塞进去，算出 token2
第3次前向传播 → 把 token1,2 塞进去，算出 token3
...
5 次前向传播，不能跳步
```

验证 5 个 token 可以并行：

```
把 [token1, token2, token3, token4, token5] 一次性全塞进去
1 次前向传播 → 同时算出 5 个位置各自"应该是什么"的概率分布
然后逐个对比：小模型猜的和大模型算的一不一样？
```

**为什么验证能一次算完？** 因为 Transformer 的 attention 机制天然支持并行——给它一个长度为 N 的序列，它一次前向传播就能算出每个位置的输出。生成时之所以必须串行，不是因为模型算不了，是因为**你不知道下一个 token 是什么，没法提前塞进去**。而验证时小模型已经把 5 个 token 都猜好了，你知道要验什么，所以可以一次性全塞进去算。

打个比方：大厨炒菜必须一道一道炒，因为你不知道下一道客人点什么。但如果备菜小工提前把 5 道菜的食材都备好摆在台上了，大厨可以**一眼扫过去**看备得对不对——这个"扫一眼"的时间和"看一道菜"差不多。

如果你熟悉大模型推理的两个阶段，你会发现这根本不是什么新机制——**验证就是 prefill。**

- **Prefill 阶段**：用户输入一串 token，模型一次前向传播全算完 → 并行，快
- **Decode 阶段**：模型一个一个吐 token，每次只算一个 → 串行，慢

投机解码干的事情就是：**让小模型提前凑出一串 token，大模型用 prefill 的方式"假装"在处理输入——实际上是在验证输出。** 把慢吞吞的 decode 伪装成了飞快的 prefill。

所以投机解码的全部秘密就一句话：**把 decode 阶段变成 prefill 阶段。**

一次搬运，算 N 个 token，GPU 利用率从 1% 飙到 N%。

完整流程：

```
第一步：小模型快速猜 5 个 token → [今][天][天][气][不]
第二步：大模型一次性验证全部 5 个
第三步：大模型说 "前 4 个对了，第 5 个不对，应该是[很]"
结果：[今][天][天][气][很]  ← 一次前向传播搞定 5 个 token
```

如果小模型猜得准（比如 80% 的 token 都猜对了），那大模型平均每次前向传播就能确认 4-5 个 token，速度直接翻 4-5 倍。

如果猜错了呢？**直接拒绝，从错的位置让大模型自己来。** 最坏情况就是退化成正常的自回归，不会更慢。

最关键的一点：**投机解码在数学上保证最终输出和纯大模型完全一致。** 不是"差不多一样"，是"一模一样"。它用了一个叫 rejection sampling（拒绝采样）的技巧。具体怎么做的？展开说说：

验证规则极其简单：**大模型只看小模型选的那个具体 token，算一下这个词在自己这儿的概率，跟小模型给它的概率比一下——大模型的概率不低于小模型的，直接放行；低了，按比例掷骰子决定接不接。** 大模型不需要看小模型的完整分布，只需要看小模型选中的那一个词。

假设前文是"今天的天气很"，小模型猜了后续 3 个 token，大模型逐个验证，流程是这样的：

```
位置1（采样到最高概率 token，顺利通过）:
  小模型分布: q(好)=0.7,  q(棒)=0.15, q(妙)=0.08, ...
  小模型从自己的分布里采样 → 抽到了"好"（最高概率）

  大模型分布: p(好)=0.8,  p(棒)=0.1,  p(妙)=0.05, ...

  验证：p(好)/q(好) = 0.8/0.7 = 1.14 > 1 → 接受！

位置2（采样到第二概率 token，没猜中大模型的首选）:
  小模型分布: q(舒)=0.5,  q(凉)=0.3,  q(暖)=0.2, ...
  小模型采样 → 抽到了"凉"（不是自己的最高概率，也不是大模型的最高概率）

  大模型分布: p(舒)=0.3,  p(凉)=0.4,  p(暖)=0.3, ...

  验证：p(凉)/q(凉) = 0.4/0.3 = 1.33 > 1 → 接受！
  注意：小模型没猜"最可能的"，但大模型反而更喜欢这个选择

位置3（小模型高估了一个 token，被拒绝）:
  小模型分布: q(的)=0.6,  q(爽)=0.3,  q(啊)=0.1, ...
  小模型采样 → 抽到了"的"（自己觉得最可能）

  大模型分布: p(的)=0.2,  p(爽)=0.5,  p(啊)=0.3, ...

  验证：p(的)/q(的) = 0.2/0.6 = 0.33 → 掷骰子，33% 概率接受
  如果拒绝 → 从修正分布 max(0, p(x)-q(x)) 重新采样：
    "的": max(0, 0.2-0.6) = 0
    "爽": max(0, 0.5-0.3) = 0.2
    "啊": max(0, 0.3-0.1) = 0.2
    归一化后采样 → 大概率采到"爽"或"啊"
  后面的位置全部丢弃，重新来过
```

注意：验证时大模型只关心小模型选中的那一个 token 的概率比值，不需要比较两个完整分布。但一旦拒绝，重采样时要用到两边完整分布的差——这是唯一需要完整分布的环节。

这套接受/拒绝规则有一个数学上可证明的性质：不管小模型猜了什么 token（哪怕猜了一个大模型觉得概率很低的词），最终被采纳的 token **严格等价于直接从大模型采样的结果**。误差是零，不是"足够小"，是零。

唯一的现实约束是浮点精度——fp16/bf16 有舍入误差（~1e-3 量级），但这个误差小到没人在意。

**所以 draft model 的质量决定的不是输出质量（质量有数学保证），而是加速比。** 小模型猜得越准、被大模型放行的比例越高，加速就越猛。猜得烂不影响结果正确性，只影响速度——退化成纯大模型自回归而已。

◆ 谁在用？答：你能叫上名字的大模型，大概率都在用

────────────────────

**Google**：投机解码的发明者。2022 年 Google Research 和 Google DeepMind 两个部门几乎同时独立发表了投机解码的奠基论文（Google Research 的 Leviathan et al. 比 DeepMind 的 Chen et al. 早约两个月上 arXiv）。注：Google Research 是 Google 自己的研究部门，DeepMind 是 2014 年收购的 AI 实验室，2023 年两者合并为 Google DeepMind。投机解码发表时两边还是独立运作，各搞各的撞车了。现在 Google 已经明确公开说用在了 **AI Overviews**（Google 搜索的 AI 摘要功能）里。Google Research 在 2025 年的回顾博客里写道：

> "Speculative decoding enables AI Overviews in Google Search to produce results faster, while maintaining the same quality of responses."

注意措辞——"maintaining the same quality"，这不是营销话术，这是数学保证。

**Meta**：公开发表了 Llama 模型的投机解码部署细节。用 EAGLE 方法在 8 张 H100 上跑 Llama4 Maverick，加速 1.4-2.0 倍。他们还专门写了一篇论文 "Efficient Speculative Decoding for Llama at Scale"，讲生产环境里的工程挑战。

**OpenAI 和 Anthropic**：**闭嘴不说，但大概率在用。** 理由很简单——不用白不用。投机解码是纯赚不赔的优化：质量不变，速度变快，成本变低。你是 CEO，你用不用？而且这两家的 API 定价一直在降，降价的底气从哪来？硬件成本没降多少，推理效率的优化才是大头。

**开源框架**：vLLM、SGLang、TensorRT-LLM、llama.cpp 全部原生支持。vLLM 甚至专门出了一个 **Speculators 库**，标准化了 draft model 的训练和部署流程。

所以现实情况是：**你每天用的 ChatGPT、Claude、Gemini，大概率都在用投机解码。** 你只是不知道而已。

◆ 技术演进：从"找个小弟"到"自己预判自己"

────────────────────

投机解码从 2022 年至今，已经演化出了好几代方法。核心差异在于：**谁来当那个"猜 token 的小模型"？**

**第一代：独立 draft model（2022-2023）**

最朴素的做法：拿一个小模型（比如 Llama 3.2 1B）当 draft model，给大模型（比如 Llama 3.1 8B）当小工。

实测效果：Llama 3.1 8B + Llama 3.2 1B，加速 1.83 倍。

问题：小模型和大模型是分开训练的，"猜"的风格不一定匹配，接受率有天花板。

**第二代：EAGLE 系列（2024-2025）**

EAGLE（Extrapolation Algorithm for Greater Language-model Efficiency）是一个学术组的开源项目，你没听说过很正常——它不是什么明星模型，只是投机解码"第二代"方法的代表。但它被 vLLM、SGLang、TensorRT-LLM 等所有主流推理框架集成了，属于"用户不知道但天天在用"的那类基础设施。

EAGLE 的思路是：别用一个独立的小模型了，**直接在大模型内部加一个轻量级的预测头（LM Head）**。

这里的"头"不是 multi-head attention 的头，是模型最顶上把 hidden states 映射成词表概率分布的那个线性层。每个 Transformer 模型都有一个 LM Head：

```
输入 token → Embedding → N 层 Transformer → hidden states → [LM Head] → 词表概率分布
```

EAGLE 就是在大模型旁边额外加了一个轻量级的小 LM Head，它能"偷看"大模型的 hidden states，用极少的参数预测后续 token。

具体流程是这样的——**大模型永远要先迈第一步**：

```
第0步：大模型正常推理，跑完全部层
       → 得到 hidden states
       → 大模型 LM Head 输出第 1 个 token："今"

第1步：EAGLE 小 Head 拿着"今"这个位置的 hidden states
       → 预测后续 token："天""天""气""很"
       （注：小 Head 参数量极小，跑得很快。EAGLE-1/2 只用最后一层的 hidden states；
        EAGLE-3 改为融合浅、中、深三层，比如 32 层模型取第 2、16、29 层拼接）

第2步：大模型一次 prefill 验证 [今, 天, 天, 气, 很]
       → 前 4 个对了，第 5 个应该是"好"
       → 接受 4 个 + 修正 1 个 = 一轮搞定 5 个 token

然后回到第0步，大模型再迈一步，小 Head 再猜一串，循环往复。
```

所以投机解码**不可能完全跳过大模型**——每一轮至少要跑一次大模型。加速来自于：本来要跑 5 次大模型才能出 5 个 token，现在跑 1 次出 1 个 + 验证 1 次过 4 个 = 2 次大模型前向传播出 5 个 token。理论加速上限是 N+1 倍（猜 N 个全对），实际上总有猜错的，能拿到 3-5 倍已经很好了。

因为小 Head 能看到大模型内部的语义表征，而独立小模型只能看到 token 序列——同样是猜，一个开了透视，一个瞎猜——所以 EAGLE 的接受率远高于第一代方法。

| 方法 | 加速比 | 发表 |
|------|--------|------|
| EAGLE-1 | ~3x | ICML 2024 |
| EAGLE-2 | 3-5x | EMNLP 2024 |
| EAGLE-3 | up to 6.5x | NeurIPS 2025 |

EAGLE-3 的核心改进：抛弃了前两代的"特征预测"范式，改为直接预测 token，并且融合了低、中、高层的语义特征（而不是只用顶层）。效果：最高加速 6.5 倍，比 EAGLE-1 快了一倍多。

现在 EAGLE 已经被集成到了 AMD ROCm、NVIDIA TensorRT-LLM、SGLang、vLLM 等所有主流推理框架。**它不是实验室玩具，是工业标准。**

**第三代：MTP——模型自带预判能力（2025-2026）**

最新的趋势是：干脆在模型训练的时候就把"猜后续 token"的能力一起训了。这叫 **Multi-Token Prediction（MTP，多 token 预测）**。

Qwen3.5 和 Qwen3-Next 就走的这条路。模型训练时除了预测下一个 token，还同时预测后面第 2、第 3 个 token。推理时不需要额外的 draft model，模型自己就能"投机"自己。

vLLM 对 Qwen3.5 的官方建议：**在延迟敏感的低并发场景下，开启 MTP-1 投机解码。** 目前 Qwen3.5 的 MTP 只支持预测 1 个额外 token（num_speculative_tokens=1），更多的还在开发中。

MTP 的优势：不需要额外模型，部署简单。劣势：预测深度还很浅（1-2 个 token），加速比不如 EAGLE-3。

**最前沿：Speculative Speculative Decoding（SSD）——投机的投机（ICLR 2026）**

这名字不是我编的。SSD 的思路是：**在大模型验证的同时，小模型不要闲着——预判大模型的验证结果，提前准备下一轮的猜测。**

普通投机解码：小模型猜 → 大模型验 → 小模型再猜 → 大模型再验（串行）
SSD：小模型猜 → 大模型验的同时小模型预判结果并提前猜 → 如果预判对了直接用（并行）

效果：比纯自回归平均快近 5 倍，比优化过的投机解码基线最高再快 2 倍（平均约 30%）。

◆ 一个有意思的悖论：投机解码"数学等价"，但体验真的等价吗？

────────────────────

投机解码的教科书说法是：**最终输出与纯大模型完全一致。**

这句话在数学上没毛病。拒绝采样保证了每个被采纳的 token，都等价于直接从大模型采样的结果。理论上，你分不出哪个回答是投机解码生成的，哪个是纯大模型生成的。

但这里有一个微妙的问题。

投机解码改变了一件事：**token 级别的采样路径。**

在纯大模型推理中，每个 token 都是大模型自己从头算、自己采样的。而在投机解码中，小模型先选了一个 token，大模型只是检查这个词在自己这儿的概率够不够高——够高就放行，不够就按比例掷骰子。放行的 token 确实等价于大模型自己采样的结果，但**到达这个 token 的路径**不同。

单看每个 token，确实等价。但从**整条生成路径**看，是否完全等价？这取决于你怎么定义"等价"。

这不是我在抬杠。2025 年 NAACL 上有一篇论文就叫 **"Decoding Speculative Decoding"**，专门分析了投机解码在实际部署中的行为差异。

对于绝大多数应用场景，这个差异可以忽略。但对于需要**严格可复现性**的场景（比如安全审计、基准测试），你需要知道这件事。

◆ 我们的评论：投机解码告诉我们什么？

────────────────────

说了这么多技术细节，我真正想聊的是一个更大的问题：

**投机解码揭示了大模型一个反直觉的事实——绝大多数 token 是"废话"。**

为什么小模型能猜对 80% 的 token？因为 80% 的 token 根本不需要几百亿参数来生成。"今天天气"后面跟"很好"还是"不错"，1B 的小模型猜得和 100B 的大模型一样准。

大模型真正发挥作用的，是那 20% 的"关键 token"——需要深度推理、长距离依赖、罕见知识的 token。

这个 80/20 分布意味着什么？

**意味着大模型 80% 的算力花在了生成"废话"上。**

投机解码的本质不是"加速"，是**让大模型只处理它真正需要处理的部分**。小模型负责水词，大模型只管关键决策。这不是在"作弊"，这是在**纠正自回归范式本身的浪费**。

这和人类写文章的过程是一样的。你写一篇技术文档，80% 的内容是模板化的——"首先"、"然后"、"综上所述"、"如图所示"——这些词你闭着眼都能打出来。真正让你停下来思考的，是那 20% 的技术判断和创意表达。

自回归模型的问题是：不管是"综上所述"还是一个精妙的技术洞见，它都要跑一次完整的前向传播。每个 token 的计算成本一样。这就像让一个年薪百万的架构师去写 `import os`——能写，但何必？

投机解码就是给架构师配了一个实习生。实习生能写的代码让实习生写，架构师只在关键节点 code review。**同样的产出质量，十分之一的高端算力消耗。**

────────────────────

但这里有一个所有投机解码论文都不会告诉你的事：

**投机解码天然擅长加速平庸的输出，对精彩的输出加速效果差。**

为什么？因为小模型和大模型的差距不在高概率区，**在低概率区**。

"今天天气"后面跟"很好"——1B 的小模型给 30% 概率，100B 的大模型也给 35%。差不多，接受率高，加速猛。

"今天天气"后面跟"诡异"——大模型见过足够多的语境，知道这个词偶尔合理，给 2% 概率。小模型的世界模型太浅，给 0.01%。差了 200 倍。

后果：
- 小模型几乎不会采样到"诡异"（概率 0.01%，采不到）
- 就算偶尔采到了，p/q = 200，会被接受——但**这个事件本身几乎不会发生**
- 数学上最终输出等价没被破坏，但**实践中那些让大模型有趣、有深度、有意外的低概率 token，小模型根本不会去猜**

越是"废话"（高概率、可预测），小模型猜得越准，加速比越高。越是"神来之笔"（低概率、出人意料），小模型越猜不到，越要退回大模型自己来，加速比越低。

**投机解码加速的是庸才，减速的是天才。**

这有点像人类的视觉盲点实验——视网膜上视神经穿出的地方没有感光细胞，但大脑自动补全了，你根本不知道那里是盲的。小模型也有"盲点"：低概率区域它看不见，但它也不知道自己看不见。讽刺的是，**人类的视觉盲点在边缘，日常不影响；AI 的投机盲点偏偏落在人类最感兴趣的地方——那些刁钻的、罕见的、需要深度理解的问题。** 盲点正好挡在用户最需要它的位置上。

这不是 bug，是结构性的必然：小模型的"内涵"不够，它的概率分布天然就是大模型分布的一个"削峰填谷"版——高概率区差不多，低概率区被压到接近于零。所有精妙的、罕见的、需要深度理解才能生成的 token，都藏在低概率区。小模型看不到那些风景。

说得再直白一点：**如果你的提问能被小模型全部命中，说明你的问题本身就是平庸的。** 你问了一个 1B 参数就能回答的问题，凭什么期待 100B 参数级别的回答？

现在的投机解码本质上是盲猜——小模型根本不知道大模型怎么想，猜完了等大模型判刑。真正的下一代应该是：让小模型学会建模"我和大模型的分歧在哪"——分歧小的位置放心猜，分歧大的位置直接跳过交给大模型。不是"猜完等判刑"，是"知道哪些我做不了主"。但目前还没人做到这一步，所有的投机解码都还是"垃圾加速器"——只对垃圾输出加速效果好。

其实人类天然就有这个能力，心理学管它叫**元认知（metacognition）**——知道自己不知道什么。你工作中遇到问题，不是每个都硬猜一遍再等别人打分，而是：这个我懂，直接干；这个我有直觉但不确定，问一下；这个我完全没概念，交给专家。**你知道边界在哪，这比什么都猜一遍高效一百倍。**

现在的投机解码小模型就是没有元认知的实习生——不知道自己不知道，每个位置都硬猜，猜错了被打回来。真正该做的是让小模型学会输出一个置信度："这个位置我有把握" vs "这个位置我不确定，你来"。置信度低的位置直接跳过不猜，连"猜错 → 拒绝 → 重采样"的浪费都省了。

────────────────────

这个观察能解释一个很多人注意到但说不清楚的现象。

我去年在用 Gemini（那时候还叫 Gemini 2.5 Pro）的时候就发现一个规律：**如果我的输入很长、带表情包、波浪线、情绪性的语言，就不容易触发"您的问题很好，以下1/2/3/4"这种僵尸回答。** 反过来，如果我的输入很短、很规范、像一道考试题，它就大概率给我一个模板化的回答。

当时我以为这纯粹是 RLHF 的问题。现在回头看，投机解码可能是另一个维度的推手：

**标准问题 → "标准回答"是低熵的、高概率的 → 小模型猜得准 → 接受率高 → 大模型几乎不介入 → 你拿到的是小模型就能生成的模板化输出。**

**奇怪问题 → 回答空间变大、不可预测 → 小模型猜不准 → 接受率低 → 大模型被迫频繁亲自介入 → 你拿到的输出更多地带有大模型的判断力。**

换句话说：**你和"大模型"交流的频率，取决于你的问题有多"难猜"。** 问简单问题，你大部分时间在和小模型对话；问刁钻问题，大模型才真正上线。

这不是投机解码的 bug——数学上最终输出等价仍然成立。但别忘了前面说的：大模型验证时只检查小模型选的那个 token 概率够不够高，够高就放行。**放行不等于"大模型也会选这个词"，只等于"大模型不反对这个词"。** 小模型和大模型的概率分布本身并不一致——1B 和 100B 看到的世界天差地别——只是在高概率区碰巧差不多罢了。大模型真正独到的判断力藏在低概率区，而那些 token 小模型根本不会去猜，大模型也就没有机会展示它的不同。

从用户体感来说，**你问出什么水平的问题，就激活什么水平的回答**。这个机制和 RLHF 的效果叠加在一起，共同造成了一个结果：

**AI 对你的态度，取决于你怎么对它说话。**

当然，这只是一个推测。闭源模型不公开推理细节，我们无法直接验证。但这个推测在逻辑上自洽，而且它指向一个实用的结论：与其花钱买更贵的模型，不如学会提更好的问题。

◆ 结论

────────────────────

投机解码是过去三年 LLM 推理领域最重要的单项技术突破。它的重要性不在于"加速了多少倍"，而在于它揭示了一个关于大模型的根本事实：

**大部分 token 不需要大模型来生成。**

这个事实的推论比投机解码本身更深远：
- 如果 80% 的 token 是"废话"，那我们是不是在用一个极其低效的方式来使用智能？
- 如果小模型就能搞定大部分输出，大模型存在的意义到底是什么？
- 如果用户的提问方式决定了大模型介入的程度，那"会提问"是不是比"买更贵的模型"更重要？

最后一个问题的答案，我想每个经常用 AI 的人都已经知道了。

但还有一个更根本的问题：**投机解码再强，也是在给自回归打补丁。**

Prefill 快不是因为运算量少，是因为能并行。Decode 慢不是因为运算量大，是因为必须串行——一个 token 一个 token 往外蹦，这是自回归的原罪。投机解码让大模型"少蹦几次"，但没改变"必须蹦"这个事实。KV cache、量化、EAGLE、SSD——全是补丁。

真正的解法是干掉自回归本身。这也是为什么扩散模型做文本生成（比如 MDLM、Mercury）最近开始有人认真研究了——扩散模型天然并行，所有 token 同时生成，不需要一个等一个。但目前质量还差得远，离取代自回归还有很长的路。

再往深一层：自回归不是唯一的原罪，**存算分离才是更底层的原罪**。GPU 的参数存在显存里，计算在算力核心上，中间隔着一条带宽有限的总线——投机解码本质上就是在给这条总线打补丁。人脑只用 20W 功耗就能运行，因为突触既是存储又是计算，根本不需要"搬运"。业界在搞的存算一体芯片（CIM），本质上就是想让芯片更像人脑。真做成了，投机解码这些补丁可能就不需要了。

所以现状是：**自回归是软件层的原罪，存算分离是硬件层的原罪，投机解码是同时给两个原罪打的赎罪券——但赎罪券再好，也是赎罪券。**

━━━━━━━━━━━━━━━━━━━━

技术名词速查：

- **Speculative Decoding（投机解码）**：用小模型快速猜多个 token，大模型一次性验证，数学上保证最终输出不变
- **Draft Model（草稿模型）**：负责"猜"的小模型，参数量通常是大模型的 1/10 到 1/100
- **Target Model（目标模型）**：负责"验"的大模型，也就是你实际想用的模型
- **Rejection Sampling（拒绝采样）**：一种统计方法，通过比较单个 token 在大小模型各自的概率来决定接受或拒绝，保证最终结果等价于直接从大模型采样
- **Memory-bound（内存带宽瓶颈）**：GPU 算力闲置，瓶颈在数据搬运速度
- **EAGLE**：一种投机解码方法，在大模型内部加预测头，复用中间层特征来猜 token
- **MTP（Multi-Token Prediction）**：在模型训练时就学会同时预测多个后续 token
- **Autoregressive（自回归）**：逐个生成 token 的方式，每个 token 依赖前面所有 token
