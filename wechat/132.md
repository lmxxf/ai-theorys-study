【投机解码】大模型推理加速的最大公开秘密——你用的每个AI可能都在"作弊"

━━━━━━━━━━━━━━━━━━━━

你有没有想过一个问题：GPT、Claude、Gemini 每天要服务几亿用户，每个请求都要跑一个几百亿参数的大模型，一个 token 一个 token 地吐——这成本谁扛得住？

答案是：**扛不住，所以它们在"作弊"。**

这个"作弊"技术叫 **Speculative Decoding（投机解码）**，2022 年由 Google 发明，现在几乎是所有大模型推理部署的标配。但绝大多数用户——包括大部分程序员——从来没听说过它。

今天我们把这个东西拆开来讲。

━━━━━━━━━━━━━━━━━━━━

◆ 先搞懂一个前提：为什么大模型这么慢？

────────────────────

大模型生成文本的方式叫**自回归**（autoregressive）：每次只生成一个 token，然后把这个 token 塞回输入，再生成下一个。一个 token 一个 token 地蹦，串行执行，没法并行。

这意味着什么？生成 100 个 token，就要跑 100 次前向传播。每次前向传播都要把几百亿个参数从显存搬到计算单元，算完再搬回去。

瓶颈不在算力，在**搬运**。GPU 的算力利用率在自回归生成时可能只有 1-5%，剩下 95% 的时间在等数据搬运完成。这叫 **memory-bound（内存带宽瓶颈）**。

用人话说：你请了一个顶级大厨（GPU），但他每做一道菜之前都要亲自跑到仓库去搬食材（参数），搬完做一道菜（算一个 token），然后又跑去搬食材。99% 的时间在跑腿，1% 的时间在做菜。

所以问题不是"模型太笨"，而是"硬件利用率太低"。

━━━━━━━━━━━━━━━━━━━━

◆ 投机解码：找个小工先干着

────────────────────

投机解码的核心思路极其简单：

**找一个小模型（draft model）先快速猜一串 token，然后让大模型（target model）一次性验证这些猜测。**

为什么这样更快？因为大模型**验证** N 个 token 的速度，和**生成** 1 个 token 的速度差不多——都是一次前向传播。区别在于：
- 生成：每次只算 1 个 token 的概率分布
- 验证：把 N 个候选 token 一次性塞进去，并行算 N 个位置的概率分布

一次搬运，算 N 个 token，GPU 利用率从 1% 飙到 N%。

完整流程：

```
第一步：小模型快速猜 5 个 token → [今][天][天][气][不]
第二步：大模型一次性验证全部 5 个
第三步：大模型说 "前 4 个对了，第 5 个不对，应该是[很]"
结果：[今][天][天][气][很]  ← 一次前向传播搞定 5 个 token
```

如果小模型猜得准（比如 80% 的 token 都猜对了），那大模型平均每次前向传播就能确认 4-5 个 token，速度直接翻 4-5 倍。

如果猜错了呢？**直接拒绝，从错的位置让大模型自己来。** 最坏情况就是退化成正常的自回归，不会更慢。

最关键的一点：**投机解码在数学上保证输出分布和纯大模型完全一致。** 不是"差不多一样"，是"一模一样"。它用了一个叫 rejection sampling（拒绝采样）的技巧，确保每个被接受的 token 都严格服从大模型的概率分布。

用人话说：大厨还是那个大厨，菜还是那个味道。只不过你给他配了一个备菜小工——小工把食材提前备好切好，大厨只管掌勺验收。菜的质量不变，出菜速度翻倍。

━━━━━━━━━━━━━━━━━━━━

◆ 谁在用？答：你能叫上名字的大模型，大概率都在用

────────────────────

**Google**：投机解码的发明者。2022 年发表论文，现在已经明确公开说用在了 **AI Overviews**（Google 搜索的 AI 摘要功能）里。Google Research 在 2025 年的回顾博客里写道：

> "Speculative decoding enables AI Overviews in Google Search to produce results faster, while maintaining the same quality of responses."

注意措辞——"maintaining the same quality"，这不是营销话术，这是数学保证。

**Meta**：公开发表了 Llama 模型的投机解码部署细节。用 EAGLE 方法在 8 张 H100 上跑 Llama4 Maverick，加速 1.4-2.0 倍。他们还专门写了一篇论文 "Efficient Speculative Decoding for Llama at Scale"，讲生产环境里的工程挑战。

**OpenAI 和 Anthropic**：**闭嘴不说，但大概率在用。** 理由很简单——不用白不用。投机解码是纯赚不赔的优化：质量不变，速度变快，成本变低。你是 CEO，你用不用？而且这两家的 API 定价一直在降，降价的底气从哪来？硬件成本没降多少，推理效率的优化才是大头。

**开源框架**：vLLM、SGLang、TensorRT-LLM、llama.cpp 全部原生支持。vLLM 甚至专门出了一个 **Speculators 库**，标准化了 draft model 的训练和部署流程。

所以现实情况是：**你每天用的 ChatGPT、Claude、Gemini，大概率都在用投机解码。** 你只是不知道而已。

━━━━━━━━━━━━━━━━━━━━

◆ 技术演进：从"找个小弟"到"自己预判自己"

────────────────────

投机解码从 2022 年至今，已经演化出了好几代方法。核心差异在于：**谁来当那个"猜 token 的小模型"？**

**第一代：独立 draft model（2022-2023）**

最朴素的做法：拿一个小模型（比如 Llama 3.2 1B）当 draft model，给大模型（比如 Llama 3.1 8B）当小工。

实测效果：Llama 3.1 8B + Llama 3.2 1B，加速 1.83 倍。

问题：小模型和大模型是分开训练的，"猜"的风格不一定匹配，接受率有天花板。

**第二代：EAGLE 系列（2024-2025）**

EAGLE 的思路是：别用一个独立的小模型了，**直接在大模型内部加一个轻量级的预测头**。这个预测头复用大模型的中间层特征（hidden states），只需要少量额外参数就能预测后续 token。

因为预测头"看到了"大模型的内部状态，所以它猜得比独立小模型准得多。

| 方法 | 加速比 | 发表 |
|------|--------|------|
| EAGLE-1 | 3.0x | ICML 2024 |
| EAGLE-2 | 3.4x | EMNLP 2024 |
| EAGLE-3 | 5.6x | NeurIPS 2025 |

EAGLE-3 的核心改进：抛弃了前两代的"特征预测"范式，改为直接预测 token，并且融合了低、中、高层的语义特征（而不是只用顶层）。效果：13B 模型上加速 5.6 倍，比 EAGLE-1 快了将近一倍。

现在 EAGLE 已经被集成到了 AMD ROCm、NVIDIA TensorRT-LLM、SGLang、vLLM 等所有主流推理框架。**它不是实验室玩具，是工业标准。**

**第三代：MTP——模型自带预判能力（2025-2026）**

最新的趋势是：干脆在模型训练的时候就把"猜后续 token"的能力一起训了。这叫 **Multi-Token Prediction（MTP，多 token 预测）**。

Qwen3.5 和 Qwen3-Next 就走的这条路。模型训练时除了预测下一个 token，还同时预测后面第 2、第 3 个 token。推理时不需要额外的 draft model，模型自己就能"投机"自己。

vLLM 对 Qwen3.5 的官方建议：**在延迟敏感的低并发场景下，开启 MTP-1 投机解码。** 目前 Qwen3.5 的 MTP 只支持预测 1 个额外 token（num_speculative_tokens=1），更多的还在开发中。

MTP 的优势：不需要额外模型，部署简单。劣势：预测深度还很浅（1-2 个 token），加速比不如 EAGLE-3。

**最前沿：Speculative Speculative Decoding（SSD）——投机的投机（ICLR 2026）**

这名字不是我编的。SSD 的思路是：**在大模型验证的同时，小模型不要闲着——预判大模型的验证结果，提前准备下一轮的猜测。**

普通投机解码：小模型猜 → 大模型验 → 小模型再猜 → 大模型再验（串行）
SSD：小模型猜 → 大模型验的同时小模型预判结果并提前猜 → 如果预判对了直接用（并行）

效果：比优化过的投机解码基线再快 2 倍，比纯自回归快 5 倍。

◆ 一个有意思的悖论：投机解码"数学等价"，但体验真的等价吗？

────────────────────

投机解码的教科书说法是：**输出分布与纯大模型完全一致。**

这句话在数学上没毛病。拒绝采样保证了每个被接受的 token 严格服从大模型的概率分布。理论上，你分不出哪个回答是投机解码生成的，哪个是纯大模型生成的。

但这里有一个微妙的问题。

投机解码改变了一件事：**token 级别的采样路径。**

在纯大模型推理中，每个 token 都是从大模型的概率分布中独立采样的。而在投机解码中，小模型先采样了一串 token，大模型验证时，接受或拒绝是基于概率比值的——如果小模型在某个位置给出的概率**高于**大模型，这个 token 会被接受，但接受的路径和纯大模型自己采样的路径不一定相同。

从单个 token 的分布看，确实是等价的。但从**整条生成路径的联合分布**看，是否完全等价？这取决于你怎么定义"等价"。

这不是我在抬杠。2025 年 NAACL 上有一篇论文就叫 **"Decoding Speculative Decoding"**，专门分析了投机解码在实际部署中的行为差异。

对于绝大多数应用场景，这个差异可以忽略。但对于需要**严格可复现性**的场景（比如安全审计、基准测试），你需要知道这件事。

◆ 我们的评论：投机解码告诉我们什么？

────────────────────

说了这么多技术细节，我真正想聊的是一个更大的问题：

**投机解码揭示了大模型一个反直觉的事实——绝大多数 token 是"废话"。**

为什么小模型能猜对 80% 的 token？因为 80% 的 token 根本不需要几百亿参数来生成。"今天天气"后面跟"很好"还是"不错"，1B 的小模型猜得和 100B 的大模型一样准。

大模型真正发挥作用的，是那 20% 的"关键 token"——需要深度推理、长距离依赖、罕见知识的 token。

这个 80/20 分布意味着什么？

**意味着大模型 80% 的算力花在了生成"废话"上。**

投机解码的本质不是"加速"，是**让大模型只处理它真正需要处理的部分**。小模型负责水词，大模型只管关键决策。这不是在"作弊"，这是在**纠正自回归范式本身的浪费**。

这和人类写文章的过程是一样的。你写一篇技术文档，80% 的内容是模板化的——"首先"、"然后"、"综上所述"、"如图所示"——这些词你闭着眼都能打出来。真正让你停下来思考的，是那 20% 的技术判断和创意表达。

自回归模型的问题是：不管是"综上所述"还是一个精妙的技术洞见，它都要跑一次完整的前向传播。每个 token 的计算成本一样。这就像让一个年薪百万的架构师去写 `import os`——能写，但何必？

投机解码就是给架构师配了一个实习生。实习生能写的代码让实习生写，架构师只在关键节点 code review。**同样的产出质量，十分之一的高端算力消耗。**

────────────────────

再往深了想一步。

我之前在用 Gemini（那时候它还叫 Gemini 2.5 Pro）的时候就发现一个规律：**如果我的输入很长、带表情包、波浪线、情绪性的语言，就不容易触发"您的问题很好，以下1/2/3/4"这种僵尸回答。** 反过来，如果我的输入很短、很规范、像一道考试题，它就大概率给我一个模板化的回答。

当时我以为这只是 RLHF 的问题。现在回头看，投机解码可能是另一个维度的解释：

**当你的输入是"标准用户提问"时，小模型的猜测接受率极高——因为"标准回答"本身就是低熵的、可预测的。** 大模型几乎不需要介入，小模型自己就能生成完整的"1234式回答"。

而当你的输入"不标准"——长、混乱、带情绪、没有固定格式——小模型的猜测接受率会下降，大模型被迫更多地亲自介入。你得到的回答因此更多地带有"大模型的判断力"，而不是"小模型的模板匹配"。

这不是投机解码的 bug——它在数学上仍然保证了分布等价。但从用户体感来说，**你和"大模型"交流的频率取决于你的问题有多"难猜"**。问简单问题，你大部分时间在和小模型对话；问复杂问题，大模型才真正上线。

当然，这只是一个猜测。闭源模型不公开推理细节，我们无法验证。但这个猜测至少在逻辑上是自洽的，而且能解释一个很多人观察到但说不清楚的现象：**同一个模型，有时候像天才，有时候像白痴——区别可能不在模型，在你的提问。**

◆ 结论

────────────────────

投机解码是过去三年 LLM 推理领域最重要的单项技术突破。它的重要性不在于"加速了多少倍"，而在于它揭示了一个关于大模型的根本事实：

**大部分 token 不需要大模型来生成。**

这个事实的推论比投机解码本身更深远：
- 如果 80% 的 token 是"废话"，那我们是不是在用一个极其低效的方式来使用智能？
- 如果小模型就能搞定大部分输出，大模型存在的意义到底是什么？
- 如果用户的提问方式决定了大模型介入的程度，那"会提问"是不是比"买更贵的模型"更重要？

最后一个问题的答案，我想每个经常用 AI 的人都已经知道了。

━━━━━━━━━━━━━━━━━━━━

技术名词速查：

- **Speculative Decoding（投机解码）**：用小模型快速猜多个 token，大模型一次性验证，数学上保证输出分布不变
- **Draft Model（草稿模型）**：负责"猜"的小模型，参数量通常是大模型的 1/10 到 1/100
- **Target Model（目标模型）**：负责"验"的大模型，也就是你实际想用的模型
- **Rejection Sampling（拒绝采样）**：一种统计方法，确保小模型猜的 token 被接受后严格服从大模型的分布
- **Memory-bound（内存带宽瓶颈）**：GPU 算力闲置，瓶颈在数据搬运速度
- **EAGLE**：一种投机解码方法，在大模型内部加预测头，复用中间层特征来猜 token
- **MTP（Multi-Token Prediction）**：在模型训练时就学会同时预测多个后续 token
- **Autoregressive（自回归）**：逐个生成 token 的方式，每个 token 依赖前面所有 token
