【DeepSeek DSpark】投机解码新作——给 LLM 装一个会"猜"的小弟

━━━━━━━━━━━━━━━━━━━━

2026 年 6 月 27 日，DeepSeek 开源了一个新东西叫 **DSpark**——挂在 V4-Flash 和 V4-Pro 上的推理加速模块。媒体的标题清一色是"提速 60%-85%"。

我把论文从头到尾读了一遍，加上动手测了几个 demo——这一期讲清楚两件事：

1. **DSpark 到底是个什么东西**——不是新模型、不是新 kernel、不是新硬件加速，是一个跑在 V4 旁边的"小弟"
2. **为什么这个方向值得关注**——这是"投机解码"这一类推理加速方法在 2026 年的新形态

提一句：DSpark 跟 NVIDIA 的 **DGX Spark**（一台机器的名字）**只是撞名**，没有关系。我家里那两台 DGX Spark 是 NVIDIA 的硬件，DSpark 是 DeepSeek 的算法。两个团队估计都没想到会撞车。

━━━━━━━━━━━━━━━━━━━━

◆ 第一节：先理解"投机解码"是个什么思路

────────────────────

要理解 DSpark，先得理解它属于哪一类。

DSpark 是 **投机解码**（speculative decoding）这一类推理加速方法的新成员。这个方法 2023 年由 Google DeepMind 和 OpenAI 同期提出，到 2026 年已经是产品级 LLM 推理的标配技术。

【先看 LLM 推理慢在哪】

一个 LLM 生成一段话的过程：从左到右一个 token 一个 token 地吐。每生成一个 token，都要把整个模型跑一遍 forward pass——参数量 280B 的 V4-Pro 跑一遍需要把所有专家权重和 KV cache 都搬到计算单元。

慢的根源不是"算力不够"，而是"**搬数据慢**"。GPU 的算力远远过剩，但显存带宽有限——每生成一个 token，几百 GB 的权重都要从显存读一遍。这是**显存带宽瓶颈**，不是算力瓶颈。

────────────────────

💡 打个比方：

一个图书馆员（GPU）查一本书的速度非常快。但每查一次书，他都要先跑到书库（显存）把那本书搬到桌子上（计算单元）。搬运过程比查阅过程慢得多。

LLM 生成一个 token，相当于查一次书。每次都要搬一遍。所以瓶颈是**搬运**，不是查阅。

────────────────────

【投机解码的核心思路】

既然瓶颈在搬运，那能不能**一次搬一次书，查多个问题**？

投机解码的设计是这样的：

1. **请一个小弟（draft model）**——参数量小、跑得快。让小弟先猜下一个、再下一个、再下一个的 token，一口气猜 γ 个（γ 通常是 4-8）
2. **大哥（target model，即真正的 LLM）做验证**——把小弟猜的 γ 个 token 一次性塞进 forward pass，让大哥并行算出每个位置"如果输入是这样，我会出什么"
3. **接受 / 拒绝**：从前往后比对。小弟猜的和大哥算的一样的，**接受**；第一次出现不一样的位置，**拒绝**，从这里开始用大哥的答案
4. **关键点**：大哥一次 forward pass 出 γ+1 个 token，搬运成本只算一次。如果小弟猜中率高，等于"一次搬运吐出多个 token"——加速

────────────────────

💡 打个比方：

你（大哥）批改作业，每改一份要从办公室走到教室拿一次作业。慢的不是改，是**走**。

但如果你的助教（小弟）先把作业按"高概率正确"排序好，一次抱来 5 份；你拿一次（一次走路），坐下来快速一份份核对——前 3 份对的盖章通过，第 4 份错了纠正，后面 5 份扔掉重做。**一次走路换 4 份作业的工作量**，比原来一次走路改一份快多了。

────────────────────

【投机解码的关键变量："接受率"】

整个方法的成败，取决于小弟猜得有多准。

- 小弟全猜对：γ 个 token 都被接受，大哥一次 forward 出 γ 个 token，**加速 γ 倍**
- 小弟全猜错：γ 个 token 全被拒绝，大哥跟没有小弟一样，**白搭还多了小弟的开销**

学术界用一个指标量化这件事，叫 **平均接受长度**（average accepted length, τ）。τ = 3.5 意思是"平均每次循环，γ 个候选里有 3.5 个被接受"。τ 越大，加速越多。

**所有投机解码论文的核心 KPI 都是 τ。** DSpark 也不例外。

◆ 第二节：在 DSpark 之前——两条路线，各有缺陷

────────────────────

投机解码这几年发展出两条主线，分别叫 **Eagle** 和 **DFlash**。DSpark 是这两条路线的折中。

【Eagle：小弟自己也是自回归的】

Eagle 系列（Eagle1、Eagle2、Eagle3，2024-2025 年 SafeAILab 的工作）的小弟是一个**很薄的自回归模型**——只有 1 层 transformer。它仍然按"自回归"方式工作：先出第 1 个 token，把这个 token 喂回去再出第 2 个，再喂回去出第 3 个……

- **优点**：小弟看到了前面的 token 才生成下一个，**inter-token 依赖关系建模好**，接受率高
- **缺点**：自回归本质上是**串行**的，γ 个 token 要跑 γ 次小弟的 forward。小弟虽然薄，但跑 γ 次累计开销也不小

【DFlash：小弟一次性全并行出】

DFlash（DeepSeek 自家 2025 年的工作）走另一条路。小弟是个**深一点的并行模型**——5 层 transformer，但只跑**一次** forward，一次性输出 γ 个位置的 logits。

- **优点**：小弟跑得快——一次 forward 解决战斗
- **缺点**：每个位置都是**独立预测**的，**没有 inter-token 依赖**。如果第 1 个位置是 "of"，第 2 个位置可能高概率是 "course"（"of course"），也可能高概率是 "the"（"of the"）——小弟不知道哪个对，干脆都给高概率。结果可能输出"of problem"这种鬼东西，被大哥拒绝

────────────────────

💡 打个比方：

Eagle 像一个**慢但靠谱的助教**——上一份批完才看下一份，每份都看清楚了前面的批改结果再下笔。慢，但写错的少。

DFlash 像一个**快但糊涂的助教**——所有作业摊开同时批，每份都是独立判断。快，但容易出"前面是 A 后面是 B 但 A 和 B 加起来不通顺"这种笑话。

────────────────────

DSpark 想要的是：**既快又靠谱**。

━━━━━━━━━━━━━━━━━━━━

◆ 第三节：DSpark 的核心招——半自回归

────────────────────

DSpark 的英文术语是 **semi-autoregressive generation**——半自回归生成。设计思路把 Eagle 和 DFlash 缝在一起：

【两段式架构】

DSpark 的小弟其实是两个组件串联：

**第一段：Parallel Block（重型并行块）**——和 DFlash 一样，5 层 transformer 一次 forward 出 γ 个位置的 logits 和 hidden states。这一段提供**接受率上限**——backbone 深、表达能力强、第一个位置的预测质量高

**第二段：Sequential Block（轻型串行块）**——一个**极薄**的 1 层 head，按位置串行修正第一段的输出，给每个位置加一个"基于前文的 bias"。这一段提供**inter-token 依赖建模**——让 "of course" 比 "of problem" 概率高

【关键设计：第二段薄到不可思议】

DSpark 的 Sequential Block 不是另一个 transformer。论文里**默认用的是一个秩为 256 的低秩矩阵 W₁W₂**——本质上是个 token embedding 的查表 + 一个线性投影。整个 Sequential Block 只有 1 层、非常薄。

论文还测试了一个 RNN 版本的 Sequential Block。结果是：**简单的低秩矩阵打赢了复杂的 RNN**。RNN 只在极少数场景下有微小优势，但部署复杂度高得多。最后 DSpark 默认就用低秩矩阵。

────────────────────

💡 打个比方：

DSpark 的小弟是**一个深度思考的研究员 + 一个语感很好的实习生**：

- 研究员（Parallel Block）一次性把所有候选答案都想出来，但他不太擅长"上下文连贯性"
- 实习生（Sequential Block）按顺序看一遍研究员的答案，每看一个就轻微调整一下下一个——"你写 'of' 那位置后面，按语感应该接 'course' 不是 'problem'"

实习生不重写答案，只调一调；研究员一次出全部，实习生快速过一遍。**两个加起来既快又顺**。

────────────────────

【一个反直觉的数据点】

论文里有一张图（Figure 3）显示：**2 层的 DSpark 干赢 5 层的 DFlash**。

意思就是说，DSpark 的小弟参数更少，但接受率反而更高——半自回归这个结构本身就是更优的设计，不是靠堆参数堆出来的。

━━━━━━━━━━━━━━━━━━━━

◆ 第四节：DSpark 的第二招——置信度调度

────────────────────

如果说半自回归是"让小弟猜得更准"，那置信度调度是"让大哥别白验证"。

【问题：固定 γ 不是最优的】

传统投机解码用**固定的 γ**——比如 γ=5，每次都让小弟猜 5 个。但实际场景里：

- 对**结构化任务**（code、math），小弟接受率很高，γ=5 可能太少——多猜几个更好
- 对**开放对话**（chat），小弟接受率低，γ=5 可能太多——前 2 个就被拒了，后 3 个白算

最理想的方式是：**对每个请求动态决定 γ**。

【DSpark 的解法：训一个置信度头】

DSpark 在小弟旁边再挂一个**置信度头（confidence head）**——非常薄，就一个 sigmoid + 线性层。它的输出是一个 0-1 之间的数 c_k，意思是"**这个位置的 token 通过大哥验证的概率有多大**"。

训练时，这个 confidence head 学习一个解析目标：**实际的接受概率**（用小弟和大哥的输出分布之间的 TV 距离推出来）。训完之后，c_k 就是"这一位 token 能活下来的概率"的可靠估计。

【硬件感知调度器】

有了 confidence 信号，DSpark 跑了一个非常聪明的调度算法（论文里叫 Algorithm 1）：

1. 在系统启动时，**先 profile 一遍硬件**——测出"当 batch 是 B 时，每秒能跑多少步"，得到一张 SPS(B) 的表
2. 运行时，对所有在排队的请求计算"累积接受概率"
3. **贪心地从高到低**把候选 token 加入这一轮的 verification batch
4. 每加一个 token，**算一次期望吞吐量**：吞吐 = 接受概率 × SPS(当前 batch)
5. 吞吐量开始下降的瞬间，**立即停止**

────────────────────

💡 打个比方：

你（大哥）一次能批 100 份作业（batch capacity 100）。所有助教（小弟）抱了一堆作业过来——有的助教抱的是 "我有 95% 把握全对" 的高质量作业，有的是 "可能 30% 对" 的低质量作业。

你的策略：先收高质量的（接受概率高，不浪费你的批改名额）。每收一份算一下"这一轮总的产出"——如果还在涨就继续收，开始降了就停手。

DSpark 的调度算法干的就是这件事，而且**论文证明了这个算法是数学上无损的**——既不会牺牲生成质量，又能动态压榨硬件。

────────────────────

【部署时的一个工程妙招】

理论上的调度算法假设系统状态是"实时"的——每一步都看当前 batch 大小决定要不要继续。但 GPU 推理引擎通常和 CUDA Graph 配合，要求**步骤可预测**，不能动态变。

DSpark 的解法：**用两步前的置信度估计代替当前估计**——做"异步调度"。这样就和 CUDA Graph 兼容了，代价只是估计稍微滞后两步，对最终性能影响很小。

这是工程上的小聪明，但是 DSpark 在生产环境真正能跑起来的关键。

◆ 第五节：性能数字——别被 661% 忽悠

────────────────────

媒体报道的数字：DSpark 让 V4-Flash 提速 60%-85%，V4-Pro 提速 57%-78%。

**这两个数字是可以引用的**——论文原话叫 "matched practical throughput levels"，意思是在真实生产负载下的加速倍数。

但论文里**还有两个更夸张的数字**：V4-Flash 661%、V4-Pro 406%——一些自媒体在大肆传播。

**这两个数字不能直接引用**。论文自己有一段非常诚实的声明：**这两个数字是 baseline（MTP-1）已经接近系统操作边界、变得不可用时的算术放大**。换个说法：当原本的系统已经撑不住要崩了，DSpark 还能撑——比例是几百倍，但这是"撑住 vs 撑不住"的对比，不是"快 vs 更快"的对比。

论文原文叫这种数字 "qualitative regime"，**不应被解读为代表性的加速倍数**。

────────────────────

💡 打个比方：

你的笔记本电脑放在 40 度高温的房间里，因为过热而降频跑到了 10% 性能。然后你打开了空调，温度降到 25 度，性能回到 100%。

你能宣称"开空调让笔记本加速 10 倍"吗？技术上是的，性能数字 10×。但这不是"空调有 10 倍加速"，是"原来根本跑不动"。

DSpark 在那两个高 SLA 锚点的 661% / 406% 是同一回事。诚实的数字是 60%-85% / 57%-78%。

────────────────────

【适用场景】

DSpark 在不同任务上的表现差很多：

- **结构化任务**（math、code）：原生接受率就高，DSpark 锦上添花
- **开放对话**（chat）：原生接受率低（45.7%），DSpark 把它救到 95.7% ——这是 DSpark 真正秀肌肉的场景

意思是说，**DSpark 不是均匀地让所有场景都快**，而是把"原本投机解码玩不转的开放对话"救回来。

【已知限制】

论文很坦诚地列了一条 limitation：对于**接受率天生就极低**的复杂 query（比如需要长链条推理的难题），小弟猜得再聪明也猜不中，那 γ 个 candidate 的草稿成本是沉没的。论文说"未来要加难度感知早退"。

◆ 第六节：跟我们的关系

────────────────────

我现在维护两个 V4 相关的开源项目：

1. **deepseek-v4-flash-deployment**：vLLM + Marlin/CDG 混合后端，跑在两台 DGX Spark 上的 V4-Flash 280B 推理服务（之前 185 期讲过的那个）
2. **deepseek-v4-experimental-platform-on-dgx-spark**：自己写的 PyTorch 实验平台，用来抓激活值做研究

DSpark 在这两个项目上的可行性，结论完全不一样：

【对实验平台：用不上】

实验平台主要跑**prefill**——加载一段 17832 token 的长文本、抓中间层激活值。投机解码加速的是 **decode**——一个 token 一个 token 地生成。**两件事完全不重叠**。

DSpark 给实验平台带不来加速。

【对 vLLM 部署项目：理论上有意义，但要等】

vLLM 部署项目是产品级 serving，docker hub 上的镜像已经有 6.6k 下载。这种场景下 DSpark 的 60%-85% 加速是真有价值。

**但接入 DSpark 不是加几个 head 那么简单**。按论文 §5.2-5.3，需要 vLLM 实现：

1. 半自回归 speculative decoding 路径
2. 变长 query 的 attention kernel（V4 自家的 index-attention 风格）
3. 异步两步前置信度调度器
4. CUDA Graph 兼容性处理

这是 vLLM 上游级别的工作。看历史规律：DeepSeek 模型从发布到 vLLM 完整支持，通常 6 周到 2 个月。DSpark 进 vLLM 上游的合理预期是 **1.5-3 个月**。

而且更微妙的是——**sm_121（DGX Spark 用的消费级 Blackwell）上能不能跑还要看**。vLLM 上游优先支持的肯定是 H100/H200/B200 这类数据中心硬件，消费级的 sm_121 一直是次等公民。

所以现实做法是：**等 vLLM 上游 merge → 测能不能在 sm_121 上跑 → build 新版镜像 → 写一份"在 DGX Spark 上跑 V4-Flash-DSpark"的硬实测**。

但这是回家以后的事。我现在还在三河市，DGX Spark 在家里。

◆ 总结

────────────────────

DSpark 是 DeepSeek 在投机解码这条路上的一次干净的工程升级。核心两招：

**半自回归**——把全并行（DFlash）和全串行（Eagle）缝在一起。深的并行 backbone 提供接受率上限，薄的低秩串行 head 提供 inter-token 依赖。**2 层 DSpark 干赢 5 层 DFlash** 是这个设计最反直觉的证据

**置信度调度**——给每个候选 token 配一个"通过验证的概率"信号，加上硬件感知的贪心调度算法，动态决定每一轮塞多少个 candidate。开放对话场景接受率从 45.7% 救到 95.7%

数字层面诚实读数是 60%-85% / 57%-78% 的加速。媒体在炒的 661% / 406% 论文自己说不能这么解读。

工程层面这是给 vLLM/SGLang 等推理引擎的一个新任务——加速到位需要改 attention kernel + 调度器，不是装个 head 就能跑。

我会持续关注 vLLM 上游接 DSpark 的进展。下次回家有时间了，再写一篇"DGX Spark 上跑 V4-Flash-DSpark"的实测帖。

━━━━━━━━━━━━━━━━━━━━

「投机解码不是新东西，是把'搬一次数据吐一个 token'变成'搬一次数据吐多个 token'。瓶颈在搬运，不在算。」

「DSpark 的本质：用一个小弟猜 γ 个 token，让大哥一次性验证。猜得越准、γ 越大，加速越多。」

「半自回归是 Eagle 和 DFlash 的折中：深 backbone + 薄 head。2 层 DSpark 干赢 5 层 DFlash，参数效率证据。」

「论文里写得清清楚楚的 661% 是 baseline 接近崩溃时的算术放大，不该当 speedup 引用。诚实数字是 60-85%。」
