【AI架构原理】CODA——把 Transformer 的碎活儿全塞进矩阵乘法的"下班前五分钟"

━━━━━━━━━━━━━━━━━━━━

◆ 起因

━━━━━━━━━━━━━━━━━━━━

5 月 22 日，FlashAttention 的核心作者 Tri Dao 在社交媒体转发了一篇论文，配文写道：

"经过一些数学重写，结果发现 Transformer 的所有内容都是一系列 GEMM + epilogue（矩阵乘法加尾声）。给定一些优化的原语，LLM（以及新手）就可以为所有 Transformer 操作编写光速内核！"

论文叫 CODA: Rewriting Transformer Blocks as GEMM-Epilogue Programs，作者来自 MIT、Princeton、Together AI、Meta。第一反应是标题很学术，但 Tri Dao 的翻译非常到位——"矩阵乘法加尾声"，这六个字就是整篇论文的核心。

为什么值得写？因为这篇论文解决的问题，恰好是我们之前在 DGX Spark 上做 Triton kernel 融合时踩过的坑。当时朱雀帮我做的那些手工 epilogue 融合，和 CODA 的思路完全一致——只是我们是"一个个手搓"，CODA 是"系统化地搓"。

这篇文章是给自己补课用的。从最基础的概念讲起。

━━━━━━━━━━━━━━━━━━━━

◆ GPU 的核心矛盾：算得太快，搬得太慢

━━━━━━━━━━━━━━━━━━━━

先说一个直觉。

你的 GPU 有两种能力：**算数**和**搬数据**。

算数能力叫 compute（计算吞吐量），单位是 TFLOPS——每秒能做多少万亿次浮点运算。H100 的 BF16 计算能力大约是 990 TFLOPS。

搬数据能力叫 bandwidth（带宽），单位是 TB/s——每秒能从显存（HBM，High Bandwidth Memory）里读写多少数据。H100 的 HBM 带宽大约是 3.35 TB/s。

做个除法：990 TFLOPS ÷ 3.35 TB/s ≈ 295。这意味着 GPU 每从显存搬 1 字节数据，就能做大约 295 次浮点运算。

**如果你的操作需要搬 1 字节数据、做 300 次运算，那 GPU 刚好被喂饱——算完的时候数据也搬完了，没有浪费。**

**如果你的操作需要搬 1 字节数据、只做 1 次运算，那 GPU 在等数据的时间里有 99.7% 的算力在空转。**

前者叫 compute-bound（计算密集），后者叫 memory-bound（带宽密集）。

💡 翻译成人话：GPU 像一个算数极快的会计，但他的账本锁在一个开门很慢的保险柜里。如果每次开柜只拿一个数算一下就放回去，大部分时间花在开柜关柜上。如果每次开柜拿一大堆数据、算几百步再放回去，会计才能全力运转。

矩阵乘法（GEMM）就是那种"拿一大堆、算几百步"的操作——compute-bound，GPU 的最爱。

但 Transformer 里不只有矩阵乘法。

━━━━━━━━━━━━━━━━━━━━

◆ Transformer 里的"碎活儿"

━━━━━━━━━━━━━━━━━━━━

一个标准的 Transformer block 大致做这些事：

1. **RMSNorm**：对每一行向量算均方根，然后除以它（归一化）
2. **QKV Projection**：三个大矩阵乘法，算出 Q、K、V
3. **RoPE**：给 Q 和 K 加位置编码（旋转一下）
4. **Attention**：Q 和 K 算相似度、对 V 做加权求和
5. **Output Projection**：一个大矩阵乘法
6. **残差连接**：把输入加回来
7. **又一个 RMSNorm**
8. **Gate/Up Projection**：两个大矩阵乘法
9. **SwiGLU 激活**：一个非线性变换
10. **Down Projection**：一个大矩阵乘法
11. **又一个残差连接**

其中 2、4、5、8、10 是大矩阵乘法——compute-bound，GPU 跑得飞快。

但 1、3、6、7、9、11 呢？RMSNorm 要对每行算均方根（逐元素操作），RoPE 要对每对元素做三角函数（逐元素操作），SwiGLU 要做激活函数和逐元素乘法，残差连接就是两个矩阵逐元素相加。

**这些操作的共同特点：算数很少，但每个都得从 HBM 读一遍输入、写一遍输出。**

一个大矩阵 4096 × 4096、BF16 精度，占 32 MB。从 HBM 读一遍 32 MB、做个逐元素操作、写回 32 MB，总共 64 MB 的数据搬运，但计算量只有几千万次浮点运算——远远不到"搬 1 字节算 295 次"的平衡点。GPU 大部分时间在等数据搬完。

更要命的是，这些碎活儿不是一个，而是**一串**。做完矩阵乘法 → 写回 HBM → 读出来做 RMSNorm → 写回 HBM → 读出来做残差加 → 写回 HBM → 读出来做下一个矩阵乘法。**每次"写回再读出"都是一次不必要的 HBM 往返。**

💡 翻译成人话：会计算完一笔大账（矩阵乘法），把结果放回保险柜。然后开柜拿出来加个 1（残差），放回去。开柜拿出来除个数（RMSNorm），放回去。开柜拿出来转个角度（RoPE），放回去……每次开关柜门的时间比算数本身长十倍。

**如果能让会计算完大账之后，结果还拿在手上的时候，顺便把加 1、除数、转角度全做了再放回保险柜，那些开关柜门的时间就全省了。**

这就是 CODA 的核心思路。

━━━━━━━━━━━━━━━━━━━━

◆ GEMM 的三段式：prologue → main loop → epilogue

━━━━━━━━━━━━━━━━━━━━

要理解 CODA，先要理解一个 GEMM kernel 内部是怎么跑的。

GEMM 就是矩阵乘法 C = A × B。GPU 不会一次性算完整个矩阵，而是把输出矩阵 C 切成很多小块（tile），每个 tile 由一组线程（thread block）负责。

一个 GEMM kernel 的执行分三步：

**第一步：Prologue（序幕）**
把这个 tile 需要的 A 和 B 的数据块从 HBM 搬到片上存储（shared memory 或寄存器）。

**第二步：Main loop（主循环）**
在片上做矩阵乘法的核心计算。A 和 B 很大，不能一次装完，所以要分成很多小块，循环地搬一块、乘一块、累加到结果里。这是 GEMM 的主体，对应 GPU Tensor Core 的密集计算。

**第三步：Epilogue（尾声）**
主循环结束后，这个 tile 的计算结果还在寄存器里——还没写回 HBM。epilogue 就是在"结果还在手上"的这个时间窗口里，对结果做一些额外操作，然后才写回 HBM。

最简单的 epilogue 就是"啥也不做，直接写回"。稍微复杂一点可以做个 bias 加法（C = A × B + bias），或者做个激活函数（C = ReLU(A × B)）。

**关键洞察：epilogue 阶段数据在寄存器里，不需要额外的 HBM 读写。任何能在 epilogue 里做的操作，都是"免费"的——不增加额外的带宽开销。**

💡 翻译成人话：会计从保险柜拿出账本（prologue），噼里啪啦算了一通（main loop），算完了结果还写在手边的白板上（寄存器里）。这时候如果你说"顺便把这个数乘以 0.5 再加个 3"，他低头就能写，不用再开保险柜。这个"结果还在白板上、会计还没站起来"的阶段，就是 epilogue。

问题是：哪些操作能塞进 epilogue？

如果一个操作的输入就是 GEMM 的输出（逐元素或逐行的变换），那天然就能塞进去。残差加法、激活函数、RoPE——它们的输入都是前一个 GEMM 的输出，逐元素做就行。

但 RMSNorm 看起来不行。因为 RMSNorm 是在 GEMM **之前**做的——先 norm，再乘矩阵。你不能在矩阵乘法还没开始的时候做 epilogue。

除非……你能用数学把顺序倒过来。

━━━━━━━━━━━━━━━━━━━━

◆ 核心数学技巧：RMSNorm 的顺序交换

━━━━━━━━━━━━━━━━━━━━

这是 CODA 最精彩的部分。

RMSNorm 做的事：给一个向量 x，算它的均方根 `rms(x) = √(mean(x²) + ε)`，然后把 x 除以这个均方根，再乘一个可学习的缩放因子 γ。

标准 Transformer 里有这样一个计算链：

```
输入 → 上一个 GEMM 的输出 h₀ + 残差 z → RMSNorm → 下一个 GEMM
```

写成公式（省略偏置等细节）：

```
h₁ = h₀ + z                        （残差连接）
r = 1 / rms(h₁)                    （算缩放因子）
h₂ = r · h₁ ⊙ γ                    （归一化 + 缩放）
y = h₂ · W₁                        （下一个矩阵乘法）
```

看起来必须先算完 r 和 h₂，才能做矩阵乘法 y = h₂ · W₁。因为 h₂ 依赖 r，r 依赖 h₁ 的全行统计量。

但 CODA 发现了一个代数性质：**r 是一个行级标量**。同一行的每个元素被乘以相同的 r。

而矩阵乘法的本质是"每一行和权重矩阵的每一列做内积"。如果一行的每个元素都乘了同一个 r，那等价于先做内积、再把结果乘以 r：

```
(r · x) · W = r · (x · W)
```

**这就是交换律（commutativity）。行级标量乘法和矩阵乘法可以交换顺序。**

利用这一点，计算链可以重新排列：

```
h₁ = h₀ + z                        （残差连接——上一个 GEMM 的 epilogue）
h₂ = h₁ ⊙ γ                        （逐元素缩放——上一个 GEMM 的 epilogue）
r̂ = partialRMS(h₁)                 （部分均方根统计——上一个 GEMM 的 epilogue）
── 小辅助 kernel：把各 tile 的 r̂ 合并成完整的 r ──
y_raw = h₂ · W₁                    （矩阵乘法——下一个 GEMM 的 main loop）
y = r · y_raw                       （缩放——下一个 GEMM 的 epilogue）
```

对比一下前后变化：

**之前**：GEMM₁ → 写回 HBM → 读出来做残差 → 写回 → 读出来做 RMSNorm → 写回 → 读出来做 GEMM₂

**之后**：GEMM₁（epilogue 里顺便做残差 + 缩放 + 部分统计）→ 写回 → 小 kernel 合并统计量 → GEMM₂（epilogue 里顺便乘 r）

**中间那些 HBM 往返全没了。**

唯一的代价是一个很小的辅助 kernel：把各个 tile 算出的部分均方根统计量合并成一行的完整统计量。这个 kernel 处理的数据量很小（只有行数那么多个标量），可以忽略不计。

💡 翻译成人话：本来的流程是"拿出来 → 加盐 → 放回去 → 拿出来 → 称重 → 放回去 → 拿出来 → 炒菜"。现在变成"拿出来 → 加盐（顺手称个重）→ 放回去 → 拿出来 → 炒菜（出锅前按重量调味）"。来回搬了更少的次数。

**关键：这个交换律，编译器推不出来。** 编译器只知道"这两个 kernel 之间有数据依赖"，不知道"这个依赖是一个行级标量、它和矩阵乘法可以交换"。得人类用代数性质手动证明等价性，再告诉系统"这样重排是安全的"。

◆ 其他碎活儿怎么塞进 epilogue

━━━━━━━━━━━━━━━━━━━━

RMSNorm 是最复杂的，因为需要交换律。其他操作更直接：

────────────────────

### RoPE（旋转位置编码）

输入：QKV projection GEMM 的输出。
操作：对每对相邻元素做三角函数旋转。
为什么能做 epilogue：GEMM 算完 Q/K/V 后结果还在寄存器里，对相邻元素做旋转是逐对操作，直接做就行。CODA 特意把相邻特征排列成匹配 Hopper Tensor Core 的累加器布局——每个线程恰好持有一对相邻值，在寄存器级别就能完成旋转，不需要任何额外的数据搬运。

────────────────────

### SwiGLU（门控激活函数）

输入：Gate projection 和 Up projection 两个 GEMM 的输出。
操作：对 gate 部分做 SiLU 激活函数，然后和 up 部分逐元素相乘。
为什么能做 epilogue：把 gate 和 up 两个投影合并成一个大 GEMM（拼接权重矩阵），输出的相邻特征对恰好对应 gate 和 up 的配对值。epilogue 里直接做 SiLU + 逐元素乘。

────────────────────

### 残差连接

输入：前一个子层的输入（需要从 HBM 读入 epilogue）。
操作：逐元素相加。
为什么能做 epilogue：GEMM 输出还在寄存器里，从 HBM 加载残差流数据到 epilogue，逐元素加完再写回。虽然仍需读一次残差数据，但省掉了单独的残差 kernel 和中间结果的额外写入。

────────────────────

### 交叉熵损失

输入：语言模型 head 的 GEMM 输出（logits）。
操作：在线计算 log-sum-exp 统计量。
为什么能做 epilogue：用有状态变换（stateful transform），在 epilogue 里维护运行中的 max 和 sum-exp，避免把整个 logits 矩阵写出再读回。

━━━━━━━━━━━━━━━━━━━━

◆ 五类可组合的 epilogue 原语

━━━━━━━━━━━━━━━━━━━━

CODA 不是每种操作写一个专用 kernel。它定义了**五类可组合的 epilogue 原语**，像乐高积木一样拼出各种融合 kernel：

**1. 逐元素/成对变换（Elementwise & Pairwise Maps）**
对累加器里的值做局部变换。包括残差加法、激活函数、RoPE 式旋转、SwiGLU 式门控。

**2. 向量加载/存储（Vector Loads & Stores）**
加载行向量或列向量，广播到整个 tile 上。比如 bias 加法（同一个 bias 向量加到每一行），或者保存行级统计量。

**3. Tile 加载/存储（Tile Loads & Stores）**
加载或存储矩阵 tile，用 TMA（Tensor Memory Accelerator）传输。用来读残差流、保存反向传播需要的激活值。

**4. Tile 规约（Tile Reductions）**
对 tile 内的行或列做部分求和/求最大值等规约操作。比如 RMSNorm 的部分均方根统计——每个 tile 算自己那部分的平方和，之后一个小辅助 kernel 把各 tile 的部分和合并成整行的统计量。

**5. 有状态变换（Stateful Transforms）**
在 epilogue 内维护运行状态。比如在线 log-sum-exp——边算边更新 max 和 sum-exp，处理完整行后直接得到 softmax 的分母，不需要多趟遍历。交叉熵损失也用这种机制。

**GEMM 的 main loop 完全不动。** 你只需要定义"GEMM 算完之后做什么"——选几块乐高拼起来就行。底层基于 NVIDIA 的 CUTLASS 库和 CuTeDSL（一个 Python 级别的 kernel 编程接口），GEMM 主循环的排布、流水线、Tensor Core 调度全由库搞定。

◆ 整体效果：一个 Transformer block 变成一串 GEMM

━━━━━━━━━━━━━━━━━━━━

把上面的融合全做了之后，一个标准 Transformer block 的前向传播变成这样：

**注意力子层：**
- GEMM₁（QKV Projection） + epilogue（RoPE 旋转）
- Attention（FlashAttention 负责，CODA 不管）
- GEMM₂（Output Projection） + epilogue（残差加 + RMSNorm 的部分统计 + 逐元素缩放）
- 辅助 kernel（合并 RMSNorm 统计量）

**MLP 子层：**
- GEMM₃（Gate+Up Projection，拼成一个大矩阵）+ epilogue（SwiGLU 激活 + RMSNorm 缩放 r）
- GEMM₄（Down Projection）+ epilogue（残差加 + 下一层 RMSNorm 的部分统计 + 逐元素缩放）
- 辅助 kernel（合并 RMSNorm 统计量）

**中间没有独立的 RMSNorm kernel，没有独立的 RoPE kernel，没有独立的 SwiGLU kernel，没有独立的残差 kernel。** 所有碎活儿都被吸收进了相邻 GEMM 的 epilogue。

注意一个跨模块边界的设计：**GEMM-Residual-RMSNorm-GEMM 模式**。这个模式出现在每两个相邻子层之间——上一个子层的输出投影 GEMM 的 epilogue 做残差和 RMSNorm 的前半段，下一个子层的输入投影 GEMM 的 epilogue 做 RMSNorm 的后半段（乘 r）。这打破了传统框架里 Attention 模块和 MLP 模块的边界——优化需要跨模块思考。

◆ 性能与精度

━━━━━━━━━━━━━━━━━━━━

论文在单张 H100 上做了系统测试，覆盖 LLaMA 系列的 1B、7B、70B 规模配置（对应隐藏维度 d = 2048、4096、8192），batch size 16K tokens。

基线对比包括 cuBLAS + torch.compile、Liger Kernels、FlashInfer、QuACK 等。

**关键结果：**

论文侧重 kernel 级和 block 级的加速比数据，没有报端到端训练吞吐的单一数字。几个有代表性的数据点：

- **GEMM-RMSNorm-GEMM 融合 kernel**：比 cuBLAS + torch.compile 的独立 kernel 组合有显著加速，核心原因就是省掉了中间的 HBM 往返。
- **交叉熵 epilogue**：在 d=8192（70B 级别）配置下，CODA 的融合交叉熵 kernel 比 Liger Kernels 快约 1.6 倍——传统做法要把完整 logits 矩阵写出再读回，CODA 在 epilogue 里用有状态变换在线算完。
- **完整 Transformer block 级别**（前向 + 反向，不含注意力），在 1B/7B/70B 配置上稳定提速。
- **精度**：BF16 下与 FP32 参考值对比，CODA 的重参数化路径的相对误差**低于标准 PyTorch 路径**。不仅更快，还更准——因为 CODA 的 GEMM 主循环用 FP32 累加器做中间计算，而中间写回 HBM 时的 BF16 截断误差被避免了。

CODA 覆盖的是**注意力之外的几乎所有计算**（前向和反向）。注意力本身由 FlashAttention 负责——两者互补，不冲突。FA 管注意力内部的融合（Q·K 相似度 → softmax → 和 V 的加权求和），CODA 管注意力之外的一切。

◆ 让 LLM 写 kernel

━━━━━━━━━━━━━━━━━━━━

论文里有一个有趣的工程细节：CODA 用 **Claude Code** 来生成大部分 kernel。

这听起来像吹牛，但逻辑是成立的：因为 CODA 的编程模型已经把问题简化为"定义 epilogue"。GEMM 的主循环、数据搬运、流水线调度——这些最难写也最容易出错的部分——全由 CUTLASS 库搞定，不需要 LLM 操心。LLM 只需要回答"GEMM 算完之后做什么"，本质上是写一段逐元素变换逻辑。

具体做法：给 Claude Code 一份书面规格说明 + 精选的示例代码 + 一份实现技巧日志，加上轻量的人类监督。仓库本身就是一个不断增长的示例集——写得越多，LLM 能参考的模式越多，新 kernel 写得越快。

**这不是"AI 自动写出 CUDA 神 kernel"的故事。是"好的抽象让问题简单到 LLM 也能做"的故事。** 抽象做对了，门槛降下来的不只是 LLM，新手工程师也一样受益。

━━━━━━━━━━━━━━━━━━━━

◆ 和我们 DGX Spark 经验的呼应

━━━━━━━━━━━━━━━━━━━━

之前在 DGX Spark（sm_121）上做 DeepSeek V4 Flash 推理时，朱雀帮我在 Triton 里手工做过类似的事。最典型的例子：INT8 量化模型的推理——`tl.dot(int8, int8)` 算完矩阵乘法后，结果还在寄存器里，直接乘 scale 因子转成 BF16 再写回。如果把 dequant 做成独立 kernel，就得多一次 HBM 写读。FP4 的 `tl.dot_scaled` 也一样——矩阵乘法和 block scale 在同一个 kernel 里完成。

跟 CODA 一模一样的思路。但我们是一个一个 case 手搓——看到哪个操作能塞进去就手动塞，没有系统化的原语体系。CODA 做的事情是把这种手搓经验提炼成五类可组合原语，从此不用每次从头想"这个操作能不能融合进 epilogue"，而是从积木箱里挑几块拼起来。

从"case by case 手工融合"到"定义原语、组合使用"，这是工程成熟度的一次跃迁。不过 CODA 依赖 CUTLASS 3.x（需要 sm_90+ 的 Hopper 或 sm_100+ 的 Blackwell 数据中心卡），DGX Spark 的 sm_121 没有 CUTLASS 3.x 需要的 wgmma/tcgen05 指令，所以 CODA 的代码在我们机器上跑不了——但思路完全通用。

技术名词速查：

- **GEMM**（General Matrix Multiply）：通用矩阵乘法。GPU 上最核心的计算原语。
- **HBM**（High Bandwidth Memory）：高带宽显存。GPU 的主存储器，容量大但读写速度相对计算速度而言是瓶颈。
- **Epilogue**（尾声）：GEMM kernel 执行完矩阵乘法后、结果写回 HBM 前的一个时间窗口。数据还在寄存器/片上缓存里，可以"免费"做额外计算。
- **Memory-bound**（带宽密集）：瓶颈在数据搬运而非计算的操作。Norm、激活函数、残差加法都是典型 memory-bound 操作。
- **Compute-bound**（计算密集）：瓶颈在计算而非数据搬运的操作。大矩阵乘法是典型 compute-bound 操作。
- **RMSNorm**（Root Mean Square Normalization）：一种归一化方法，对向量除以其均方根值。
- **RoPE**（Rotary Position Embedding）：旋转位置编码，对 Q/K 的特征对做二维旋转来编码位置信息。
- **SwiGLU**：一种门控激活函数，SiLU(gate) ⊙ up，用在 Transformer 的 MLP 层。
- **Tile**：GPU 计算时把大矩阵切成的小块，每个 tile 由一组线程负责。
- **CUTLASS / CuTeDSL**：NVIDIA 的 CUDA 模板库和 Python 级 kernel 编程接口。
