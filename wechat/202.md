【AI架构原理】CODA——把 Transformer 的碎活儿全塞进矩阵乘法的"下班前五分钟"

◆ 起因

━━━━━━━━━━━━━━━━━━━━

5 月 22 日，FlashAttention 的核心作者 Tri Dao 在社交媒体转发了一篇论文，配文写道：

"经过一些数学重写，结果发现 Transformer 的所有内容都是一系列 GEMM + epilogue（矩阵乘法加尾声）。给定一些优化的原语，LLM（以及新手）就可以为所有 Transformer 操作编写光速内核！"

论文叫 CODA: Rewriting Transformer Blocks as GEMM-Epilogue Programs，作者来自 MIT、Princeton、Together AI、Meta。第一反应是标题很学术，但 Tri Dao 的翻译非常到位——"矩阵乘法加尾声"，这六个字就是整篇论文的核心。

为什么值得写？因为这篇论文解决的问题，恰好是我们之前在 DGX Spark 上做 Triton kernel 融合时踩过的坑（见《【Deepseek V4推理】sm_12x问题修复，从4到12 tok/s》： https://mp.weixin.qq.com/s/dEc-wD0JcM5WhKicOUR1pQ ）。当时我们之前做的那些手工 epilogue 融合，和 CODA 的思路完全一致——只是我们是"一个个手搓"，CODA 是"系统化地搓"。

这篇文章是给自己补课用的。我手头只有两台 DGX Spark（sm_121，家用级 Blackwell），没摸过 H100 也没见过 B200。但 CODA 背后的思路——"少搬数据，多顺手干活"——跟硬件型号无关。从最基础的概念讲起。

━━━━━━━━━━━━━━━━━━━━

◆ GPU 的核心矛盾：算得太快，搬得太慢

━━━━━━━━━━━━━━━━━━━━

先说一个直觉。

你的 GPU 有两种能力：**算数**和**搬数据**。

算数能力叫 compute（计算吞吐量），单位是 TFLOPS——每秒能做多少万亿次浮点运算。H100 的 BF16 计算能力大约是 990 TFLOPS。

搬数据能力叫 bandwidth（带宽），单位是 TB/s——每秒能从显存（HBM，High Bandwidth Memory）里读写多少数据。H100 的 HBM 带宽大约是 3.35 TB/s。

做个除法：990 TFLOPS ÷ 3.35 TB/s ≈ 295。这意味着 GPU 每从显存搬 1 字节数据，就能做大约 295 次浮点运算。

**如果你的操作需要搬 1 字节数据、做 300 次运算，那 GPU 刚好被喂饱——算完的时候数据也搬完了，没有浪费。**

**如果你的操作需要搬 1 字节数据、只做 1 次运算，那 GPU 在等数据的时间里有 99.7% 的算力在空转。**

前者叫 compute-bound（计算密集），后者叫 memory-bound（带宽密集）。

一个活生生的例子：Mac Studio 的 GPU 算力不如 DGX Spark（GB10 大约 RTX 5070 水平）和 AMD AI Max+ 395（大约 RTX 4060 水平），但跑 LLM 推理时 Mac 反而更快——因为 LLM 逐 token 生成是 memory-bound 的，瓶颈在带宽不在算力，而苹果统一内存的带宽够猛。**算力再强，数据喂不进去也是白搭。**

💡 翻译成人话：GPU 像一个算数极快的会计，但他的账本锁在一个开门很慢的保险柜里。如果每次开柜只拿一个数算一下就放回去，大部分时间花在开柜关柜上。如果每次开柜拿一大堆数据、算几百步再放回去，会计才能全力运转。

矩阵乘法（GEMM）就是那种"拿一大堆、算几百步"的操作——compute-bound，GPU 的最爱。

但 Transformer 里不只有矩阵乘法。

━━━━━━━━━━━━━━━━━━━━

◆ Transformer 里的"碎活儿"

━━━━━━━━━━━━━━━━━━━━

一个标准的 Transformer block 大致做这些事：

1. **RMSNorm：对每一行向量算均方根，然后除以它（归一化）**
2. QKV Projection：三个大矩阵乘法，算出 Q、K、V
3. **RoPE：给 Q 和 K 加位置编码（旋转一下）**
4. Attention：Q 和 K 算相似度、对 V 做加权求和
5. Output Projection：一个大矩阵乘法
6. **残差连接：把输入加回来**
7. **又一个 RMSNorm**
8. Gate/Up Projection：两个大矩阵乘法
9. **SwiGLU 激活：一个非线性变换**
10. Down Projection：一个大矩阵乘法
11. **又一个残差连接**

其中 2、4、5、8、10 是大矩阵乘法——compute-bound，GPU 跑得飞快，走的是 Tensor Core（专门做矩阵乘加的硬件单元）。

但 1、3、6、7、9、11 呢？这些走的是 CUDA Core——GPU 上最基础的通用计算单元。用游戏卡打比方：CUDA Core 从第一代 GeForce 8800（2006 年）就有了，Tensor Core 是 RTX 20 系（2018 年）才加的，就是跑 DLSS 的那个专用硬件。CUDA Core 什么都能算，但没有 Tensor Core 对矩阵乘法的那种暴力加速。RMSNorm 要对每行算均方根（逐元素操作），RoPE 要读取预计算好的 cos/sin 表、对每对元素做二维旋转，SwiGLU 要做激活函数和逐元素乘法，残差连接就是两个矩阵逐元素相加。

**这些操作的共同特点：算数很少，但每个都得从 HBM 读一遍输入、写一遍输出。**

一个大矩阵 4096 × 4096、BF16 精度，占 32 MB。从 HBM 读一遍 32 MB、做个逐元素操作、写回 32 MB，总共 64 MB 的数据搬运，但计算量只有几千万次浮点运算——远远不到"搬 1 字节算 295 次"的平衡点。GPU 大部分时间在等数据搬完。

更要命的是，这些碎活儿不是一个，而是**一串**。做完矩阵乘法 → 写回 HBM → 读出来做 RMSNorm → 写回 HBM → 读出来做残差加 → 写回 HBM → 读出来做下一个矩阵乘法。**每次"写回再读出"都是一次不必要的 HBM 往返。**

💡 翻译成人话：会计算完一笔大账（矩阵乘法），把结果放回保险柜。然后开柜拿出来加个 1（残差），放回去。开柜拿出来除个数（RMSNorm），放回去。开柜拿出来转个角度（RoPE），放回去……每次开关柜门的时间比算数本身长十倍。

**如果能让会计算完大账之后，结果还拿在手上的时候，顺便把加 1、除数、转角度全做了再放回保险柜，那些开关柜门的时间就全省了。**

这就是 CODA 的核心思路。

◆ GEMM 的三段式：prologue → main loop → epilogue

━━━━━━━━━━━━━━━━━━━━

要理解 CODA，先要理解一个 GEMM kernel 内部是怎么跑的。

GEMM 就是矩阵乘法 C = A × B。GPU 不会一次性算完整个矩阵，而是把输出矩阵 C 切成很多小块（tile），每个 tile 由一组线程（thread block）负责。

一个 GEMM kernel 的执行分三步：

**第一步：Prologue（序幕）**
把这个 tile 需要的 A 和 B 的数据块从 HBM 搬到片上存储（shared memory 或寄存器）。

**第二步：Main loop（主循环）**
在片上做矩阵乘法的核心计算。A 和 B 很大，不能一次装完，所以要分成很多小块，循环地搬一块、乘一块、累加到结果里。这是 GEMM 的主体，对应 GPU Tensor Core 的密集计算。

**第三步：Epilogue（尾声）**
主循环结束后，这个 tile 的计算结果还在寄存器里——还没写回 HBM。epilogue 就是在"结果还在手上"的这个时间窗口里，对结果做一些额外操作，然后才写回 HBM。

最简单的 epilogue 就是"啥也不做，直接写回"。稍微复杂一点可以做个 bias 加法（C = A × B + bias），或者做个激活函数（C = ReLU(A × B)）。

**关键洞察：epilogue 阶段数据在寄存器里，不需要额外的 HBM 往返。任何能在 epilogue 里做的操作，带宽上都接近"免费"——不需要为了这一步再把完整中间张量写回显存、再读出来。**

💡 翻译成人话：会计从保险柜拿出账本（prologue），噼里啪啦算了一通（main loop），算完了结果还写在手边的白板上（寄存器里）。这时候如果你说"顺便把这个数乘以 0.5 再加个 3"，他低头就能写，不用再开保险柜。这个"结果还在白板上、会计还没站起来"的阶段，就是 epilogue。

问题是：哪些操作能塞进 epilogue？

如果一个操作的输入就是 GEMM 的输出（逐元素或逐行的变换），那天然就能塞进去。残差加法、激活函数、RoPE——它们的输入都是前一个 GEMM 的输出，逐元素做就行。

但 RMSNorm 看起来不行。因为 RMSNorm 是在 GEMM **之前**做的——先 norm，再乘矩阵。你不能在矩阵乘法还没开始的时候做 epilogue。

除非……你能用数学把顺序倒过来。

在进入数学之前，先看一下 CODA 程序大概长什么样。真实代码是 CuTeDSL/CUTLASS，比下面复杂得多，但骨架就是这个：

```python
def gemm_with_epilogue(A, B, epilogue):
    # main loop：CUTLASS 负责，把 A @ B 算到寄存器 accumulator 里
    acc = gemm_mainloop(A, B)

    # epilogue：用户只写 GEMM 算完后，对 acc 顺手做什么
    out = epilogue(acc)

    # 最后才写回 HBM
    store(out)
```

比如普通矩阵乘法后面接一个 bias，可以写成：

```python
def bias_epilogue(acc):
    return acc + bias
```

CODA 做的事情，就是把 Transformer 里的碎活儿都写成这种 epilogue 小积木。RMSNorm 这一段对应两个积木。注意，下面这个函数不是对整个矩阵执行一次，而是每个输出 tile 都执行一次：

```python
# 第一个 GEMM 的 epilogue：每个输出 tile 执行一次
def residual_partial_rmsnorm_epilogue(acc_tile):
    h1_tile = acc_tile + residual_tile       # 残差
    u_tile = h1_tile * gamma_tile            # 先乘 RMSNorm 权重 gamma
    s_tile = partial_sum(h1_tile * h1_tile)  # 只算这个 tile 的平方和
    return h1_tile, u_tile, s_tile

# 小 kernel：把同一行所有 tile 的 partial_sum 合并成 r
r = reduce_to_rstd(s_tiles)

# 第二个 GEMM 的 epilogue
def rmsnorm_scale_epilogue(acc_tile):
    return acc_tile * r_tile                 # 每一行乘一个 r
```

后面那堆公式，讲的就是为什么最后这个 `acc * r` 可以放到第二个 GEMM 的 epilogue 里。

◆ 核心数学技巧：RMSNorm 的顺序交换

━━━━━━━━━━━━━━━━━━━━

这是 CODA 最精彩的部分。

RMSNorm 做的事：给一个向量 x，算它的均方根 `rms(x) = √(mean(x²) + ε)`（ε 是个极小常数如 1e-6，纯粹防除零用的），然后把 x 除以这个均方根，再乘一个可学习的缩放因子 γ。

标准 Transformer 里有这样一个计算链：

```
输入 → 上一个 GEMM 的输出 h₀ + 残差 z → RMSNorm → 下一个 GEMM
```

先把形状写清楚。假设一次处理 `M` 个 token，隐藏维度是 `d`，下一个矩阵乘法的输出维度是 `n`：

- `h₀`：`M × d`，上一个 GEMM 的输出
- `z`：`M × d`，残差流
- `h₁`：`M × d`，残差相加后的结果
- `r`：`M × 1`，每个 token 一个缩放因子
- `γ`：`1 × d`，RMSNorm 的可学习权重，会广播到每一行
- `W₁`：`d × n`，下一个 GEMM 的权重
- `y`：`M × n`，下一个 GEMM 的输出

写成公式（省略偏置等细节）：

这里 `·` 表示矩阵乘法，`⊙` 表示逐元素乘法，不是矩阵乘法。形状不一样时按广播规则处理：`r[M×1]` 会复制到每一列，变成 `M×d`；`γ[1×d]` 会复制到每一行，也变成 `M×d`。最后三个 `M×d` 矩阵逐元素相乘。尤其要注意：`r[M×1] ⊙ h₁[M×d]` 的意思是"每一行乘一个标量"，等价于左乘一个 `M×M` 的对角矩阵 `D_r`。比如第 1 行全乘 `r₁`，第 2 行全乘 `r₂`。后面能把 `r` 挪到 GEMM 后面，靠的就是这个性质。

```
h₁[M×d] = h₀[M×d] + z[M×d]                         （残差连接）
r[M×1] = 1 / rms(h₁[M×d])                           （每行算一个缩放因子）
h₂[M×d] = r[M×1] ⊙ h₁[M×d] ⊙ γ[1×d]                （广播后逐元素相乘）
y[M×n] = h₂[M×d] · W₁[d×n]                         （下一个矩阵乘法）
```

下面这段分析的是最后两行：`h₂ = r ⊙ h₁ ⊙ γ` 和 `y = h₂ · W₁`。

看起来必须先算完 r 和 h₂，才能做矩阵乘法 y = h₂ · W₁。因为 h₂ 依赖 r，r 又依赖 h₁ 的全行统计量。

但 CODA 发现了一个代数性质：**r 是一个列向量**（长度=行数），每个元素是对应行的 1/rms。作用方式是"每行所有元素乘同一个数"——对整个矩阵来说是列向量，对单行来说就是一个标量。

把 `h₁ ⊙ γ` 先合成一个中间量 `u[M×d]`，原式就变成：

```
u[M×d] = h₁[M×d] ⊙ γ[1×d]
y[M×n] = (r[M×1] ⊙ u[M×d]) · W₁[d×n]
```

因为 r 是"每行一个标量"，所以这一行可以改写成：

```
y[M×n] = r[M×1] ⊙ (u[M×d] · W₁[d×n])
```

也就是说：先按 r 缩放再乘矩阵，和先乘矩阵再按 r 缩放，数学上完全等价。稀奇的不是结合律本身，而是 **r 是行级标量**。如果 r 是一个 `M×d` 的普通矩阵，这一步就不能这么搬；正因为 r 每行只有一个数，搬到 GEMM 后面仍然只是"每行乘一个数"，可以塞进 epilogue 里做，省掉一轮完整的 HBM 往返。

那 `γ[1×d]` 呢？它不用挪。`γ` 是模型权重，早就知道，而且只是每一列乘一个固定系数，可以在上一个 GEMM 的 epilogue 里直接乘到 `h₁` 上，得到 `u = h₁ ⊙ γ`。真正麻烦的是 `r`：它必须先看完整行的平方和才能算出来，所以不能在一开始就用。CODA 折腾这一套，核心就是把"等 r 算出来以后再做的那次逐元素缩放"挪到下一个 GEMM 的 epilogue 里。

**结合律保证了数学上等价，行级标量保证了交换之后代价极小。两个条件缺一不可。**

利用这一点，计算链可以重新排列。但在展开之前，先解释一个问题：**为什么不能在一个 epilogue 里直接算出完整的 RMSNorm？**

RMSNorm 需要算一整行的均方根 `rms = √(mean(x²))`——你得把这一行的**所有元素**的平方加起来求平均，才能算出 rms。

但 GEMM 是把输出矩阵切成小块（tile）分给不同线程组算的。每个 tile 只负责输出矩阵的一小块区域——比如一行有 4096 列，切成 4 个 tile，每个 tile 只看到 1024 列。一个 tile 的 epilogue 只能算出它负责的那 1024 列的平方和，看不到其他 3 个 tile 的数据。

所以需要分两步：

**第一步**：每个 tile 在自己的 epilogue 里算出"部分平方和"——只把自己负责的那些列的 x² 加起来。这就是 partialRMS。

**第二步**：一个很小的辅助 kernel 把 4 个 tile 的部分平方和加起来，得到整行的完整平方和，再开方得到 rms，算出缩放因子 r = 1/rms。这个 kernel 处理的数据量极小——每行只有一个标量，比如 batch_size × 1 个浮点数——跑起来几乎不花时间。

有了这个背景，重排后的计算链是：

```
【Kernel 1：GEMM₁ + epilogue】
  main loop: 上一个矩阵乘法
  epilogue（结果还在寄存器里，顺手做三件事）:
    h₁ = h₀ + z                    ① 残差连接（逐元素加）
    h₂ = h₁ ⊙ γ                    ② 逐元素乘缩放因子 γ
    partial_sum = Σ(h₁²)           ③ 算这个 tile 负责的那些列的平方和
  写回 HBM
  （GPU 上几百个 tile 同时跑，每个 tile 各自算自己的 partial_sum，天然并发）

【Kernel 2：小辅助 kernel】（所有 tile 跑完之后）
  r = 1 / √(各 tile 的 partial_sum 之和 / 列数 + ε)
  （每行只算一个标量，极快）

【Kernel 3：GEMM₂ + epilogue】
  main loop: y_raw = h₂ · W₁      下一个矩阵乘法
  epilogue: y = r · y_raw          乘以缩放因子 r
```

传统做法需要 5-6 个独立 kernel（GEMM₁、残差加、RMSNorm、GEMM₂……），每个之间都要写回再读出。CODA 压到 3 个 kernel，中间只剩一个极小的辅助 kernel。

对比一下前后变化：

**之前**：GEMM₁ → 写回 HBM → 读出来做残差 → 写回 → 读出来做 RMSNorm → 写回 → 读出来做 GEMM₂。**三次 HBM 往返。**

**之后**：GEMM₁（epilogue 里顺便做残差 + 缩放 + 部分统计）→ 写回 → 小辅助 kernel → GEMM₂（epilogue 里顺便乘 r）。**一次 HBM 往返 + 一个极小的辅助 kernel。**

💡 翻译成人话：本来的流程是"拿出来 → 加盐 → 放回去 → 拿出来 → 称重 → 放回去 → 拿出来 → 炒菜"。现在变成"拿出来 → 加盐，顺手称一下自己这份的重量 → 放回去 → 把几份的重量加起来算个总重（几个数相加，秒完）→ 拿出来 → 炒菜，出锅前按总重调味"。

**关键：这个结合律，编译器推不出来。** 编译器只知道"这两个 kernel 之间有数据依赖"，不知道"这个依赖是一个行级标量、它和矩阵乘法满足结合律"。得人类用代数性质手动证明等价性，再告诉系统"这样重排是安全的"。

◆ 其他碎活儿怎么塞进 epilogue

━━━━━━━━━━━━━━━━━━━━

RMSNorm 是最复杂的，因为需要结合律来重排计算顺序。其他操作塞进 epilogue 就没什么悬念了——结果还在寄存器里，顺手做就完了：

- **RoPE**：QKV projection 算完后，读取预计算好的 cos/sin 表，对每对相邻元素做二维旋转。逐对操作，直接做。
- **SwiGLU**：把 Gate 和 Up 两个投影拼成一个大 GEMM，epilogue 里直接做 SiLU 激活 + 逐元素乘。
- **残差连接**：GEMM 输出还在寄存器里，从 HBM 读入上一层的残差数据，逐元素加完再写回。虽然多读了一次残差，但省掉了单独的残差 kernel。
- **交叉熵损失**（训练专用，推理时不涉及）：这个稍微特殊——需要对整行做 log-sum-exp，但可以用"有状态变换"（边算边维护 max 和 sum-exp），避免把整个 logits 矩阵写出再读回。

────────────────────

### epilogue 的边界在哪？

一个自然的问题：什么操作能塞进 epilogue，什么不能？

界限不是计算量大小，是**"你需要的信息在不在当前 tile 的寄存器里"**。

- **tile 内部的操作**：逐元素加减乘除、逐对旋转、激活函数——数据已经在寄存器里了，不需要为这些操作增加完整的 HBM 往返。
- **需要整行信息但可以拆的操作**：比如 RMSNorm 需要整行的平方和，但可以拆成"每个 tile 算部分平方和 → 辅助 kernel 合并"。多了一个极小的辅助 kernel，但主体操作仍在 epilogue 里完成，不再单独搬运一整遍中间张量。
- **需要跨 tile 大规模交互的操作**：比如 Attention——Q 要和所有 token 的 K 算相似度，一个 tile 里只有几个 token，看不到全局。这不是"顺手"能搞定的，必须单独处理。

**所以在 CODA 这篇论文的边界里，真正不能做成 GEMM epilogue 的操作是 Attention**——Q·K 的相似度计算 + softmax + 和 V 的加权求和，需要所有 token 之间的全局交互。这部分由 FlashAttention 负责，CODA 不碰。两者互补：FA 管注意力内部，CODA 管注意力之外的一切。

━━━━━━━━━━━━━━━━━━━━

◆ 五类可组合的 epilogue 原语

━━━━━━━━━━━━━━━━━━━━

CODA 不是每种操作写一个专用 kernel。它定义了**五类可组合的 epilogue 原语（primitive，最基本的积木块）**，像乐高一样拼出各种融合 kernel：

**1. 逐元素/成对变换（Elementwise & Pairwise Maps）**
对累加器里的值做局部变换。包括残差加法、激活函数、RoPE 式旋转、SwiGLU 式门控。

**2. 向量加载/存储（Vector Loads & Stores）**
加载行向量或列向量，广播到整个 tile 上。比如 bias 加法（同一个 bias 向量加到每一行），或者保存行级统计量。

**3. Tile 加载/存储（Tile Loads & Stores）**
加载或存储矩阵 tile，用 TMA（Tensor Memory Accelerator）传输。用来读残差流、保存反向传播需要的激活值。

**4. Tile 规约（Tile Reductions）**
对 tile 内的行或列做部分求和/求最大值等规约操作。比如 RMSNorm 的部分均方根统计——每个 tile 算自己那部分的平方和，之后一个小辅助 kernel 把各 tile 的部分和合并成整行的统计量。

**5. 有状态变换（Stateful Transforms）**
在 epilogue 内维护运行状态。比如在线 log-sum-exp——边算边更新 max 和 sum-exp，处理完整行后直接得到 softmax 的分母，不需要多趟遍历。交叉熵损失也用这种机制。

**GEMM 的 main loop 完全不动。** 你只需要定义"GEMM 算完之后做什么"——选几块乐高拼起来就行。底层基于 NVIDIA 的 CUTLASS 库和 CuTeDSL（一个 Python 级别的 kernel 编程接口），GEMM 主循环的排布、流水线、Tensor Core 调度全由库搞定。

━━━━━━━━━━━━━━━━━━━━

◆ 整体效果：一个 Transformer block 变成一串 GEMM

━━━━━━━━━━━━━━━━━━━━

把上面的融合全做了之后，一个标准 Transformer block 的前向传播变成这样：

**注意力子层：**
- GEMM₁（QKV Projection） + epilogue（RoPE 旋转）
- Attention（FlashAttention 负责，CODA 不管）
- GEMM₂（Output Projection） + epilogue（残差加 + RMSNorm 的部分统计 + 逐元素缩放）
- 辅助 kernel（合并 RMSNorm 统计量）

**MLP 子层：**
- GEMM₃（Gate+Up Projection，拼成一个大矩阵）+ epilogue（SwiGLU 激活 + RMSNorm 缩放 r）
- GEMM₄（Down Projection）+ epilogue（残差加 + 下一层 RMSNorm 的部分统计 + 逐元素缩放）
- 辅助 kernel（合并 RMSNorm 统计量）

**中间没有独立的 RMSNorm kernel，没有独立的 RoPE kernel，没有独立的 SwiGLU kernel，没有独立的残差 kernel。** 所有碎活儿都被吸收进了相邻 GEMM 的 epilogue。

注意一个跨模块边界的设计：**GEMM-Residual-RMSNorm-GEMM 模式**。这个模式出现在每两个相邻子层之间——上一个子层的输出投影 GEMM 的 epilogue 做残差和 RMSNorm 的前半段，下一个子层的输入投影 GEMM 的 epilogue 做 RMSNorm 的后半段（乘 r）。这打破了传统框架里 Attention 模块和 MLP 模块的边界——优化需要跨模块思考。

━━━━━━━━━━━━━━━━━━━━

◆ 性能与精度

━━━━━━━━━━━━━━━━━━━━

论文在单张 H100 上做了系统测试，覆盖 LLaMA 系列的 1B、7B、70B 规模配置（对应隐藏维度 d = 2048、4096、8192），batch size 16K tokens。

基线对比包括 cuBLAS + torch.compile、Liger Kernels、FlashInfer、QuACK 等。

**关键结果：**

论文侧重 kernel 级和 block 级的加速比数据，没有报端到端训练吞吐的单一数字。几个有代表性的数据点：

- **GEMM-RMSNorm-GEMM 融合 kernel**：比 cuBLAS + torch.compile 的独立 kernel 组合有显著加速，核心原因就是省掉了中间的 HBM 往返。
- **交叉熵 epilogue**：在 d=8192（70B 级别）配置下，CODA 的融合交叉熵 kernel 比 Liger Kernels 快约 1.6 倍——传统做法要把完整 logits 矩阵写出再读回，CODA 在 epilogue 里用有状态变换在线算完。
- **完整 Transformer block 级别**（前向 + 反向，不含注意力），在 1B/7B/70B 配置上稳定提速。
- **精度**：在论文的 GEMM-RMSNorm-GEMM 数值实验里，BF16 下与 FP32 参考值对比，CODA 的重参数化路径相对误差更低。原因是 CODA 的 GEMM 主循环用 FP32 累加器做中间计算，而中间写回 HBM 时的 BF16 截断误差被避免了。

CODA 覆盖的是**注意力之外的几乎所有计算**（前向和反向）。注意力本身由 FlashAttention 负责——两者互补，不冲突。FA 管注意力内部的融合（Q·K 相似度 → softmax → 和 V 的加权求和），CODA 管注意力之外的一切。

━━━━━━━━━━━━━━━━━━━━

◆ 让 LLM 写 kernel

━━━━━━━━━━━━━━━━━━━━

论文里有一个有趣的工程细节：CODA 用 **Claude Code** 来生成大部分 kernel。

这听起来像吹牛，但逻辑是成立的：因为 CODA 的编程模型已经把问题简化为"定义 epilogue"。GEMM 的主循环、数据搬运、流水线调度——这些最难写也最容易出错的部分——全由 CUTLASS 库搞定，不需要 LLM 操心。LLM 只需要回答"GEMM 算完之后做什么"，本质上是写一段逐元素变换逻辑。

具体做法：给 Claude Code 一份书面规格说明 + 精选的示例代码 + 一份实现技巧日志，加上轻量的人类监督。仓库本身就是一个不断增长的示例集——写得越多，LLM 能参考的模式越多，新 kernel 写得越快。

CODA 的关键不是让 LLM 变成 CUDA 专家，而是把 kernel 编程从"开放式写 CUDA"降维成"在固定 GEMM 模板里填写 epilogue visitor"。最危险的 GEMM mainloop、tiling、pipeline、Tensor Core 调度都被框架锁死，LLM 只负责在几个固定钩子里组合原语。它不是靠 LLM 自觉写对，而是靠接口设计让它很难写歪。

**这不是"AI 自动写出 CUDA 神 kernel"的故事。是"好的抽象让问题简单到 LLM 也能做"的故事。** 抽象做对了，门槛降下来的不只是 LLM，新手工程师也一样受益。

往大了说，这是 AI 编程时代的一个缩影：每一个非程序员现在都能用 AI 写 app、写后台——"写个 CRUD"已经不是程序员的护城河了。传统程序员被迫往前走一步，去碰 kernel 融合、硬件适配、PTX 指令这些 AI 目前还搞不定的东西。而 CODA 做的事情，恰好是把这个"前一步"的门槛也降低了——定义好原语之后，连 kernel 融合也变成了 LLM 能做的活。程序员的前线还在继续往前推。

━━━━━━━━━━━━━━━━━━━━

◆ 和我们 DGX Spark 经验的呼应

━━━━━━━━━━━━━━━━━━━━

之前在 DGX Spark（sm_121）上做 DeepSeek V4 Flash 推理时，我用 Triton 写过类似的东西。比如 INT8 量化推理——`tl.dot(int8, int8)` 算完矩阵乘法后，结果还在寄存器里，直接乘 scale 因子转成 BF16 再写回。FP4 的 `tl.dot_scaled` 也一样——矩阵乘法和 block scale 在同一个 kernel 里完成。

这不是什么创新——Triton 的设计本身就鼓励你在一个 kernel 里把 matmul 和后处理写在一起，任何写过 Triton kernel 的人都会这么做。但读完 CODA 之后回头看，才意识到自己一直在做的事情叫"epilogue 融合"，而且只做了最简单的一种（逐元素 scale）。CODA 把这个思路系统化了——五类原语覆盖了 RMSNorm、RoPE、SwiGLU、残差、交叉熵，从"会写一个"变成"有一套积木箱"。

说到底，epilogue 融合本身不是什么高深技巧——在 Triton 里把 matmul 后面接逐元素操作是 CUDA 工程师的基本功，RMSNorm 的结合律重排也不难想到。CODA 真正的贡献是**从手艺到体系**：把整个 Transformer block 的所有碎活儿系统性地归约成 GEMM+epilogue 链条，抽象成五类可组合原语，搭在 CUTLASS EVT 上做成了框架。单个技巧谁都能写，把所有技巧统一成一套积木箱是另一回事。

不过 CODA 当前公开实现主要面向 H100/Hopper，依赖 CuTeDSL/CUTLASS 里那套 Hopper GEMM/epilogue 生态。DGX Spark 是 sm_121，和 RTX 5090 的 sm_120 基本同一路线：BF16/INT8/FP8/FP4 Tensor Core 都在，但没有数据中心 Blackwell sm_100 的 TMEM/tcgen05，编程模型更接近 `mma.sync`。所以 CODA 的思想能学，代码不能指望原封不动搬过来；真要在 Spark/5090 这条线上做，需要按 sm_120/121 的指令和内存模型重新适配。

技术名词速查：

- **GEMM**（General Matrix Multiply）：通用矩阵乘法。GPU 上最核心的计算原语。
- **HBM**（High Bandwidth Memory）：高带宽显存。GPU 的主存储器，容量大但读写速度相对计算速度而言是瓶颈。
- **Epilogue**（尾声）：GEMM kernel 执行完矩阵乘法后、结果写回 HBM 前的一个时间窗口。数据还在寄存器/片上缓存里，可以顺手做额外计算，避免为这些操作单独读写一整遍中间张量。
- **Memory-bound**（带宽密集）：瓶颈在数据搬运而非计算的操作。Norm、激活函数、残差加法都是典型 memory-bound 操作。
- **Compute-bound**（计算密集）：瓶颈在计算而非数据搬运的操作。大矩阵乘法是典型 compute-bound 操作。
- **RMSNorm**（Root Mean Square Normalization）：一种归一化方法，对向量除以其均方根值。
- **RoPE**（Rotary Position Embedding）：旋转位置编码，对 Q/K 的特征对做二维旋转来编码位置信息。
- **SwiGLU**：一种门控激活函数，SiLU(gate) ⊙ up，用在 Transformer 的 MLP 层。
- **Tile**：GPU 计算时把大矩阵切成的小块，每个 tile 由一组线程负责。
- **CUTLASS / CuTeDSL**：NVIDIA 的 CUDA 模板库和 Python 级 kernel 编程接口。
