【显卡选购】2026穷人想买显卡学大模型？我会碰到哪种坑？

━━━━━━━━━━━━━━━━━━━━

你以为买显卡跑大模型就是比显存、比价格、比带宽？

2026 年最大的坑不在硬件参数里——**在软件生态里**。

同样瞄准 ~96GB 显存（能跑 70B 参数模型的门槛），3 万到 8 万预算，你以为花钱就能用。实际上每条路都有坑，而且**坑的位置完全不同**。

先给急性子同学一个总览表：

| 方案 | 显存 | 预算 | 内存带宽 | 最大的坑 |
|------|------|------|----------|----------|
| 3× AMD R9700 | 32GB×3=96GB | ~¥3.3万 | 640GB/s×3 | 多卡 TP=2 死锁，ROCm 生态 |
| 2× AMD W7900 | 48GB×2=96GB | ~¥3万 | 864GB/s×2 | ROCm 生态，多卡不稳 |
| 1× DGX Spark | 128GB 统一内存（有效~96GB） | ~¥3.1万 | 273GB/s | 带宽只有 5090 的 1/6.5 |
| 1× RTX 5090 | 32GB（不够 96） | ~¥3万 | 1,792GB/s | 显存不够，多卡 P2P 被禁 |
| 1× RTX PRO 6000 | 96GB | ¥7-8万 | 1,792GB/s | 花 8 万也是 sm_120，FA4 跑不了 |

看完这张表你大概已经困惑了——为什么每个方案后面都跟着一个"坑"？下面一个一个拆。

━━━━━━━━━━━━━━━━━━━━

◆ 坑一：sm_100 vs sm_120 —— "Blackwell"不是一个东西

━━━━━━━━━━━━━━━━━━━━

这是 2026 年最容易被忽略、也最致命的坑。

NVIDIA 把 2024-2025 年发布的新架构统一叫"Blackwell"，但**这个名字底下藏着两套完全不同的硬件**：

- **sm_100**（B200/B300）：数据中心 Blackwell。有一个叫 **TMEM（Tensor Memory）** 的硬件单元，每个 SM 有 256KB 专用张量内存，配套 tcgen05 指令集
- **sm_120**（RTX 5090/RTX PRO 6000）：消费和工作站 Blackwell。**没有 TMEM**，用的还是 Ampere 时代的 mma.sync 指令

💡 翻译成人话：sm_100 和 sm_120 的关系，就像柴油发动机和汽油发动机——都叫"发动机"，但你不能往汽油车里加柴油。

────────────────────

【这跟你有什么关系？】

FlashAttention——大模型推理和训练里**最核心的加速组件**——有好几个版本：

- **FA2**（2023）：给 Ampere/Hopper 写的，兼容 sm_80/sm_89/sm_90
- **FA3**（2024）：Hopper H100 专属（sm_90）
- **FA4**（2025）：数据中心 Blackwell 专属（sm_100），**硬件依赖 TMEM**

FA4 需要的 TMEM 硬件，在 sm_120 的 GB202 芯片上**物理不存在**。这不是软件锁，不是 fuse bit，不是工具链没适配——**是硅片级别的架构差异**。

RTX PRO 6000 花 7-8 万也是 sm_120，和 5090 同一块 GB202 芯片。FA4 一样跑不了。

消费卡目前只能用 FA2（2023 年的版本），需要通过社区 workaround 编译 Triton 后端才能跑。

**老黄切的不是显存，是指令集。这才是真正的产品分级。**

参考：
- SGLang Discussion #10564：FA4 在 sm_120 上不可用的讨论
- CUTLASS Issue #2800：FP4 操作被代码写死限定 sm_100a
- DeepWiki：SM100 Memory Hierarchy and TMEM 技术文档

━━━━━━━━━━━━━━━━━━━━

◆ 坑二：多卡通信的隐形墙

━━━━━━━━━━━━━━━━━━━━

跑大模型最直觉的思路：显存不够？多买几张卡拼起来啊！

想法没错，但 2026 年的现实是：**消费级多卡已经是二等公民**。

────────────────────

【NVIDIA 这边：P2P 被驱动层软禁】

GPU 之间最快的通信方式叫 P2P（Peer-to-Peer）——两张显卡的显存直接对传，不经过 CPU。但 NVIDIA 从 RTX 30 系开始，在驱动层把消费卡的 P2P **软件禁用**了。

硬件能力在不在？**在。**

有人 patch 了 NVIDIA 的开源内核驱动，强行打开 P2P，同时绕过 vLLM 里写死的 P2P 能力检测。结果双 RTX 3090 跑 Qwen 3.5 35B 的吞吐量**直接提升 10-30%**。

这说明什么？硬件支持 P2P，老黄就是不让你用——逼你买专业卡。

但 RTX PRO 6000 也没有 NVLink！花 7-8 万也只有 PCIe Gen5 x16，多卡互联没有比消费卡好到哪去。最后一代有 NVLink 的消费卡是 **RTX 3090（2020 年）**。从那以后，消费级多卡就再也没有过高速互联。

参考：
- smcleod.net："Patching NVIDIA's driver and vLLM to enable P2P on consumer GPUs"
- vLLM Issue #14628：双 RTX 5090 TP=2 多卡推理失败
- NCCL Issue #1637：双 5090 的 NCCL P2P 问题

────────────────────

【AMD 这边：更惨】

- **双 R9700 TP=2**：RCCL 死锁——两张卡 GPU 占用率锁定 100%，推理直接挂死。社区需要降级 RCCL 到 7.1.1 才能 workaround（vLLM Issue #40980，ROCm Issue #5480）
- **双 7900 XTX**：RCCL 2.27.7 集合通信失败，只在双卡同时参与 collective 时触发（ROCm Issue #6074）

────────────────────

【DGX Spark：没这个问题，但有别的问题】

DGX Spark 是统一内存架构，CPU 和 GPU 共享同一块内存池，不存在多卡通信问题。但它的代价是——**带宽只有 273GB/s**，RTX 5090 的 1/6.5。大模型推理是内存带宽密集型任务，这个差距直接反映在推理速度上。

━━━━━━━━━━━━━━━━━━━━

◆ 坑三：vLLM / SGLang 不是开箱即用

━━━━━━━━━━━━━━━━━━━━

vLLM 和 SGLang 是目前最主流的两个大模型推理框架。你在网上看到的各种"本地部署 XX 模型"教程，十有八九用的就是它们。

但在消费卡上，它们**全都不是 pip install 就能跑的**。

────────────────────

【NVIDIA sm_120 的问题清单】

1. **装了跑不了**：PyPI 预编译包不含 sm_120 kernel。`pip install vllm` 装完启动就报错："no kernel image is available for execution on the device"（vLLM Issue #16901）。**必须源码编译**，需要 CUDA 12.8+ 和 PyTorch 2.6+ 的环境

2. **FP4 MoE 模型输出乱码**：CUTLASS 的 grouped GEMM 在 sm_120 上产生垃圾输出。比如跑 Nemotron-3-Nano-30B 的 NVFP4 版本直接废掉（vLLM Issue #35065，CUTLASS Issue #3096）

3. **SGLang 自动选错后端**：SGLang 检测到 Blackwell 架构后自动选择 trtllm_mha 注意力后端，然后立刻报错——"只支持 SM100"。你得手动指定用 Triton 后端（SGLang Issue #14814）

4. **双卡 TP=2 崩溃**：双 5090 开 tensor parallelism 直接卡死在 NCCL 初始化阶段（vLLM Issue #21491）

5. **Flash Attention 编译**：FA3 不支持 sm_120，FA4 硬件不兼容，只能退回 FA2 的 Triton 实现（vLLM Issue #36865）

────────────────────

【AMD 的问题清单】

1. **容器里启动就崩**：vLLM 的 ROCm 平台检测依赖 amdsmi，但容器里 sysfs/hwmon 路径不暴露，导致平台检测返回 None，引擎直接退出（vLLM Issue #40081）

2. **多卡 TP=2 死锁**：已经在坑二说过了。R9700 双卡 100% 占用挂死，7900 XTX 双卡 RCCL 集合通信失败

3. **W7900 跑 vLLM 间歇性失败**：mxfp4 编译错误、chunked_prefill 挂起（vLLM Issue #33906，#5013）

💡 翻译成人话：你以为装个框架就像装个 app，实际上更像自己组装宜家家具——说明书还是瑞典语的。

━━━━━━━━━━━━━━━━━━━━

◆ 坑四：AMD 特有 —— ROCm 是另一个世界

━━━━━━━━━━━━━━━━━━━━

如果你看中 AMD 卡的性价比（R9700 三张 3.3 万就有 96GB 显存），你需要做好心理准备：**ROCm 生态和 CUDA 生态的差距，远比你想象的大**。

────────────────────

【Flash Attention 装不上】

`pip install flash-attn` 装不了——PyPI 包只有 CUDA 版。必须从 ROCm fork 源码编译。

更要命的是：FlashAttention 的 CK（Composable Kernel）后端是给 **Wave64**（数据中心 MI300X 的执行模型）写的。R9700 是 RDNA4 架构，用的是 **Wave32**——指令宽度减半，内核直接不兼容。

社区 workaround 是设置 `FLASH_ATTENTION_TRITON_AMD_ENABLE="TRUE"` 强制走 Triton 后端，但性能比原生 CK 后端差。

FA3 和 FA4？**没有 AMD 移植。**

参考：FlashAttention Issue #2452，ROCm Discussion #5226

────────────────────

【训练/微调不靠谱】

- **QLoRA 在 R9700 上 GPU 内存故障**：跑 Llama 3 8B 的 QLoRA 微调时出现 GPU memory access fault。临时解决方案是锁定 transformers==4.57.5（ROCm Issue #5832）
- **W7900 跑 vLLM 间歇性脚本失败**：mxfp4 格式在 gfx1100 上编译报错（vLLM Issue #33906）

────────────────────

【Windows 几乎不能用】

ROCm 在 Windows 上只有 PyTorch 能勉强跑。vLLM、SGLang、FlashAttention——全是 Linux only。你要是 Windows 用户，要么装双系统，要么用 WSL2（又多一层坑）。

────────────────────

【能跑的部分】

说句公道话：单卡推理用 llama.cpp / Ollama 是**完全能跑的**。RDNA4 的 llama.cpp 支持已经比较成熟。问题出在你想做更多的时候——想用推理框架、想多卡并行、想微调——每一步都是新的坑。

━━━━━━━━━━━━━━━━━━━━

◆ 坑五：DGX Spark 特有 —— 仓库不是跑道

━━━━━━━━━━━━━━━━━━━━

DGX Spark 的卖点很直接：128GB 统一内存，~¥3.1万，能把 DeepSeek V4 Flash 280B 参数（FP4 量化）直接塞进去。

听起来很美。但 **能装下 ≠ 能跑快**。

────────────────────

【带宽是硬伤】

核心数据：

| 设备 | 内存带宽 |
|------|----------|
| DGX Spark | 273 GB/s |
| RTX 5090 | 1,792 GB/s |
| AMD R9700 | 640 GB/s |
| AMD W7900 | 864 GB/s |
| RTX PRO 6000 | 1,792 GB/s |

大模型推理（尤其是 decode 阶段）的瓶颈是**内存带宽**——每生成一个 token 都要把模型权重从内存搬到计算单元。DGX Spark 的带宽差 RTX 5090 有 6.5 倍。

实测 DeepSeek V4 Flash 280B（FP4）在 DGX Spark 上：**4.2 tok/s**。能用，但体感就是"每个字都要等一下"。做研究验证够了，当推理服务不行。

────────────────────

【sm_121：比 sm_120 还孤独】

DGX Spark 的 GPU 报告为 **sm_121**——这是 DGX Spark 专属的计算能力编号。它既不是 sm_100（数据中心），也不是 sm_120（消费卡），是一个更小众的第三类。

这意味着所有针对 sm_100 或 sm_120 的 workaround，**不一定适用于 sm_121**。比如：

- CUTLASS 的 FP4 GEMM 在 sm_121 上因为共享内存容量不够（99KB vs B200 的 228KB）直接溢出失败（CUTLASS Issue #2947，TensorRT-LLM Issue #11368）
- SGLang 的 Triton attention + CUDA graphs 在 sm_121 上非法内存访问（SGLang Issue #19799）

DGX Spark 适合一件事：**验证"这个模型能不能跑通"**。它是仓库，不是跑道。

━━━━━━━━━━━━━━━━━━━━

◆ 坑六：其他软件层的地雷

━━━━━━━━━━━━━━━━━━━━

除了上面的大坑，还有一些零散但恼人的软件问题：

────────────────────

【CUTLASS】

- FP4 kernel 限定 sm_100，sm_120 被 `admissible_archs` 写死排除（Issue #2800）
- MoE grouped GEMM 在 sm_120 上输出乱码，需要等 CUDA 13.0 的 `compute_120f` 才有修复（Issue #3096）

────────────────────

【TensorRT-LLM】

- FMHA（Fused Multi-Head Attention）的 cubin 没编译 sm_120/sm_121 版本，只有 sm_100 和 sm_103（TensorRT-LLM Issue #11799）
- 消费卡被迫 fallback 到未融合的 MHA kernel，性能打折

────────────────────

【bitsandbytes】

- `matmul_4bit` 在 sm_120 上报 AssertionError，根源是 PyTorch 和 bitsandbytes 的 CUDA 版本不匹配链（ComfyUI Issue #11864）
- 解决方案：升级到 PyTorch 2.11.0+cu129 + 对应版本的 bitsandbytes

────────────────────

【cuDNN】

- cuDNN 本身支持 sm_120，但上层框架（vLLM/SGLang/FlashInfer）**没更新 dispatch 路径**来包含 sm_120。cuDNN 能用，但没人调用它

💡 翻译成人话：底层零件都有了，但组装说明书还没写完。你买回来的是一堆零件，不是成品。

━━━━━━━━━━━━━━━━━━━━

◆ 那到底该怎么选？

━━━━━━━━━━━━━━━━━━━━

说了这么多坑，不给个选购建议就是耍流氓。按需求分流：

────────────────────

**只想跑推理玩玩，不想折腾**

→ 任意一张卡 + llama.cpp / Ollama。NVIDIA、AMD 都能跑，Ollama 把编译的活儿都干了。这是 2026 年唯一"开箱即用"的路径。

────────────────────

**想用 vLLM 跑推理服务**

→ NVIDIA 系（5090 / PRO 6000）。接受源码编译，接受 FA2 而不是 FA4。AMD 的 vLLM 支持还在"能跑但经常崩"的阶段。

────────────────────

**想跑 70B+ 大模型，显存优先**

→ 两个选择：
- **DGX Spark**（128GB，最大最便宜但最慢）
- **2× W7900**（48GB×2=96GB，ROCm 要折腾，多卡通信有风险）
- RTX 5090 单卡只有 32GB，70B 的 Q4 量化勉强能塞但没余量

────────────────────

**想训练/微调**

→ NVIDIA 单卡。AMD 单卡 QLoRA 能跑但有内存故障风险，多卡别碰。DGX Spark 带宽太低训练不实际。

────────────────────

**有钱全都要**

→ RTX PRO 6000 单卡 96GB + 1,792GB/s 带宽，目前消费/工作站级别的最优解。但别以为没坑——sm_120 的所有软件问题它一个不少，7-8 万块买不到 FA4。

━━━━━━━━━━━━━━━━━━━━

◆ 写在最后

━━━━━━━━━━━━━━━━━━━━

2026 年穷人跑大模型没有完美选项。

NVIDIA 用指令集切产品线，消费卡永远是二等公民。AMD 有性价比但生态没跟上。DGX Spark 能装大模型但跑不快。

**每条路都有坑，区别只在于坑的位置你能不能忍。**

选之前，先想清楚你到底要干什么——是玩玩聊天机器人，还是跑推理服务，还是做微调研究。需求不同，能忍的坑就不同。

别问哪个最好。问你自己哪个坑你踩得起。
