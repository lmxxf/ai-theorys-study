【端侧 AI 工程】给 OpenHarmony 上的 AI Agent 一个完整 Linux 工具箱——chroot 沙盒落地手记

━━━━━━━━━━━━━━━━━━━━

◆ 缘起：Agent 想"写个程序跑跑"，可惜端侧没有 Python

━━━━━━━━━━━━━━━━━━━━

我在 OpenHarmony 上做了一个跑大模型的 AI Agent。这种 agent 在 PC / 云端，干起活儿来是这样的：

> 用户："帮我看看这个目录下哪些 .log 文件超过 100 MB。"
> Agent：（自动写一行 shell）`find /var/log -name '*.log' -size +100M`
> Agent：（自己跑，自己看结果，自己用人话告诉你）

人类工程师在终端里干的活，agent 完全可以现场写出来现场跑。它本质上就是一个**会写代码的工具调度大脑**。

> 不熟悉"LLM 怎么调工具"的同学可以先看一下早先的基础篇 [《LLM 是怎么决定"回答问题"还是"调用工具"的？》](https://mp.weixin.qq.com/s/AdwO3jDD4K0__OzyTB9XsQ)（公众号往期 No.116）——一句话总结：**LLM 自己不会执行任何东西，它只会"说话"输出一段特定格式的 JSON，应用代码解析这段 JSON 去真正跑命令，然后把结果塞回 LLM 的上下文**。今天这篇要讲的端侧沙盒，就是 116 那篇里"被应用代码真正跑起来"的那一环——只不过这一环我把它放到了 OpenHarmony 端侧设备上。

但搬到 OpenHarmony 上有个尴尬：

**OH 端侧的命令行工具集是 toybox**——一个由 Android 团队维护、定位极简的 BSD 工具集。它的设计目标是**够用就行**：cp、mv、ls、grep 都有，但**没有 python3、没有 nodejs、没有 git、没有 jq、没有 ripgrep**。也就是说 agent 想现场写个 Python 脚本算个东西——没解释器。想 `git clone` 一个开源仓——没 git。想 `jq` 解析一段 JSON——没 jq。

这事如果发生在桌面 Linux 上，三个字解决：`apt install`。在 OH 上呢？OH 用户态空空荡荡，连 apt 都没有。

那为啥 OH 不内置这些？因为 OH 是端侧操作系统，定位是手机/物联网设备，**用户态保持极简**是设计选择——你不会希望你的智能手表自带一个 GCC 编译器。但它和"上面跑 AI agent"这个新需求撞上了。

【小白翻译】**端侧 / 端侧 AI**：相对于"云端"——把模型/智能体跑在用户身边的设备上（手机、手表、车载、边缘盒子），而不是远程数据中心。延迟低、隐私好，但资源紧、生态弱。

💡 这篇是工程实录：怎么用 100 行 Python + 一份配置文件，给 OH 上的 agent 凭空变出一个**完整的、隔离的、零虚拟化开销的 Linux 命令行环境**。不打算讲深奥的论文，全是踩坑实录。

━━━━━━━━━━━━━━━━━━━━

◆ 思路：把整个 Linux 用户态当资源包预置进去

━━━━━━━━━━━━━━━━━━━━

工具不够用，**最直接的办法是把工具搬进来**。但搬进来不能污染宿主——agent 万一手抖把宿主的 `/system` 写坏了怎么办？

业界标准答案有三种：

| 方案 | 隔离强度 | 端侧可行性 |
|---|---|---|
| **Docker 容器** | 中（namespace + cgroup） | ❌ OH 没装 dockerd，硬塞要改几十 MB |
| **虚拟机** | 强 | ❌ 端侧设备内存不够撑虚拟机 |
| **chroot 沙盒** | 弱（只隔离文件系统） | ✅ Linux 内核原生支持，零开销 |

我选了 **chroot**——一个 1979 年就存在的老古董技术。

【小白翻译】**chroot**：把进程的"根目录"换掉。进程 chroot 之后，它看见的 `/` 就不再是真实的 `/`，而是你指定的某个子目录。它从此被"困"在这个子目录里，看不到外面的世界。

为什么用最简陋的 chroot 而不是更花的容器？因为我的需求很纯粹：

- 不需要防 agent 越狱（这是自己的 agent，又不是不信任的外人）
- 不需要复杂资源限制（agent 跑个脚本能用多少资源）
- 只需要"agent 在沙盒里 `pip install` 别污染宿主、agent 装的乱七八糟东西想清就清"

chroot 完美匹配。

**具体方案**：

1. 在 PC 上把一个完整的 Alpine Linux 用户态目录（约 540 MB，含 Python 3.12、Node.js 22、Git、jq、SQLite……全套工具）打成 tar.gz
2. 这个 tar.gz 跟着 OpenHarmony 系统镜像一起烧录到设备的 `/system/etc/agent-sandbox/`
3. 设备开机时自动解压到 `/data/alpine/`
4. Agent 想跑命令时，**fork 一个子进程 → 子进程 chroot 到 `/data/alpine/` → 然后 execve 跑 shell 命令**——这一瞬间，agent 看见的就是一个完整的 Alpine Linux

整个机制核心代码不到 100 行 Python。

💡 为啥选 Alpine 而不是 Ubuntu？Alpine 用 **musl libc**，体积小（rootfs 整套压完 200 MB 多点）；Ubuntu 用 **glibc**，光基础系统就好几个 GB。OH 内核能同时跑两种 libc 的二进制（musl 二进制在 OH 上原生跑没问题——内核 ABI 是统一的，差异只在用户态库）。

◆ 三轮打包：跟 toybox tar 斗智斗勇

━━━━━━━━━━━━━━━━━━━━

打这个 tar.gz 是个**惊心动魄的故事**。

**第一轮**：用 GNU tar 配 `--transform` 把顶级路径改写成 `alpine/`，打出 194 MB 包，扔进系统镜像，烧机。点测试按钮——

> ```
> [!] 解压失败
> ```

去掉 stderr 重定向看真错：

> ```
> tar: './' not under '/data/alpine-debug'
> ```

**OH 自带的 toybox tar 有个 path traversal 防御**：拒绝解压名字叫 `./` 的 entry。我虽然用 `--transform` 把普通文件路径改写了，但 GNU tar 在包头部多塞了一个独立的 `./` 顶级 entry——这个 toybox tar 一眼就拦下。

**第二轮**：那别用 GNU tar 了，换 OH 自带的 toybox tar 打包——它自己打的总不会被自己拒了吧？

```sh
tar czf alpine.tar.gz alpine/
Signal 11
```

**段错误**。toybox tar 在 12000 多个文件的大目录上直接崩了。

**第三轮**：换 busybox tar——它的实现独立。这次跑通了，但包大小……

> 541 MB → **534 MB**

压缩比 ≈ 0。原来 busybox 内置的 gzip 是个"假压缩"实现——只走最轻量级的算法。OH `/system/bin/gzip` 也是 busybox 的 symlink，**整个 OH 用户态里就没一个能正常压缩的 gzip**。

最后定型：**必须进 Alpine chroot，用 Alpine 自带的 GNU tar + GNU gzip 打包**——但 chroot 进去后根目录就是 Alpine 自己，怎么打"alpine 自己"？

绕了个 bind mount——把 alpine 根目录自挂载到 chroot 内的 `/mnt/packroot/alpine`，然后从 `/mnt/packroot` 路径下 `tar alpine`。这样包内顶级条目就是 `alpine/...`，**绕开了 toybox 拒 `./` 的雷，又用上了 alpine 的正经压缩工具**。

最终产物：194 MB（gzip），解压后 540 MB。**OH toybox tar 能正常解开**。

> 这三轮踩坑让我明白一个事：**轻量工具集（toybox / busybox）的"够用"是真的够用——不是绝对意义的够用**。某些边角行为跟 GNU 工具差距大到能让你怀疑人生。

◆ 真正的拦路虎：SELinux 与 Mount Namespace

━━━━━━━━━━━━━━━━━━━━

包终于打好了，烧进系统镜像。开机后流程是这样的：

```
LLM → 调用 sandbox_cli_exec 工具
   → SA 进程 fork 子进程
   → 子进程 chroot 到 /data/alpine
   → execve /bin/sh
   → 跑命令
   → 返回结果
```

听起来很顺。但有两个问题等着我。

**第一个问题：SA 没权限 mount**

按惯例，进 chroot 之前你得挂上 `/proc`、`/sys`、`/dev`，否则 chroot 内的进程看不到这些虚拟文件系统，很多命令（特别是 `ps`、`top`）会崩。

但是——SA（我这个 agent 服务）在 OH 上跑在 **`u:r:sh:s0`** 这个 SELinux 域里。这个域**没有 CAP_SYS_ADMIN**（不能 mount）、**没有 CAP_CHOWN**（不能改文件属主）。

【小白翻译】**SELinux / capability**：Linux 内核里的两套权限系统。SELinux 是"基于标签的强制访问控制"——给每个进程贴个标签，标签决定它能干啥。capability 是把传统的 root 权限拆成几十个独立小权限，进程只拿需要的那几个。两者一起决定一个进程"能"和"不能"做什么——uid=0 root 也不是为所欲为。

【小白翻译】**SA**：System Ability，OH 的系统服务进程。我的 agent 就是个 SA。

直接在 SA 里 mount → 失败。

**解法**：分离权能。专门写一个**特权辅助进程**（叫 Prepare Service），跑在 `u:r:su:s0` 这个有 mount 权限的域里，**只在开机时跑一次**，负责解压沙盒。SA 本身还是受限的，但它跑的时候沙盒已经准备好了，它只需要 chroot + execve——这两个不需要任何额外权限。

```
┌────────────────────────────────────────────┐
│ OpenHarmony 开机 init                       │
│                                            │
│ ┌─────────────────┐   ┌────────────────┐   │
│ │ Prepare Service │   │ Agent SA       │   │
│ │ 特权 once=1      │   │ 受限 长跑       │   │
│ │ 解压沙盒到       │   │ chroot+execve  │   │
│ │ /data/alpine    │   │ 跑命令         │   │
│ │ 跑完即退         │   │                │   │
│ └─────────────────┘   └────────────────┘   │
└────────────────────────────────────────────┘
```

跑完即退的特权服务 + 长跑的受限服务——只在开机那 30 秒短暂存在高权能进程，之后就只剩受限的 SA 跑。这是"最小特权原则"的工程落地。

**第二个问题：Mount Namespace 隔离**

这是真正让我懵了半天的坑。

Prepare Service 挂上了 `/proc`、`/sys`、`/dev`。SA 起来后，去看——

```sh
mountpoint -q /data/alpine/proc && echo OK || echo NO
NO
```

挂的呢？我手工 hdc 进设备看，明明挂着啊：

```sh
mountpoint -q /data/alpine/proc && echo OK
OK
```

**hdc shell 能看到，SA 看不到**。

查 `/proc/<pid>/ns/mnt`——

```
hdc shell:  mnt:[4026531840]   ← namespace A
SA python:  mnt:[4026532998]   ← namespace B（不一样）
```

**OH 给 SA 服务做了独立的 mount namespace**。Prepare Service 挂的 mount 在它自己的 namespace 里，SA 看不到。

【小白翻译】**Mount Namespace**：Linux 内核机制。同一台机器上不同进程组可以有完全不同的"挂载点视图"——A 看见 `/data` 挂着 SSD，B 看见 `/data` 是空的。Docker、容器、Android 应用沙盒都靠这个隔离。OH 把它用到了系统服务级别。

这事**几乎让我放弃了沙盒方案**。namespace 隔离没法绕过——除非 SA 自己有 CAP_SYS_ADMIN 自己 mount。

但绝望之中我做了一个 dirty trick——直接试试**不挂任何 /proc /sys /dev 的纯 chroot**，看看跑命令会怎样：

```sh
chroot /data/alpine /bin/sh -c "python3 -c 'print(2+2)'"
4
```

**居然能跑**。

那 `node`、`git`、`curl`、`sqlite`、`jq`、`bash` 这些 agent 实际会用到的命令呢？**全能跑**。

只有 `htop`、`ps`、`free` 这种**显式依赖 /proc** 的命令会崩。但 agent 几乎从来不调这些——它要算东西、跑脚本、调 API，几乎不需要看进程列表。

**完全放弃挂 mount 这条路就走通了**。SA 只需要 chroot + execve，零额外权限，整个架构变得极其干净。

━━━━━━━━━━━━━━━━━━━━

◆ 第三关：Agent 框架要的不是 dict，是 ToolResponse

━━━━━━━━━━━━━━━━━━━━

mount 问题绕过了。沙盒跑起来了。Agent 框架（AgentScope）这边——

LLM 调用 `sandbox_cli_exec` 工具→工具返回了一个 Python `dict`→agent 框架直接抛错：

```
The tool function must return a ToolResponse object,
but got <class 'dict'>.
```

这个就是工程师常遇到的"框架礼仪"问题。AgentScope 要求工具返回它定义的 `ToolResponse` 对象——里面有规范的 content / metadata / 流式标志等字段——而不接受裸 dict。包一下就行：

```python
async def sandbox_cli_exec(cmd: str, timeout: int = 30):
    r = SandboxManager.instance().exec(cmd, timeout)
    # 把 dict 包成 ToolResponse 文本
    text = f"exitcode={r['exitcode']} duration_ms={r['duration_ms']}\n"
    if r["stdout"]: text += f"--- stdout ---\n{r['stdout']}"
    if r["stderr"]: text += f"--- stderr ---\n{r['stderr']}"
    return _text_response(text)
```

包完，再让 LLM 调一次——

> 用户：用 sandbox_cli_exec 跑一下 uname -a
> Agent（调用工具）：→ 沙盒里跑 `uname -a` →
> ```
> Linux localhost 5.10.184 #1 SMP PREEMPT Mon Jun 22 ... aarch64 GNU/Linux
> ```
> Agent（用人话）：当前沙盒系统是 Linux，内核 5.10.184，aarch64 架构。

**通了**。

完整链路：用户在 App 上说一句中文 → 端侧 LLM 解析 → 决定调 sandbox_cli_exec 工具 → SA 接到调用 → fork+chroot+execve 进沙盒跑 shell → 拿到 stdout → LLM 用中文总结回复用户。

从用户敲完字到看到回复，**几秒钟**。

◆ 真机演示：让 agent 算个阶乘

━━━━━━━━━━━━━━━━━━━━

我做了几个测试按钮，最直观的：

**按钮 1：「沙盒 uname」**

> 用户：用 sandbox_cli_exec 工具跑一下 uname -a，把输出告诉我。

Agent 干的：

1. 决定调 sandbox_cli_exec
2. cmd 参数 = `"uname -a"`
3. SA 执行 → 沙盒返回 `Linux localhost 5.10.184 ... aarch64 GNU/Linux`
4. LLM 把这行原样告诉用户

**按钮 2：「沙盒阶乘」**

> 用户：用 sandbox_cli_exec 工具跑 `python3 -c "import math; print(math.factorial(8))"`，告诉我结果。

Agent 干的：

1. 调 sandbox_cli_exec
2. cmd 参数 = `'python3 -c "import math; print(math.factorial(8))"'`
3. 沙盒里 Python 3.12 解释执行 → 返回 `40320`
4. LLM：8 的阶乘是 40320

40320 这个数字**不是 LLM 自己算的**——它是沙盒里那个真实的 Python 解释器算出来的，LLM 只是把结果"读"出来念给你听。

> 这个区别非常关键：**LLM 不需要"会算阶乘"，它只需要"会写出算阶乘的命令"**——计算让真实的程序去做。模型不靠不靠谱无所谓，结果是真的。

◆ 这事的意义：Agent 的能力从"懂"变成"会查工具"

━━━━━━━━━━━━━━━━━━━━

讲完工程，讲点更大的事。

很多人对 AI 的能力有个误解：**模型越大、参数越多，能力就越强**。

但实际上当一个 LLM 接上工具（tool calling）之后，它的能力上限就**不再由参数决定**了。一个 30B 参数的小模型 + 一个完整的 Linux 沙盒，跑出来的效果可以比 600B 参数的纯模型还强——因为：

- 算 8 的阶乘？30B 自己可能算错，但它会调 `python3 math.factorial`，**沙盒返回的是确定答案**
- 解析一段 JSON？30B 可能漏字段，但它会调 `jq`，jq 不会漏
- 查一个开源仓的 README？30B 不知道这个仓，但它会调 `git clone` 然后 `cat README.md`

LLM 的角色从"无所不知的智慧体"变成了**"会用工具的研究员"**。它不需要自己懂所有事，它只需要知道**哪个工具能解决眼前的问题**，然后把工具跑起来、读结果、把结果翻译给用户。

这个范式转变在云端 agent（Claude Code、Cursor 那种）上已经走得很远了。但**端侧 agent 一直走得磕磕绊绊**——因为端侧没工具。

这次的工作，就是把这个**"工具腰包"**给端侧的 agent 配齐。沙盒里有 Python，agent 就能现场算。沙盒里有 git，agent 就能现场拉代码。沙盒里有 sqlite，agent 就能现场查数据库。沙盒里有 curl，agent 就能现场调 API。

**没有这个腰包，30B 的端侧模型就是个会写字的鹦鹉。有了这个腰包，它是个能干活的助手**。

◆ 一些没解决的事

━━━━━━━━━━━━━━━━━━━━

诚实说，还有几个问题悬着：

1. **Prepare Service 自启动一旦失败就 stuck**。我用 `once=1`，跑一次失败就不再重试。首次开机如果出了奇怪问题，得 reboot 设备让它再跑一次。这个其实可以做得更鲁棒（带退避重试），还没做。

2. **沙盒和宿主某些目录的共享**。比如 agent 想看一下 `/data/log/system.log`——它现在在沙盒里看不到这个宿主目录。要让它看到，需要在 prepare 阶段 bind mount。但 bind mount 又回到那个 mount namespace 隔离的问题——SA 看不到 prepare 挂的东西。这条路目前没绕通。

3. **沙盒里也跑不起 htop / ps**。因为没挂 /proc。如果未来真有需要看进程的场景，得想办法让 SA 自己挂——这意味着要给 SA 一些 mount 权能，又是 SELinux 策略一通改。

4. **多 agent 并发调用**。目前用一个 Python 全局锁串行化 exec。同时只能一个命令在跑。够用，但不够漂亮。

但这些都是"锦上添花"。**核心能力——端侧 agent 能现场写 Python 跑出真实结果——已经稳定可用了**。

- **demo App 测试按钮**：20 行 ArkTS

总共**不到 700 行**。但它要解决的问题——让一个端侧 AI agent 在 OpenHarmony 上拥有完整的 Linux 命令行能力——在两个月前我还会觉得几乎不可能。

不是因为技术多深奥。是因为这事涉及到**操作系统、AI 框架、SELinux 策略、init 启动顺序、Linux capability、namespace 隔离**这一长串各自独立、但必须每一段都对的链条。**任何一段不通，整套塌**。

我花了一个下午接通最后一段链路——SA 把 dict 包成 ToolResponse 的那两行代码——然后整条线"哗"地一下亮起来。LLM 真的会去用 Python 算阶乘了。

那一刻挺神奇的。

不是因为模型多聪明，是因为**整个系统第一次像一个完整的人在工作**：你说一句中文，它理解、决定用什么工具、写出工具调用、跑起来、看结果、用中文回你。

而它跑的那个工具，是一个 1979 年发明的 chroot + 一个 2009 年发布的 Alpine Linux + 一个 1992 年的 Python 解释器。AI 时代最新的部件，是被几十年前的老技术抬上去的。

这大概就是 2026 年的味道。
