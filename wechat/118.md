【鸿蒙AI+】在 OpenHarmony 开发板上跑 Python + LangChain

——三个 Segfault 和一行修复：把 Python 从静态链接改成动态链接的破案过程

━━━━━━━━━━━━━━━━━━━━

◆ 背景：Python 程序到底需要编译什么

━━━━━━━━━━━━━━━━━━━━

在聊具体的坑之前，先说清楚一件事：**"Python 不需要编译"是个误解。**

你在 Ubuntu 上 `pip install langchain` 一键搞定，那是因为别人替你把脏活干完了。Python 程序能跑起来，底下其实有好几层东西：

```
你写的 Python 代码（.py）
    ↑ 运行在
CPython 解释器（python3 命令本身，50 万行 C 代码编译出来的）
    ↑ 依赖
C 标准库（glibc 或 musl，程序和 Linux 内核之间的翻译层）
    ↑ 调用
Linux 内核（syscall）
```

在 Ubuntu 上，这些全是现成的——系统自带 glibc，自带 python3，pip 仓库里有编好的二进制轮子（wheel）。你感觉不到"编译"的存在。

**但在 OpenHarmony 上，以上每一层都没有。** 没有 python3，没有 pip，甚至 C 标准库都不一样（OH 用的是魔改版 musl，不是 glibc）。所以要在 OH 上跑 Python 程序，得从地基开始搭：

```
第 1 层：交叉编译 OpenSSL、zlib 等 C 库（给 Python 的网络/压缩功能用）
第 2 层：交叉编译 CPython 解释器本身（链接上面的 C 库）
第 3 层：把纯 Python 包（.py 文件）直接拷进去 ← 这步不用编译
第 4 层：有 C/Rust 扩展的包（.so 文件）→ 每个都要交叉编译
```

**"交叉编译"**是什么意思？就是在 x86 电脑上编译出能在 ARM 开发板上跑的程序。你不能在板子上现场编译——板子性能太弱，而且上面连编译器都没有。

━━━━━━━━━━━━━━━━━━━━

◆ 两条路线：搬 glibc 还是重新编译

━━━━━━━━━━━━━━━━━━━━

把 Python 移植到 OpenHarmony，业界有两种做法：

**路线 A：搬 glibc 兼容层上去**

不重新编译任何东西。把 Linux 上的 glibc（GNU C 标准库）连同编好的 Python 二进制一起搬到 OH 上，靠兼容层让 glibc 程序在 musl 环境里硬跑。

好处很明显——省事，依赖链上几十个库都不用重新编译，搬上去就完了。但代价是：OH 上同时跑着两套 C 库（musl + glibc），内存多占一份，动态链接器也是两套，碰到 OH 内核的安全沙箱或定制 syscall 可能莫名其妙崩。**本质上是在鸿蒙里塞了个 Linux 子系统。** 能用，但脏。

**路线 B：用 OH 工具链从源码交叉编译**

痛苦路线。OpenSSL、zlib、CPython，每一个都用 OH 的 clang 编译器重新编译，链接 OH 自己的 musl。出来的产物是 OH 原生二进制，不带任何外挂。

坏处也很明显——每碰到一个有 C/Rust 扩展的库（pydantic-core、cryptography……）都得再交叉编译一次。

**我们选的是路线 B。** 虽然苦，但编出来的东西是 OH 的原生公民，不需要兼容层，不埋隐患。反正要编的硬骨头也就那几个，编一次就完事。

第 1、2 层我们上一篇已经搞定了。这篇要解决的是：**第 2 层的一个关键架构决策——静态链接还是动态链接——选错了，导致第 4 层走不通。**

━━━━━━━━━━━━━━━━━━━━

◆ 问题的起因

━━━━━━━━━━━━━━━━━━━━

之前在 OpenHarmony 6.0 的 RK3568 开发板上，用静态链接编出了一个 11MB 的 CPython 3.11.11 二进制，能跑 DeepSeek API。当时觉得"大功告成"。（整个交叉编译过程是我和 Claude Code 协作完成的——我提需求、验板子，它写脚本、排故障。下面的排查过程也是。）

结果一想需要的 LangChain 功能——得在 OH 上跑最新版才行。

如果你在 OH 上移植了 Python 和 pip，尝试 `pip install langchain`，你会发现：**装上的不是最新版，而是 0.0.x 这种远古版本。** 原因是这条依赖链：

```
LangChain 最新版（v0.3+）
  → 强制依赖 pydantic v2
    → 强制依赖 pydantic-core（Rust 编译的 .so 动态库）
      → PyPI 上没有 aarch64-linux-ohos 平台的预编译 wheel
        → pip 找不到能用的包
          → 自动降级到不依赖 pydantic v2 的最老版本
```

PyPI（Python 包仓库）上的预编译 wheel 只覆盖主流平台：`linux-x86_64`、`linux-aarch64`（标准 Linux）、`macosx`、`win_amd64`。**OpenHarmony 不在其中。** pip 找不到匹配当前平台的 pydantic-core wheel，又不能从源码现场编译（板子上没有 Rust 编译器），就只能 fallback 到全部由纯 Python 组成的旧版本。**0.0.x 版的 LangChain 和现在的 v0.3 已经不是同一个东西了**——API 全变了，功能差了几代。装上也没什么实际意义。

**大多数人到这里就放弃了。** 要跑最新版，就必须自己动手交叉编译那几个 `.so`。

pydantic v2 的核心是 pydantic-core，用 Rust 写的，编出来是个 `.so` 动态库。而且 pydantic 官方态度很明确：**没有纯 Python fallback，不做，别问。**

问题来了：静态链接的 Python 不能 `dlopen`。不能 `dlopen` 就不能加载 `.so`。不能加载 `.so` 就没有 pydantic-core。没有 pydantic-core 就没有 pydantic v2。没有 pydantic v2 就没有 LangChain。

**一环扣一环，全卡在"静态 vs 动态"这一个选择上。**

━━━━━━━━━━━━━━━━━━━━

◆ 为什么当初选了静态链接

━━━━━━━━━━━━━━━━━━━━

当时的思路很"合理"：OpenHarmony 用的是魔改过的 musl libc，有个叫 `ld-musl-namespace` 的机制——动态库加载受命名空间隔离限制。听起来很可怕，感觉动态链接上去就会被系统拦住。

静态链接多省事——把 musl、OpenSSL、zlib 全部打进一个二进制，11MB，往板子上一扔就跑。不用管动态链接器路径，不用管命名空间，不用管 `LD_LIBRARY_PATH`。

**但省事是有代价的：静态链接的 python 是个"封闭系统"，编译时没打进去的模块，运行时永远加载不了。**

当时把 31 个关键模块全编成了内建（builtin），塞进二进制。够用了——对标准库来说。但 pydantic-core 这种第三方 Rust 扩展，不可能在编译 CPython 时就预知并打进去。

所以：**要装 LangChain，必须让 Python 能加载 .so，必须改成动态链接。**

━━━━━━━━━━━━━━━━━━━━

◆ 恐惧是多余的

━━━━━━━━━━━━━━━━━━━━

改之前最大的顾虑就是 `ld-musl-namespace`。

实际去读了 OH 的命名空间配置文件 `ld-musl-namespace-aarch64.ini`——

发现 `/data/local/tmp` 走的是 `acquiescence`（默认）命名空间，**没有任何隔离限制**。`LD_LIBRARY_PATH` 直接能用。

💡 就好比你一直以为小区门口有保安查身份证，绕了一大圈走后门进来的。结果有一天正面走过去——保安亭根本没人。

**之前选静态链接绕开 namespace 隔离，是在绕一个不存在的障碍。**

这是个教训：对未知系统的恐惧，经常比系统本身更可怕。不如直接试一下。

| 对比项 | 静态链接版 | 动态链接版 |
|:---|:---|:---|
| python3 大小 | 11MB（全打包） | 4.8KB（launcher）+ 4.8MB libpython |
| 扩展模块 | 31 个内建，写死在二进制里 | 64 个 .so，lib-dynload/ 目录 |
| 能否加载第三方 .so | 不能 | 能 |
| 需要 LD_LIBRARY_PATH | 不需要 | 需要 |
| 适合场景 | 跑标准库够用 | 装 pip 包、跑框架 |

◆ 三个 Segfault——像破案一样排查

━━━━━━━━━━━━━━━━━━━━

动态版编出来了，64 个 `.so` 扩展模块。推到板子上一跑——

`math` OK，`_json` OK，`_struct` OK……

`_ssl` **segfault**。`_hashlib` **segfault**。`_sha256` **segfault**。

程序都没报错信息，直接崩。segfault 是 C 语言世界里最让人头疼的错误——段错误，内存访问越界，系统直接杀进程。没有 traceback，没有 "FileNotFoundError"，就一个冷冰冰的 "Segmentation fault"。

开始排查。（以下排查过程由 Claude Code 主导——我描述现象，它提假设、写测试、跑工具。）

────────────────────

**假设 1：OpenSSL 没加 -fPIC**

动态库必须用 `-fPIC`（位置无关代码）编译——这是常识。之前编的 OpenSSL 是 `.a` 静态库，给静态链接用的，可能没加 `-fPIC`。

重新编了一遍 OpenSSL，确认加了 `-fPIC`。

推上去一跑——还是 segfault。

❌ 假设 1 排除。

────────────────────

**假设 2：OpenSSL 静态 .a 链进 .so 有问题**

`.a` 静态库被链接进 `.so` 动态库时，可能有 ABI（应用二进制接口）兼容性问题。那就不静态链了，把 OpenSSL 也编成 `.so` 动态库。

重新编了一遍 OpenSSL——这次是 `--shared`，出 `libssl.so.3` 和 `libcrypto.so.3`。

推上去一跑——还是 segfault。

❌ 假设 2 排除。

────────────────────

**假设 3：OpenSSL 本身在 OH 上跑不了**

是不是 OpenSSL 在 OHOS 的 musl 上根本有问题？写个独立 C 程序，直接调 OpenSSL API 试试。

```c
// test_openssl.c — 最小验证
#include <openssl/evp.h>
int main() {
    EVP_MD_CTX *ctx = EVP_MD_CTX_new();
    // ... 算个 SHA256 ...
    printf("OpenSSL works!\n");
    return 0;
}
```

交叉编译，推上板子一跑——**跑得好好的。**

❌ 假设 3 排除。

────────────────────

三个假设全排除了。OpenSSL 没问题，编译选项没问题，那问题出在哪？

━━━━━━━━━━━━━━━━━━━━

◆ 破案：能跑的和不能跑的，区别在哪？

━━━━━━━━━━━━━━━━━━━━

冷静下来，重新梳理事实：

- `math.so` → OK
- `_json.so` → OK
- `_md5.so` → OK
- `_blake2.so` → OK
- `_sha256.so` → **crash**
- `_sha1.so` → **crash**
- `_ssl.so` → **crash**

等等。`_md5` 和 `_blake2` 能跑，但 `_sha256` 不行？它们都是 hash 模块，逻辑差不多。区别到底是什么？

翻 CPython 的构建系统：`_md5` 和 `_blake2` 是通过 `setup.py` 编译的，`_sha256` 和 `_sha1` 是通过 `Setup.stdlib`（走 Makefile）编译的。

让 Claude 用 `readelf` 看看这些 `.so` 的依赖：

```bash
readelf -d _md5.so | grep NEEDED
# → libpython3.11.so.1.0 ✅

readelf -d _sha256.so | grep NEEDED
# → （没有 libpython） ❌
```

**找到了。能跑的模块链接了 `libpython3.11.so`，crash 的没链。**

────────────────────

为什么会这样？

OH 官方有一个 CPython 交叉编译的 patch，里面在 `setup.py` 里加了 `-lpython3.11`——这样通过 `setup.py` 编译的模块（`_md5`、`_blake2`、`math`、`_json`……）都会自动链接 libpython。

但 CPython 3.11 有些模块不走 `setup.py`，走 `Setup.stdlib`（一个配置文件），编译时走 Makefile 的规则。`_sha256`、`_sha1`、`_sha3`、`_hashlib`、`_ssl` 都在这批里。

Makefile 的 `BLDSHARED` 变量定义了编 `.so` 时的链接命令。默认值里**没有 `-lpython3.11`**。

OH 的 patch 只改了 `setup.py`，**漏了 Makefile**。

━━━━━━━━━━━━━━━━━━━━

◆ 根因：OHOS musl 的 dlopen 不继承符号

━━━━━━━━━━━━━━━━━━━━

但等一下——在标准 Linux（glibc）上，CPython 的 `.so` 扩展模块从来不需要显式链接 `libpython`。为什么？

因为 `python3` 进程本身就是 `libpython` 编出来的（或者动态链接了它）。当 Python 调 `dlopen("_sha256.so")` 时，glibc 的动态链接器会让新加载的 `.so` "看到" 进程里已有的所有符号。`_sha256.so` 里用到的 `PyModule_Create`、`PyArg_ParseTuple` 这些 Python C API 函数，已经在进程的符号表里了，直接就能找到。

💡 打个比方：在普通 Linux 上，`dlopen` 就像你进了一间大办公室，办公室里所有人你都认识，需要谁的帮忙直接喊一声就行。

**但 OHOS 的 musl 不是这样。** 它的 `dlopen` 不会让新加载的 `.so` 自动"看到"调用者进程的符号。每个 `.so` 必须自己声明"我需要 libpython"——也就是在链接时加 `-lpython3.11`。

💡 在 OHOS 上，`dlopen` 更像是你被空降到一个陌生的办公室，谁也不认识。你得自带通讯录（显式链接依赖），不然谁也找不到。

这就是根因：

| | 标准 Linux (glibc) | OpenHarmony (musl) |
|:---|:---|:---|
| dlopen 行为 | 新 .so 能看到进程已有符号 | 新 .so **看不到**进程符号 |
| 扩展模块需不需要链 libpython | 不需要 | **必须** |
| 不链会怎样 | 正常运行 | **segfault** |

━━━━━━━━━━━━━━━━━━━━

◆ 一行修复

━━━━━━━━━━━━━━━━━━━━

知道了根因，修复就简单了。在 Makefile 的 `BLDSHARED` 里加上 `-L. -lpython3.11`：

```makefile
BLDSHARED = $(CC) -shared $(PY_CORE_LDFLAGS) -L. -lpython3.11
```

这样所有通过 Makefile 编译的 `.so` 模块也会链接 libpython。

重新 `make`，推上板子：

```
_ssl: OpenSSL 3.3.2 ✅
_hashlib: OK ✅
_sha256: OK ✅
socket: OK ✅
ALL DYNAMIC IMPORTS OK!
```

DeepSeek API 调用——成功。

**从 segfault 到全通，就改了一行。** 但找到这一行，人机协作花了大半天。

━━━━━━━━━━━━━━━━━━━━

◆ 依赖链分析：哪些要编，哪些不用编

━━━━━━━━━━━━━━━━━━━━

segfault 修完了，动态链接版 CPython 能加载 `.so` 了。下一步是 LangChain。

在动手编译之前，先得搞清楚 LangChain 的依赖树里到底有多少硬骨头。不能上来就一个个编——先分析，再动手。

Claude Code 把 langchain-core 的完整依赖树拉出来，逐个分类。结果比预想的乐观得多：

**需要交叉编译的（4 个）：**

| 库 | 底层语言 | 构建工具 | 作用 |
|:---|:---|:---|:---|
| pydantic-core | Rust | maturin + PyO3 | 数据校验引擎，LangChain 骨架 |
| uuid-utils | Rust | maturin + PyO3 | 快速 UUID 生成 |
| jiter | Rust | maturin + PyO3 | 快速 JSON 解析 |
| xxhash | C | 手动编译 | 快速哈希 |

**有纯 Python fallback、不用编的（4 个）：**

| 库 | 本来的底层语言 | 为什么不用编 |
|:---|:---|:---|
| orjson | Rust | langsmith 有完整的 json fallback，import 失败自动走标准库 json |
| zstandard | C | 压缩传输用的，没有就不压缩，功能不受影响 |
| pyyaml | C (libyaml) | 没有 libyaml 时自动退回纯 Python 解析器 |
| charset-normalizer | C | 有纯 Python 实现，C 版只是加速 |

**纯 Python 直接丢进去的（20+ 个）：**

pydantic、langchain-core、langsmith、openai、httpx、httpcore、requests、anyio、certifi、idna、h11、sniffio、jsonpatch、packaging、tenacity、typing-extensions、typing-inspection、annotated-types、distro、tqdm……

**4 个要编的，4 个不用编的，20 多个直接丢。** 需要交叉编译的硬骨头比想象中少得多。Python 生态的一个好习惯是：很多库的 C/Rust 扩展只是加速层，纯 Python 版本作为 fallback 始终保留着。

类比一下：**搭厨房。灶台（CPython）、燃气管（OpenSSL）、几口锅（pydantic-core 等 4 个编译的 .so）需要安装一次。之后做菜（装纯 Python 包）就是往锅里丢食材，不用再装管道了。**

━━━━━━━━━━━━━━━━━━━━

◆ 第一关：pydantic-core Rust 交叉编译

━━━━━━━━━━━━━━━━━━━━

pydantic-core 是整条依赖链里唯一没有 fallback 的硬骨头。必须编。

好消息是 `aarch64-unknown-linux-ohos` 是 Rust 官方 Tier 2 支持的 target。一行命令就能装：

```bash
rustup target add aarch64-unknown-linux-ohos
```

构建工具用 maturin——这是 PyO3 生态的标准构建器，专门把 Rust 代码编译成 Python 扩展模块（`.so`）。工具链：Rust 1.94 + maturin 1.12.6，编译 pydantic-core 2.41.5。

Rust 这边需要在 `~/.cargo/config.toml` 里告诉 cargo：编 ohos target 时，链接器不要用默认的 `cc`，用 OH 的 clang：

```toml
[target.aarch64-unknown-linux-ohos]
linker = "/path/to/ohos-sdk/native/llvm/bin/clang"
```

然后 maturin build，指定 target：

```bash
maturin build --release --target aarch64-unknown-linux-ohos --skip-auditwheel
```

`--skip-auditwheel` 是因为 auditwheel（wheel 兼容性检查工具）不认识 ohos 平台，会报错。跳过就好。

编译过程很顺利——Rust 编译器吃完 pydantic-core 的几万行 Rust 代码，几分钟后吐出一个 `.so`。

推上板子，`import pydantic_core`——

```
ImportError: ... PyInterpreterState_Get: symbol not found
```

────────────────────

**老朋友又来了。**

`PyInterpreterState_Get` 是 CPython 内部的 C API 函数。这个报错的意思是：pydantic_core.so 在运行时找不到 CPython 的符号。

听着耳熟吗？**和前面 `_sha256.so` segfault 是同一个根因**——OHOS musl 的 `dlopen` 不继承调用者进程的符号表。`_sha256.so` 找不到 `PyModule_Create` 会 segfault，`pydantic_core.so` 找不到 `PyInterpreterState_Get` 会报 ImportError（Rust 的 PyO3 做了更好的错误处理，不会直接 segfault，而是抛异常）。

**病因一模一样：`.so` 没有显式链接 libpython。** 之前编 CPython 自己的模块时，修法是在 Makefile 的 `BLDSHARED` 加 `-lpython3.11`。现在编第三方 Rust 扩展，修法是告诉 Rust 编译器在链接时加上 libpython：

```bash
export RUSTFLAGS="-L /path/to/libpython -l python3.11"
```

重新 `maturin build`，推上板子：

```python
>>> import pydantic_core
>>> pydantic_core.__version__
'2.41.5'
```

**过了。** 第一次遇到 OHOS musl dlopen 的坑时，花了大半天排查三个假设才找到根因。第二次遇到，10 分钟解决。**踩过的坑会变成直觉。**

━━━━━━━━━━━━━━━━━━━━

◆ 第二关 + 第三关：uuid-utils、jiter、xxhash

━━━━━━━━━━━━━━━━━━━━

有了 pydantic-core 的模板，后面两个 Rust 包就是复制粘贴。

**uuid-utils**（Rust/maturin）和 **jiter**（Rust/maturin）：完全照搬 pydantic-core 的编译配置——同样的 cargo config 指定 OH clang 做 linker，同样的 `RUSTFLAGS="-L ... -l python3.11"`，同样的 `maturin build --target aarch64-unknown-linux-ohos --skip-auditwheel`。一次过，没踩新坑。

**xxhash**（C 扩展）：这个更简单。xxhash 的 Python 绑定把 xxhash.c 源码直接内嵌在包里了。一行 OH clang 命令就能编出 `.so`：

```bash
${OHOS_CLANG} -shared -fPIC -O3 \
    -I${PYTHON_INCLUDE} \
    -L${PYTHON_LIB} -lpython3.11 \
    xxhash/*.c -lxxhash -o xxhash/_xxhash.so
```

注意 `.so` 要放到 `xxhash/` 包目录里面，不是放到顶层。Python 的 import 机制是按包目录找的。

四个需要编译的包，从开始到全部编完，不到 20 分钟。

━━━━━━━━━━━━━━━━━━━━

◆ 一波推上板子

━━━━━━━━━━━━━━━━━━━━

4 个编译好的 `.so` + 20 多个纯 Python 包，打个 tar 包，hdc 推上板子。

```bash
tar czf langchain_deps.tar.gz pydantic/ pydantic_core/ langchain_core/ \
    openai/ httpx/ httpcore/ langsmith/ ... # 省略一堆
hdc file send langchain_deps.tar.gz /data/local/tmp/
```

然后在板子上解压到 Python 的 `site-packages` 目录。

第一次 `import langchain_core`——报错，缺 `typing_inspection`。补上。
再跑——缺 `annotated_types`。补上。
再跑——缺 `_sysconfigdata`。这是 CPython 编译时生成的系统配置模块，之前打包时漏掉了。补上。

**这种"跑一次缺一个"的循环在移植工作里太常见了。** 不是偷懒，是依赖树太深，总有漏网之鱼。好在每次缺的都是纯 Python 文件，从宿主机上拷过去就行，不用重新编译。

◆ 复盘

━━━━━━━━━━━━━━━━━━━━

整个过程分两个阶段：前半段花了大半天，把 CPython 从静态链接改成动态链接，排查三个 segfault，最终定位到 OHOS musl 的 dlopen 不继承符号这个根因，改了一行 Makefile 修好。后半段只花了半小时——编 4 个 Rust/C 扩展，装 20 多个纯 Python 包，LangChain 全链路打通。

后半段之所以快，是因为前半段已经把最难的坑踩完了。pydantic-core 推上板子报 `PyInterpreterState_Get: symbol not found`，和前面 `_sha256` segfault 是同一个根因，10 分钟就解决了。**踩过的坑会变成直觉。**

回头看，几个教训值得记住：

**不要基于恐惧做架构决策。** 选静态链接是怕 `ld-musl-namespace`。没验证就怕了。结果 `/data/local/tmp` 根本没有隔离。直接试一下比猜半天强。

**排查 bug 的最有效手段不是猜原因，是找差异。** 三个假设都在查"有什么坏了"——OpenSSL 有没有 -fPIC？.a 有没有 ABI 问题？OpenSSL 本身行不行？全是在外围转圈。真正打开突破口的是一个简单的观察：`_md5` 能跑但 `_sha256` 不行。同类模块，一个行一个不行，区别一定在编译方式上。`readelf` 一看就破案了。

**平台差异藏在 C 库层。** glibc 和 musl 的 `dlopen` 行为不一样。这种差异不写在文档首页，不会有编译报错，不会有 warning。它藏在"程序在一个平台能跑，换一个就 segfault"的现象背后。做跨平台移植，C 库层面的行为差异是最隐蔽的坑。

**官方 patch 不一定全。** OH 的 CPython patch 改了 `setup.py` 加 `-lpython3.11`，但漏了 Makefile。这种"改了一半"的 patch 很常见——写 patch 的人可能只测了 `setup.py` 编出来的模块，没测 `Setup.stdlib` 编出来的。不能假设官方 patch 覆盖了所有情况。

**人机分工**——整个过程是我提需求、在板子上验证，Claude Code 写编译脚本、分析依赖树、排查报错。板子上 `import xxx` 报什么错，贴过来，Claude Code 判断原因、给修法。这种"一个人拿着板子，一个人看着代码"的分工，比一个人两头跑快得多。

◆ 术语速查

━━━━━━━━━━━━━━━━━━━━

| 术语 | 解释 |
|:---|:---|
| 静态链接 | 编译时把所有依赖库打包进一个二进制文件。大、自包含、不能动态加载新模块 |
| 动态链接 | 运行时从 .so 文件加载库。小、灵活、需要 LD_LIBRARY_PATH 指路 |
| dlopen | C 语言的动态库加载函数。`import _ssl` 底层就是调它 |
| segfault | Segmentation Fault，段错误。程序访问了不该访问的内存，被操作系统直接杀掉。没有错误信息，只有死亡 |
| -fPIC | Position Independent Code，位置无关代码。编动态库必须加的编译选项 |
| musl | 一种轻量级 C 标准库，Alpine Linux 和 OpenHarmony 都用它（但 OH 魔改了） |
| glibc | GNU C Library，Ubuntu/CentOS 等主流 Linux 发行版用的 C 标准库 |
| ld-musl-namespace | OH 的动态库命名空间隔离机制。限制哪些目录的程序能加载哪些 .so |
| readelf | 查看 ELF 二进制文件信息的工具。`readelf -d xxx.so` 能看它依赖了哪些库 |
| BLDSHARED | CPython Makefile 里的变量，定义编译 .so 扩展模块时的链接命令 |
| pydantic-core | pydantic v2 的核心引擎，用 Rust 写的 .so 动态库。LangChain 的硬依赖 |
| Setup.stdlib | CPython 3.11 新增的模块配置文件，走 Makefile 编译，不经过 setup.py |
| maturin | Rust 生态的 Python 扩展构建工具。把 Rust 代码编译成 `.so`，打包成 wheel |
| PyO3 | Rust 和 Python 之间的桥梁库。让 Rust 代码能直接调用 Python C API |
| wheel | Python 的二进制分发格式（`.whl` 文件）。本质是个 zip，里面装着编好的 `.so` 和 `.py` |
| --skip-auditwheel | maturin 的选项，跳过 wheel 兼容性检查。ohos 平台不在检查列表里，不跳会报错 |
| RUSTFLAGS | Rust 编译器的环境变量，用来传递额外的链接选项。比如 `-L path -l python3.11` |
| fallback | 后备方案。很多 Python 库的 C/Rust 扩展只是加速层，import 失败时自动退回纯 Python 实现 |
